PROJECT 3: THE DEVELOPER WHO COULDN'T ACCESS THE SERVER — EC2 Instance Launch & SSH Configuration

🔴 HYPOTHETICAL PROBLEM

→ ShopNow (from Project 1 and 2) now has their S3 bucket secured with IAM policies
→ The development team says: "We need a server to run our product catalog web application"
→ Previously — they ran the app on a physical server under the CTO's desk
→ That server overheated twice, crashed once, and nobody could access it from home
→ The CTO decides: "Let's move to AWS — spin up a virtual server"
→ priya-dev gets the task — she launches an EC2 instance from the console
→ Instance shows "Running" — but she CANNOT connect to it via SSH
→ She tries the public IP → "Connection timed out"
→ She tries the key pair → "Permission denied (publickey)"
→ She restarts the instance → the public IP CHANGES → bookmarks break
→ She stops the instance overnight to save money → starts it next morning → IP changed again
→ She terminates the instance by accident thinking "Terminate" means "Stop" → instance is GONE permanently
→ The CTO asks: "Why is something as simple as launching a server this complicated?"

🟡 HOW TO APPROACH IT

→ EC2 (Elastic Compute Cloud) is AWS's virtual server service
→ It gives you a virtual machine (VM) running on AWS physical hardware
→ You control the operating system, applications, networking, and storage

→ The problems Priya faced map to specific EC2 concepts:

Burstable Instances (T-family) — Critical to understand:

→ t3.micro has a BASELINE CPU performance (e.g., 10% of a full vCPU)
→ When your app needs more → it "bursts" above baseline using CPU credits
→ CPU credits accumulate when instance is BELOW baseline
→ CPU credits are consumed when instance is ABOVE baseline
→ If credits run out → performance is capped at baseline → app becomes slow

→ For ShopNow's dev server → t3.micro is perfect (low traffic, occasional bursts)
→ For production with consistent load → use m family (no bursting concept — fixed performance)

For ShopNow:
→ Choose: t3.micro (1 vCPU, 1 GB RAM, free tier eligible)

STEP 3 — UNDERSTAND KEY PAIRS (SSH AUTHENTICATION)

→ AWS EC2 uses KEY-BASED authentication — NOT password-based
→ This is MORE secure than passwords because:

  • Keys are 2048-bit or 4096-bit RSA/ED25519 — virtually uncrackable
  • No password to guess, brute-force, or phish
  • Private key never leaves your machine

STEP 6 — UNDERSTAND PUBLIC IP vs ELASTIC IP

The Problem:
→ Priya stops her instance at night → starts it in the morning → IP changed
→ All SSH configs, bookmarks, DNS entries break

Why does the IP change?

→ When you launch with "Auto-assign public IP: Enable"
→ AWS assigns a PUBLIC IP from its pool
→ This IP is EPHEMERAL — it belongs to AWS, not to you
→ When you stop the instance → the IP is released back to AWS's pool
→ When you start again → AWS assigns a DIFFERENT IP from the pool

Solution: Elastic IP (EIP)

→ An Elastic IP is a STATIC public IPv4 address that YOU own
→ It stays the same until you explicitly release it
→ You can attach it to any instance and detach/reattach freely

Configuration:

→ EC2 Console → Elastic IPs (left sidebar under Network & Security) → "Allocate Elastic IP address"

SettingValue
Network border groupap-south-1 (your region)
Public IPv4 address poolAmazon's pool of IPv4 addresses

→ Click "Allocate"
→ You get an IP like 13.234.176.89
→ This IP is NOW YOURS until you release it

Associate Elastic IP to Instance:

→ Select the Elastic IP → Actions → "Associate Elastic IP address"

SettingValue
Resource typeInstance
Instanceshopnow-product-catalog-dev (select from dropdown)
Private IP addressAuto-populated (the instance's private IP)
Allow reassociationCheck (allows moving this EIP to another instance)

→ Click "Associate"
→ Now the instance has a FIXED public IP: 13.234.176.89
→ Stop/start the instance → IP stays the same
→ This solves Priya's changing IP problem

CRITICAL Pricing Rule:

EIP StateCost
Associated with a RUNNING instanceFREE
Associated with a STOPPED instanceCHARGED (~$3.65/month)
Allocated but NOT associated with any instanceCHARGED (~$3.65/month)

→ AWS charges for WASTED Elastic IPs — to discourage hoarding IPv4 addresses
→ IPv4 addresses are scarce → AWS wants you to use them or release them
→ Additionally: Starting February 2024, AWS charges $0.005/hour for ALL public IPv4 addresses (including auto-assigned)

Connection Verification Checklist:

CHECK 1: Is the instance in "Running" state?
         EC2 Console → Instances → Status: "Running" ✓

CHECK 2: Does the Security Group allow inbound SSH?
         Instance → Security tab → Inbound rules → Port 22 from your IP ✓

CHECK 3: Is the public IP / Elastic IP assigned?
         Instance → Details → Public IPv4 address: 13.234.176.89 ✓

CHECK 4: Is the key file permissions correct?
         Local machine → file permission is 400 (read-only owner) ✓

CHECK 5: Are you using the correct username?
         Amazon Linux = ec2-user (NOT root, NOT admin, NOT ubuntu) ✓

CHECK 6: Is port 22 open on the OS-level firewall?
         Amazon Linux 2023 → SSH enabled by default ✓
         (Some AMIs may have iptables/firewalld blocking — rare for AWS AMIs)

→ If ALL 6 checks pass → SSH connection succeeds
→ Priya's failures mapped to: CHECK 2 (no SG rule), CHECK 4 (wrong permissions), CHECK 5 (wrong username)

→ Every EC2 instance can query information ABOUT ITSELF
→ This is called the Instance Metadata Service (IMDS)
→ Available at a special link-local address: http://169.254.169.254/latest/meta-data/
→ This address is ONLY accessible from WITHIN the instance — not from the internet

IMDSv1 vs IMDSv2:

VersionHow It WorksSecurity Level
IMDSv1Simple GET request to 169.254.169.254Vulnerable to SSRF attacks (Server-Side Request Forgery)
IMDSv2Requires a TOKEN first (PUT request), then GET with token in headerProtected against SSRF — token cannot be forwarded

IMDSv2 Flow:

Step 1: Request a token
        PUT http://169.254.169.254/latest/api/token
        Header: X-aws-ec2-metadata-token-ttl-seconds: 21600
        → Returns: temporary token (valid for specified TTL)

Step 2: Use token to query metadata
        GET http://169.254.169.254/latest/meta-data/instance-id
        Header: X-aws-ec2-metadata-token: {token from step 1}
        → Returns: i-0abcdef1234567890

→ We configured "Metadata version: V2 only (IMDSv2)" during launch
→ This BLOCKS IMDSv1 requests — only token-based (IMDSv2) requests work
→ Always use IMDSv2 — IMDSv1 is a known attack vector (Capital One breach 2019 exploited IMDSv1)

Why Metadata Matters:
→ Applications running on EC2 use metadata to discover their own environment
→ IAM Role credentials are delivered through metadata (Project 11)
→ Auto-scaling scripts use metadata to identify which instance they're running on
→ Configuration management tools use metadata for dynamic setup

✅ WHAT SHOULD DO

→ Always enable Termination Protection on important instances
→ Always use Elastic IPs for instances that need stable public addresses
→ Always use IMDSv2 only — disable IMDSv1
→ Always set "Shutdown behavior" to "Stop" not "Terminate"
→ Always use key pairs — never rely on password-based SSH
→ Always restrict Security Group SSH source to specific IPs — never 0.0.0.0/0
→ Always store private key files securely — treat them like passwords
→ Always tag instances with Name, Environment, Owner, Department
→ Always choose gp3 over gp2 for new EBS volumes — better performance, lower cost
→ Always right-size instances — start small, monitor, scale up if needed
→ Always use Session Manager when possible — reduces network attack surface
→ Always verify the correct default SSH username for your chosen AMI

❌ WHAT SHOULD NOT DO

→ Never launch instances without a key pair (unless using Session Manager exclusively)
→ Never set SSH Security Group source to 0.0.0.0/0 — bots scan for open port 22 constantly
→ Never confuse Stop with Terminate — Terminate is irreversible
→ Never leave Elastic IPs allocated but unassociated — you pay for waste
→ Never use t2 family for new instances — t3 is superior in every way (newer, cheaper, better networking)
→ Never store private key files in shared locations, email, or version control (Git)
→ Never leave IMDSv1 enabled — it's an SSRF attack vector
→ Never launch in the default VPC for production workloads — use custom VPC (Project 5)
→ Never skip Termination Protection for production instances
→ Never assume instance store data survives a stop — it doesn't (only EBS persists)
→ Never choose an instance type without understanding if you need burstable (t) or fixed (m) performance

2 → THE NITRO SYSTEM (AWS's Secret Weapon)

→ Pre-Nitro (before ~2017): AWS used Xen hypervisor

  • Xen consumed host CPU/RAM for virtualization overhead
  • Networking and storage I/O went through the hypervisor
  • Security relied on software isolation

→ Nitro System (current):

  • Custom hardware cards handle networking, storage, and security
  • The hypervisor is stripped to bare minimum — almost all host resources go to customer VMs
  • This is why modern instances (t3, m6i, c6i) are faster and cheaper than older generations
Nitro ComponentFunction
Nitro CardsOffload VPC networking, EBS storage I/O, instance monitoring
Nitro Security ChipHardware root of trust, prevents unauthorized firmware changes
Nitro HypervisorMinimal hypervisor — CPU and memory only, everything else offloaded
Nitro EnclavesIsolated compute environments for sensitive data processing

4 → HOW NETWORKING WORKS (ENI)

→ Every EC2 instance has at least one Elastic Network Interface (ENI)
→ The ENI is the actual network identity of the instance

ENI PropertyDescription

6 → WHAT HAPPENS DURING STOP vs TERMINATE INTERNALLY

STOP:

1. EC2 API sends "stop" signal to hypervisor
2. Guest OS receives ACPI shutdown signal
3. OS performs graceful shutdown
4. VM is deallocated from physical host
5. EBS volumes are DETACHED from compute host but remain intact
6. Private IP is RETAINED (stays associated with ENI)
7. Public IP is RELEASED (back to AWS pool)
8. Elastic IP stays associated (if one was attached)
9. Instance state: "Stopped"
10. Compute billing STOPS — EBS billing CONTINUES

TERMINATE:

1. EC2 API sends "terminate" signal
2. Guest OS receives shutdown signal
3. VM is deallocated and destroyed
4. EBS root volume is DELETED (if "delete on termination" = Yes)
5. Additional EBS volumes: depends on their "delete on termination" setting
6. ENI is deleted (primary ENI)
7. Private IP is released back to subnet pool
8. Public IP / Elastic IP is disassociated
9. Instance state: "Terminated" → eventually disappears from console
10. ALL billing stops
11. IRREVERSIBLE — instance cannot be recovered

📈 SCALABILITY CONSIDERATIONS

→ Vertical Scaling (Scale Up)

  • Stop instance → Change instance type → Start instance
  • Example: t3.micro → t3.large → m6i.xlarge
  • Requires downtime (stop/start)
  • Simple but has upper limits (largest instance: u-24tb1.metal with 24 TB RAM)

→ Horizontal Scaling (Scale Out)

  • Launch MORE instances behind a load balancer
  • Each instance handles a portion of traffic
  • No upper limit (theoretically)
  • Requires application to be stateless or use shared state (database, cache)
  • Auto Scaling Groups automate this (Project 33)

→ Instance Limits

  • Default: 20 On-Demand instances per instance family per region (soft limit)
  • Can request increase via Service Quotas console
  • Spot instances have separate limits

→ EBS Scalability

  • Maximum EBS volume size: 64 TiB
  • Can modify volume size, type, IOPS without detaching (online modification)
  • Takes time to apply (minutes to hours depending on size)

→ Network Scalability

  • Instance type determines network bandwidth
  • t3.micro: Up to 5 Gbps
  • m6i.32xlarge: 50 Gbps
  • Enhanced Networking (ENA): Higher PPS (packets per second), lower latency — enabled by default on modern instances
