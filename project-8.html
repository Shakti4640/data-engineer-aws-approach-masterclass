→ CRITICAL UNDERSTANDING: CloudWatch basic monitoring sees EC2 from the OUTSIDE (hypervisor level)
→ It CANNOT see INSIDE the guest OS
→ CPU utilization is visible because the hypervisor allocates CPU
→ Memory and disk are NOT visible because they're internal to the guest OS
→ This is why ShopNow's disk-full incident wasn't detected — no disk metric existed

→ The CloudWatch Agent (Project 19) runs INSIDE the EC2 instance → publishes memory, disk, and custom metrics

STEP 2 — UNDERSTAND NAMESPACES, DIMENSIONS, AND PERIODS

NAMESPACES:

→ A namespace is a CONTAINER for metrics — groups metrics by source
→ Format: AWS/{ServiceName} for AWS services

NamespaceSource
AWS/EC2EC2 instances
AWS/SQSSQS queues
AWS/SNSSNS topics
AWS/RDSRDS databases
AWS/S3S3 buckets

Custom/ShopNowYour custom application metrics

→ Custom namespaces: You choose the name (convention: Custom/{AppName})
→ Namespaces prevent metric name collisions (both EC2 and RDS have CPUUtilization but in different namespaces)

DIMENSIONS:

→ Dimensions are KEY-VALUE pairs that IDENTIFY which specific resource the metric belongs to
→ Think of dimensions as FILTERS

NamespaceMetricDimensionsWhat It Identifies
AWS/EC2CPUUtilizationInstanceId: i-0abc123CPU of THIS specific instance
AWS/EC2CPUUtilizationAutoScalingGroupName: shopnow-web-asgAVERAGE CPU across all instances in the ASG
AWS/SQSApproximateNumberOfMessagesVisibleQueueName: shopnow-orders.fifoMessages in THIS specific queue
AWS/RDSCPUUtilizationDBInstanceIdentifier: shopnow-dbCPU of THIS specific database
AWS/S3BucketSizeBytesBucketName: shopnow-productimages-prod-aps1, StorageType: StandardStorageSize of THIS bucket in Standard storage

→ A single metric name (e.g., CPUUtilization) with DIFFERENT dimensions = DIFFERENT metric streams
→ CPUUtilization for instance A is a completely separate data stream from CPUUtilization for instance B
→ When creating alarms → you MUST specify the correct dimensions to monitor the right resource

Dimension Combinations:

→ EC2 publishes CPUUtilization with multiple dimension sets:

  • Per instance: {InstanceId: i-0abc123}
  • Per Auto Scaling Group: {AutoScalingGroupName: shopnow-web-asg}
  • Per Image (AMI): {ImageId: ami-0abc123}
  • Per Instance Type: {InstanceType: t3.micro}

→ This allows you to create alarms at different granularities:

  • "Alert if THIS instance CPU > 90%" (instance dimension)
  • "Alert if AVERAGE CPU across the ASG > 80%" (ASG dimension)

PERIODS:

→ The time interval over which a data point is aggregated
→ Minimum: 1 second (high-resolution custom metrics)
→ Standard: 60 seconds (1 minute) or 300 seconds (5 minutes)

Monitoring TypePeriodCostAvailable For
Basic Monitoring300 seconds (5 minutes)FreeEC2 (default)
Detailed Monitoring60 seconds (1 minute)$3.50/instance/monthEC2 (must enable)
High Resolution1 secondCustom metric pricingCustom metrics only

→ Basic monitoring: Data point every 5 minutes → coarse-grained
→ Detailed monitoring: Data point every 1 minute → fine-grained
→ For production EC2 instances → enable Detailed Monitoring (catches spikes faster)

STEP 3 — BASIC vs DETAILED MONITORING

Basic Monitoring (Default for EC2):

→ Data points every 5 MINUTES
→ FREE — no extra cost
→ Sufficient for development/testing

Problem with 5-minute intervals:

Time 03:00 — CPU: 12% (data point published)
Time 03:01 — CPU spikes to 100% (NOT recorded yet)
Time 03:02 — CPU: 100% (NOT recorded yet)
Time 03:03 — CPU: 100% (NOT recorded yet)
Time 03:04 — CPU: 100% (NOT recorded yet)
Time 03:05 — CPU: 100% (data point published — AVERAGE of last 5 min = ~82%)

CloudWatch sees: 12% at 03:00, 82% at 03:05
Reality: CPU was 100% for 4 minutes before CloudWatch noticed

→ A 4-minute outage could be missed or delayed in detection

Detailed Monitoring (Must Enable):

→ Data points every 1 MINUTE
→ Costs ~$3.50 per instance per month
→ Catches problems 5x faster

How to Enable:

→ During EC2 launch: Advanced details → Detailed monitoring: Enable
→ After launch: EC2 Console → Select instance → Actions → Monitor and troubleshoot → Manage detailed monitoring → Enable

→ For ShopNow production instances → ALWAYS enable Detailed Monitoring
→ For dev/test → Basic Monitoring is acceptable

Understanding Statistics:

→ Within each period, multiple data points may exist (especially with detailed monitoring)
→ The STATISTIC determines how those data points are aggregated into a single value

StatisticWhat It ComputesBest For
AverageMean of all data points in the periodCPU utilization, queue depth (typical state)
SumTotal of all data pointsRequest count, error count (total volume)
MinimumLowest data pointFinding if a metric EVER dropped (e.g., healthy host count)
MaximumHighest data pointFinding if a metric EVER spiked (e.g., latency)

SampleCountNumber of data pointsChecking if data is being reported

→ For CPU → Average makes sense (overall utilization)
→ For latency → p99 is better (catches tail latency that Average hides)
→ For error counts → Sum makes sense (total errors in the period)

Step 4c: Conditions (Threshold)

SettingValueExplanation
Threshold typeStaticFixed threshold value (vs Anomaly Detection)
Whenever CPUUtilization is...Greater thanDirection of breach
...than90Threshold value (percent)
Datapoints to alarm3 out of 5How many periods must breach before alarm triggers

Understanding "Datapoints to Alarm" (M out of N):

→ This prevents false alarms from brief spikes
→ "3 out of 5" means: Look at the last 5 evaluation periods → alarm ONLY if 3 or more breached the threshold

Period 1: CPU 45% → Below threshold ✓
Period 2: CPU 92% → ABOVE threshold ✗
Period 3: CPU 88% → Below threshold ✓
Period 4: CPU 95% → ABOVE threshold ✗
Period 5: CPU 91% → ABOVE threshold ✗

Breaching periods: 3 out of 5 → ALARM TRIGGERED 🚨

Compare with:
Period 1: CPU 45% → Below ✓
Period 2: CPU 92% → ABOVE ✗
Period 3: CPU 88% → Below ✓
Period 4: CPU 65% → Below ✓
Period 5: CPU 91% → ABOVE ✗

Breaching periods: 2 out of 5 → NOT triggered (need 3)

→ "1 out of 1" = Most sensitive — triggers on single breach (may cause alert fatigue)
→ "3 out of 3" = Conservative — requires 3 consecutive breaches (may delay detection)
→ "3 out of 5" = Balanced — allows 2 non-breaching periods (handles intermittent spikes)

STEP 5 — UNDERSTAND ALARM STATES AND EVALUATION LOGIC

Three Alarm States:

┌─────────────┐    Threshold    ┌─────────────┐
│             │    Breached      │             │
│     OK      │───────────────►│    ALARM     │
│             │                 │             │
│  (Metric is │◄───────────────│  (Metric is │
│   within    │    Threshold    │   beyond    │
│   limits)   │    Recovered    │   limits)   │
│             │                 │             │
└──────┬──────┘                 └──────┬──────┘
       │                               │
       │    No Data                     │    No Data
       │                               │
       ▼                               ▼
┌─────────────────────────────────────────────┐
│           INSUFFICIENT_DATA                  │
│                                             │
│  (Not enough data points to evaluate)       │
│  (Instance stopped? Metric not published?)  │
│                                             │
└─────────────────────────────────────────────┘
StateMeaningColor in Console
OKMetric is within acceptable rangeGreen
ALARMMetric has breached the thresholdRed
INSUFFICIENT_DATANot enough data to evaluateGrey

When Does INSUFFICIENT_DATA Occur?

CauseExample
Alarm just createdNot enough data points collected yet
Instance stoppedNo CPU metrics being published
Metric reporting gapCloudWatch Agent crashed → no custom metrics
Wrong dimensionAlarm references non-existent instance ID

→ INSUFFICIENT_DATA is often more concerning than ALARM
→ ALARM means something is wrong but you KNOW about it
→ INSUFFICIENT_DATA means you're BLIND — you don't know the state

How to Handle Missing Data:

→ During alarm creation → "Missing data treatment" setting:

OptionBehaviorUse Case
Treat missing data as missingDon't change alarm state — maintain current stateDefault — conservative
Treat missing data as breachingConsider missing = threshold breached → may trigger alarmInstance monitoring (if no data, instance may be down)
Treat missing data as not breachingConsider missing = within threshold → alarm goes to OKBatch job metrics (no data between runs is normal)
Treat missing data as ignoreSkip evaluation — maintain current stateSame as "missing" but explicitly stated

→ For EC2 CPU alarm → "Treat missing data as breaching"

  • If instance is dead → no CPU data → we WANT an alarm
    → For batch job metrics → "Treat missing data as not breaching"
  • No data between job runs is normal → don't alarm

Evaluation Logic Deep Dive:

Alarm Configuration:
  Metric: CPUUtilization
  Threshold: > 90%
  Datapoints to alarm: 3 out of 5
  Period: 300 seconds (5 minutes)
  Missing data treatment: Treat as missing

Evaluation Window = 5 periods × 300 seconds = 25 minutes

Timeline:
Period 1 (03:00): CPU 45% → NOT breaching
Period 2 (03:05): CPU 92% → BREACHING
Period 3 (03:10): (missing)  → Treated as MISSING (skipped)
Period 4 (03:15): CPU 95% → BREACHING
Period 5 (03:20): CPU 91% → BREACHING

Evaluation: Out of 5 periods, 3 are breaching, 1 is not, 1 is missing
Missing is skipped → 3 out of 4 evaluated → 3 breaching
3 >= 3 required → ALARM STATE TRIGGERED 🚨

→ When missing data is treated as "missing" → it's excluded from the count
→ The evaluation becomes: "3 out of 4 evaluated" instead of "3 out of 5"
→ If ALL periods are missing → alarm goes to INSUFFICIENT_DATA

STEP 8 — COMPOSITE ALARMS

→ Individual alarms can be noisy
→ A composite alarm combines MULTIPLE alarms with AND/OR logic

Problem:
→ CPU spikes briefly during deployments → triggers alarm → false alert
→ But CPU spike + queue depth spike = REAL problem
→ We want to alert ONLY when BOTH conditions are true simultaneously

Creating a Composite Alarm:

→ CloudWatch Console → Alarms → "Create alarm" → "Create composite alarm"

SettingValue
Alarm ruleALARM("shopnow-web-01-cpu-high") AND ALARM("shopnow-orders-queue-depth-high")
Alarm nameshopnow-critical-overload
ActionSNS: shopnow-ops-alerts

Alarm Rule Syntax:

ALARM("alarm-name")        → True when the named alarm is in ALARM state
OK("alarm-name")           → True when the named alarm is in OK state
INSUFFICIENT_DATA("alarm") → True when in INSUFFICIENT_DATA state

Operators: AND, OR, NOT, parentheses for grouping

Examples:

RuleMeaning
ALARM("cpu-high") AND ALARM("queue-deep")Both must be alarming
ALARM("cpu-high") OR ALARM("memory-high")Either one alarming triggers composite
ALARM("cpu-high") AND NOT ALARM("deployment-in-progress")CPU high but NOT during deployment
ALARM("a") AND (ALARM("b") OR ALARM("c"))A must be alarming AND either B or C

→ Composite alarms REDUCE alert fatigue by requiring multiple conditions
→ They make alerts more ACTIONABLE — if composite triggers, it's definitely a real issue

Suppressor Alarm:

→ A special feature of composite alarms
→ When the suppressor alarm is in ALARM state → the composite alarm actions are SUPPRESSED

SettingValue
Suppressor alarmshopnow-maintenance-window
Wait period60 seconds

→ During maintenance → set shopnow-maintenance-window to ALARM state
→ All composite alarms with this suppressor → won't trigger notifications
→ After maintenance → set suppressor back to OK → normal alerting resumes

✅ WHAT SHOULD DO

→ Always create alarms for critical metrics: CPU, status checks, queue depth, error rates
→ Always use "Datapoints to alarm" (M out of N) to prevent false alarms from brief spikes
→ Always configure BOTH "In alarm" AND "OK" notifications — know when problems START and RESOLVE
→ Always use appropriate statistics: Average for utilization, Sum for counts, Maximum for peaks
→ Always enable Detailed Monitoring for production EC2 instances (1-minute resolution)
→ Always set meaningful alarm names with resource identifier included
→ Always add descriptive alarm descriptions with remediation steps
→ Always configure "Missing data treatment" appropriately per metric type
→ Always create a production dashboard with all critical metrics on one screen
→ Always use composite alarms to reduce alert fatigue
→ Always monitor the monitoring — ensure alarms themselves are working
→ Always connect alarms to SNS topics (Project 6) for multi-channel notification
→ Always tag alarms for organizational tracking

❌ WHAT SHOULD NOT DO

→ Never rely solely on basic monitoring (5-minute) for production — use detailed monitoring
→ Never set "1 out of 1" datapoints for volatile metrics — creates alert fatigue
→ Never forget to configure "OK" actions — you need to know when problems resolve
→ Never use only email for critical alerts — add SMS for on-call (Project 6)
→ Never create alarms without descriptive names — debugging at 3 AM with "Alarm-1" is useless
→ Never ignore INSUFFICIENT_DATA state — it often means the resource is unreachable
→ Never set thresholds too aggressively — 80% CPU is normal for some workloads
→ Never create dozens of individual alarms without composite alarms — alert fatigue kills response time
→ Never monitor only infrastructure metrics — also monitor business metrics (order count, error rate)
→ Never forget that memory and disk metrics require CloudWatch Agent (Project 19) — not available by default
→ Never assume all metrics are free — custom metrics and detailed monitoring have costs
→ Never create alarms for metrics you won't act on — every alarm should have a runbook

Static Threshold vs Anomaly Detection:

ApproachHow It WorksBest For
Static threshold"Alert if CPU > 90%"Known, fixed limits
Anomaly DetectionML model learns normal pattern → alerts on deviationMetrics with daily/weekly patterns (traffic varies by time of day)

→ Anomaly Detection: CloudWatch builds a model of expected metric behavior
→ Example: Traffic is normally low at 3 AM and high at 3 PM
→ If traffic is unusually HIGH at 3 AM → anomaly → alarm
→ With static threshold → 3 AM traffic would be "normal" (below threshold) but actually abnormal for that time

Configuring Anomaly Detection:

→ When creating alarm → Threshold type: "Anomaly detection"
→ CloudWatch uses 2 weeks of historical data to build the model
→ "Band" width controls sensitivity (1 = very sensitive, 10 = less sensitive)
→ Good for: Request counts, latency, traffic patterns
→ Not good for: Brand new metrics with no history

1 → HOW METRICS ARE COLLECTED AND STORED

AWS SERVICE (e.g., EC2)
        │
        │  Every 5 min (basic) or 1 min (detailed):
        │  Publishes metric data point to CloudWatch
        │
        ▼
CLOUDWATCH METRICS SERVICE
        │
        ├── Receives data point:
        │   Namespace: AWS/EC2
        │   MetricName: CPUUtilization
        │   Dimensions: {InstanceId: i-0abc123}
        │   Value: 78.5
        │   Timestamp: 2025-01-15T03:17:00Z
        │   Unit: Percent
        │
        ├── Stores in TIME SERIES DATABASE
        │   (Distributed across multiple AZs)
        │
        └── Data retention:
            1-second resolution: 3 hours
            60-second resolution: 15 days
            300-second (5-min) resolution: 63 days
            3600-second (1-hour) resolution: 455 days (15 months)

Data Retention — CRITICAL to understand:

Original ResolutionKept ForThen Aggregated ToKept For
1 second (high-resolution)3 hours60 seconds15 days
60 seconds (detailed/standard)15 days5 minutes63 days
5 minutes63 days1 hour455 days

→ After 15 days → 1-minute data points are AGGREGATED into 5-minute summaries
→ You LOSE the 1-minute granularity after 15 days
→ After 63 days → 5-minute data is aggregated into 1-hour summaries
→ After 455 days → data is DELETED

→ This means: If you're investigating an incident from 2 months ago → you only have 5-minute granularity
→ For long-term storage → export metrics to S3 (via Metric Stream or GetMetricData API)

2 → HOW ALARM EVALUATION WORKS

Alarm: shopnow-web-01-cpu-high
  Metric: CPUUtilization, InstanceId: i-0abc123
  Period: 300 seconds
  Statistic: Average
  Threshold: > 90
  Datapoints to alarm: 3 out of 5

EVALUATION CYCLE (runs every Period = 300 seconds):

Time 03:25:00 — Evaluation triggered
        │
        ▼
Step 1: Retrieve last 5 data points (evaluation range)
        │
        ├── 03:05 → Average CPU: 45.2%
        ├── 03:10 → Average CPU: 89.1%
        ├── 03:15 → Average CPU: 95.3%
        ├── 03:20 → Average CPU: 92.7%
        └── 03:25 → Average CPU: 91.5%
        │
        ▼
Step 2: Compare each to threshold (> 90)
        │
        ├── 03:05: 45.2% → NOT breaching
        ├── 03:10: 89.1% → NOT breaching
        ├── 03:15: 95.3% → BREACHING ✗
        ├── 03:20: 92.7% → BREACHING ✗
        └── 03:25: 91.5% → BREACHING ✗
        │
        ▼
Step 3: Count breaching data points: 3
        Required: 3 out of 5
        3 >= 3 → THRESHOLD MET
        │
        ▼
Step 4: Check current alarm state
        │
        ├── If currently OK → Change to ALARM → Execute ALARM actions
        │   → Send SNS notification: "ALARM: shopnow-web-01-cpu-high"
        │   → Execute EC2 actions (if configured)
        │
        └── If already ALARM → No state change → No re-notification
            (Unless "Actions enabled" is set to trigger on every evaluation)
        │
        ▼
Step 5: Record state change in alarm history
        Visible in: CloudWatch Console → Alarms → Select alarm → History tab

→ Alarms evaluate on a FIXED schedule (every Period)
→ They do NOT evaluate continuously — there's a delay up to one full period
→ With 5-minute periods → alarm may take up to 5 minutes to detect a breach

3 → HOW ALARM ACTIONS WORK

Alarm transitions to ALARM state
        │
        ▼
Step 1: Check "Actions enabled" setting
        │
        ├── Disabled → No actions taken (useful during maintenance)
        │
        └── Enabled → Proceed to execute actions
        │
        ▼
Step 2: Execute notification actions
        │
        ├── SNS Publish: CloudWatch publishes to the configured SNS topic
        │   Message includes: alarm name, state, reason, timestamp, metric data
        │   SNS then fans out to subscribers (Project 6)
        │
        └── Multiple SNS topics can be targeted (one alarm → multiple topics)
        │
        ▼
Step 3: Execute service-specific actions
        │
        ├── EC2: Stop/Terminate/Reboot/Recover (API call to EC2)
        ├── Auto Scaling: Execute scaling policy (API call to Auto Scaling)
        └── Systems Manager: Run automation document (API call to SSM)

🚨 COMMON PITFALLS AND FAILURE MODES

#PitfallWhat HappensHow to Avoid
1Assuming memory/disk metrics exist by defaultAlarm on non-existent metric → INSUFFICIENT_DATA foreverInstall CloudWatch Agent for OS-level metrics (Project 19)
2Setting "1 out of 1" for volatile metricsEvery brief spike triggers alarm → alert fatigue → team ignores all alarmsUse "3 out of 5" or similar for volatile metrics
3Not configuring "OK" notificationTeam never knows when problem resolves → keeps investigating resolved issuesAlways add "OK" action to SNS topic
4Wrong statistic for metric typeAverage hides spikes, Sum doesn't make sense for percentagesAverage for utilization, Sum for counts, Maximum for peaks, p99 for latency
5Alarm on wrong dimensionAlarm watches aggregate when you want per-instanceAlways verify dimension matches the specific resource
6Missing data treated as "not breaching" for critical metricsInstance dies → no data → alarm stays OK → nobody knowsUse "breaching" treatment for instance health metrics
7Too many alarms → all going to one SNS topicOps team receives 50 emails/hour → ignores all of themUse filter policies (Project 6), composite alarms, and severity-based routing
8Alarms created but actions disabledAlarm transitions to ALARM but nobody is notifiedVerify "Actions enabled" is ON for all production alarms
9Dashboard not auto-refreshingTeam looks at stale data → thinks everything is fineSet auto-refresh to 1 minute on production dashboards
10Monitoring only AWS metrics, not application metricsInfrastructure is healthy but application is returning errorsPublish custom metrics for application health (error rate, latency, success rate)
11Forgot SNS topic access policy for CloudWatchAlarm triggers but SNS rejects the publish → silent failureAdd CloudWatch service principal to topic access policy (Project 6 Step 6)
12Metric resolution mismatch with alarm period5-minute metric with 1-minute alarm period → INSUFFICIENT_DATAAlarm period must be >= metric resolution

📈 SCALABILITY CONSIDERATIONS

→ Metric Scalability

  • CloudWatch handles billions of data points across millions of metrics
  • No provisioning needed — fully managed
  • Custom metrics scale to thousands per account
  • For very high cardinality → use Embedded Metric Format (EMF) in logs → extracted as metrics automatically

→ Alarm Scalability

  • 5,000 alarms per region per account (can request increase)
  • For large fleets → use composite alarms to reduce individual alarm count
  • For dynamic infrastructure (Auto Scaling) → create/delete alarms programmatically
  • Consider: Alarm per instance vs alarm per Auto Scaling Group
    • Per instance: More granular but more alarms to manage
    • Per ASG: Fewer alarms but less granular

→ Dashboard Scalability

  • 500 widgets per dashboard
  • Multiple dashboards for different teams/purposes
  • Cross-account dashboards for centralized monitoring (Project 60)
  • For very large fleets → use CloudWatch Explorer (auto-discovers resources)

→ Cost Scalability

  • Costs scale linearly with: number of metrics, alarms, dashboard count
  • Optimization: Use metric math to derive metrics instead of publishing redundant custom metrics
  • Optimization: Disable detailed monitoring on non-production instances
  • Optimization: Use CloudWatch Metric Streams to export to third-party tools (Datadog, Grafana) instead of duplicating monitoring

→ Multi-Region / Multi-Account Scalability

  • CloudWatch is REGIONAL — each region has its own metrics and alarms
  • Cross-region dashboards: Can add metrics from other regions to a single dashboard
  • Cross-account observability: CloudWatch supports sharing metrics across accounts (Project 60)
  • For enterprise: Centralized monitoring account with cross-account access

→ How to test an alarm WITHOUT waiting for a real incident:

Method 1: Use set-alarm-state CLI command
  → Manually force alarm to ALARM state
  → CloudWatch Console → Select alarm → Actions → "Set alarm state" → ALARM
  → Triggers all configured actions → verifies end-to-end notification
  → NOTE: This is temporary — alarm will re-evaluate on next cycle

Method 2: Stress test the resource
  → Run a CPU stress tool on EC2 → CPU spikes → alarm triggers naturally
  → This tests the full path: metric → alarm → action

Method 3: Check alarm evaluation in history
  → Select alarm → History tab → Look for "Alarm updated" and "State change" entries
  → Shows exactly which data points were evaluated and why the state changed

→ How to debug "alarm didn't fire when it should have":

STEP 1: Check alarm state
        Is it in OK, ALARM, or INSUFFICIENT_DATA?

STEP 2: Check alarm history
        Did it evaluate? What data points did it see?

STEP 3: Check metric data
        CloudWatch → Metrics → Find the metric → Graph it
        Did the metric actually breach the threshold?

STEP 4: Check statistic and period
        Are you using Average when Maximum would catch the spike?
        Is the period appropriate for the metric resolution?

STEP 5: Check "Datapoints to alarm"
        Did enough consecutive/near-consecutive breaches occur?
        "3 out of 5" requires 3 breaching — 2 breaches won't trigger

STEP 6: Check "Missing data treatment"
        If data points are missing — how are they treated?
        Missing treated as "not breaching" could prevent alarm

STEP 7: Check dimension
        Is the alarm watching the RIGHT resource?
        Wrong InstanceId → watching a different server

STEP 8: Check actions
        Is "Actions enabled" = Yes?
        Is the SNS topic ARN correct?
        Does CloudWatch have permission to publish to the SNS topic?
        (Topic access policy must allow cloudwatch.amazonaws.com — Project 6)

→ Metric Math for Derived Metrics:

→ CloudWatch supports mathematical expressions on metrics
→ Useful for: error rates, ratios, derived metrics

ExpressionMeaning
