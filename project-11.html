PROJECT 11: THE EC2 APP THAT NEEDED S3 WITHOUT HARDCODED KEYS — IAM Roles for EC2

🔴 HYPOTHETICAL PROBLEM

→ ShopNow (from Projects 1-10) now has:

  • S3 bucket with bucket policies enforcing encryption and HTTPS (Projects 1, 10)
  • IAM users with proper permissions (Project 2)
  • EC2 instances in a properly architected VPC (Projects 3, 4, 5)
  • SNS topics and SQS queues for messaging (Projects 6, 7)
  • CloudWatch monitoring (Project 8)
  • RDS MariaDB in private subnets (Project 9)

→ The web application running on EC2 needs to:

  • READ product images from S3 (serve to customers)
  • WRITE order confirmation PDFs to S3
  • SEND messages to SQS (order queue — Project 7)
  • PUBLISH notifications to SNS (alert topics — Project 6)
  • READ database credentials from Secrets Manager (future — Project 37)

→ Problem 1: priya-dev created an IAM user app-service-account with access keys
→ She placed the Access Key ID and Secret Access Key in a configuration file on the EC2 instance
→ The config file: /opt/shopnow/config/aws-credentials.conf
→ File permissions were set to 644 (readable by everyone on the server)
→ A compromised web application vulnerability allowed an attacker to read local files
→ The attacker read the credentials file → used the keys to download the ENTIRE S3 bucket
→ The keys were LONG-LIVED (never rotated) → the attacker had access for weeks before detection

→ Problem 2: The application was deployed to 5 EC2 instances behind a load balancer
→ Each instance had the SAME access keys hardcoded
→ When the keys needed rotation (security policy: every 90 days):

  • New keys had to be generated
  • Config files on ALL 5 instances had to be updated
  • Application had to be restarted on ALL 5 instances
  • During rotation → brief window where old keys are invalidated → application errors
  • One instance was missed during rotation → it stopped working → 20% of traffic failed

→ Problem 3: The development team copied the same credentials to the staging environment
→ The staging application accidentally wrote to PRODUCTION S3 bucket
→ Test data mixed with production data → customer reports contained garbage data
→ There was no way to distinguish which instance or environment made which API call

→ Problem 4: arjun-intern (read-only user — Project 2) found the credentials file during debugging
→ He used them to upload a test file to S3 → the credentials had broader permissions than his IAM user
→ The service account credentials bypassed his IAM restrictions

→ The CTO demands: "NO hardcoded credentials on ANY server. EVER. I want:

  • Credentials that auto-rotate without application changes
  • Credentials that are DIFFERENT per instance
  • Credentials that are TEMPORARY — even if stolen, they expire
  • No configuration files with secrets on any server
  • Full audit trail of which instance accessed which resource"

🟡 HOW TO APPROACH IT

→ This is a CREDENTIAL MANAGEMENT problem for compute resources
→ The solution: IAM Roles + Instance Profiles

→ An IAM Role is an identity that is NOT a user — it has no password and no permanent access keys
→ Instead, it provides TEMPORARY security credentials to whoever ASSUMES it
→ An EC2 instance can ASSUME a role → receive temporary credentials → use them to access AWS services

BEFORE (Hardcoded Keys — BROKEN):
┌───────────────────────────────┐
│ EC2 Instance                  │
│                               │
│ Config file:                  │
│   ACCESS_KEY=AKIA...          │  ← PERMANENT credentials
│   SECRET_KEY=wJal...          │  ← NEVER rotate automatically
│                               │  ← If stolen → unlimited access
│ Application reads config      │
│ → Uses keys for S3/SQS/SNS   │
└───────────────────────────────┘

AFTER (IAM Role — CORRECT):

│ EC2 Instance                  │
│                               │

│                               │  ← If stolen → useless after expiry
│ Application uses AWS SDK      │
│ → SDK auto-reads from IMDS    │
│ → No config files needed      │
└───────────────────────────────┘

STEP 1 — UNDERSTAND IAM ROLES vs IAM USERS

AspectIAM User (Project 2)IAM Role
Identity typePermanent identity for a person or serviceTemporary identity assumed by entities
AuthenticationPassword (console) + Access Keys (API)No password, no permanent keys
CredentialsLONG-LIVED (until rotated manually)TEMPORARY (auto-expire, auto-rotate)
Who uses itHumans, legacy applicationsEC2, Lambda, Glue, other AWS services, cross-account access
Can be assumed byNobody — it IS the identityEC2, Lambda, other roles, users, AWS services

→ KEY INSIGHT: IAM Users are WHO you are. IAM Roles are WHAT you can become.
→ A person IS a user. An EC2 instance ASSUMES a role.
→ Think of it like:

  • IAM User = Your permanent employee badge (always identifies you)
  • IAM Role = A temporary visitor pass (anyone authorized can wear it, expires after hours)

Components of an IAM Role:

ComponentPurposeAnalogy

Permission PolicyWHAT the role can do"What doors does this pass open?"
Session DurationHOW LONG the temporary credentials last"When does this pass expire?"
Instance ProfileBRIDGE between role and EC2"The badge holder that clips the pass to the EC2 instance"

STEP 2 — UNDERSTAND TRUST POLICIES

→ A Trust Policy is a JSON document that defines WHO can assume the role
→ It's attached to the ROLE (not to the user or instance)
→ It uses the same policy language but with specific "assume" actions

Trust Policy for EC2:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowEC2ToAssume",
      "Effect": "Allow",
      "Principal": {
        "Service": "ec2.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}

Every Field Explained:

→ Principal: Service: ec2.amazonaws.com

  • This says: "The EC2 service is allowed to assume this role"
  • It does NOT say which specific EC2 instance — ANY instance with this role attached can assume it
  • The restriction on WHICH instance comes from the Instance Profile attachment (Step 5)

→ Action: sts:AssumeRole

  • STS = Security Token Service
  • AssumeRole = "Give me temporary credentials for this role"
  • This is the ONLY action in a trust policy

Trust Policy Principals for Different Services:

ServiceTrust Policy PrincipalUse Case
EC2ec2.amazonaws.comEC2 instances accessing AWS services
Lambdalambda.amazonaws.comLambda functions (Project 26)
Glueglue.amazonaws.comGlue crawlers and jobs (Project 17)
Redshiftredshift.amazonaws.comRedshift cluster accessing S3 (Project 22)
RDSrds.amazonaws.comRDS Enhanced Monitoring
S3s3.amazonaws.comS3 replication (Project 30)
CloudWatchmonitoring.amazonaws.comCloudWatch Enhanced Monitoring
SNSsns.amazonaws.comSNS delivery logging
ECS Tasksecs-tasks.amazonaws.comContainer tasks
Another IAM Userarn:aws:iam::123456789012:user/priya-devUser assuming role for elevated access

→ Each service has its OWN service principal
→ A role can trust MULTIPLE principals (multiple statements or array of principals)
→ A role trusted by ec2.amazonaws.com CANNOT be assumed by Lambda → and vice versa

Trust Policy with Conditions:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ec2.amazonaws.com"
      },
      "Action": "sts:AssumeRole",
      "Condition": {
        "StringEquals": {
          "aws:RequestedRegion": "ap-south-1"
        }
      }
    }
  ]
}

→ Conditions in trust policies add extra restrictions on WHO can assume
→ This example: Only EC2 instances in ap-south-1 can assume this role
→ More condition examples will appear in cross-account scenarios (Project 40)

→ An Instance Profile is a CONTAINER that holds exactly ONE IAM Role
→ It's the BRIDGE between the IAM Role and the EC2 instance
→ You cannot attach a Role directly to EC2 — you attach an Instance Profile which contains the Role

Why Instance Profiles Exist (Historical Reason):

→ When IAM Roles were first created → AWS designed them to potentially hold multiple roles
→ The Instance Profile was the container that could theoretically hold multiple roles
→ In practice → Instance Profiles hold EXACTLY ONE role → the concept is mostly invisible
→ When you create a role through the AWS Console → an Instance Profile with the SAME NAME is auto-created
→ When you create a role through CLI/API → you must create the Instance Profile separately and add the role

Console Behavior:

ActionWhat Happens Behind the Scenes
Create role via ConsoleRole created + Instance Profile with same name auto-created
Attach role to EC2 via ConsoleInstance Profile is attached (console shows "IAM Role" but actually attaches the profile)
Create role via CLIONLY the role is created — you must separately create Instance Profile

→ In the Console → you never see "Instance Profile" → you always see "IAM Role"
→ In the CLI/API → you work with Instance Profiles explicitly
→ The Console abstracts the Instance Profile away for simplicity

STEP 6 — HOW CREDENTIALS ARE DELIVERED VIA IMDS

→ This connects directly to Instance Metadata (Project 3 — Step 8)
→ When an Instance Profile is attached → IMDS provides temporary credentials

Credential Delivery Flow:

EC2 Instance boots with Instance Profile attached
        │
        ▼
Step 1: EC2 service contacts STS (Security Token Service)
        "I am instance i-0abc123, I have role shopnow-web-server-role attached"
        │
        ▼
Step 2: STS validates:
        - Does this instance have this role attached? YES
        - Does the role trust ec2.amazonaws.com? YES (trust policy)
        │
        ▼
Step 3: STS generates TEMPORARY credentials:
        - AccessKeyId: ASIA... (starts with ASIA for temporary creds, not AKIA)
        - SecretAccessKey: (temporary secret)
        - SessionToken: (required for temporary creds — proves they're from STS)
        - Expiration: (typically ~6 hours from now)
        │
        ▼
Step 4: Credentials are placed in Instance Metadata Service (IMDS)
        Available at: http://169.254.169.254/latest/meta-data/iam/security-credentials/shopnow-web-server-role
        │
        ▼
Step 5: Application (or AWS SDK) reads credentials from IMDS
        │
        ▼
Step 6: Application uses credentials to make AWS API calls
        │
        ▼
Step 7: BEFORE credentials expire → EC2 service automatically requests NEW credentials from STS
        New credentials are placed in IMDS
        Application reads new credentials on next API call
        OLD credentials remain valid for a brief overlap period
        │
        ▼
Step 8: Continuous rotation — credentials are ALWAYS fresh

What the IMDS Response Looks Like:

GET http://169.254.169.254/latest/meta-data/iam/security-credentials/shopnow-web-server-role

Response:
{
  "Code": "Success",
  "LastUpdated": "2025-01-15T10:00:00Z",
  "Type": "AWS-HMAC",
  "AccessKeyId": "[REDACTED:AWS_ACCESS_KEY]",
  "SecretAccessKey": "abc123def456ghi789jkl012mno345",
  "Token": "FwoGZXIvYXdzEBYaDL3k9N...(very long string)...",
  "Expiration": "2025-01-15T16:00:00Z"
}

→ AccessKeyId starts with ASIA → indicates TEMPORARY credentials (permanent keys start with AKIA)
→ Token (SessionToken) → MUST be included with every API call → proves the credentials are from STS
→ Expiration → when these credentials become invalid → new ones are generated before this time
→ LastUpdated → when credentials were last refreshed

IMDSv2 Requirement (Project 3 Reference):

→ Credentials are served via IMDS → IMDS must be secured
→ IMDSv2 (token-required) prevents SSRF attacks from stealing credentials
→ We configured "Metadata version: V2 only" during instance launch (Project 3)
→ This means: Even if an attacker exploits an SSRF vulnerability → they CANNOT reach IMDS without a valid token
→ The token is generated with a PUT request → SSRF attacks typically use GET → blocked

Credential Refresh Timeline:

Time 10:00 — Credentials generated (expire at 16:00)
Time 14:30 — EC2 service pre-emptively requests new credentials
             (about 1.5 hours before expiry)
Time 14:31 — New credentials placed in IMDS (expire at 20:31)
             Old credentials (expire 16:00) still valid until 16:00
Time 16:00 — Old credentials expire
             New credentials continue working until 20:31
Time 19:00 — Another refresh cycle begins

→ There is ALWAYS a valid set of credentials in IMDS
→ The overlap period prevents any gap in access
→ Applications using AWS SDK never notice the rotation

STEP 7 — THE CREDENTIAL CHAIN (HOW AWS SDK FINDS CREDENTIALS)

→ AWS SDKs (used in application code) search for credentials in a specific ORDER
→ This is called the "Default Credential Provider Chain"
→ Understanding this chain prevents credential confusion

Credential Chain Order (SDK searches top to bottom, uses FIRST found):

PrioritySourceHow It's SetPersistence
1Explicit codeHardcoded in application codePERMANENT (DON'T DO THIS)
2Environment variablesAWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_SESSION_TOKENUntil shell session ends
3Shared credentials file~/.aws/credentials (profiles)Until file is changed
4AWS config file~/.aws/config (profiles with role_arn)Until file is changed
5Container credentialsECS task role (via container metadata)TEMPORARY
6Instance Profile (IMDS)IAM Role attached to EC2TEMPORARY — AUTO-ROTATED
7SSO credentialsIAM Identity CenterTEMPORARY

Why This Order Matters:

SCENARIO: Developer troubleshooting on the EC2 instance
→ Developer runs: export AWS_ACCESS_KEY_ID=AKIAXXXXX (their personal keys)
→ Developer runs an AWS CLI command → uses their PERSONAL keys (Priority 2)
→ Developer forgets to unset the environment variables
→ Application restarts → picks up environment variables → uses WRONG credentials
→ Application now uses the developer's keys instead of the role → WRONG permissions

SOLUTION: Never set environment variables on EC2 instances with IAM Roles
→ Let the SDK naturally fall through to Priority 6 (Instance Profile)
→ No environment variables → No credentials file → SDK uses IMDS → Correct role credentials

How the SDK Uses Instance Profile Credentials:

Application makes AWS API call (e.g., S3 GetObject)
        │
        ▼
AWS SDK: Do I have cached credentials that are still valid?
        │
        ├── YES (and not near expiry) → Use cached credentials
        │
        └── NO (or near expiry) → Refresh credentials
            │
            ▼
            SDK calls IMDS:
            GET http://169.254.169.254/latest/meta-data/iam/security-credentials/
            → Returns role name: "shopnow-web-server-role"
            │
            GET http://169.254.169.254/latest/meta-data/iam/security-credentials/shopnow-web-server-role
            → Returns: AccessKeyId, SecretAccessKey, Token, Expiration
            │
            ▼
            SDK caches these credentials
            SDK includes AccessKeyId + SecretAccessKey + Token in the API request
            (SigV4 signature — Project 2)
            │
            ▼
            AWS service receives request → validates credentials → authorizes via role's policies

→ This is COMPLETELY AUTOMATIC — no application code needed to manage credentials
→ The SDK handles: finding credentials, caching, refreshing before expiry
→ The application just says: "Get this object from S3" → SDK handles all credential logistics

         From the instance → AWS CLI: aws sts get-caller-identity

            Account: 123456789012
            Arn: arn:aws:sts::123456789012:assumed-role/shopnow-web-server-role/i-0abc123def456

Understanding the Assumed Role ARN:

arn:aws:sts::123456789012:assumed-role/shopnow-web-server-role/i-0abc123def456
     │                      │              │                       │
     │                      │              │                       └── Session name (instance ID)

STEP 10 — UNDERSTANDING ROLE SWITCHING AND SESSION DURATION

Session Duration:

→ When EC2 assumes a role → credentials are valid for a default period
→ EC2 Instance Profile credentials: ~6 hours (managed by AWS, auto-refreshed)
→ For manual AssumeRole calls (cross-account — Project 40): 1-12 hours (configurable)

Role Maximum Session Duration:

→ IAM Console → Roles → Select role → "Edit" → Maximum session duration

SettingOptionsDefault
Maximum session duration1 hour, 2 hours, 4 hours, 8 hours, 12 hours1 hour

→ This limits how long manually-assumed credentials can last
→ Does NOT affect EC2 Instance Profile auto-rotation (always ~6 hours)
→ Affects: CLI aws sts assume-role, cross-account assumption (Project 40)

Role Chaining:

→ A role can assume ANOTHER role → this is called "role chaining"
→ Maximum session duration for chained roles: 1 hour (hard limit)
→ Example: User assumes Role A → Role A assumes Role B → Role B credentials last max 1 hour
→ Used in complex cross-account scenarios (Project 40)

✅ WHAT SHOULD DO

→ Always use IAM Roles for EC2 instead of hardcoded access keys — no exceptions
→ Always follow least privilege — the role should have ONLY the permissions the application needs
→ Always use specific resource ARNs in permission policies — avoid Resource: "*" where possible
→ Always enforce IMDSv2 on all instances (Project 3) — prevents SSRF credential theft
→ Always use the AWS SDK's default credential chain — don't manually read from IMDS
→ Always create SEPARATE roles for different application tiers (web role, processor role, admin role)
→ Always name roles descriptively: {application}-{component}-role
→ Always attach the SSM managed policy for Session Manager access — eliminates SSH key management
→ Always verify role permissions with aws sts get-caller-identity from the instance
→ Always test DENIED actions to confirm least privilege is working
→ Always tag roles with Environment, Application, and ManagedBy
→ Always create roles through the Console for simplicity (auto-creates Instance Profile)

❌ WHAT SHOULD NOT DO

→ Never hardcode access keys in application code, config files, or environment variables on EC2
→ Never attach AdministratorAccess to an EC2 role — if the instance is compromised, the attacker owns your account
→ Never use Resource: "*" when you know the specific resource ARNs
→ Never share the same role between production and development environments
→ Never set environment variables (AWS_ACCESS_KEY_ID) on instances with IAM Roles — they override IMDS
→ Never create IAM Users for applications running on AWS services — always use Roles
→ Never attach more than one Instance Profile to an EC2 instance — it's not supported
→ Never forget to update the bucket policy (Project 10) to reference the role as Principal
→ Never skip IMDSv2 enforcement — IMDSv1 is vulnerable to SSRF attacks
→ Never use the same role for web servers and database servers — different trust levels
→ Never forget that RDS access is via Security Groups (network) not IAM actions (API) — IAM Role doesn't grant SQL access
→ Never leave unused roles attached to terminated instances — clean up regularly

3 → HOW EFFECTIVE PERMISSIONS ARE CALCULATED WITH ROLES

Application on EC2 tries: s3:PutObject to shopnow-bucket/orders/order-001.pdf

PERMISSION EVALUATION:

Layer 1: SCP (Service Control Policy — if in Organization)
  → Does the SCP allow s3:PutObject? → YES (or not restricted) → Continue

Layer 2: Permission Boundary (if set on the role)
  → Does the boundary allow s3:PutObject? → Not set → Continue

Layer 3: Session Policy (if provided during AssumeRole)
  → EC2 Instance Profile doesn't use session policies → Continue

Layer 4: Identity-based Policy (role's permission policies)
  → shopnow-web-server-policy Statement 2:
    Allow s3:PutObject on arn:aws:s3:::shopnow-productimages-prod-aps1/orders/*
  → order-001.pdf matches orders/* → ALLOW found

Layer 5: Resource-based Policy (bucket policy — Project 10)
  → Statement 1: Allow Principal shopnow-web-server-role s3:GetObject
    → Action is PutObject, not GetObject → No match
  → Statement 2: Deny without KMS encryption
    → Is encryption header present with aws:kms? → YES → Deny doesn't apply
  → No explicit deny found

Layer 6: Bucket ACL (disabled — Project 10)
  → Not applicable

FINAL RESULT:
  No explicit deny found ✓
  Explicit allow found in role's permission policy ✓
  (Same account — either IAM or bucket policy allow is sufficient)
  → ACCESS GRANTED ✓

📈 SCALABILITY CONSIDERATIONS

→ Role Reusability

  • ONE role can be used by HUNDREDS of EC2 instances
  • All instances with the same Instance Profile share the same permissions
  • Each instance gets its own unique temporary credentials (different AccessKeyId)
  • Each instance has its own session name (instance ID) → distinguishable in logs

→ Permission Management at Scale

  • Use AWS Managed Policies for common patterns (CloudWatchAgent, SSM)
  • Use Customer Managed Policies for application-specific permissions
  • Update the policy ONCE → ALL instances with that role get updated permissions
  • No need to deploy to individual instances

  • Web servers: shopnow-web-server-role (read S3, send SQS, publish SNS)

  • Admin: shopnow-admin-role (broader access for maintenance tasks)
  • Monitoring: shopnow-monitoring-role (read CloudWatch, write metrics)
  • Each tier has EXACTLY the permissions it needs → blast radius is limited

→ Auto Scaling Integration

  • Auto Scaling Launch Template includes Instance Profile
  • Every new instance automatically gets the correct role
  • No manual credential distribution → scales infinitely
  • Terminated instances lose credentials immediately (IMDS is destroyed)

→ Multi-Account Scalability

  • Cross-account roles (Project 40) enable centralized access patterns
  • Roles in Account A can be assumed by identities in Account B
  • No need to create IAM users in every account

→ How to debug "Unable to locate credentials":

STEP 1: Is an Instance Profile attached?
        EC2 Console → Instance → Details → IAM Role
        If blank → attach Instance Profile

STEP 2: Is IMDS reachable?
        From instance: curl -w "\n%{http_code}" http://169.254.169.254/latest/meta-data/
        If timeout → check IMDS settings (httpPutResponseHopLimit, HttpTokens)

STEP 3: Are credentials available in IMDS?
        IMDSv2 token → then GET /latest/meta-data/iam/security-credentials/
        Should return role name
        Then GET /latest/meta-data/iam/security-credentials/{role-name}
        Should return JSON with AccessKeyId

STEP 4: Are environment variables overriding IMDS?
        Check: env | grep AWS_
        If AWS_ACCESS_KEY_ID is set → it overrides IMDS → unset it

STEP 5: Is ~/.aws/credentials present?
        Check: cat ~/.aws/credentials
        If present with credentials → they override IMDS → remove or rename

STEP 6: Is the SDK version too old?
        Very old SDK versions may not support IMDSv2
        Update to latest SDK version

→ How to debug "Access Denied" with correct role:

STEP 1: Confirm which credentials are being used
        aws sts get-caller-identity
        → Is it the expected role? Or different credentials?

STEP 2: Check the role's permission policies
        IAM Console → Role → Permissions
        Does ANY policy allow the specific action + resource?

STEP 3: Check for explicit Deny in role policies
        Look for any Deny statements that match

STEP 4: Check the resource's policy (bucket policy, SQS policy, etc.)
        Does it explicitly Deny the role?
        For same-account: Either IAM or resource policy Allow is sufficient
        For cross-account: BOTH must Allow

STEP 5: Check Permission Boundary (if set on role)
        Effective permissions = Role policy ∩ Boundary

STEP 6: Check SCP (if in AWS Organization)
        SCP might deny the action at the organization level

STEP 7: Check KMS permissions
        If the resource is encrypted with KMS
        → Role needs kms:Decrypt (for reading) or kms:GenerateDataKey (for writing)

STEP 8: Use IAM Policy Simulator
        IAM Console → Policy Simulator → Select role → Test action + resource

→ Roles provide TEMPORARY credentials that AUTO-ROTATE — no keys to manage, steal, or leak
→ The Trust Policy defines WHO can assume the role (EC2, Lambda, Glue, another account)
→ The Permission Policy defines WHAT the role can do (which actions on which resources)
→ Instance Profiles are the bridge between IAM Roles and EC2 instances

→ Connection to Project 2: IAM Roles use the SAME policy language as IAM Users

  • Different: Roles have Trust Policies, Users don't
  • Different: Roles provide temporary credentials, Users have permanent keys

→ Connection to Project 10: Bucket policies reference the role as Principal
