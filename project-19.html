PROJECT 19: THE APPLICATION LOGS NOBODY COULD FIND — CloudWatch Logs from EC2 via Unified Agent

🔴 HYPOTHETICAL PROBLEM

→ ShopNow (from previous projects) now runs a Java-based order processing application on an EC2 instance
→ The application writes logs to /var/log/shopnow/application.log
→ The Nginx reverse proxy writes access logs to /var/log/nginx/access.log and error logs to /var/log/nginx/error.log
→ The operating system writes critical messages to /var/log/messages and /var/log/secure
→ Last week — the application threw 500 errors for 45 minutes during peak traffic
→ The on-call engineer SSH'd into the instance to check logs
→ Problem 1: The instance was part of an Auto Scaling group — the problematic instance had already been TERMINATED and replaced
→ Problem 2: The logs were gone forever — terminated instance = deleted EBS volume = deleted logs
→ Problem 3: Even on running instances, logs had rotated — only last 24 hours available
→ Problem 4: With 6 EC2 instances running, the engineer had to SSH into each one individually to grep logs
→ Problem 5: No way to search across all instances simultaneously
→ Problem 6: No way to set alarms on log content (e.g., alert when "OutOfMemoryError" appears)
→ The CTO demands: "I need ALL logs from ALL instances in ONE place, searchable, retained for 90 days, and alertable"

🟡 HOW TO APPROACH IT

→ First — understand the core limitation
→ EC2 instance logs live on the LOCAL disk of that instance
→ If the instance terminates, scales down, or the disk fails — logs vanish permanently
→ SSH-based log access does NOT scale beyond 2–3 instances
→ You cannot search across multiple instances from a terminal

→ The solution requires CENTRALIZING logs OFF the instance into a durable, searchable service
→ AWS CloudWatch Logs is that service

→ But CloudWatch Logs doesn't magically pull logs from EC2
→ You need an AGENT running ON the EC2 instance that:

  • Reads specified log files continuously
  • Ships log lines to CloudWatch Logs service in real-time
  • Handles batching, retry, and buffering

→ That agent is the CloudWatch Unified Agent

→ The approach should be:

  • Step 1 — Understand CloudWatch Logs hierarchy (Log Group → Log Stream → Log Event)
  • Step 2 — Create an IAM Role with CloudWatch Logs permissions (builds on Project 11)
  • Step 3 — Attach role to EC2 instance
  • Step 4 — Install the CloudWatch Unified Agent on EC2
  • Step 5 — Configure the agent (which files to ship, where to send them)
  • Step 6 — Start the agent and verify logs appear
  • Step 7 — Set retention policy
  • Step 8 — Create Metric Filters for log-based alarms

🟢 HOW CAN WE SOLVE IT — Step-by-Step Configuration

STEP 1 — UNDERSTAND CLOUDWATCH LOGS HIERARCHY

→ CloudWatch Logs has a 3-level hierarchy — this is foundational

CloudWatch Logs
  └── Log Group          (container — one per application/log type)
        └── Log Stream    (one per source — typically one per instance)
              └── Log Events   (individual log lines with timestamps)

Log Group
→ Think of it as a "folder" for a specific type of log
→ Example Log Groups:

  • /shopnow/application
  • /shopnow/nginx/access
  • /shopnow/nginx/error
  • /ec2/system/messages
  • /ec2/system/secure
    → Retention policy is set at the Log Group level
    → Encryption is set at the Log Group level
    → Access control (IAM) can target specific Log Groups

Log Stream
→ Think of it as "one source feeding into the group"
→ Typically — one Log Stream per EC2 instance per Log Group
→ Stream name is usually the instance ID: i-0a1b2c3d4e5f67890
→ When that instance terminates — the stream remains (logs preserved!)
→ When a new instance launches — a new stream is created

Log Event
→ A single log line with two components:

  • Timestamp (millisecond precision)
  • Raw message (the actual log line text)
    → Example: 2025-01-15T14:32:05.123Z ERROR OutOfMemoryError: Java heap space

→ Why this hierarchy matters:

  • You search within a Log Group (across ALL streams = across ALL instances simultaneously)
  • You set retention on the Log Group (all streams inherit it)
  • You create Metric Filters on the Log Group (pattern matching across all instances)

STEP 2 — CREATE IAM POLICY FOR CLOUDWATCH AGENT

→ The agent runs ON the EC2 instance
→ It needs permissions to WRITE logs to CloudWatch Logs
→ It also needs permissions to WRITE custom metrics (the Unified Agent also collects system metrics)
→ As learned in Project 11 — EC2 gets permissions via IAM Role + Instance Profile

Option A: Use AWS Managed Policy
→ Navigate to IAM → Policies → Search for CloudWatchAgentServerPolicy
→ This is an AWS-managed policy that grants:

logs:CreateLogGroupAgent creates Log Groups if they don't exist
logs:CreateLogStreamAgent creates Log Streams (one per instance)
logs:PutLogEventsAgent pushes actual log lines
logs:DescribeLogStreamsAgent checks existing streams
logs:DescribeLogGroupsAgent checks existing groups
ec2:DescribeTagsAgent reads instance tags (useful for dynamic naming)
ec2:DescribeVolumesAgent reads volume info for disk metrics
ec2:DescribeInstancesAgent reads instance metadata
ssm:GetParameterAgent can retrieve its config from SSM Parameter Store

→ This is the recommended approach for most use cases

Option B: Custom Policy (Least Privilege)
→ If you want tighter control — restrict to specific Log Groups

→ Custom policy structure:

→ This allows writing ONLY to Log Groups starting with /shopnow/
→ The agent cannot create random Log Groups or write to others

✅ WHAT SHOULD DO

→ Always use IAM Roles (Project 11) for agent permissions — never hardcode access keys on the instance
→ Always use {instance_id} as Log Stream name — guarantees uniqueness and traceability
→ Always set retention policies on every Log Group — "Never Expire" silently accumulates massive costs
→ Always configure multi_line_start_pattern for Java/Python logs — stack traces span multiple lines
→ Always store agent config in SSM Parameter Store — enables fleet-wide config management
→ Always separate log types into different Log Groups — application logs, access logs, error logs, system logs
→ Always use a naming convention for Log Groups

PatternExample
/{company}/{app}/{log-type}/shopnow/orderservice/application
/{environment}/{app}/{log-type}/prod/orderservice/error

→ Always create Metric Filters for critical error patterns on Day 1
→ Always verify agent is running after instance reboot — configure it as a system service that starts on boot
→ Always test with a deliberate log write — generate a test log line and verify it appears in CloudWatch within 30 seconds

❌ WHAT SHOULD NOT DO

→ Never leave retention as "Never Expire" without a conscious decision — 1 TB of logs/month at $0.03/GB = $30/month growing forever
→ Never use {hostname} as stream name in Auto Scaling groups — hostnames can be recycled
→ Never put ALL log types into one Log Group — makes searching, retention, and access control impossible to differentiate
→ Never install the OLD CloudWatch Logs Agent (awslogs) — it is deprecated; always use the UNIFIED Agent (amazon-cloudwatch-agent)
→ Never forget the timestamp format — wrong format = all events get ingestion time instead of actual event time → makes time-based searching useless
→ Never set collection interval below 10 seconds in production — excessive API calls, higher costs, potential throttling
→ Never skip multi-line configuration for Java apps — a single exception with stack trace becomes 30 separate log events instead of 1
→ Never grant logs:* in IAM — follow least privilege; agent only needs PutLogEvents, CreateLogGroup, CreateLogStream, DescribeLogStreams
→ Never ignore the agent's own log file — if logs aren't appearing in CloudWatch, the answer is in /opt/aws/amazon-cloudwatch-agent/logs/amazon-cloudwatch-agent.log
→ Never assume CloudWatch Agent is pre-installed — verify on your AMI or include installation in User Data / launch template

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

1 → AGENT INTERNALS — HOW LOGS MOVE FROM FILE TO CLOUD

[Log File on Disk]
       ↓
 Agent TAILS the file (like Linux 'tail -f')
       ↓
 Agent reads new lines since last checkpoint
       ↓
 Agent parses timestamp using configured format
       ↓
 Agent batches multiple log events together
       ↓
 Agent calls PutLogEvents API
       ↓
 CloudWatch Logs service receives and indexes events
       ↓
 Events appear in Log Stream within the Log Group

Checkpointing:
→ The agent maintains a STATE FILE on disk
→ Location: /opt/aws/amazon-cloudwatch-agent/logs/state/
→ This file records: "I have read up to byte offset 45678 of /var/log/shopnow/application.log"
→ If the agent restarts → it resumes from the checkpoint → no duplicate or missed logs
→ If the file rotates (logrotate) → agent detects the inode change → starts reading the new file

Batching:
→ Agent does NOT send one API call per log line — that would be extremely expensive
→ Agent batches events and sends them in bulk
→ Batch constraints:

ConstraintLimit
Maximum batch size1 MB
Maximum events per batch10,000
Maximum event size256 KB (single log event)
Flush intervalConfigurable — default ~5 seconds

→ Agent accumulates log events in memory buffer
→ When buffer hits 1 MB OR 5 seconds elapse → flush to CloudWatch Logs API
→ This means typical log delivery latency: 5–15 seconds from file write to CloudWatch visibility

Retry Logic:
→ If PutLogEvents fails (network issue, throttling) → agent retries with exponential backoff
→ Events are buffered in memory during retry
→ If agent crashes during retry → checkpointed events are safe, un-checkpointed events may be lost
→ Risk is minimal — window of potential loss is typically < 5 seconds of logs

2 → CLOUDWATCH LOGS STORAGE INTERNALS

Ingestion:
→ PutLogEvents API receives batches
→ Events are validated (timestamps must be within 14 days past to 2 hours future)
→ Events are indexed by timestamp for time-range queries
→ Events are stored durably (replicated within the region)

Storage Tiers (as of 2024):

TierNamePurposeCost
StandardStandard ClassFull features — Logs Insights, Metric Filters, alarmsHigher
Infrequent AccessIA ClassReduced cost, supports Logs Insights, no Metric Filters~50% cheaper ingestion

→ When creating a Log Group → choose the class:

  • Application error logs (need Metric Filters + alarms) → Standard
  • Verbose debug logs (rarely searched) → Infrequent Access

Sequence Token Mechanism:
→ Each Log Stream has a sequenceToken
→ Every PutLogEvents call must include the current sequence token
→ The response returns the NEXT sequence token
→ This ensures ordered, non-duplicate writes to a stream
→ The agent handles this automatically — you never interact with sequence tokens directly
→ If two agents write to the SAME stream simultaneously → one gets InvalidSequenceTokenException
→ This is why: one stream per instance — never share streams

3 → METRIC FILTER INTERNALS

→ Metric Filters are NOT queries that run periodically
→ They are PATTERN MATCHERS applied to EVERY incoming log event in real-time

Log Event arrives at Log Group
       ↓
 CloudWatch checks ALL Metric Filters on that Log Group
       ↓
 For each filter — does the event match the pattern?
       ↓
 If YES → emit a CloudWatch metric data point
       ↓
 Metric data point flows into CloudWatch Metrics
       ↓
 Alarm evaluates the metric (as per Project 8 mechanics)

→ This is a PUSH model — not a pull/polling model
→ Every log event is evaluated against every Metric Filter on ingestion
→ Maximum Metric Filters per Log Group: 100
→ Metric Filters do NOT scan historical data — only events AFTER the filter is created
→ If you create a filter now — it won't count yesterday's errors

4 → LOGS INSIGHTS INTERNALS

→ Logs Insights is a query engine that scans log data
→ It is NOT a database — there is no pre-built index on your log content
→ When you run a query:

  • CloudWatch identifies which log data falls in your time range
  • It scans ALL events in that range across selected Log Groups
  • Pattern matching and aggregation happen during scan
  • Results are returned (up to 10,000 events for display, 20 per limit default)

→ Pricing: Charged per GB of data scanned
→ Narrower time range = less data scanned = lower cost
→ This is exactly the same pricing model as Athena (Project 18) — scan-based billing

→ Logs Insights automatically discovers fields in JSON-formatted logs
→ For non-JSON logs — use parse command to extract fields from text patterns

📈 SCALABILITY CONSIDERATIONS

→ Instance Fleet Scaling

  • Agent runs independently on each instance
  • Auto Scaling group launches new instance → User Data installs agent → agent fetches config from SSM → logs flow
  • No central coordination needed
  • 6 instances or 600 instances — same architecture

→ Log Volume Scaling

  • CloudWatch Logs can ingest up to 5,000 PutLogEvents requests per second per Log Group
  • Each request can contain up to 10,000 events or 1 MB
  • For higher throughput → split into multiple Log Groups (per microservice, per component)

→ API Throttling

APIDefault LimitScalable?
PutLogEvents800 TPS per account per regionYes — request increase
CreateLogGroup10 TPSRarely hit unless mass-deploying
DescribeLogGroups10 TPSCan be bottleneck in automation scripts
FilterLogEvents25 TPSLogs Insights is better for ad-hoc queries
GetLogEvents25 TPSUse Logs Insights instead

→ Cost Scaling

ComponentPricing Model
Log Ingestion$0.50 per GB ingested (Standard class)
Log Ingestion$0.25 per GB ingested (Infrequent Access class)
Log Storage$0.03 per GB per month
Logs Insights$0.005 per GB of data scanned

→ Example: 100 GB/day ingestion × 30 days = 3,000 GB/month

  • Ingestion: 3,000 × $0.50 = $1,500/month
  • Storage (90-day retention): ~9,000 GB × $0.03 = $270/month
  • Total: ~$1,770/month

→ Cost optimization strategies:

  • Reduce log verbosity (INFO not DEBUG in production)
  • Use Infrequent Access class for verbose logs
  • Set aggressive retention (30 days instead of 90 for access logs)
  • Export to S3 for long-term retention (S3 storage: $0.023/GB vs CW $0.03/GB)
  • Filter at the agent level — don't ship logs you'll never search

→ Cross-Region Scaling

  • CloudWatch Logs is regional — logs stay in the region of the instance
  • For multi-region deployments → each region has its own Log Groups
  • Cross-region log aggregation → use Subscription Filters → Kinesis → S3 in central region
