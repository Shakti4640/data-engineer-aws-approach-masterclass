PROJECT 29: THE DATABASE THAT COULDN'T AFFORD DOWNTIME — Multi-AZ RDS MariaDB with Read Replica

🔴 HYPOTHETICAL PROBLEM

→ ShopNow's RDS MariaDB instance (Project 9) is the operational backbone:

  • E-commerce application reads/writes orders, customers, products, payments
  • 50,000 orders/day flow through this single database instance
  • Peak: 500 transactions/second during flash sales
  • Current configuration: Single-AZ deployment in ap-south-1a

→ Incident Report — The Tuesday Outage:

→ Problem 1: Single Point of Failure

  • One AZ failure = complete database outage
  • Application has ZERO redundancy for the database tier
  • EC2 instances span multiple AZs (Project 5) — but database is in only one
  • The weakest link determines system availability

→ Problem 2: No Automatic Failover

  • When the AZ recovered, the instance came back on its own
  • But what if the AZ didn't recover for 4 hours? Or the EBS volume was corrupted?
  • The engineer would need to manually restore from snapshot — 30-60 minutes
  • During restoration — choosing a new AZ, restoring data, updating DNS, reconnecting application
  • Manual process = human error risk + longer downtime

→ Problem 3: Read Scaling Crisis

  • Despite the ban (Project 24), analysts are querying MariaDB directly again
  • Reason: Glue extraction runs once daily at 02:00 UTC — analysts need data from 10:00 AM
  • 15 analysts running complex analytical queries during business hours
  • These queries scan millions of rows — consuming 40% of CPU
  • Production application response time degraded from 50ms to 300ms
  • DBA monitoring shows:

→ Problem 4: Glue Extraction Impact (Revisited from Project 24)

  • Project 24 recommended using a Read Replica for Glue extraction
  • But no Read Replica exists yet
  • The nightly Glue extraction job (02:00 UTC) runs against the primary
  • During extraction: 5 parallel JDBC connections scanning large tables
  • CPU spikes to 60% for 35 minutes — overlapping with early morning orders from global customers

→ Problem 5: Business Continuity Requirements

  • After the Tuesday outage, the board demands:
    • RTO (Recovery Time Objective): < 2 minutes (time to recover)
    • RPO (Recovery Point Objective): 0 data loss (no transactions lost)
    • Read scaling: Analysts must have a separate database endpoint
    • Extraction safety: Glue must not touch the primary database

→ The CTO says: "I need the database to survive AZ failures automatically, with zero data loss, zero manual intervention, and analyst queries must NEVER touch the production database again"

Synchronous Replication — What It Means:

Application sends: INSERT INTO orders VALUES (...)
       ↓
PRIMARY receives the write
       ↓
PRIMARY writes to local EBS storage
       ↓
PRIMARY sends the write to STANDBY (synchronous)
       ↓
STANDBY writes to its local EBS storage
       ↓
STANDBY acknowledges to PRIMARY: "Write confirmed"
       ↓
PRIMARY acknowledges to APPLICATION: "INSERT successful"
       ↓
ONLY NOW does the application know the write succeeded

→ Key insight: The application does NOT get acknowledgment until BOTH copies are written
→ This guarantees: If PRIMARY fails after acknowledging → STANDBY has the data
→ RPO = 0 (zero data loss) — because no write is acknowledged unless both copies exist

Trade-off: Synchronous replication adds latency
→ Each write must wait for standby acknowledgment
→ Typical additional latency: 2-5ms (cross-AZ network)
→ For ShopNow: 50ms average query time → 55ms with Multi-AZ → acceptable

Failover Process:

PRIMARY in AZ-a fails (hardware, network, or storage failure)
       ↓
RDS detects failure (health checks every few seconds)
       ↓
RDS initiates automatic failover (NO human intervention)
       ↓
STEP 1: STANDBY in AZ-b is promoted to PRIMARY (30-60 seconds)
  ├── Standby stops accepting replication
  ├── Standby completes any pending writes
  └── Standby transitions to PRIMARY role (read + write)
       ↓
STEP 2: DNS record updated (30-60 seconds)
  ├── shopnow-prod-db.xxxxx.rds.amazonaws.com
  ├── Previously resolved to: AZ-a IP (old primary)
  └── Now resolves to: AZ-b IP (new primary = former standby)
       ↓
STEP 3: Application reconnects
  ├── Application's next database call uses the DNS name
  ├── DNS resolves to new primary IP
  ├── Connection established to new primary
  └── Application resumes normal operation
       ↓
STEP 4: RDS provisions new standby (background — minutes to hours)
  ├── New standby created in AZ-a (or another AZ)
  ├── Data synchronized from new primary
  └── Multi-AZ protection restored
       ↓
Total failover time: 60-120 seconds
Data loss: ZERO (synchronous replication)
Manual intervention: NONE

Asynchronous Replication — What It Means:

Application sends: INSERT INTO orders VALUES (...)
       ↓
PRIMARY writes to local storage
       ↓
PRIMARY acknowledges to APPLICATION: "INSERT successful"
  (Application does NOT wait for replica)
       ↓
PRIMARY sends the write to READ REPLICAS (asynchronously)
       ↓
Replicas receive the write (typically < 1 second later)
       ↓
Replicas apply the write to their local storage

→ Key difference from Multi-AZ:

  • Multi-AZ: Application waits for standby acknowledgment (synchronous)
  • Read Replica: Application does NOT wait — replica catches up later (asynchronous)

→ Consequence: Replica lag

  • There is a brief period where the replica is "behind" the primary
  • Typical lag: < 1 second under normal load
  • Under heavy write load: 1-5 seconds
  • Under extreme load: 10-60 seconds (concerning)

→ For analyst queries: 1-second lag is completely irrelevant
→ For Glue extraction: 1-second lag is irrelevant
→ For application reads requiring absolute consistency: Use primary, not replica

✅ WHAT SHOULD DO

→ Always enable Multi-AZ for production databases — the cost of downtime far exceeds the cost of a standby
→ Always create at least one Read Replica for analyst query offloading — protect primary performance
→ Always create a dedicated Read Replica for ETL extraction — isolate extraction from all other workloads
→ Always use separate security groups for each replica — enforce least-privilege access
→ Always use separate parameter groups for each replica — optimize for workload type (OLTP vs analytical vs ETL)
→ Always monitor ReplicaLag — alarm at 30 seconds (warning) and 300 seconds (critical)
→ Always set up RDS Event Subscriptions for failover events — immediate notification
→ Always test failover before you need it — scheduled failover tests quarterly
→ Always update Glue JDBC connections to point to ETL replica (Project 24 update)
→ Always create dedicated database users for each access pattern (app, analyst, glue)
→ Always verify application handles connection drops gracefully — failover causes brief disconnection
→ Always increase backup retention when enabling Multi-AZ — 14+ days recommended

❌ WHAT SHOULD NOT DO

→ Never run production databases without Multi-AZ — single AZ is a ticking time bomb
→ Never let analysts query the primary database — always redirect to Read Replica
→ Never let Glue ETL extract from the primary — always use a dedicated replica
→ Never assume Read Replica is a backup — it's a live copy, not a point-in-time recovery mechanism
→ Never ignore ReplicaLag alarms — sustained high lag means replica is falling behind (may indicate under-provisioning)
→ Never use the same security group for primary and replicas — different access patterns need different rules
→ Never use the same parameter group for OLTP primary and analytical replica — different workloads need different tuning
→ Never promote a Read Replica unless it's a true disaster scenario — promotion is irreversible
→ Never skip failover testing — an untested failover is not a failover plan
→ Never forget to update Glue connections after creating ETL replica — Glue still points to primary otherwise
→ Never create Read Replicas without monitoring — an unmonitored lagging replica serves stale data silently
→ Never set max_connections on replica higher than it can handle — more connections ≠ better performance

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

1 → MULTI-AZ SYNCHRONOUS REPLICATION ENGINE

APPLICATION WRITE:
  INSERT INTO orders (order_id, total) VALUES (12345, 99.99)
       ↓
PRIMARY INSTANCE (AZ-a):
  ├── Write to InnoDB buffer pool (memory)
  ├── Write to InnoDB redo log (WAL — Write-Ahead Log)
  ├── Flush redo log to EBS (primary's disk)
  │
  ├── SIMULTANEOUSLY: Send redo log entry to STANDBY
  │     ├── Network transfer: AZ-a → AZ-b (2-5ms latency)
  │     ├── Standby receives redo log entry
  │     ├── Standby writes to its redo log on its EBS
  │     └── Standby acknowledges: "Received and persisted"
  │
  ├── PRIMARY receives standby acknowledgment
  └── PRIMARY acknowledges to APPLICATION: "COMMIT successful"
       ↓
STANDBY INSTANCE (AZ-b):
  ├── Applies redo log entry to its InnoDB buffer pool
  ├── Eventually flushes to data files (background)
  └── Standby is now consistent with primary
       ↓
GUARANTEE: If primary fails after acknowledging COMMIT:
  ├── Standby has the redo log entry
  ├── Standby can apply it during promotion
  └── ZERO data loss

→ The replication is at the STORAGE level (redo log), not the SQL level
→ This is more efficient than SQL-based replication — no need to re-execute queries

2 → FAILOVER DNS MECHANISM

NORMAL STATE:
  DNS: shopnow-prod-db.xxxxx.rds.amazonaws.com
    → CNAME → primary-instance.ap-south-1a.rds.amazonaws.com
      → A record → 10.0.1.50 (primary's private IP in AZ-a)
  
  TTL (Time-To-Live): 5 seconds
  (DNS record cached for only 5 seconds)

FAILOVER INITIATED:
  T+0s: Primary failure detected
  T+30s: Standby promoted to primary
  T+45s: DNS CNAME updated:
    shopnow-prod-db.xxxxx.rds.amazonaws.com
      → CNAME → promoted-instance.ap-south-1b.rds.amazonaws.com
        → A record → 10.0.2.75 (new primary's private IP in AZ-b)
  
  T+50s: DNS TTL expires (was 5 seconds)
  T+50s: Application's next DNS lookup resolves to new IP
  T+51s: Application connects to new primary
  T+51s: Service restored

WHY TTL = 5 SECONDS MATTERS:
  ├── If TTL = 300 seconds (5 minutes): Application would cache old IP for 5 minutes after failover
  ├── If TTL = 5 seconds: Application gets new IP within 5 seconds of DNS update
  └── RDS sets TTL intentionally low for fast failover

→ Critical application requirement: Application must NOT cache DNS longer than TTL
→ Most modern connection pools and SDKs respect DNS TTL
→ If application caches DNS indefinitely → failover doesn't work → must restart application

Application connection handling during failover:

Application connection pool has 50 active connections to old primary
       ↓
Primary fails → all 50 connections are broken
       ↓
Connection pool detects broken connections (socket errors)
       ↓
Connection pool attempts to create new connections
       ↓
DNS resolution returns NEW primary IP
       ↓
New connections established to new primary
       ↓
Application resumes operation
       ↓
TOTAL APPLICATION DISRUPTION: 60-120 seconds

→ Well-designed connection pools handle this automatically
→ Application may see brief errors during the 60-120 second window
→ Retry logic in the application handles these transient failures

3 → READ REPLICA ASYNCHRONOUS REPLICATION ENGINE

PRIMARY WRITE:
  INSERT INTO orders (order_id, total) VALUES (12345, 99.99)
       ↓
PRIMARY:
  ├── Writes to InnoDB + redo log
  ├── Acknowledges to APPLICATION immediately
  │     (Does NOT wait for replicas)
  └── Writes to BINARY LOG (binlog)
       ↓
BINARY LOG REPLICATION:
  ├── Replica's I/O thread connects to primary's binlog
  ├── Reads new binlog events
  ├── Writes events to replica's RELAY LOG
  └── This is the NETWORK transfer step (< 1 second typically)
       ↓
REPLICA SQL THREAD:
  ├── Reads relay log events
  ├── Applies SQL statements to replica's database
  └── Replica data is now updated
       ↓
REPLICA LAG = Time between:
  ├── Primary commits the write
  └── Replica applies the write
  ├── Components: Network transfer time + SQL apply time
  └── Typical: < 1 second

Why replicas can lag:

CauseMechanismTypical Lag
Network latency (same region)Cross-AZ transfer time1-5ms
Network latency (cross-region)Internet/backbone transfer50-200ms
Replica CPU overloadedSQL thread can't apply fast enoughSeconds to minutes
Long-running query on replicaBlocks SQL thread applicationDuration of query
Primary write burstBinlog events accumulate faster than replica appliesSeconds
Large transaction on primarySingle large binlog eventSeconds to minutes

4 → READ REPLICA PROMOTION MECHANICS

PROMOTION INITIATED (manual action):
       ↓
STEP 1: Replica stops replication
  ├── I/O thread stops reading from primary's binlog
  ├── SQL thread finishes applying remaining relay log events
  └── Replica is now at a consistent point
       ↓
STEP 2: Replica reconfigured
  ├── Read-only mode disabled → writes now allowed
  ├── Binary logging enabled (was disabled on replica)
  ├── Replica becomes a standalone RDS instance
  └── New endpoint retains the replica's original endpoint name
       ↓
STEP 3: Instance restarts (brief)
  ├── Configuration changes require restart
  └── Takes 1-3 minutes
       ↓
STEP 4: Instance available
  ├── Status: Available
  ├── Role: Standalone (no longer a replica)
  ├── Read/Write: Both enabled
  └── No connection to former primary
       ↓
POST-PROMOTION:
  ├── Former replica is now an independent database
  ├── No automatic replication from old primary
  ├── Must create NEW replicas from this promoted instance
  ├── Must update application connection strings if using as new primary
  └── Old primary (if still running) continues independently

→ Promotion is ONE-WAY — cannot be reversed
→ After promotion, the instance is fully independent
→ Must rebuild the replica infrastructure from scratch
