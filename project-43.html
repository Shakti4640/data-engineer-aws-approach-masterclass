PROJECT 43: THE WAREHOUSE DATA THAT NEEDED TO FLOW BACK TO OPERATIONS — Redshift to MariaDB Data Sync via Glue

🔴 HYPOTHETICAL PROBLEM

→ ShopNow's data architecture has matured significantly:

  • Raw data flows INTO the data lake (S3) via Glue ETL (Project 21)
  • Operational data flows FROM MariaDB TO S3 via Glue JDBC (Project 24)
  • S3 data is loaded INTO Redshift via COPY (Project 22)
  • Redshift performs complex aggregations, ML scoring, and business metrics computation
  • Cross-account access to the data lake is governed via Lake Formation (Project 42)

→ But ALL data flows are ONE-DIRECTIONAL: Operational → Analytical

CURRENT FLOW (One-Way):
MariaDB (Operational) → Glue ETL → S3 (Data Lake) → Redshift (Warehouse)
                                                          │
                                                          ▼
                                                    Analytics/BI
                                                    
MISSING FLOW (Reverse):
Redshift (Warehouse) → ??? → MariaDB (Operational)

The Business Need:

→ Redshift computes several high-value derived datasets that the operational application NEEDS:

Redshift DatasetWhat It ContainsWho Needs ItWhy
customer_lifetime_valueCLV score per customer (ML-computed)MariaDB → Web ApplicationDisplay personalized offers based on CLV tier
product_recommendationsRecommended products per customerMariaDB → Web ApplicationShow "Recommended for You" section
inventory_forecastPredicted inventory needs per product per weekMariaDB → Inventory Management SystemAuto-generate purchase orders
sales_daily_summaryDaily sales aggregation by region/categoryMariaDB → Operations DashboardReal-time operations dashboard
fraud_risk_scoresFraud probability per transactionMariaDB → Payment Processing SystemFlag high-risk transactions

Current Workaround (Broken):

→ A data analyst manually runs Redshift UNLOAD (Project 23) to export CSV files to S3
→ Another analyst downloads the CSV from S3
→ The DBA manually imports the CSV into MariaDB using a MySQL client
→ This process happens once a week (should be daily)
→ It takes 4 hours of manual work each time
→ Data is always 1-7 days stale in MariaDB
→ The web application shows week-old recommendations — customers see irrelevant products
→ The fraud detection system uses week-old scores — new fraud patterns are missed for days

Specific Technical Challenges:

Challenge 1 — Schema Mismatch:
→ Redshift uses different data types than MariaDB

Data ElementRedshift TypeMariaDB TypeIssue
Customer IDBIGINTINT UNSIGNEDPotential overflow
CLV ScoreDECIMAL(18,6)DECIMAL(10,2)Precision loss
Recommendation ListVARCHAR(65535) (Redshift SUPER or JSON)TEXT or JSONFormat conversion needed
TimestampTIMESTAMP WITHOUT TIME ZONEDATETIMETimezone handling
Boolean flagsBOOLEANTINYINT(1)Type mapping

Challenge 2 — Incremental vs Full Load:
→ customer_lifetime_value has 5 million rows — full reload takes 45 minutes
→ Only ~50,000 rows change daily
→ Need incremental sync — only push changed rows
→ But Redshift doesn't have native change data capture

Challenge 3 — Concurrency:
→ MariaDB serves live web traffic — 500 requests/second
→ Bulk writing 50,000 rows while the application reads from the same tables
→ Risk of lock contention, slow queries, and application timeouts

Challenge 4 — Failure Recovery:
→ If the sync fails halfway through — MariaDB has partial data
→ Some customers have new recommendations, others have old ones
→ Need atomic updates — either ALL rows are updated or NONE

The Data Engineering Lead asks:

"I need an automated daily pipeline that:
Reads computed datasets from Redshift,
Transforms data types for MariaDB compatibility,
Writes to MariaDB incrementally (only changed rows),
Does this without disrupting live application traffic,
Recovers cleanly from failures,
And runs automatically every night."

→ This requires a Glue ETL job that reads from Redshift (as source) and writes to MariaDB (as sink)
→ The REVERSE of Project 24 (which read from MariaDB and wrote to S3)

🟡 HOW TO APPROACH IT

→ First — understand the architectural pattern being implemented

This is "Reverse ETL":

Traditional ETL:     Operational DB → Transform → Data Warehouse
Reverse ETL:         Data Warehouse → Transform → Operational DB

→ Reverse ETL is increasingly common as warehouses become compute engines (ML scoring, aggregations)
→ The warehouse computes insights → those insights must flow back to operational systems

Why Glue for Reverse ETL (not direct Redshift → MariaDB):

ApproachProsCons
Redshift UNLOAD → S3 → Lambda → MariaDBUses existing patterns (Project 23, 34)Complex orchestration, no built-in type mapping
Redshift Federated Query write-backDirect SQL (Project 48 preview)Read-only — federated query cannot WRITE to external sources
DMS (Database Migration Service)Built for replicationDesigned for migration, not selective aggregation sync
Glue ETL Job (This project)Built-in JDBC connectors for both Redshift and MariaDB, type mapping, bookmarking, schedulingRequires VPC networking for both connections

→ Glue is the best fit because:

  • It has native JDBC connectors for BOTH Redshift and MariaDB
  • It supports DynamicFrame transformations for type conversion
  • It supports job bookmarking for incremental processing (Project 46 preview)
  • It integrates with Glue Workflow for orchestration (Project 31)
  • It runs within VPC for private connectivity to both databases

→ The approach:

  • Step 1 — Create Glue JDBC Connection to Redshift (source)
  • Step 2 — Verify existing Glue JDBC Connection to MariaDB (sink — from Project 24)
  • Step 3 — Configure VPC networking for dual-database connectivity
  • Step 4 — Create the Glue ETL Job for Redshift → MariaDB
  • Step 5 — Handle schema/type mapping between Redshift and MariaDB
  • Step 6 — Implement incremental sync strategy
  • Step 7 — Implement safe write strategy for live MariaDB
  • Step 8 — Configure job scheduling and orchestration
  • Step 9 — Configure monitoring and alerting

STEP 6 — IMPLEMENT INCREMENTAL SYNC STRATEGY

→ This addresses Challenge 2 — avoiding full table reload every day

Three Incremental Strategies:

StrategyHow It WorksWhen to UseComplexity
Full ReplaceDrop + Recreate target table every runSmall tables (< 100K rows)Simplest
Timestamp-Based IncrementalRead only rows WHERE computed_at > last_run_timestampTables with reliable timestamp columnMedium
Glue Job BookmarkGlue tracks what was read last time — only reads new dataS3 sources (limited JDBC support)Medium

For ShopNow's Tables:

TableRow CountDaily Change VolumeStrategy
customer_lifetime_value5 million~50,000 changedTimestamp-Based Incremental
product_recommendations5 million~200,000 changedTimestamp-Based Incremental
inventory_forecast50,000Full recompute dailyFull Replace
sales_daily_summary365 rows (one per day)1 new row + updates to last 7Full Replace
fraud_risk_scores1 million~100,000 new + updatesTimestamp-Based Incremental

Timestamp-Based Incremental Implementation:

Prerequisites:
→ Redshift tables must have a computed_at TIMESTAMP column
→ This column is set when Redshift computes/updates the row
→ Glue reads only rows WHERE computed_at > last_successful_sync_timestamp

Tracking Last Sync Timestamp:

→ Store the last successful sync timestamp in SSM Parameter Store:

Parameter NameValuePurpose
/shopnow/reverse-etl/customer_lifetime_value/last_sync2025-01-15T02:00:00ZLast successful sync timestamp
/shopnow/reverse-etl/product_recommendations/last_sync2025-01-15T02:00:00ZLast successful sync timestamp
/shopnow/reverse-etl/fraud_risk_scores/last_sync2025-01-15T02:00:00ZLast successful sync timestamp

Why SSM Parameter Store instead of Secrets Manager?
→ This is NOT a credential — it's a configuration value
→ SSM Parameter Store is free for standard parameters
→ Secrets Manager costs $0.40/month per secret — unnecessary for timestamps
→ Same pattern discussed in Project 37's "When to use what" section

Incremental Read Flow:

Glue Job starts
    │
    ├── Read last_sync timestamp from SSM Parameter Store
    │     Value: 2025-01-14T02:00:00Z (yesterday's sync)
    │
    ├── Construct Redshift query:
    │     SELECT * FROM customer_lifetime_value
    │     WHERE computed_at > '2025-01-14T02:00:00Z'
    │
    ├── Redshift UNLOAD filtered data to S3 staging
    │     Only 50,000 changed rows (not 5 million)
    │
    ├── Glue reads staged data → transforms → writes to MariaDB
    │
    ├── If SUCCESS:
    │     Update SSM Parameter: last_sync = 2025-01-15T02:00:00Z
    │
    └── If FAILURE:
          Do NOT update SSM Parameter
          Next run will re-read from same timestamp (idempotent retry)

Why Update Timestamp ONLY on Success:
→ If the job fails after reading but before writing → next run re-reads the same data
→ This ensures no data is lost between runs
→ The write to MariaDB must be IDEMPOTENT — writing the same row twice should produce the same result
→ This is achieved through UPSERT (Step 7)

Full Replace Implementation:

→ For small tables like inventory_forecast and sales_daily_summary:

Glue Job starts
    │
    ├── Read ALL data from Redshift table
    │     SELECT * FROM inventory_forecast
    │
    ├── Transform data
    │
    ├── Write to MariaDB STAGING table:
    │     inventory_forecast_staging (temporary copy)
    │
    ├── Atomic table swap in MariaDB:
    │     RENAME TABLE inventory_forecast TO inventory_forecast_old,
    │                  inventory_forecast_staging TO inventory_forecast
    │
    ├── Drop old table:
    │     DROP TABLE inventory_forecast_old
    │
    └── Application never sees partial data — atomic swap

→ The RENAME TABLE operation in MariaDB is ATOMIC — takes milliseconds
→ Application reads from inventory_forecast throughout the process
→ At the moment of rename → application instantly sees new data
→ No lock contention, no partial reads

STEP 7 — IMPLEMENT SAFE WRITE STRATEGY FOR LIVE MARIADB

→ This addresses Challenge 3 (concurrency) and Challenge 4 (failure recovery)

Three Write Strategies:

StrategyHow It WorksImpact on Live TrafficFailure Recovery
Direct INSERT/UPDATEWrite directly to production tableHIGH — lock contentionPartial data on failure
Staging Table + SwapWrite to staging → atomic renameZERO — no locks on productionClean — staging dropped on failure

For ShopNow — Recommended Strategy Per Table:

TableStrategyWhy
customer_lifetime_valueBatch UPSERT (1000 rows/batch)50K rows — too many for single transaction, too few for staging swap overhead
product_recommendationsBatch UPSERT (1000 rows/batch)200K rows — same reasoning
inventory_forecastStaging Table + SwapSmall table, full replace, needs atomic consistency
sales_daily_summaryStaging Table + SwapSmall table, full replace
fraud_risk_scoresBatch UPSERT (1000 rows/batch)100K rows — time-sensitive, needs incremental

Batch UPSERT Configuration:

→ Glue writes to MariaDB using JDBC with UPSERT semantics

UPSERT in MariaDB/MySQL:

→ MariaDB supports INSERT ... ON DUPLICATE KEY UPDATE
→ This means:

  • If the row's PRIMARY KEY doesn't exist → INSERT new row
  • If the row's PRIMARY KEY already exists → UPDATE the existing row

Connectionshopnow-mariadb-connectionMariaDB JDBC connection
Databaseshopnow_prodTarget database

Understanding Glue JDBC Write Modes:

ModeGlue SettingWhat It DoesMariaDB Effect
Appendmode = "append"INSERT only — fails on duplicate keyErrors if row exists
Overwritemode = "overwrite"DROP TABLE + CREATE TABLE + INSERTDestroys existing data
CustomPre/Post actionsExecute custom SQL before/after data writeFull control

→ Neither append nor overwrite supports UPSERT natively
→ The solution: Use the STAGING TABLE pattern even for incremental writes

Incremental UPSERT via Staging Table Pattern:

Step 1: Glue creates staging table in MariaDB
        CREATE TABLE customer_lifetime_value_staging LIKE customer_lifetime_value

Step 2: Glue writes incremental data (50K rows) to staging table
        INSERT INTO customer_lifetime_value_staging VALUES (...)
        (Glue's standard append mode — no conflicts because staging is empty)

Step 3: Glue executes MERGE from staging to production
        INSERT INTO customer_lifetime_value 
        SELECT * FROM customer_lifetime_value_staging
        ON DUPLICATE KEY UPDATE
          clv_score = VALUES(clv_score),
          clv_tier = VALUES(clv_tier),
          computed_at = VALUES(computed_at),
          etl_batch_id = VALUES(etl_batch_id),
          etl_loaded_at = VALUES(etl_loaded_at)

Step 4: Drop staging table
        DROP TABLE customer_lifetime_value_staging

Step 5: Update SSM Parameter with new sync timestamp

Configuring Pre/Post Actions in Glue:

→ In the Glue job configuration → Advanced properties → JDBC target settings:

SettingValue
Pre-action SQLCREATE TABLE IF NOT EXISTS {table}_staging LIKE {table}; TRUNCATE TABLE {table}_staging;
Post-action SQLINSERT INTO {table} SELECT * FROM {table}_staging ON DUPLICATE KEY UPDATE ...; DROP TABLE {table}_staging;

→ These SQL statements execute BEFORE and AFTER the Glue data write
→ Glue writes to the staging table → post-action merges staging into production

✅ WHAT SHOULD DO

→ Always use the native Redshift connector (not generic JDBC) for reading large datasets — uses S3 staging for parallel reads
→ Always create a dedicated MariaDB user for reverse ETL with explicit WRITE permissions — separate from read-only ETL user
→ Always use staging table + atomic swap pattern for full-replace tables — prevents partial reads by applications
→ Always use UPSERT via staging table for incremental tables — handles both new and updated rows cleanly
→ Always track sync timestamps in SSM Parameter Store — update ONLY on successful completion
→ Always add etl_batch_id and etl_loaded_at metadata columns — enables troubleshooting and data lineage
→ Always configure lifecycle rules on the S3 staging bucket — prevent storage cost accumulation
→ Always schedule reverse ETL AFTER warehouse computation completes — ensure fresh data
→ Always use Step Functions for orchestration — provides visual execution history and error handling
→ Always monitor both source (Redshift) and target (MariaDB) health during sync — prevent resource exhaustion
→ Always use batch sizes of 1000 rows for MariaDB JDBC writes — balances throughput and lock contention
→ Always test type mappings with sample data before running full pipeline — catch precision/overflow issues early

❌ WHAT SHOULD NOT DO

→ Never write directly to production MariaDB tables in large batches — causes lock contention with live traffic
→ Never use Glue's overwrite mode on production tables — it drops and recreates the table (causes downtime)
→ Never update SSM sync timestamp before confirming successful MariaDB write — causes data loss on retry
→ Never skip the staging table pattern for tables serving live traffic — partial writes corrupt application data
→ Never use generic JDBC for reading large Redshift tables — row-by-row JDBC is 10-100x slower than native connector
→ Never share the same MariaDB user between forward ETL (read) and reverse ETL (write) — violates least privilege
→ Never schedule reverse ETL during peak application traffic hours — competes for MariaDB resources
→ Never skip security group updates for the new Glue job — connectivity failures are the #1 cause of job failures
→ Never ignore type mismatches between Redshift and MariaDB — silent truncation or precision loss corrupts data
→ Never run full-replace on large tables (> 100K rows) — use incremental sync instead
→ Never forget to clean up staging tables on job failure — orphaned staging tables accumulate

MECHANIC 1 — GLUE NATIVE REDSHIFT CONNECTOR FLOW

Glue Job starts with Redshift source configuration
        │

Glue generates a Redshift UNLOAD command:

  TO 's3://shopnow-glue-redshift-staging/temp/{job_run_id}/'

  - Each compute node writes its portion to S3 in parallel
  - Creates multiple Parquet files (one per slice per node)
  - Files: part-00000.parquet, part-00001.parquet, ...
        │
        ▼
Glue reads the Parquet files from S3:
  - Spark workers read files in parallel
  - Each worker processes a subset of files
  - Data loaded into Spark DynamicFrames
        │
        ▼
Glue performs transformations:
  - Type casting (BOOLEAN → INT, SUPER → STRING)
  - Add metadata columns (etl_batch_id, etl_loaded_at)
  - Handle NULLs and defaults
        │

  - Writes to staging table
        │
        ▼
Glue executes post-action SQL:
  - MERGE staging into production (UPSERT)
  - DROP staging table
        │

        ▼

MECHANIC 2 — JDBC WRITE PERFORMANCE INTERNALS

How Glue Writes to MariaDB via JDBC:

Glue Spark Job has N partitions of data
    │
    ├── Partition 1 (Worker 1):
    │     Opens JDBC connection to MariaDB
    │     Begins batch:
    │       INSERT INTO staging VALUES (row1), (row2), ..., (row1000)  ← 1 batch = 1000 rows
    │       INSERT INTO staging VALUES (row1001), ..., (row2000)
    │       ... continues until partition exhausted
    │     Closes connection
    │
    ├── Partition 2 (Worker 2):
    │     Opens JDBC connection to MariaDB (SEPARATE connection)
    │     Same batch INSERT pattern
    │
    ... (N partitions = N concurrent JDBC connections)
    │
    └── All partitions complete → post-action SQL executes

Performance Tuning Levers:

SettingImpactRecommendation
Batch sizeLarger batches = fewer round trips = faster1000-5000 rows per batch
Number of partitionsMore partitions = more parallel connections5-10 for MariaDB (too many = connection exhaustion)
MariaDB max_connectionsLimits concurrent JDBC connectionsEnsure headroom: app connections + Glue connections < max
MariaDB innodb_buffer_pool_sizeLarger = better write performanceEnsure sufficient for batch inserts
Network latency (Glue → MariaDB)Lower = faster per-batch round tripSame AZ deployment recommended

Connection Pooling Consideration:

→ Each Spark partition opens its OWN JDBC connection
→ 10 partitions = 10 concurrent connections to MariaDB
→ If MariaDB max_connections = 150 and application uses 100 → only 50 available for Glue
→ Set Glue partitions to < 50 to avoid connection exhaustion
→ Monitor RDS DatabaseConnections metric during sync

MECHANIC 3 — ATOMIC TABLE SWAP INTERNALS

Before swap:
  MariaDB tables:
    inventory_forecast (PRODUCTION — application reads from this)
    inventory_forecast_staging (NEW DATA — Glue just finished writing)

RENAME TABLE command:
  RENAME TABLE inventory_forecast TO inventory_forecast_old,
               inventory_forecast_staging TO inventory_forecast;

What happens internally:
  1. MariaDB acquires metadata lock on both tables
  2. Metadata lock duration: microseconds (only changes table name pointers)
  3. No data is moved — only the name-to-file mapping changes
  4. Application queries that were mid-flight:
     - Queries that started BEFORE rename → complete on old data (still valid)
     - Queries that start AFTER rename → read new data
  5. NO query fails — zero downtime

After swap:
  MariaDB tables:
    inventory_forecast (was staging — NOW PRODUCTION with new data)
    inventory_forecast_old (was production — NOW OLD, ready for deletion)

→ The RENAME TABLE operation is the SAFEST way to replace data in a live table
→ It's atomic at the metadata level — no data copying occurs
→ Application sees either ALL old data or ALL new data — never partial

📈 SCALABILITY CONSIDERATIONS

→ Data Volume Scaling

  • < 100K rows → Single Glue job, G.1X workers, 2-3 workers
  • 100K - 1M rows → 5 workers, batch UPSERT
  • 1M - 10M rows → 10 workers, consider partitioning by customer segment

  • 10M rows → Consider DMS CDC (Project 58) for continuous replication instead of batch

→ Table Count Scaling

  • 5 tables → Single workflow with parallel jobs (current design)
  • 10-20 tables → Multiple workflows grouped by priority/dependency
  • 50+ tables → Parameterized generic job with table-specific configurations in DynamoDB or SSM
  • Avoid creating 50 separate Glue jobs — use one parameterized job with different arguments

→ Frequency Scaling

  • Daily sync → Standard Glue job (this project)
  • Hourly sync → Reduce data volume per run (narrower timestamp window), watch MariaDB load
  • Near-real-time (< 5 min) → NOT Glue — switch to DMS CDC (Project 58) or Kinesis (Project 52)

→ MariaDB Write Scaling

  • Single RDS instance → Max ~5,000 writes/second sustained
  • With Glue batch UPSERT → ~50K rows in 30 seconds (well within limits)
  • If write volume exceeds capacity → Use RDS Proxy for connection pooling
  • If MariaDB becomes bottleneck → Consider Aurora MySQL for higher write throughput

→ Cost Scaling

  • Costs scale linearly with DPU-hours
  • Double the data → approximately double the DPU-hours
  • Optimize: Use Auto Scaling workers (Glue 4.0 feature) — start with minimum, scale up as needed
