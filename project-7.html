PROJECT 7: THE ORDERS THAT DISAPPEARED DURING PEAK HOURS — SQS Queue & Message Lifecycle

🔴 HYPOTHETICAL PROBLEM

→ ShopNow (from Projects 1-6) is growing rapidly
→ They launched a flash sale — "60% off on all electronics for 2 hours"
→ Within 15 minutes — 5,000 orders flooded in simultaneously
→ The order processing system runs on a single EC2 instance (Project 3)
→ It can process 10 orders per second → needs 500 seconds (8+ minutes) for 5,000 orders
→ But orders arrive at 300+ per second during peak

→ Problem 1: The web application calls the order processor SYNCHRONOUSLY
→ Web app sends order → waits for processor to finish → then responds to customer
→ When processor is overwhelmed → web app times out → customer sees "Error 504"
→ Customer tries again → creates DUPLICATE orders → inventory becomes incorrect

→ Problem 2: At 11:32 AM the order processor CRASHES (out of memory)
→ 2,847 orders were "in flight" — they hadn't been processed yet
→ The web app had already told customers "Order placed successfully"
→ But those 2,847 orders were in the processor's memory → LOST when it crashed
→ No way to recover them → customers paid but never received products
→ Customer support receives 2,847 angry calls

→ Problem 3: After restarting the processor → it starts processing NEW orders
→ The 2,847 lost orders are gone forever — nobody even knows WHICH orders were lost
→ No audit trail, no recovery mechanism

→ Problem 4: During non-peak hours → the processor sits idle consuming resources
→ During peak → it's completely overwhelmed
→ There's no way to absorb traffic spikes without losing messages

→ The CTO demands: "We need a system where:

  • Orders are NEVER lost — even if the processor crashes
  • The web app responds IMMEDIATELY — doesn't wait for processing
  • Orders WAIT in a queue if the processor is busy
  • When processor restarts — it picks up unprocessed orders automatically
  • During peak — orders queue up safely instead of being dropped"

🟡 HOW TO APPROACH IT

→ This is a MESSAGE QUEUING problem
→ The solution pattern: ASYNCHRONOUS DECOUPLING

Synchronous (Current — Broken):

Customer → Web App → Order Processor → Database
                     ↑
           Web app WAITS here
           If processor is slow/dead → customer gets error

Asynchronous (Solution — Queue-Based):

Customer → Web App → SQS QUEUE → Order Processor → Database
                  ↑              ↑
           Web app puts          Processor reads
           message in queue      messages at its
           → responds            own pace
           immediately

→ The QUEUE is the buffer between producer (web app) and consumer (processor)
→ Think of it like a restaurant kitchen:

  • Waiter (web app) takes order → puts ticket on the order rail (queue) → goes back to customer
  • Chef (processor) picks tickets from the rail at their own speed
  • If the chef is busy → tickets queue up on the rail → none are lost
  • If the chef goes home → tickets stay on the rail → next chef picks them up

→ AWS SQS (Simple Queue Service) is purpose-built for this pattern

→ The approach should be:

  • Step 1 — Understand SQS queue types (Standard vs FIFO)
  • Step 2 — Create an SQS queue
  • Step 3 — Understand the message lifecycle
  • Step 4 — Understand visibility timeout (the most critical concept)
  • Step 5 — Understand polling (short vs long)
  • Step 6 — Configure message retention and delivery delay
  • Step 7 — Understand message attributes
  • Step 8 — Configure queue access policies

🟢 HOW CAN WE SOLVE IT — Step-by-Step Configuration

STEP 1 — UNDERSTAND STANDARD vs FIFO QUEUES

→ SQS offers TWO fundamentally different queue types
→ Choosing wrong = architectural problems later

FeatureStandard QueueFIFO Queue
Message orderingBEST-EFFORT ordering (mostly ordered, not guaranteed)STRICT first-in-first-out ordering
Delivery guaranteeAT-LEAST-ONCE (message may be delivered more than once)EXACTLY-ONCE processing (within deduplication window)
ThroughputNearly UNLIMITED (thousands of messages/second)300 messages/second (3,000 with batching)
DeduplicationNo built-in deduplicationBuilt-in 5-minute deduplication window
Use caseHigh throughput, order not criticalOrder matters, no duplicates allowed
NamingAny nameMUST end with .fifo suffix

Which Should ShopNow Use?

→ Analysis:

RequirementStandardFIFO
Orders processed in exact sequence?Not critical — Order #1001 can process before #1000Would guarantee sequence
Duplicate processing acceptable?DANGEROUS — customer charged twicePrevents duplicates
Volume: 5,000+ orders in minutesHandles easily300/sec might be borderline
Cost sensitivityCheaper~25% more expensive

→ Decision: FIFO Queue
→ Reason: Duplicate orders are UNACCEPTABLE (customer charged twice)
→ The throughput limit (300/sec, 3,000 with batching) is sufficient for ShopNow's volume
→ If throughput were an issue → Standard Queue with APPLICATION-LEVEL deduplication

When to Use Standard:
→ Log processing — duplicates are harmless, order doesn't matter
→ Fan-out processing — SNS → SQS where throughput matters most
→ Thumbnail generation — processing image twice is wasteful but not dangerous

When to Use FIFO:
→ Financial transactions — duplicate charges are unacceptable
→ Order processing — sequence and uniqueness matter
→ Inventory updates — out-of-order updates corrupt stock counts

PHASE 4: MESSAGE EXPIRATION
────────────────────────────
If message sits in queue for longer than retention period (7 days)
→ SQS automatically deletes it → LOST
This should NEVER happen in normal operations — it means something is broken

The Three Critical API Calls:

API CallWho Calls ItWhat It Does
SendMessageProducer (web app)Puts a message into the queue
ReceiveMessageConsumer (processor)Reads message(s) from the queue — starts visibility timeout
DeleteMessageConsumer (processor)Permanently removes a successfully processed message

→ The consumer MUST explicitly call DeleteMessage after successful processing
→ If it doesn't → the message comes back (visibility timeout expires) → gets processed AGAIN
→ This is the AT-LEAST-ONCE delivery guarantee in Standard queues
→ In FIFO queues → deduplication prevents re-processing within the 5-minute window

STEP 4 — VISIBILITY TIMEOUT DEEP DIVE

→ This is the MOST MISUNDERSTOOD concept in SQS
→ Getting it wrong causes either duplicate processing or lost messages

What Is Visibility Timeout?

→ When Consumer A reads a message → the message becomes INVISIBLE to Consumer B, C, D
→ This prevents multiple consumers from processing the SAME message simultaneously
→ The invisibility lasts for the configured visibility timeout duration

Visual Timeline:

Time 0s:     Consumer A reads Msg 1 → Msg 1 becomes invisible
             Visibility timeout starts: 60 seconds
             
Time 0-30s:  Consumer A is processing Msg 1
             Consumer B calls ReceiveMessage → does NOT get Msg 1 (invisible)
             Consumer B gets Msg 2 instead
             
Time 30s:    Consumer A finishes → calls DeleteMessage → Msg 1 GONE ✓
             (Happy path — processed and deleted within timeout)

ALTERNATIVE SCENARIO:

Time 0s:     Consumer A reads Msg 1 → invisible for 60s
Time 45s:    Consumer A CRASHES
Time 60s:    Visibility timeout EXPIRES → Msg 1 becomes VISIBLE again
Time 61s:    Consumer B reads Msg 1 → processes it → deletes it ✓
             (Recovery path — message NOT lost despite crash)

PROBLEMATIC SCENARIO:

Time 0s:     Consumer A reads Msg 1 → invisible for 60s
Time 55s:    Consumer A is STILL processing (slow processing)
Time 60s:    Visibility timeout EXPIRES → Msg 1 becomes VISIBLE again
Time 61s:    Consumer B reads Msg 1 → starts processing
Time 65s:    Consumer A finishes → deletes Msg 1
Time 90s:    Consumer B finishes → tries to delete → message already gone
             RESULT: Msg 1 processed TWICE → customer charged twice! 💥

How to Set Visibility Timeout Correctly:

Processing TimeRecommended Visibility TimeoutWhy
5 seconds30 seconds6x buffer for occasional slow processing
30 seconds90 seconds3x buffer
5 minutes15 minutes3x buffer
Unknown/variableUse ChangeMessageVisibility APIExtend timeout dynamically during processing

→ Rule of thumb: Visibility timeout = 3-6x average processing time
→ Too short → messages reappear → duplicate processing
→ Too long → crashed messages take forever to become available again → processing delay

ChangeMessageVisibility — Dynamic Extension:

→ If the consumer realizes processing will take longer than expected:
→ Call ChangeMessageVisibility BEFORE the timeout expires
→ This RESETS the visibility timeout to a new duration
→ The consumer can keep extending as needed (heartbeat pattern)

Time 0s:     Consumer reads message (visibility = 60s)
Time 45s:    Consumer still processing → calls ChangeMessageVisibility(120s)
             Timeout now: 120 seconds from NOW (not from original read)
Time 150s:   Consumer finishes → deletes message ✓

→ This is the RECOMMENDED pattern for variable-duration processing
→ Consumer sends "heartbeat" extensions every N seconds while still processing

STEP 5 — SHORT POLLING vs LONG POLLING

→ Polling is HOW the consumer retrieves messages from the queue
→ SQS is a PULL-based system — consumer ASKS for messages (unlike SNS which PUSHES)

Short Polling (Default if receive wait time = 0):

Consumer: "Any messages?"
SQS: Checks a SUBSET of servers → "No" (even if messages exist on other servers)
Consumer: "Any messages?"
SQS: Checks another subset → "No"
Consumer: "Any messages?"
SQS: Checks another subset → "Here's one message"

Problem: Multiple empty responses → wasted API calls → wasted money
Problem: Messages on unchecked servers → temporary invisibility → delay

→ Short polling queries only a SUBSET of SQS servers
→ May return empty response even when messages exist
→ Results in unnecessary API calls and costs

Long Polling (receive wait time = 1-20 seconds):

Consumer: "Any messages? I'll wait up to 20 seconds"
SQS: Queries ALL servers → waits → message arrives at second 3
SQS: "Here's a message" (returns after 3 seconds)

If no message arrives within 20 seconds:
SQS: "No messages" (returns after 20 seconds)

→ Long polling queries ALL SQS servers
→ Waits until a message arrives OR the wait time expires
→ Reduces empty responses → fewer API calls → lower cost
→ Returns messages faster (as soon as one is available)

Comparison:

AspectShort PollingLong Polling
Wait time setting0 seconds1-20 seconds
Servers queriedSubsetALL
Empty responsesFrequentRare (only if truly empty for full wait period)
API costHigher (many calls)Lower (fewer calls)
Latency to get messageMay miss messages temporarilyGets messages as soon as available
CPU usage on consumerHigher (constant polling loop)Lower (waits between polls)

→ ALWAYS use Long Polling (20 seconds) for production
→ There is NO downside to long polling — it's strictly better
→ The only exception: If you need to check multiple queues rapidly and can't wait 20 seconds per queue

How to Configure:

→ Queue-level: Set "Receive message wait time" to 20 seconds during queue creation
→ This applies to ALL ReceiveMessage calls on this queue
→ Per-request: Consumer can override by specifying WaitTimeSeconds in the API call
→ Per-request setting overrides queue-level setting

STEP 7 — FIFO-SPECIFIC CONCEPTS

Message Group ID:

→ FIFO queues guarantee ordering WITHIN a message group
→ Messages in DIFFERENT groups can be processed in parallel

Message Group: "CUST-5678"
  → Order A (sent first)
  → Order B (sent second)
  → Order C (sent third)
  → MUST be processed: A → B → C (in order)

Message Group: "CUST-9012"
  → Order X (sent first)
  → Order Y (sent second)
  → MUST be processed: X → Y (in order)

But Order A and Order X can be processed SIMULTANEOUSLY
(different groups = independent ordering)

→ ShopNow: Use customer_id as Message Group ID
→ Orders for the same customer process in sequence (prevent inventory conflicts)
→ Orders for different customers process in parallel (maximize throughput)

Choosing Message Group ID:

StrategyGroup IDOrdering GuaranteeThroughput
Single group for everything"all-orders"ALL orders in strict sequenceLOW (300 msg/sec max)
Per customercustomer_idPer-customer orderingHIGH (300 × num_customers)
Per productproduct_idPer-product orderingHIGH (300 × num_products)
Per regionregion_idPer-region orderingMEDIUM (300 × num_regions)

→ Single group = safest ordering but lowest throughput
→ Per customer = best balance for ShopNow

Message Deduplication ID:

→ Prevents the same message from being enqueued twice within 5 minutes
→ Two options:

MethodHow It Works
Content-based deduplication (enabled in Step 2)SQS hashes the message body → if identical body sent within 5 minutes → second message discarded
Explicit deduplication IDProducer provides MessageDeduplicationId parameter → SQS checks if this ID was used in last 5 minutes

→ Content-based: Simple but only catches IDENTICAL bodies
→ Explicit ID: More control — use order_id as deduplication ID

Example:
→ Customer double-clicks "Place Order"
→ Web app sends same order message twice in 1 second
→ Both messages have order_id: ORD-2025-00001 in the body
→ Content-based deduplication: Body hash is identical → second message DISCARDED
→ Or explicit: Both have MessageDeduplicationId: ORD-2025-00001 → second DISCARDED

→ Controls WHO can send to and receive from the queue

→ Receipt Handle is CRITICAL:

  • It's unique to THIS specific receive action
  • You MUST use it to delete the message or change its visibility
  • It's different each time the same message is received
  • It EXPIRES when the visibility timeout expires
  • After expiry → the receipt handle becomes INVALID → must receive the message again to get a new one

Deleting a Message (After Successful Processing):

→ Consumer processes the order successfully
→ Consumer calls DeleteMessage with the receipt handle
→ Message is PERMANENTLY removed from the queue

If Consumer Doesn't Delete:
→ Visibility timeout expires → message becomes visible again
→ Another consumer (or same consumer) receives it again
→ ApproximateReceiveCount increments
→ This cycle continues until:

  • Message is successfully processed and deleted
  • Message reaches the Dead Letter Queue threshold (Project 20)
  • Message retention period expires (7 days) → auto-deleted

✅ WHAT SHOULD DO

→ Always use FIFO queues when message ordering or deduplication matters
→ Always use Long Polling (receive wait time = 20 seconds) — reduces cost and improves responsiveness
→ Always set visibility timeout to 3-6x your average processing time
→ Always delete messages AFTER successful processing — never before
→ Always use the ChangeMessageVisibility API for variable-duration processing (heartbeat pattern)
→ Always encrypt queues with at minimum SSE-SQS
→ Always monitor ApproximateAgeOfOldestMessage — growing age means consumers can't keep up
→ Always use meaningful Message Group IDs in FIFO queues — they determine parallelism
→ Always use the claim-check pattern for large payloads (store data in S3, send reference in SQS)
→ Always add queue access policy conditions (SourceArn, SourceAccount) when allowing services to send
→ Always configure Dead Letter Queues for poison messages (Project 20)
→ Always tag queues with Name, Environment, Owner, Application

❌ WHAT SHOULD NOT DO

→ Never use Short Polling in production — wastes API calls and money
→ Never set visibility timeout too short — causes duplicate processing
→ Never set visibility timeout too long — delayed recovery when consumers crash
→ Never forget to delete messages after processing — they'll come back and be processed again
→ Never use a single Message Group ID for all messages in FIFO — creates a bottleneck (300 msg/sec for everything)
→ Never rely on message retention period as a normal operating parameter — it's a safety net
→ Never store sensitive data in messages without encryption
→ Never assume Standard queue ordering is guaranteed — it's best-effort only
→ Never put large payloads directly in the message — use claim-check pattern for > 50 KB
→ Never forget that FIFO queue names MUST end with .fifo — the API will reject the request otherwise
→ Never ignore the ApproximateReceiveCount — high counts indicate processing failures
→ Never assume SQS is a database — it's a transient message buffer, not permanent storage

5 → HOW FIFO ORDERING WORKS INTERNALLY

FIFO Queue with Message Group ID "CUST-5678":

Messages in group (in order):
  Msg-A (Seq: 1) → Msg-B (Seq: 2) → Msg-C (Seq: 3)

Consumer calls ReceiveMessage:
  → Returns Msg-A (Seq: 1)
  → Msg-B and Msg-C are BLOCKED for this group
  → They will NOT be returned until Msg-A is deleted or visibility expires

Consumer deletes Msg-A:
  → Msg-B becomes available
  → Consumer calls ReceiveMessage → Returns Msg-B
  
This guarantees: A before B before C (FIFO within group)

SIMULTANEOUSLY:
  Message Group "CUST-9012":
  Msg-X (Seq: 4) → Msg-Y (Seq: 5)
  
  A different consumer can receive Msg-X while first consumer has Msg-A
  Different groups are INDEPENDENT

→ FIFO ordering is enforced per MESSAGE GROUP, not per queue
→ The queue blocks subsequent messages in the SAME group until current message is processed
→ Different groups can be processed concurrently by different consumers
→ This is why choosing the right Message Group ID is critical for throughput

6 → HOW STANDARD QUEUE "AT-LEAST-ONCE" WORKS

Standard Queue — Message stored on servers S1, S2, S3 (replicas)

Consumer A reads from S1 → Gets Msg-1 → S1 marks it invisible
Consumer B reads from S2 → S2 hasn't received the "invisible" signal yet
                           → S2 returns Msg-1 to Consumer B too!

Result: Both Consumer A and Consumer B process Msg-1
This is the "at-least-once" delivery behavior

FIFO queues PREVENT this by:
  → Centralizing visibility tracking
  → Using sequence numbers for ordering
  → Ensuring only ONE consumer gets each message
  → This is why FIFO has lower throughput (more coordination needed)

→ Standard queues trade consistency for throughput
→ FIFO queues trade throughput for consistency
→ This is a fundamental distributed systems tradeoff

📈 SCALABILITY CONSIDERATIONS

→ Throughput Scalability

  • Standard queues: Nearly unlimited throughput → scales automatically
  • FIFO queues: 300 msg/sec per group → scale by using more message groups
  • Batch operations: 10 messages per batch → effective 3,000 msg/sec per FIFO group
  • Multiple queues: Distribute workload across queues for extreme throughput

→ Consumer Scalability

  • Multiple consumers can read from the same queue simultaneously
  • Each consumer gets different messages (SQS ensures this via visibility timeout)
  • Add more consumers to process faster → horizontal scaling
  • For FIFO: Multiple consumers can process different message groups in parallel
  • Auto Scaling consumers based on queue depth (ApproximateNumberOfMessagesVisible)

→ Storage Scalability

  • No limit on number of messages in a queue
  • Messages are distributed across SQS infrastructure
  • No provisioning needed — queue grows and shrinks automatically
  • Retention up to 14 days for buffering extended outages

→ Cost Scalability

  • Costs scale linearly with request count
  • Batch operations reduce costs by up to 10x
  • Long polling reduces unnecessary requests
  • First million requests/month are free → small workloads are essentially free

→ Multi-Consumer Patterns

Pattern 1: Competing Consumers (Work Distribution)
─────────────────────────────────────────────────
  Queue → Consumer A gets Msg 1
       → Consumer B gets Msg 2
       → Consumer C gets Msg 3
  Each message processed by ONE consumer
  Used for: parallel processing, load distribution

Pattern 2: Fan-Out (via SNS → Multiple SQS Queues)
──────────────────────────────────────────────────
  SNS Topic → SQS Queue A → Consumer A (image resize)
           → SQS Queue B → Consumer B (catalog update)
           → SQS Queue C → Consumer C (analytics)
  Each message processed by ALL consumers
  Used for: multiple processing pipelines (Project 13)
