PROJECT 22: THE DATA WAREHOUSE THAT NEEDED BULK LOADING — Redshift COPY from S3

🔴 HYPOTHETICAL PROBLEM

→ ShopNow's data team has built an impressive pipeline so far:

  • Raw CSV/JSON lands in S3 (Project 1)
  • Glue Crawler catalogs it (Project 17)
  • Glue ETL converts it to optimized Parquet (Project 21)
  • Athena queries the data lake directly (Project 18)

→ But the business analysts have new requirements that Athena cannot efficiently handle:

→ Problem 1: COMPLEX JOINS AT SCALE

  • The CFO wants a report joining orders + customers + products + clickstream
  • 4-table JOIN across 50 GB of data
  • Athena can do it — but it takes 8 minutes and costs $0.25 per execution
  • The report runs 20 times/day (analysts tweaking parameters) = $5/day = $150/month for ONE report

→ Problem 2: DASHBOARD CONCURRENCY

  • 15 analysts run queries simultaneously against the same data
  • Athena has per-account concurrency limits (default 20-25 concurrent queries)
  • During morning standup — everyone runs their dashboards at 9 AM → queries queue up
  • Some queries timeout → dashboards show stale data

→ Problem 3: SUB-SECOND AGGREGATIONS

  • The VP of Sales wants a dashboard that refreshes every 30 seconds
  • Athena's minimum query time is 2-5 seconds even for simple queries (S3 scan overhead)
  • Redshift stores data locally on compute nodes → sub-second for pre-loaded data

→ Problem 4: HISTORICAL ANALYTICS WITH COMPLEX WINDOW FUNCTIONS

  • "Show me the rolling 90-day average revenue per customer segment, ranked by growth rate"
  • Complex window functions across billions of rows
  • Athena can do this but extremely slowly — Redshift is purpose-built for it

→ The data engineering lead decides: "We need to LOAD our optimized Parquet data from S3 into Redshift for high-performance analytics"

→ The Redshift cluster from Project 16 is running — but the tables are empty
→ The question: How do we efficiently bulk-load terabytes of data from S3 into Redshift?

→ The naive approach: INSERT statements one row at a time
→ Result: 38 GB of data would take ~72 hours to load row-by-row
→ The correct approach: Redshift COPY command — designed for massively parallel bulk loading

🟡 HOW TO APPROACH IT

→ First — understand why COPY is fundamentally different from INSERT

ApproachMechanismSpeed for 38 GBWhy
INSERT (row-by-row)Single leader node processes each row sequentially~72 hoursNo parallelism, network round-trip per row
INSERT via multi-row batchesLeader node processes batches~12 hoursBetter but still single-node bottleneck
COPY from S3ALL compute nodes read from S3 in PARALLEL~15 minutesMassively parallel — each node loads its portion

→ COPY is not just "faster INSERT" — it's a completely different loading architecture
→ Every compute node in the Redshift cluster reaches directly into S3 and pulls data simultaneously

→ The approach should be:

  • Step 1 — Understand Redshift's architecture for data loading (Leader vs Compute nodes)
  • Step 2 — Create IAM Role for Redshift to access S3
  • Step 3 — Associate the IAM Role with the Redshift cluster
  • Step 4 — Create target tables in Redshift (schema design)
  • Step 5 — Configure and execute the COPY command
  • Step 6 — Understand COPY parameters for different formats
  • Step 7 — Handle errors and validation
  • Step 8 — Optimize loading performance (file splitting, sort keys, distribution)
  • Step 9 — Automate the loading process

STEP 1 — UNDERSTAND REDSHIFT DATA LOADING ARCHITECTURE

→ Recall from Project 16: Redshift has a Leader Node and Compute Nodes

                     │   LEADER NODE    │
                     │                  │
                     │  Receives COPY   │
                     │  command         │
                     │  Plans loading   │
                     │  Distributes     │
                     │  work to compute │
                     │  nodes           │
                     └────────┬─────────┘
                              │
              ┌───────────────┼───────────────┐
              │               │               │
     ┌────────▼──────┐ ┌─────▼───────┐ ┌─────▼───────┐
     │ COMPUTE NODE 1│ │COMPUTE NODE 2│ │COMPUTE NODE 3│
     │               │ │              │ │              │
     │  Reads files  │ │ Reads files  │ │ Reads files  │
     │  from S3      │ │ from S3      │ │ from S3      │

     │               │ │              │ │              │
     │  Parses data  │ │ Parses data  │ Parses data   │
     │  Loads into   │ │ Loads into   │ │ Loads into   │

     └───────────────┘ └──────────────┘ └──────────────┘
              │               │               │
              ▼               ▼               ▼
         S3 Bucket (multiple files read in parallel)

→ Key insight: Compute nodes read DIRECTLY from S3
→ The Leader Node does NOT funnel data through itself
→ Each compute node has multiple "slices" (processing units)
→ Each slice can read one file independently

→ This means: more files in S3 = more parallelism = faster loading

Optimal file count for COPY:

Cluster ConfigurationSlices per NodeTotal SlicesOptimal File Count
2 × dc2.large244 or multiples of 4
4 × dc2.large288 or multiples of 8
2 × ra3.xlplus244 or multiples of 4
2 × ra3.4xlarge488 or multiples of 8
4 × ra3.4xlarge41616 or multiples of 16

→ Rule: Number of input files should be a MULTIPLE of the total number of slices
→ This ensures every slice gets equal work — no idle slices while others are still loading

→ From Project 21 — Glue ETL writes multiple Parquet files per partition
→ If Glue wrote 12 files and cluster has 4 slices → 12/4 = 3 files per slice → good distribution
→ If Glue wrote 1 giant file → only 1 slice reads it → 3 slices idle → 4x slower

Redshift-Specific Table Properties (Critical for Performance):

Property 1: DISTSTYLE (Distribution Style)

→ How Redshift distributes rows across compute nodes

DISTSTYLEBehaviorBest For
AUTORedshift chooses based on table sizeDefault — good starting point
EVENRows distributed round-robin across all slicesLarge tables not frequently joined
KEYRows with same key value go to same nodeTables frequently joined on that key
ALLEntire table copied to every nodeSmall dimension tables (< 5 million rows)

→ For orders table: DISTSTYLE KEY with DISTKEY customer_id
→ Why? Most analytical queries join orders with customers on customer_id
→ If both tables use customer_id as DISTKEY → joined rows are on the SAME node → no network shuffle

→ For products table (small — 50,000 rows): DISTSTYLE ALL
→ Every node has a complete copy → joins with any table require no network transfer

Property 2: SORTKEY

→ How data is physically ordered on disk within each slice

SORTKEY TypeBehaviorBest For
COMPOUNDData sorted by key1, then key2, then key3Queries that filter by leading key columns
INTERLEAVEDEqual weight to all key columnsQueries that filter by any combination of keys
AUTORedshift choosesDefault — reasonable starting point

→ For orders table: COMPOUND SORTKEY (order_date)
→ Why? Most queries filter by date range: "Show me orders from last 30 days"
→ With date as sort key:

  • Redshift stores data sorted by date on disk
  • Zone maps (min/max per block) allow skipping entire blocks
  • "Last 30 days" query reads only recent blocks — skips 95% of data

→ For clickstream table: COMPOUND SORTKEY (event_date, user_id)
→ Primary filter is date, secondary filter is user

Property 3: ENCODE (Compression)

→ Redshift compresses columns individually

SettingBehavior
AUTORedshift chooses best encoding per column during COPY
RAWNo compression (wastes space)
Specific (AZ64, LZO, ZSTD, etc.)Manual override

→ Use AUTO — Redshift analyzes the first 100,000 rows during COPY and chooses optimal encoding
→ This is called "automatic compression analysis"

What happens internally when COPY executes:

COPY command received by Leader Node
       ↓
Leader reads S3 file listing from the specified path
       ↓
Leader identifies all Parquet files (recursively in subfolders)
       ↓
Leader distributes files across compute node slices
  ├── Slice 1 → files [1, 5, 9, ...]
  ├── Slice 2 → files [2, 6, 10, ...]
  ├── Slice 3 → files [3, 7, 11, ...]
  └── Slice 4 → files [4, 8, 12, ...]
       ↓
Each slice reads its assigned files from S3 IN PARALLEL
       ↓
Each slice parses Parquet → extracts rows
       ↓
Automatic compression analysis (first COPY only):
  ├── Analyze first 100,000 rows
  ├── Choose optimal encoding per column
  └── Apply compression
       ↓
Each slice applies distribution:
  ├── Rows matching this node's DISTKEY range → stored locally
  └── Rows for other nodes → sent via network (redistribution)
       ↓
Each slice sorts data by SORTKEY within its portion
       ↓
Data is written to Redshift's columnar storage
       ↓
COPY completes → returns row count

STEP 7 — HANDLE ERRORS AND VALIDATION

Understanding MAXERROR:

MAXERROR ValueBehavior
0 (default)ANY error aborts the entire COPY — zero tolerance
100Up to 100 rows can fail — rest still loads
-1No limit — load whatever you can (dangerous in production)

→ For initial data load: MAXERROR = 0 — catch ALL issues
→ For ongoing incremental loads: MAXERROR = 100 — tolerate occasional bad records
→ Never use -1 in production — silently ignoring thousands of errors = data quality disaster

→ Updates table statistics used by the query optimizer

✅ WHAT SHOULD DO

→ Always use COPY for bulk loading — never INSERT for large datasets
→ Always use an IAM Role for S3 access — never embed access keys in the COPY command
→ Always split source data into multiple files — one file per slice minimum
→ Always use Parquet as source format when possible — schema embedded, compressed, fastest loading
→ Always set MAXERROR appropriately — 0 for initial loads, small number for incremental
→ Always check stl_load_errors after every COPY — even "successful" loads may have skipped rows
→ Always run VACUUM and ANALYZE after large loads
→ Always design tables with appropriate DISTKEY and SORTKEY BEFORE loading
→ Always load smallest tables first to validate the process
→ Always use COMPUPDATE ON for first load, OFF for subsequent loads
→ Always verify row counts match between source (S3/Athena) and target (Redshift)
→ Always consider file sizes — 100 MB to 1 GB per file is optimal

❌ WHAT SHOULD NOT DO

→ Never use INSERT for bulk loading — COPY is 100x+ faster
→ Never embed AWS credentials in the COPY command — use IAM Roles
→ Never load from a single giant file — it eliminates parallelism
→ Never skip error checking after COPY — stl_load_errors reveals silent data issues
→ Never forget VACUUM after loads — unsorted data degrades query performance over time
→ Never set MAXERROR to -1 in production — you'll silently lose data
→ Never load CSV when Parquet is available — CSV requires more configuration and is slower
→ Never ignore DISTKEY design — wrong distribution causes data skew → slow queries
→ Never load into a table without SORTKEY — full table scans become necessary
→ Never run COPY during peak query hours if avoidable — COPY consumes cluster resources
→ Never assume COPY is atomic across all files — if one file fails with MAXERROR exceeded, partial data may be loaded

BehaviorMAXERROR Not ExceededMAXERROR Exceeded
Successfully loaded rowsCommittedRolled back — nothing loaded
Error rowsSkipped (logged in stl_load_errors)N/A — entire COPY aborted

→ Never use FLOAT/REAL for monetary values — use DECIMAL for precision

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

1 → COPY EXECUTION PHASES

PHASE 1: PLANNING (Leader Node)
  ├── Parse COPY command syntax
  ├── Validate IAM Role permissions
  ├── Connect to S3 and list files in specified path
  ├── If manifest specified → read manifest → validate files exist
  ├── Count total files → plan distribution across slices
  └── Send file assignments to compute nodes

PHASE 2: READING (Compute Nodes — Parallel)
  ├── Each slice opens connection to S3
  ├── Each slice reads its assigned files
  ├── For Parquet: read file footer → identify relevant row groups
  ├── For CSV: read line by line, split by delimiter
  ├── For compressed files: decompress on-the-fly
  └── Parse into columnar format

PHASE 3: DISTRIBUTING (Compute Nodes)
  ├── For each row, compute DISTKEY hash
  ├── If row belongs to this node → store locally
  ├── If row belongs to another node → send via interconnect
  └── Receive rows from other nodes destined for this node

PHASE 4: SORTING (Compute Nodes)
  ├── Sort received rows by SORTKEY within each slice
  ├── Merge sorted runs
  └── Write to columnar storage blocks (1 MB each)

PHASE 5: COMPRESSION (First COPY only, if COMPUPDATE ON)
  ├── Analyze first 100,000 rows per column
  ├── Test multiple encoding algorithms
  ├── Choose encoding with best compression ratio
  └── Apply chosen encoding to all data

PHASE 6: COMMIT (Leader Node)
  ├── Verify all slices completed successfully
  ├── Check error count vs MAXERROR threshold
  ├── If within threshold → COMMIT transaction
  ├── If exceeded → ROLLBACK entire load
  ├── Update table statistics (if STATUPDATE ON)
  └── Return row count to client

→ Parquet COPY is:

  • Faster (no string parsing)
  • Simpler (fewer parameters)
  • More reliable (schema embedded, types pre-validated)
  • More efficient (columnar read — only needed columns)

→ This is why Project 21 (CSV → Parquet conversion) is a prerequisite for optimal Redshift loading

3 → DISTRIBUTION KEY MECHANICS DURING COPY

→ When COPY loads data into a table with DISTSTYLE KEY:

Row arrives: {customer_id: "CUST-1234", order_id: "ORD-5678", ...}
       ↓
Hash function applied to DISTKEY value:
  hash("CUST-1234") = 7482931
       ↓
Hash mapped to slice:
  7482931 mod 4 (total slices) = slice 3
       ↓
Row sent to slice 3 on compute node 2
       ↓
If the reading slice IS slice 3 → store locally (fast)
If the reading slice is NOT slice 3 → send over network (slower)

→ This redistribution happens during COPY
→ If source files are already organized by DISTKEY → less redistribution → faster load
→ If source files are random → significant redistribution → still fast but slightly slower

→ For tables with DISTSTYLE ALL (like products):

  • Every row is copied to EVERY node
  • Load time is slightly longer
  • But ALL subsequent joins with this table are local (no network)

4 → ZONE MAPS AND SORTKEY DURING COPY

→ As data is loaded and sorted by SORTKEY, Redshift creates "zone maps"

SORTKEY = order_date

Block 1: [2024-01-01 to 2024-01-15] ← zone map: min=2024-01-01, max=2024-01-15
Block 2: [2024-01-16 to 2024-01-31] ← zone map: min=2024-01-16, max=2024-01-31
Block 3: [2024-02-01 to 2024-02-14] ← zone map: min=2024-02-01, max=2024-02-14
...
Block 24: [2025-01-01 to 2025-01-15] ← zone map: min=2025-01-01, max=2025-01-15

→ Query: "WHERE order_date >= '2025-01-01'"
→ Redshift checks zone maps:

  • Block 1: max = 2024-01-15 < 2025-01-01 → SKIP
  • Block 2: max = 2024-01-31 < 2025-01-01 → SKIP
  • ... (skip blocks 1-23)
  • Block 24: min = 2025-01-01 → READ
    → 23 out of 24 blocks skipped → 96% less I/O

→ This is why SORTKEY on the most filtered column is critical
→ Without SORTKEY → every query reads every block → no zone map benefit

5 → AUTOMATIC COMPRESSION ANALYSIS DETAILS

→ When COMPUPDATE ON (first load):

COPY reads first 100,000 rows
       ↓
For each column, Redshift tests multiple encodings:
  ├── RAW (no compression)
  ├── AZ64 (Amazon's proprietary — best for numeric/date)
  ├── LZO (fast, moderate ratio)
  ├── ZSTD (balanced speed and ratio)
  ├── BYTEDICT (for low-cardinality columns — < 256 unique values)
  ├── DELTA (for sequential values — timestamps, auto-increment)
  ├── MOSTLY8/16/32 (for integers that mostly fit in smaller types)
  ├── RUNLENGTH (for columns with many consecutive repeated values)
  └── TEXT255/TEXT32K (for short/long text columns)
       ↓
For each column, compare compressed size
       ↓
Choose encoding with smallest compressed size
       ↓
Apply to the column for ALL data (not just sample)

→ Incremental Loading Pattern

PatternMechanismComplexity
Full reloadTRUNCATE → COPY all dataSimple but slow for large tables

→ For ShopNow daily loads:

  • Orders: Append + occasional upsert (order status updates)
  • Clickstream: Append only (events never change)
  • Products: Full reload (small table, changes unpredictably)
  • Customers: Upsert (customer info updates)

🎯 KEY TAKEAWAY

→ COPY is the ONLY way to efficiently bulk-load data into Redshift — INSERT is 100x+ slower for large datasets
→ COPY's power comes from PARALLEL reading — every compute node slice reads from S3 simultaneously
→ The pipeline is now: Raw CSV → Glue ETL (Project 21) → Parquet in S3 → COPY → Redshift

→ Post-load maintenance (VACUUM, ANALYZE) is mandatory — not optional
→ File splitting strategy directly impacts load speed — match file count to slice count
