🟡 HOW TO APPROACH IT

→ CloudWatch Events was the original event routing service

→ CloudWatch Events rules are visible in EventBridge console (they're the same underlying service)
→ New features (custom event buses, schema registry, archives, partner integrations) are EventBridge-only

Component 1 — Event Bus:
→ A channel where events are published and consumed
→ Three types of event buses:

Bus TypeNamePurposeCost
DefaultdefaultReceives ALL AWS service events automaticallyEvents from AWS services are FREE
CustomYou name it (e.g., shopnow-app-events)Application-generated events$1.00 per million events published
PartnerThird-party name (e.g., aws.partner/datadog.com/...)Events from SaaS partnersVaries

→ CRITICAL: AWS service events (S3, Glue, RDS, etc.) ONLY go to the default event bus
→ You CANNOT redirect AWS service events to a custom bus directly
→ Custom buses are for YOUR application events

Component 2 — Rules:
→ A rule matches events based on a pattern and routes them to targets
→ Each rule has:

  • An event pattern (what to match) OR a schedule expression (when to trigger)
  • One or more targets (where to send matching events)
  • An IAM role (permissions to invoke the target)
  • Optional input transformation (modify the event before sending to target)

Component 3 — Targets:
→ The destination service that receives the matched event

Supported TargetUse Case
Lambda functionCustom processing, transformation, orchestration
SNS topicFan-out notifications (Project 6, 13)
SQS queueBuffered processing (Project 7)

Kinesis Data StreamReal-time streaming (Project 52 preview)
ECS taskContainer-based processing
CodePipelineCI/CD trigger
Systems Manager AutomationOperational remediation
Another EventBridge busCross-account or cross-region routing
API GatewayHTTP endpoint invocation
Redshift Data APIExecute Redshift queries
CloudWatch LogsLog events for analysis

→ A single rule can have up to 5 targets
→ All targets receive the SAME event (fan-out within a rule)
→ For different targets based on different patterns → use separate rules

Type 1 — Rate Expression:

ExpressionMeaning
rate(5 minutes)Every 5 minutes

Type 2 — Cron Expression:

FieldValuesAllowed
Minute0-59
Hour0-23
Day-of-month1-31
Month1-12 or JAN-DEC
Day-of-week1-7 or SUN-SAT
Year1970-2199
ExpressionMeaning
cron(0 2 * * ? *)Every day at 2:00 AM UTC
cron(0 6 ? * MON-FRI *)Every weekday at 6:00 AM UTC
cron(0 0 1 * ? *)First day of every month at midnight
cron(0/15 * * * ? *)Every 15 minutes
cron(0 18 ? * FRI *)Every Friday at 6:00 PM UTC

CRITICAL: Cron uses ? for either day-of-month or day-of-week
→ You cannot specify BOTH day-of-month and day-of-week
→ One must be ? (no value)
→ cron(0 2 15 * ? *) = 15th of every month at 2 AM ✅
→ cron(0 2 ? * MON *) = Every Monday at 2 AM ✅
→ cron(0 2 15 * MON *) = ❌ Invalid — cannot specify both

CRITICAL: All schedule times are UTC
→ If ShopNow is in India (IST = UTC+5:30)
→ 2:00 AM IST = 8:30 PM previous day UTC
→ Cron expression for 2 AM IST: cron(30 20 * * ? *)

STEP 7 — CONFIGURE EVENT ARCHIVE AND REPLAY

→ One of EventBridge's most powerful features — impossible with S3 notifications or CloudWatch alarms

Why Archive Matters:
→ A Glue job failed 3 days ago → You fixed the bug → Now you want to reprocess the SAME S3 events that triggered the failure
→ Without archive → Those events are gone — you'd have to manually re-upload files or create fake events
→ With archive → Replay the exact events from 3 days ago → pipeline reprocesses naturally

Create an Archive:

→ Navigate to EventBridge → "Archives" → "Create archive"

SettingValue
Archive nameshopnow-default-bus-archive
Source event busdefault
DescriptionArchive all events from default bus for replay capability
Retention period90 days (configurable: 0 = indefinite)
Event pattern filter(Leave blank to archive ALL events) OR filter specific patterns

Filtering Archive (Optional — to reduce storage cost):

StrategyArchive PatternUse Case
Archive everythingNo filterFull replay capability — higher storage cost
Archive S3 events onlyFilter: source = aws.s3Only need to replay data pipeline triggers
Archive failures onlyFilter: detail.state = FAILEDOnly need to replay failed workflows

→ For ShopNow → Archive everything with 90-day retention
→ Storage cost is minimal — events are small JSON documents

How to Replay Events:

→ Navigate to EventBridge → "Replays" → "Start new replay"

SettingValue
Replay namereplay-glue-failures-jan-15
Source archiveshopnow-default-bus-archive
Start time2025-01-15T00:00:00Z
End time2025-01-15T23:59:59Z
Destination event busdefault (same bus — rules will re-trigger)
Event pattern filterFilter to only replay specific events (e.g., S3 orders uploads)

What Happens During Replay:
→ EventBridge reads events from the archive for the specified time window
→ Events are re-published to the destination event bus
→ Rules on that bus match and trigger targets as if the events just happened
→ Replayed events have a replay-name field added — so targets can distinguish replay from live

Replay Considerations:

ConsiderationDetail
IdempotencyTargets MUST handle duplicate processing — replay sends the same event again
OrderEvents are replayed approximately in order but NOT guaranteed exactly in order
RateReplay rate is throttled — not instantaneous for large time windows
Live eventsLive events continue processing normally during replay — they coexist

STEP 8 — CREATE A CUSTOM EVENT BUS FOR APPLICATION EVENTS

→ AWS service events go to default bus automatically
→ But ShopNow's application also generates events:

  • "Order placed" — from the order processing microservice
  • "Payment confirmed" — from the payment gateway integration
  • "Inventory threshold breached" — from the inventory monitoring service

→ These are CUSTOM application events — they should go to a SEPARATE bus

Why separate bus?
→ Isolation: Application events don't pollute the AWS service event stream
→ Access control: Different IAM policies for who can publish to each bus
→ Cost tracking: Custom bus events are billed separately
→ Cross-account: Custom buses can accept events from other AWS accounts (Project 41 preview)

✅ WHAT SHOULD DO

→ Always enable S3 EventBridge notifications on buckets that need content-based routing
→ Always use the default event bus for AWS service events — it's free and automatic
→ Always use custom event buses for application-generated events — isolation and access control
→ Always use input transformations to simplify payloads for targets
→ Always enable event archiving for critical event flows — replay is invaluable for debugging
→ Always use descriptive rule names following a convention: {company}-{source}-{event}-{action}
→ Always set up rules for FAILURE events (Glue failed, rotation failed) — not just success
→ Always use UTC in cron expressions — convert from local timezone carefully
→ Always create separate rules for different routing paths — don't overload one rule with complex logic
→ Always attach tags to rules and custom buses for cost tracking and organization
→ Always configure dead-letter queues on rules for failed target invocations (see DLQ section below)

❌ WHAT SHOULD NOT DO

→ Never rely on CloudTrail-based events for time-sensitive workflows — 5-15 minute delay
→ Never put application events on the default bus — use custom buses for separation
→ Never create overly broad rules (e.g., match ALL S3 events) without specific need — noisy targets
→ Never forget that scheduled rules use UTC — IST 2:00 AM ≠ UTC 2:00 AM
→ Never use ? incorrectly in cron — one of day-of-month or day-of-week MUST be ?
→ Never assume events are delivered in strict order — EventBridge is near real-time but not ordered
→ Never use EventBridge for high-throughput streaming (>millions of events/second) — use Kinesis (Project 52)
→ Never skip DLQ configuration on rules — failed target invocations are silently lost without DLQ
→ Never hardcode event patterns — use variables and templates for maintainability
→ Never create duplicate rules that route the same event to the same target — causes double processing
→ Never forget that EventBridge rules have a 5-target limit per rule — use multiple rules if needed
→ Never skip input transformation when targets only need a subset of the event — reduces payload size and processing complexity

MECHANIC 2 — EVENT PATTERN MATCHING ENGINE

How EventBridge evaluates patterns:
→ Pattern matching is PREFIX-based comparison at each level of the JSON tree
→ Only fields SPECIFIED in the pattern are checked — unspecified fields are ignored
→ This means: An empty pattern {} matches ALL events

Evaluation logic:

For each field in the event pattern:
    Find the corresponding field in the event
    If field doesn't exist in event → NO MATCH
    If field exists → compare value:
        String: exact match or prefix/suffix/anything-but
        Number: numeric comparison (>, <, >=, <=, =, range)
        Array: event value must match AT LEAST ONE pattern value
    All fields must match → OVERALL MATCH

Important matching behaviors:

BehaviorExampleResult
Array = OR logicPattern: "state": ["FAILED", "ERROR"] → Event: "state": "FAILED"✅ Match
Missing fieldPattern has "detail.severity" but event has no severity field❌ No Match
Extra fieldsEvent has detail.extraField not in pattern✅ Match (ignored)
Nested matchingPattern: "detail": {"bucket": {"name": ["my-bucket"]}}Matches nested structure
Null matchingPattern: "detail.value": [null]Matches events where value IS null
Exists matchingPattern: "detail.value": [{"exists": true}]Matches events where field exists (any value)

📈 SCALABILITY CONSIDERATIONS

→ Event Throughput Scaling

  • Default bus: Effectively unlimited for AWS service events
  • Custom bus: Default 10,000+ events/second (can request increase)
  • For > 100,000 events/second → consider Kinesis Data Streams (Project 52) as event source with EventBridge as routing layer
  • EventBridge is NOT designed for millions of events/second — that's Kinesis territory

→ Rule Count Scaling

  • Default: 300 rules per bus (soft limit)
  • Can increase to 2,000+ via support request
  • If you need > 2,000 rules → reconsider architecture
  • Consolidate rules with broader patterns → use target Lambda to sub-route

→ Target Scaling

  • 5 targets per rule — use multiple rules for more targets
  • Each target invocation is independent — one slow target doesn't block others
  • Lambda targets scale automatically (concurrency)
  • SQS targets scale automatically (unlimited writes)
  • SNS targets scale automatically
  • Step Functions: Default 1,300 state machine starts/second (can increase)

→ Archive Scaling

  • Archive storage scales automatically (S3-backed internally)
  • No limit on archive size — limited only by retention period and storage cost
  • Replay throughput is throttled — not suitable for replaying millions of events quickly

→ Cross-Account Scaling (Preview for Project 41)

  • Custom bus can receive events from up to 100 AWS accounts (configurable via resource policy)
  • Default bus can send events to other accounts via cross-account rules
  • Organizational event bus pattern: Central account collects events from all accounts

→ Multi-Region Scaling

  • EventBridge is regional — rules and buses exist in one region
  • For cross-region: Use EventBridge Global Endpoints (active-active)
  • Or configure rules that target another region's event bus

🎯 KEY TAKEAWAY

→ EventBridge is the CENTRAL NERVOUS SYSTEM of AWS event-driven architecture
→ It replaces fragmented notification mechanisms (S3 Event Notifications, CloudWatch Alarms → SNS, manual polling) with ONE unified routing system
→ The three pillars are: Event Bus (where events flow), Rules (what to match), Targets (where to send)
→ Content-based routing via event patterns enables INTELLIGENT routing — different file types → different pipelines from the same S3 bucket
→ Scheduled rules replace unreliable EC2 cron jobs and fragmented CloudWatch Events with centralized, managed schedules
→ Event archiving + replay provides debugging and reprocessing capability that NO other notification mechanism offers
→ Input transformation simplifies target payloads — targets receive ONLY what they need
→ DLQ configuration on rules is ESSENTIAL — without it, failed target invocations are silently lost
→ Custom event buses isolate application events from AWS service events — separate access control and billing
→ EventBridge is at-least-once delivery — targets MUST be idempotent
→ EventBridge is NOT a message queue (SQS), NOT a streaming platform (Kinesis), NOT an orchestrator (Step Functions) — it is a ROUTER that connects sources to the right targets
