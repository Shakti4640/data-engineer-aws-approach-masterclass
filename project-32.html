PROJECT 32: THE PIPELINE THAT NEEDED HUMAN APPROVAL — Step Functions Multi-Service Orchestration

🔴 HYPOTHETICAL PROBLEM

→ ShopNow's Glue Workflow (Project 31) runs perfectly every morning at 6 AM
→ But the Head of Data keeps getting escalations:

Escalation 1 — No Post-ETL Actions:
→ After the Glue Workflow completes, someone must manually run a Redshift COPY command (Project 22 pattern) to load enriched data into the warehouse
→ The analyst who does this forgets every other day
→ Dashboards show stale data → CEO frustrated

Escalation 2 — No Human Approval Gate:
→ One morning, the enrichment Job produced garbage data (MariaDB source had corrupt records)
→ The data flowed straight into Redshift → wrong numbers in board presentation
→ The Data Quality team says: "We MUST review a sample before data goes to Redshift"
→ They want a manual approval step — pipeline pauses, human reviews, approves or rejects

Escalation 3 — No Failure Notification:
→ The Glue Workflow fails silently (as noted in Project 31 — no native alerting)
→ Nobody knows until analysts complain about missing data at 10 AM
→ The team needs SNS notification on any step failure

Escalation 4 — No Retry Logic:
→ The MariaDB enrichment Job sometimes fails because the database connection times out during peak hours
→ If retried 5 minutes later, it usually works
→ Currently — failure means the entire pipeline stops and requires manual intervention
→ They need automatic retry with increasing wait times (exponential backoff)

Escalation 5 — No Dynamic Branching:
→ On month-end days, data volume is 10x normal
→ The standard Glue Job config (5 DPUs) takes 3 hours on month-end vs 20 minutes normally
→ They want: "If it's the last day of the month → trigger the large-capacity Job; otherwise → trigger the standard Job"
→ Glue Workflow's DAG is static — no runtime decisions possible

Escalation 6 — Cross-Service Coordination:
→ The pipeline now spans: Glue Crawlers → Glue Jobs → Redshift → SNS → Human Approval
→ Glue Workflow can only orchestrate Glue components
→ They need an orchestrator that coordinates ANY AWS service

→ The CTO says: "We need a state machine — something that manages the entire flow with conditions, retries, approvals, and notifications across all services"

→ This is EXACTLY what AWS Step Functions solves

🟡 HOW TO APPROACH IT

→ First — understand the fundamental difference between Glue Workflow and Step Functions

DimensionGlue Workflow (Project 31)Step Functions
ScopeGlue Crawlers + Glue Jobs ONLYANY AWS service + HTTP endpoints
BranchingStatic DAG — no runtime decisionsDynamic — Choice state with runtime conditions
RetryDelegated to individual Job retry settingsBuilt-in per-state Retry with backoff configuration
Error HandlingPipeline stops on failure — no catch mechanismCatch blocks per state — route errors to fallback states
Human ApprovalNot supportedWait for Callback pattern — pauses until external signal
NotificationsNot built-inNative SNS/SQS/Lambda integration as states
Parallel ExecutionSupported via fan-out triggersNative Parallel state with branch-level error handling
LoopsNot supportedSupported via Choice + state transitions
Execution ModelEvent-driven pollingState machine with explicit transitions
CostFree (pay for Jobs/Crawlers)Pay per state transition
Definition LanguageConsole visual builderAmazon States Language (ASL) — JSON-based

→ Step Functions uses a STATE MACHINE model
→ Every step in the pipeline is a "State"
→ States have types — each type does something different

Core State Types:

State TypeWhat It DoesShopNow Use Case
TaskExecutes an action (calls an AWS service)Start Glue Job, Run Redshift query, Send SNS message
ChoiceMakes a runtime decision based on input data"Is today month-end?" → branch to large or small Job
WaitPauses execution for a duration or until a timestampWait 5 minutes before retry
ParallelExecutes multiple branches simultaneouslyRun two independent crawlers at the same time
MapIterates over a list and processes each itemProcess each file in a batch
PassPasses input to output (optionally transforms data)Inject static values or restructure JSON
SucceedMarks execution as successful and stopsTerminal state for happy path
FailMarks execution as failed with error/causeTerminal state for error path

→ Step Functions comes in TWO workflow types:

TypeExecution DurationExecution RateCostUse Case
StandardUp to 1 year2,000 per second$0.025 per 1,000 transitionsLong-running ETL pipelines, human approval
ExpressUp to 5 minutes100,000 per secondPer execution + durationHigh-volume event processing

→ ShopNow needs Standard — their pipeline runs 30+ minutes and includes human approval (which could take hours)

Understanding Integration Patterns — CRITICAL CONCEPT:

PatternBehaviorSyntax SuffixUse Case
Request ResponseCalls the API and immediately moves to next state — does NOT wait for completionNone (default)Fire-and-forget actions like SNS Publish
Run a Job (.sync)Calls the API and WAITS until the job/crawler completes before moving to next state.syncGlue Jobs, Glue Crawlers, ECS Tasks — anything long-running
Wait for Callback (.waitForTaskToken)Pauses execution, sends a task token, waits until external system sends success/failure signal.waitForTaskTokenHuman approval, external system callbacks

→ For Crawler → Use .sync pattern
→ Step Functions calls StartCrawler → then automatically polls GetCrawler until status = READY
→ Only then does it transition to the next state

Understanding Choice State Rules:

→ Choice state evaluates CONDITIONS against the state input
→ Each condition has a comparison operator and a "Next" state

How the input gets the date information:
→ The state machine input (provided when execution starts) includes:

Input to state machine:
{
  "processing_date": "2025-01-31",
  "is_month_end": true,
  "source_path": "s3://shopnow-data-lake/raw/orders/"
}

→ This input is passed by EventBridge when triggering the execution (configured in Step 13)
→ OR can be enriched by a Lambda Task state that determines if today is month-end

Choice Rules Configuration:

Rule #VariableComparisonValueNext State
Rule 1$.is_month_endBooleanEqualstrueRun-Large-Enrichment-Job
Default(no condition — fallback)——Run-Standard-Enrichment-Job

Understanding Choice Operators:

OperatorUse CaseExample
StringEqualsExact string match$.status equals "approved"
NumericGreaterThanNumeric comparison$.record_count > 1000000
BooleanEqualsBoolean check$.is_month_end equals true
TimestampLessThanTime comparison$.deadline < "2025-02-01T00:00:00Z"
IsPresentCheck if field exists$.optional_field is present
StringMatchesWildcard matching$.filename matches "*.csv"
And/Or/NotCompound conditions$.count > 100 AND $.type equals "premium"

→ Choice state has NO Retry or Catch — it's a pure routing decision
→ If no rule matches AND no Default → execution fails with States.NoChoiceMatched error

SettingValue

Topic ARNarn:aws:sns:REGION:ACCOUNT:shopnow-pipeline-approval-request

Message Configuration:
→ The message sent to SNS must include the Task Token
→ The DQ team uses this token to signal approval or rejection

Message structure:

{
  "pipeline": "shopnow-daily-orders",
  "execution_id": provided by context object,
  "processing_date": from $.processing_date,
  "task_token": from $$.Task.Token,
  "message": "Please review enriched data and approve or reject",
  "approve_instructions": "Call SendTaskSuccess API with this task token",
  "reject_instructions": "Call SendTaskFailure API with this task token"
}

→ $$.Task.Token — this is a SPECIAL context variable
→ Step Functions automatically generates a unique token for each .waitForTaskToken invocation
→ The $$ prefix means "context object" (metadata about the execution), not the state input ($)

Timeout Configuration:

SettingValueReasoning
Timeout Seconds28800 (8 hours)If nobody approves within 8 hours → timeout → catch → failure notification
Heartbeat Seconds3600 (1 hour)Optional — if no heartbeat received in 1 hour, assume stuck

→ Without timeout → execution could wait FOREVER → costs accumulate → state machine stuck

How the DQ Team Responds:

→ Option 1 — API Call (recommended for automation):

  • Approve: Call SendTaskSuccess with taskToken and output {"decision": "approved"}
  • Reject: Call SendTaskFailure with taskToken and error "DataQualityRejection"

→ Option 2 — Build a simple approval UI:

  • API Gateway + Lambda that receives approve/reject click
  • Lambda calls SendTaskSuccess or SendTaskFailure
  • This is the production pattern — but requires API Gateway (covered in Project 59)

→ Option 3 — CLI (for testing):

  • AWS CLI: aws stepfunctions send-task-success --task-token "TOKEN" --task-output '{"decision":"approved"}'

After callback received:
→ The state resumes
→ Output includes whatever was passed in SendTaskSuccess → {"decision": "approved"}
→ Next state can use this output for further routing

✅ WHAT SHOULD DO

→ Always use .sync integration pattern for long-running tasks (Glue, ECS, Redshift)

  • Without .sync → Step Functions fires the task and immediately moves on → you lose sequencing

→ Always add Retry with exponential backoff on EVERY Task state

  • Transient errors (network timeouts, throttling) often resolve on retry
  • Pattern: specific errors first → States.ALL last

→ Always add Catch on EVERY Task state pointing to a failure notification state

  • Uncaught errors terminate the execution silently

→ Always set execution timeout on the state machine level

  • Prevents runaway executions from accumulating cost

→ Always set timeout on .waitForTaskToken states

  • Without timeout → execution waits forever → stuck indefinitely

→ Always enable CloudWatch Logs with level ALL during development

  • Switch to ERROR level in production to reduce log volume and cost

→ Always enable X-Ray tracing for debugging latency issues across states

→ Always use ResultPath and OutputPath to control data flow between states

  • ResultPath: $.result → appends task output to input under "result" key
  • OutputPath: $.result → passes only the "result" key to next state
  • Without these → output replaces input → you lose context

→ Always version-control your ASL definition in Git

  • The console visual builder is great for design — but production deployments should use Infrastructure as Code

→ Always use least-privilege IAM role — scope permissions to specific Glue Jobs, SNS Topics, Redshift clusters

❌ WHAT SHOULD NOT DO

→ Never use Express workflow for long-running ETL — 5-minute limit will kill your execution
→ Never forget to activate the EventBridge rule after creation — inactive rule = no triggers
→ Never pass sensitive data (credentials, tokens) through state machine input/output

  • State data is visible in execution history and CloudWatch Logs
  • Use Secrets Manager (Project 37) for credentials

→ Never create deeply nested state machines (30+ sequential states)

  • Break into child state machines using StepFunctions:StartExecution (.sync) — nested execution pattern

→ Never use ResultPath: null unless you intentionally want to discard task output

  • Common mistake: task succeeds but output is thrown away → next state has no data

→ Never rely on execution history alone for auditing — history is purged after 90 days

  • Always enable CloudWatch Logs for permanent retention

→ Never hardcode ARNs in the ASL definition

  • Use CloudFormation/CDK parameters or SSM Parameter Store references

→ Never use Choice state without a Default branch

  • If no rule matches and no Default → States.NoChoiceMatched error → execution fails

→ Never put business logic inside the state machine definition

  • State machines should ORCHESTRATE — not compute
  • Business logic belongs in Lambda functions or Glue Jobs

→ Never ignore the state machine execution cost

  • Every state transition costs money
  • A state machine with 100 states running 1000 times/day = significant cost

🔴 WHEN NOT TO USE STEP FUNCTIONS

→ When you have a simple event → action pattern with no orchestration

  • S3 upload → Lambda → done? → Just use S3 Event Notification + Lambda (Project 15, 26)

→ When you need sub-second execution with thousands of concurrent workflows

  • Standard workflows have 2,000/sec start rate → use Express for high volume

→ When your pipeline is exclusively Glue components with no external services

  • Glue Workflow (Project 31) is free and simpler for Glue-only chains

→ When you need a full-featured workflow scheduler with backfill capabilities

  • Step Functions doesn't natively support backfill
  • Consider Amazon MWAA (Managed Apache Airflow) for complex scheduling needs

→ When your workflow exceeds 25,000 execution history events

  • Standard workflow limit per execution
  • For extremely long workflows → use child executions (nested state machines)

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

MECHANIC 1 — THE STATE MACHINE EXECUTION MODEL

→ When StartExecution is called:

  1. Step Functions creates an Execution object with unique Execution ARN
  2. The ASL definition is parsed and the start state is identified
  3. The input JSON is passed to the start state
  4. Each state executes → produces output → the engine evaluates "Next" field
  5. Output of current state becomes input of next state (unless modified by ResultPath/OutputPath)
  6. Execution continues until it reaches a Succeed or Fail terminal state
  7. All state transitions are durably recorded in execution history

→ KEY PROPERTY: Step Functions is DURABLE

  • Every state transition is checkpointed
  • If the Step Functions infrastructure itself has a failure → it resumes from the last checkpoint
  • This is why it can run for up to 1 year — it doesn't keep a persistent process running
  • It's event-driven internally — "wake up when this thing completes"

MECHANIC 2 — THE .sync INTEGRATION PATTERN (Deep Dive)

→ When a Task state uses .sync (e.g., Glue StartJobRun.sync):

Step Functions                     AWS Glue
     │                                │
     ├─ Call StartJobRun API ────────►│ Returns JobRunId
     │                                │
     ├─ Store JobRunId internally     │ Job starts running
     │                                │
     ├─ Poll GetJobRun API ──────────►│ Returns RUNNING
     │  (periodic polling)            │
     ├─ Poll GetJobRun API ──────────►│ Returns RUNNING
     │                                │
     ├─ Poll GetJobRun API ──────────►│ Returns SUCCEEDED
     │                                │
     ├─ State transitions to Next     │
     ▼                                │
  Next State                          │

→ The polling is handled by Step Functions internally — you don't configure polling intervals
→ Step Functions uses an optimized polling strategy (not fixed interval — adapts based on expected duration)
→ You are NOT charged for polling API calls — they're part of the Step Functions service
→ But you ARE charged for the state transition when the task completes

MECHANIC 3 — THE .waitForTaskToken PATTERN (Deep Dive)

→ When a Task state uses .waitForTaskToken:

Step Functions                     External World
     │                                │
     ├─ Generate unique Task Token    │
     │  (UUID-like string)            │
     ├─ Execute the API (e.g.,        │
     │  SNS:Publish) with token       │
     │  embedded in the payload ─────►│ Human receives email with token
     │                                │
     ├─ PAUSE ◄─── execution stops    │
     │  here — no polling, no cost    │ Human reviews data
     │  for waiting (Standard only)   │
     │                                │
     │                                │ Human calls SendTaskSuccess
     │◄──── (token, output) ──────────┤ with the task token
     │                                │
     ├─ Validate token matches        │
     ├─ Resume execution with         │
     │  provided output               │
     ▼                                │
  Next State                          │

→ The Task Token is generated by Step Functions — guaranteed unique
→ The token is available via $$.Task.Token in the state definition
→ Whoever has the token can call SendTaskSuccess or SendTaskFailure
→ Security implication: Treat task tokens as sensitive — anyone with the token can resume the execution
→ Standard workflows: No charge during wait time (event-driven, not polling)
→ Express workflows: CANNOT use .waitForTaskToken — not supported

MECHANIC 4 — DATA FLOW BETWEEN STATES

→ This is the most misunderstood mechanic — and the #1 source of debugging pain

Core Concept: Every state receives INPUT and produces OUTPUT

→ By default: The OUTPUT of state N becomes the INPUT of state N+1
→ This can be controlled with 4 filters:

FilterApplied WhenWhat It Does
InputPathBefore task executionSelects a portion of the state input to pass to the task
ParametersBefore task executionConstructs a new JSON object to pass to the task (can mix static values + input references)
ResultPathAfter task executionDetermines WHERE the task result is placed within the original input
OutputPathAfter ResultPathSelects a portion of the combined result to pass to the next state

Execution Order:

State Input
    │
    ▼ InputPath (filter input)
    │
    ▼ Parameters (construct task input)
    │
    ▼ [Task Executes]
    │
    ▼ ResultSelector (filter task output)
    │
    ▼ ResultPath (merge result into original input)
    │
    ▼ OutputPath (filter final output)
    │
    ▼ Next State Input

Common Patterns:

GoalConfiguration
Pass entire input through, append task resultResultPath: $.taskResult
Discard task result, keep original inputResultPath: null
Pass only the task result to next stateResultPath: $.taskResult, OutputPath: $.taskResult
Replace entire input with task resultDefault (no configuration needed)

→ ShopNow Example:

  • Determine-Date-Context Lambda returns {"processing_date": "2025-01-31", "is_month_end": true}
  • This becomes the input for ALL subsequent states (flows through)
  • Crawl-Raw-CSV-Data completes → its output (crawler details) is irrelevant to the next state
  • Set ResultPath: $.crawler1_result → appends crawler output but KEEPS the original date context
  • Next state still has $.processing_date and $.is_month_end available

MECHANIC 5 — RETRY INTERNAL BEHAVIOR

→ When Retry is configured on a state:

Task executes → FAILS with error "Glue.JobRunException"
     │
     ▼
Retry evaluator checks retry rules IN ORDER:
     │
     ├─ Rule 1: Error matches "Glue.JobRunException"? → YES
     │   ├─ Attempts remaining? (MaxAttempts - current attempt)
     │   │   ├─ YES → Wait (IntervalSeconds × BackoffRate^attempt) → Re-execute task
     │   │   └─ NO → Move to Catch evaluator
     │
     ├─ Rule 2: (next rule if Rule 1 didn't match)
     │
     └─ No rules matched → Move to Catch evaluator

Backoff Calculation Example:

AttemptIntervalSecondsBackoffRateActual Wait
1st retry3002.0300 seconds (5 min)
2nd retry3002.0600 seconds (10 min)
3rd retry3002.01200 seconds (20 min)

→ Formula: wait = IntervalSeconds × (BackoffRate ^ retryAttemptNumber)
→ Optional: MaxDelaySeconds caps the maximum wait time

→ Important: Each retry is a FULL re-execution of the task
→ The task receives the SAME input as the original attempt
→ Retries are NOT visible as separate state transitions in the visual graph
→ They appear in the detailed execution event history

MECHANIC 6 — CATCH INTERNAL BEHAVIOR

→ When all retries are exhausted OR no retry rule matches:

Catch evaluator checks catch rules IN ORDER:
     │
     ├─ Rule 1: Error matches "States.ALL"? → YES
     │   ├─ Store error info at ResultPath (e.g., $.error)
     │   ├─ Transition to "Next" state (e.g., Notify-Pipeline-Failure)
     │   └─ DONE
     │
     └─ No catch rules matched → Execution FAILS with unhandled error

Error Information Structure (stored at ResultPath):

{
  "Error": "Glue.JobRunException",
  "Cause": "Job run failed with status FAILED: OutOfMemoryError in executor"
}

→ The Catch "Next" state receives the original input + error information
→ The failure notification state can include $.error.Error and $.error.Cause in the SNS message

📈 SCALABILITY CONSIDERATIONS

→ Concurrent Executions

  • Standard: Up to 1,000,000 open executions per account per region
  • Express: Up to 100,000 execution starts per second
  • For ShopNow's daily pipeline → 1 execution/day → nowhere near limits

→ State Machine Complexity

  • No hard limit on number of states
  • ASL definition size limit: 1 MB
  • Execution history limit: 25,000 events per execution
  • For very complex pipelines → use nested state machines (child executions)

→ Cost Scaling

Pipeline ComplexityStatesDaily RunsMonthly TransitionsMonthly Cost
ShopNow (this project)~12 states1~360< $0.01
Medium pipeline30 states24 (hourly)~21,600~$0.54
Large pipeline50 states100 (event-driven)~150,000~$3.75
High-volume Express10 states1,000,000/dayN/A (per execution pricing)Varies significantly

→ Standard workflow cost is negligible for most ETL pipelines
→ Cost becomes significant for high-volume event-driven patterns → use Express

→ Payload Size Scaling

  • Maximum payload between states: 256 KB
  • If you need to pass large data → store in S3, pass the S3 path as reference
  • Never pass actual data content through state machine — only references

→ Timeout Scaling

  • Standard: Up to 1 year execution duration
  • Plan timeout budgets: Sum of all possible wait times + task durations + human approval wait
  • Set execution-level timeout as safety net

→ Regional Scaling

  • State machines are regional
  • For multi-region patterns → use EventBridge cross-region to trigger state machines in other regions
  • Cross-region state machine orchestration explored in Project 55 (DR patterns)
