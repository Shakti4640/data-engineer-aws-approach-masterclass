The Head of Data's requirement:

→ This is an EVENT-DRIVEN pipeline:

  • Event: File lands in S3
  • Trigger: S3 Event Notification

Why VPC permissions are needed — Deep Explanation:

→ Redshift cluster sits in a PRIVATE subnet (Project 16 configuration)
→ Lambda by default runs OUTSIDE any VPC — in AWS-managed network space
→ Lambda outside VPC CANNOT reach resources inside your VPC (Redshift, RDS)
→ To access Redshift → Lambda must be configured to run INSIDE the VPC
→ When Lambda runs in VPC → AWS creates an Elastic Network Interface (ENI) in your subnet
→ This ENI gives Lambda a private IP address within your VPC
→ With that private IP → Lambda can reach Redshift via Security Group rules

→ But there's a CATCH (explored fully in Project 45):
→ Lambda in VPC LOSES direct internet access
→ If Lambda needs to call S3, SNS, Secrets Manager (all public endpoints) → it needs either:

  • NAT Gateway in the VPC (from Project 5) → routes to internet → reaches public services
  • VPC Endpoints for S3, SNS, Secrets Manager (from Project 27) → private path, no internet needed

→ For this project → we need BOTH:

  • VPC access for Redshift (private subnet connectivity)
  • Internet/endpoint access for S3, SNS, Secrets Manager

Lambda's Processing Logic:

STEP 1: EXTRACT VENDOR NAME FROM S3 KEY
    │
    Input: "vendor-feeds/vendorA/products-2025-01-15.csv"
    Extract: "vendorA"
    │
    ▼
STEP 2: VALIDATE FILE
    │
    ├─ Check file extension (.csv expected)
    ├─ Check file size (> 0 bytes, < 500 MB)
    ├─ Download first 1000 rows to /tmp
    ├─ Check column count matches expected schema
    ├─ Check required columns exist
    ├─ Check for NULL values in required fields
    │
    ├─ VALID → Continue to Step 3
    └─ INVALID → Move file to quarantine/ → Send SNS failure alert → EXIT
    │
    ▼
STEP 3: DETERMINE TARGET REDSHIFT TABLE
    │
    Mapping (from environment variable or configuration table):
    │
    vendorA → schema: vendor_data, table: vendor_a_products
    vendorB → schema: vendor_data, table: vendor_b_products  
    vendorC → schema: vendor_data, table: vendor_c_catalog
    ...
    │
    ▼
STEP 4: EXECUTE REDSHIFT COPY VIA DATA API
    │
    Call: redshift-data.execute_statement()
    Parameters:
        ClusterIdentifier: from env var
        Database: from env var
        SecretArn: from env var (Secrets Manager — Project 37)
        Sql: "COPY vendor_data.vendor_a_products 
              FROM 's3://shopnow-data-lake/vendor-feeds/vendorA/products-2025-01-15.csv' 
              IAM_ROLE 'arn:aws:iam::ACCOUNT:role/RedshiftS3ReadRole'

              REGION 'ap-south-1'

              DATEFORMAT 'auto';"
    │
    Returns: Statement ID (async — COPY starts but hasn't completed)
    │
    ▼

    │

        Call: redshift-data.describe_statement(Id=statement_id)

            STARTED → Wait 10 seconds → Poll again

    │
    ▼

    │

STEP 9 — CONFIGURE LAMBDA CONCURRENCY CONTROLS

→ What happens when 5 vendors upload files simultaneously at 6 AM?

Default behavior:
→ S3 sends 5 separate events → Lambda creates 5 concurrent executions
→ Each execution independently processes its vendor file
→ 5 simultaneous Redshift COPY commands execute

Problem:
→ Redshift has connection limits (based on cluster size)
→ 5 concurrent COPY commands + regular analyst queries → might exceed Redshift's WLM queue slots
→ Could cause "Too many connections" or queue timeout errors

Solution: Lambda Reserved Concurrency

→ Lambda Console → shopnow-vendor-feed-processor → Configuration → Concurrency

SettingValueMeaning
Reserved Concurrency3Maximum 3 simultaneous executions of this function

→ What happens when 5 events arrive but concurrency is 3:

  1. Events 1, 2, 3 → Lambda invocations start immediately
  2. Events 4, 5 → Placed in Lambda's internal retry queue
  3. When execution 1 finishes → Event 4 starts
  4. When execution 2 finishes → Event 5 starts

Understanding Lambda Concurrency Types:

TypeBehaviorUse Case
UnreservedUses shared account concurrency pool (default 1000)General functions without specific limits
ReservedGuarantees N concurrent executions AND limits to N✅ Protect downstream services (Redshift) from overload
ProvisionedPre-initializes N execution environments (warm)Latency-sensitive functions — eliminates cold starts

→ Reserved concurrency of 3 means:

  • This function can NEVER exceed 3 simultaneous executions
  • This function ALWAYS has 3 slots reserved (other functions can't consume them)
  • Excess events are queued (async invocation from S3) and retried

S3 → Lambda Async Invocation Retry Behavior:

→ S3 event notifications invoke Lambda ASYNCHRONOUSLY
→ If Lambda can't process (throttled, error), the async invocation model handles retries:

Retry BehaviorConfiguration
Maximum retries2 (total 3 attempts: original + 2 retries)
Maximum event age6 hours (event discarded if not processed within 6 hours)
On failure destinationCan send failed events to SQS DLQ or SNS

✅ WHAT SHOULD DO

→ Always validate vendor files BEFORE loading into Redshift — garbage in = garbage out
→ Always move processed files to a separate prefix — prevents reprocessing and enables audit trail
→ Always move invalid files to quarantine — enables investigation without blocking pipeline
→ Always use Redshift Data API for Lambda-to-Redshift communication — avoids VPC complexity
→ Always use Secrets Manager for Redshift credentials — never hardcode in Lambda environment variables
→ Always set Lambda reserved concurrency to protect Redshift from connection overload
→ Always configure S3 event prefix/suffix filters to prevent infinite trigger loops
→ Always set up a Dead Letter Queue for failed Lambda invocations — no event should be silently lost
→ Always include MAXERROR in COPY commands — allows partial loads while tracking bad rows
→ Always use COMPUPDATE OFF and STATUPDATE OFF for frequent small loads — run maintenance separately
→ Always add CloudWatch Alarms on Lambda Errors, Throttles, and Duration
→ Always log the full processing details (vendor, file, rows loaded, duration, errors) for audit
→ Always use S3 Lifecycle policies for quarantine cleanup and processed file archival

❌ WHAT SHOULD NOT DO

→ Never trigger Lambda on s3:ObjectCreated:* without prefix/suffix filters — catches ALL uploads including processed/quarantine moves → infinite loops
→ Never put Lambda in VPC when only using Redshift Data API — unnecessary complexity and cold start penalty
→ Never set Lambda timeout shorter than expected COPY duration — timeout kills Lambda → COPY continues in Redshift → Lambda reports failure but data loads → inconsistent state
→ Never hardcode vendor-to-table mappings in Lambda code — use environment variables, SSM Parameter Store, or DynamoDB lookup table
→ Never set reserved concurrency to 0 — this DISABLES the function entirely (0 means "no executions allowed")
→ Never skip the DLQ configuration — without DLQ, failed events are lost after retries
→ Never use MAXERROR with a very high value without monitoring STL_LOAD_ERRORS — bad data silently enters warehouse
→ Never run ANALYZE/VACUUM in the same Lambda as the COPY — ANALYZE is expensive and should be batched
→ Never ignore the S3 event notification limit: maximum 1 event notification configuration per prefix+suffix+event type combination per bucket
→ Never store Lambda execution results only in CloudWatch Logs — also persist to DynamoDB or S3 for structured querying of processing history
→ Never assume vendor files arrive in order or exactly once — vendors might re-upload, upload late, or upload partial files
→ Never process files with .tmp or .partial suffix — vendors might upload in stages; filter on .csv only

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

MECHANIC 1 — S3 EVENT NOTIFICATION DELIVERY

→ When a file is uploaded to S3:

1. S3 receives PutObject API call
2. Object is stored durably (replicated across 3 AZs — Project 1 mechanic)
3. S3 returns HTTP 200 to uploader
4. S3 Event Notification system detects the event
5. Event matches notification configuration (prefix + suffix + event type)
6. S3 constructs event payload (JSON with bucket, key, size, timestamp)
7. S3 invokes Lambda ASYNCHRONOUSLY

Key Timing:
→ Event delivery is NOT instant — typically 1-10 seconds after upload completes
→ In rare cases → up to 60 seconds
→ S3 provides AT-LEAST-ONCE delivery — the same event MAY be delivered more than once
→ Lambda must be IDEMPOTENT — processing the same file twice should produce the same result

What "at-least-once" means practically:

ScenarioWhat HappensHow to Handle
Normal1 event → 1 Lambda invocationProcess normally
Duplicate delivery1 upload → 2 Lambda invocationsLambda must check if file already processed (check processed/ prefix)
Delayed deliveryUpload at 2:00 AM → event at 2:00:45 AMUsually not a problem — 45 seconds is acceptable

→ Idempotency check in Lambda:

  • Before processing → check if file already exists in processed/ prefix
  • If yes → skip processing → log "already processed" → exit
  • If no → proceed with processing

MECHANIC 2 — LAMBDA ASYNC INVOCATION MODEL

→ S3 invokes Lambda asynchronously — this has specific implications:

S3 → Lambda Async Invocation Queue → Lambda Execution
         │
         ├─ If Lambda available → Execute immediately
         ├─ If Lambda throttled → Queue event → Retry
         └─ If Lambda fails → Retry (up to 2 times)
              │
              └─ After all retries exhausted → Send to DLQ

Internal Queue Behavior:

AspectBehavior
Queue typeInternal AWS-managed queue (not visible to you)
Queue retentionUp to 6 hours (configurable via Maximum Event Age)
OrderingNOT guaranteed — events may be processed out of order
DeduplicationNOT guaranteed — same event may be delivered multiple times
Throttle behaviorEvents queue up and retry automatically
Error behaviorFailed invocation retried up to MaxRetries times
Final failureEvent sent to configured DLQ (SQS or SNS)

→ Contrast with SYNCHRONOUS invocation (e.g., API Gateway → Lambda):

  • Synchronous: Caller waits for response → caller handles retries
  • Asynchronous: S3 fires and forgets → Lambda service handles retries

MECHANIC 4 — REDSHIFT COPY INTERNAL MECHANICS (Extending Project 22)

→ When Redshift receives a COPY command:

1. Redshift leader node parses COPY command
2. Leader reads manifest/S3 path → determines file(s) to load
3. Leader distributes work to compute nodes
4. Each compute node:
   a. Assumes the IAM Role specified in COPY command
   b. Reads assigned portion of S3 file(s) in parallel
   c. Parses CSV/JSON/Parquet format
   d. Applies data type conversions
   e. Validates against MAXERROR threshold
   f. Distributes rows to correct slice based on distribution key
   g. Writes to table storage
5. Leader collects results from all compute nodes
6. COPY completes → rows visible for queries

Why COPY is fast:
→ COPY is massively parallel — all compute nodes read from S3 simultaneously
→ If file is 100 MB and cluster has 4 compute nodes → each reads ~25 MB in parallel
→ This is why COPY is preferred over INSERT for bulk loading

COPY vs INSERT performance:

Method1 Million RowsUse Case
COPY from S310-30 seconds✅ Bulk loading
Individual INSERTs30-60 minutesNever for bulk — only for single-row operations

MECHANIC 5 — LAMBDA COLD START IN VPC vs NON-VPC

→ Even though we chose non-VPC for this project, understanding the mechanics matters:

Non-VPC Lambda Cold Start:

1. Lambda service receives invocation
2. Provision execution environment (download code, set up runtime)
3. Initialize function (import libraries, run initialization code)
4. Execute handler
Total cold start: 200ms - 2 seconds

VPC Lambda Cold Start:

1. Lambda service receives invocation
2. Create/attach Elastic Network Interface (ENI) in VPC subnet
3. Assign private IP address from subnet CIDR
4. Configure security group rules on ENI
5. Provision execution environment
6. Initialize function
7. Execute handler
Total cold start: 1-10 seconds (ENI creation was slow, improved with VPC-to-VPC NAT in 2019)

→ AWS improved VPC Lambda cold starts significantly in 2019 (Hyperplane)
→ Current VPC cold start: ~1-2 seconds additional (down from 10+ seconds historically)
→ But still measurable — for latency-sensitive applications, non-VPC is preferred when possible

🎯 KEY TAKEAWAY

→ Event-driven pipelines are fundamentally different from orchestrated pipelines
→ Orchestrated (Project 31, 32): "Run these 5 steps in order every day at 6 AM"
→ Event-driven (this project): "When THIS happens → do THAT → immediately"

→ The S3 → Lambda → Redshift pattern is the SIMPLEST production data loading pipeline:

  • No orchestrator needed (no Step Functions, no Glue Workflow)
  • No scheduling needed (S3 event triggers instantly)
  • No servers to manage (Lambda is serverless)
  • No connection management (Redshift Data API handles it)
  • Costs essentially nothing ($0.41/month for 8 daily files)

→ The CRITICAL architecture decision in this project: VPC vs No-VPC for Lambda

  • Redshift Data API = HTTPS call = NO VPC needed = simpler, faster, cheaper
  • Direct JDBC to Redshift = TCP to port 5439 = VPC needed = complex networking
  • ALWAYS evaluate whether VPC is truly required before adding it
  • This decision point recurs in Project 45 (Lambda accessing RDS — where VPC IS required)

→ The CRITICAL operational decisions:

  • S3 prefix/suffix filters prevent infinite trigger loops
  • Reserved concurrency protects Redshift from connection overload
  • DLQ captures failed events that would otherwise be lost
  • Idempotency checks prevent duplicate data loading
  • File movement (to processed/quarantine) prevents reprocessing and enables audit

→ What this project proves:

  • Not every pipeline needs an orchestrator
  • Simple event → process → load patterns work perfectly with just S3 + Lambda
  • The decision between orchestrated and event-driven depends on COMPLEXITY, not preference
  • When the pipeline grows beyond Lambda's capabilities → migrate to Step Functions (Project 32)
