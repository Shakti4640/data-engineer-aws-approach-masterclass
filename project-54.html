STEP 4 — DESIGN AND ATTACH SERVICE CONTROL POLICIES (SCPs)

Understanding SCPs — The Most Powerful Governance Tool:

AspectIAM PolicyService Control Policy (SCP)
ScopeSingle account, single principalOrganization-wide or OU-wide
OverrideCan be overridden by another IAM policyCANNOT be overridden by ANY IAM policy
EffectGrants permissionsSets MAXIMUM ALLOWED permissions (guardrails)
Who it affectsSpecific users/rolesALL principals in the account (including root)
Exception—Management account is NEVER affected by SCPs

How SCPs Work (Critical Understanding):

SCP says: "DENY ec2:RunInstances in us-west-2"
IAM Policy says: "ALLOW ec2:RunInstances in us-west-2"

Result: DENIED ❌

→ SCP is a CEILING — it defines the MAXIMUM permissions
→ IAM policy is the ACTUAL permission within that ceiling
→ Even if IAM says ALLOW — SCP DENY wins
→ Even the ACCOUNT ROOT USER is bound by SCPs
→ ONLY the MANAGEMENT ACCOUNT root is exempt from SCPs

SCP Strategy: Deny-List vs Allow-List:

StrategyHow It WorksWhen to Use
Deny-list (Recommended)Start with full access → Deny specific dangerous actionsMost organizations — easier to manage
Allow-listStart with no access → Allow only specific servicesMaximum security — but complex to maintain

→ PayScale uses Deny-list strategy — block dangerous actions, allow everything else

SCP 1: Deny Region Restriction (ALL OUs except Sandbox)

Purpose: Prevent resource creation outside approved regions

SettingValue
SCP Namescp-deny-unapproved-regions
Attached toRoot (affects all OUs)

Policy Logic:

DENY all actions on all services
WHERE condition: aws:RequestedRegion NOT IN [ap-south-1, us-east-1]
EXCEPT for: Global services (IAM, Organizations, CloudFront, Route 53, STS, Support)

Why us-east-1 is Included:
→ Some global services (S3 bucket creation for CloudFront, ACM certificates for CloudFront) require us-east-1
→ CloudWatch Events for global services route through us-east-1
→ Billing and Cost Explorer are in us-east-1
→ Without us-east-1 → Some legitimate operations fail

Excluded Services (Must NOT be region-restricted):

ServiceWhy Excluded
IAMGlobal service — no region concept
STSGlobal service — federation and role assumption
OrganizationsGlobal service — org management
CloudFrontGlobal service — edge distribution
Route 53Global service — DNS
WAF (Global)Global WAF policies
SupportAWS Support is global
BudgetsGlobal service
Cost ExplorerGlobal service
HealthAWS Health events are global

SCP 2: Deny CloudTrail Modification (ALL OUs)

Purpose: Prevent anyone from disabling, deleting, or modifying CloudTrail

SettingValue
SCP Namescp-deny-cloudtrail-modification
Attached toRoot (affects all OUs)

Policy Logic:

DENY the following actions:
    cloudtrail:DeleteTrail
    cloudtrail:StopLogging
    cloudtrail:UpdateTrail
    cloudtrail:PutEventSelectors (removing event selectors)
    
ON ALL resources
FOR ALL principals (including root of member accounts)

→ Even the root user of Account A CANNOT disable CloudTrail
→ Only the Management Account (Account L) root can modify the organizational trail
→ This ensures audit trail integrity across the entire organization

SCP 3: Deny Unencrypted Resources (Production + Infrastructure OUs)

Purpose: Enforce encryption at rest for all data services

SettingValue
SCP Namescp-deny-unencrypted-resources
Attached toOU: Production, OU: Infrastructure

Policy Logic:

DENY s3:CreateBucket
WHERE condition: s3:x-amz-server-side-encryption is NOT present

DENY rds:CreateDBInstance
WHERE condition: rds:StorageEncrypted = false

DENY redshift:CreateCluster
WHERE condition: redshift:Encrypted = false

DENY ec2:CreateVolume
WHERE condition: ec2:Encrypted = false

DENY ec2:RunInstances
WHERE condition: ec2:Encrypted = false (for EBS volumes)

→ Any attempt to create unencrypted S3 bucket, RDS instance, Redshift cluster, or EBS volume → DENIED
→ This is a PREVENTIVE control — stops the problem BEFORE it happens
→ Compared to AWS Config rules (DETECTIVE control) — Config detects AFTER creation

SCP 4: Deny IAM User Creation with Console Access (ALL OUs)

Purpose: Force SSO usage — no long-lived IAM credentials

SettingValue
SCP Namescp-deny-iam-console-users
Attached toRoot (affects all OUs)

Policy Logic:

DENY iam:CreateLoginProfile (creates console password for IAM user)
DENY iam:CreateAccessKey (creates long-lived access keys)

EXCEPT when performed by:
    arn:aws:iam::*:role/OrganizationAccountAccessRole (bootstrap role)
    arn:aws:iam::*:role/CICDPipelineRole (automated deployments)

→ No human can create IAM users with passwords or access keys
→ Exception for automated roles that need service account access keys
→ All human access must go through AWS IAM Identity Center (SSO)

SCP 5: Deny Expensive Instance Types (Non-Production + Sandbox OUs)

Purpose: Prevent accidental launch of expensive resources in dev/staging

SettingValue
SCP Namescp-deny-expensive-instances
Attached toOU: Non-Production, OU: Sandbox

Policy Logic:

DENY ec2:RunInstances
WHERE condition: ec2:InstanceType IN [
    p4d.24xlarge, p3.16xlarge, p3.8xlarge,     -- GPU instances ($10-32/hour)
    x2idn.*, x2iedn.*, x1e.*,                  -- Memory-optimized ($13-26/hour)
    inf1.24xlarge, inf2.*,                      -- Inference ($4-12/hour)
    dl1.24xlarge,                               -- Deep Learning ($13/hour)
    *.metal                                      -- Bare metal instances
]

DENY redshift:CreateCluster
WHERE condition: redshift:NodeType IN [
    ra3.16xlarge, dc2.8xlarge                   -- Large Redshift nodes
]

DENY rds:CreateDBInstance
WHERE condition: rds:DBInstanceClass IN [
    db.r6g.16xlarge, db.r5.24xlarge,           -- Large RDS instances
    db.x2g.*                                    -- Memory-optimized RDS
]

→ Developers cannot launch $32/hour GPU instances in dev accounts
→ The $786/day surprise bill from Account G → PREVENTED
→ If they genuinely need expensive instances → Request exception through the Platform Team

SCP 7: Deny Public S3 Buckets (ALL OUs)

Purpose: Prevent creation of publicly accessible S3 buckets

SettingValue
SCP Namescp-deny-public-s3
Attached toRoot

Policy Logic:

DENY s3:PutBucketPublicAccessBlock
WHERE condition: s3:PublicAccessBlockConfiguration.BlockPublicAcls = false
    OR s3:PublicAccessBlockConfiguration.BlockPublicPolicy = false
    OR s3:PublicAccessBlockConfiguration.IgnorePublicAcls = false
    OR s3:PublicAccessBlockConfiguration.RestrictPublicBuckets = false

DENY s3:PutBucketPolicy
WHERE condition: Policy contains Principal = "*" (public access)

→ Nobody in any account can make an S3 bucket public
→ The Customer Success team's public bucket (from the problem statement) → PREVENTED
→ If a team genuinely needs a public bucket (static website) → Request exception from Security OU

SCP Inheritance:

Root SCP applies to ALL OUs and ALL accounts (except Management Account)
    |
    ├── Security OU: Root SCPs + Security-specific SCPs
    │   └── Account K: Inherits all SCPs from Root + Security OU
    |
    ├── Production OU: Root SCPs + Production-specific SCPs
    │   └── Account A: Inherits all SCPs from Root + Production OU
    |
    └── Non-Production OU: Root SCPs + Non-Prod-specific SCPs
        └── Account F: Inherits all SCPs from Root + Non-Prod OU

→ SCPs are CUMULATIVE — a principal must be allowed by ALL SCPs in the path from Root to the account
→ If Root SCP denies an action → OU-level SCP cannot override it
→ SCPs only RESTRICT — they never GRANT permissions

✅ WHAT SHOULD DO

→ Always use a DEDICATED management account with no application workloads
→ Always enable "All features" when creating the organization — never just consolidated billing
→ Always design OU structure BEFORE inviting accounts — moving accounts between OUs changes their SCPs
→ Always use Deny-list SCP strategy — start permissive, add specific denials
→ Always protect CloudTrail with SCPs — audit trail integrity is non-negotiable
→ Always enforce encryption at rest via SCPs in production and infrastructure OUs
→ Always restrict regions via SCPs — prevent data residency violations
→ Always configure consolidated billing with cost allocation tags from day one
→ Always delegate security service administration to a dedicated security account
→ Always implement account vending for automated new account provisioning
→ Always start tag policies in "report only" mode — switch to enforcement after cleanup
→ Always protect the management account root with hardware MFA
→ Always use AWS Control Tower for organizations with 10+ accounts
→ Always set budget alerts at organization, OU, and account levels
→ Always keep the management account minimal — no applications, no data, no experiments

❌ WHAT SHOULD NOT DO

→ Never use an application account as the management account — compromise risk is too high
→ Never choose "consolidated billing only" feature set — you lose all governance capabilities
→ Never apply SCPs to the management account — they have no effect (by design) — protect it with IAM instead
→ Never create Allow-list SCPs without thorough testing — you WILL break legitimate operations
→ Never forget that SCPs affect the ACCOUNT ROOT USER — test carefully before deploying
→ Never deploy SCPs to production without testing in Sandbox/Non-Prod first
→ Never allow member accounts to leave the organization without approval
→ Never skip the region restriction SCP — one resource in the wrong region = compliance violation
→ Never let individual accounts manage their own CloudTrail — use organizational trail
→ Never grant AdministratorAccess in member accounts without SCP guardrails — SCPs are the safety net
→ Never ignore tag policy compliance — untagged resources are unattributable costs
→ Never use personal email addresses for account root users — use distribution lists
→ Never create new accounts manually when account vending is available — inconsistent baseline
→ Never assume SCPs are retroactive — they prevent NEW actions but don't fix EXISTING resources

The Complete Permission Decision Flow:

Request arrives
    |
    ↓
SCP Check (Organization level):
    ALL SCPs from Root to OU to Account must ALLOW (or not DENY)
    Any DENY in any SCP → Request DENIED → STOP
    |
    ↓
IAM Check (Account level):
    IAM policy must ALLOW the action
    No ALLOW → Request DENIED → STOP
    |
    ↓
Resource Policy Check (if applicable):
    Resource policy (S3 bucket policy, KMS key policy) evaluated
    |
    ↓
Permission Boundary Check (if applicable):
    Permission boundary on IAM entity evaluated
    |
    ↓
Session Policy Check (if applicable):
    Session policy on assumed role evaluated
    |
    ↓
FINAL: ALLOW only if ALL checks pass

→ SCP is the FIRST check — it acts as a pre-filter before IAM evaluation
→ This is why SCPs are so powerful — they cannot be bypassed by any IAM configuration

→ The Management Account is EXEMPT from SCPs — protect it with IAM and extreme access restriction
→ Organizational Units (OUs) enable differentiated governance — production gets encryption enforcement, dev gets cost limits
→ Consolidated billing provides single invoice + volume discounts + cost allocation via tags

→ Delegated administration allows security team to manage GuardDuty/Security Hub/Config organization-wide
→ Account vending automates new account creation from 2 weeks to 30 minutes with baseline security
→ SCPs are NOT retroactive — they prevent NEW non-compliant actions but don't fix EXISTING resources → Use AWS Config for detection + remediation
→ Always test SCPs in Sandbox → Non-Prod → Production (staged rollout)
