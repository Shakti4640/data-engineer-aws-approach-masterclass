PROJECT 60: THE ENTERPRISE DATA MESH — Multi-Account Producer-Consumer Pattern with Full Governance & Observability

🔴 HYPOTHETICAL PROBLEM

→ DataPulse Corp has grown dramatically over the past 18 months:

  • Acquired InsightWorks (Project 58) and integrated their data
  • Built a complete serverless analytics platform (Project 59)
  • Hardened security across 5 accounts (Project 57)
  • Optimized costs to save 54% (Project 56)
  • Implemented cross-account governance via Organizations (Project 54)

→ But the organization has hit a SCALING WALL — not technical, but ORGANIZATIONAL:

The Central Data Team Bottleneck:

TeamData RequestWait TimeCurrent Status
MarketingCampaign attribution dataset with hourly refresh3 weeks in backlogPending — data team overloaded
FinanceQuarterly reconciliation dataset joining 4 sources5 weeks in backlogNot started
InsightWorks IntegrationRegional sales normalization2 weeks in backlogIn progress
ProductUser behavior analytics from clickstream6 weeks in backlogNot prioritized
Customer SuccessChurn prediction feature dataset4 weeks in backlogPending
Partner IntegrationPartner-facing aggregated API datasets8 weeks in backlogBlocked by security review

→ The central data team has 6 engineers
→ They maintain 47 Glue ETL jobs, 23 Crawlers, 15 Redshift schemas, 12 Athena workgroups
→ Every new dataset request flows through them
→ Average time from request to production: 4-8 weeks
→ Business teams are frustrated — they know their domain data better than the central team

The Ownership Problem:
→ Marketing team understands campaign data better than anyone
→ But they cannot create their own Glue jobs, Athena tables, or Redshift schemas
→ They must explain their domain to the central team → central team builds it → marketing validates → iterations take weeks
→ The central team becomes a translation layer between domain experts and infrastructure

The Governance Problem:
→ 5 AWS accounts with data spread across them
→ No clear ownership of datasets — "who is responsible for data quality of the customer table?"
→ No data contracts — downstream consumers break when upstream schemas change
→ No data discoverability — teams don't know what data exists in other accounts
→ Lake Formation permissions (Project 47) exist but are managed ad-hoc

The Observability Problem:
→ CloudWatch metrics are per-account — no cross-account unified view
→ When a Glue job fails in Account 3 → Account 1's dashboard doesn't update → users see stale data for hours
→ No end-to-end pipeline observability from ingestion to consumption
→ SLA monitoring doesn't exist — nobody knows if data freshness commitments are met

The CTO's Vision:
→ "Each domain team should OWN their data end-to-end"
→ "They should PUBLISH data products that other teams can discover and consume"
→ "A central governance team should provide guardrails — not gatekeeping"
→ "We need to see the health of the ENTIRE data platform from one dashboard"
→ "This must work across all our AWS accounts with proper security"

→ This is the Data Mesh architecture pattern
→ Coined by Zhamak Dehghani — it applies product thinking and domain ownership to data

Step C5 — Implement Data Quality Validation

→ Each data product has quality checks that run after ETL:

Glue Data Quality Rules for campaign_attribution:

RuleCheckThreshold
CompletenessCompleteness "campaign_id" >= 0.9999% of rows have campaign_id

Row CountRowCount >= 10000At least 10K rows (prevents empty table publication)
Column TypeColumnDataType "revenue" = DoubleRevenue column is numeric

→ Quality validation Lambda:

  1. Runs Glue Data Quality checks
  2. Calculates composite quality score (weighted average of all rules)
  3. If score ≥ 95% → data product is "HEALTHY" → publish notification
  4. If score 80-95% → data product is "DEGRADED" → publish warning

✅ WHAT SHOULD DO

→ Always assign clear domain ownership — every data product has ONE owner team
→ Always define data product standards BEFORE teams start publishing
→ Always use Lake Formation TBAC for scalable permission management
→ Always require schema registration in Glue Schema Registry — backward compatibility prevents consumer breakage
→ Always implement data quality checks in every domain pipeline — quality is the producer's responsibility
→ Always publish data product events via EventBridge — enables event-driven consumption
→ Always maintain a central Data Product Registry — discoverability is critical
→ Always use resource links (not data copying) for cross-account access — keeps data in producer's control
→ Always implement cross-account CloudWatch observability — single pane of glass for platform health
→ Always enforce SLAs with automated monitoring and alerting — trust comes from reliability
→ Always provide self-serve tooling for domain teams — reduce dependency on central platform team
→ Always version schemas — breaking changes must be communicated and managed
→ Always separate governance (rules + standards) from gatekeeping (approval bottleneck)
→ Always start with 2-3 domain teams as pilots — don't transform all teams simultaneously
→ Always tag resources with Domain and Classification — foundation for TBAC and cost allocation

❌ WHAT SHOULD NOT DO

→ Never allow domains to publish directly to consumers without governance registration — creates shadow data products
→ Never copy data between accounts when Lake Formation sharing works — duplication causes inconsistency
→ Never let the central team own domain-specific ETL — they become a bottleneck
→ Never skip data quality checks because "the data is obviously correct" — quality degrades silently
→ Never use IAM-only access control for cross-account data sharing — Lake Formation provides finer granularity
→ Never create one giant account for all domains — defeats the purpose of domain ownership
→ Never implement all 4 Data Mesh principles at once — start with domain ownership, then add incrementally
→ Never force ALL data into the mesh — some shared infrastructure data (IAM, networking) stays centralized
→ Never let domains define their own standards independently — federated means SHARED standards, not NO standards
→ Never skip the discovery portal — if teams can't find data products, they'll build their own (duplication)
→ Never grant blanket access to all data products — least privilege applies even in a data mesh
→ Never ignore cross-domain dependencies — if Finance depends on Marketing's data, that's a contractual relationship
→ Never let domain teams skip EventBridge notifications — consumers need to know when data is fresh
→ Never use cross-account S3 bucket policies alone for data sharing — Lake Formation provides audit trail and column-level security
→ Never implement data mesh without executive sponsorship — organizational change requires leadership alignment

  2. Are there column restrictions? (Filter columns)
  3. Are there row-level filters? (Apply filter expression)
           │
           ▼ (If authorized)
Lake Formation returns:
  • Temporary S3 credentials (scoped to specific S3 prefixes)
  • Column filter (if CLS configured)
  • Row filter expression (if row-level security configured)
           │
           ▼
Athena uses temporary credentials to read S3 objects
  • Credentials are short-lived (expire in minutes)
  • Credentials are scoped to ONLY the specific S3 paths for this table
  • Even if consumer tries to access other S3 paths → ACCESS DENIED

→ This is fundamentally different from cross-account S3 bucket policies (Project 39):

  • Bucket policies grant broad access to entire buckets or prefixes
  • Lake Formation grants narrow access to specific table data only
  • Lake Formation adds column and row filtering on top
  • Lake Formation provides centralized audit trail

3 → Tag-Based Access Control (TBAC) Evaluation

Consumer queries: SELECT * FROM dp_marketing.campaign_attribution
           │
           ▼
Lake Formation evaluates TBAC:
           │
           ├── Resource Tags:
           │   campaign_attribution has tags:
           │   • classification = "internal"
           │   • domain = "marketing"
           │   • tier = "gold"
           │
           ├── Principal Tags:
           │   Consumer role has tags:
           │   • classification = ["internal", "public"]
           │   • domain = ["marketing", "finance", "product"]
           │
           ├── Tag Expression Evaluation:
           │   For each tag key:
           │     Resource tag value ∈ Principal tag values?
           │
           │   classification: "internal" ∈ ["internal", "public"] → ✅ MATCH
           │   domain: "marketing" ∈ ["marketing", "finance", "product"] → ✅ MATCH
           │
           └── ALL tag keys match → ACCESS GRANTED
           
If even ONE tag key doesn't match → ACCESS DENIED

Example: Consumer tries to query dp_finance.cost_data
  Resource tags: classification = "confidential"
  Principal tags: classification = ["internal", "public"]
  "confidential" ∈ ["internal", "public"] → ❌ NO MATCH → ACCESS DENIED

→ TBAC evaluation happens at QUERY TIME — not at grant time
→ Adding a new tag to a principal IMMEDIATELY changes their access
→ Adding a new tag to a resource IMMEDIATELY changes who can access it
→ No need to modify individual grants — tags drive everything

2 → Compatibility Check Algorithm

Change TypeBACKWARD Compatible?FORWARD Compatible?FULL Compatible?
Add optional column✅ Yes❌ No❌ No
Add required column (with default)✅ Yes❌ No❌ No
Remove column❌ No✅ Yes❌ No
Rename column❌ No❌ No❌ No
Change column type (widening: INT→LONG)✅ Yes❌ No❌ No
Change column type (narrowing: LONG→INT)❌ No✅ Yes❌ No
Add new enum value✅ Yes❌ No❌ No
Remove enum value❌ No✅ Yes❌ No

→ BACKWARD is recommended for Data Mesh:

  • Producers can ADD columns freely (new features)
  • Producers CANNOT remove or rename columns (breaks existing consumers)
  • Forces producers to think about consumer impact before schema changes
  • If a column must be removed → deprecate first (add _deprecated suffix) → remove after all consumers migrate
