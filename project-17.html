PROJECT 17: THE DATA LAKE THAT NEEDED A CATALOG — Glue Crawler Cataloging S3 Data

🔴 HYPOTHETICAL PROBLEM

→ ShopNow (from Projects 1-16) now has:

  • S3 bucket with event-driven pipelines (Projects 1, 12, 15)
  • RDS MariaDB for transactional operations (Projects 9, 14)
  • Redshift for analytical queries (Project 16)
  • Full VPC, messaging, monitoring infrastructure (Projects 3-8, 13)

→ The data engineering team manages data from FIVE sources landing in S3:

Data SourceS3 LocationFormatFilesUpdate Frequency
Orders exports3://shopnow-data-lake/raw/orders/CSV365 daily filesDaily
Product catalogs3://shopnow-data-lake/raw/products/JSON12 monthly snapshotsMonthly
Clickstreams3://shopnow-data-lake/raw/clickstream/Parquet8,760 hourly filesHourly
Marketing campaignss3://shopnow-data-lake/raw/campaigns/CSV52 weekly filesWeekly
Customer demographicss3://shopnow-data-lake/raw/customers/CSV4 quarterly filesQuarterly

→ Problem 1: Nobody knows the SCHEMA of each dataset
→ The orders CSV has 22 columns — but which columns? What data types?
→ The clickstream Parquet has nested structures — what's inside?
→ The product JSON has arrays and objects — how to query it?
→ Each time an analyst wants to query data → they download a sample → open in Excel → manually inspect columns
→ This takes 30-60 minutes per dataset → repeated by EVERY analyst

→ Problem 2: To query S3 data → they must CREATE TABLE statements manually

Example: To query orders CSV in Athena (future — Project 18):
  CREATE EXTERNAL TABLE orders (
    order_id BIGINT,
    customer_id BIGINT,
    order_date STRING,
    ... (22 columns — must type each one correctly)
  )
  ROW FORMAT DELIMITED FIELDS TERMINATED BY ','
  STORED AS TEXTFILE
  LOCATION 's3://shopnow-data-lake/raw/orders/'

→ Writing this for 5 datasets × 22+ columns each = tedious and error-prone
→ If the schema changes (new column added) → manual table definition is WRONG → queries fail
→ Nobody maintains these definitions → they become stale

→ Problem 3: New data files arrive constantly
→ Clickstream: 24 new Parquet files per day
→ Orders: 1 new CSV per day
→ When a new file arrives with a NEW column → existing table definitions don't include it
→ Queries on the new column fail → analyst says "the data is broken"
→ Actually the TABLE DEFINITION is broken — the data is fine

→ Problem 4: The team wants to use BOTH Athena (serverless SQL on S3) AND Redshift Spectrum (query S3 from Redshift)
→ Both services need a METADATA CATALOG that describes:

  • Where the data is in S3 (location)
  • What format it's in (CSV, Parquet, JSON)
  • What columns exist (schema)
  • What data types each column has
  • How files are partitioned (by date, by region, etc.)
    → Without a shared catalog → each service needs separate table definitions → inconsistency

→ Problem 5: Data is partitioned by date in S3:

s3://shopnow-data-lake/raw/orders/year=2024/month=01/day=15/orders-20240115.csv
s3://shopnow-data-lake/raw/orders/year=2024/month=01/day=16/orders-20240116.csv
s3://shopnow-data-lake/raw/orders/year=2024/month=02/day=01/orders-20240201.csv

→ This Hive-style partitioning (key=value/) improves query performance
→ But someone must REGISTER each partition in the catalog
→ With 365 days × multiple years = thousands of partitions → manual registration is impossible

→ The CTO demands:

  • AUTOMATIC schema discovery — no manual column typing
  • CENTRALIZED metadata catalog — one source of truth for all services
  • AUTOMATIC partition registration — new partitions discovered without manual work
  • Schema evolution handling — new columns detected automatically
  • Shared catalog for Athena, Redshift Spectrum, Glue ETL, and future services

Hive-Style Partitioning:

→ year=2024/month=01/day=15/ is called Hive-style partitioning
→ The folder names contain key=value format
→ Glue Crawler AUTOMATICALLY detects this pattern → creates partition columns
→ Athena/Spectrum can then filter by partition → reads ONLY relevant files

Query: SELECT * FROM orders WHERE year = 2024 AND month = 1

WITHOUT partitioning: Scans ALL 365+ files across all dates → slow, expensive
WITH partitioning: Scans ONLY files in year=2024/month=01/ → ~31 files → fast, cheap

→ Partitioning is the MOST important performance optimization for S3-based analytics
→ Always partition time-series data by date (year/month/day at minimum)

STEP 7 — UNDERSTAND SCHEMA EVOLUTION AND PARTITION MANAGEMENT

Schema Evolution: New Column Added

BEFORE: orders CSV has 22 columns
  order_id, customer_id, order_date, ..., order_status

AFTER: Source system adds a new column "loyalty_points"
  order_id, customer_id, order_date, ..., order_status, loyalty_points

What happens on next crawler run:
  1. Crawler reads new CSV file → discovers 23 columns (was 22)
  2. New column "loyalty_points" detected (type: int)
  3. Crawler behavior: "Add new columns only"
  4. Table updated: loyalty_points (int) added to schema
  5. Old files: loyalty_points = NULL (column didn't exist)
  6. New files: loyalty_points = actual value

→ Schema evolution is handled GRACEFULLY
→ Old data → NULL for new columns → no errors
→ New data → full values → no schema mismatch

Schema Evolution: Column Type Change

BEFORE: pincode column is "string" (correct — pincodes have leading zeros)
AFTER: New data file has pincode as numeric (lost leading zeros)

Crawler behavior:
  "Add new columns only" → Does NOT change existing column type
  pincode stays as "string" → new numeric values are read as strings → works

  "Update all new and existing columns" → Would change to "bigint"
  → Old string values with leading zeros would fail to parse → DANGEROUS

→ "Add new columns only" protects against type changes → safest option
→ If you NEED to change a type → manually update the table definition in the Catalog

Partition Management: New Partitions

BEFORE: Crawler ran on Jan 15, 2025 — registered partitions up to 2025/01/15
AFTER: New data arrives for Jan 16, 17, 18

What happens on next crawler run:
  Subsequent runs setting: "Crawl new sub-folders only"
  1. Crawler checks for new sub-folders since last run
  2. Discovers: year=2025/month=01/day=16/, day=17/, day=18/
  3. Creates 3 new partitions in the catalog
  4. Does NOT re-crawl existing partitions → fast

→ With "Crawl new sub-folders only" → each subsequent run is FAST
→ Only new data is discovered → minutes instead of hours for large datasets

Alternative: MSCK REPAIR TABLE (Athena)

→ Instead of running a crawler → you can register new partitions directly in Athena:

SQL (Athena): MSCK REPAIR TABLE shopnow_raw.orders

→ This scans S3 for new Hive-style partitions → adds them to the catalog
→ Faster than running a full crawler for partition-only updates
→ But doesn't detect schema changes — only new partitions

Alternative: ALTER TABLE ADD PARTITION (Manual)

SQL (Athena):
  ALTER TABLE shopnow_raw.orders ADD
    PARTITION (year='2025', month='01', day='16')
    LOCATION 's3://shopnow-data-lake/raw/orders/year=2025/month=01/day=16/'

✅ WHAT SHOULD DO

→ Always create separate Glue databases for each data zone (raw, curated, refined)
→ Always use Hive-style partitioning (key=value/) for time-series data in S3
→ Always set crawler schema change behavior to "Add new columns only" — safest option
→ Always set subsequent crawl behavior to "Crawl new sub-folders only" — faster incremental crawls
→ Always create a dedicated IAM Role for Glue with least-privilege S3 and Catalog permissions
→ Always schedule crawlers to run after data is expected to arrive — not during data loading
→ Always verify crawled table schemas against expected schemas — crawlers can misinterpret types
→ Always use meaningful database and table names — they appear in Athena and Spectrum
→ Always use Parquet format where possible — self-describing, compressed, columnar → best performance
→ Always partition data by the most commonly filtered dimension (usually date)
→ Always check crawler logs in CloudWatch for errors — silent failures are common
→ Always tag crawlers and databases for cost tracking and organizational management

❌ WHAT SHOULD NOT DO

→ Never let crawlers create tables in the default database — use purpose-specific databases
→ Never use "Update all new and existing columns" for production — can change column types unexpectedly
→ Never crawl the entire bucket with one crawler — separate crawlers per data source for isolation
→ Never schedule crawlers during data loading — may read partial/incomplete files
→ Never assume crawler-inferred data types are always correct — dates are often inferred as strings
→ Never use non-Hive-style partition folders expecting automatic partition detection — crawler won't recognize them
→ Never forget to grant the Glue role access to KMS if S3 data is KMS-encrypted
→ Never skip Glue Database creation — tables without databases are hard to organize
→ Never run crawlers more frequently than data arrives — wastes DPU costs
→ Never ignore crawler errors — a failed crawler means stale catalog → wrong query results
→ Never create hundreds of individual crawlers when a few multi-source crawlers suffice
→ Never forget that Glue Catalog is REGIONAL — tables in ap-south-1 are not visible in us-east-1

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

1 → HOW A CRAWLER DISCOVERS SCHEMAS

Crawler: shopnow-orders-crawler starts
        │
        ▼
Step 1: ENUMERATE — List all objects in the S3 path
        s3:ListBucket on s3://shopnow-data-lake/raw/orders/
        Discovers: 380 sub-folders (year=.../month=.../day=.../), 380 CSV files
        │
        ▼
Step 2: CLASSIFY — Determine file format
        Reads first few KB of a sample file
        Built-in classifiers try to match:
          - CSV classifier: Tries comma, tab, pipe delimiters → comma works ✓
          - JSON classifier: Not valid JSON → skip
          - Parquet classifier: Not Parquet magic bytes → skip
        Classification result: CSV (comma-delimited, with header row)
        │
        ▼
Step 3: SAMPLE — Read sample data for schema inference
        Reads first N rows (configurable, default varies by format)
        For CSV: Reads header row → column names
        For each column → analyzes values to infer data type:
          "12345" → all rows are numeric → bigint
          "999.99" → decimal values → double
          "2024-01-15" → looks like date but stored as string → string
          "Shoes" → text → string
          "" (empty) → could be any type → string (safest)
        │
        ▼
Step 4: DETECT PARTITIONS — Identify Hive-style partitions
        Folder structure: year=2024/month=01/day=15/
        Pattern detected: key=value/ → Hive-style partitions
        Partition keys: year (string), month (string), day (string)
        Enumerates all unique partition value combinations → 380 partitions
        │
        ▼
Step 5: GROUP — Determine table grouping
        All files in raw/orders/ have the same schema → single table "orders"
        If files had DIFFERENT schemas → would create separate tables
        │
        ▼
Step 6: CATALOG — Write to Glue Data Catalog
        │
        ├── Table already exists?
        │   ├── NO → CREATE new table with discovered schema
        │   └── YES → UPDATE based on schema change behavior
        │       ├── "Add new columns only" → Merge new columns, keep existing
        │       ├── "Update all" → Replace schema with new discovery
        │       └── "Log only" → Record changes, don't modify
        │
        ├── Register partitions
        │   For each partition: {year=2024, month=01, day=15} → s3://path/
        │   Batch create partitions in the catalog
        │
        └── Write crawler metrics
            Tables created/updated, partitions created, duration, DPU-hours
        │
        ▼
Step 7: LOG — Write results to CloudWatch Logs
        Log group: /aws-glue/crawlers
        Contains: Discovery details, errors, warnings, statistics

🚨 COMMON PITFALLS AND FAILURE MODES

#PitfallWhat HappensHow to Avoid
1Crawler infers date columns as stringsQueries can't use date functions on string columnsManually update column type in catalog or use CAST in queries
2Crawler misinterprets numeric strings (pincode) as bigintLeading zeros lost → "110001" becomes 110001Use custom classifier or manually fix type to string
3Crawler runs during data uploadReads partial/incomplete file → wrong schema or corrupt dataSchedule crawler AFTER data loading completes
4Non-Hive-style partitions (date=20240115 instead of year=2024/month=01/day=15)Crawler doesn't detect partitions → single table with all data → slow queriesUse Hive-style key=value/ partition naming
5Missing IAM permissions on S3Crawler fails with "Access Denied" → no tables createdVerify role has s3:GetObject and s3:ListBucket on the data path
6KMS-encrypted S3 without KMS permissions on Glue roleCrawler can't read encrypted data → fails silentlyAdd kms:Decrypt to Glue role for the S3 encryption key
7Mixed file formats in same S3 pathCrawler creates multiple tables or picks wrong formatKeep one format per S3 path
8Schema change breaks existing queriesNew column added but old Athena views reference fixed columnsUse "Add new columns only" + test queries after crawl
9Too many small partitionsCatalog API becomes slow, query planning overhead increasesUse appropriate partition granularity (daily, not per-minute)
10Crawler creates tables with auto-generated namesTable names like "raw_orders" or "orders_csv" — confusingUse table name prefix setting or rename after creation
11Empty S3 path → crawler creates no tableNo table in catalog → queries fail → confusionEnsure at least one data file exists before first crawl
12Forgot to run crawler after new data → stale catalogAnalysts query old data → wrong business decisionsSchedule crawlers on appropriate frequency

📈 SCALABILITY CONSIDERATIONS

→ Data Volume Scalability

  • Crawlers can scan PBs of data in S3
  • "Crawl new sub-folders only" keeps incremental crawls fast
  • Partition count: Up to 10 million per table → supports years of granular data
  • Catalog storage: Up to 1 million tables per database

→ Crawler Scalability

  • Up to 1,000 crawlers per account
  • Up to 100 concurrent crawler runs
  • Each crawler uses minimum 2 DPUs → auto-scales based on data volume
  • Multi-source crawlers reduce crawler count → one crawler, many tables

→ Catalog Query Scalability

  • Catalog API handles millions of requests per month
  • First million requests free → ample for most organizations
  • Athena and Spectrum batch partition requests efficiently
  • Partition indexing (available for very large partition counts) improves GetPartitions performance

→ Multi-Account Scalability

  • Each account has its own catalog
  • Cross-account catalog sharing via Lake Formation (Project 42)
  • Resource links enable consumer accounts to reference producer account tables
  • AWS RAM (Resource Access Manager) for organizational sharing

→ Schema Evolution Scalability

  • Crawlers handle column additions gracefully
  • Table versioning tracks schema changes over time
  • Consumers (Athena, Spectrum) handle NULL values for missing columns automatically
