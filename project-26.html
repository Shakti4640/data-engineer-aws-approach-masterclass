PROJECT 26: THE ORDER PROCESSING SYSTEM THAT NEEDED DECOUPLING — SNS → SQS → Lambda Event-Driven Pipeline

🔴 HYPOTHETICAL PROBLEM

→ ShopNow's order processing system has grown significantly
→ The current architecture (from Project 13) uses SNS → SQS fan-out:

Order placed on website
       ↓
Application on EC2 publishes to SNS topic: order-events
       ↓
SNS fans out to 3 SQS queues:
  ├── order-fulfillment    (+ DLQ from Project 20)
  ├── order-invoicing      (+ DLQ from Project 20)
  └── order-analytics      (+ DLQ from Project 20)

→ Currently — EC2-based consumer applications poll each SQS queue continuously
→ Each consumer runs on a dedicated EC2 instance 24/7

→ Problem 1: Wasted Compute During Off-Peak Hours

  • Order volume: 50,000 orders/day
  • Peak: 5,000 orders/hour (10 AM - 8 PM)
  • Off-peak: 200 orders/hour (midnight - 6 AM)
  • Three EC2 instances (t3.medium) run 24/7 for the three consumers
  • During off-peak (10 hours/day) → instances are 95% idle
  • Monthly cost: 3 × t3.medium × 730 hours = ~$93/month
  • Actual compute needed during off-peak: $5 worth of processing
  • $88/month wasted on idle instances

→ Problem 2: Scaling Complexity

  • During flash sales → order volume spikes to 20,000 orders/hour (4x normal peak)
  • Single EC2 consumer can process ~3,000 messages/hour
  • Queue depth grows → orders delayed by 2-4 hours during flash sales
  • Auto Scaling (Project 33 concept) could help — but takes 5-10 minutes to launch new instances
  • By the time new instances are ready → the spike may be over

→ Problem 3: Consumer Code Deployment Overhead

  • Each consumer is a full Java application running on EC2
  • Deployment requires: build JAR → upload to S3 → deploy to EC2 → restart service
  • Three separate applications to maintain, monitor, and patch
  • OS patches, Java updates, dependency management for each instance
  • The fulfillment consumer is 15 lines of actual business logic wrapped in 500 lines of boilerplate

→ Problem 4: New Event-Driven Requirements

  • The data team wants: "When a new file lands in S3 → automatically trigger Glue ETL"
  • The operations team wants: "When an order status changes → send customer notification"
  • The security team wants: "When a new IAM user is created → log to compliance system"
  • Each requirement needs a new consumer — another EC2 instance, another application

→ Problem 5: Error Handling Gaps

  • When a consumer crashes → messages pile up in the queue
  • The DLQ (Project 20) catches poison messages — but who processes DLQ messages?
  • Currently: Manual investigation by an engineer
  • Desired: Automatic processing of DLQ messages (log, alert, store for analysis)

→ The CTO says: "I want event-driven processing that scales to zero when idle, scales instantly during spikes, requires zero server management, and deploys in seconds"

STEP 1 — UNDERSTAND LAMBDA FUNDAMENTALS

What is Lambda?
→ A compute service that runs your code in response to EVENTS
→ No servers to provision, manage, or scale
→ You provide: code + configuration
→ AWS provides: compute, scaling, availability, monitoring

Execution Model:

EVENT occurs (SQS message, S3 upload, API call, schedule)
       ↓
Lambda service receives the event
       ↓
Lambda provisions a lightweight compute environment ("execution environment")
       ↓
Your code runs inside this environment
       ↓
Code completes (success or failure)
       ↓
Environment may be REUSED for next invocation (warm start)
  OR destroyed after idle period (cold start on next invocation)

Key Lambda Concepts:

Execution environmentThe container running your codeA temporary VM
Cold startFirst invocation — new environment createdBooting a fresh computer
Warm startSubsequent invocation — reuses existing environmentComputer already running
InvocationOne execution of your functionOne function call
ConcurrencyNumber of simultaneous executionsHow many parallel workers

What happens after configuration:

Lambda Event Source Mapping (managed by AWS):
       ↓
Continuously polls SQS queue using long polling
       ↓
When messages available:
  ├── Collects up to [batch size] messages
  ├── OR waits up to [batch window] seconds
  ├── (whichever comes first)
  └── Invokes Lambda function with the batch
       ↓
Lambda function receives event containing:
  ├── Records[0]: First SQS message (body, attributes, messageId, ...)
  ├── Records[1]: Second SQS message
  ├── ...
  └── Records[9]: Tenth SQS message (if batch size = 10)
       ↓
Function processes all messages in the batch
       ↓
IF function returns SUCCESS:
  └── Lambda automatically DELETES all messages from the queue
       (No need for your code to call DeleteMessage!)

IF function throws an error/exception:
  └── Lambda does NOT delete any messages
  └── Messages return to queue after visibility timeout
  └── ReceiveCount increments → eventually reaches DLQ (Project 20)

→ Critical insight: Lambda manages the ENTIRE SQS polling lifecycle
→ Your code receives messages, processes them, and returns success/failure
→ No polling logic, no DeleteMessage calls, no visibility timeout management in your code

STEP 5 — CONFIGURE BATCH PROCESSING SETTINGS

Batch Size:

Batch SizeBehaviorUse Case
1One message per Lambda invocationSimple processing, low volume, need per-message isolation
5Five messages per invocationModerate volume, moderate processing time
10Ten messages per invocation (default max for standard SQS)High volume, efficient processing
10,000Maximum (for standard SQS)Very high volume, batch-oriented processing

→ For FIFO queues: Maximum batch size = 10
→ For Standard queues: Maximum batch size = 10,000

→ Larger batch = fewer Lambda invocations = lower request cost
→ But: if ANY message in the batch fails → entire batch is returned to queue

→ For order fulfillment: Batch size = 10
→ Processing 10 orders per invocation is efficient
→ If one fails → 9 successfully processed orders are also returned to queue → reprocessed (duplicate concern)
→ Solution: "Report Batch Item Failures" (Step 6)

Batch Window (Maximum Batching Window):

WindowBehaviorUse Case
0 seconds (default)Invoke immediately with available messagesLow latency — process ASAP
5 secondsWait up to 5 seconds to collect batch size messagesBalance latency and efficiency
30 secondsWait up to 30 secondsHigh efficiency, acceptable latency
300 seconds (5 min)Maximum waitBatch-heavy processing, analytics

→ Without batch window: Lambda invokes with 1 message even if 9 more are arriving
→ With 5-second window: Lambda waits up to 5 seconds to fill the batch to 10

→ For order fulfillment: Batch window = 5 seconds
→ 5-second delay is acceptable for order processing
→ Reduces invocations by ~50% during steady traffic

STEP 6 — CONFIGURE ERROR HANDLING

→ This is where Lambda + SQS + DLQ (Project 20) integrate tightly

Scenario: One message in a batch of 10 fails

Default behavior (WITHOUT Report Batch Item Failures):

Lambda receives batch of 10 messages
  ├── Message 1: ✅ Processed successfully
  ├── Message 2: ✅ Processed successfully
  ├── ...
  ├── Message 7: ❌ Processing FAILS (bad data)
  ├── ...
  └── Message 10: ✅ Processed successfully

Function throws exception due to Message 7
  ↓
Lambda treats ENTIRE BATCH as failed
  ↓
ALL 10 messages return to queue (including 9 that succeeded)
  ↓
All 10 messages will be reprocessed → 9 messages processed TWICE
  ↓
Eventually Message 7 hits maxReceiveCount → goes to DLQ
  But Messages 1-6, 8-10 were also retried unnecessarily

→ This is wasteful and creates duplicate processing

Improved behavior (WITH Report Batch Item Failures):

→ Navigate to Lambda → Event Source Mapping → Edit

SettingValueExplanation
Report batch item failuresEnabledFunction can report individual failures

→ Now your function returns a response identifying WHICH specific messages failed

Function response format:

{
  "batchItemFailures": [
    {"itemIdentifier": "message-id-of-failed-message-7"}
  ]
}

→ Lambda processes the response:

  • Messages NOT in batchItemFailures → DELETED from queue (successfully processed)
  • Messages IN batchItemFailures → RETURNED to queue (will be retried)
Lambda receives batch of 10 messages
  ├── Message 1: ✅ Processed → deleted
  ├── Message 2: ✅ Processed → deleted
  ├── ...
  ├── Message 7: ❌ Failed → returned to queue (ReceiveCount++)
  ├── ...
  └── Message 10: ✅ Processed → deleted

Only Message 7 retries → eventually goes to DLQ
Other 9 messages processed exactly once ✅

→ This is the RECOMMENDED configuration for all production SQS → Lambda setups
→ Without it → batch failures cause cascading retries and duplicates

→ Lambda Destinations route invocation results to another service
→ Note: Destinations work for ASYNC invocations, NOT for SQS event source mappings
→ For SQS → Lambda, error handling is via batch item failures + SQS DLQ (Project 20)

However — for ASYNC Lambda invocations (e.g., S3 event → Lambda):

Destination TypeTarget ServiceUse Case

STEP 7 — CONFIGURE CONCURRENCY CONTROLS

Understanding Lambda Concurrency:

1 message arrives → 1 concurrent execution
10 messages arrive simultaneously → up to 10 concurrent executions
1000 messages in queue → Lambda scales up to process them

Scaling behavior:
  ├── Initial burst: Up to 1,000 concurrent executions immediately
  ├── After burst: Scales up 500 additional concurrent executions per minute
  └── Maximum: Account-level concurrency limit (default 1,000)

→ For SQS specifically:

  • Lambda starts with 5 concurrent invocations (5 pollers)
  • Every minute, Lambda adds 60 more concurrent invocations if queue has messages
  • Scales until queue is drained or concurrency limit is reached

Concurrency Types:

TypeConfigurationBehavior
UnreservedDefault — shares account poolFunction uses whatever concurrency is available
ReservedSet specific limit per functionGuarantees AND caps concurrency for this function
ProvisionedPre-warm execution environmentsEliminates cold starts (at a cost)

Reserved Concurrency:

→ Navigate to Lambda → Function → Configuration → Concurrency

SettingValueExplanation
Reserved concurrency50Maximum 50 simultaneous executions for this function

→ Why set a limit?

  • Protect downstream systems from overload
  • Order fulfillment calls an inventory API → API can handle 50 concurrent requests
  • Without limit → Lambda could scale to 1000 concurrent → overwhelm the API
  • Also prevents one function from consuming all account concurrency

Reserved Concurrency for each function:

FunctionReserved ConcurrencyRationale
order-fulfillment-processor50Inventory API limit
order-invoicing-processor30Invoice generation system limit
order-analytics-processor100S3 writes — can handle more parallelism
Unreserved pool820 (1000 - 50 - 30 - 100)Available for other functions

→ Sum of all reserved concurrency must be < account limit (1000)
→ Leave at least 100 unreserved for other functions

Provisioned Concurrency (Eliminating Cold Starts):

SettingValueExplanation
Provisioned concurrency5Keep 5 execution environments always warm

→ Cold start for Python: ~200ms → may be acceptable for SQS processing
→ Provisioned concurrency eliminates cold starts but costs $0.000004646 per GB-second even when idle
→ For SQS consumers: Usually NOT needed — cold starts are acceptable
→ For API Gateway backends: Often needed — cold starts affect user experience

→ ShopNow decision: No provisioned concurrency — 200ms cold start is invisible in SQS processing

✅ WHAT SHOULD DO

→ Always enable "Report batch item failures" — prevents entire batch retry on single message failure
→ Always set Lambda timeout < SQS visibility timeout — prevents duplicate processing
→ Always set reserved concurrency — protects downstream systems from Lambda's aggressive scaling
→ Always use arm64 (Graviton2) architecture — 20% cheaper, same or better performance
→ Always start with minimal memory (256 MB) and optimize based on CloudWatch metrics
→ Always use environment variables for configuration — never hardcode ARNs, bucket names, or settings
→ Always create function-specific IAM Roles with least privilege — don't share roles across functions
→ Always set CloudWatch Log retention — Lambda logs can grow rapidly (especially with DEBUG logging)
→ Always monitor Iterator Age for SQS event sources — it's the best indicator of processing health
→ Always test with a small batch first — send 10 test messages, verify processing, then enable for production
→ Always keep Lambda functions focused — one function per responsibility (single responsibility principle)
→ Always use batch window > 0 for non-latency-critical processing — reduces invocations and cost

❌ WHAT SHOULD NOT DO

→ Never set Lambda timeout > SQS visibility timeout — causes duplicate message processing
→ Never use batch size > 1 without "Report batch item failures" — one failure retries entire batch
→ Never put secrets in environment variables — use Secrets Manager (Project 37)
→ Never set reserved concurrency to 0 — this DISABLES the function entirely (0 = throttle everything)
→ Never forget the Lambda resource-based policy for S3 triggers — S3 can't invoke without permission
→ Never use x86_64 architecture without reason — arm64 is 20% cheaper
→ Never deploy Java runtime for SQS consumers unless cold start is acceptable — 2-5 second cold starts
→ Never run long-running processes (> 15 minutes) in Lambda — use ECS/Fargate or Step Functions
→ Never give Lambda sqs:* permissions — only ReceiveMessage, DeleteMessage, GetQueueAttributes
→ Never ignore throttle metrics — throttled invocations mean messages are not being processed
→ Never process and delete messages manually in Lambda code when using SQS event source — Lambda handles deletion
→ Never create a Lambda function without AWSLambdaBasicExecutionRole — you lose all logging capability

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

1 → LAMBDA SQS EVENT SOURCE MAPPING INTERNALS

EVENT SOURCE MAPPING (managed by Lambda service):
       ↓
Lambda creates INTERNAL pollers (you don't see or manage these)
       ↓
Initial state: 5 concurrent pollers
       ↓
Each poller calls SQS ReceiveMessage with long polling (20 seconds)
       ↓
SCALING UP:
  ├── If queue has messages AND current pollers are all busy
  ├── Lambda adds 60 more pollers per minute
  ├── Each poller can invoke one Lambda function
  └── Scales until: queue is empty OR concurrency limit reached
       ↓
SCALING DOWN:
  ├── If queue is empty
  ├── Pollers receive no messages
  ├── Lambda gradually reduces poller count
  └── Eventually: back to 5 minimum pollers
       ↓
MESSAGE PROCESSING:
  ├── Poller receives batch of messages (up to batch size)
  ├── Poller invokes Lambda function synchronously
  ├── Passes messages as event payload
  ├── Waits for function to complete
  │     ├── SUCCESS: Poller calls DeleteMessageBatch for all messages
  │     ├── PARTIAL FAILURE (batch item failures): Delete successful, return failed
  │     └── FAILURE: Poller does nothing → visibility timeout expires → messages retry
  └── Poller returns to polling

Key detail: Lambda polls SQS — SQS does NOT push to Lambda
→ Despite the "trigger" terminology, Lambda is PULLING from SQS
→ The Event Source Mapping is a managed poller — Lambda manages the polling infrastructure
→ This is important because: polling costs apply to Lambda's SQS API calls, not yours
→ These polling API calls are FREE — Lambda absorbs the cost

2 → COLD START VS WARM START

COLD START (First invocation or after idle period):
  ├── Lambda provisions new execution environment
  │     ├── Download function code from S3 (or container image from ECR)
  │     ├── Create runtime environment (Python interpreter, libraries)
  │     ├── Execute initialization code (imports, global variables)
  │     └── Total time: 100ms - 5s (depends on runtime, package size)
  ├── Execute handler function with event
  └── Return result
  
  Timeline: |---cold start (200ms)---|---handler execution (500ms)---|
  Total: 700ms

WARM START (Subsequent invocation, environment reused):
  ├── Environment already exists
  ├── Runtime already initialized
  ├── Execute handler function with event
  └── Return result
  
  Timeline: |---handler execution (500ms)---|
  Total: 500ms

ENVIRONMENT LIFECYCLE:
  ├── Created on cold start
  ├── Reused for subsequent invocations (warm)
  ├── Kept alive for ~5-15 minutes after last invocation
  ├── Frozen between invocations (CPU suspended, memory preserved)
  └── Destroyed after idle timeout

Cold start by runtime:

3 → CONCURRENCY AND SCALING DEEP DIVE

TIME: 00:00 — Queue is empty
  Active pollers: 5 (minimum)
  Concurrent executions: 0

TIME: 00:01 — Flash sale starts, 10,000 orders/minute flood the queue
  Queue depth: 10,000 messages
  Active pollers: 5 → detect full queue
  Concurrent executions: 5 (each processing batch of 10)

TIME: 00:02 — Lambda scales up
  Active pollers: 5 + 60 = 65
  Concurrent executions: 50 (reserved concurrency limit)
  Queue depth: 10,000 - 500 (processed) + 10,000 (new) = 19,500

TIME: 00:03 — Steady state at reserved concurrency
  Concurrent executions: 50 (capped)
  Processing rate: 50 × 10 messages × 1 invocation/second = 500 messages/second
  Queue depth: Growing (input > processing)

TIME: 00:10 — Flash sale ends, input rate drops
  Queue depth: 15,000 (backlog)
  Concurrent executions: 50 (still processing backlog)

TIME: 00:30 — Backlog cleared
  Queue depth: 0
  Concurrent executions: Scaling down
  Active pollers: Reducing

TIME: 01:00 — Back to baseline
  Active pollers: 5 (minimum)
  Concurrent executions: 0
  Cost during idle: $0

→ Reserved concurrency (50) acts as both GUARANTEE and CAP:

  • Guarantee: 50 concurrent executions are always available for this function
  • Cap: Never exceeds 50, protecting downstream systems

4 → SQS FIFO + LAMBDA BEHAVIOR

→ For FIFO queues (Project 7), Lambda's behavior is different:

AspectStandard QueueFIFO Queue
Concurrent executionsUp to concurrency limitOne concurrent execution per Message Group ID
Ordering guaranteeNoneMessages in same group processed in order
ScalingBased on queue depthBased on number of unique Message Group IDs
Maximum batch size10,00010

→ FIFO + Lambda:

  • 100 messages across 10 message groups → up to 10 concurrent Lambda invocations
  • 100 messages all in 1 message group → only 1 concurrent Lambda invocation (ordered processing)
  • More message groups = more parallelism

→ Design implication: Use meaningful Message Group IDs that distribute load

  • Good: customer_id as group ID (thousands of unique values = high parallelism)
  • Bad: "all" as group ID (single group = no parallelism)

5 → S3 → LAMBDA INVOCATION INTERNALS

File uploaded to S3
       ↓
S3 detects the upload event
       ↓
S3 checks Event Notification configuration:
  ├── Does prefix match? raw/vendor-products/ ✅
  ├── Does suffix match? .csv ✅
  └── Destination: Lambda function ARN
       ↓
S3 calls Lambda Invoke API (asynchronous invocation)
  ├── Payload: S3 event (bucket, key, size, timestamp)
  ├── Lambda checks resource-based policy: Is S3 allowed to invoke? ✅
  └── Lambda queues the invocation
       ↓
Lambda processes the invocation:
  ├── Provisions execution environment (cold start if needed)
  ├── Invokes handler with S3 event
  └── Handler processes the event
       ↓
If handler fails:
  ├── Lambda retries up to 2 more times (async retry policy)
  ├── If all retries fail → sends to On Failure destination (if configured)
  └── Event is discarded after max retries
       ↓
KEY DIFFERENCE from SQS:
  ├── SQS → Lambda: SYNCHRONOUS (Lambda polls, waits for result)
  ├── S3 → Lambda: ASYNCHRONOUS (S3 fires and forgets)
  ├── SQS: Lambda manages retries via ReceiveCount
  └── S3: Lambda manages retries via async retry policy (max 2)

6 → LAMBDA EXECUTION ENVIRONMENT REUSE

Invocation 1 (Cold Start):
  ├── INIT phase: Import libraries, initialize clients, read config
  │     (This code runs ONCE per cold start)
  │     Example: Create S3 client, create SNS client, parse environment variables
  ├── HANDLER phase: Process the specific event
  │     (This code runs EVERY invocation)
  └── Environment frozen (CPU suspended, memory preserved)

Invocation 2 (Warm Start):
  ├── INIT phase: SKIPPED (already done)
  ├── HANDLER phase: Process the new event
  │     (S3 client, SNS client from Invocation 1 are still in memory!)
  └── Environment frozen again

Invocation 3 (Warm Start):
  ├── Same pattern — reuses initialized resources
  └── ...

After 5-15 minutes of no invocations:
  └── Environment destroyed → next invocation = cold start

→ Best practice: Initialize expensive resources (SDK clients, DB connections) OUTSIDE the handler
→ These resources persist across warm invocations → faster processing
→ The handler should only contain per-invocation logic
