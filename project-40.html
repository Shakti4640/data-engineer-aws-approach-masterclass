The Problems That Have Emerged:

Problem 1 — Manual Role Pair Explosion:
→ Project 39 created ONE cross-account role (HQ → Europe)
→ Now the team needs:

→ That's 11 cross-account role pairs — each requiring trust policy + IAM policy + KMS key policy
→ Managing these individually is error-prone and unscalable

Problem 2 — Different Permission Levels Per Account:
→ The Shared Services account needs DIFFERENT permission levels depending on which account it accesses:

  • In HQ → Deploy Glue jobs, manage S3, but NOT access Redshift directly
  • In Europe → Full infrastructure deployment, but NOT read customer data
  • In US → Full infrastructure deployment, but NOT read PII data
  • In Japan → Read-only monitoring, NO deployment (Japan team manages their own infra)

→ Creating one role per permission level per account = more role explosion

Problem 3 — Session Duration Mismatches:
→ DevOps engineer doing a 5-minute configuration check → needs 15-minute session
→ CI/CD pipeline running 4-hour deployment → needs extended session
→ Overnight batch job running 8 hours → needs maximum session duration
→ All three use the same cross-account role → but need different durations

Problem 4 — Role Chaining for Transitive Access:
→ A CI/CD pipeline in Shared Services needs to:

  1. Read source code configuration from HQ's S3
  2. Deploy the processed artifact to Europe's S3
  3. Trigger Europe's Glue job
    → This requires: Shared Services → HQ role → read S3 → then Shared Services → Europe role → deploy
    → But what if the pipeline needs to access BOTH accounts in the SAME operation?
    → Can the HQ role then assume the Europe role? (Role chaining)

Problem 5 — Audit Confusion:
→ 15 DevOps engineers all assume the same cross-account role
→ CloudTrail shows assumed-role/SharedServicesDeploy/session-name
→ Without meaningful session names → impossible to tell WHICH engineer performed WHICH action
→ Compliance audit requires individual accountability

Problem 6 — Credential Leak Risk:
→ A junior engineer accidentally prints temporary credentials in CI/CD logs
→ Those credentials are valid for 1 hour with full deployment permissions
→ How to minimize blast radius?

The DevOps Lead asks:

"I need a systematic cross-account role architecture that:
handles different permission levels cleanly,
supports both short and long sessions,
provides individual accountability,
minimizes blast radius of credential exposure,
and scales to 20+ accounts without exponential complexity."

→ This requires mastering STS (Security Token Service) in depth — beyond the basics of Project 39

STEP 3 — CONFIGURE SESSION DURATION MANAGEMENT

→ Session duration is controlled at THREE levels — understanding precedence is critical

Duration Hierarchy:

Level 1: ROLE MAXIMUM (set on the role in Account A)
  ↓ caps
Level 2: CALLER REQUEST (DurationSeconds in AssumeRole call)
  ↓ defaults to
Level 3: DEFAULT (1 hour if not specified)

How it works:

Role MaximumCaller RequestsActual DurationWhy
4 hours2 hours2 hoursCaller request within maximum
4 hours8 hours❌ ErrorCaller request exceeds maximum → API returns error
4 hours(not specified)1 hourDefault is 1 hour when not specified
1 hour30 minutes30 minutesCaller request within maximum
15 minutes1 hour❌ ErrorCannot exceed role's maximum

Configuration per tier:

TierRole Maximum Session DurationTypical Caller RequestRationale
Tier 1 — Monitor1 hour (3600s)1 hourQuick observability checks
Tier 2 — DataRead1 hour (3600s)1 hourAnalytics queries
Tier 3 — Deploy4 hours (14400s)2-4 hoursCI/CD pipeline duration
Tier 4 — Emergency15 minutes (900s)15 minutesMinimum blast radius

Setting Maximum Session Duration on a Role:

→ Navigate to IAM → Roles → Select role → "Edit"
→ Scroll to "Maximum session duration"
→ Select from dropdown: 1 hour, 2 hours, 4 hours, 8 hours, 12 hours
→ Default is 1 hour

Duration Limits by STS API:

STS APIMinimumMaximumDefault
AssumeRole15 minutes (900s)12 hours (43200s) — capped by role's maximum1 hour (3600s)
AssumeRole with role chaining15 minutes (900s)1 hour (3600s) — HARD LIMIT1 hour (3600s)
AssumeRoleWithSAML15 minutes (900s)12 hours (43200s)1 hour (3600s)
AssumeRoleWithWebIdentity15 minutes (900s)12 hours (43200s)1 hour (3600s)
GetSessionToken15 minutes (900s)36 hours (129600s)12 hours (43200s)

CRITICAL: Role Chaining Duration Limit:
→ When Role A assumes Role B which then assumes Role C (chaining)
→ The CHAINED session (Role B → Role C) is HARD LIMITED to 1 hour
→ Cannot be extended regardless of role maximum setting
→ This is an AWS security constraint — prevents indefinite credential chains

STEP 4 — IMPLEMENT ROLE CHAINING FOR TRANSITIVE ACCESS

→ This addresses Problem 4 — CI/CD pipeline needing access to multiple accounts

Scenario:
→ CI/CD pipeline in Shared Services needs to:

  1. Read deployment config from HQ's S3
  2. Deploy Lambda function to Europe's account

Without Role Chaining (Recommended Approach):

SharedServices-CICDRole
    │
    ├── AssumeRole → CrossAccount-Shared-Deploy (in HQ Account)
    │       └── Read config from HQ's S3
    │       └── Release credentials
    │
    ├── AssumeRole → CrossAccount-Shared-Deploy (in Europe Account)
    │       └── Deploy Lambda to Europe
    │       └── Release credentials
    │
    (Two SEPARATE AssumeRole calls — NOT chained)

→ This is the PREFERRED approach — each assumption is independent
→ No chaining limitations apply
→ Each session can have full duration (up to role maximum)
→ Clear audit trail — two separate AssumeRole events in CloudTrail

With Role Chaining (When Needed):

SharedServices-CICDRole
    │
    └── AssumeRole → CrossAccount-Shared-Deploy (in HQ Account)
            │
            └── AssumeRole → CrossAccount-HQ-To-Europe-Deploy (in Europe Account)
                    │
                    └── Deploy to Europe using HQ's role as intermediary

→ This creates a chain: Shared → HQ → Europe
→ The SECOND assumption (HQ → Europe) is limited to 1 hour maximum
→ Europe's trust policy must trust HQ's deploy role (not Shared Services directly)

When Role Chaining is Actually Needed:

ScenarioChain Required?Better Alternative
Pipeline needs access to 2 accounts sequentially❌ No — use separate AssumeRole callsTwo independent sessions
Account A can only reach Account C through Account B✅ Yes — if direct trust doesn't existCreate direct trust (preferred)
Temporary delegation — Account A delegates its permissions to Account B✅ Yes — Account B acts as Account ADirect cross-account role (preferred)
Hub-and-spoke model — central account mediates all access⚠️ Possible but limits duration to 1 hourDirect trust from spoke to spoke

→ For ShopNow → Use SEPARATE AssumeRole calls (no chaining)
→ Each destination account trusts the Shared Services account directly
→ No need for transitive trust through HQ

STEP 5 — IMPLEMENT SESSION POLICIES FOR SCOPED-DOWN ACCESS

→ This addresses Problem 6 — minimizing blast radius

What Session Policies Do:
→ When calling AssumeRole, the caller can pass an INLINE session policy
→ This policy FURTHER RESTRICTS the assumed role's permissions for THIS session only
→ The effective permissions are the INTERSECTION of the role policy and the session policy

Effective Permissions = Role IAM Policy ∩ Session Policy

Example — Scoping Down Deploy Role for Specific Deployment:

→ CrossAccount-Shared-Deploy in HQ allows:

  • s3:PutObject on shopnow-deployment-artifacts/*
  • glue:* on all Glue resources
  • lambda:UpdateFunctionCode on specific functions
  • events:* on all EventBridge resources

→ But THIS specific deployment ONLY needs to update one Lambda function
→ Pass a session policy that ONLY allows:

Session Policy:

ActionResource
lambda:UpdateFunctionCodearn:aws:lambda:ap-south-1:111111111111:function:shopnow-orders-processor
lambda:GetFunctionSame function ARN

→ Even though the role allows glue:*, s3:PutObject, events:* → this session CANNOT access them
→ If these temporary credentials are leaked → attacker can ONLY update one specific Lambda function
→ Blast radius: Full role permissions → narrowed to one function

Session Policy Configuration (in AssumeRole call):

ParameterValue
RoleArnarn:aws:iam::111111111111:role/CrossAccount-Shared-Deploy
RoleSessionNamecicd-deploy-orders-processor-20250115
ExternalIdshopnow-shared-deploy-2025
DurationSeconds1800 (30 minutes — only need 30 min for this deployment)
PolicySession policy JSON (inline — max 2048 chars URL-encoded)
PolicyArnsOR reference up to 10 managed policy ARNs

Two Ways to Pass Session Policies:

MethodLimitUse Case
Inline Policy (JSON string)2048 characters (URL-encoded)Simple, one-off restrictions
PolicyArns (managed policy ARNs)Up to 10 managed policies, total 2048 chars packedReusable, standardized restrictions

Session Policy Best Practices:

PracticeWhy
Always scope to specific resourcesarn:aws:lambda:...:function:specific-function not *
Include only necessary actionsDon't include lambda:* if only UpdateFunctionCode is needed
Keep it small2048 char limit — focus on the specific task
Use PolicyArns for repeated patternsCreate managed policies for common scoped-down patterns

STEP 7 — CONFUSED DEPUTY PREVENTION AT SCALE

→ Project 39 introduced External ID for one role pair
→ With 15+ cross-account roles → External ID management becomes important

External ID Management Strategy:

ApproachHowWhen
One External ID per partner accountshopnow-europe-2025 for all Europe rolesSimplest — sufficient for trusted subsidiaries

External ID Rotation Process:

Step 1: Create new External ID: "shopnow-europe-2026"
Step 2: Update trust policy to accept BOTH old and new:
        Condition: sts:ExternalId = ["shopnow-europe-2025", "shopnow-europe-2026"]
Step 3: Update all callers in Account B to use new External ID
Step 4: Verify all callers use new External ID (monitor CloudTrail)
Step 5: Remove old External ID from trust policy:
        Condition: sts:ExternalId = "shopnow-europe-2026"
Step 6: Document new External ID securely

→ Dual-ID window allows zero-downtime rotation
→ Similar to Secrets Manager credential rotation concept (Project 37)

Storing External IDs Securely:

→ For ShopNow → Store External IDs in Secrets Manager in the Shared Services account

STEP 8 — IMPLEMENT SOURCE IDENTITY FOR DEEP TRACEABILITY

→ AWS introduced SourceIdentity — a stronger version of session naming

What SourceIdentity Does:

FeatureRoleSessionNameSourceIdentity
Set by caller✅ Yes✅ Yes
Persists through role chaining❌ No — each hop has its own session name✅ Yes — propagates through ALL chained roles
Immutable once set❌ No — each hop can set a new session name✅ Yes — cannot be changed once set
CloudTrail visibilityIn assumed role ARNIn sourceIdentity field of CloudTrail event

Why SourceIdentity Matters:

Without SourceIdentity:
  Priya assumes Role A (session: "priya-session")
    → Role A assumes Role B (session: "roleA-session")
      → Role B accesses S3
      → CloudTrail shows: assumed-role/RoleB/roleA-session
      → WHO IS "roleA-session"? Is it Priya or David? → UNKNOWN

With SourceIdentity:
  Priya assumes Role A (sourceIdentity: "priya@shopnow.com")
    → Role A assumes Role B (sourceIdentity PROPAGATES automatically)
      → Role B accesses S3
      → CloudTrail shows: sourceIdentity = "priya@shopnow.com"
      → CLEAR: Priya performed this action through role chain

Configuring SourceIdentity:

→ In the AssumeRole call:

ParameterValue
SourceIdentitypriya@shopnow.com (or employee ID)

→ In the trust policy — enforce SourceIdentity must be set:

Condition KeyOperatorValue
sts:SourceIdentityStringLike*@shopnow.com

→ In the role's PERMISSION policy — allow setting SourceIdentity:

ActionResource
sts:SetSourceIdentity*

→ Without this permission in the role's policy → AssumeRole with SourceIdentity fails

SourceIdentity Restrictions:

RestrictionLimit
Minimum length2 characters
Maximum length64 characters
Allowed characters[a-zA-Z0-9=,.@-]
ImmutabilityOnce set → cannot be changed in chained role assumptions
PropagationAutomatically propagates through ALL role chains

STEP 9 — IMPLEMENT EMERGENCY ACCESS BREAK-GLASS PATTERN

→ For situations where normal access paths are broken or insufficient

Break-Glass Architecture:

Normal Operations:
  Engineer → Shared Services → Cross-Account Deploy Role → Deploy
  (Automated, audited, session-policy scoped)

Emergency (Production Down):
  Engineer → MFA Authentication → Shared Services → Emergency Admin Role → Full Access
  (Manual, heavily audited, 15-minute sessions, requires justification)

Break-Glass Role Configuration in Each Account:

→ Already created in Step 2 as CrossAccount-Emergency-Admin (Tier 4)

Additional Safeguards:

Safeguard 1 — EventBridge Alert on Emergency Role Assumption:

→ In each destination account → EventBridge rule (Project 38):

SettingValue
Rule nameshopnow-emergency-access-alert
Event pattern sourceaws.sts (via CloudTrail)
detail-typeAWS API Call via CloudTrail
detail.eventNameAssumeRole
detail.requestParameters.roleArnContains Emergency-Admin
Target 1SNS → shopnow-security-critical-alerts (pages security team)
Target 2CloudWatch Log → /shopnow/security/emergency-access

→ EVERY emergency role assumption triggers an immediate page to the security team
→ This ensures accountability even during emergencies

Safeguard 2 — CloudTrail Insight for Anomaly Detection:

→ Enable CloudTrail Insights in each account
→ Insights automatically detect unusual API activity
→ Emergency admin performing unusual volume of API calls → automatic alert

Safeguard 3 — Session Duration Enforcement:

→ Maximum session: 15 minutes
→ Engineer must re-authenticate with MFA every 15 minutes
→ Each re-authentication creates a new CloudTrail event
→ Creates a detailed timeline of emergency access

Safeguard 4 — Post-Incident Review:

→ After every emergency access:

  • CloudTrail logs reviewed for all actions taken
  • Security team verifies actions were necessary
  • Incident report documents why emergency access was needed
  • Process improvements identified to prevent future emergencies

✅ WHAT SHOULD DO

→ Always use tiered permission model — Monitor, DataRead, Deploy, Emergency — not one-size-fits-all
→ Always set maximum session duration on each role appropriate to its tier
→ Always enforce meaningful RoleSessionName via trust policy conditions
→ Always implement SourceIdentity for role chain traceability
→ Always use session policies to scope down permissions per task
→ Always require MFA for emergency/admin cross-account roles
→ Always store External IDs in Secrets Manager — not in wikis or code
→ Always set up EventBridge alerts for emergency role assumption
→ Always centralize CloudTrail logs in one account for unified audit
→ Always use separate AssumeRole calls instead of role chaining when possible
→ Always rotate External IDs annually with dual-ID transition window
→ Always document which roles exist in which accounts and their permission levels
→ Always use descriptive role names that indicate source account and permission tier

❌ WHAT SHOULD NOT DO

→ Never create a single "god role" with *:* permissions for all cross-account access
→ Never use the same External ID across all roles and all partner accounts
→ Never set maximum session duration to 12 hours for admin roles — keep it minimal
→ Never allow default SDK session names (botocore-session-*) — enforce naming conventions
→ Never skip MFA for emergency admin roles — this is the last line of defense
→ Never use role chaining when direct trust relationships can be established
→ Never forget that chained role sessions are HARD LIMITED to 1 hour
→ Never grant iam:* in cross-account roles — prevents privilege escalation
→ Never grant sts:AssumeRole on * in any account — always specify exact target role ARNs
→ Never allow cross-account roles to create other roles — prevents persistent backdoor creation
→ Never skip centralized CloudTrail — without it, cross-account activity is invisible
→ Never assume session policies EXPAND permissions — they can ONLY restrict
→ Never reuse session names across different engineers or pipelines — destroys accountability

MECHANIC 2 — TEMPORARY CREDENTIAL ANATOMY

What the SessionToken contains (encoded, not human-readable):

Encoded FieldPurpose
Role ARNWhich role was assumed
Session NameWho assumed it
Source IdentityOriginal caller identity (if set)
Expiration TimeWhen credentials become invalid
Session PolicyScoped-down permissions (if provided)
MFA Authentication InfoWhether MFA was used and when
External IDWhat External ID was provided

→ The SessionToken is cryptographically signed by STS
→ Any tampering → signature verification fails → API call rejected
→ The token is opaque — services cannot read its contents directly
→ Only STS can verify and decode it

How AWS Services Validate Temporary Credentials:

API Call arrives at S3 (or any AWS service)
        │
        ▼
Service extracts AccessKeyId + SessionToken from request
        │
        ▼
Service calls STS internal validation:
  1. Is the SessionToken signature valid? → Tamper check
  2. Has the token expired? → Timestamp check
  3. Has the source role been modified/deleted? → Real-time check
  4. Is there an active SCP denial? → Organization check
        │
        ▼
If all valid → extract the effective permissions from:
  Role IAM Policy ∩ Session Policy ∩ Permission Boundary ∩ SCP
        │
        ▼
Evaluate the specific API action against effective permissions

MECHANIC 3 — ROLE CHAINING INTERNAL FLOW

Step 1: SharedServices-CICDRole assumes CrossAccount-Shared-Deploy (in HQ)
        │
        STS returns credentials with:
          sourceIdentity: "cicd-pipeline-orders"
          sessionName: "cicd-orders-build-4521"
          expiration: 4 hours from now (role maximum)
        │
Step 2: Using HQ credentials, assume CrossAccount-HQ-To-Europe (in Europe)
        │

          Caller = assumed-role/CrossAccount-Shared-Deploy/cicd-orders-build-4521
          Target trust policy must allow THIS assumed role ARN
          sourceIdentity: "cicd-pipeline-orders" (PROPAGATED — cannot change)
          
        CRITICAL DIFFERENCE:
          Maximum duration = 1 HOUR (hard limit for chained sessions)
          Even if Europe's role maximum is 4 hours → capped at 1 hour
          
        STS returns NEW credentials with:
          sourceIdentity: "cicd-pipeline-orders" (same — propagated)
          sessionName: (new session name for this hop)
          expiration: 1 hour from now (HARD LIMIT)

Why 1-Hour Hard Limit for Chaining?
→ AWS designed this to prevent indefinite credential extension
→ Without this limit → Chain A → B → C → D → E → each extending duration
→ Could create effectively permanent credentials through chaining
→ 1-hour limit forces re-authentication through the original identity periodically

MECHANIC 4 — EFFECTIVE PERMISSION CALCULATION

→ The most complex but most important mechanic to understand

Effective Permissions = 
    INTERSECTION of:
        ┌── Role IAM Policy (attached to the assumed role)
        ├── Session Policy (passed during AssumeRole — if any)
        ├── Permission Boundary (attached to the assumed role — if any)
        └── SCP (from AWS Organizations — if member account)
    
    MINUS:
        Any EXPLICIT DENY in any of the above

Visual:

│    │   │  │       SCP         │  │   │    │

→ The effective permissions are the INNERMOST overlap
→ Adding a session policy can ONLY make this smaller — never larger
→ This is why session policies are safe — they can ONLY restrict

→ Session policy only allowed s3:GetObject products/* → everything else is implicitly denied

📈 SCALABILITY CONSIDERATIONS

→ Account Count Scaling

  • 2-5 accounts → Individual cross-account roles work fine (this project's approach)
  • 5-20 accounts → Use naming conventions + Infrastructure-as-Code templates
  • 20-50 accounts → Use AWS Organizations with organizational conditions
  • 50-100 accounts → Use AWS Organizations + SCPs + AWS SSO (IAM Identity Center)
  • 100+ accounts → Use AWS Control Tower + Account Factory + centralized governance
  • Project 54 addresses the 20+ account scenario

→ Role Count Scaling

  • IAM roles per account: Default 1,000 (soft limit — can increase to 5,000+)
  • With 4 tiers × 10 source accounts = 40 cross-account roles per destination account
  • Well within limits
  • Use tags for role categorization and automated policy management

→ STS API Scaling

  • AssumeRole: No published rate limit — designed for high-volume use
  • STS endpoints are regional — use regional endpoints for lower latency
  • Global endpoint (sts.amazonaws.com) works but routes to us-east-1
  • Regional endpoint (sts.ap-south-1.amazonaws.com) is faster for local callers

→ Trust Policy Size Scaling

  • Maximum trust policy size: 2048 characters
  • Each principal ARN adds ~70 characters
  • Practical limit: ~20-25 specific role ARNs per trust policy
  • If more → Use account-level trust (root) and control via IAM policies in source account
  • Or use aws:PrincipalOrgID condition to trust entire Organization (Project 54)

→ Credential Caching Scaling

  • AssumeRole returns credentials valid for the requested duration
  • Cache credentials in memory — don't call AssumeRole for every API request
  • Refresh credentials when ~80% of duration has elapsed
  • AWS SDKs handle credential refresh automatically when configured with AssumeRole provider

→ CloudTrail Centralization Scaling

  • Centralized bucket receives logs from all accounts
  • At 50+ accounts → CloudTrail log volume becomes significant (TBs per month)
  • Use S3 lifecycle policies (Project 12) to transition old logs to Glacier
  • Use Athena with partitioned tables for efficient querying
  • Consider CloudTrail Lake for managed query experience
