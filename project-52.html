Partition Key Strategy:

→ When producing to Kinesis → Each event needs a PARTITION KEY
→ The partition key determines WHICH SHARD receives the event
→ Events with the same partition key go to the same shard (ordered within shard)

Partition Key ChoiceDistributionOrderingBest For
transaction_idExcellent (unique per event)No ordering guaranteeMaximum throughput distribution
customer_idGood (many customers)All events for same customer → same shard → orderedCustomer-level ordering needed
merchant_idModerate (fewer merchants than customers)All events for same merchant → same shardMerchant-level analysis
Constant value (e.g., "all")❌ Terrible — all events to one shardPerfect orderingNever — creates hot shard

→ PayScale uses customer_id as partition key
→ Reason: Fraud detection needs to see a customer's events in order (rapid successive transactions = suspicious)
→ This means all events for customer "cust-88421" go to the same shard and are ordered by arrival time

Dynamic Partitioning — The Critical Feature:

→ Without dynamic partitioning → Firehose writes ALL events to ONE S3 prefix
→ Result: One giant file per buffer interval → No partition pruning in Athena

→ With dynamic partitioning → Firehose routes events to S3 prefixes based on event content or timestamp
→ Result: Hive-style partitioned output → Athena partition pruning works perfectly

STEP 6 — CONFIGURE ATHENA TO QUERY STREAMING + BATCH DATA TOGETHER

The Unified Query Problem:

→ Historical batch data (3 years) is in: curated_fraud.transactions_clean
→ Real-time streaming data (last 60 seconds) is in: raw_fraud.transactions_streaming
→ An analyst wants to query: "Show me all transactions for customer X in the last 30 days including the latest ones"
→ This requires querying BOTH tables

Solution — Athena View (UNION ALL):

CREATE VIEW unified_fraud.all_transactions AS
SELECT 
    transaction_id, customer_id, amount, currency,
    merchant_id, merchant_name, city, country,
    status, event_timestamp,
    'batch' as data_source
FROM curated_fraud.transactions_clean

UNION ALL

SELECT 
    transaction_id, customer_id, amount, currency,
    merchant_id, merchant_name, city, country,
    status, event_timestamp,
    'streaming' as data_source
FROM raw_fraud.transactions_streaming
WHERE year = CAST(YEAR(CURRENT_DATE) AS VARCHAR)
    AND month = LPAD(CAST(MONTH(CURRENT_DATE) AS VARCHAR), 2, '0')
    AND day = LPAD(CAST(DAY(CURRENT_DATE) AS VARCHAR), 2, '0');

How the View Works:

Analyst queries:
    SELECT * FROM unified_fraud.all_transactions
    WHERE customer_id = 'cust-88421'
    AND event_timestamp > DATE_ADD('day', -30, CURRENT_TIMESTAMP)
    ORDER BY event_timestamp DESC;

Athena expands the view:
    ├── Scans curated_fraud.transactions_clean (last 30 days partitions)
    │   → Returns batch records from yesterday and earlier
    │
    └── Scans raw_fraud.transactions_streaming (today's partitions)
        → Returns streaming records from the last few hours
    
    UNION ALL combines both result sets
    ORDER BY event_timestamp sorts them together
    
Result: Seamless timeline from 30 days ago to 60 seconds ago

→ The data_source column tells the analyst whether each row came from batch or streaming
→ This is useful for debugging and understanding data freshness

Deduplication in the View:

→ Some transactions may exist in BOTH batch (yesterday's ETL) and streaming (today's Firehose)
→ Overlap period: Events from the last batch run to the current streaming window

Enhanced View with Deduplication:
CREATE VIEW unified_fraud.all_transactions_deduped AS
SELECT * FROM (
    SELECT *,
        ROW_NUMBER() OVER (
            PARTITION BY transaction_id 
            ORDER BY 
                CASE data_source 
                    WHEN 'batch' THEN 1      -- Prefer batch (cleaned/enriched)
                    WHEN 'streaming' THEN 2   -- Streaming as fallback
                END
        ) as row_num
    FROM unified_fraud.all_transactions
)
WHERE row_num = 1;

→ If a transaction exists in both batch and streaming → Batch version wins (it's been cleaned and enriched)
→ If a transaction exists only in streaming → Streaming version is used (it's the only available)

✅ WHAT SHOULD DO

→ Always use On-demand capacity mode unless traffic patterns are perfectly predictable
→ Always use a meaningful partition key (customer_id for fraud use cases) — never a constant value
→ Always enable format conversion in Firehose (JSON → Parquet) — reduces downstream query costs by 80%+
→ Always enable dynamic partitioning in Firehose — enables Athena partition pruning
→ Always add hour-level partitioning for streaming data — day-level is too coarse for real-time queries
→ Always create a Glue table BEFORE configuring Firehose format conversion — Firehose needs the schema
→ Always configure buffer interval based on latency requirements — 60s for near-real-time, 300s for cost optimization
→ Always monitor IteratorAgeMilliseconds — it tells you if consumers are falling behind
→ Always implement both analytics (Firehose → S3) and alerting (Lambda) consumers for fraud detection
→ Always use Enhanced Fan-Out for latency-sensitive Lambda consumers
→ Always create a unified view (UNION ALL) for querying batch + streaming data together
→ Always handle deduplication when batch and streaming data overlap
→ Always encrypt the Kinesis stream with KMS for PCI-DSS compliance
→ Always set maximum record age on Lambda triggers — prevents reprocessing very old events

❌ WHAT SHOULD NOT DO

→ Never write one S3 object per event — creates millions of tiny files → Athena performance nightmare
→ Never use a constant partition key — all events go to one shard → Hot shard → Throughput exceeded
→ Never skip Parquet conversion — JSON in S3 costs 10x more to query with Athena
→ Never set buffer interval to 900 seconds if real-time freshness is required — that's 15 minutes stale
→ Never forget the Glue table for Firehose format conversion — Firehose fails without schema reference
→ Never process streaming data through the full Curated ETL pipeline synchronously — adds latency
→ Never ignore IteratorAge metrics — growing IteratorAge means data is getting staler over time
→ Never use provisioned capacity without monitoring — under-provisioned shards drop events
→ Never hardcode shard count for variable workloads — use On-demand or auto-scaling
→ Never allow Lambda consumer to fall behind without alerting — fraud detection loses value if delayed
→ Never skip the DLQ for Lambda Kinesis triggers — failed events are silently lost without DLQ
→ Never query the streaming table without partition filters — full scan of 525,600 files/year is extremely expensive

1 → KINESIS DATA STREAM INTERNAL ARCHITECTURE

Kinesis Data Stream: "fraud-transactions-stream"
    |
    ├── Shard 0: Handles partition keys hashing to range [0, 170141183460469...]
    │   ├── Record 1 (customer_id: cust-001 → hash → falls in shard 0)
    │   ├── Record 2 (customer_id: cust-005 → hash → falls in shard 0)
    │   └── Record 3 ...
    │   Capacity: 1 MB/s write, 2 MB/s read
    │   
    ├── Shard 1: Handles partition keys hashing to range [170141183460470..., 340282366920938...]
    │   ├── Record 4 (customer_id: cust-088 → hash → falls in shard 1)
    │   └── Record 5 ...
    │   Capacity: 1 MB/s write, 2 MB/s read
    │
    └── (On-demand: Shards auto-scale based on traffic)

How Partition Key Determines Shard:

Producer sends event with partition_key = "cust-88421"
    |
    ↓
Kinesis computes MD5 hash: MD5("cust-88421") = 0x7A3F...
    |
    ↓
Hash maps to a HASH KEY RANGE
    |
    ↓
Each shard owns a CONTIGUOUS range of hash keys
    Shard 0: [0x0000... to 0x7FFF...]
    Shard 1: [0x8000... to 0xFFFF...]
    |
    ↓
0x7A3F... falls in Shard 0's range → Event goes to Shard 0

→ All events for cust-88421 ALWAYS go to the same shard
→ Within that shard → Events are ORDERED by arrival time
→ Across shards → NO ordering guarantee
→ This is why partition key choice matters (from Step 2)

Data Retention Mechanics:

Event arrives at Shard 0 at Time T
    |
    ↓
Event stored in Shard 0's buffer
    |
    ├── Retention = 24 hours → Event available until T + 24h
    │   → After 24h → Event automatically deleted
    │   → If consumer hasn't read it → Event is LOST
    |
    └── Consumer reads event at Time T + 5 seconds
        → Event still available for other consumers until T + 24h
        → Multiple consumers can read the SAME event independently

→ Kinesis is a RETENTION-BASED log — not a queue
→ Unlike SQS (from Project 7) where reading REMOVES the message
→ Kinesis events REMAIN in the stream until retention expires
→ Multiple consumers read independently at their own pace

Firehose reads from Kinesis Data Stream
    |
    ↓
Internal Buffer (managed by Firehose):
    ├── Buffer accumulates events
    ├── Buffer condition check (every second):
    │   ├── Buffer size >= 64 MB? → FLUSH
    │   ├── Buffer time >= 60 seconds? → FLUSH
    │   └── Neither? → Continue accumulating
    |
    ↓ FLUSH triggered
    |
    ↓
Format Conversion (if enabled):
    ├── Read Glue table schema
    ├── Parse JSON records
    ├── Map JSON fields to Parquet columns
    ├── Apply Snappy compression
    └── Generate Parquet file in memory
    |
    ↓
Dynamic Partitioning (if enabled):
    ├── Extract partition keys from each record (JQ expressions)
    ├── Group records by partition key values
    ├── Route each group to appropriate S3 prefix
    └── Write separate Parquet files per partition
    |
    ↓
S3 Delivery:
    ├── PutObject to each partition prefix
    ├── Retry on failure (built-in exponential backoff)
    ├── If persistent failure → Write to error prefix
    └── Update delivery metrics

GuaranteeValue

    |

    │       ├── part-00002.snappy.parquet

            │   ├── micro-batch-14-15.snappy.parquet

→ Both batch and streaming data use the SAME partitioning scheme
→ Athena queries seamlessly span both — partition pruning works identically
→ The only difference: batch files are larger (optimized), streaming files are smaller (frequent writes)

File Compaction (Important for Long-Term Performance):

→ Over time → Streaming creates many small files (1440/day × 365 days = 525,600 files)
→ Many small files degrade Athena performance (each file = S3 GET request)
→ Solution: Nightly compaction job

Nightly Compaction (Glue ETL):
    Input: yesterday's streaming micro-batch files (1440 small Parquet files)
    Process: Read all → Merge → Write optimized large Parquet files
    Output: 3-5 large Parquet files (100-500 MB each)
    Cleanup: Delete the 1440 small files

→ Compaction runs AFTER the nightly batch ETL (which also processes yesterday's data from MariaDB)
→ After compaction → Yesterday's data exists as optimized large files
→ Today's data exists as streaming micro-batch files (small but fresh)
→ The table always has: Large files (historical) + Small files (today) — both queryable

4 → HOW MULTIPLE CONSUMERS READ FROM THE SAME STREAM

Kinesis Data Stream: fraud-transactions-stream
    |
    Shard 0: [Event1, Event2, Event3, Event4, Event5, ...]
    |
    ├── Consumer 1: Firehose
    │   Iterator position: After Event3 (has read 1-3, processing 4-5)
    │   Reading at: 1 MB/s (shared throughput)
    │
    ├── Consumer 2: Lambda (standard)
    │   Iterator position: After Event5 (has read 1-5, caught up)
    │   Reading at: 1 MB/s (shared throughput)
    │
    └── Consumer 3: Lambda (Enhanced Fan-Out)
        Iterator position: After Event5 (has read 1-5, caught up)
        Reading at: 2 MB/s (DEDICATED throughput — not shared)

Standard vs Enhanced Fan-Out:

Standard (Polling):
    Consumer polls GetRecords every 200ms
    All standard consumers SHARE 2 MB/s per shard
    2 standard consumers = 1 MB/s each
    5 standard consumers = 400 KB/s each → May cause IteratorAge growth

Enhanced Fan-Out (Push):
    Consumer subscribes via SubscribeToShard
    Kinesis PUSHES records to consumer
    Each EFO consumer gets DEDICATED 2 MB/s per shard
    5 EFO consumers = 2 MB/s each (no sharing)
    Latency: ~70ms vs ~200ms for standard

→ Firehose always uses standard consumer (built-in)
→ Lambda can use either standard or Enhanced Fan-Out
→ For PayScale: Firehose (standard) + Lambda fraud alerting (Enhanced Fan-Out)
