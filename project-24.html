PROJECT 24: THE OPERATIONAL DATABASE THAT NEEDED A BACKUP IN S3 — MariaDB to S3 Export via Glue ETL

🔴 HYPOTHETICAL PROBLEM

→ ShopNow's operational stack runs on EC2 instances (Project 3) connected to an RDS MariaDB database (Project 9)
→ The MariaDB database is the single source of truth for live business operations:

TableRecordsGrowth RatePurpose
customers2.5 million5,000/dayCustomer profiles, addresses, preferences
orders15 million50,000/dayOrder transactions, statuses, amounts
order_items45 million150,000/dayLine items per order
products50,000200/weekProduct catalog, pricing, inventory
inventory_movements80 million300,000/dayStock in/out, warehouse transfers
payments12 million40,000/dayPayment records, refunds

→ Problem 1: Analytics Team Cannot Access Operational Data

  • The data lake (S3) has clickstream and historical order snapshots
  • But LIVE customer data, real-time inventory levels, and payment records are ONLY in MariaDB
  • Analysts want to join clickstream behavior with customer profiles and purchase history
  • They cannot query MariaDB directly — no credentials, no network access, no SQL knowledge

→ Problem 2: Production Database Performance Degradation

  • A junior analyst was given read-only MariaDB credentials for a "quick report"
  • They ran a full table scan on order_items (45M rows) with a complex JOIN to products
  • The query consumed 80% of the RDS instance's CPU for 12 minutes
  • During that time — the e-commerce application experienced 3-second response times
  • 200 customers abandoned their shopping carts
  • The CTO banned ALL direct analyst access to the production database

→ Problem 3: No Historical State Tracking

  • MariaDB stores CURRENT state only — when a customer updates their address, the old address is overwritten
  • When a product price changes, the old price is gone
  • When an order status changes from "processing" to "shipped," the processing timestamp may be overwritten
  • The analytics team needs historical state for trend analysis
  • "What was the price of product X on January 1st?" — MariaDB cannot answer this

→ Problem 4: Disaster Recovery Gap

  • RDS automated backups retain snapshots for 35 days maximum
  • Compliance requires 7 years of transaction records
  • If MariaDB data is corrupted or accidentally deleted — recovery is limited to 35 days
  • Exporting to S3 creates a long-term, immutable archive

→ Problem 5: Data Warehouse Gap

  • Redshift (Project 16) has historical data loaded from S3 (Project 22)
  • But Redshift doesn't have the LATEST operational data — it's always 24+ hours behind
  • The daily pipeline is: S3 raw files → Glue ETL (Project 21) → S3 transformed → COPY (Project 22) → Redshift
  • MariaDB's live data never enters this pipeline

→ The data engineering lead says: "We need to extract data FROM MariaDB, transform it, and land it in S3 as Parquet — without touching the production database during peak hours, and without giving analysts database access"

→ Glue must CONNECT to MariaDB over the network — this requires:

  1. Network path from Glue to RDS (VPC configuration)
  2. Authentication credentials (username/password)
  3. JDBC driver (MariaDB/MySQL compatible)
  4. Proper security group rules (firewall)

→ The approach should be:

  • Step 1 — Understand Glue's VPC networking requirements for JDBC
  • Step 2 — Create a Glue Connection to MariaDB
  • Step 3 — Configure security groups for Glue ↔ RDS communication
  • Step 4 — Test the connection
  • Step 5 — Create Glue Crawler with JDBC source (catalog the database)
  • Step 6 — Create Glue ETL Job with JDBC source and S3 target
  • Step 7 — Optimize extraction (parallel reads, push-down predicates)
  • Step 8 — Schedule during off-peak hours
  • Step 9 — Integrate output with existing data lake pipeline

🟢 HOW CAN WE SOLVE IT — Step-by-Step Configuration

STEP 1 — UNDERSTAND GLUE VPC NETWORKING FOR JDBC

→ This is the most complex and most commonly misconfigured part of Glue JDBC jobs
→ Understanding this thoroughly prevents hours of debugging

The problem:
→ Glue is a MANAGED service — you don't control where Glue workers run
→ RDS MariaDB sits in a PRIVATE subnet (Project 5, Project 9) — not accessible from the internet
→ How does Glue reach a private database?

The solution: Glue Connection with VPC Configuration

→ When you create a Glue Connection, you specify:

  • VPC ID
  • Subnet ID
  • Security Group ID

→ Glue then creates Elastic Network Interfaces (ENIs) in YOUR VPC, in YOUR subnet
→ These ENIs give Glue workers a private IP address in your VPC
→ From this private IP, Glue can reach the RDS instance on port 3306

┌─────────────────────────────────────────────────────────────┐
│                        YOUR VPC                              │
│                                                              │
│  ┌──────────────────────┐    ┌───────────────────────────┐  │
│  │   PRIVATE SUBNET A    │    │   PRIVATE SUBNET B        │  │
│  │                        │    │                           │  │
│  │  ┌──────────────────┐ │    │  ┌──────────────────────┐ │  │
│  │  │  RDS MariaDB      │ │    │  │  Glue ENI            │ │  │
│  │  │  (Primary)         │ │    │  │  (created by Glue)   │ │  │
│  │  │  10.0.1.50         │ │    │  │  10.0.2.100          │ │  │
│  │  │  Port 3306         │◀┼────┼──│  Connects via JDBC   │ │  │
│  │  └──────────────────┘ │    │  └──────────────────────┘ │  │
│  │                        │    │                           │  │
│  └──────────────────────┘    └───────────────────────────┘  │
│                                                              │
└─────────────────────────────────────────────────────────────┘

→ Key insight: Glue workers run in AWS-managed infrastructure OUTSIDE your VPC
→ But the ENI bridges them INTO your VPC
→ The ENI gets a private IP from your subnet's CIDR range
→ Traffic flows: Glue worker → ENI (in your VPC) → Private network → RDS

Critical networking requirements:

RequirementWhyConsequence if Missing
Subnet must have available IP addressesGlue creates ENIs that need IPsConnection fails — "unable to create ENI"
Subnet should be PRIVATE (no internet gateway route)Security best practiceNot required but strongly recommended
Security group must allow OUTBOUND to RDSGlue ENI → RDS communicationConnection timeout
RDS security group must allow INBOUND from Glue SGRDS must accept Glue's connectionConnection refused
Subnet must have NAT Gateway OR S3 VPC EndpointGlue needs to write to S3 after extractionJob runs but FAILS on S3 write

→ That last point is critical and frequently missed:

  • Glue ENI is in a PRIVATE subnet
  • Glue needs to write Parquet files to S3
  • S3 is a public service — requires internet access OR VPC endpoint
  • Without NAT Gateway or S3 VPC endpoint → Glue can read from MariaDB but CANNOT write to S3

Two solutions for S3 access from Glue ENI:

SolutionMechanismCostRecommended
NAT Gateway (Project 5)Routes S3 traffic through NAT → Internet Gateway → S3~$32/month + data transferWorks but more expensive
S3 VPC Gateway Endpoint (Project 27 concept)Direct private path to S3 — no internetFree✅ Preferred

→ If NAT Gateway already exists from Project 5 → Glue will use it automatically
→ If not → create an S3 VPC Gateway Endpoint (free, private, more secure)

→ S3 source: Each worker reads a separate FILE → more workers = more parallel file reads → faster
→ JDBC source: ALL workers read from the SAME database → more workers = more concurrent database connections

STEP 8 — OPTIMIZE JDBC EXTRACTION

→ This is where JDBC extraction differs significantly from S3 file reading
→ By default, Glue reads the ENTIRE table through a single JDBC connection — no parallelism

Default behavior (BAD for large tables):

Glue Worker 0: SELECT * FROM orders    ← Single connection, reads ALL 15M rows
Glue Worker 1: Idle
Glue Worker 2: Idle
Glue Worker 3: Idle
Glue Worker 4: Idle

→ 4 out of 5 workers do nothing during extraction → waste of resources
→ 15M rows through single connection → 30-45 minutes

Optimized behavior (Partitioned JDBC Read):

→ Configure "hash partitioning" or "range partitioning" on the source

Option A: Hash Partitioning (Glue Native)

SettingValueExplanation
Enable partitioningYesSplit table read across workers
Partition columnorder_idColumn to partition by (must be numeric)
Number of partitions5Match to worker count
Lower bound1Minimum value of partition column
Upper bound15000000Maximum value (approximate)

→ Glue generates 5 parallel queries:

Worker 0: SELECT * FROM orders WHERE order_id >= 1 AND order_id < 3000001
Worker 1: SELECT * FROM orders WHERE order_id >= 3000001 AND order_id < 6000001
Worker 2: SELECT * FROM orders WHERE order_id >= 6000001 AND order_id < 9000001
Worker 3: SELECT * FROM orders WHERE order_id >= 9000001 AND order_id < 12000001
Worker 4: SELECT * FROM orders WHERE order_id >= 12000001 AND order_id <= 15000000

→ 5 parallel reads → 5x faster extraction → ~6-9 minutes instead of 30-45

Partition column selection:

Good Partition ColumnWhyBad Partition ColumnWhy
Primary key (auto-increment)Evenly distributed, indexedstatus (3 values)Extreme skew — one partition gets 90% of data
Sequential IDGood range distributioncountry (string)Not numeric — can't range-partition
Date as integer (epoch)Good for time-range partitioningboolean_flagOnly 2 values — only 2 partitions

→ order_id (auto-increment INT) is ideal — evenly distributed, indexed, numeric

Option B: Predicate Push-Down

→ Instead of extracting the ENTIRE table, push a WHERE clause to the database

SettingValuePurpose
Additional SQL conditionorder_date >= '2025-01-01'Only extract recent data

→ Glue translates this into the JDBC query:
SELECT * FROM orders WHERE order_date >= '2025-01-01'
→ MariaDB filters at the database level → less data transferred → faster extraction
→ Combine with partitioning for maximum optimization

Option C: Custom SQL Query

→ Instead of reading a table, specify a custom SELECT:

SettingValue
Source typeJDBC (direct, not Catalog)
Custom SQLSELECT o.*, c.customer_name, c.email FROM orders o JOIN customers c ON o.customer_id = c.customer_id WHERE o.order_date >= '2025-01-01'

→ This performs the JOIN in MariaDB (where indexes exist) rather than in Glue Spark
→ Reduces data transfer — only needed columns and rows are extracted
→ BUT: Cannot combine custom SQL with Glue's hash partitioning
→ Trade-off: Join efficiency vs extraction parallelism

STEP 9 — IMPACT MITIGATION ON PRODUCTION DATABASE

→ This is unique to JDBC extraction — S3 sources (Project 21) have no such concern

Strategy 1: Use Read Replica Instead of Primary

→ From Project 29 (Read Replica concept):

  • RDS supports read replicas — asynchronous copies of the primary database
  • Point Glue at the READ REPLICA endpoint instead of the primary
  • Extraction load goes to the replica → primary is unaffected
  • Application performance is completely isolated
SettingChange
JDBC URL (original)jdbc:mysql://shopnow-prod-db.xxxxxxx.rds.amazonaws.com:3306/shopnow_db
JDBC URL (replica)jdbc:mysql://shopnow-prod-db-replica.xxxxxxx.rds.amazonaws.com:3306/shopnow_db

→ This is the #1 recommended approach for production database extraction
→ If no replica exists → create one specifically for analytics extraction (Project 29)

Strategy 2: Schedule During Off-Peak Hours

Time PeriodDatabase LoadExtraction Safety
02:00 - 05:00 UTCLowest traffic✅ Safest window
09:00 - 11:00 UTCPeak morning traffic❌ Avoid
12:00 - 14:00 UTCModerate (lunch dip)⚠️ Possible with small tables
18:00 - 22:00 UTCEvening shopping peak❌ Avoid

→ Schedule Glue job at 02:00 UTC → completes by 03:30 UTC → before peak hours

STEP 11 — FULL EXTRACTION VS INCREMENTAL EXTRACTION

→ This is a critical architectural decision for each table

StrategyMechanismData VolumeUse Case
Full extractionExtract ALL rows every runEntire table every timeSmall tables, tables with complex updates
Incremental extractionExtract only NEW/CHANGED rows since last runOnly deltaLarge tables, append-only or timestamp-tracked

Full Extraction:

→ Simple: Extract entire table → overwrite target S3 path
→ Advantage: Always complete, always consistent
→ Disadvantage: Wasteful for large tables (80M rows extracted daily when only 300K are new)

→ Appropriate for:

TableRowsFull Extract TimeRationale
products50K30 secondsTiny — full extract is trivial
customers2.5M3 minutesModerate — updates happen anywhere in the table

Incremental Extraction:

→ Complex: Extract only rows where updated_at >= last_extract_timestamp
→ Advantage: 95%+ less data extracted, faster, less database impact
→ Disadvantage: Requires reliable timestamp column, merge logic for updates

→ Appropriate for:

TableRowsNew Rows/DayIncremental Extract TimeRationale
orders15M50K30 secondsHas created_at and updated_at timestamps
order_items45M150K45 secondsHas created_at timestamp
inventory_movements80M300K60 secondsAppend-only (movements are never updated)
payments12M40K20 secondsHas created_at and updated_at

How Glue Job Bookmarks Enable Incremental JDBC Extraction:

→ Job Bookmark (introduced in Project 21, deep dive in Project 46) tracks extraction state
→ For JDBC sources: Bookmark stores the last extracted value of a key column

RunBookmark StateExtracted Range
Run 1 (first run)No bookmark → extract allorder_id 1 to 15,000,000
Run 2Bookmark: order_id = 15,000,000order_id 15,000,001 to 15,050,000
Run 3Bookmark: order_id = 15,050,000order_id 15,050,001 to 15,100,000

→ Each run extracts ONLY new records → fast, minimal database impact
→ Bookmark is stored by Glue automatically — no manual tracking needed

Limitation: Job Bookmarks work on monotonically increasing columns (auto-increment IDs, timestamps)
→ If records are UPDATED (not just inserted) → bookmark misses updates to existing rows
→ For tables with updates → use updated_at timestamp approach with custom SQL:
WHERE updated_at >= '{last_extract_time}'

✅ WHAT SHOULD DO

→ Always create a DEDICATED read-only database user for Glue — never use admin credentials
→ Always use a Read Replica for extraction — protect production database performance
→ Always configure the self-referencing security group rule on Glue SG — without it, Spark workers can't communicate
→ Always test the Glue Connection BEFORE creating jobs — connection issues are the #1 failure mode
→ Always use hash partitioning for large tables (> 1M rows) — single-connection extraction is too slow
→ Always schedule extraction during off-peak hours — minimize production impact
→ Always add extract_date metadata column — enables historical state tracking
→ Always monitor RDS CloudWatch metrics during extraction — detect performance impact immediately
→ Always ensure the Glue ENI subnet has S3 access (NAT Gateway or VPC Endpoint) — otherwise output write fails
→ Always use incremental extraction for large, append-heavy tables — reduces extraction time by 95%+
→ Always separate MariaDB-sourced data from file-sourced data in S3 — clear lineage
→ Always use Secrets Manager for database credentials — never plaintext in Glue Connection (Project 37)
→ Always set fetchSize explicitly (1000-10000) — default behavior varies and may cause memory issues

❌ WHAT SHOULD NOT DO

→ Never point Glue directly at the production primary database for large extractions — use Read Replica
→ Never use the database admin/root user for Glue — blast radius is too large
→ Never forget the self-referencing security group rule — most common Glue JDBC failure
→ Never set partition count > database max connections — overwhelms the database
→ Never extract entire 80M row table daily when only 300K rows are new — wasteful and slow
→ Never store database passwords in plaintext in Glue Connection — use Secrets Manager
→ Never run extraction during peak business hours without Read Replica — risks application performance
→ Never forget S3 access from Glue ENI subnet — job succeeds on extract but fails on write
→ Never use fetchSize = 0 for large tables — may attempt to load entire result set into memory
→ Never extract sensitive data (SSN, credit cards) without masking/encrypting — compliance violation
→ Never skip the connection test — debugging a failed job is 10x harder than testing the connection first
→ Never use the same Glue Connection for dev and prod databases — create separate connections with clear naming

2 → JDBC FETCHSIZE MECHANICS

Worker opens JDBC connection
       ↓
Worker sends: SELECT * FROM orders WHERE order_id >= 1 AND order_id < 3000001
       ↓
MariaDB begins executing query
       ↓
fetchSize = 1000:
  ├── Worker requests first 1000 rows
  ├── MariaDB sends 1000 rows over network
  ├── Worker processes 1000 rows in Spark
  ├── Worker requests next 1000 rows
  ├── MariaDB sends next 1000 rows
  └── ... repeats until all 3M rows fetched
       ↓
fetchSize = 0 (DEFAULT — DANGEROUS):
  ├── MariaDB sends ALL 3M rows at once
  ├── Worker must buffer ALL rows in memory
  └── If rows exceed worker memory → OutOfMemoryError → Job fails

fetchSize impact:

fetchSizeMemory UsageNetwork Roundtrips (for 3M rows)Speed
0 (all at once)Entire result in memory1Fastest (if memory sufficient)
100Low30,000Slow (too many roundtrips)
1,000Moderate3,000Good balance
10,000Higher300Faster, more memory
100,000High30Very fast, may cause memory pressure

→ Recommended: 1,000 - 10,000 depending on row size
→ Wide rows (many columns, large text fields) → use smaller fetchSize
→ Narrow rows (few columns, numeric) → use larger fetchSize

4 → JOB BOOKMARK FOR JDBC SOURCES

FIRST RUN (no bookmark exists):
  ├── Glue reads entire table: SELECT * FROM orders
  ├── Processes all 15M rows
  ├── After successful completion:
  │     Bookmark saved: {table: "orders", column: "order_id", value: 15000000}
  └── Next run will start from order_id > 15000000

SECOND RUN (bookmark exists):
  ├── Glue reads bookmark: {column: "order_id", value: 15000000}
  ├── Generates query: SELECT * FROM orders WHERE order_id > 15000000
  ├── Only new rows (50,000) are extracted
  ├── After successful completion:
  │     Bookmark updated: {column: "order_id", value: 15050000}
  └── Next run will start from order_id > 15050000

FAILED RUN (bookmark NOT updated):
  ├── Job fails mid-extraction
  ├── Bookmark remains at previous value: 15000000
  ├── Next run re-extracts from order_id > 15000000
  └── No data loss — failed run's partial data may create duplicates (handled by dedup)

→ Bookmark column must be:

  • Monotonically increasing (new records have higher values)
  • Never updated on existing records (otherwise bookmark misses updates)
  • Typically: auto-increment primary key or created_at timestamp

→ For tables with UPDATES (not just inserts):

  • Bookmark on order_id misses updated rows (their order_id didn't change)
  • Solution: Use updated_at column — but requires custom extraction logic
  • Or: Full extraction for tables with frequent updates (customers, products)
