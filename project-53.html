Namespace — The "What" (Data):
→ Contains your databases, schemas, tables, and data
→ Think of it as the "persistent identity" of your warehouse
→ Survives workgroup changes — data is never lost
→ Managed storage — scales automatically, no capacity planning
→ Encrypted with KMS key (from Project 36)

Workgroup — The "How" (Compute):
→ Contains the compute resources (RPUs) that execute queries
→ Think of it as the "engine" that processes your SQL
→ Scales up and down automatically based on query demand
→ Has network configuration (VPC, subnets, security groups)
→ Has the endpoint that applications connect to

RPU (Redshift Processing Unit):
→ The unit of compute in Serverless
→ Measured in RPU-hours — you pay per RPU per second of usage
→ Minimum: 8 RPUs
→ Maximum: 512 RPUs (default limit — can request increase)
→ Scales in increments of 8 RPUs (8, 16, 24, 32, ... 512)

RPU CountApproximate EquivalentUse Case
8 RPUs (minimum)~2 dc2.large nodesLight queries, small datasets
32 RPUs~4 dc2.large nodesMedium workload (PayScale's current)
128 RPUs~16 ra3.xlplus nodesHeavy analytics, large concurrent queries
512 RPUs~64 ra3.xlplus nodesEnterprise-scale, massive concurrency

How RPU Auto-Scaling Works:

No queries running → 0 RPUs → $0 cost (idle)
    |
    ↓ Analyst submits query
    |
Serverless provisions base RPUs (e.g., 32 RPUs) → Query starts
    |
    ↓ Query is complex, needs more compute
    |
Serverless auto-scales to 64 RPUs → Query completes faster
    |
    ↓ Query finishes, no more queries
    |
RPUs scale down → After idle timeout → 0 RPUs → $0 cost

→ Scale-up happens in SECONDS — no resize operation, no downtime
→ Scale-down happens after a configurable idle period
→ You are billed per RPU-SECOND — granular billing

STEP 4 — CONFIGURE RPU SCALING AND USAGE LIMITS

RPU Configuration:

SettingValueWhy
Base RPU capacity32 RPUsStarting capacity — matches current cluster equivalent
Maximum RPU capacity128 RPUsAllow scaling up during audit periods

How Base and Maximum RPU Work:

Base RPU = 32:
    → When queries start → Serverless provisions 32 RPUs
    → If 32 RPUs are sufficient → Stays at 32

    → If query demand exceeds 32 RPU capacity:
        → Serverless auto-scales UP (to 64, 96, 128...)
        → Scaling happens in seconds, no downtime
        → Maximum: 128 RPUs (configured limit)

    → If queries finish and no new queries arrive:
        → Serverless scales DOWN after idle timeout
        → Eventually reaches 0 RPUs (no cost)

Usage Limits (Critical for Cost Control):

→ Navigate to Workgroup → Limits tab → Create usage limit

Limit 1: Daily RPU-Hours Cap

Limit 3: Monthly Cost Cap (Safety Net)

SettingValueWhy

Amount15000 RPU-hours~$5,400/month maximum (absolute ceiling)

Why Usage Limits Are Critical:

→ Without limits → A single runaway query or misconfigured dashboard can consume unlimited RPUs
→ Example nightmare scenario:

  • Dashboard configured to refresh every 10 seconds
  • Each refresh runs a full table scan (no partition filter)
  • 128 RPUs × 24 hours × 30 days = 92,160 RPU-hours = $33,177/month
  • vs expected $150/month

→ Usage limits prevent this — queries are stopped when limits are reached
→ Always configure at LEAST a monthly RPU-hour limit

Idle Timeout Configuration:

→ Not directly configurable in Serverless (AWS manages it)
→ Serverless automatically scales to zero after a period of inactivity
→ Typical idle timeout: ~5-10 minutes of no queries
→ First query after idle: "Cold start" of ~10-30 seconds (RPU provisioning)

    w.lifetime_value,

Three-Source Query in Serverless (Local + Spectrum + Federated):

SELECT 
    w.customer_id,
    w.lifetime_risk_score,                    -- Local Serverless table
    s.daily_transaction_count,                 -- S3 data lake via Spectrum
    m.current_status,                          -- Live MariaDB via Federation
    m.last_login
FROM fraud_schema.customer_risk_profile w      -- LOCAL
JOIN spectrum_fraud.daily_aggregates s         -- SPECTRUM (S3)
    ON w.customer_id = s.customer_id
    AND s.year = '2025' AND s.month = '01'
JOIN mariadb_live.customers m                  -- FEDERATED (MariaDB)
    ON w.customer_id = m.customer_id
WHERE w.lifetime_risk_score > 0.8
ORDER BY w.lifetime_risk_score DESC
LIMIT 50;

→ This single query accesses THREE different data stores simultaneously
→ Serverless auto-scales RPUs to handle the join complexity
→ No provisioning, no capacity planning — it just works

✅ WHAT SHOULD DO

→ Always start with the MINIMUM base RPU (8) and let auto-scaling handle peaks
→ Always configure usage limits (daily, weekly, monthly) — runaway queries can cost thousands
→ Always use 3 subnets in 3 AZs for Serverless workgroup — it's a hard requirement
→ Always enable Enhanced VPC Routing for Spectrum and Federated Query access
→ Always migrate via snapshot restore for simplest path — data integrity is preserved
→ Always run provisioned and Serverless in parallel for 2+ weeks before decommissioning
→ Always update ALL connection strings before decommissioning provisioned cluster
→ Always keep the final provisioned cluster snapshot for 90+ days — rollback safety net
→ Always use Spectrum for historical data (> 90 days) — keep only recent data locally
→ Always monitor RPU consumption patterns for the first month — adjust base RPU accordingly
→ Always tag the workgroup and namespace with cost allocation tags
→ Always configure CloudWatch alarms for RPU consumption and usage limit approaches
→ Always use Redshift Query Editor v2 for testing — it works with both provisioned and Serverless

❌ WHAT SHOULD NOT DO

→ Never set base RPU to 512 "just in case" — you'll pay $0.36 × 512 = $184/hour during any query
→ Never skip usage limits — a single misconfigured dashboard can generate $30,000/month in RPU charges
→ Never delete the provisioned cluster without thorough verification — no undo once deleted
→ Never use Serverless for 24/7 heavy workloads (> 70% utilization) — provisioned with Reserved Instances is cheaper
→ Never ignore the cold start latency (~10-30 seconds) — first query after idle period is slower
→ Never forget to update Step Functions pipeline (from Project 49) — Redshift Data API calls must target Serverless
→ Never assume Serverless and provisioned have identical performance — some edge cases differ
→ Never skip Enhanced VPC Routing — Spectrum and Federated Query require it
→ Never use only 2 subnets — Serverless requires minimum 3 AZs
→ Never expose Serverless endpoint publicly — keep in private VPC with security groups
→ Never forget that Spectrum scan costs ($5/TB) are ADDITIONAL to RPU costs — optimize partition pruning
→ Never set breach action to "turn off queries" on daily limits during critical business periods — use "alert" instead

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

1 → RPU PROVISIONING AND SCALING MECHANICS

STATE: IDLE (0 RPUs, $0/second)
    |
    ↓ Query arrives
    |
STATE: COLD START (~10-30 seconds)
    Serverless provisions base RPUs (e.g., 8 RPUs)
    Allocates compute from AWS managed pool
    Loads metadata and query compilation cache
    |
    ↓ RPUs ready
    |
STATE: ACTIVE (8 RPUs, $0.36 × 8 / 3600 = $0.0008/second)
    Query execution begins
    |
    ├── Query needs more compute?
    │   ↓ YES
    │   Serverless auto-scales to 16, 24, 32... RPUs
    │   Scaling happens in SECONDS (not minutes)
    │   No query restart — scaling is transparent
    │   Query continues on expanded compute
    │
    └── Query completes
        |
        ↓
STATE: WARM IDLE (8 RPUs maintained for ~5-10 minutes)
    Ready for next query — no cold start
    Still charged at $0.0008/second
    |
    ├── New query arrives within idle timeout?
    │   → Execute immediately (no cold start) ✅
    │
    └── No queries for ~5-10 minutes?
        → Scale down to 0 RPUs
        → $0/second
        → Next query will have cold start

Key Insight — RPU Scaling is TRANSPARENT:
→ Unlike provisioned cluster resize (which causes brief downtime)
→ Serverless RPU scaling happens DURING query execution
→ The query doesn't restart — it simply gets more compute mid-flight
→ Users never know scaling happened

2 → HOW MANAGED STORAGE WORKS IN SERVERLESS

Serverless Namespace Storage:
    |
    ├── Hot tier (SSD-backed, in-compute)
    │   → Recently accessed data blocks
    │   → Automatically cached based on access patterns
    │   → Fastest access — same as local SSD in provisioned
    │
    ├── Warm tier (S3-backed, managed)
    │   → Less frequently accessed data
    │   → Automatically tiered by Redshift
    │   → Slightly higher latency than hot tier
    │   → Most data lives here
    │
    └── Snapshot tier (S3-backed, compressed)
        → Point-in-time recovery data
        → Managed automatically
        → Used for restore operations

→ You do NOT manage these tiers — Redshift handles data placement automatically
→ Hot data is cached close to compute for fast access
→ Cold data is in S3-backed managed storage — still fast but slightly higher latency
→ This is the same architecture as RA3 nodes — just fully managed in Serverless

3 → HOW CONCURRENCY SCALING WORKS

Normal Load: 5 concurrent queries
    → 8 RPUs handle all 5 queries → Fine

Peak Load: 50 concurrent queries arrive simultaneously
    |
    ↓
Serverless evaluates:
    → Current 8 RPUs can handle ~5-8 concurrent queries
    → 50 queries need more compute
    |
    ↓
Auto-scale RPUs: 8 → 32 → 64 → 128 (based on demand)
    → All 50 queries execute concurrently
    → No WLM queuing (unlike provisioned clusters)
    → No manual concurrency scaling configuration needed
    |
    ↓
Queries complete → Load drops to 2 concurrent queries
    → Scale down: 128 → 32 → 8 RPUs
    → Cost returns to base level

Provisioned vs Serverless Concurrency:

AspectProvisionedServerless
Concurrency managementManual WLM configuration (from Project 16)Automatic
Queuing behaviorQueries queue when slots fullQueries trigger scaling — minimal queuing
Concurrency scalingMust enable and configureBuilt-in, automatic
Maximum concurrent queriesLimited by cluster size + WLM slotsLimited by RPU maximum (configurable)

4 → HOW BILLING WORKS (PER-SECOND GRANULARITY)

Timeline:

14:00:00 — First query arrives → Cold start begins
14:00:15 — 8 RPUs provisioned → Query starts
14:00:15 to 14:02:30 — Query executes (2 min 15 sec = 135 seconds)
14:02:30 — Query completes

Billing for this query:
    RPU-seconds: 8 RPUs × 135 seconds = 1,080 RPU-seconds
    Cost: 1,080 / 3600 × $0.36 = $0.108

14:02:30 to 14:07:30 — Idle (warm) — 5 minutes = 300 seconds
    RPU-seconds: 8 RPUs × 300 seconds = 2,400 RPU-seconds
    Cost: 2,400 / 3600 × $0.36 = $0.24

14:07:30 — No more queries → Scale to 0 → Billing stops

Total session cost: $0.108 + $0.24 = $0.348

→ You pay for the WARM IDLE period too — not just query execution
→ This is why optimizing base RPU matters — lower base = lower idle cost
→ Frequent short queries with gaps = more warm idle cost
→ Batching queries together = less warm idle waste

→ How to debug performance issues:

Step 1: Is the query slow due to cold start?
    → Check: Was this the first query after idle period?
    → Fix: Schedule warm-up query OR accept 10-30s initial latency

Step 2: Is the query slow due to insufficient RPUs?
    → Check: Workgroup monitoring → RPU utilization during query
    → Fix: Increase base RPU or maximum RPU limit

Step 3: Is the query slow due to Spectrum scan?
    → Check: Query plan → Spectrum scan bytes
    → Fix: Add partition filters (WHERE year=, month=, day=)

Step 4: Is the query slow due to federated query?
    → Check: Query plan → Foreign Scan duration
    → Fix: Optimize MariaDB indexes (from Project 48)

Step 5: Is the query slow due to poor table design?
    → Check: SVV_TABLE_INFO → Sort key, distribution key
    → Fix: Optimize table design (sort keys, distribution keys)

Step 6: Is the query blocked by usage limits?
    → Check: Workgroup → Limits → Current usage
    → Fix: Increase limits or optimize query patterns
