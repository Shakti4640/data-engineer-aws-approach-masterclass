PROJECT 25: THE COMPANY THAT WANTED SELF-SERVICE ANALYTICS — S3 → Glue Crawler → Glue Catalog → Athena End-to-End Pipeline

🔴 HYPOTHETICAL PROBLEM

→ ShopNow has built an impressive set of individual components over the last 24 projects
→ The data lake has three data origins flowing into S3:

OriginPipelineData Available
File uploadsRaw CSV/JSON → Glue ETL (Project 21) → ParquetClickstream, historical orders
Redshift computationsRedshift → UNLOAD (Project 23) → ParquetDaily revenue summaries, customer segments
MariaDB operationsMariaDB → Glue JDBC (Project 24) → ParquetCustomers, orders, products, payments, inventory

→ But the data engineering team is drowning in manual work:

→ Problem 1: Manual Pipeline Orchestration

  • Every morning the data engineer must:
    1. Verify raw files landed in S3 (uploaded overnight by external vendors)
    2. Manually trigger the Glue Crawler on raw data
    3. Wait for Crawler to complete
    4. Manually trigger the Glue ETL job
    5. Wait for ETL to complete
    6. Manually trigger another Crawler on transformed data
    7. Wait for Crawler to complete
    8. Notify analysts that data is ready
  • This takes 45 minutes of babysitting every single morning
  • If the engineer is sick → pipeline doesn't run → analysts have stale data

→ Problem 2: Analysts Can't Self-Serve

  • 15 analysts need different views of the same data
  • Marketing wants: customer profiles joined with purchase behavior
  • Finance wants: revenue by region, category, and payment method
  • Product team wants: inventory levels joined with sales velocity
  • Each analyst writes their own complex JOIN queries in Athena
  • Same 4-table JOIN is written 15 different ways → inconsistent results
  • When schema changes → all 15 analysts must update their queries individually

→ Problem 3: Athena Performance on Complex Queries

  • Analysts run multi-table JOINs directly in Athena on base Parquet tables
  • A 4-table JOIN across 50GB of data takes 45 seconds and scans 12 GB → $0.06/query
  • This query runs 200 times/day across all analysts → $12/day → $360/month for ONE report pattern
  • Pre-computing the JOIN result would reduce this to scanning 2 GB → $0.01/query → $60/month

→ Problem 4: Partition Management Overhead

  • New partitions appear daily (year=2025/month=01/day=16, day=17, day=18...)
  • Glue Crawler must run AFTER every ETL to register new partitions
  • If Crawler doesn't run → Athena returns incomplete results (missing the latest day)
  • Running Crawler on large tables (millions of partitions) takes 10+ minutes itself
  • There must be a better way to handle partitions without constant Crawler runs

→ Problem 5: No Single Source of Truth

  • Three separate Glue Catalog databases exist:
    • shopnow_raw_db (raw CSV tables — Project 17)
    • shopnow_transformed_db (Parquet tables — Project 21)
    • shopnow_mariadb_catalog (MariaDB tables — Project 24)
  • Plus a curated database from Redshift UNLOAD:
    • shopnow_curated_db (aggregated tables — Project 23)
  • Analysts don't know which database to query
  • "Should I use transformed.orders or mariadb_orders or curated.daily_revenue?"
  • No governance layer defining WHICH tables are the official "blessed" versions

→ The CTO says: "I want a self-service analytics platform where any analyst can find, understand, and query any data — without needing a data engineer to run pipelines or explain schemas"

🟡 HOW TO APPROACH IT

→ This project is NOT about introducing new AWS services
→ It's about CONNECTING services already learned into a cohesive, automated, self-service platform
→ Every component was introduced individually — now we orchestrate them together

→ The approach has five layers:

LayerPurposeComponents
1. Automated PipelineEliminate manual stepsCrawler schedules + ETL schedules + dependency ordering
2. Catalog OrganizationSingle source of truthDatabase naming, table naming, descriptions, tags
3. Athena ViewsReusable query definitionsNamed views that analysts reference instead of writing JOINs
4. CTAS (Create Table As Select)Pre-computed resultsMaterialized query results stored as optimized Parquet
5. Partition ProjectionEliminate Crawler for partitionsAthena infers partitions from patterns without Crawler

→ The approach should be:

  • Step 1 — Design the catalog organization (naming conventions, databases, descriptions)
  • Step 2 — Automate the end-to-end pipeline (schedule + dependency chain)
  • Step 3 — Create Athena Views for common query patterns
  • Step 4 — Use CTAS to pre-compute expensive JOINs
  • Step 5 — Implement Partition Projection to eliminate Crawler dependency
  • Step 6 — Build the analyst self-service interface
  • Step 7 — Implement cost controls and governance

Scheduling Method 1: Individual Glue Schedules (Simple but Fragile)

→ Each job has its own cron schedule
→ Time-based dependencies: "ETL starts at 03:00 because extraction finishes by 02:45"

ProblemImpact
Extraction runs late (large data day)ETL starts before extraction finishes → processes stale data
ETL runs lateCrawler runs on incomplete data
Any step failsDownstream steps run anyway on incomplete data

→ Time-based scheduling is FRAGILE — it assumes fixed durations
→ Better approach: Event-driven dependencies (Glue Workflow — Project 31)

Scheduling Method 2: Glue Workflow (Recommended — Preview for Project 31)

→ Glue Workflow chains jobs and crawlers with TRIGGERS based on completion events

Workflow: shopnow-daily-pipeline
  │
  ├── TRIGGER: Scheduled (02:00 UTC daily)
  │     └── STARTS: Extraction Jobs (parallel)
  │
  ├── TRIGGER: On completion of ALL extraction jobs (SUCCESS)
  │     └── STARTS: Crawler - operational data
  │
  ├── TRIGGER: On completion of operational Crawler (SUCCESS)
  │     └── STARTS: ETL Jobs (parallel)
  │
  ├── TRIGGER: On completion of ALL ETL jobs (SUCCESS)
  │     └── STARTS: Crawler - staging data
  │
  ├── TRIGGER: On completion of staging Crawler (SUCCESS)
  │     └── STARTS: CTAS refresh queries (via Lambda)
  │
  └── TRIGGER: On completion of CTAS refresh (SUCCESS)
        └── STARTS: SNS notification

→ Event-driven: Each step starts ONLY when the previous step SUCCEEDS
→ If extraction fails → nothing downstream runs → no stale data corruption
→ If extraction takes 60 minutes instead of 45 → ETL waits until extraction completes

→ Glue Workflow deep dive is in Project 31
→ For now — understand that THIS is the production-grade orchestration method

Scheduling Method 3: EventBridge + Step Functions (Advanced — Project 32, 38)

→ For complex branching logic, parallel paths with independent error handling, and human approval steps
→ More powerful than Glue Workflow but more complex to configure
→ Used when pipeline spans beyond Glue (Lambda, Redshift, SNS, etc.)

→ For ShopNow's current needs → Glue Workflow (Method 2) is sufficient
→ We'll implement it fully in Project 31

STEP 3 — CREATE ATHENA VIEWS FOR COMMON QUERY PATTERNS

→ An Athena View is a SAVED query definition that acts like a virtual table
→ No data is stored — the View's SQL executes every time the View is queried
→ Views are stored in the Glue Data Catalog as table entries with TableType: VIRTUAL_VIEW

Why Views solve the analyst problem:

  • Write the complex JOIN once → analysts reference the View by name
  • Schema changes → update the View definition → all analysts automatically get the new schema
  • Consistent results — everyone queries the same logic
  • Simpler queries — analysts write SELECT * FROM v_customer_360 instead of a 40-line JOIN

STEP 4 — USE CTAS TO PRE-COMPUTE EXPENSIVE QUERIES

→ CTAS = Create Table As Select
→ Unlike a View (virtual — executes on every query), CTAS MATERIALIZES results as physical Parquet files in S3
→ The result is a NEW table in the Glue Catalog backed by Parquet files

When to use View vs CTAS:

CriteriaUse ViewUse CTAS
Underlying data changes frequently✅ Always shows latest❌ Shows snapshot at creation time
Query is expensive (large JOINs)❌ Expensive on every access✅ Pre-computed — fast reads
Many users run the same query❌ Each user pays full scan cost✅ All users read pre-computed result
Result set is small (< 100 MB)✅ Negligible scan costEither works
Result set is large (> 1 GB)❌ Expensive repeated scans✅ Scan pre-computed result instead
Freshness requirement: real-time✅ Always current❌ Only as fresh as last CTAS refresh
Freshness requirement: dailyEither works✅ Refresh daily via scheduled CTAS

CTAS Refresh Pattern:

→ CTAS creates a table ONCE — but data goes stale
→ To refresh, you must:

  1. Drop the existing table (removes Catalog entry)
  2. Delete the S3 files (removes old data)
  3. Re-run the CTAS (creates new data + catalog entry)

→ Or use INSERT INTO pattern:

  1. Keep existing table
  2. Use INSERT INTO with new partition values
  3. Old partitions remain, new partition added

For daily append (sales funnel):

→ Each day adds a new partition: funnel_date=2025-01-16
→ Use INSERT INTO to add new partition without touching historical data

For full refresh (customer segments):

→ Segments change as customers' behavior changes
→ Full CTAS refresh monthly — drop + recreate

STEP 5 — IMPLEMENT PARTITION PROJECTION

→ This is a game-changer for eliminating Crawler dependency for partitioned tables

The Problem with Crawlers for Partitions:

IssueImpact
Crawler must run after every ETL to discover new partitionsAdds 5-15 minutes to pipeline
Large tables with thousands of partitions → Crawler is slow10,000 partitions → 10+ minute Crawler run
If Crawler fails or is delayed → Athena shows stale dataMissing latest partition = missing latest data
Crawler creates Glue Catalog entries for each partitionMillions of partition entries = Catalog bloat
Crawler costs moneyEach Crawler run is billed by DPU-hour

The Solution: Partition Projection

→ Instead of CRAWLING S3 to discover partitions → TELL Athena the partition pattern
→ Athena INFERS partitions mathematically — no Crawler needed
→ No Glue Catalog partition entries created — Athena generates them on the fly

How it works conceptually:

WITHOUT Partition Projection:
  Athena: "What partitions exist for this table?"
  → Asks Glue Catalog → Catalog returns list from last Crawler run
  → If new partition not yet crawled → Athena doesn't know about it

WITH Partition Projection:
  Athena: "What partitions exist for this table?"
  → Checks table properties: "year is integer 2020-2030, month is integer 1-12, day is integer 1-31"
  → Generates ALL possible partition combinations mathematically
  → For each partition in the query's WHERE clause → constructs S3 path → checks if files exist
  → No Catalog lookup, no Crawler needed

Configuring Partition Projection:

→ Navigate to Glue Console → Data Catalog → Table → Edit Table → Table Properties

OR set via Athena ALTER TABLE

Example: Orders table with year/month/day partitions

Table Properties to Add:

Property KeyValuePurpose
projection.enabledtrueEnables partition projection
projection.year.typeintegerYear is an integer
projection.year.range2020,2030Valid year range
projection.month.typeintegerMonth is an integer
projection.month.range1,12Valid month range
projection.day.typeintegerDay is an integer
projection.day.range1,31Valid day range
storage.location.templates3://shopnow-data-lake/transformed/mariadb/orders/year=${year}/month=${month}/day=${day}/S3 path pattern

What these properties mean:

→ projection.year.range = 2020,2030 → Athena knows years go from 2020 to 2030
→ projection.month.range = 1,12 → Months go from 1 to 12
→ projection.day.range = 1,31 → Days go from 1 to 31
→ storage.location.template → For year=2025, month=1, day=15 → Athena constructs path:
s3://shopnow-data-lake/transformed/mariadb/orders/year=2025/month=1/day=15/

Query behavior:

Query: SELECT * FROM orders WHERE year = 2025 AND month = 1 AND day = 15

WITHOUT Partition Projection:
  1. Athena asks Glue Catalog for partition year=2025/month=1/day=15
  2. If Crawler hasn't run → partition not in Catalog → returns 0 results
  3. Even if data exists in S3

WITH Partition Projection:
  1. Athena generates path: s3://…/orders/year=2025/month=1/day=15/
  2. Athena checks S3 directly → files exist
  3. Athena reads files → returns results
  4. No Catalog lookup, no Crawler dependency

TypeUse CaseConfiguration
integerNumeric partitions (year, month, day)range: min,max; optional digits: zero-padding
dateDate-based partitionsrange: start,NOW; format: yyyy-MM-dd; interval: 1; interval.unit: DAYS
enumFixed set of valuesvalues: value1,value2,value3
injectedDynamic values from queryNo range — value comes from WHERE clause

✅ WHAT SHOULD DO

→ Always create a dedicated analytics database for analyst-facing Views and CTAS tables
→ Always add descriptions to every table and column in Glue Catalog — this IS your data catalog
→ Always use Views for lightweight, frequently-changing queries — real-time freshness
→ Always use CTAS for expensive JOINs that multiple users run — pre-compute saves 90%+ cost
→ Always enable Partition Projection on date-partitioned tables — eliminates Crawler dependency
→ Always create Workgroups per team — cost isolation, query limits, encryption enforcement
→ Always enable query result reuse — 30-50% cost savings for common queries
→ Always require partition filters in analyst Workgroups — prevents accidental full table scans
→ Always schedule CTAS refreshes AFTER upstream pipeline completes — stale CTAS = stale analytics
→ Always use consistent naming conventions (v_ for Views, t_ for CTAS) — analysts instantly understand table type
→ Always version-control View definitions — store CREATE VIEW statements in git repository
→ Always test Views after upstream schema changes — broken Views return errors to analysts

❌ WHAT SHOULD NOT DO

→ Never let analysts query raw or staging databases directly — use the analytics layer
→ Never create Views with SELECT * from huge tables without filters — View executes on every access
→ Never forget to refresh CTAS tables — stale pre-computed data is worse than no pre-computed data
→ Never rely solely on Crawlers for partition discovery when patterns are predictable — use Partition Projection
→ Never skip Workgroup configuration — one analyst without limits can scan the entire lake
→ Never create Views that join more than 5-6 tables — performance degrades exponentially
→ Never use CTAS without specifying format as Parquet — default may be less optimal
→ Never store CTAS output in the same S3 prefix as source data — creates confusion
→ Never skip table/column descriptions — undocumented tables become "dark data" nobody uses
→ Never assume Views are free — every View query scans underlying data; complex Views on large tables are expensive
→ Never create circular View dependencies (View A references View B which references View A) — Athena will error
→ Never enable Partition Projection with wrong path template — Athena constructs incorrect S3 paths → returns 0 results

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

1 → ATHENA VIEW EXECUTION INTERNALS

Analyst runs: SELECT * FROM v_customer_360 WHERE customer_segment = 'high_value'
       ↓
Athena resolves View definition from Glue Catalog
  ├── Catalog entry type: VIRTUAL_VIEW
  ├── Stored SQL: The full CREATE VIEW SELECT statement
  └── Athena REPLACES the View reference with the actual SQL
       ↓
Expanded query becomes the full JOIN with WHERE clause
  ├── FROM shopnow_operational.mariadb_customers c
  ├── LEFT JOIN shopnow_operational.mariadb_orders o ON c.customer_id = o.customer_id
  ├── LEFT JOIN shopnow_staging.clickstream cl ON c.customer_id = cl.user_id
  ├── WHERE computed_segment = 'high_value'
  └── (full aggregation and CASE logic)
       ↓
Athena query optimizer analyzes expanded query
  ├── Identifies partition filters → applies partition pruning
  ├── Identifies column usage → applies column pruning (Parquet benefit)
  └── Generates distributed execution plan
       ↓
Athena executes plan against S3 data
  ├── Reads only needed columns from Parquet files
  ├── Reads only needed partitions
  └── Performs JOINs, aggregations, CASE logic
       ↓
Results returned to analyst
       ↓
COST: Same as running the expanded query directly
  (View adds zero overhead — it's pure SQL substitution)

→ Key insight: Views have ZERO runtime overhead — they are syntactic sugar
→ The query optimizer sees through the View and optimizes the underlying query
→ This is why expensive Views on large data are still expensive — there's no magic caching

2 → CTAS EXECUTION INTERNALS

CTAS command: CREATE TABLE t_customer_segments AS SELECT ...
       ↓
Athena executes the SELECT query (same as any query)
       ↓
Instead of returning results to client:
  ├── Results are formatted as Parquet
  ├── Results are compressed with Snappy
  ├── Results are partitioned by specified columns
  └── Results are written to specified S3 location
       ↓
Athena creates Glue Catalog entry:
  ├── Table name: t_customer_segments
  ├── Database: shopnow_analytics
  ├── Location: s3://shopnow-data-lake/analytics/customer-segments/
  ├── Format: Parquet
  ├── Columns: (from SELECT output schema)
  └── Partitions: (from PARTITIONED BY clause)
       ↓
Table is immediately queryable
       ↓
COST: Data scanned for the SELECT + S3 storage for output
  (Same scan cost as running the query once)
  (But subsequent reads of the CTAS table scan ONLY the pre-computed result)

CTAS File Output Behavior:

SettingBehavior
Default file countAthena determines based on data size and parallelism
File formatSpecified in CTAS (Parquet, ORC, JSON, CSV, AVRO)
File sizeTypically 64-256 MB per file
CompressionSnappy for Parquet (default), GZIP for CSV
Partition layoutHive-style key=value/ directories

CTAS vs INSERT INTO behavior:

OperationCreates New Table?Appends to Existing?Overwrites?
CTASYes — fails if table existsNoNo
INSERT INTONo — table must existYes — adds new dataNo
DROP + CTASYes (after drop)NoEffectively yes

→ For full refresh: DROP TABLE → CTAS
→ For incremental append: INSERT INTO existing table

3 → PARTITION PROJECTION INTERNALS

Query: SELECT * FROM orders WHERE year = 2025 AND month = 1 AND day = 15

WITHOUT Partition Projection:
  1. Athena → Glue Catalog: "List partitions for orders table matching year=2025, month=1, day=15"
  2. Glue Catalog → Scans partition index → Returns matching partition entry
  3. Partition entry contains S3 location: s3://…/orders/year=2025/month=1/day=15/
  4. Athena reads from that S3 location
  
  Problem: If Crawler hasn't discovered this partition → Catalog returns nothing → 0 results

WITH Partition Projection:
  1. Athena reads table properties:
       projection.enabled = true
       projection.year.type = integer, range = 2020,2030
       projection.month.type = integer, range = 1,12
       projection.day.type = integer, range = 1,31
       storage.location.template = s3://…/orders/year=${year}/month=${month}/day=${day}/
  
  2. Athena validates: year=2025 within [2020,2030]? ✅
                       month=1 within [1,12]? ✅
                       day=15 within [1,31]? ✅
  
  3. Athena constructs path: s3://…/orders/year=2025/month=1/day=15/
  
  4. Athena reads from that S3 path directly
  
  5. NO Glue Catalog partition lookup
     NO Crawler dependency
     Works immediately when data lands in S3

Partition Projection Performance:

MetricWithout ProjectionWith Projection
Partition discovery time100ms - 5s (Catalog lookup)< 10ms (mathematical generation)
New partition availabilityAfter Crawler runsImmediately when data lands in S3
Catalog storageOne entry per partition (millions possible)Zero partition entries
GetPartitions API callsProportional to partition countZero

→ For tables with thousands of partitions → Projection is dramatically faster
→ For tables with millions of partitions → Projection is the ONLY viable option (Catalog can't handle millions)

Edge case: What if projection generates a path that doesn't exist in S3?
→ Athena constructs the path → S3 returns "no objects found" → Athena returns 0 rows for that partition
→ No error — just empty result
→ This is fine — not every day may have data (e.g., no orders on a holiday)

4 → WORKGROUP QUERY LIMIT ENFORCEMENT

Analyst submits query: SELECT * FROM large_table (no WHERE clause)
       ↓
Athena estimates data scan size: ~50 GB
       ↓
Workgroup per-query limit: 10 GB
       ↓
50 GB > 10 GB → QUERY REJECTED before execution
       ↓
Error: "Query would scan more than the configured maximum"
       ↓
Analyst adds WHERE clause: WHERE year = 2025 AND month = 1
       ↓
Athena estimates data scan size: ~2 GB
       ↓
2 GB < 10 GB → QUERY ACCEPTED → executes normally

→ Limits are enforced BEFORE query execution — no partial scans
→ Athena uses table statistics and partition information to estimate scan size
→ Estimation may be slightly off — actual scan could be higher than estimate
→ If actual scan exceeds limit during execution → query is terminated

Daily workgroup limit:

09:00 - Analyst A runs query: scans 5 GB → Daily total: 5 GB
09:15 - Analyst B runs query: scans 8 GB → Daily total: 13 GB
09:30 - Analyst A runs query: scans 3 GB → Daily total: 16 GB
...
15:00 - Analyst C runs query: estimated 90 GB → Daily total would be 106 GB
         Workgroup daily limit: 100 GB → QUERY REJECTED
         "Workgroup has exceeded its daily data scan limit"

→ Resets at midnight UTC (or configurable)
→ Prevents runaway costs from automated or repeated queries

5 → QUERY RESULT REUSE INTERNALS

09:00 - Analyst A: SELECT count(*) FROM v_daily_revenue WHERE order_date = '2025-01-15'
  ├── Athena checks result cache: No cached result
  ├── Executes query → Scans 500 MB → Takes 3 seconds
  ├── Returns result: 15,234
  └── Stores result in cache with hash of: query text + table versions + partition metadata

09:15 - Analyst B: SELECT count(*) FROM v_daily_revenue WHERE order_date = '2025-01-15'
  ├── Athena checks result cache:
  │     ├── Query text hash matches ✅
  │     ├── Underlying table metadata unchanged ✅
  │     ├── Cache age (15 min) < max age (60 min) ✅
  │     └── CACHE HIT
  ├── Returns cached result: 15,234
  ├── Data scanned: 0 bytes (no S3 read)
  └── Cost: $0.00

09:45 - ETL job updates underlying table → table metadata changes
10:00 - Analyst A: SELECT count(*) FROM v_daily_revenue WHERE order_date = '2025-01-15'
  ├── Athena checks result cache:
  │     ├── Query text hash matches ✅
  │     ├── Underlying table metadata CHANGED ❌ (ETL updated data)
  │     └── CACHE MISS (stale)
  ├── Executes query fresh → Scans 500 MB
  └── Returns updated result: 15,567

→ Cache invalidation is automatic when underlying data changes
→ No risk of returning stale results after ETL updates
→ Cache key includes: query text + database + table + partition metadata + workgroup
