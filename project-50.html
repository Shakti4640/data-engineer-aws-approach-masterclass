Understanding Key Settings:

Amazon side ASN (64512):
→ ASN = Autonomous System Number — used in BGP routing
→ Only relevant if you connect the TGW to on-premises via VPN or Direct Connect
→ For VPC-only connectivity → Default 64512 is fine
→ Must be in the private ASN range: 64512-65534 or 4200000000-4294967294

DNS Support:
→ When enabled → VPCs attached to the TGW can resolve each other's private DNS names
→ Same concept as VPC Peering DNS resolution (from Project 44, Step 5)
→ Requires enableDnsSupport and enableDnsHostnames on each VPC (from Project 5)

Default Route Table Association vs Propagation:

ConceptWhat It Does
AssociationLinks a VPC attachment to a specific TGW route table — determines which route table is used for OUTBOUND routing from that VPC
PropagationAllows a VPC attachment to ADVERTISE its CIDR into a TGW route table — determines which route table knows about that VPC's CIDR

→ These are TWO separate concepts — confusion between them is the #1 TGW misconfiguration

VPC A (10.0.0.0/16) attached to TGW
    |
    ├── ASSOCIATED with Route Table X
    |   → When traffic FROM VPC A enters TGW, Route Table X decides where it goes
    |
    └── PROPAGATED to Route Table Y
        → Route Table Y now has entry: 10.0.0.0/16 → VPC A attachment
        → Other VPCs associated with Route Table Y can now reach VPC A

STEP 4 — CREATE VPC ATTACHMENTS FROM EACH ACCOUNT

Understanding VPC Attachments:

→ A VPC attachment is the CONNECTION between a VPC and the Transit Gateway
→ Each VPC that needs TGW connectivity must create its own attachment
→ The attachment specifies WHICH SUBNETS in the VPC will be used for TGW traffic

Subnet Selection for TGW Attachment:

→ TGW creates an ENI (Elastic Network Interface) in each selected subnet
→ Best practice: Create DEDICATED subnets for TGW attachments

Subnet TypeUse for TGW Attachment?Why
Public subnet❌ NoTGW traffic should not flow through public subnets
Private application subnet⚠️ Possible but not idealMixes application and transit traffic
Private database subnet❌ NoDatabase subnets should be isolated
Dedicated TGW subnet✅ Best practiceClean separation, small CIDR, clear purpose

❌ WHAT SHOULD NOT DO

→ Never delete all peering connections at once during migration — migrate one at a time
→ Never use TGW for same-VPC communication — TGW is for INTER-VPC, local route handles intra-VPC
→ Never ignore TGW costs — $0.05/hour per attachment × 8 attachments = $292/month

→ Never create a single flat route table with all CIDRs — defeats the purpose of segmentation
→ Never skip TGW flow logs — without them, debugging cross-account traffic issues is nearly impossible
→ Never allow TGW subnets to have routes to the internet directly — TGW subnets should only route to NAT or stay local

3 → ASSOCIATION vs PROPAGATION — DETAILED MECHANICS

ASSOCIATION determines OUTBOUND routing:

VPC A sends packet → Enters TGW → Which route table to use?
    → TGW checks: "VPC A attachment is ASSOCIATED with Production Route Table"
    → Uses Production Route Table for routing decision

PROPAGATION determines INBOUND reachability:

VPC B wants to be reachable from Production VPCs
    → VPC B attachment is PROPAGATED to Production Route Table
    → Production Route Table automatically gets: 10.1.0.0/16 → VPC B attachment
    → Any VPC associated with Production Route Table can now reach VPC B

The Relationship:

Association = "When I SEND traffic, which route table decides where it goes?"
Propagation = "Which route tables know I EXIST and can send traffic TO me?"

For VPC A to reach VPC B:
    1. VPC A must be ASSOCIATED with a route table that has VPC B's CIDR
    2. VPC B must be PROPAGATED into that route table

For VPC B to reach VPC A (return traffic):
    1. VPC B must be ASSOCIATED with a route table that has VPC A's CIDR
    2. VPC A must be PROPAGATED into that route table

→ Association and propagation are INDEPENDENT operations
→ A VPC can be associated with Route Table X but propagated into Route Table Y
→ This is the foundation of segmentation

4 → STATIC ROUTES vs PROPAGATED ROUTES

Route TypeHow CreatedUse CasePriority
PropagatedAutomatic — from VPC attachment CIDRNormal inter-VPC routingLower
StaticManual — admin creates the routeDefault routes (0.0.0.0/0), blackholes, specific overridesHigher

Priority Rule:
→ If a static route and propagated route have the SAME destination CIDR → Static route WINS
→ If routes have different prefix lengths → Longest prefix match wins (regardless of static/propagated)

Blackhole Routes (Static):

→ You can create a STATIC route with target = "Blackhole"
→ Any traffic matching this route is SILENTLY DROPPED
→ Useful for explicitly blocking specific CIDRs
