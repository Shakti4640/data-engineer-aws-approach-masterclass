🔴 HYPOTHETICAL PROBLEM

Problem: SQS delivers each message to ONLY ONE consumer
When Payment Service reads the message → it's invisible to Inventory Service
Only ONE system processes each order → the others never see it

→ SQS is POINT-TO-POINT: One message → one consumer
→ They need PUBLISH-SUBSCRIBE + QUEUING: One message → multiple consumers, each with their own queue

→ Problem 3: priya-dev tried using SNS (Project 6) directly

Web App → SNS Topic → Payment Service (HTTP endpoint)
                     → Inventory Service (HTTP endpoint)
                     → Notification Service (HTTP endpoint)
                     → Analytics Service (HTTP endpoint)
                     → Logistics Service (HTTP endpoint)

→ This WORKS for fan-out BUT:

  • If Analytics Service is down → SNS retries 3 times → gives up → message LOST for analytics
  • If Logistics Service is slow → SNS delivers but service can't process fast enough → dropped
  • NO BUFFERING: Each consumer must process immediately or risk losing the message
  • NO RETRY AT CONSUMER PACE: SNS pushes — consumer can't pull when ready
  • During flash sale: 5,000 orders/minute → each service must handle 5,000/minute → no backpressure

→ Problem 4: Different consumers need different processing rates:

Payment Service1,000/minOverwhelmed — orders dropped

Logistics Service800/minOverwhelmed

→ Without buffering → the SLOWEST consumer becomes the bottleneck
→ The CTO demands: "Each system must receive EVERY order, process at its OWN pace, and NEVER lose a message — even if a system is down for hours"

🟡 HOW TO APPROACH IT

→ This is a FAN-OUT + BUFFERING problem
→ The solution: SNS → SQS Fan-Out Pattern

→ Combines the BEST of both services:

  • SNS provides: FAN-OUT (one message → many subscribers)
  • SQS provides: BUFFERING (each consumer has its own queue, processes at own pace)

Why Standard (Not FIFO) Topic:

→ FIFO topics limit throughput to 300 msg/sec
→ FIFO topics can ONLY fan out to FIFO SQS queues
→ Standard topics fan out to BOTH Standard and FIFO queues
→ For ShopNow: Order sequence matters at the QUEUE level (per-customer) — not at the topic level
→ Each consumer's SQS queue can be FIFO if that consumer needs ordering
→ The topic itself doesn't need to guarantee order — the queues handle it

FIFO Topic vs Standard Topic Fan-Out:

FeatureStandard Topic → Standard QueuesFIFO Topic → FIFO Queues
ThroughputVirtually unlimited300 msg/sec (topic-level)
Message orderingBest-effort at topic, per-group in FIFO queuesStrict FIFO end-to-end
DeduplicationNone at topic level5-minute dedup at topic
Queue compatibilityStandard or FIFO queuesONLY FIFO queues
Use caseHigh-throughput fan-outLow-volume, strict ordering required

→ For most fan-out patterns → Standard topic is the right choice

STEP 3 — CONFIGURE SQS ACCESS POLICIES ALLOWING SNS

→ This is the CRITICAL step — without this, SNS cannot send messages to SQS
→ By default, only the queue owner can send messages (Project 7)
→ We must explicitly allow the SNS topic to send

→ For EACH queue → SQS Console → Select queue → "Access policy" tab → "Edit"

Access Policy for shopnow-order-payment.fifo:

Statement:
  Sid: AllowSNSToSendMessage
  Effect: Allow
  Principal:
    Service: sns.amazonaws.com
  Action: sqs:SendMessage
  Resource: arn:aws:sqs:ap-south-1:123456789012:shopnow-order-payment.fifo
  Condition:
    ArnEquals:
      aws:SourceArn: arn:aws:sns:ap-south-1:123456789012:shopnow-order-events

Every Field Explained:

→ Principal: Service: sns.amazonaws.com — The SNS service is the sender
→ Action: sqs:SendMessage — Allow sending messages to this queue
→ Resource — This specific queue's ARN (not wildcard)
→ Condition: ArnEquals: aws:SourceArn — ONLY from this specific SNS topic

→ Without the condition → ANY SNS topic in ANY account could send to this queue
→ The condition ensures only shopnow-order-events can send

Apply the SAME policy pattern to ALL 5 queues:
→ Change only the Resource ARN for each queue
→ The SourceArn condition stays the same (all receive from the same topic)

If Queue Uses SSE-KMS Encryption:

→ When the SQS queue is encrypted with KMS (not SSE-SQS):
→ SNS needs permission to use the KMS key to encrypt the message

→ Additional KMS key policy required:

Statement:
  Sid: AllowSNSToUseKMSKey
  Effect: Allow
  Principal:
    Service: sns.amazonaws.com
  Action:
    - kms:GenerateDataKey
    - kms:Decrypt
  Resource: "*"
  Condition:
    ArnEquals:
      aws:SourceArn: arn:aws:sns:ap-south-1:123456789012:shopnow-order-events

→ This is added to the KMS key policy (not the SQS policy)
→ With SSE-SQS encryption → no KMS policy needed (simpler)
→ KMS deep dive in Project 36

STEP 5 — UNDERSTAND RAW MESSAGE DELIVERY

→ By default, SNS wraps the original message in a JSON envelope before sending to SQS
→ This envelope contains SNS metadata that the consumer must parse through

Default Behavior (Raw Message Delivery = OFF):

→ Publisher sends message body: {"order_id": "ORD-001", "amount": 4497}
→ SQS receives this wrapped envelope:

{
  "Type": "Notification",
  "MessageId": "sns-message-id-123",
  "TopicArn": "arn:aws:sns:ap-south-1:123456789012:shopnow-order-events",
  "Subject": "New Order",
  "Message": "{\"order_id\": \"ORD-001\", \"amount\": 4497}",
  "Timestamp": "2025-01-15T14:30:00.000Z",
  "SignatureVersion": "1",
  "Signature": "EXAMPLEpH+...",
  "SigningCertURL": "https://sns.ap-south-1.amazonaws.com/...",
  "UnsubscribeURL": "https://sns.ap-south-1.amazonaws.com/...",
  "MessageAttributes": {
    "order_type": {
      "Type": "String",
      "Value": "flash-sale"
    }
  }
}

→ NOTICE: The actual message is INSIDE the "Message" field as an ESCAPED JSON string
→ The consumer must: Parse SQS message → Extract "Message" field → Parse AGAIN (double-parse)
→ This is ANNOYING for consumers that just want the order data

Raw Message Delivery (Enabled):

→ SNS Console → Select subscription → "Edit" → Enable "Raw message delivery"

→ With raw message delivery ON → SQS receives the ORIGINAL message body directly:

{"order_id": "ORD-001", "amount": 4497}

→ Consumer only needs to parse ONCE — much simpler
→ BUT: SNS metadata (TopicArn, Timestamp, Signature) is NOT included
→ AND: Message attributes are delivered as SQS message attributes (not in the body)

When to Use Raw Message Delivery:

ScenarioRaw DeliveryDefault (Wrapped)
Consumer only needs the message body✅ ONRequires double-parsing
Consumer needs SNS metadata (topic ARN, signature)❌ OFFMetadata included in envelope
Consumer needs to verify message authenticity❌ OFFSignature in envelope
Simple processing pipeline✅ ONUnnecessary complexity
Multiple protocol subscribers need consistent format❌ OFFSame format for all

→ For ShopNow → Enable Raw Message Delivery on ALL SQS subscriptions
→ Consumers just want the order data — they don't need SNS metadata

Enabling Raw Message Delivery:

→ SNS Console → Subscriptions → Select each SQS subscription → "Edit"
→ Raw message delivery: ✅ Enable
→ Save

→ IMPORTANT: Raw message delivery is ONLY available for SQS and Firehose subscriptions
→ Email, SMS, HTTP, Lambda subscriptions ALWAYS receive the wrapped envelope

STEP 6 — SUBSCRIPTION FILTER POLICIES

→ Not every consumer needs EVERY message
→ Use filter policies (from Project 6) to route specific messages to specific queues

Scenario: ShopNow has three order types:

Order TypeAttribute ValueWho Needs It
Regular orderregularAll consumers
Flash sale orderflash-saleAll consumers + special analytics
Return/refundreturnPayment + Inventory + Notification (NOT analytics or logistics)

Filter Policy on Analytics Queue:

→ SNS Console → Select analytics subscription → "Edit" → Subscription filter policy

{
  "order_type": ["regular", "flash-sale"]
}

→ Analytics queue receives regular and flash-sale orders
→ Return/refund orders are NOT delivered to analytics → reduces noise

Filter Policy on Logistics Queue:

{
  "order_type": ["regular", "flash-sale"]
}

→ Logistics doesn't need return/refund events — those are handled differently

Filter Policy on Payment Queue:

No filter policy (receives ALL messages)

→ Payment must process everything — returns need refund processing

Filter Policy on Inventory Queue:

No filter policy (receives ALL messages)

→ Inventory must update for all event types — returns increase stock

Filter Policy on Notification Queue:

No filter policy (receives ALL messages)

→ Customer needs notification for every order event type

How Message Attributes Must Be Set by Publisher:

→ When publishing to the SNS topic → include message attributes:

Message Body:
  {"order_id": "ORD-001", "customer_id": "CUST-5678", "items": [...], "amount": 4497}

Message Attributes:
  order_type: String = "flash-sale"
  region: String = "south-india"
  priority: Number = 1

→ SNS evaluates filter policies BEFORE delivery
→ If a subscription's filter doesn't match → message is NOT delivered to that queue
→ This is more efficient than delivering to all queues and having consumers discard irrelevant messages

Filter Policy Evaluation Flow:

Message published with attributes: order_type = "return"
        │
        ▼
SNS evaluates each subscription's filter policy:
        │
        ├── Payment queue: No filter → DELIVER ✓
        ├── Inventory queue: No filter → DELIVER ✓
        ├── Notification queue: No filter → DELIVER ✓
        ├── Analytics queue: Filter = ["regular", "flash-sale"]
        │   "return" NOT in list → DO NOT DELIVER ✗
        └── Logistics queue: Filter = ["regular", "flash-sale"]
            "return" NOT in list → DO NOT DELIVER ✗

Result: 3 queues receive the message, 2 are filtered out

STEP 8 — HANDLE FAILURE SCENARIOS AND DEAD LETTER QUEUES

Scenario 1: SQS Queue Is Unavailable When SNS Tries to Deliver

→ SQS is a managed service — it's almost never unavailable
→ But if the SQS access policy is misconfigured → SNS gets "Access Denied"
→ SNS treats this as a delivery failure → retries based on retry policy

SNS Retry Policy for SQS Subscriptions:

PhaseRetriesInterval
Immediate3 retriesNo delay (back-to-back)
Pre-backoff2 retries1 second apart
Backoff10 retriesExponential: 1s, 2s, 4s, ... up to 20s
Post-backoff100,000+ retries20 seconds apart
Total duration~23 daysBefore giving up

→ SNS retries SQS delivery aggressively — over 100,000 retries across 23 days
→ It's EXTREMELY unlikely for a properly configured SQS queue to lose a message from SNS

Scenario 2: Consumer Cannot Process a Message (Poison Message)

→ Payment service reads an order but can't process it (invalid data, external API failure)
→ Visibility timeout expires → message becomes visible → payment service reads it again → fails again
→ This loops forever → "poison message" consumes processing resources

→ Solution: Dead Letter Queue (DLQ) on EACH SQS queue

Create DLQ for Payment Queue:

→ SQS Console → "Create queue"

SettingValue
TypeFIFO (must match source queue type)
Nameshopnow-order-payment-dlq.fifo
Message retention14 days
Visibility timeout60 seconds

Configure Redrive Policy on Source Queue:

→ SQS Console → Select shopnow-order-payment.fifo → "Edit" → "Dead-letter queue"

SettingValueExplanation
Dead-letter queueshopnow-order-payment-dlq.fifoWhere failed messages go
Maximum receives3After 3 failed processing attempts → move to DLQ

How It Works:

Attempt 1: Consumer reads message → processing fails → doesn't delete
           Visibility timeout expires → message becomes visible again
           ApproximateReceiveCount = 1

Attempt 2: Consumer reads message → processing fails again
           ApproximateReceiveCount = 2

Attempt 3: Consumer reads message → processing fails again
           ApproximateReceiveCount = 3 (= maxReceiveCount)

SQS: "This message has been received 3 times without successful deletion"
     → Move message to DLQ: shopnow-order-payment-dlq.fifo
     → Message removed from source queue
     → Message appears in DLQ

Operations team:
     → Monitor DLQ depth (CloudWatch alarm — Project 8)
     → Investigate failed messages
     → Fix the issue
     → Redrive messages back to source queue (or process manually)

Source QueueDLQmaxReceiveCount

Scenario 3: SNS Cannot Deliver to SQS After All Retries (Extremely Rare)

→ Configure a DLQ on the SNS SUBSCRIPTION (not just the SQS queue)
→ This catches messages that SNS itself couldn't deliver to SQS

→ SNS Console → Select SQS subscription → "Edit" → "Redrive policy (dead-letter queue)"

SettingValue
Dead-letter queuearn:aws:sqs:ap-south-1:123456789012:shopnow-sns-delivery-dlq

→ The SNS DLQ is a SEPARATE queue from the SQS consumer DLQ
→ SNS DLQ catches: SNS → SQS delivery failures
→ SQS DLQ catches: Consumer processing failures
→ Two different failure points → two different DLQs

FAILURE POINT 1: SNS → SQS delivery failure
  SNS cannot send to SQS (access denied, queue deleted, etc.)
  → Message goes to SNS subscription DLQ
  → Monitor and fix SQS access policy

FAILURE POINT 2: SQS → Consumer processing failure
  Consumer reads from SQS but can't process (application error)
  → Message goes to SQS source queue's DLQ
  → Monitor, fix consumer, redrive

STEP 9 — ADDING A NEW CONSUMER (EXTENSIBILITY)

→ One month later → ShopNow adds a fraud detection system
→ It needs to receive ALL order events and check for fraudulent patterns
→ How much effort to add?

Steps to Add a New Consumer:

Step 1: Create SQS queue: shopnow-order-fraud-detection
Step 2: Add SQS access policy allowing SNS to send
Step 3: Create SNS subscription: SQS protocol → fraud-detection queue
Step 4: (Optional) Add filter policy if only certain order types need checking
Step 5: Deploy the fraud detection consumer application

CHANGES TO EXISTING SYSTEMS: ZERO
→ The publisher (web app) is unchanged
→ Other consumers are unchanged
→ No code changes, no redeployment
→ The new consumer just subscribes and starts receiving

→ THIS is the power of fan-out architecture
→ Adding consumers is ADDITIVE — never modifies existing components
→ This is the Open/Closed Principle applied to distributed systems

✅ WHAT SHOULD DO

→ Always create a SEPARATE SQS queue per consumer — never share queues between different processing concerns
→ Always configure SQS access policies with aws:SourceArn condition — prevents unauthorized topics from sending
→ Always enable Raw Message Delivery for SQS subscriptions — simplifies consumer code
→ Always create DLQs for BOTH levels: SNS subscription DLQ + SQS consumer DLQ
→ Always monitor DLQ depth with CloudWatch alarms (Project 8)
→ Always use filter policies when consumers don't need every message — reduces processing waste
→ Always include message attributes when publishing — enables future filter policies without publisher changes
→ Always use Standard topics for fan-out unless you need end-to-end FIFO
→ Always test the complete flow: publish → verify all queues → verify filter policies
→ Always set maxReceiveCount based on criticality: low for critical queues, higher for tolerant ones
→ Always document which queues subscribe to which topics — this becomes complex at scale
→ Always consider queue type per consumer: FIFO for critical, Standard for tolerant

❌ WHAT SHOULD NOT DO

→ Never use a single SQS queue for multiple independent consumers — SQS is point-to-point per message
→ Never skip SQS access policy configuration — SNS silently fails to deliver without permission
→ Never use FIFO topics when you need high-throughput fan-out — 300 msg/sec limit
→ Never forget DLQs — without them, poison messages loop forever consuming resources
→ Never set maxReceiveCount too high for critical queues — delays detection of processing failures
→ Never set maxReceiveCount to 1 — transient failures need at least 2-3 retries
→ Never forget that FIFO DLQs must match source FIFO queues — Standard DLQ with FIFO source doesn't work
→ Never publish without message attributes — you lose the ability to add filters later
→ Never assume SNS → SQS delivery is instant — there can be sub-second delays
→ Never create circular dependencies: Queue A → processes → publishes to Topic → delivers to Queue A
→ Never mix KMS-encrypted queues with unencrypted SNS without proper key policies
→ Never forget the ops email subscription for visibility — invaluable for debugging

2 → HOW SNS DELIVERS TO FIFO SQS QUEUES

→ When an SNS Standard topic delivers to a FIFO SQS queue → SNS must provide:

  • MessageGroupId (required for FIFO queues)
  • MessageDeduplicationId (required if content-based dedup is off)

How SNS Generates These for FIFO Queue Subscriptions:

AttributeHow SNS Sets ItOverride Possible?
MessageGroupIdFrom the MessageGroupId message attribute set by publisherYES — publisher must set it
MessageDeduplicationIdFrom the SNS MessageId (auto-generated, unique)YES — if content-based dedup is enabled on queue

→ CRITICAL: If publishing to a Standard topic that fans out to FIFO queues:

  • Publisher MUST include a MessageGroupId message attribute
  • Without it → SNS cannot send to the FIFO queue → delivery fails silently

Publisher Must Include:

Message Attributes (when publishing to SNS topic):
  order_type: String = "regular"
  region: String = "south-india"
  MessageGroupId: String = "CUST-5678"    ← REQUIRED for FIFO queue subscribers

→ The MessageGroupId attribute is used by SNS when delivering to FIFO queues
→ For non-FIFO queue subscribers → this attribute is ignored
→ This allows a Standard topic to fan out to a MIX of Standard and FIFO queues

3 → HOW CONCURRENT DELIVERY WORKS

SNS delivers to 5 SQS queues CONCURRENTLY (not sequentially):

Timeline:
0ms:    SNS starts delivery to all 5 queues simultaneously
5ms:    Payment queue confirms receipt ✓
8ms:    Notification queue confirms receipt ✓
12ms:   Logistics queue confirms receipt ✓
15ms:   Inventory queue confirms receipt ✓
45ms:   Analytics queue confirms receipt ✓ (slightly slower)

Total fan-out time: ~50ms (determined by SLOWEST delivery)

→ SNS does NOT wait for one delivery to complete before starting the next
→ All deliveries happen in parallel → total time = max(individual delivery times)
→ This is why fan-out is fast even with many subscribers

4 → HOW MESSAGE ORDERING WORKS END-TO-END

Standard Topic → Standard Queue (No ordering guarantee):

Publisher sends: Order A, Order B, Order C (in sequence)
SNS topic receives: A, B, C
SNS delivers to Standard queue: Might arrive as B, A, C or A, C, B
Consumer reads: No guaranteed order

→ Consumer must handle out-of-order messages
→ Or consumer must use message timestamps to reconstruct order

Standard Topic → FIFO Queue (Ordering within message group):

Publisher sends: Order A (group: CUST-1), Order B (group: CUST-1), Order C (group: CUST-2)
SNS delivers to FIFO queue:
  Group CUST-1: A then B (in order)
  Group CUST-2: C (independent)
Consumer reads from CUST-1: A first, then B (guaranteed)
Consumer reads from CUST-2: C (independent of CUST-1)

→ FIFO queue preserves order within each message group
→ But the ORDER in which groups are processed is not guaranteed
→ CUST-2's Order C might be processed before CUST-1's Order A

FIFO Topic → FIFO Queue (Strictest ordering):

Publisher sends: Order A (group: CUST-1), Order B (group: CUST-1)
FIFO topic guarantees: A delivered before B
FIFO queue guarantees: A processed before B

Full end-to-end ordering within the group
BUT limited to 300 msg/sec at the topic level

→ For ShopNow: Standard topic → FIFO queues for critical consumers → good balance

📈 SCALABILITY CONSIDERATIONS

→ Throughput Scalability

  • Standard SNS topic: Virtually unlimited publishing rate
  • Standard SQS queues: Virtually unlimited receive/send rate
  • FIFO SQS queues: 300 msg/sec per message group (3,000 with batching)
  • Fan-out itself adds minimal latency (50-200ms for all deliveries)
  • No pre-provisioning needed — scales automatically

→ Consumer Scalability

  • Each consumer can have MULTIPLE instances reading from the same queue
  • Competing consumers pattern: 5 EC2 instances reading from payment queue → 5x throughput
  • Auto Scaling (Project 33) can scale consumers based on queue depth
  • Each consumer scales independently — slow analytics doesn't affect fast payment processing

→ Subscriber Scalability

  • Up to 12.5 million subscriptions per SNS topic
  • Practical limit: Hundreds of SQS queues per topic is manageable
  • Each additional subscriber = ONE new SQS queue + ONE subscription → zero change to publisher

→ Multi-Region Fan-Out

  • SNS can deliver to SQS queues in DIFFERENT regions
  • Example: Topic in ap-south-1 → Queue in us-east-1 (cross-region delivery)
  • Cross-region delivery adds latency but enables global processing

→ Cost Scalability

  • Messaging costs are NEGLIGIBLE compared to compute costs
  • At 1 million messages/day → messaging cost is ~$10/month
  • Real cost is in CONSUMER COMPUTE (EC2/Lambda) — not in SNS/SQS

→ How to debug "queue not receiving messages from SNS":

STEP 1: Is the subscription confirmed?
        SNS Console → Subscriptions → Status column
        If "Pending" or "Deleted" → recreate subscription

STEP 2: Is the SQS access policy correct?
        SQS Console → Queue → Access policy
        Does it allow sqs:SendMessage from sns.amazonaws.com?
        Does the SourceArn condition match the exact topic ARN?

STEP 3: Is the filter policy filtering out the message?
        Check subscription filter policy
        Compare with message attributes used during publish
        Publish without attributes → does message arrive? (confirms filter is the issue)

STEP 4: Is the queue encrypted with KMS?
        If yes → does the KMS key policy allow sns.amazonaws.com?
        Try with SSE-SQS encryption → if it works, KMS policy is the issue

STEP 5: Is the topic encrypted with KMS?
        If yes → SQS subscription needs access to the topic's KMS key
        Check key policy

STEP 6: Check SNS delivery status logs
        CloudWatch Logs → sns/ → Look for delivery failures
        Shows: MessageId, delivery status, error message

STEP 7: For FIFO queues → check MessageGroupId
        Is the publisher including MessageGroupId as message attribute?
        Without it → SNS can't deliver to FIFO queues

STEP 8: Check CloudTrail for SNS:Publish events
        Was the publish call successful?
        What attributes were included?

🎯 KEY TAKEAWAY

→ SNS → SQS Fan-Out is the MOST POWERFUL messaging pattern in AWS — one publish, many independent consumers
→ SNS provides FAN-OUT (one-to-many distribution) — SQS provides BUFFERING (each consumer's own pace)
→ Together they solve: decoupling, reliability, independent scaling, and graceful failure handling
→ SQS access policy is the CRITICAL configuration — without it, SNS silently fails to deliver
→ Raw message delivery simplifies consumers by removing the SNS envelope wrapper
→ Filter policies enable selective delivery — not every consumer needs every message
→ DLQs are needed at TWO levels: SNS subscription DLQ (delivery failures) + SQS consumer DLQ (processing failures)
→ Adding new consumers is ADDITIVE — zero changes to publisher or existing consumers
→ Standard topic → mix of Standard and FIFO queues gives the best balance of throughput and correctness
→ For FIFO queue subscribers → publisher MUST include MessageGroupId as a message attribute
