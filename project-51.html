STEP 2 — DESIGN THE DATA ZONE ARCHITECTURE

The Four-Zone Model:

LANDING ZONE → RAW ZONE → CURATED ZONE → REFINED ZONE
(Ingest)       (Preserve)   (Transform)    (Serve)
ZonePurposeData StateFormatWho WritesWho ReadsRetention
LandingTemporary staging for incoming dataAs-received (CSV, JSON, Excel, API dumps)Original format — untouchedProducers (automated pipelines)Glue ETL only7-30 days (auto-delete)
RawPermanent archive of source dataConverted to standard format but NOT transformedParquet (columnar, compressed)Glue ETL (from Landing)Data engineers, complianceIndefinite (Glacier after 1 year)
CuratedBusiness-ready, cleaned, enriched dataDeduplicated, validated, joined, enrichedParquet (partitioned, optimized)Glue ETL (from Raw)Analysts, data scientists, BI toolsIndefinite
RefinedPre-aggregated datasets for specific use casesAggregated, summarized, pivotedParquet (highly optimized)Glue ETL or Athena CTAS (from Curated)Dashboards, APIs, executivesRefresh cycle (daily/hourly)

Why Four Zones?

QuestionAnswer
Why not just Landing + Curated?Raw zone preserves original data — if curated transformation has a bug, you can reprocess from Raw without re-ingesting from source
Why not just one big bucket?Different zones have different access patterns, retention policies, and governance requirements
Why Landing is temporary?Landing data is in its original format (CSV, JSON) — not optimized for querying — it's a staging area only
Why Raw is Parquet but untransformed?Parquet is efficient for storage and querying — converting format early saves downstream costs — but business logic is NOT applied yet
Why Curated is separate from Refined?Curated is the "single source of truth" — Refined is derived for specific use cases — multiple Refined datasets can come from one Curated dataset

Why Hive-Style Partitioning (key=value)?

→ Athena and Redshift Spectrum recognize key=value prefix patterns as PARTITIONS automatically
→ Glue Crawlers detect and catalog these as partition columns
→ Partition projection (from Project 25) eliminates the need for MSCK REPAIR TABLE
→ Athena queries with WHERE year=2025 AND month=01 scan ONLY that partition — massive cost savings

Partition Projection Configuration (Athena — from Project 25):

PropertyValuePurpose

projection.year.typeintegerYear is integer type
projection.year.range2020,2030Valid year range
projection.month.typeintegerMonth is integer type
projection.month.range1,12Valid month range
projection.month.digits2Zero-padded (01, 02, ...)
projection.day.typeintegerDay is integer type
projection.day.range1,31Valid day range
projection.day.digits2Zero-padded
storage.location.templates3://payscale-datalake-curated/domain=fraud/entity=transactions/year=${year}/month=${month}/day=${day}/S3 path template

→ With partition projection → Athena NEVER calls glue:GetPartitions
→ Athena constructs the S3 path directly from the query filter
→ This eliminates Glue API throttling for tables with millions of partitions

STEP 7 — DESIGN THE CROSS-ACCOUNT ACCESS MODEL

Three Layers of Cross-Account Access:

Layer 1: Network Connectivity
    → Transit Gateway (from Project 50) connects all accounts
    → VPC Endpoints for S3 (from Project 27) in each account
    → NOT needed for Athena → Athena uses AWS-managed endpoints

Layer 2: Glue Catalog Sharing
    → Lake Formation cross-account grants (from Project 42, 47)
    → RAM shares the Glue Catalog resources
    → Consumer accounts create Resource Links

Layer 3: S3 Data Access
    → Lake Formation credential vending (from Project 47)
    → Consumer accounts do NOT need direct S3 access
    → Lake Formation's registered IAM role accesses S3 on behalf of consumers

✅ WHAT SHOULD DO

→ Always create a DEDICATED data lake account — never mix with application workloads
→ Always implement the four-zone architecture (Landing → Raw → Curated → Refined) — each zone has a clear purpose
→ Always use Hive-style partitioning (key=value/) for Athena and Spectrum compatibility
→ Always use Parquet format with Snappy compression for all zones except Landing
→ Always register ALL S3 data locations with Lake Formation before granting permissions
→ Always use Tag-Based Access Control for scalable governance — named grants don't scale beyond 20 tables
→ Always implement data quality checks at each zone transition — not just at ingestion
→ Always use partition projection in Athena for time-partitioned tables — eliminates Glue API bottleneck
→ Always create separate Athena workgroups per team — enables cost tracking and data limits
→ Always implement a quarantine pattern for failed quality records — never silently drop data
→ Always use cross-account Lake Formation grants (not bucket policies) for consumer access — governance is centralized
→ Always enable CloudTrail with S3 Data Events for the data lake account — audit trail is mandatory for compliance
→ Always design for domain ownership — each team governs their own data, central team provides the platform
→ Always use Resource Links for cross-account catalog access — direct table references don't work in consumer Athena
→ Always version your Glue ETL scripts in source control — data lineage requires knowing what transformation was applied

❌ WHAT SHOULD NOT DO

→ Never store the data lake in an application account — blast radius and billing confusion
→ Never skip the Raw zone — without it, you lose the ability to reprocess from source
→ Never allow producers to write directly to Curated zone — all data must go through ETL quality gates
→ Never grant IAMAllowedPrincipals on data lake databases — defeats the purpose of Lake Formation
→ Never use CSV as the storage format in Raw/Curated zones — Parquet is 10x more efficient for analytics
→ Never create one giant database for everything — use zone_domain naming convention
→ Never let consumers query the Landing zone — it contains unvalidated, potentially corrupt data
→ Never skip partition projection for large time-series tables — Glue GetPartitions API becomes the bottleneck
→ Never allow cross-account access via bucket policies alone — Lake Formation provides column/row-level control
→ Never forget to grant Glue ETL service role Lake Formation permissions — crawlers and jobs fail silently
→ Never mix production and development data in the same data lake — use separate buckets/accounts for dev
→ Never ignore data quality — one bad dataset destroys trust in the entire data lake
→ Never grant "Grantable" permissions to domain owners for other domains — each domain governs only itself
→ Never store Athena query results in the data lake buckets — use separate result buckets per account

3 → HOW TBAC RESOLVES ACROSS ZONES AND DOMAINS

User: JuniorFraudAnalystRole
TBAC Grant: zone IN (curated, refined) AND domain = fraud AND sensitivity IN (public, internal, confidential)

Query: SELECT * FROM curated_fraud.transactions_clean WHERE year=2025 AND month=01

Resolution:
    Table: curated_fraud.transactions_clean
    Table Tags: zone=curated ✅, domain=fraud ✅, sensitivity=confidential ✅
    → TABLE ACCESS GRANTED ✅
    
    Column: transaction_id → Tag: sensitivity=internal → ✅ INCLUDED
    Column: customer_id → Tag: sensitivity=internal → ✅ INCLUDED
    Column: amount → Tag: sensitivity=internal → ✅ INCLUDED
    Column: credit_card_number → Tag: sensitivity=restricted → ❌ EXCLUDED
    Column: merchant → Tag: sensitivity=public → ✅ INCLUDED
    Column: risk_score → Tag: sensitivity=confidential → ✅ INCLUDED
    Column: customer_tier → Tag: sensitivity=internal → ✅ INCLUDED
    
    Result: Query returns all columns EXCEPT credit_card_number

Same user queries: raw_fraud.transactions
    Table Tags: zone=raw → ❌ NOT IN GRANT (grant only covers curated, refined)
    → TABLE ACCESS DENIED ❌

Same user queries: curated_platform.payments_reconciled
    Table Tags: domain=platform → ❌ NOT IN GRANT (grant only covers fraud)
    → TABLE ACCESS DENIED ❌

→ TBAC evaluates BOTH table tags AND column tags
→ The grant covers zone + domain + sensitivity combinations
→ Any mismatch at any level → Access denied

4 → HOW CROSS-ACCOUNT DATA FLOW WORKS WITHOUT DIRECT S3 ACCESS

Consumer (Account C) has NO S3 permissions on Account J's buckets
Consumer (Account C) has NO IAM policy for s3:GetObject on Account J

How does the query work?

Step 1: Athena (Account C) receives query
Step 2: Athena calls Glue Catalog via Resource Link
Step 3: Lake Formation (Account C) validates local permission on resource link
Step 4: Lake Formation (Account J) validates cross-account permission on target table
Step 5: Lake Formation (Account J) generates TEMPORARY S3 credentials
    → These credentials are scoped to ONLY the permitted columns and partitions
    → Credentials expire after the query completes
Step 6: Athena uses these temporary credentials to read Parquet files from S3
Step 7: Athena applies column projection (skip excluded columns in Parquet reader)
Step 8: Athena applies row filter (if data filter exists)
Step 9: Results returned to Account C

The consumer NEVER has direct S3 access
Lake Formation's IAM role (registered in Step 6d) is the ONLY entity reading S3
Credential vending creates a security boundary between consumers and storage

→ This is fundamentally different from cross-account bucket policies (Project 39)
→ With bucket policies → Consumer has direct S3 access → Cannot enforce column/row filtering
→ With Lake Formation → Consumer has NO direct S3 access → Column/row filtering enforced by Lake Formation
