PROJECT 16: THE ANALYST WHO NEEDED A DATA WAREHOUSE — Redshift Single-Node Cluster Setup

🔴 HYPOTHETICAL PROBLEM

→ ShopNow (from Projects 1-15) now has:

  • S3 bucket with event-driven processing pipelines (Projects 1, 12, 15)
  • RDS MariaDB for transactional operations (Projects 9, 14)
  • Full messaging infrastructure with SNS → SQS fan-out (Projects 6, 7, 13)
  • VPC with proper networking and security (Projects 3-5)
  • IAM Roles for all services (Project 11)

→ The business intelligence (BI) team has 5 analysts who need to answer questions like:

  • "What were total sales by product category for the last 12 months?"
  • "Which regions had the highest return rates?"
  • "What is the week-over-week trend in average order value?"
  • "How many unique customers purchased in each month of 2024?"
  • "What is the correlation between discount percentage and order volume?"

→ Problem 1: The BI team runs these queries against RDS MariaDB (Project 9)
→ The query: SELECT category, SUM(amount) FROM orders JOIN products ... GROUP BY category
→ This query scans 50 million rows in the orders table
→ MariaDB takes 47 MINUTES to return results
→ During execution → table locks slow down the web application (Project 14 — Problem 6)
→ Customers experience latency while analysts run reports

→ Problem 2: The BI team wants to JOIN data from multiple sources:

  • Orders data (in RDS MariaDB)
  • Product catalog (in RDS MariaDB)
  • Website clickstream (CSV files in S3)
  • Marketing campaign results (CSV files in S3)
  • Customer demographics (CSV files in S3)
    → MariaDB can't efficiently query CSV files in S3
    → The team manually downloads CSVs → imports into MariaDB → joins → runs query
    → This manual ETL takes 2 hours before they can even START analyzing

→ Problem 3: Data volume is growing rapidly:

→ Even with Read Replicas (Project 29) → row-based scanning is fundamentally slow for analytics

→ Problem 4: The CTO asks: "Why is our transactional database so slow for analytics?"
→ The answer: MariaDB (and all OLTP databases) store data in ROWS

ROW-BASED STORAGE (MariaDB):
┌─────────────────────────────────────────────────────────────┐
│ Row 1: | order_id=1 | date=2024-01-15 | category=Shoes | amount=999 | region=South |
│ Row 2: | order_id=2 | date=2024-01-15 | category=Electronics | amount=2499 | region=North |
│ Row 3: | order_id=3 | date=2024-01-16 | category=Shoes | amount=1299 | region=West |
│ ...
│ Row 50,000,000: | order_id=50M | date=2025-01-15 | ... |
└─────────────────────────────────────────────────────────────┘

Query: SELECT category, SUM(amount) FROM orders GROUP BY category

MariaDB must: Read ALL columns of ALL 50 million rows
              Even though query only needs "category" and "amount" (2 out of 20 columns)
              Reads 20x more data than necessary
              Sequential scan → 47 minutes

→ The CTO demands:

  • Analytics queries must complete in SECONDS, not minutes
  • Analytical workloads must NOT impact the production transactional database
  • Must handle terabytes of data efficiently
  • Must support loading data from BOTH MariaDB AND S3
  • Must integrate with BI tools (QuickSight, Tableau, etc.)

🟡 HOW TO APPROACH IT

→ This is a DATA WAREHOUSING problem
→ The solution: Amazon Redshift — a columnar, massively parallel processing (MPP) data warehouse

OLTP vs OLAP — Fundamental Difference:

CharacteristicOLTP (MariaDB — Project 9)OLAP (Redshift)
PurposeProcess transactionsAnalyze data
Query patternSingle-row lookups, inserts, updatesAggregations over millions of rows
Data modelNormalized (3NF — many tables, JOINs)Denormalized (star/snowflake schema)
Storage formatRow-basedColumn-based
Data volumeGBs to low TBsTBs to PBs
ConcurrencyThousands of concurrent small queriesDozens of concurrent complex queries
Response timeMilliseconds per transactionSeconds per analytical query
Example querySELECT * FROM orders WHERE id = 12345SELECT category, SUM(amount) FROM orders GROUP BY category

Why Columnar Storage is Faster for Analytics:

COLUMNAR STORAGE (Redshift):
┌────────────────────────────────────────┐
│ Column: order_id   │ 1, 2, 3, 4, ... 50M │
│ Column: date       │ 2024-01-15, 2024-01-15, ... │
│ Column: category   │ Shoes, Electronics, Shoes, ... │  ← Query reads ONLY this
│ Column: amount     │ 999, 2499, 1299, ... │           ← And this
│ Column: region     │ South, North, West, ... │
│ Column: customer_id│ 5678, 9012, 3456, ... │
│ ... (14 more columns stored separately) │
└────────────────────────────────────────┘

Query: SELECT category, SUM(amount) FROM orders GROUP BY category

Redshift: Reads ONLY "category" column (50M values) + "amount" column (50M values)
          Skips 18 other columns entirely
          Reads 2/20 = 10% of the data
          Compressed columnar data → even less I/O
          Parallel processing across nodes → distributed computation
          Result: 5 seconds (vs 47 minutes in MariaDB)

Single-Node Cluster (What ShopNow Will Start With):

→ When you create a cluster with 1 node → that node acts as BOTH leader AND compute
→ No separate leader node → simpler but limited to one node's resources
→ Sufficient for ShopNow's initial data volume (~520 GB)

Multi-Node Cluster (Future Growth):

→ When data grows beyond one node → add compute nodes
→ Leader node is added automatically (separate from compute)
→ Data is distributed across compute nodes
→ Queries run in parallel across all nodes → linear performance scaling

Enhanced VPC Routing — Why Enable:

→ Without Enhanced VPC Routing:

  • Redshift COPY/UNLOAD traffic goes through the PUBLIC internet
  • Data travels: Redshift → Internet → S3 endpoint
  • Even though S3 is in the same region → traffic exits VPC

→ With Enhanced VPC Routing:

  • All COPY/UNLOAD traffic stays within the VPC
  • Data travels: Redshift → VPC → VPC Endpoint or NAT Gateway → S3
  • Enables: VPC flow logs, VPC endpoints, NACLs for Redshift traffic
  • Required for compliance (data never leaves private network)

→ IMPORTANT: With Enhanced VPC Routing enabled:

  • Redshift must have a route to S3 (via NAT Gateway or VPC Endpoint)
  • VPC Endpoint for S3 (Project 27) is STRONGLY recommended → free, faster, no NAT Gateway cost
  • Without a route to S3 → COPY/UNLOAD commands FAIL

STEP 7 — UNDERSTAND COMPRESSION AND ENCODING

→ Redshift automatically compresses data in columnar format
→ Compression reduces storage AND improves query performance (less I/O)

How Columnar Compression Works:

UNCOMPRESSED column "category" (50 million values):
  Shoes, Electronics, Shoes, Clothing, Shoes, Electronics, Clothing, ...
  Size: ~500 MB

COMPRESSED column "category" with LZO encoding:
  Dictionary: {1=Shoes, 2=Electronics, 3=Clothing, 4=Books, ...}
  Values: 1, 2, 1, 3, 1, 2, 3, 1, 2, ...
  Size: ~50 MB (10:1 compression ratio)

COMPRESSED column "amount" with AZ64 encoding:
  Optimized for numeric data with patterns
  Size reduction: ~4:1 typical for numeric columns

Encoding Types:

EncodingBest ForHow It Works
AZ64Numeric, date/time columnsAWS proprietary — best for most numeric data
LZOVARCHAR, large text columnsGeneral-purpose compression
ZSTDMost column typesExcellent compression ratio, slightly more CPU
BYTEDICTLow-cardinality columns (< 256 unique values)Dictionary encoding
DELTAColumns with sequential/incremental valuesStores differences between values
RUNLENGTHColumns with many consecutive repeated valuesStores value + count
RAWNo compressionUse when data is already compressed or very small

Automatic Compression:

→ When you load data into Redshift using COPY command (Project 22):
→ Redshift can AUTOMATICALLY choose the best encoding for each column
→ COPY ... COMPUPDATE ON → analyzes sample data → applies optimal encoding

→ For ShopNow: Let Redshift auto-select encodings during initial data load
→ Manual encoding optimization in Project 22

✅ WHAT SHOULD DO

→ Always use RA3 node types for new clusters — managed storage separates compute from storage
→ Always place Redshift in private subnets — never publicly accessible for production
→ Always enable Enhanced VPC Routing — keeps COPY/UNLOAD traffic within VPC
→ Always enable encryption at rest — compliance requirement
→ Always enforce SSL connections — set require_ssl = true
→ Always associate an IAM Role for S3 access — needed for COPY/UNLOAD (Projects 22, 23)
→ Always enable audit logging — query history for compliance and performance analysis
→ Always set statement_timeout — prevent runaway queries from consuming resources
→ Always start with Auto WLM — switch to manual only when specific tuning is needed
→ Always enable concurrency scaling for burst handling — absorbs query spikes
→ Always monitor PercentageDiskSpaceUsed — cluster becomes read-only at 100%
→ Always create a Cluster Subnet Group with subnets in multiple AZs
→ Always take a manual snapshot before major changes (resize, version upgrade)

❌ WHAT SHOULD NOT DO

→ Never use Redshift for OLTP workloads (individual row lookups, frequent small inserts) → Use RDS (Project 9)
→ Never make Redshift publicly accessible — always use VPC + Security Groups
→ Never use DC2 nodes for growing datasets — storage is fixed and tied to compute
→ Never skip Enhanced VPC Routing — data will traverse the public internet
→ Never let PercentageDiskSpaceUsed reach 100% — cluster becomes read-only
→ Never run heavy ETL during peak analytics hours — use WLM queues or schedule off-hours
→ Never use the admin account for application queries — create separate users

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

1 → HOW COLUMNAR STORAGE WORKS

ROW-BASED (MariaDB):
  Disk Block 1: [order_id=1, date=2024-01-15, category=Shoes, amount=999, region=South]
  Disk Block 2: [order_id=2, date=2024-01-15, category=Electronics, amount=2499, region=North]
  Disk Block 3: [order_id=3, date=2024-01-16, category=Shoes, amount=1299, region=West]

  Query: SELECT category, SUM(amount) GROUP BY category
  → Must read: Block 1 (entire row) + Block 2 (entire row) + Block 3 (entire row)
  → Reads ALL columns even though only "category" and "amount" are needed
  → I/O wasted on: order_id, date, region (unused columns)

COLUMNAR (Redshift):
  Column Block "category": [Shoes, Electronics, Shoes, Clothing, Shoes, ...]
  Column Block "amount": [999, 2499, 1299, 599, 1899, ...]
  Column Block "order_id": [1, 2, 3, 4, 5, ...]     ← NOT read
  Column Block "date": [2024-01-15, 2024-01-15, ...]  ← NOT read
  Column Block "region": [South, North, West, ...]     ← NOT read

  Query: SELECT category, SUM(amount) GROUP BY category
  → Reads: ONLY "category" block + "amount" block
  → Skips: order_id, date, region blocks entirely
  → I/O reduced by: (20 columns - 2 used) / 20 = 90% less I/O

Additional Columnar Benefits:

BenefitHow It Works
Better compressionSame column = same data type = similar values = compresses well
Zone mapsRedshift tracks min/max per block → skips blocks that can't match WHERE clause
Late materializationOnly assembles full rows at the very end (after filtering)
Vectorized executionProcesses batches of column values using CPU SIMD instructions

2 → HOW MASSIVELY PARALLEL PROCESSING (MPP) WORKS

Query: SELECT category, SUM(amount) FROM orders WHERE year = 2024 GROUP BY category

LEADER NODE:
  Step 1: Parse SQL
  Step 2: Create execution plan
  Step 3: Distribute plan to compute nodes

COMPUTE NODE 1 (Slices 0-1):
  Holds: orders where customer_id hashed to slices 0-1
  Scans: ~12.5 million rows (25% of 50M)
  Filters: year = 2024 → ~6 million rows
  Aggregates locally: {Shoes: 50M, Electronics: 80M, ...}
  Sends partial results to leader

COMPUTE NODE 2 (Slices 2-3):
  Holds: orders where customer_id hashed to slices 2-3
  Scans: ~12.5 million rows (25% of 50M)
  Same process → partial results to leader

(All nodes work SIMULTANEOUSLY — parallel execution)

LEADER NODE:
  Step 4: Receive partial results from all nodes
  Step 5: Merge/aggregate: {Shoes: 50M+48M+52M+49M, Electronics: ...}
  Step 6: Return final result to client

Total time: ~5 seconds (each node processes 25% of data in parallel)
Compare: MariaDB processes 100% sequentially → 47 minutes

3 → HOW REDSHIFT MANAGED STORAGE (RMS) WORKS

RA3 NODE ARCHITECTURE:
┌────────────────────────────────────────────────────────┐
│ RA3 Compute Node                                       │
│                                                        │
│ ┌──────────────────────────────────────────────────┐   │
│ │ LOCAL SSD CACHE (Managed by Redshift)             │   │
│ │                                                    │   │
│ │ Hot data: Frequently accessed columns/blocks       │   │
│ │ Automatic: Redshift decides what to cache           │   │
│ │ Fast: Local NVMe SSD → microsecond access           │   │
│ └──────────────────────────────┬─────────────────────┘   │
│                                │                         │
│                                │ Cache miss              │
│                                ▼                         │
│ ┌──────────────────────────────────────────────────┐   │
│ │ REDSHIFT MANAGED STORAGE (S3-backed)              │   │
│ │                                                    │   │
│ │ All data: Complete dataset stored in S3             │   │
│ │ Durable: 11 9s durability (S3 — Project 1)         │   │
│ │ Scalable: Up to 64 TB per node                      │   │
│ │ Transparent: Appears as local storage to queries    │   │
│ └──────────────────────────────────────────────────┘   │
│                                                        │
└────────────────────────────────────────────────────────┘

→ RA3 nodes separate compute from storage
→ Frequently queried data → cached on local SSD → fast access
→ Rarely queried data → in S3 → fetched on demand → cached for next time
→ You pay for: Compute (per node-hour) + Storage (per GB-month in RMS)
→ Storage grows independently of compute → add data without adding nodes

📈 SCALABILITY CONSIDERATIONS

→ Compute Scalability

  • Add nodes: 2 → 4 → 8 → 16 → 128 nodes (classic resize or elastic resize)
  • Elastic resize: Adds nodes in minutes (brief unavailability)
  • Classic resize: Redistributes data (hours for large clusters)
  • Concurrency scaling: Automatic burst capacity for query spikes (seconds to activate)

→ Storage Scalability

  • RA3: Up to 64 TB per node → 128 nodes × 64 TB = 8+ PB theoretical
  • Managed storage grows automatically — no provisioning
  • Redshift Spectrum (Project 28): Query S3 data directly → unlimited "storage"

→ Query Scalability

  • Auto WLM: Up to 50 concurrent queries
  • Concurrency scaling: Adds temporary clusters for burst queries
  • Materialized views: Pre-compute common aggregations → instant results
  • Result caching: Repeated queries return cached results in milliseconds

→ Data Loading Scalability

  • COPY command: Parallel loading from S3 across all slices
  • Multiple files → parallel ingestion → TB/hour loading rates
  • Streaming ingestion: Near-real-time loading from Kinesis (Project 52)
