PROJECT 36: THE AUDITOR WHO FOUND UNENCRYPTED DATA EVERYWHERE — KMS Encryption for S3, RDS & Redshift

🔴 HYPOTHETICAL PROBLEM

→ ShopNow is preparing for their first enterprise compliance audit (SOC 2 Type II)
→ The auditor "David" runs a security assessment across their AWS environment
→ His findings are alarming:

Finding 1 — S3 Data Lake
→ 12 S3 buckets exist across the organization
→ 8 buckets use SSE-S3 encryption (configured in Project 1)
→ 4 buckets have NO default encryption at all
→ Nobody can answer: "Who accessed the encryption keys? When? For which objects?"
→ SSE-S3 provides encryption but ZERO audit trail — AWS manages the keys invisibly

Finding 2 — RDS MariaDB
→ The production MariaDB instance (Project 9) was launched WITHOUT encryption at rest
→ At the time, the team thought "it's in a private subnet, so it's secure"
→ The auditor explains: "Network isolation is NOT encryption. If someone copies the underlying EBS volume, data is readable in plain text"
→ Snapshots of this database are also unencrypted — anyone with snapshot access reads raw data

Finding 3 — Redshift Cluster
→ The Redshift cluster (Project 16) was launched without encryption
→ Contains 18 months of customer purchase history, PII (names, emails, phone numbers)
→ The auditor marks this as a CRITICAL finding

Finding 4 — No Key Management Strategy
→ No centralized encryption key management exists
→ No key rotation policy
→ No key access audit trail
→ No separation between "who can use a key" and "who can manage a key"
→ No ability to revoke encryption access independently from data access

The Auditor's Mandate:

"Encrypt everything at rest using keys YOU control with full audit trail.
Implement key rotation. Separate key administration from key usage.
Prove you can answer: WHO used WHICH key, WHEN, for WHAT purpose."

→ This requires AWS KMS (Key Management Service) as the centralized encryption backbone

🟡 HOW TO APPROACH IT

→ First — understand WHY SSE-S3 (from Project 1) is insufficient for compliance

SSE-S3 (What we configured in Project 1):
→ AWS generates the encryption key
→ AWS manages the encryption key
→ AWS rotates the encryption key
→ You have ZERO visibility into key usage
→ You CANNOT audit who decrypted what
→ You CANNOT restrict decryption separately from S3 access
→ You CANNOT disable or revoke the key
→ For basic protection — sufficient
→ For compliance, governance, auditing — insufficient

SSE-KMS (What this project introduces):
→ YOU create the encryption key (or AWS creates one you control)
→ YOU define who can use the key (key policy)
→ YOU define who can manage the key (key administrators)
→ Every key usage is logged in CloudTrail automatically
→ You CAN rotate the key automatically
→ You CAN disable or schedule deletion of the key
→ You CAN grant/revoke access independently from the data service

→ The approach:

  • Step 1 — Understand KMS key hierarchy and types
  • Step 2 — Create a Customer Managed Key (CMK) for S3
  • Step 3 — Create a Customer Managed Key for RDS
  • Step 4 — Create a Customer Managed Key for Redshift
  • Step 5 — Apply KMS encryption to S3 buckets
  • Step 6 — Handle the RDS encryption problem (cannot encrypt existing unencrypted instance)
  • Step 7 — Handle the Redshift encryption migration
  • Step 8 — Configure key rotation
  • Step 9 — Verify audit trail in CloudTrail

🟢 HOW CAN WE SOLVE IT — Step-by-Step Configuration

STEP 1 — UNDERSTAND KMS KEY HIERARCHY AND TYPES

→ Before creating anything — understand what KMS actually manages

The Encryption Hierarchy:

                    ┌──────────────────────┐
                    │   KMS Master Key     │
                    │   (CMK / KMS Key)    │
                    │   Never leaves KMS   │
                    └──────────┬───────────┘
                               │
                    Generates / Encrypts
                               │
                    ┌──────────▼───────────┐
                    │   Data Encryption    │
                    │   Key (DEK)          │
                    │   Used to encrypt    │
                    │   your actual data   │
                    └──────────┬───────────┘
                               │
                    Encrypts
                               │
                    ┌──────────▼───────────┐
                    │   Your Data          │
                    │   (S3 object, RDS    │
                    │   volume, Redshift   │
                    │   block)             │
                    └──────────────────────┘

This is called "Envelope Encryption":
→ The KMS Master Key encrypts the Data Encryption Key (DEK)
→ The DEK encrypts the actual data
→ The encrypted DEK is stored alongside the encrypted data
→ The KMS Master Key NEVER leaves KMS — it never touches your data directly
→ To decrypt: KMS decrypts the DEK → DEK decrypts the data

Why envelope encryption?
→ KMS has API rate limits (typically 5,500–30,000 requests/second depending on key type and region)
→ If KMS encrypted every byte directly — it would be a bottleneck
→ Instead — KMS only encrypts/decrypts the small DEK
→ The DEK handles the bulk encryption locally (in S3, RDS, Redshift infrastructure)
→ This combines KMS security with high throughput

Three Types of KMS Keys:

Key TypeWho CreatesWho ManagesWho PaysAudit TrailKey Policy ControlRotation Control
AWS Owned KeysAWSAWSFree❌ No❌ NoAWS decides
AWS Managed KeysAWS (auto-created per service)AWSFree (but usage charges may apply)✅ Yes (CloudTrail)❌ No (cannot modify policy)Auto every year (mandatory)
Customer Managed KeysYouYou$1/month per key + $0.03 per 10,000 API calls✅ Yes (CloudTrail)✅ Full controlYou configure (optional auto-rotate)

What the auditor requires → Customer Managed Keys (CMKs)
→ Full control over key policy
→ Full audit trail
→ Ability to disable/revoke
→ Configurable rotation
→ Separation of duties (key admin ≠ key user)

AWS Managed Key Naming Convention:
→ When a service like S3 or RDS first uses KMS, AWS auto-creates an AWS Managed Key
→ These appear as aws/s3, aws/rds, aws/redshift in the KMS console
→ You CANNOT modify their key policy
→ You CANNOT disable them
→ You CANNOT delete them
→ They exist per service per region per account
→ The auditor considers these INSUFFICIENT because you cannot control access granularity

→ Navigate to the created key → "Key policy" tab

Key Policy Structure (Conceptual — Settings view):

StatementPrincipalActionsPurpose
Enable IAM User Permissionsarn:aws:iam::123456789012:rootkms:*Allows the account root to manage this key — SAFETY NET
Key AdministratorsSecurityTeamAdminRole, security-admin-davidkms:Create*, kms:Describe*, kms:Enable*, kms:List*, kms:Put*, kms:Update*, kms:Revoke*, kms:Disable*, kms:Get*, kms:Delete*, kms:TagResource, kms:UntagResource, kms:ScheduleKeyDeletion, kms:CancelKeyDeletionManage the key lifecycle
Key UsersGlueETLRole, AthenaQueryRole, EC2AppRole, etc.kms:Encrypt, kms:Decrypt, kms:ReEncrypt*, kms:GenerateDataKey*, kms:DescribeKeyUse the key for data encryption/decryption
Grant PermissionsSame Key Userskms:CreateGrant, kms:ListGrants, kms:RevokeGrantAllow AWS services to use the key on behalf of the user (explained below)

CRITICAL: The Root Account Statement
→ The first statement grants kms:* to arn:aws:iam::123456789012:root
→ This does NOT mean the root user can bypass everything
→ It means: IAM policies in this account CAN grant KMS permissions for this key
→ Without this statement → Even IAM Admin cannot grant anyone access to this key
→ This is the SAFETY NET — if you lock yourself out of the key policy, IAM can still recover access
→ NEVER remove this statement — you risk permanent key lockout

CRITICAL: KMS Grants
→ The "Grant Permissions" statement allows key users to create KMS Grants
→ Grants are temporary, scoped permissions that AWS services create automatically
→ Example: When S3 needs to encrypt an object, it creates a grant to use the key for that specific operation
→ Grants are how AWS services interact with KMS without you manually invoking KMS APIs
→ Without grant permissions → S3, RDS, Redshift CANNOT use the key even if listed as key users

→ Navigate to S3 → Select shopnow-datalake-prod bucket
→ Properties tab → Default encryption → Edit

SettingPrevious Value (Project 1)New Value

→ Save changes

What "Bucket Key" Does (Important cost optimization):

WITHOUT Bucket Key:
    Every object upload → S3 calls KMS GenerateDataKey → unique DEK per object
    1 million objects = 1 million KMS API calls
    Cost: 1,000,000 × $0.03/10,000 = $3.00

WITH Bucket Key:
    S3 generates a bucket-level key FROM KMS → uses that to derive per-object keys locally
    1 million objects ≈ a few KMS API calls (for the bucket key)
    Cost: Negligible
    Savings: Up to 99% reduction in KMS API costs

→ Bucket Key is a MUST for high-volume buckets
→ Trade-off: CloudTrail logs show the bucket-level key ARN, not individual object-level KMS calls
→ For most compliance needs — this is acceptable
→ If you need per-object KMS audit → Disable Bucket Key (very rare requirement)

What Happens to Existing Objects?
→ Changing default encryption does NOT re-encrypt existing objects
→ Existing objects remain encrypted with SSE-S3
→ Only NEW objects uploaded after the change use SSE-KMS
→ To re-encrypt existing objects → Use S3 Batch Operations or copy objects in place

Re-encrypting Existing Objects:
→ Navigate to S3 → Management tab → "Create Batch Operations job"
→ Or use the simpler approach:

  • S3 → Select objects → Actions → "Copy" → Copy to same bucket/same key
  • Under encryption → Select SSE-KMS with your key
  • This creates a new version (if versioning enabled from Project 12) encrypted with KMS

→ Repeat for all buckets that need KMS encryption

STEP 6 — HANDLE THE RDS ENCRYPTION PROBLEM

→ This is the most painful finding — and a critical lesson

THE HARD TRUTH:
→ You CANNOT enable encryption on an existing unencrypted RDS instance
→ There is NO "toggle encryption ON" button
→ This is an AWS architectural limitation — encryption is set at volume creation time
→ The underlying EBS volumes were created without encryption — they cannot be retroactively encrypted

        ↓

STEP 7 — HANDLE THE REDSHIFT ENCRYPTION MIGRATION

→ Redshift has a BETTER story than RDS — encryption CAN be enabled on existing clusters

Two Options:

Option A — Modify Existing Cluster (Preferred for Redshift)
→ Navigate to Redshift Console → Select cluster → Actions → "Modify"

SettingPrevious ValueNew Value
EncryptionDisabledEnable
KMS KeyN/Ashopnow-redshift-warehouse-key

→ Click "Modify cluster"

What Happens Internally:
→ Redshift performs an online encryption migration
→ The cluster remains available during the migration
→ Behind the scenes:

  • Redshift creates a new encrypted cluster
  • Migrates all data from old to new
  • Swaps the endpoint to the new cluster
  • Deletes the old unencrypted cluster
    → The endpoint DOES NOT change — applications are unaffected
    → Duration depends on data volume — can take hours for large clusters
    → Cluster status shows "resizing" during migration

Option B — Snapshot and Restore (If Option A fails or for cross-region)
→ Similar to RDS approach:

  • Create snapshot → Copy with encryption → Restore
  • But this creates a NEW endpoint (unlike Option A)

→ Option A is preferred because it preserves the endpoint

STEP 8 — CONFIGURE KEY ROTATION

→ Navigate to KMS → Select each key → "Key rotation" tab

For Customer Managed Keys:

SettingValue
Automatic key rotation✅ Enable
Rotation periodEvery 365 days (only option for automatic rotation)

What Rotation Actually Does:

Year 1: KMS Key has backing key version 1 (BK-1)
         All new data encrypted with BK-1
         
Year 2: Rotation creates backing key version 2 (BK-2)
         New data encrypted with BK-2
         Old data still encrypted with BK-1
         The KMS Key ARN/Alias does NOT change
         KMS automatically knows which backing key to use for decryption
         
Year 3: BK-3 created → new data uses BK-3
         BK-1 and BK-2 data still decryptable
         Old backing keys are NEVER deleted (as long as key exists)

→ This is TRANSPARENT rotation — no re-encryption needed
→ The key ID/ARN remains the same
→ Applications need ZERO changes
→ Old data remains accessible — KMS tracks which backing key encrypted which data
→ Cost: No additional charge for rotated backing keys

Why the Auditor Cares:
→ If a backing key is ever compromised, new data uses a new backing key
→ Rotation limits the blast radius of key compromise
→ Compliance frameworks (SOC2, PCI-DSS, HIPAA) require key rotation policies

✅ WHAT SHOULD DO

→ Always use Customer Managed Keys (CMK) for compliance-sensitive workloads
→ Always separate key administrators from key users — enforce least privilege
→ Always enable automatic key rotation on every Customer Managed Key
→ Always enable S3 Bucket Key when using SSE-KMS — reduces cost by up to 99%
→ Always keep the root account statement in key policies — prevents lockout
→ Always use separate KMS keys per service category (S3, RDS, Redshift, CloudTrail)
→ Always verify that service IAM roles have both IAM permissions AND key policy permissions
→ Always tag KMS keys with service, environment, compliance framework
→ Always test encryption by uploading a test object and verifying you can read it back
→ Always update Secrets Manager secrets when migrating to encrypted RDS instances (new endpoint)
→ Always check that Glue, Athena, Lambda roles have kms:Decrypt and kms:GenerateDataKey on the S3 KMS key
→ Always verify CloudTrail is capturing KMS events — this IS the audit trail

❌ WHAT SHOULD NOT DO

→ Never remove the root account statement from key policy — you risk permanent key lockout
→ Never use the same KMS key for data encryption AND audit log encryption — defeats audit integrity
→ Never grant kms:* to application roles — only grant specific actions (Encrypt, Decrypt, GenerateDataKey)
→ Never assume changing S3 default encryption re-encrypts existing objects — it only affects NEW objects
→ Never attempt to enable encryption on existing unencrypted RDS — it is architecturally impossible
→ Never forget to re-create Read Replicas after encrypting the primary RDS (replicas must be recreated)
→ Never delete a KMS key that still has encrypted data — all data becomes PERMANENTLY unreadable
→ Never schedule key deletion with the minimum 7-day waiting period without thorough impact analysis
→ Never hardcode KMS key ARNs in application code — use aliases or parameter store references
→ Never create one key per object or per table — this creates key management explosion
→ Never disable a key "temporarily" without communicating to all teams — it instantly breaks all services using it
→ Never skip Bucket Key for high-volume S3 buckets — KMS API costs can become significant

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

MECHANIC 1 — ENVELOPE ENCRYPTION FLOW (S3 Example)

Encryption (Object Upload):

Application uploads object to S3 bucket with SSE-KMS
        ↓
S3 calls KMS: GenerateDataKey(KeyId = shopnow-s3-datalake-key)
        ↓
KMS returns TWO things:
  1. Plaintext Data Encryption Key (DEK)
  2. Encrypted DEK (encrypted using the CMK)
        ↓
S3 uses the plaintext DEK to encrypt the object data using AES-256-GCM
        ↓
S3 stores:
  - Encrypted object data
  - Encrypted DEK (as object metadata)
        ↓
S3 DISCARDS the plaintext DEK from memory
        ↓
KMS logs the GenerateDataKey call in CloudTrail

Decryption (Object Download):

Application requests object from S3
        ↓
S3 reads the encrypted DEK from object metadata
        ↓
S3 calls KMS: Decrypt(CiphertextBlob = encrypted DEK)
        ↓
KMS checks:
  1. Is the caller authorized in the key policy? → Yes/No
  2. Is the caller authorized in IAM policy? → Yes/No
  3. Is the key enabled? → Yes/No
        ↓
If all checks pass:
  KMS returns the plaintext DEK
        ↓
S3 uses the plaintext DEK to decrypt the object data
        ↓
S3 returns decrypted data to the application
        ↓
S3 DISCARDS the plaintext DEK from memory
        ↓
KMS logs the Decrypt call in CloudTrail

Key insight: The CMK NEVER leaves KMS. The CMK never touches the data. The CMK only encrypts/decrypts the DEK.

MECHANIC 2 — DOUBLE AUTHORIZATION MODEL

→ To use a KMS key, the caller must be authorized in BOTH:

┌──────────────────────┐     ┌──────────────────────┐
│    KMS Key Policy     │     │    IAM Policy         │
│  (attached to key)    │ AND │  (attached to role)   │
│                       │     │                       │
│  Must ALLOW the       │     │  Must ALLOW the       │
│  caller to perform    │     │  caller to perform    │
│  the KMS action       │     │  the KMS action       │
└──────────────────────┘     └──────────────────────┘
              │                          │
              └──────────┬───────────────┘
                         ↓
                  BOTH must allow
                  Either denying = DENIED

Example Scenario:

IAM PolicyKey PolicyResult
Allows kms:Decrypt on key ARNLists role as Key User✅ Access Granted
Allows kms:Decrypt on key ARNDoes NOT list role❌ Access Denied
Does NOT allow kms:DecryptLists role as Key User❌ Access Denied
Allows kms:Decrypt on *Lists role as Key User✅ Access Granted (but IAM * is bad practice)

→ This dual-check is WHY KMS provides stronger access control than SSE-S3
→ With SSE-S3 → If you have S3 GetObject permission, you automatically get decryption
→ With SSE-KMS → You need S3 GetObject permission AND KMS Decrypt permission → Two independent gates

MECHANIC 3 — KMS GRANT MECHANISM (How AWS Services Use Keys)

→ When RDS or Redshift starts up with a KMS-encrypted configuration:

RDS Service starts encrypted instance
        ↓
RDS calls KMS: CreateGrant(
    KeyId = shopnow-rds-mariadb-key
    GranteePrincipal = RDS internal service principal
    Operations = [Decrypt, Encrypt, GenerateDataKey, ReEncrypt, CreateGrant, RetireGrant]
    RetiringPrincipal = RDS internal service principal
)
        ↓
KMS creates a Grant — a temporary, scoped permission
        ↓
RDS uses this Grant for all encryption/decryption operations
during the instance lifecycle
        ↓
When instance is deleted:
    RDS calls RetireGrant → Grant is removed

→ Grants are separate from the Key Policy
→ Grants allow services to operate without modifying the key policy
→ You can view active grants: KMS Console → Key → "Grants" tab
→ Each grant shows: Grantee, Operations allowed, Creation date

CRITICAL WARNING:
→ Key deletion is the MOST DANGEROUS operation in KMS
→ If you delete a key that encrypted S3 objects → Those objects become permanently unreadable
→ If you delete a key that encrypted RDS volumes → That database becomes permanently inaccessible
→ The minimum waiting period is 7 days — use the maximum 30 days for production keys
→ During the waiting period — monitor CloudTrail for any Decrypt calls using this key
→ If ANY calls appear → Cancel deletion — someone still needs this key

MECHANIC 5 — RDS ENCRYPTION INTERNALS

How RDS Uses KMS:

RDS Instance with KMS Encryption
        ↓
Underlying EBS Volumes are encrypted with:
  - A volume-specific DEK
  - The DEK is encrypted by your KMS CMK
        ↓
Every I/O operation:
  - EBS decrypts data transparently
  - The DEK is cached in memory on the host
  - KMS is NOT called for every read/write (that would be too slow)
  - KMS is only called at volume mount time (instance start)
        ↓
RDS Snapshots:
  - Snapshots inherit the encryption key of the source instance
  - Encrypted instance → encrypted snapshot (automatically)
  - You CAN copy a snapshot with a DIFFERENT KMS key (for cross-account sharing)
        ↓
Read Replicas:
  - Must use the SAME KMS key as primary (same region)
  - Cross-region replica CAN use a different KMS key (key is region-specific)

MECHANIC 6 — REDSHIFT ENCRYPTION INTERNALS

How Redshift Uses KMS:

Redshift Cluster with KMS Encryption
        ↓
Four-tier encryption hierarchy:
  1. KMS CMK (your Customer Managed Key)
  2. Cluster Encryption Key (CEK) — encrypted by CMK
  3. Database Encryption Key (DEK) — encrypted by CEK
  4. Data Blocks — encrypted by DEK using AES-256
        ↓
Key rotation in Redshift:
  - When you rotate the KMS key → Redshift re-encrypts the CEK
  - The DEK and data blocks are NOT re-encrypted (too expensive)
  - This is transparent and does not require downtime
        ↓
Snapshots:
  - Encrypted snapshots use the same KMS key
  - Cross-region snapshot copy requires a "snapshot copy grant"
    in the destination region with a KMS key in that region

MECHANIC 7 — CROSS-SERVICE KMS PERMISSION FLOW

→ When Athena reads SSE-KMS encrypted Parquet files from S3 (combining Projects 18 and 36):

Analyst runs Athena query
        ↓
Athena assumes the Workgroup execution role (e.g., AthenaQueryRole)
        ↓
AthenaQueryRole needs:
  1. s3:GetObject on the S3 bucket (IAM policy) — from Project 18
  2. kms:Decrypt on the S3 KMS key (IAM policy + Key Policy) — from THIS project
        ↓
For each Parquet file:
  Athena calls S3 GetObject
  S3 calls KMS Decrypt (using AthenaQueryRole's permissions)
  KMS checks both key policy AND IAM policy
  If authorized → DEK decrypted → Parquet file decrypted → data returned to Athena

→ If the AthenaQueryRole is listed as a key user in the key policy BUT the IAM policy attached to the role does NOT include kms:Decrypt → Access Denied
→ BOTH must align — this is the most common source of "everything was working with SSE-S3 but broke when I switched to SSE-KMS"

📈 SCALABILITY CONSIDERATIONS

→ KMS API Request Quotas

Request TypeDefault Quota (per second per key)Notes
Cryptographic operations (Encrypt, Decrypt, GenerateDataKey)5,500 – 30,000 (varies by region and key type)Shared across all users of the key
Symmetric key operationsHigher quotaMost common for AWS service encryption
Asymmetric key operationsLower quotaNot used for service encryption

→ With S3 Bucket Key enabled → Quota is rarely an issue
→ Without Bucket Key + millions of objects per hour → Can hit throttling
→ Solution: Enable Bucket Key, request quota increase via AWS Support

→ Key Count Scaling

  • No hard limit on number of Customer Managed Keys per account
  • Soft limit: Typically 100,000 keys per account per region
  • Cost: $1/month per key (including disabled keys awaiting deletion)
  • Best practice: Use per-service or per-classification keys, NOT per-resource keys
  • Example: 1 key for all S3 buckets, not 1 key per bucket

→ Cross-Account Key Scaling (Preview for Project 39)

  • Customer Managed Keys CAN be shared cross-account via key policy
  • Add the external account's root ARN to the key policy
  • External account then grants access to specific roles via IAM
  • AWS Managed Keys CANNOT be shared cross-account
  • This is why compliance workloads MUST use Customer Managed Keys

→ Multi-Region Key Scaling

  • KMS supports Multi-Region Keys (MRK) — replicas of a key in multiple regions
  • Useful when data is replicated cross-region (S3 CRR from Project 30, cross-region RDS from Project 55)
  • Multi-Region Keys have the same key material in all regions
  • Data encrypted in Region A can be decrypted in Region B without cross-region KMS calls

→ Encryption Performance Impact

ServicePerformance Impact of KMS Encryption
S3Negligible — encryption/decryption handled at infrastructure level
RDSNegligible — EBS encryption is transparent, DEK cached in memory
RedshiftNegligible for most workloads — slight overhead during key rotation
AthenaNegligible — S3 handles decryption before data reaches Athena
GlueNegligible — same as S3/RDS
LambdaMay add 10-50ms per KMS API call if not using Bucket Key
