PROJECT 45: THE LAMBDA THAT NEEDED PRIVATE DATABASE AND PUBLIC BUCKET — Lambda in VPC Accessing RDS + S3

🔴 HYPOTHETICAL PROBLEM

→ PayScale (from Project 44) has a critical business requirement
→ The Risk Team in Account B runs a fraud scoring Lambda function
→ This Lambda function must do THREE things in sequence:

Step 1: Read transaction data from MariaDB in Account B's PRIVATE subnet
        (RDS MariaDB configured in Project 9, placed in private subnet in Project 14)

Step 2: Call an external third-party fraud scoring API over the internet
        (https://api.fraudcheck.io/v2/score)

Step 3: Write the fraud score results to an S3 bucket in Account B
        (S3 bucket configured in Project 1)

→ The Lambda function was deployed without VPC configuration (default Lambda)
→ Step 3 works perfectly — Lambda can reach S3 over the internet ✅
→ Step 2 works perfectly — Lambda can reach external APIs over the internet ✅
→ Step 1 FAILS — Lambda CANNOT reach the MariaDB in the private subnet ❌

Error: Connection timed out: 10.1.3.45:3306

→ The developer moves Lambda INTO the VPC (attaches it to the same VPC as MariaDB)
→ Now Step 1 works — Lambda can reach MariaDB ✅
→ But Step 2 FAILS — Lambda can no longer reach the internet ❌
→ And Step 3 FAILS — Lambda can no longer reach S3 ❌

Error: Task timed out after 30 seconds (trying to reach S3 or external API)

→ The developer is stuck in a paradox:

  • Lambda OUTSIDE VPC → Can reach internet + S3, CANNOT reach private RDS
  • Lambda INSIDE VPC → Can reach private RDS, CANNOT reach internet + S3

→ The CTO says: "This function must access ALL THREE — private database, public S3, and external API — without compromise"

🟡 HOW TO APPROACH IT

→ First — understand WHY this paradox exists

Lambda Default (No VPC):
→ Lambda runs in an AWS-managed VPC (invisible to you)
→ It has a public IP assigned by AWS
→ It can reach ANY public endpoint (S3, DynamoDB, external APIs)
→ It CANNOT reach anything inside YOUR VPC's private subnets
→ Because YOUR private subnets are isolated — that's the entire point of private subnets (from Project 5)

Lambda in VPC:
→ Lambda creates an ENI (Elastic Network Interface) inside YOUR VPC subnet
→ The ENI gets a PRIVATE IP from your subnet's CIDR range
→ Lambda now behaves like an EC2 instance in that subnet
→ If the subnet is PRIVATE (no Internet Gateway route) → Lambda has NO internet access
→ S3 is a PUBLIC service — its API endpoint is s3.ap-south-1.amazonaws.com — reachable only via internet or VPC endpoint

The Solution Architecture:

Lambda (in Private Subnet)
    |
    ├──→ MariaDB (Private Subnet, same VPC) → Direct via private IP ✅
    |
    ├──→ S3 (Public Service) → Via VPC Gateway Endpoint (from Project 27) ✅
    |
    └──→ External API (Internet) → Via NAT Gateway (from Project 5) ✅

→ The approach should be:

  • Step 1 — Understand Lambda ENI mechanics
  • Step 2 — Configure Lambda VPC settings (subnet + security group)
  • Step 3 — Ensure NAT Gateway exists for internet access
  • Step 4 — Ensure VPC Gateway Endpoint exists for S3 access
  • Step 5 — Configure Security Groups for Lambda → RDS communication
  • Step 6 — Configure Lambda Execution Role with correct IAM permissions
  • Step 7 — Handle cold start implications
  • Step 8 — Test all three connectivity paths

→ Why NOT in public subnet?

  • Lambda ENIs only get private IPs — unlike EC2 which can have Elastic IPs

  • Lambda needs NAT Gateway access for external API calls
  • Placing Lambda in the DB subnet would block internet access entirely

  • Lambda gets private IP → can reach RDS in DB subnet (same VPC, routable)
  • Lambda can reach internet via NAT Gateway → external API access
  • Lambda can reach S3 via VPC Gateway Endpoint → S3 access without internet

→ Always create a VPC Gateway Endpoint for S3 — saves NAT Gateway processing costs and reduces latency
→ Always attach AWSLambdaVPCAccessExecutionRole managed policy to the Lambda execution role
→ Always create a DEDICATED security group for Lambda — do not reuse application server or RDS security groups
→ Always set Lambda timeout to accommodate VPC networking overhead (minimum 30–60 seconds)
→ Always use Secrets Manager (from Project 37) for database credentials — never hardcode in Lambda configuration
→ Always monitor ENI usage in the VPC — Lambda consumes ENIs from your subnet's IP space
→ Always test all three connectivity paths independently before deploying to production
→ Always use VPC Flow Logs to debug connectivity issues — filter by Lambda ENI's private IP
→ Always tag Lambda security groups clearly: Purpose: Lambda-FraudScoring-ENI

❌ WHAT SHOULD NOT DO

→ Never place Lambda in a public subnet expecting internet access — Lambda ENIs NEVER get public IPs regardless of subnet type
→ Never place Lambda in database/isolated subnets if it needs internet or S3 access
→ Never forget AWSLambdaVPCAccessExecutionRole — Lambda deployment succeeds but invocations fail with ENI creation errors
→ Never select only ONE subnet — AZ failure will make Lambda non-functional
→ Never route S3 traffic through NAT Gateway when a VPC Endpoint exists — wastes money and adds latency
→ Never use overly broad security groups (0.0.0.0/0 inbound) on Lambda — Lambda doesn't need inbound rules
→ Never set Lambda timeout to 3 seconds for VPC Lambda — VPC networking adds overhead
→ Never hardcode RDS endpoint IPs — use DNS names (IPs change during failover, from Project 29)
→ Never forget to account for Lambda ENI IP consumption in subnet CIDR planning
→ Never assume Lambda VPC config is instant — first invocation after config change may take longer
→ Never put Lambda in VPC unnecessarily — if Lambda only needs S3 and DynamoDB (public endpoints), keep it outside VPC

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

1 → THE ENI LIFECYCLE (How Lambda Gets Network Access)

Pre-2019 Behavior (Legacy — Understanding for Context):

Cold Start (OLD):
1. Lambda receives invocation
2. Lambda service creates a NEW ENI in your VPC subnet
3. ENI creation takes 10–30 seconds (!!)
4. ENI gets a private IP from your subnet
5. Lambda execution environment attaches to the ENI
6. Function code executes
7. After idle timeout — ENI is deleted

→ This caused MASSIVE cold starts (10-30 seconds)
→ Made VPC Lambda nearly unusable for latency-sensitive workloads

Post-2019 Behavior (Current — Hyperplane):

When Lambda VPC Config is SAVED (not at invocation time):
1. Lambda pre-creates ENIs using AWS Hyperplane (a shared NAT-like system)
2. These ENIs are shared across ALL concurrent executions of the function
3. ENIs persist as long as the Lambda function exists with VPC config

Cold Start (NEW):
1. Lambda receives invocation
2. Lambda attaches execution environment to EXISTING Hyperplane ENI
3. Attachment takes ~1-2 seconds (vs 10-30 seconds previously)
4. Function code executes
5. Subsequent invocations reuse the same Hyperplane network tunnel

→ Cold start overhead reduced from 10-30 seconds to ~1-2 seconds
→ ENIs are NOT created per-invocation — they are shared infrastructure

Hyperplane Architecture:

                    AWS Hyperplane (Shared Network Infrastructure)
                    ┌─────────────────────────────────────────┐
                    │                                         │
Lambda Exec Env 1 ──┤   Hyperplane ENI (10.1.2.50)           │
Lambda Exec Env 2 ──┤   in subnet-private-1a                 │── Your VPC
Lambda Exec Env 3 ──┤                                         │
     ...           │   Hyperplane ENI (10.1.3.60)           │
Lambda Exec Env N ──┤   in subnet-private-1b                 │
                    └─────────────────────────────────────────┘

→ Multiple Lambda execution environments SHARE a single Hyperplane ENI
→ This is why Lambda no longer consumes one ENI per concurrent execution
→ IP consumption is significantly reduced compared to the old model

2 → IP ADDRESS CONSUMPTION MODEL

Old Model (Pre-2019):
→ Each concurrent Lambda execution = 1 ENI = 1 private IP
→ 1000 concurrent executions = 1000 IPs consumed from your subnet
→ A /24 subnet (256 IPs, 251 usable) would be exhausted by 251 concurrent Lambdas

New Model (Hyperplane):
→ Lambda creates a small number of Hyperplane ENIs per function per subnet
→ Exact formula is not publicly documented but roughly:

  • Initial: ~2 ENIs per subnet
  • Scales based on concurrent execution count but MUCH fewer than 1:1
    → A /24 subnet can now support thousands of concurrent Lambda executions

However — still plan for ENI capacity:

Subnet CIDRTotal IPsAWS ReservedUsable IPsComfortable Lambda Capacity
/242565251High — thousands of concurrent executions
/251285123Medium — still comfortable
/2664559Low — monitor carefully
/2732527Very Low — risk of IP exhaustion
/2816511Dangerous — only for small functions

→ AWS reserves 5 IPs per subnet:

  • .0 = Network address
  • .1 = VPC router
  • .2 = DNS server
  • .3 = Reserved for future use
  • .255 = Broadcast address (last IP in range)

4 → HOW VPC GATEWAY ENDPOINT INTERCEPTS S3 TRAFFIC

→ When Lambda calls S3 → it resolves s3.ap-south-1.amazonaws.com to a PUBLIC IP
→ The route table has a PREFIX LIST entry for S3 IP ranges

Lambda: "Send packet to 52.219.62.1 (S3 IP)"
    |
    ↓
Route Table Evaluation:
    - 10.1.0.0/16 → local? NO (52.219.x.x is not in VPC CIDR)
    - pl-xxx (S3 prefix list) → vpce-xxx? YES ✅ (52.219.x.x is in S3 IP range)
    - 0.0.0.0/0 → nat-xxx? NOT EVALUATED (more specific match found above)
    |
    ↓
Packet goes to VPC Endpoint → AWS backbone → S3

→ The PREFIX LIST is maintained by AWS — it automatically includes all S3 IP ranges for the region
→ Route table evaluation follows LONGEST PREFIX MATCH — the prefix list is more specific than 0.0.0.0/0
→ This is why S3 traffic goes through the endpoint, NOT through NAT Gateway

What if VPC Endpoint does NOT exist?

Lambda: "Send packet to 52.219.62.1 (S3 IP)"

    |
    ↓

    |
    ↓
WORKS ✅ but costs NAT processing fees + higher latency

→ Both paths WORK — but the VPC Endpoint path is cheaper and faster

📈 SCALABILITY CONSIDERATIONS

→ Concurrency Scalability

  • Lambda concurrent executions: 1,000 default per region per account (soft limit — requestable to tens of thousands)
  • Hyperplane ENIs scale automatically — no manual intervention
  • However — each concurrent execution opens a DATABASE CONNECTION to MariaDB
  • MariaDB max_connections is limited (default ~150 for small instances, from Project 9)
  • 500 concurrent Lambda executions = 500 DB connections → MariaDB crashes

→ The Database Connection Problem:

Lambda ConcurrencyDB Connections OpenedMariaDB db.t3.medium max_connectionsResult
1010~150✅ Fine
100100~150⚠️ Near limit
500500~150❌ Connection refused errors
10001000~150❌ Database overwhelmed

→ Solution: RDS Proxy

Without RDS ProxyWith RDS Proxy
Each Lambda → Direct DB connectionEach Lambda → RDS Proxy → Pooled DB connections
500 Lambdas = 500 DB connections500 Lambdas = ~20 pooled DB connections
DB overwhelmedDB comfortable

→ RDS Proxy Configuration:

  • Navigate to RDS Console → Proxies → Create Proxy
  • Engine: MariaDB
  • Target: Your RDS MariaDB instance
  • VPC: Same VPC as Lambda
  • Secrets Manager secret for DB credentials (from Project 37)
  • IAM authentication: Optional (adds security layer)
  • Lambda connects to RDS Proxy endpoint instead of RDS endpoint

→ NAT Gateway Scalability

  • Single NAT Gateway: Up to 45 Gbps bandwidth
  • If Lambda generates massive outbound internet traffic → NAT becomes bottleneck
  • Solution: Multiple NAT Gateways across AZs (each subnet's route table points to its AZ's NAT)
  • NAT Gateway scales automatically up to 45 Gbps — no manual scaling needed

→ VPC Endpoint Scalability

  • Gateway Endpoints (S3): No bandwidth or request limits — scales automatically
  • Interface Endpoints: Governed by ENI bandwidth (~10 Gbps per ENI per AZ)
  • For S3 → Gateway Endpoint has no practical limit

→ Cost Scalability

ComponentCost DriverOptimization
LambdaInvocations + Duration + MemoryRight-size memory, reduce timeout

S3 VPC Endpoint (Gateway)FREEAlways use for S3

ENIsFREE (no charge for Lambda Hyperplane ENIs)No optimization needed

→ Biggest cost trap: Lambda writing 1 TB/month to S3 through NAT Gateway = $45/month NAT processing
→ Same traffic through VPC Gateway Endpoint = $0
→ Always create VPC Endpoints BEFORE deploying VPC Lambda

    |
    ↓

    |
    ↓

🎯 KEY TAKEAWAY

→ Lambda in VPC is a networking configuration — not a code change
→ The paradox (need private + public access) is solved with architecture, not workarounds
→ Private resources (RDS) → Direct via VPC private routing
→ AWS public services (S3) → Via VPC Gateway Endpoint (free, fast)

→ The most dangerous scalability issue is Lambda exhausting RDS connections → RDS Proxy solves this

🔗 BRIDGE TO NEXT PROJECT

→ PayScale's Lambda is now working — reading from MariaDB, calling external APIs, writing to S3
→ But the Glue ETL jobs (from Project 21, 24) have a different problem
→ They run a daily job that extracts new transactions from MariaDB and converts to Parquet in S3
→ Every day, the Glue job re-reads ALL records from MariaDB — including records already processed yesterday
→ Day 1: 10,000 records → Processed in 2 minutes
→ Day 30: 300,000 records → Processed in 45 minutes (re-reading 290,000 old records!)
→ Day 365: 3.6 million records → Job takes 4 hours and costs $50 per run
→ The data engineer asks: "How do I make Glue only process NEW data since the last run?"

→ This is the exact problem Project 46 solves
→ Project 46 introduces Glue Job Bookmarks — the mechanism for incremental ETL
→ It tracks what data has already been processed and only reads new/changed data on subsequent runs
→ Without bookmarks → every Glue job is a full re-extraction → costs scale linearly with total data volume
→ With bookmarks → costs scale only with NEW data volume
