🟡 HOW TO APPROACH IT

→ This is an EVENT-DRIVEN ARCHITECTURE problem
→ S3 can AUTOMATICALLY generate events when objects are created, deleted, or modified
→ These events can be sent to: SNS topics, SQS queues, Lambda functions, or EventBridge

S3 BUCKET
    │
    │ Object created/deleted/restored
    │
    ▼
S3 EVENT NOTIFICATION ENGINE
    │
    │ Matches event against configured rules
    │ (prefix filter + suffix filter + event type)
    │
    ├──► SNS Topic → Fan-out to multiple subscribers (Project 13 pattern)
    │
    ├──► SQS Queue → Direct to a single consumer
    │
    ├──► Lambda Function → Serverless processing (Project 26)
    │
    └──► EventBridge → Complex event routing (Project 38)

→ S3 events are the TRIGGER that starts automated workflows
→ Combined with SNS fan-out (Project 13) → one upload can trigger MULTIPLE independent actions
→ This eliminates polling, cron jobs, and manual checking

→ The approach should be:

  • Step 1 — Understand S3 event types
  • Step 2 — Configure destination permissions (SNS topic policy, SQS queue policy)
  • Step 3 — Create event notification configurations with prefix/suffix filters
  • Step 4 — Configure multiple notification rules for different prefixes
  • Step 5 — Understand the event message format
  • Step 6 — Build the complete event-driven pipeline
  • Step 7 — Test and verify end-to-end event flow
  • Step 8 — Handle edge cases and failure scenarios

STEP 1 — UNDERSTAND S3 EVENT TYPES

→ S3 generates events for specific object lifecycle actions
→ You choose WHICH event types trigger notifications

Event Type Categories:

Object Created Events:

Event TypeTriggered When
s3:ObjectCreated:*ANY creation method (wildcard — covers all below)
s3:ObjectCreated:PutObject uploaded via PUT API (single upload)
s3:ObjectCreated:PostObject uploaded via POST API (form-based upload)

STEP 2 — CONFIGURE DESTINATION PERMISSIONS

→ This is the step that MOST people miss → causes "Unable to validate destination" error
→ S3 needs EXPLICIT permission to publish to SNS topics and SQS queues
→ The permission is granted via the DESTINATION's resource policy (not IAM — not S3's policy)

Statement:

CRITICAL: Configure Permissions BEFORE Creating Event Notifications

→ When you create an S3 event notification → S3 VALIDATES the destination
→ S3 sends a test message to the destination → if it fails → notification creation is REJECTED
→ Error: "Unable to validate the following destination configurations"
→ This means: The topic/queue policy doesn't allow S3 to publish
→ ALWAYS set up destination permissions FIRST → then create event notifications

Filter Rules and Limitations:

RuleDetail
Prefix filterMatches the beginning of the object key
Suffix filterMatches the end of the object key
CombinedPrefix AND suffix must both match (AND logic)
Case sensitivityPrefix and suffix are CASE-SENSITIVE
Maximum notifications per bucketNo hard limit on number of notification configs
Overlap restrictionCannot have overlapping prefix+suffix+event type combinations for the SAME destination type

Overlap Rule — CRITICAL:

→ S3 does NOT allow two notification configs with the same prefix+suffix+event type if they go to the SAME destination type (both to SNS, or both to SQS)
→ BUT: You CAN have the same prefix+suffix+event type going to DIFFERENT destination types (one to SNS AND one to SQS)

Config AConfig BAllowed?
Prefix: data/, Event: Created, Dest: SNS-APrefix: data/, Event: Created, Dest: SNS-B❌ NO (same prefix+event, both SNS)
Prefix: data/, Event: Created, Dest: SNS-APrefix: data/, Event: Created, Dest: SQS-A✅ YES (different destination types)
Prefix: data/, Event: Created, Dest: SNS-APrefix: data/raw/, Event: Created, Dest: SNS-B✅ YES (different prefixes — data/ vs data/raw/)
Prefix: data/, Event: Created, Dest: SNS-APrefix: data/, Event: Removed, Dest: SNS-B✅ YES (different event types)

→ To send the SAME event to MULTIPLE SNS topics → use ONE SNS topic → fan out to multiple SQS queues (Project 13)
→ Or use EventBridge (Project 38) which doesn't have this limitation

URL Encoding in Object Key:

→ CRITICAL: The object key is URL-ENCODED in the event message
→ Spaces become +, special characters become %XX
→ Example: products/images/shoe 001.jpg → products/images/shoe+001.jpg
→ Consumers MUST URL-DECODE the key before using it to access S3
→ Forgetting this → GetObject call fails with "NoSuchKey" → confusing bug

Sequencer Field:

→ The sequencer is a hexadecimal value that increases with each event for the same object key
→ Use it to determine EVENT ORDER for the same object
→ If you receive two events for the same key → compare sequencers → higher = later
→ This is important when: Upload v1 → Upload v2 → events arrive out of order
→ Sequencer comparison ensures you process the LATEST version

STEP 8 — HANDLE EDGE CASES AND FAILURE SCENARIOS

Edge Case 1: High-Volume Uploads (Burst Scenario)

→ Marketing team bulk-uploads 500 images at once (drag-and-drop)
→ 500 S3 events generated in rapid succession
→ All 500 published to SNS topic → all 500 delivered to each SQS queue

→ This is FINE because:

  • SNS Standard topics handle millions of messages per second
  • SQS Standard queues handle unlimited messages per second
  • Consumers process at their own pace → queues buffer the burst
  • This is exactly the pattern designed in Project 13

→ POTENTIAL ISSUE: If using FIFO queues → 300 msg/sec limit per message group
→ Solution: Use Standard queues for image processing (duplicates are harmless — resize same image twice = same result)

Edge Case 2: Event Delivery Guarantee

→ S3 event notifications provide AT-LEAST-ONCE delivery
→ Rare possibility: Same event delivered TWICE to the destination
→ This can happen during: S3 internal retries, network issues

→ Impact on consumers:

  • Image resize: Idempotent — resizing same image twice produces same result ✓
  • Catalog update: Must be idempotent — INSERT should use INSERT IGNORE or UPSERT ✓
  • Compliance alert: Duplicate alert is acceptable (better safe than sorry) ✓
  • Security alert: Duplicate alert is acceptable ✓

→ DESIGN PRINCIPLE: All consumers of S3 events must be IDEMPOTENT
→ Processing the same event twice should produce the same result as processing it once

Edge Case 3: Event Ordering

→ S3 does NOT guarantee event ordering across different objects
→ Upload A.jpg then B.jpg → B.jpg event might arrive before A.jpg event
→ For SAME object: Upload v1 then v2 → v2 event might arrive before v1 event
→ Use the sequencer field to determine the correct order for the same object key

→ For ShopNow: Event ordering doesn't matter for image processing
→ Each image is processed independently → order is irrelevant

Edge Case 4: Event Delivery Timing

→ S3 events are typically delivered within SECONDS of the object action
→ AWS documentation states: "Typically delivered in seconds, but can occasionally take a minute or longer"
→ This is NOT real-time streaming → it's near-real-time event notification
→ For time-critical applications → account for possible delays

→ For ShopNow: Seconds-to-minutes delay is acceptable
→ "Within minutes of upload" requirement is met ✓

Edge Case 5: Large Objects and Multipart Uploads

→ A 5 GB video uploaded via multipart upload
→ S3 generates the event ONLY when CompleteMultipartUpload is called
→ No event during individual part uploads
→ The event fires AFTER the complete object is available

→ This means: When the consumer receives the event → the object is fully uploaded and readable
→ No risk of reading a partially uploaded object

Edge Case 6: Versioning + Events

→ With versioning enabled (Project 12):

  • Upload v1 → event with versionId of v1
  • Upload v2 (overwrite) → NEW event with versionId of v2
  • Delete (simple) → event with eventName ObjectRemoved:DeleteMarkerCreated
  • Delete (version-specific) → event with eventName ObjectRemoved:Delete

→ Each version action generates its OWN event
→ Consumers receive the versionId in the event → can process the specific version

Edge Case 7: SNS Topic or SQS Queue Unavailable

→ If the destination is temporarily unavailable when S3 tries to publish:
→ S3 retries the event delivery
→ Retry behavior: Multiple retries over a period of time (not precisely documented)
→ If ALL retries fail → the event is LOST

→ Mitigation:

  • SNS and SQS are highly available (99.9%+) → delivery failures are extremely rare
  • For critical events → also enable S3 → EventBridge (Project 38) as a backup destination
  • EventBridge has its own retry and DLQ mechanisms
  • CloudTrail logs ALL S3 API calls → even if events are lost, the audit trail exists

Edge Case 8: Destination Validation Failure

→ If you modify the SNS topic policy AFTER creating the notification → remove S3 permission
→ S3 continues to TRY publishing events → events fail silently
→ No error appears in S3 Console → events are just lost
→ S3 only validates permissions at CONFIGURATION TIME, not at runtime

→ Mitigation:

  • Never modify destination policies after creating notifications
  • If you must modify → test by uploading a file and verifying event delivery
  • Monitor SNS NumberOfNotificationsFailed metric

✅ WHAT SHOULD DO

→ Always configure destination permissions (topic/queue policy) BEFORE creating event notifications
→ Always include BOTH aws:SourceArn AND aws:SourceAccount conditions in destination policies
→ Always use specific prefix and suffix filters to reduce noise → avoid triggering on irrelevant objects
→ Always design consumers to be IDEMPOTENT → S3 events are at-least-once delivery
→ Always URL-decode the object key from event messages before using it
→ Always use the sequencer field when processing multiple events for the same object key
→ Always test with actual uploads after configuration → don't assume it works
→ Always monitor SNS/SQS metrics to verify events are flowing
→ Always use SNS fan-out when multiple consumers need the same event → don't create duplicate S3 notifications
→ Always enable Raw Message Delivery on SQS subscriptions (Project 13)
→ Always create DLQs for consumer queues → handle processing failures gracefully
→ Always document which prefixes trigger which pipelines → event routing becomes complex

❌ WHAT SHOULD NOT DO

→ Never create event notifications without configuring destination permissions first → "Unable to validate" error
→ Never forget to include SourceAccount condition → prevents confused deputy attacks
→ Never assume events arrive instantly → allow for seconds-to-minutes delay
→ Never assume events arrive exactly once → design idempotent consumers
→ Never assume events arrive in order → use sequencer for same-key ordering
→ Never create overlapping prefix+suffix+event configs for the same destination type → S3 rejects it
→ Never use overly broad filters (no prefix, no suffix) → triggers on every object → overwhelming volume
→ Never hardcode bucket names in consumer code → read from the event message instead
→ Never modify destination policies after creating notifications without testing → events may silently fail
→ Never rely solely on S3 events for audit → use CloudTrail as the authoritative audit log
→ Never use S3 event notifications for complex event routing → use EventBridge (Project 38) instead
→ Never forget that the object key is URL-encoded → common source of "NoSuchKey" bugs

1 → HOW S3 EVENT GENERATION WORKS

User uploads: products/images/new-shoe-001.jpg
        │
        ▼

        Stores the object across multiple AZs (Project 1 — 11 9s durability)
        Returns HTTP 200 to the uploader
        │

Step 2: S3 Event Notification Engine activates

        │
        ▼

        │   Key "products/images/new-shoe-001.jpg":
        │   Prefix match: "products/images/" ✓
        │   Suffix match: ".jpg" ✓
        │   Event match: "ObjectCreated:Put" matches "ObjectCreated:*" ✓
        │   → GENERATE EVENT for this config
        │
        ├── Config 2: Prefix "data/raw/", Suffix ".csv", Event "ObjectCreated:*"
        │   Prefix match: "data/raw/" vs "products/images/" ✗
        │   → NO EVENT
        │
        ├── Config 3: Prefix "invoices/", Event "ObjectRemoved:*"
        │   Event match: "ObjectCreated" ≠ "ObjectRemoved" ✗
        │   → NO EVENT
        │
        └── Config 4: Prefix "restricted/", Event "ObjectCreated:*"
            Prefix match: "restricted/" vs "products/images/" ✗
            → NO EVENT
        │
        ▼
Step 4: For each matching config → construct event message
        (JSON with Records array — as shown in Step 5)
        │
        ▼
Step 5: Deliver event to configured destination
        │
        ├── Config 1 destination: SNS topic shopnow-s3-image-events
        │   S3 calls sns:Publish
        │   SNS receives → fans out to subscribers
        │
        └── Delivery complete

2 → HOW PREFIX AND SUFFIX MATCHING WORKS

Notification Config:
  Prefix: "products/images/"
  Suffix: ".jpg"

Object Key: "products/images/category/shoes/new-shoe-001.jpg"

Prefix Match:
  Key starts with "products/images/"? 
  "products/images/category/shoes/new-shoe-001.jpg"
   ^^^^^^^^^^^^^^^^
  YES ✓ (prefix matches the beginning)

Suffix Match:
  Key ends with ".jpg"?
  "products/images/category/shoes/new-shoe-001.jpg"
                                              ^^^^
  YES ✓ (suffix matches the end)

RESULT: MATCH → Generate event

Important: Prefix matches ANY depth:

Prefix: "products/"

These ALL match:
  products/shoe.jpg ✓
  products/images/shoe.jpg ✓
  products/images/category/shoes/shoe.jpg ✓
  products/a ✓

These do NOT match:
  marketing/shoe.jpg ✗ (different prefix)
  Products/shoe.jpg ✗ (case-sensitive: "P" ≠ "p")

→ Prefix matching is a SIMPLE string comparison on the beginning of the key
→ It does NOT understand "folder" hierarchy → products/ matches anything starting with products/
→ Case-sensitive → Products/ ≠ products/

3 → HOW EVENT DELIVERY TO SNS WORKS

S3 Event Engine has event to deliver to SNS topic
        │
        ▼
Step 1: S3 constructs the event JSON message
        │
        ▼
Step 2: S3 calls sns:Publish API

4 → HOW S3 → EVENTBRIDGE DIFFERS

→ Starting 2021, S3 can send ALL events to EventBridge (not just configured events)
→ Must be explicitly enabled: S3 Console → Properties → Amazon EventBridge → "On"

S3 Event (with EventBridge enabled):

Step 1: Object action occurs
Step 2: S3 Event Notification Engine (prefix/suffix matching — as before)
        → Delivers to configured SNS/SQS/Lambda destinations
Step 3: ADDITIONALLY: S3 sends event to EventBridge event bus

        → EventBridge rules handle filtering

EventBridge advantages:
  → Filter on ANY field in the event JSON (not just prefix/suffix)

  → No overlap restrictions

#PitfallWhat HappensHow to Avoid
1Missing destination access policy"Unable to validate destination" error on creationConfigure topic/queue policy BEFORE creating notification
2URL-encoded object key not decodedConsumer tries to GetObject with encoded key → "NoSuchKey"Always URL-decode the key from event message
3Overlapping prefix+suffix+event for same destination typeS3 rejects the notification configurationUse single SNS topic → fan-out to multiple queues
4Processing non-idempotentlyDuplicate event → duplicate processing → data corruptionDesign all consumers to be idempotent
5Case-sensitive prefix mismatchPrefix "Products/" doesn't match key "products/image.jpg"Verify exact case of prefix matches actual S3 key structure
6Event fires for lifecycle transitionsLifecycle moves object to Glacier → triggers ObjectCreated event → consumer confusedFilter by specific event types, not just ObjectCreated:*

9Destination policy removed after configurationEvents silently fail — no error in S3 ConsoleMonitor SNS/SQS delivery metrics continuously

📈 SCALABILITY CONSIDERATIONS

→ Event Generation Scalability

  • S3 can generate millions of events per second
  • No limit on event generation rate
  • Each object action generates exactly one event per matching configuration
  • Bulk uploads → bulk events → SNS/SQS handle the throughput

→ Destination Scalability

  • SNS Standard topics: Virtually unlimited throughput → handles any burst
  • SQS Standard queues: Virtually unlimited throughput → buffers any burst
  • Lambda: Concurrent execution limits apply → may need to increase limits
  • EventBridge: 400 events/second per account per region (soft limit)

→ Consumer Scalability

  • Each SQS queue independently buffers events
  • Multiple consumers per queue → horizontal scaling
  • Auto Scaling based on queue depth (Project 33) → automatic scale-out during bursts
  • Different consumers process at different rates → no bottleneck

→ Multi-Pipeline Scalability

  • One bucket can have dozens of notification configurations
  • Each configuration routes to a different pipeline
  • Adding new pipelines requires: new notification config + new destination → zero change to existing pipelines
  • This is the same additive extensibility as SNS fan-out (Project 13)

→ Cost Scalability

  • Event generation and delivery are FREE for SNS/SQS
  • Cost is entirely in consumer compute
  • At 1 million events/day → consumer cost depends on processing complexity, not event volume

        Check SNS topic policy / SQS queue policy

        Was the policy accidentally modified?

        EventBridge rules might be consuming events

        CloudTrail → Event history → Filter: PutObject
        Confirms the upload actually happened

STEP 7: Try a different file
        Upload a completely new file (not overwrite)
        Some edge cases with versioning may behave differently
