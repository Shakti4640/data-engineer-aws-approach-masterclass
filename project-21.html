PROJECT 21: THE CSV FILES THAT SLOWED EVERYTHING DOWN — Glue ETL Job for S3 CSV to Parquet Transformation

🔴 HYPOTHETICAL PROBLEM

→ ShopNow's data team has been using the pipeline built in Projects 17 and 18
→ Glue Crawler catalogs raw CSV files in S3 → Athena queries the catalog
→ The data landscape today:

DatasetFormatSizeLocationQuery Frequency
Order transactionsCSV8 GB (growing 500 MB/day)s3://shopnow-data-lake/raw/orders/40 queries/day
Product catalogCSV2 GBs3://shopnow-data-lake/raw/products/20 queries/day
Customer profilesJSON3 GBs3://shopnow-data-lake/raw/customers/15 queries/day
Clickstream eventsCSV25 GB (growing 2 GB/day)s3://shopnow-data-lake/raw/clickstream/10 queries/day

→ Problem 1: COST EXPLOSION

  • Athena charges $5 per TB scanned (Project 18)
  • Every query on the 8 GB orders CSV scans the ENTIRE 8 GB — even if the query only needs 2 columns
  • 40 queries/day × 8 GB × $5/TB = $1.60/day just for orders
  • Clickstream: 10 queries × 25 GB × $5/TB = $1.25/day
  • Total across all datasets: ~$120/month and growing rapidly

→ Problem 2: QUERY SPEED

  • A simple aggregation on orders takes 45 seconds scanning 8 GB of CSV
  • The CEO's dashboard times out because underlying queries are too slow
  • CSV is ROW-based — Athena must read every row even when only 3 of 25 columns are needed

→ Problem 3: DATA QUALITY

  • Raw CSV files have inconsistent date formats (some MM/DD/YYYY, some YYYY-MM-DD)
  • Some numeric columns contain text values ("N/A", "null", empty strings)
  • Duplicate records exist across files uploaded on different days
  • Column names have spaces and special characters (Order ID, Ship-Date, Customer Name)

→ Problem 4: NO PARTITIONING

  • All order files sit in one flat prefix: raw/orders/
  • A query for "January 2025 orders" still scans ALL months of data
  • No way to skip irrelevant data

→ The data engineering lead says: "We need an ETL process that cleans, transforms, and converts our raw data into an optimized format — and we need it automated"

→ The target state:

FormatCSV / JSONParquet
CompressionNoneSnappy
SchemaInconsistentClean, typed, standardized
PartitioningNoneBy year/month/day
Column namesSpaces, special charssnake_case
Data qualityDuplicates, nulls, mixed formatsDeduplicated, null-handled, consistent
Storage size38 GB~8 GB (Parquet + Snappy compression)
Athena scan cost~$120/month~$15/month

🟡 HOW TO APPROACH IT

→ This is an ETL (Extract-Transform-Load) problem
→ Extract: Read raw CSV/JSON from S3
→ Transform: Clean, convert types, standardize columns, deduplicate, partition
→ Load: Write optimized Parquet files back to S3 in a structured layout

→ AWS Glue ETL Job is the service purpose-built for this

→ Understanding what Glue ETL is (vs what we already know):

  • Project 17: Glue CRAWLER — discovers schema, creates catalog tables (reads metadata)
  • This project: Glue ETL JOB — reads data, transforms it, writes it back (processes actual data)
  • The Crawler is about CATALOGING → The Job is about TRANSFORMING
  • They are complementary — often used together in a pipeline

→ The approach should be:

  • Step 1 — Understand Glue Job types and when to use which
  • Step 2 — Understand the Glue execution environment (what runs under the hood)
  • Step 3 — Create IAM Role for the Glue Job
  • Step 4 — Create the Glue Job with source/target configuration
  • Step 5 — Configure job parameters (DPUs, timeout, retries)
  • Step 6 — Configure transformations via Glue Studio Visual Editor
  • Step 7 — Configure output format (Parquet + Snappy + Partitioning)
  • Step 8 — Run the job and verify output
  • Step 9 — Re-crawl the output to update the Glue Catalog
  • Step 10 — Query optimized data via Athena and compare costs

STEP 2 — UNDERSTAND THE GLUE EXECUTION ENVIRONMENT

→ When you "run" a Glue Job, here's what happens behind the scenes:

You click "Run Job"
       ↓
Glue provisions a Spark cluster in AWS-managed infrastructure
       ↓
Cluster has: 1 Driver node + N Worker nodes (you configure N)
       ↓
Driver node reads your job configuration / script
       ↓
Driver creates a Spark execution plan
       ↓
Workers read data from S3 in PARALLEL (each worker reads a portion)
       ↓
Workers apply transformations in parallel
       ↓
Workers write output to S3 in PARALLEL
       ↓
Job completes → Cluster is TERMINATED → You stop paying

→ Key insight: You do NOT manage servers, clusters, or Spark installations
→ Glue provisions everything → runs your job → tears it down
→ This is "serverless Spark"

→ Cold start time: 30 seconds to 2 minutes (cluster provisioning)
→ For frequently-run jobs — Glue offers "Flex" execution (uses spare capacity, cheaper, longer cold start)

Glue Execution Classes:

ClassCold StartCostBest For
Standard~1 minuteFull priceTime-sensitive jobs, SLA-bound
Flex2-5 minutes~35% cheaperNon-urgent batch jobs, overnight processing

→ ShopNow's daily ETL runs overnight → Flex is ideal (save 35%)

STEP 6 — CONFIGURE TRANSFORM NODES (Transform)

→ This is where the data quality issues get fixed
→ In the Visual Editor → Add Transform nodes between Source and Target

Transform 1: Change Schema (Rename & Retype Columns)

→ Add node: "Change Schema" (also called "ApplyMapping")

Original ColumnNew Column NameOriginal TypeNew TypeAction
Order IDorder_idstringstringRename (remove space)
Customer Namecustomer_namestringstringRename
Ship-Dateship_datestringstringRename (remove hyphen)
Order Dateorder_datestringdateRename + Retype
Total Amounttotal_amountstringdoubleRename + Retype
QuantityquantitystringintRetype
Discount%discount_pctstringdoubleRename (remove special char)
Internal Notes(drop)string—Remove column entirely

→ Why rename?

  • Athena and Redshift don't handle spaces/special characters in column names well
  • snake_case is the industry standard for data lake columns
  • Consistent naming prevents query errors downstream

→ Why retype?

  • CSV stores everything as strings — Crawler may have inferred some correctly, others not
  • Explicit type casting ensures: total_amount is numeric (can be summed, averaged)
  • Date types enable date functions in Athena (YEAR, MONTH, date arithmetic)

→ Why drop columns?

  • Internal Notes is not needed for analytics
  • Removing it reduces file size and improves query performance
  • Parquet is columnar — fewer columns = less data per query

Transform 2: Filter (Remove Invalid Records)

→ Add node: "Filter"

Filter ConditionPurpose
total_amount is not nullRemove records with missing amounts
total_amount > 0Remove zero/negative amounts (likely errors)
order_date is not nullRemove records without dates
order_id is not nullRemove records without order IDs

→ Filter evaluates conditions and DROPS rows that don't match
→ This is data quality enforcement at the ETL layer

Transform 3: Drop Duplicates

→ Add node: "Drop Duplicates"

SettingValue
Columns to matchorder_id

→ If two records have the same order_id → keep only the first one
→ This handles the duplicate records problem from overlapping file uploads

→ For more complex deduplication (keep latest based on timestamp):

  • Switch to Script mode
  • Use window functions to rank records by updated_at within each order_id
  • Keep only rank = 1

→ Add node: "Derived Column" (or "Custom Transform")

Transform 5: Resolve Choice (Handle Ambiguous Types)

→ Glue's DynamicFrame has a unique feature: "ChoiceType"
→ When a column has mixed types (some rows are int, some are string), Glue creates a ChoiceType
→ Example: quantity column has values 5, 10, N/A, null

→ Add node: "Resolve Choice"

SettingValueBehavior
Choice resolutioncast:intForce all values to int — non-castable become null

→ cast:int is usually the right choice for numeric columns
→ "N/A" values become null → handled by the Filter transform above or by downstream logic

Understanding Parquet — Why It Changes Everything:

AspectCSVParquet
Storage modelRow-based (entire row stored together)Column-based (each column stored separately)
Reading 3 of 25 columnsMust read ALL 25 columns for every rowReads ONLY the 3 needed columns
CompressionPoor (text data, redundant headers)Excellent (similar values grouped, dictionary encoding)
SchemaExternal (no type info in file)Embedded (types stored in file footer)
SplittableYes (line-by-line)Yes (row groups)
File size (typical)1x (baseline)0.2x – 0.3x (70-80% smaller)
Athena scanFull file sizeOnly relevant columns + row groups

→ Concrete example for ShopNow orders:

  • 25 columns in the orders table
  • Typical Athena query uses 3 columns: order_date, total_amount, customer_id
  • CSV: Scan 8 GB (all columns, all rows)
  • Parquet: Scan ~0.6 GB (3/25 columns × compression × row group pruning)
  • Cost reduction: 8 GB × $5/TB = $0.04/query → 0.6 GB × $5/TB = $0.003/query → 13x cheaper

Understanding Snappy Compression:

CompressionSpeedRatioSplittableUse Case
SnappyVery fastModerate (~2-4x)Yes (with Parquet)Default for analytics (Athena, Spark, Redshift Spectrum)
GZIPSlowHigh (~5-8x)No (standalone), Yes (with Parquet)Maximum compression, cold storage
LZOFastLow (~2x)YesLegacy Hadoop workloads
ZSTDFastHigh (~4-7x)Yes (with Parquet)Modern alternative to GZIP, better speed
NoneN/A1xYesWhen speed matters more than size

→ Snappy is the recommended default for Parquet in AWS analytics
→ Athena, Redshift Spectrum, and EMR all natively support Snappy-compressed Parquet
→ GZIP gives better compression but slower read/write — use for archival
→ ZSTD is emerging as a strong alternative — supported in newer Glue versions

Understanding Partitioning — The Other Half of Cost Optimization:

→ Partitioning means physically organizing files in S3 by partition key values

Without partitioning:

s3://shopnow-data-lake/transformed/orders/
  ├── part-00000.parquet    (contains all dates mixed together)
  ├── part-00001.parquet
  ├── part-00002.parquet
  └── ... (100 files, ~2 GB total)

→ Query: "Show me January 2025 orders"
→ Athena scans ALL 2 GB — cannot skip any files

With partitioning by year/month/day:

s3://shopnow-data-lake/transformed/orders/
  ├── year=2024/
  │     ├── month=01/
  │     │     ├── day=01/
  │     │     │     ├── part-00000.parquet
  │     │     │     └── part-00001.parquet
  │     │     ├── day=02/
  │     │     │     └── part-00000.parquet
  │     │     └── ...
  │     ├── month=02/
  │     └── ...
  ├── year=2025/
  │     ├── month=01/
  │     │     ├── day=01/
  │     │     │     └── part-00000.parquet
  │     │     └── ...
  │     └── ...
  └── ...

→ Query: "Show me January 2025 orders"
→ Athena reads ONLY files under year=2025/month=01/ → skips all other months/years
→ Instead of scanning 2 GB → scans ~60 MB (one month out of ~36 months of data)
→ Cost: 2 GB × $5/TB = $0.01 → 60 MB × $5/TB = $0.0003 → 33x cheaper

Partition key format in S3:
→ Hive-style partitioning: key=value/ in the path
→ Athena and Glue Catalog recognize this format natively
→ When you specify year, month, day as partition keys in the Glue Job target:

  • Glue automatically creates the year=XXXX/month=XX/day=XX/ folder structure
  • Glue writes each record's Parquet file into the matching partition folder
  • The partition columns are NOT stored inside the Parquet files (they're in the path)

Choosing partition keys:

ConsiderationGuidance
Most common filter in queriesPartition by that column (e.g., date for time-series)
Cardinality too highDon't partition by order_id (millions of unique values = millions of folders)
Cardinality too lowDon't partition by country if 95% of data is one country
Sweet spot100-10,000 partitions total
Common patternyear/month or year/month/day for time-series data

→ For ShopNow orders: year/month/day → ~365 partitions per year → good cardinality

Understanding Job Bookmarks (Preview — Deep Dive in Project 46):

→ Without bookmarks: Every job run processes ALL data from the beginning
→ With bookmarks: Glue tracks which data was already processed → only processes NEW data
→ For S3 sources: Bookmark tracks which files have been processed (by file path + timestamp)
→ For JDBC sources: Bookmark tracks the last processed value of a key column
→ Enable now → understand deeply in Project 46

✅ WHAT SHOULD DO

→ Always convert CSV/JSON to Parquet for analytical workloads — the cost savings alone justify the ETL
→ Always partition by the most commonly filtered column (usually date)
→ Always use Snappy compression with Parquet — it's the standard for AWS analytics
→ Always separate raw and transformed data in different S3 prefixes — never overwrite source

ZonePathPurpose
Raws3://bucket/raw/Immutable landing zone — source of truth
Transformeds3://bucket/transformed/Cleaned, optimized, partitioned
Curateds3://bucket/curated/Business-ready, aggregated (future projects)

→ Always enable Job Bookmarks for incremental processing
→ Always set job timeout to prevent runaway costs
→ Always enable CloudWatch metrics and continuous logging for observability
→ Always right-size workers — start small, monitor metrics, scale up if needed
→ Always verify output file sizes — target 64-256 MB per file for Athena
→ Always use Flex execution class for non-time-sensitive batch jobs (35% savings)
→ Always use a separate Glue Catalog database for transformed data — don't mix with raw catalog
→ Always test with a small subset first — run job on 1 day's data before processing entire history

❌ WHAT SHOULD NOT DO

→ Never query raw CSV directly in Athena for production workloads — it's slow and expensive
→ Never write transformed output to the SAME S3 path as raw input — you lose the ability to reprocess
→ Never skip partitioning — unpartitioned Parquet is better than CSV but still wastes scan costs
→ Never over-partition — partitioning by order_id creates millions of tiny files (worse performance)
→ Never under-provision workers for large datasets — job runs for hours, costs more in DPU-hours than adding workers
→ Never over-provision workers for small datasets — 50 workers for 1 GB of data wastes cold start time and money
→ Never ignore the Spark UI when jobs are slow — it reveals data skew, shuffle bottlenecks, and serialization issues
→ Never leave job timeout as default (2880 minutes = 48 hours) — a stuck job runs for 2 days before failing
→ Never use Standard execution class for nightly batch jobs — Flex is 35% cheaper with acceptable cold start
→ Never create a Parquet file per input record — use coalesce/repartition to control output file count
→ Never forget to update the Glue Catalog after the job — Athena can't query data it doesn't know about
→ Never use G.4X or G.8X workers unless you've proven smaller workers are insufficient — they're 4-8x more expensive

3 → HOW PARQUET WRITES HAPPEN INTERNALLY

Glue Worker has transformed records in memory
       ↓
Records are organized into Row Groups (default ~128 MB each)
       ↓
Within each Row Group, data is organized by COLUMN:
  ├── Column 1 (order_id): all values for this row group
  ├── Column 2 (customer_name): all values for this row group
  ├── Column 3 (total_amount): all values for this row group
  └── ... (25 columns)
       ↓
Each column chunk is compressed with Snappy
       ↓
Column statistics are computed:
  ├── Min value, Max value
  ├── Null count
  └── Distinct count (approximate)
       ↓
File footer is written:
  ├── Schema (column names, types)
  ├── Row group metadata
  ├── Column chunk offsets
  └── Column statistics
       ↓
File is written to S3 as: part-XXXXX-YYYYY.snappy.parquet

Why column statistics matter:
→ When Athena reads Parquet:

  • It reads the file footer FIRST (tiny — a few KB)
  • It checks column statistics: "Does this row group contain any order_date in January 2025?"
  • If min/max of order_date in a row group is [2024-06-01, 2024-06-30] → SKIP entire row group
  • This is called "predicate pushdown" or "row group pruning"
  • Works IN ADDITION to partition pruning → double optimization

4 → HOW PARTITIONED WRITES HAPPEN

Glue Worker has record: {order_id: "ORD-5678", order_date: "2025-01-15", ...}
       ↓
Worker extracts partition values:
  year = 2025, month = 01, day = 15
       ↓
Worker determines target S3 path:
  s3://shopnow-data-lake/transformed/orders/year=2025/month=01/day=15/
       ↓
Worker writes the record to a Parquet file in that path
  NOTE: partition columns (year, month, day) are NOT in the Parquet file
        They exist only in the S3 path — Athena reconstructs them from the path
       ↓
Each unique partition value combination gets its own S3 "directory"
       ↓
After all records are written:
  Glue updates the Catalog with partition metadata
  Each partition is registered: {year=2025, month=01, day=15} → S3 location

Partition registration in Catalog:
→ The Glue Catalog stores a mapping:

Partition ValuesS3 Location
year=2024, month=01, day=01s3://…/orders/year=2024/month=01/day=01/
year=2024, month=01, day=02s3://…/orders/year=2024/month=01/day=02/
year=2025, month=01, day=15s3://…/orders/year=2025/month=01/day=15/

→ When Athena queries with WHERE year = 2025 AND month = 1:

  • Athena checks the Catalog for matching partitions
  • Only partitions matching the filter are returned
  • Athena reads files ONLY from those S3 paths
  • All other partitions are completely skipped

5 → HOW GLUE READS CSV INTERNALLY

Glue Worker receives task: "Read files from s3://shopnow-data-lake/raw/orders/"
       ↓
Worker lists all files in the S3 prefix
       ↓
Files are distributed across workers (each worker gets a subset)
       ↓
Each worker reads its assigned files:
  ├── Downloads file content from S3
  ├── Splits by newline (row delimiter)
  ├── Splits each line by comma (column delimiter)
  ├── Applies header row (first line) as column names
  ├── Infers types OR uses Catalog schema
  └── Creates DynamicFrame rows
       ↓
All workers have their portions of data in memory
  → Ready for transformations

CSV-specific challenges Glue handles:
→ Quoted fields (commas inside quotes): Handled by CSV parser
→ Multi-line values (newlines inside quotes): Handled with multiline option
→ Different encodings (UTF-8, Latin-1): Configurable via job parameters
→ Files with different schemas (extra columns, missing columns): DynamicFrame handles gracefully
