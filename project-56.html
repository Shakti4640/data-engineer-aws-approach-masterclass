PROJECT 56: THE CFO WHO DEMANDED 40% COST REDUCTION — S3 Intelligent Tiering + Redshift RA3 + Reserved Capacity

🔴 HYPOTHETICAL PROBLEM

→ "DataPulse Corp" is a mid-size analytics company running on AWS for 3 years
→ Monthly AWS bill has grown from $12,000 to $87,000
→ The CFO reviews the bill and finds alarming patterns:

S3 Costs — $31,000/month
→ 180 TB across 47 buckets
→ 92% of objects are in S3 Standard — even data from 2021 that nobody has touched in 2 years
→ Lifecycle policies (Project 12) exist on only 3 buckets
→ Multiple teams upload data and forget about it — "data gravity" problem
→ Nobody knows which buckets cost the most or which objects are accessed

Redshift Costs — $38,000/month
→ Running 6x dc2.8xlarge nodes on-demand pricing
→ Cluster was sized for Black Friday peak 18 months ago
→ Average CPU utilization: 22% on weekdays, 4% on weekends
→ 8 TB of data — but 60% is historical data queried once a quarter
→ Team considered Redshift Serverless (Project 53) but needs persistent cluster for SLA-bound dashboards

EC2 Costs — $11,000/month
→ 25 EC2 instances running 24/7
→ Auto Scaling configured (Project 33) but minimum capacity set too high
→ 8 instances are development/staging — running on weekends when nobody works
→ All on-demand pricing — no reservations, no savings plans

Other Costs — $7,000/month
→ NAT Gateway data processing charges: $2,800
→ CloudWatch Logs retention set to "Never expire" on all log groups
→ Unattached EBS volumes and unused Elastic IPs
→ Cross-region data transfer for a replication that was "temporary" 14 months ago

→ CFO mandate: "Reduce the AWS bill by 40% within 90 days without impacting production performance"
→ That means reducing from $87,000 → ~$52,000/month
→ Target savings: ~$35,000/month

🟡 HOW TO APPROACH IT

→ Cost optimization is NOT about cutting services — it's about RIGHT-SIZING and RIGHT-PRICING
→ AWS pricing has 4 levers:

LeverMeaningExample
Right-SizingUse the correct size/tier for actual workloaddc2.8xlarge → ra3.xlplus
Right-PricingUse commitment discounts for predictable workloadsOn-Demand → Reserved or Savings Plan
Right-TieringMove data to appropriate storage classS3 Standard → Intelligent-Tiering
Right-EliminatingRemove unused/orphaned resourcesUnattached EBS, idle instances

→ Approach must be data-driven — not guesswork
→ Step 1 — Gain visibility (what is costing what and why)
→ Step 2 — Identify quick wins (unused resources — immediate savings)
→ Step 3 — Optimize storage tiers (S3 — largest spend area)
→ Step 4 — Right-size compute (Redshift, EC2)
→ Step 5 — Apply pricing models (Reserved, Savings Plans)
→ Step 6 — Optimize data transfer and auxiliary costs
→ Step 7 — Implement governance to prevent cost regression

SECTION D — S3 COST OPTIMIZATION: INTELLIGENT-TIERING & LIFECYCLE

Step D1 — Understanding S3 Intelligent-Tiering (Deep Dive)

→ Introduced in Project 1 (storage class table) and Project 12 (lifecycle policies)
→ Now we configure it at scale for cost optimization

→ S3 Intelligent-Tiering is NOT a lifecycle rule — it is a storage class with BUILT-IN automation

How Intelligent-Tiering Works Internally:

Access TierActivationTriggerCost vs Standard
Frequent AccessAutomatic (default)Object recently accessedSame as S3 Standard
Infrequent AccessAutomaticNot accessed for 30 consecutive days40% cheaper
Archive Instant AccessAutomaticNot accessed for 90 consecutive days68% cheaper
Archive AccessOPT-IN requiredNot accessed for 90–730 days (configurable)71% cheaper
Deep Archive AccessOPT-IN requiredNot accessed for 180–730 days (configurable)95% cheaper

→ Key behavior:

  • When an object in any lower tier is accessed → automatically moved back to Frequent Access
  • No retrieval fees for moving between tiers (unlike Standard-IA which charges retrieval)
  • Small monitoring fee: $0.0025 per 1,000 objects/month
  • Objects < 128 KB are always stored in Frequent Access tier (not monitored — too small)

Step D2 — Configure Intelligent-Tiering Archive Settings

→ S3 Console → Intelligent-Tiering → Archive Access Configuration
→ Create Configuration:

SettingValue
Configuration Namedatapulse-archive-config
Configuration ScopeEntire bucket (or filter by prefix/tag)
Archive Access TierEnable → Days since last access: 90
Deep Archive Access TierEnable → Days since last access: 180

→ This opt-in is CRITICAL — without it, Intelligent-Tiering only uses Frequent + Infrequent + Archive Instant
→ With opt-in → objects untouched for 180 days move to Deep Archive pricing automatically
→ And they STILL auto-restore when accessed (unlike manual Glacier which requires restore requests)

SECTION E — REDSHIFT COST OPTIMIZATION: RA3 + RESERVED NODES

Step E1 — Understanding the Current Problem

→ Current: 6x dc2.8xlarge on-demand
→ dc2.8xlarge per node: ~$4.80/hour on-demand
→ 6 nodes × $4.80 × 730 hours/month = ~$21,024/month
→ Plus storage costs embedded in dc2 (SSD-based, storage coupled to compute)
→ Total Redshift on-demand: ~$38,000/month (includes data transfer, Spectrum, etc.)

→ Problems with dc2:

  • Storage is COUPLED to compute — to add storage, you must add compute nodes
  • 60% of data is historical (queried quarterly) but lives on expensive SSD
  • Cannot scale compute independently of storage
  • Cluster sized for peak (Black Friday) but average utilization is 22%

Step E2 — Understanding Redshift RA3 vs DC2

CharacteristicDC2 (Current)RA3 (Target)
Storage TypeLocal SSD (coupled)Managed Storage (S3-backed)
Storage ScalingAdd nodes to add storageStorage scales independently — virtually unlimited
Compute ScalingAdd nodesElastic Resize + Concurrency Scaling
Data LocationAll on local SSDHot data cached on local SSD, warm/cold data in S3
Storage CostIncluded in node price$0.024/GB/month for Redshift Managed Storage
Best ForSmall datasets, consistent workloadsLarge datasets, variable workloads, cost optimization
Minimum Nodes12

→ RA3 key advantage: SEPARATES compute from storage
→ You can have 8 TB of data but only pay for 2 nodes of compute
→ Historical data automatically moves to S3-backed managed storage
→ Frequently queried data stays cached on local SSD for performance

Step E3 — Migrate from DC2 to RA3

→ Redshift Console → Clusters → Select cluster → Actions → Resize

Classic Resize (for node type change):

SettingValue
Resize TypeClassic Resize
New Node Typera3.xlplus (4 vCPU, 32 GB RAM, 32 TB managed storage per node)
Number of Nodes4 (down from 6 — RA3 has more memory per node)

→ Why ra3.xlplus instead of ra3.4xlarge?

  • DataPulse average utilization is 22% — they don't need 12 vCPU per node
  • ra3.xlplus: $1.086/hour on-demand per node
  • ra3.4xlarge: $3.26/hour on-demand per node
  • Start smaller → monitor → resize if needed

→ Classic Resize takes the cluster offline for a period (minutes to hours depending on data volume)
→ Schedule during maintenance window

Alternative — Elastic Resize:
→ If staying within same node type family — elastic resize is faster (minutes)
→ But DC2 → RA3 requires Classic Resize (different node type family)

Post-Migration Configuration:

→ After resize → Configure Managed Storage behavior
→ Redshift automatically manages data tiering:

  • Frequently accessed blocks → local SSD cache
  • Infrequently accessed blocks → S3 managed storage
  • No manual configuration needed — Redshift's auto-tiering handles it

Step E4 — Right-Size the Cluster with Redshift Advisor

  • Sort keys missing on 8 frequently filtered tables → adding sort keys reduces scan time by 60%

Step E5 — Configure Concurrency Scaling

→ Instead of keeping 6 nodes for peak periods:
→ Redshift Console → Workload Management → Edit WLM Configuration

SettingValue
Concurrency Scaling ModeAuto
Max Concurrency Scaling Clusters3

→ How it works:

  • During normal hours → 4 RA3 nodes handle all queries
  • During peak periods → Redshift automatically adds transient clusters
  • Concurrency scaling clusters are FREE for up to 1 hour per day per main cluster node
  • 4 nodes = 4 free concurrency scaling hours/day
  • Beyond free tier: charged per-second at on-demand rate

→ This eliminates the need to size the cluster for peak permanently

Step E6 — Configure Pause/Resume Schedule

→ Redshift Console → Select Cluster → Actions → Modify → Scheduled Actions

SettingValue
Action 1: PauseSchedule: Every Saturday 8 PM
Action 2: ResumeSchedule: Every Monday 6 AM

→ When paused:

  • Compute charges STOP (you only pay for managed storage)
  • No queries can run
  • Cluster state is preserved
  • Resume takes 30–60 seconds

→ DataPulse runs no analytics on weekends
→ Savings: 2 days/week = ~28% reduction in compute hours

→ For development/staging Redshift clusters:

  • Pause every weekday evening (7 PM → 7 AM) + all weekend
  • Compute runs only 50 hours/week instead of 168
  • 70% compute savings on non-production clusters

✅ WHAT SHOULD DO

→ Always gain visibility BEFORE optimizing — Cost Explorer + Storage Lens + Trusted Advisor first
→ Always start with quick wins (waste elimination) — zero risk, immediate savings
→ Always enable S3 Intelligent-Tiering for data with unpredictable access patterns
→ Always opt-in to Archive Access tiers in Intelligent-Tiering configuration
→ Always right-size BEFORE applying reservations — don't reserve the wrong size
→ Always test RA3 migration in a non-production cluster first
→ Always use Pause/Resume for clusters with predictable idle periods
→ Always apply Compute Savings Plans (not EC2 Instance SP) unless you are 100% certain of instance family
→ Always implement budgets and alerts AFTER optimization — prevent regression
→ Always enforce tagging — cost allocation without tags is guesswork
→ Always schedule non-production resources — paying 24/7 for 8-hour workloads is pure waste
→ Always review costs monthly — optimization is continuous, not one-time

❌ WHAT SHOULD NOT DO

→ Never buy Reserved Instances/Savings Plans before right-sizing — you'll lock in the wrong commitment
→ Never apply Glacier Deep Archive to data that might need quick access — 12-48 hour retrieval
→ Never assume Intelligent-Tiering is free — monitoring fee of $0.0025/1,000 objects adds up with billions of tiny objects
→ Never skip the Pause/Resume capability assessment — "we might need it on weekends" is usually false
→ Never set Savings Plan commitment at 100% of current usage — leave 20% headroom for variability
→ Never delete data without confirming compliance retention requirements
→ Never optimize only once — costs drift upward without governance
→ Never ignore data transfer costs — they are often the hidden surprise on AWS bills
→ Never use dc2 node types for data volumes > 2 TB per node — RA3 is almost always cheaper
→ Never rely solely on Trusted Advisor — it catches obvious waste but misses architectural optimization
→ Never apply Spot Instances to stateful, non-fault-tolerant workloads — interruption will cause data loss

🔴 WHEN NOT TO USE

→ Do NOT use Intelligent-Tiering for objects < 128 KB — they stay in Frequent tier permanently but still incur monitoring fee
→ Do NOT use Reserved Instances for workloads you plan to decommission within 6 months
→ Do NOT use 3-year reservations unless you have high confidence in architecture stability
→ Do NOT use Spot for databases, primary web servers, or single points of failure
→ Do NOT use S3 Glacier for data that users expect to download immediately
→ Do NOT use Pause/Resume on clusters with 24/7 SLA requirements
→ Do NOT use RA3 for clusters < 2 nodes — minimum is 2 for RA3
→ Do NOT use Cost Explorer alone for S3 optimization — it doesn't show per-object access patterns (use Storage Lens)
→ Do NOT use tag-based budgets without tag enforcement — untagged resources become invisible

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

S3 INTELLIGENT-TIERING INTERNAL MECHANICS

1 → Monitoring Engine
→ S3 maintains an access-tracking metadata layer for every object in Intelligent-Tiering
→ Every GET/HEAD request on an object resets its "last access" timer
→ A background process evaluates every object daily:

  • Object accessed in last 30 days → Frequent Access tier
  • Object NOT accessed for 30 days → move to Infrequent Access tier
  • Object NOT accessed for 90 days → move to Archive Instant Access tier
  • Object NOT accessed for configured days → move to Archive/Deep Archive tier
    → Movement between tiers is AUTOMATIC — no lifecycle rules needed
    → Movement has ZERO retrieval cost and ZERO operational overhead

2 → Archive Tier Restore Mechanics
→ Objects in Archive Instant Access → served immediately (milliseconds)
→ Objects in Archive Access tier → require restore before access

  • But unlike manual Glacier, Intelligent-Tiering auto-initiates restore on access
  • Restore time: 3–5 hours for Archive Access
  • Restore time: 12 hours for Deep Archive Access
    → After access → object moves back to Frequent Access tier automatically

3 → Cost Calculation Internally
→ Each tier has a different per-GB/month rate
→ AWS bills based on the tier each object was in during that billing period
→ If an object moved from Frequent → Infrequent on day 15:

  • Days 1–14: billed at Frequent Access rate
  • Days 15–30: billed at Infrequent Access rate
    → Monitoring fee is per-object regardless of tier

REDSHIFT RA3 MANAGED STORAGE INTERNAL MECHANICS

1 → Tiered Storage Architecture

┌─────────────────────────────────┐
│      REDSHIFT RA3 NODE          │
│                                 │
│  ┌───────────────────────────┐  │
│  │   LOCAL SSD CACHE         │  │     ← Hot data (frequently accessed blocks)
│  │   (Large NVMe drives)     │  │     ← Managed automatically by Redshift
│  └────────────┬──────────────┘  │
│               │                 │
│  ┌────────────▼──────────────┐  │
│  │   REDSHIFT MANAGED        │  │     ← Warm/Cold data
│  │   STORAGE (S3-backed)     │  │     ← Virtually unlimited
│  │   $0.024/GB/month         │  │     ← Automatic data tiering
│  └───────────────────────────┘  │
│                                 │
└─────────────────────────────────┘

2 → Auto-Tiering Between SSD and Managed Storage
→ Redshift tracks block-level access patterns for every table
→ Frequently scanned blocks → kept on local SSD (high IOPS)
→ Infrequently scanned blocks → migrated to S3-backed managed storage
→ If a query needs data from managed storage → fetched to SSD cache automatically
→ Cache is large enough that most interactive queries hit SSD
→ Historical/archive data stays in cheap S3-backed storage until queried

3 → Why RA3 Decouples Compute from Storage
→ DC2 nodes: Storage capacity = function of number of nodes

  • Need more storage? Add more nodes (even if CPU is idle)
    → RA3 nodes: Storage is independent
  • Need more storage? Managed storage grows automatically
  • Need more compute? Add nodes (storage stays separate)
  • Need less compute? Remove nodes (storage unaffected)

4 → Pause/Resume Internal Mechanics
→ When you PAUSE a Redshift cluster:

  • Cluster state is saved
  • Compute resources are released (you stop paying for compute)
  • Data remains in Managed Storage (you keep paying for storage)
  • Snapshots are NOT taken — data is already in S3-backed storage
    → When you RESUME:
  • Compute resources are re-provisioned
  • Cluster metadata is restored
  • Local SSD cache is cold (first queries may be slower until cache warms up)
  • Resume typically takes 30–60 seconds

SAVINGS PLANS INTERNAL MECHANICS

1 → How Commitment Works
→ You commit to a specific $/hour of compute usage
→ Example: Commit $4.50/hour for 1 year
→ AWS applies the Savings Plan discount to your usage up to $4.50/hour
→ Usage above $4.50/hour is billed at on-demand rates
→ Usage below $4.50/hour — you still pay $4.50/hour (you lose the unused portion)

2 → Application Order
→ AWS applies Savings Plans in this order:

  1. Savings Plans are applied to usage with the highest savings percentage first
  2. This maximizes your discount automatically
  3. If you have both EC2 Instance SP and Compute SP → EC2 Instance SP applied first (higher discount)

3 → Compute Savings Plan Flexibility
→ You commit $4.50/hour
→ Today you run m5.xlarge in us-east-1
→ Tomorrow you switch to c6i.2xlarge in eu-west-1
→ Savings Plan still applies — it's instance-family-agnostic and region-agnostic
→ Even applies to Fargate and Lambda usage
→ This is why Compute SP is recommended over EC2 Instance SP for most organizations

📈 SCALABILITY CONSIDERATIONS

→ Cost Optimization Scales Differently Than Performance

  • At $10K/month — manual review is sufficient
  • At $100K/month — need automated tooling (Budgets, Anomaly Detection, Storage Lens)
  • At $1M/month — need dedicated FinOps team + third-party tools (CloudHealth, Spot.io, Kubecost)
  • At $10M/month — need organizational FinOps practice with per-team accountability

→ S3 Cost at Scale

  • 1 TB: Storage class doesn't matter much (~$23/month Standard)
  • 100 TB: Storage class matters significantly (~$2,300/month Standard vs $400/month Glacier)
  • 1 PB: Storage class is a strategic decision (~$23,000/month Standard vs $1,000/month Deep Archive)
  • Intelligent-Tiering monitoring fee at 1 billion objects: ~$2,500/month — still worth it if it saves $20,000+ in tiering

→ Redshift Cost at Scale

  • < 500 GB: Consider Redshift Serverless (Project 53) — may be cheaper
  • 500 GB – 10 TB: RA3.xlplus with Reserved Nodes — sweet spot
  • 10 TB – 100 TB: RA3.4xlarge or RA3.16xlarge with Reserved Nodes
  • 100 TB+: RA3 with aggressive use of Spectrum (Project 28) for cold data

→ Savings Plans at Scale

  • Small organization: One Compute Savings Plan covering baseline
  • Large organization: Layered approach:
    • EC2 Instance Savings Plans for known stable workloads (highest discount)
    • Compute Savings Plans for flexible workloads (medium discount)
    • On-demand for burst/unpredictable (no commitment)

→ Multi-Account Cost Governance (builds on Project 54)

  • AWS Organizations consolidated billing → single payer account
  • Volume discounts apply across all accounts
  • Savings Plans purchased in management account → shared across member accounts
  • Reserved Instances can be shared across accounts in the same Organization
