PROJECT 18: THE ANALYST WHO DIDN'T WANT TO SPIN UP A SERVER — Athena Querying S3 via Glue Catalog

🔴 HYPOTHETICAL PROBLEM

→ ShopNow (from Projects 1-17) now has:

  • S3 data lake with raw data from 5 sources (Project 17)
  • Glue Data Catalog with 5 tables cataloged (Project 17)
  • Redshift cluster for heavy analytical workloads (Project 16)
  • RDS MariaDB for transactional operations (Projects 9, 14)

→ The BI team has 5 analysts who need to answer ad-hoc questions QUICKLY:

  • "How many orders were placed last Tuesday?"
  • "What's the average clickstream session duration by device type?"
  • "Which marketing campaign had the highest conversion rate?"
  • "How many customers from Maharashtra ordered in January?"

→ Problem 1: The Redshift cluster costs ~₹1,30,000/month ($1,598)
→ The BI team uses it for 8 hours a day → 16 hours idle → paying for unused compute
→ The CFO says: "Why are we paying for a warehouse 24/7 when analysts work 8 hours?"
→ For ad-hoc queries that run a few times per day → a full cluster is OVERKILL

→ Problem 2: Loading data INTO Redshift takes time
→ Data arrives in S3 → must run COPY command (Project 22) → data loads into Redshift
→ This ETL pipeline introduces delay → analysts can't query FRESH S3 data immediately
→ They want to query the CSV/Parquet files in S3 DIRECTLY — the moment they land

→ Problem 3: Some queries scan a FEW megabytes of data
→ "How many orders yesterday?" → scans 1 partition → ~5 MB of data
→ Running this on Redshift (2 RA3.xlplus nodes) is like driving a truck to the grocery store
→ They need a LIGHTWEIGHT query engine for simple ad-hoc analysis

→ Problem 4: The data engineering team doesn't want to MANAGE another service
→ Redshift requires: Cluster sizing, WLM tuning, VACUUM, ANALYZE, monitoring
→ They want: "Write SQL → Get results → Pay per query → Zero management"

→ Problem 5: Non-technical stakeholders (product managers, marketing leads) need self-service analytics
→ They can write basic SQL but can't manage infrastructure
→ They need a service where they just type SQL and get answers

→ The CTO demands:

  • Analysts must query S3 data WITHOUT loading it anywhere
  • Pay ONLY for queries actually run — not for idle compute
  • ZERO infrastructure to manage — no servers, no clusters, no scaling
  • Use the Glue Data Catalog (Project 17) as the metadata source
  • Queries must leverage partitioning for cost and performance optimization

🟡 HOW TO APPROACH IT

→ This is a SERVERLESS ANALYTICS problem
→ AWS Athena is a serverless, interactive query service that runs SQL on S3 data

Query 2: Aggregation with Partition Pruning

SELECT 
    order_status,
    COUNT(*) as order_count,
    SUM(total_amount) as total_revenue,
    AVG(total_amount) as avg_order_value
FROM shopnow_raw.orders
WHERE year = '2024' AND month = '12'
GROUP BY order_status
ORDER BY total_revenue DESC;

→ This query scans ONLY December 2024 data (1 month out of 15+ months)
→ Partition pruning: WHERE year = '2024' AND month = '12' → Athena reads only 31 partitions
→ Without partition filter → Athena would scan ALL 380+ partitions → 12x more data → 12x more cost

Query 3: JOIN Across Tables

SELECT 
    p.category,
    COUNT(DISTINCT o.customer_id) as unique_customers,
    SUM(o.total_amount) as total_revenue
FROM shopnow_raw.orders o
JOIN shopnow_raw.products p ON o.product_id = p.product_id
WHERE o.year = '2025' AND o.month = '01'
GROUP BY p.category
ORDER BY total_revenue DESC;

→ Joins orders (partitioned, CSV) with products (non-partitioned, JSON)
→ Athena handles cross-format joins seamlessly — CSV JOIN JSON → works
→ Partition filter on orders → scans only January 2025 → fast

Query 4: Clickstream Analysis (Parquet — Columnar)

SELECT 
    device_type,
    COUNT(*) as page_views,
    COUNT(DISTINCT session_id) as sessions,

STEP 4 — OPTIMIZE QUERIES WITH PARTITIONING AND COLUMNAR FORMAT

→ This is the MOST important section for cost and performance optimization
→ Athena charges per TB SCANNED → reducing data scanned = reducing cost

Optimization 1: Partition Pruning (ALWAYS Use Partition Filters)

BAD (Full table scan — expensive):
  SELECT COUNT(*) FROM shopnow_raw.orders;
  → Scans: ALL 380+ partitions → ALL 15 GB → Cost: $0.075

GOOD (Partition pruned — cheap):
  SELECT COUNT(*) FROM shopnow_raw.orders WHERE year = '2025' AND month = '01';
  → Scans: 31 partitions → ~1.3 GB → Cost: $0.0065

SAVINGS: 91% cost reduction

→ ALWAYS include partition columns in WHERE clauses
→ Athena can only prune partitions that are referenced in WHERE
→ Even WHERE year IS NOT NULL doesn't help → must use equality or range

Effective Partition Filters:

FilterPruning?Why
WHERE year = '2024'✅ YESEquality on partition key
WHERE year = '2024' AND month = '01'✅ YESMultiple partition keys
WHERE year >= '2024'✅ YESRange on partition key
WHERE year IN ('2024', '2025')✅ YESIN list on partition key
WHERE order_date = '2024-01-15'❌ NOorder_date is a COLUMN, not a partition key
WHERE year IS NOT NULL❌ NOIS NOT NULL doesn't prune
No WHERE clause❌ NOFull scan of all partitions

Optimization 2: Use Columnar Format (Parquet/ORC)

CSV file: 100 columns, query needs 3 columns
  → Athena reads ALL 100 columns → scans 100% of data

Parquet file: 100 columns, query needs 3 columns
  → Athena reads ONLY 3 column chunks → scans ~3% of data

Cost impact for 10 GB table:
  CSV: 10 GB scanned × $5/TB = $0.05
  Parquet: 0.3 GB scanned × $5/TB = $0.0015

SAVINGS: 97% cost reduction

→ Converting CSV to Parquet is done by Glue ETL (Project 21)
→ This is one of the MOST impactful optimizations in the data lake

Columnar Format Comparison:

FormatCompressionColumn PruningPredicate PushdownAthena Support
CSVNone (unless gzipped)❌ NO❌ NO✅ Full
JSONNone (unless gzipped)❌ NO❌ NO✅ Full
ParquetBuilt-in (Snappy/GZIP/ZSTD)✅ YES✅ YES✅ Full
ORCBuilt-in (ZLIB/Snappy)✅ YES✅ YES✅ Full
AvroBuilt-in (Deflate/Snappy)❌ NO (row-based)❌ NO✅ Full

→ Parquet and ORC are the BEST formats for Athena
→ Parquet is more widely used → preferred choice

Predicate Pushdown:

→ With Parquet → Athena can push WHERE filters INTO the file reader
→ Each Parquet file has min/max statistics per column per row group
→ If WHERE amount > 1000 and a row group's max(amount) = 500 → skip entire row group

Parquet file row group statistics:
  Row Group 1: amount min=10, max=500     → WHERE amount > 1000 → SKIP
  Row Group 2: amount min=200, max=5000   → WHERE amount > 1000 → READ (may contain matches)
  Row Group 3: amount min=50, max=800     → WHERE amount > 1000 → SKIP

Result: Only 1 out of 3 row groups read → 67% less I/O

Optimization 3: Compress CSV Files

→ If you MUST use CSV → compress with GZIP

Uncompressed CSV: 10 GB → Athena scans 10 GB → $0.05
GZIP compressed CSV: 2 GB → Athena scans 2 GB → $0.01

SAVINGS: 80% cost reduction

→ Athena natively reads: GZIP, BZIP2, LZ4, ZSTD, Snappy compressed files
→ No configuration needed → just upload compressed files → Athena auto-detects
→ Glue Crawler (Project 17) also auto-detects compression

→ BUT: GZIP CSV still reads ALL columns (no column pruning) → Parquet is still better

Optimization 4: Optimal File Sizing

→ Athena performs best with files between 128 MB and 512 MB
→ Too many small files → excessive S3 LIST operations → slow
→ Too few large files → limited parallelism → slow

File SizeProblemSolution
< 10 MB (many small files)Overhead per file exceeds read timeCombine into larger files
10-128 MBAcceptable but not optimalConsider combining
128-512 MBOPTIMALBest balance of parallelism and overhead
> 512 MBLimited parallelism within single fileSplit into smaller files

→ Glue ETL (Project 21) can coalesce small files into optimal sizes during transformation

Optimization 5: CTAS (Create Table As Select) for Materialized Results

-- Create a new optimized table from a complex query
CREATE TABLE shopnow_curated.monthly_revenue
WITH (
    format = 'PARQUET',
    partitioned_by = ARRAY['year', 'month'],
    external_location = 's3://shopnow-data-lake/curated/monthly_revenue/'
) AS
SELECT 
    category,
    SUM(total_amount) as revenue,
    COUNT(*) as order_count,
    year,
    month
FROM shopnow_raw.orders
GROUP BY category, year, month;

→ CTAS creates a NEW table with query results stored in S3 as Parquet
→ The new table is registered in the Glue Catalog automatically
→ Subsequent queries on monthly_revenue are MUCH faster (pre-aggregated, Parquet, partitioned)
→ This is Athena's version of a "materialized view"

✅ WHAT SHOULD DO

→ Always include partition columns in WHERE clauses — partition pruning is the #1 cost optimization
→ Always use Parquet or ORC format for frequently queried data — column pruning saves 80-97%
→ Always configure workgroup data limits — prevents accidental full-table scans from costing hundreds
→ Always set query result location with lifecycle rules — auto-delete old results
→ Always enable CloudWatch metrics on workgroups — monitor cost and performance
→ Always compress CSV files with GZIP if Parquet conversion isn't possible yet
→ Always use CTAS for repeated complex queries — pre-compute and store as optimized Parquet
→ Always create views for common JOIN patterns — simplifies analyst workflow
→ Always use specific column names in SELECT — avoid SELECT * (reads all columns)
→ Always use TRY_CAST instead of CAST for unreliable data — prevents query failures
→ Always use saved queries for standard reports — consistency across team
→ Always configure separate workgroups per team — cost isolation and control

❌ WHAT SHOULD NOT DO

→ Never run SELECT * on large unpartitioned tables — scans everything → expensive
→ Never skip partition filters on partitioned tables — full scan is 10-100x more expensive
→ Never use Athena for frequent (hundreds/day) recurring dashboard queries — Redshift is cheaper at scale
→ Never store query results indefinitely — add lifecycle rules to auto-expire
→ Never use CSV for analytical tables when Parquet is possible — CSV reads all columns always
→ Never let all users share one workgroup — no cost isolation or control
→ Never run queries without understanding the data size — check table properties first
→ Never use LIKE '%pattern%' on non-partitioned string columns — forces full scan
→ Never assume query results persist forever — they're in S3 with lifecycle rules
→ Never create tables directly in Athena when Glue Crawler can automate it
→ Never ignore QueryQueueTime metrics — growing queue time means concurrency limits are being hit
→ Never use Athena for UPDATE/DELETE operations — it's a read-mostly engine

→ Do NOT use Athena for OLTP workloads (INSERT/UPDATE/DELETE per row) → Use RDS (Project 9)
→ Do NOT use Athena for sub-second dashboard queries → Use Redshift or ElastiCache
→ Do NOT use Athena for high-concurrency workloads (100+ simultaneous queries) → Use Redshift with concurrency scaling
→ Do NOT use Athena for real-time streaming analysis → Use Kinesis Analytics
→ Do NOT use Athena as a data transformation tool → Use Glue ETL (Project 21) for large-scale transforms
→ Do NOT use Athena when the same query runs hundreds of times daily → Redshift is more cost-effective

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

1 → HOW ATHENA EXECUTES A QUERY

Analyst submits: SELECT category, SUM(amount) FROM orders WHERE year='2024' GROUP BY category
        │
        ▼
Step 1: QUERY SUBMISSION
        Athena receives the SQL text
        Validates syntax
        Checks IAM permissions: athena:StartQueryExecution, glue:GetTable, s3:GetObject
        │
        ▼
Step 2: QUERY PLANNING
        │
        ├── Athena calls Glue Catalog:
        │   GetTable("shopnow_raw", "orders")
        │   → Returns: columns, types, format (CSV), SerDe, location, partition keys
        │
        ├── Athena calls Glue Catalog:
        │   GetPartitions(filter: year='2024')
        │   → Returns: 365 partition locations (all days in 2024)
        │
        ├── Athena creates execution plan:
        │   → Scan 365 S3 files
        │   → Parse CSV using LazySimpleSerDe
        │   → Project only: category, amount (but CSV reads all columns)
        │   → Filter: (partition pruning already applied)
        │   → Aggregate: GROUP BY category, SUM(amount)
        │   → Sort: ORDER BY (if specified)
        │
        └── Plan distributed across Athena workers
        │
        ▼
Step 3: QUERY EXECUTION (Distributed)
        │
        ├── Athena allocates WORKERS from a shared pool
        │   (You don't control the number — Athena auto-scales)
        │
        ├── Worker 1: Reads S3 files for partitions day=01 to day=73
        │   → Parses CSV → Extracts category + amount → Partial aggregation
        │   → Sends partial result: {Shoes: 5M, Electronics: 8M, ...}
        │
        ├── Worker 2: Reads S3 files for partitions day=74 to day=146
        │   → Same process → partial result
        │
        ├── Worker 3-5: Similar parallel processing
        │
        └── All workers process SIMULTANEOUSLY
        │
        ▼
Step 4: RESULT AGGREGATION
        Coordinator worker receives all partial results
        Merges: {Shoes: 5M+4.8M+5.2M+..., Electronics: ...}
        Applies final ORDER BY
        │
        ▼
Step 5: RESULT STORAGE
        Writes result to: s3://shopnow-athena-results/shopnow-analytics/{query-id}.csv
        Writes metadata to: s3://.../{query-id}.csv.metadata
        │
        ▼
Step 6: RESULT DELIVERY
        Returns result to console/API
        Records metrics: data scanned, execution time, cost
        │
        ▼
Step 7: BILLING
        Data scanned: 4.5 GB (365 days × ~12 MB average per day)
        Cost: 4.5 GB × $5/TB = $0.0225
        Minimum charge: 10 MB per query

2 → HOW PARTITION PRUNING WORKS INTERNALLY

TABLE: orders
  Partition keys: year (string), month (string), day (string)
  Total partitions: 380

QUERY: WHERE year = '2024' AND month = '12'

Step 1: Athena calls GetPartitions with filter expression
        Filter: year = '2024' AND month = '12'
        │
        ▼
Step 2: Glue Catalog evaluates filter against partition index
        │
        ├── Total partitions: 380
        ├── Matching: year='2024' AND month='12' → 31 partitions (Dec 1-31)
        └── Returns: 31 partition locations
        │
        ▼
Step 3: Athena generates S3 file list from 31 partition locations
        Each partition → 1 S3 file → 31 files total
        │
        ▼
Step 4: Athena scans ONLY these 31 files
        Ignores: 349 other partitions (92% of data skipped)

WITHOUT partition filter:
  Athena calls GetPartitions without filter → ALL 380 partitions → ALL files scanned
  12x more data → 12x more cost → 12x slower

3 → HOW ATHENA READS DIFFERENT FORMATS

CSV Reading:

S3 File: orders-20240115.csv
  order_id,customer_id,order_date,category,amount,...
  1,5678,2024-01-15,Shoes,999,...
  2,9012,2024-01-15,Electronics,2499,...

Athena process:
  1. Read entire file (cannot skip columns in CSV)
  2. Parse using LazySimpleSerDe with comma delimiter
  3. Cast each value to catalog-defined type
  4. Project only needed columns (discard unused AFTER reading)
  5. Apply remaining filters (WHERE, HAVING)

Data scanned: ENTIRE file size (all columns)

Parquet Reading:

S3 File: click-20240115-00.parquet
  Column chunks: session_id, user_id, page_url, device_type, timestamp, ...

Athena process:
  1. Read Parquet footer (metadata: column locations, statistics)
  2. Identify needed columns: device_type, session_id, time_on_page
  3. Read ONLY those column chunks (skip all other columns)
  4. For each row group: check min/max statistics
     → If predicate pushdown can eliminate row group → skip
  5. Decompress read column chunks
  6. Apply remaining filters

Data scanned: ONLY the bytes of needed columns
  → 3 columns out of 35 → ~8.5% of file size

JSON Reading:

S3 File: products-snapshot-202501.json
  {"product_id": 1, "name": "Running Shoes", "category": "Shoes", "price": 999}
  {"product_id": 2, "name": "Laptop Pro", "category": "Electronics", "price": 45999}

Athena process:
  1. Read entire file (JSON is row-based — cannot skip fields efficiently)
  2. Parse each line as JSON using JsonSerDe
  3. Extract fields matching catalog columns
  4. Apply filters

Data scanned: ENTIRE file size (similar to CSV — row-based format)

→ Parquet/ORC = columnar → reads only needed columns → CHEAP
→ CSV/JSON = row-based → reads everything → EXPENSIVE
→ This is why converting to Parquet (Project 21) is so important

🚨 COMMON PITFALLS AND FAILURE MODES

#PitfallWhat HappensHow to Avoid
1SELECT * on unpartitioned tableScans entire table → expensive, slowAlways SELECT specific columns, always use partition filters
2No partition filter on partitioned tableScans all partitions → 10-100x more expensiveAlways include partition keys in WHERE clause
3Using CSV when Parquet is possibleReads all columns → 10-30x more data scannedConvert to Parquet via Glue ETL (Project 21)

5Query results accumulating in S3Storage costs grow over monthsAdd lifecycle rule to auto-delete old results

7Date columns inferred as strings by crawlerCan't use date functions (DATE_ADD, DATE_DIFF)CAST to DATE in queries or fix Glue Catalog column type
8Many small files (< 10 MB each)Slow queries due to S3 LIST overheadCoalesce small files via Glue ETL (Project 21)
9Concurrent query limit hitQueries queued → analysts wait → frustrationRequest limit increase or stagger query schedules
10Partition values as strings vs integersWHERE month = 1 doesn't match partition month = '01'Use string comparison: WHERE month = '01'

📈 SCALABILITY CONSIDERATIONS

→ Query Scalability

  • Athena auto-scales workers per query → larger data = more workers
  • No provisioning needed → handles 1 MB or 1 TB transparently
  • Concurrent query limit (25) is the bottleneck → request increase for heavy usage
  • Workgroup isolation doesn't increase total concurrency

→ Data Volume Scalability

  • Athena can query petabytes of data in S3
  • Performance scales with: partitioning, columnar format, compression
  • No data loading required → query new data immediately after it lands in S3
  • Catalog supports millions of tables and billions of partitions

→ Cost Scalability

  • Cost scales linearly with data scanned (not data stored)
  • Optimization reduces cost sub-linearly → more data but smarter queries
  • At very high query volumes → Redshift becomes more cost-effective
  • Hybrid: Use Athena for ad-hoc, Redshift for production dashboards

→ Team Scalability

  • Workgroups isolate teams → add workgroups as teams grow
  • IAM controls who can query which databases/tables
  • Views simplify access for non-technical users
  • Saved queries ensure consistency across the team

→ How to debug "query failed":

STEP 1: Read the error message
        Athena Console → Recent queries → Click failed query → Error details
        Common errors:
          - "HIVE_METASTORE_ERROR" → Glue Catalog issue (table/partition missing)
          - "HIVE_BAD_DATA" → Data doesn't match catalog schema
          - "HIVE_CANNOT_OPEN_SPLIT" → S3 access denied or file not found
          - "GENERIC_USER_ERROR" → SQL syntax error
          - "EXCEEDED_MEMORY_LIMIT" → Query too complex for available memory

STEP 2: For "HIVE_BAD_DATA"
        Data format doesn't match SerDe definition
        → Check if files are actually CSV/Parquet/JSON as catalog says
        → Check delimiter: is it comma, tab, or pipe?
        → Re-run crawler to refresh schema

STEP 3: For "HIVE_CANNOT_OPEN_SPLIT"
        → S3 files missing at the cataloged location
        → IAM role doesn't have s3:GetObject permission
        → KMS encryption without proper decrypt permissions
        → Check: Does the S3 path still contain data?

STEP 4: For "EXCEEDED_MEMORY_LIMIT"
        → Query is too complex (too many JOINs, too large GROUP BY)
        → Break into smaller queries using CTAS (materialize intermediate results)
        → Add more restrictive WHERE clauses to reduce data

STEP 5: For access denied
        → Check IAM permissions: athena:StartQueryExecution
        → Check S3 permissions: s3:GetObject on data location
        → Check S3 permissions: s3:PutObject on result location
        → Check Glue permissions: glue:GetTable, glue:GetPartitions
        → Check workgroup access: correct workgroup selected

STEP 6: For missing data (query returns empty)
        → Are partitions registered for the date you're querying?
        → Run MSCK REPAIR TABLE or check Glue Catalog partitions
        → Is the partition filter correct? (string '01' vs integer 1)
