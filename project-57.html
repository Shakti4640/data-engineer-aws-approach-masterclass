Step D3 — KMS Key Policy Structure (Critical for Zero-Trust)

→ Every KMS key has a Key Policy (resource-based policy)
→ Key Policy is the PRIMARY access control for KMS keys — IAM policies alone are NOT sufficient

→ Key Policy must explicitly:

SectionPurpose
Key AdministratorsWho can manage the key (enable/disable, modify policy, schedule deletion)
Key UsersWho can use the key for encrypt/decrypt operations
Key GrantsWho can delegate key usage to other AWS services
Root Account AccessAllow root account to manage key policy (prevents lockout)

→ Principle of Least Privilege in Key Policy:

  • Key Administrators should NOT be Key Users (separation of duties)
  • Key Users should only have kms:Encrypt, kms:Decrypt, kms:GenerateDataKey — not kms:*
  • Grant permission should be limited to specific services (e.g., RDS needs grants to encrypt storage)

4 → Log File Validation Mechanics

→ When enabled, CloudTrail creates a digest file every hour
→ Digest file contains:

  • Hash (SHA-256) of each log file delivered in that hour
  • Hash of the previous digest file (creating a chain)
  • Digital signature using CloudTrail's private key

→ Validation process:

  • Start from latest digest file
  • Verify its signature using CloudTrail's public key
  • Verify each log file hash matches what's recorded in the digest
  • Follow the chain backward to verify historical integrity
  • If any log file was modified or deleted → validation fails → tampering detected

→ This creates a cryptographic chain of custody for audit logs

5 → Organization Trail Mechanics

→ When you create an Organization trail in the management account:

  • CloudTrail automatically creates a trail in EVERY member account
  • Member accounts CANNOT modify or delete this trail (if SCP prevents it)
  • All logs flow to the MANAGEMENT account's S3 bucket
  • Bucket structure: s3://bucket/AWSLogs/{account-id}/CloudTrail/{region}/yyyy/mm/dd/
  • Each account's logs are separated by account ID prefix
  • Member accounts can ALSO have their own additional trails (for account-specific logging)

2 → GuardDuty Does NOT Read Your Data

→ Common misconception: GuardDuty reads your S3 objects or database content
→ Reality: GuardDuty only analyzes METADATA:

  • CloudTrail: API call metadata (who, what, when, from where)
  • VPC Flow Logs: Network connection metadata (source, destination, port, bytes)
  • DNS Logs: Domain name resolution queries (what domains are being resolved)

3 → Threat Intelligence Sources

→ GuardDuty uses multiple threat intelligence feeds:

  • AWS internal threat intelligence
  • CrowdStrike threat data
  • Proofpoint threat data

4 → Baseline Learning

  • Which API calls are normal for each IAM principal
  • What network communication patterns are typical
  • What DNS queries are standard

→ After baseline period → deviations from normal trigger findings
→ This is why GuardDuty may generate more false positives in the first 2 weeks
→ After baseline stabilizes → findings become more accurate

→ GuardDuty aggregates related events into a single finding

  • Finding includes count, first seen, last seen, severity
  • Finding is UPDATED if activity continues (not duplicated)

KMS INTERNAL MECHANICS (DEEP DIVE)

1 → Envelope Encryption

→ KMS does NOT encrypt your data directly with the CMK
→ It uses ENVELOPE ENCRYPTION:

Step 1: Application calls KMS → GenerateDataKey(CMK-ID)
          │
          ▼
Step 2: KMS returns TWO things:
        ┌──────────────────────────────────────┐
        │  Plaintext Data Key: abc123...        │  ← Used to encrypt data
        │  Encrypted Data Key: xyz789...        │  ← Encrypted copy of data key
        │  (encrypted using the CMK)            │     (stored alongside encrypted data)
        └──────────────────────────────────────┘
          │
          ▼
Step 3: Application encrypts data using Plaintext Data Key
          │
          ▼
Step 4: Application stores:
        ┌──────────────────────────────────────┐
        │  Encrypted Data + Encrypted Data Key  │
        └──────────────────────────────────────┘
          │
          ▼
Step 5: Application DELETES the Plaintext Data Key from memory

→ To decrypt:

Step 1: Application reads Encrypted Data Key from storage
          │
          ▼
Step 2: Application calls KMS → Decrypt(Encrypted Data Key)
          │
          ▼
Step 3: KMS decrypts the data key using the CMK → returns Plaintext Data Key
          │
          ▼
Step 4: Application decrypts data using Plaintext Data Key
          │
          ▼
Step 5: Application DELETES the Plaintext Data Key from memory

→ Why envelope encryption?

  • CMK never leaves KMS hardware (FIPS 140-2 Level 3 HSMs)
  • Only the small data key is transmitted (not your multi-GB data)
  • KMS API has a 4 KB encryption limit — cannot encrypt large data directly
  • Data keys can encrypt unlimited data locally (fast, no network calls)
  • Only key operations (GenerateDataKey, Decrypt) hit KMS API

2 → Key Rotation Internal Mechanics

→ When automatic rotation is enabled:

  • Every 365 days, KMS generates NEW key material (backing key)
  • The CMK ID, ARN, alias all remain UNCHANGED
  • Old backing key is PRESERVED indefinitely
Year 1: CMK-abc → Backing Key v1 (encrypts new data)
Year 2: CMK-abc → Backing Key v2 (encrypts new data)
                   Backing Key v1 (still available for decryption)
Year 3: CMK-abc → Backing Key v3 (encrypts new data)
                   Backing Key v2 (available for decryption)
                   Backing Key v1 (available for decryption)

→ New encryptions automatically use the latest backing key
→ Decryptions automatically use the correct backing key (KMS tracks which key encrypted which data)
→ No re-encryption needed — completely transparent to applications
→ This is why rotation is safe and should ALWAYS be enabled

3 → KMS Key Policy vs IAM Policy Interaction

→ UNIQUE to KMS: Key Policy is the PRIMARY authorization mechanism
→ Unlike S3/SQS where IAM policy alone can grant access:

  • KMS Key Policy MUST explicitly allow the account (or specific principals)
  • If Key Policy doesn't grant access → IAM policy is IGNORED
  • Key Policy is the gatekeeper — IAM policy is secondary

→ Default Key Policy includes:

StatementPurpose
Allow root account full KMS accessEnables IAM policies to work (delegates to IAM)
Allow key administratorsSpecific principals who can manage the key
Allow key usersSpecific principals who can use the key
Allow grantsSpecific principals who can create grants (for AWS services)

→ If the root account statement is removed → ONLY principals explicitly named in the Key Policy can access the key
→ This is the most restrictive configuration — used for highest security keys
→ WARNING: Removing root access from Key Policy can cause key lockout — only do with careful planning

📈 SCALABILITY CONSIDERATIONS

→ CloudTrail at Scale

  • Single Organization trail handles ALL accounts automatically
  • Log volume grows linearly with API activity
  • 5 accounts with moderate usage: ~2-5 GB/day
  • 50 accounts with heavy usage: ~50-200 GB/day
  • Cost management: Use S3 Intelligent-Tiering on CloudTrail bucket (connects to Project 56)
  • Query performance: Partition CloudTrail logs by account/region/date → Athena queries stay fast
  • Data Events: Can generate 10-100x more volume than Management Events → budget carefully

→ GuardDuty at Scale

  • Pricing: Based on volume of CloudTrail events, VPC Flow Logs, and DNS logs analyzed
  • 5 accounts: ~$50-200/month
  • 50 accounts: ~$500-5,000/month depending on activity
  • Volume discounts apply automatically
  • GuardDuty processes data independently per account → no cross-account performance impact
  • Organization-wide enablement is automatic for new accounts

→ AWS Config at Scale

  • Cost: $0.003 per configuration item recorded
  • 5 accounts with 1,000 resources each → ~$15/month
  • 50 accounts with 10,000 resources each → ~$1,500/month
  • Config rules: $0.001 per rule evaluation per resource
  • 20 rules × 10,000 resources × 50 accounts = significant cost
  • Optimization: Use periodic evaluation (every 24 hours) instead of change-triggered for non-critical rules

→ Security Hub at Scale

  • No additional charge for Security Hub itself
  • Charges come from Config rules that Security Hub enables
  • Finding ingestion: First 10,000 findings/month/account free → $0.00003 per finding after
  • Cross-account aggregation has no additional cost
  • At 50 accounts → consider delegated administrator pattern for manageable console experience

→ VPC Endpoints at Scale

  • Gateway Endpoints (S3, DynamoDB): FREE regardless of scale
  • Interface Endpoints: $0.01/hour/AZ = ~$7.20/month per AZ per endpoint
  • 10 Interface Endpoints × 3 AZs × 5 VPCs = 150 endpoints = ~$1,080/month
  • Optimization: Not every VPC needs every endpoint → only create endpoints for services actually used in that VPC
  • Shared Services VPC pattern: Central VPC with all endpoints → other VPCs access through Transit Gateway (Project 50)

→ KMS at Scale

  • Key storage: $1/month per CMK
  • API calls: $0.03 per 10,000 requests
  • S3 Bucket Key feature reduces KMS API calls by up to 99% for S3 encryption
  • At high scale (millions of encryptions/day): Enable S3 Bucket Key → use envelope encryption efficiently
  • Cross-account key sharing: Key policies must be maintained per consumer account → use Organizations conditions for simplicity
