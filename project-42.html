The Data Engineering Lead asks:

🟡 HOW TO APPROACH IT

→ First — understand what Lake Formation IS and what problem it solves

Without Lake Formation (Current State):

Access Control Stack:
  S3 Bucket Policy → controls WHO can read WHICH S3 prefixes
  IAM Policy → controls WHO can call WHICH S3 actions
  KMS Key Policy → controls WHO can decrypt WHICH keys
  Glue Catalog → ONLY controls WHO can modify catalog metadata
  
  RESULT: Access is at FILE/PREFIX level — NOT at TABLE/COLUMN level
  If you can read the S3 file → you can read ALL columns in that file

With Lake Formation (Target State):

Access Control Stack:
  Lake Formation Permissions → controls WHO can access WHICH database/table/column
  Lake Formation enforces permissions AT QUERY TIME
  Even if someone has S3 GetObject → Lake Formation blocks column access
  
  RESULT: Access is at TABLE and COLUMN level
  You can read the S3 file but Lake Formation filters out restricted columns

Lake Formation's Core Concept:

→ Lake Formation sits BETWEEN the Glue Data Catalog and the query engines (Athena, Redshift Spectrum, Glue ETL)
→ When a query runs → Lake Formation checks:

  1. Does this principal have permission to access this DATABASE?
  2. Does this principal have permission to access this TABLE?
  3. Does this principal have permission to access these specific COLUMNS?
  4. Are there any row-level filters applied?
    → If any check fails → the query is blocked or filtered

How Lake Formation Relates to IAM:

LayerWhat It ControlsGranularity
IAMWho can call AWS API actions (glue:GetTable, athena:StartQueryExecution)API-level
Lake FormationWho can access which data (databases, tables, columns, rows)Data-level

→ IAM and Lake Formation work TOGETHER:

  • IAM allows the API call (e.g., "this role can call Athena")
  • Lake Formation allows the data access (e.g., "this role can see columns A, B, C but not D")
  • BOTH must allow for the query to succeed

→ The approach:

  • Step 1 — Enable Lake Formation and configure Data Lake Administrators
  • Step 2 — Register S3 data locations with Lake Formation
  • Step 3 — Grant Lake Formation permissions on databases and tables (intra-account)
  • Step 4 — Share catalog resources cross-account using AWS RAM
  • Step 5 — Accept shared resources and create Resource Links in consumer accounts
  • Step 6 — Configure column-level security
  • Step 7 — Configure cross-account query flow
  • Step 8 — Set up governance dashboard and audit trail

Step 1b — Change Default Catalog Permissions (CRITICAL):

→ Lake Formation → "Administrative roles and tasks" → "Data catalog settings"

Default Behavior (What you MUST change):

→ By default, Lake Formation has a compatibility mode enabled:
→ "Use only IAM access control for new databases" → ✅ CHECKED (default)
→ "Use only IAM access control for new tables in new databases" → ✅ CHECKED (default)

What this means:
→ With these defaults ON → Lake Formation permissions are IGNORED
→ Access is controlled ONLY by IAM policies (same as before Lake Formation)
→ This exists for backward compatibility — so enabling Lake Formation doesn't break existing workloads

What you must change for governance:
→ UNCHECK both settings:
→ "Use only IAM access control for new databases" → ❌ UNCHECKED
→ "Use only IAM access control for new tables in new databases" → ❌ UNCHECKED

SettingDefaultChange ToWhy
Use only IAM access control for new databases✅ Checked❌ UncheckedEnable Lake Formation permissions for databases
Use only IAM access control for new tables in new databases✅ Checked❌ UncheckedEnable Lake Formation permissions for tables

WARNING: Impact of Unchecking:
→ Once unchecked → ALL new databases and tables are governed by Lake Formation
→ Existing databases and tables MAY lose access if Lake Formation permissions are not granted
→ Before unchecking → Grant Lake Formation permissions to all existing roles that need catalog access

Recommended Migration Path:

Step A: Identify all IAM roles that currently access Glue Catalog
        (GlueETLRole, AthenaQueryRole, RedshiftSpectrumRole, etc.)

Step B: Grant Lake Formation permissions to each role for each database/table they need

Step C: THEN uncheck the default IAM-only settings

Step D: Test all workloads — verify nothing broke

Step E: If something breaks → the role is missing Lake Formation permission → grant it

Understanding the Service-Linked Role:

WITHOUT Lake Formation Registration:
  AthenaQueryRole needs: s3:GetObject on bucket + kms:Decrypt on key
  GlueETLRole needs: s3:GetObject, s3:PutObject on bucket + kms:Decrypt, kms:GenerateDataKey
  RedshiftSpectrumRole needs: s3:GetObject on bucket + kms:Decrypt
  → Each role has direct S3 and KMS permissions
  → Adding a new role = adding S3 + KMS permissions to that role

WITH Lake Formation Registration:
  Lake Formation service-linked role has: s3:GetObject, s3:PutObject + kms:Decrypt, kms:GenerateDataKey
  AthenaQueryRole needs: ONLY Lake Formation Select permission on the table
  GlueETLRole needs: ONLY Lake Formation Select + Insert permission
  → Individual roles do NOT need direct S3 or KMS permissions
  → Lake Formation mediates all data access through its service role
  → Adding a new role = granting Lake Formation permission (no S3/KMS changes)

CRITICAL: Hybrid Mode vs Pure Lake Formation Mode:

→ For ShopNow → Start with Hybrid mode during migration

Europe's analyst runs:
  SELECT * FROM hq_shared_curated.products LIMIT 10

    │

    │

    ├── Table permission: Select on products → ✅
    ├── Column permissions: ALL columns EXCEPT customer_email, customer_phone, customer_address
    │
    ▼
Athena REWRITES the query internally:
  SELECT product_id, product_name, category, price, weight, ... (45 columns) 
  FROM hq_shared_curated.products LIMIT 10
  
  (customer_email, customer_phone, customer_address are REMOVED from SELECT *)
    │
    ▼
Athena reads from S3 (via Lake Formation service role)
    │
    ▼
Results returned to analyst — 45 columns visible, 3 columns invisible

What if the analyst explicitly queries a restricted column?

Europe's analyst runs:
  SELECT customer_email FROM hq_shared_curated.products LIMIT 10

Athena checks Lake Formation permissions:
    Column customer_email → EXCLUDED for this principal
    
    → Query FAILS with: "Insufficient Lake Formation permission(s): Required Select on customer_email"

→ Column-level security is ENFORCED — not just hidden in the UI
→ Even explicit column reference is blocked
→ This is fundamentally stronger than just hiding columns in a view

→ The data lives in Account 111's S3 bucket
→ Europe's Athena role does NOT need direct S3 access to Account 111's bucket
→ Lake Formation's service-linked role in Account 111 reads the S3 data
→ Data is passed to Athena through Lake Formation's internal mechanisms

✅ WHAT SHOULD DO

→ Always uncheck the default IAM-only settings AFTER granting Lake Formation permissions to existing roles
→ Always use Tag-Based Access Control (TBAC) for cross-account sharing at scale — tags automate future grants
→ Always register S3 data locations with Lake Formation for centralized access mediation
→ Always use column-level exclusion/inclusion for PII protection — stronger than views
→ Always use data filters for row-level security when regional data isolation is needed
→ Always create database-level resource links in consumer accounts — simplifies table visibility
→ Always grant both producer-side AND consumer-side Lake Formation permissions
→ Always enable AWS Organizations RAM sharing for automatic share acceptance
→ Always use Version 3 cross-account settings for full feature support
→ Always add the Lake Formation service-linked role to KMS key policies
→ Always test cross-account queries after setting up resource links
→ Always document the tag taxonomy — it becomes the governance framework

❌ WHAT SHOULD NOT DO

→ Never uncheck default IAM settings before granting Lake Formation permissions — breaks ALL existing workloads
→ Never grant Super permission to external accounts — grants full control including Drop
→ Never skip the resource link creation in consumer accounts — shared resources won't appear in Glue Catalog
→ Never modify AWS RAM shares created by Lake Formation manually — let Lake Formation manage them
→ Never grant grantable permissions to external accounts unless they need to re-delegate within their account
→ Never use named resource grants for 100+ tables — use tag-based access control instead
→ Never forget KMS key policy update for Lake Formation service-linked role — cross-account queries fail silently
→ Never assume column exclusion applies retroactively to cached results — only applies to new queries
→ Never create resource links with the same name as existing local tables — naming conflicts cause errors
→ Never skip consumer-side Lake Formation permission grants — resource links exist but roles can't use them
→ Never use Lake Formation for unstructured data (images, PDFs) — it's designed for tabular catalog data
→ Never rely solely on Lake Formation without CloudTrail — audit trail is essential for compliance

  classification IN [public, internal]

  Tags: region_access=hq_only, classification=confidential

  region_access: hq_only IN [global, europe]? → NO ❌
  → ACCESS DENIED (first non-matching tag short-circuits)

Table: shopnow_curated.products
  Tags: region_access=global, classification=internal

Evaluation:
  region_access: global IN [global, europe]? → YES ✅
  classification: internal IN [public, internal]? → YES ✅
  → ALL tags match → ACCESS GRANTED ✅

→ ALL tag conditions must match (AND logic between tag keys)
→ Multiple values per tag key are OR logic (global OR europe)
→ First non-matching tag → immediate denial

📈 SCALABILITY CONSIDERATIONS

→ Table Count Scaling

  • Named resource grants: Manageable for < 50 tables per account pair
  • Tag-based grants: Scales to thousands of tables — one grant covers all matching tags
  • When HQ adds new table with appropriate tags → auto-shared to all accounts with matching tag grants

→ Account Count Scaling

  • 5 accounts → Individual cross-account grants work
  • 20+ accounts → Use TBAC + AWS Organizations for automatic RAM acceptance
  • 100+ accounts → Use Organization-level grants with OU-based tag policies (Project 54)

→ Column/Row Filter Scaling

  • Column filters: Per-table per-principal — manageable for dozens of tables
  • Data filters: Per-table per-principal with row expressions — more complex
  • For complex filtering needs → Consider creating curated views/tables in producer account

→ Permission Evaluation Performance

  • Lake Formation permission checks add ~100-500ms to query startup
  • Does NOT affect query execution time (once permissions are evaluated)
  • Tag-based evaluation is efficient — O(number of tags) not O(number of tables)
  • Column filtering is applied at read time — minimal overhead for Parquet columnar format

→ Cross-Region Considerations

  • Lake Formation cross-account sharing works best within the same region
  • For cross-region → Consider S3 replication (Project 30) + local Lake Formation in each region
  • Glue Catalog is regional — resource links must be in the same region as the shared catalog

🎯 KEY TAKEAWAY

→ Lake Formation is the GOVERNANCE LAYER for the data lake — it sits between the Glue Data Catalog and query engines to enforce fine-grained access control
→ Tag-Based Access Control (TBAC) is the scalable way to manage cross-account sharing — one tag grant covers unlimited tables with matching tags
→ Column-level security provides PII protection that S3 bucket policies CANNOT — even if someone has S3 GetObject, Lake Formation blocks restricted columns at query time
→ Data filters provide row-level security — regional data isolation without physical data separation
→ Resource links are POINTERS, not copies — schema changes and new partitions propagate automatically from producer to consumer
→ Two-layer permission model: Producer controls WHAT external accounts can access, Consumer controls WHO within their account can use it
→ AWS RAM handles the sharing mechanism, Lake Formation handles the permissions — they work together
→ The Lake Formation service-linked role needs KMS access to decrypt S3 data on behalf of cross-account queries
→ Lake Formation cross-account queries do NOT require direct S3 cross-account access — Lake Formation mediates all data access
→ Always uncheck IAM-only defaults AFTER granting Lake Formation permissions to existing roles — order matters
