Specific Pain Points:

Pain Point 1 — Notification Silos:
→ Each account has its own SNS topics (Project 6)
→ Each account has its own SQS queues (Project 7)
→ Each account has its own EventBridge rules (Project 38)
→ The Shared Services team must log into EACH account individually to see what's happening

Pain Point 2 — Delayed Incident Response:
→ HQ's Glue job fails at 2 AM
→ HQ's SNS topic sends email to HQ data engineers
→ HQ data engineers are sleeping
→ Shared Services on-call team (24/7) has no visibility
→ Issue discovered 6 hours later when HQ team wakes up
→ Downstream pipelines in Europe and US are broken because they depend on HQ's data

Pain Point 3 — Centralized Processing:
→ Shared Services needs to:

  • Aggregate alerts from all accounts into ONE processing queue
  • Apply consistent incident classification logic
  • Route to appropriate response teams based on severity and account
  • Maintain a centralized audit trail of all operational events
    → This requires events from ALL accounts to flow into Shared Services' SQS queue

Pain Point 4 — Fan-Out Complexity:
→ When HQ's Glue job fails:

  • Shared Services monitoring should know (for dashboards)
  • Shared Services incident management should know (for ticket creation)
  • Europe team should know (their downstream pipeline depends on HQ's data)
  • US team should know (their dashboard uses HQ's data)
    → This requires fan-out ACROSS accounts — not just within one account (Project 13)

The Operations Lead asks:

"I need a centralized notification architecture where:
Events from ANY account can publish to a CENTRAL topic or queue in Shared Services.
Shared Services can also push notifications TO other accounts when needed.
All of this with proper security — no account should be able to spam another."

→ This requires cross-account SNS and SQS configuration — applying the trust model from Projects 39-40 to messaging services

Understanding Filter Policies (Extending Project 13 Fan-Out):

→ In Project 13 → All SNS subscribers received ALL messages
→ With filter policies → Each subscriber receives ONLY matching messages

SubscriberFilterReceives
Monitoring QueueNo filterALL alerts (for dashboards and analytics)
Incident Queueseverity: [CRITICAL, HIGH]Only critical and high severity alerts
Email On-Callseverity: [CRITICAL]Only critical alerts (avoid email fatigue)

How filter policies work:
→ Publisher includes message attributes when publishing
→ SNS evaluates each subscription's filter policy against message attributes
→ Only matching messages are delivered to the subscriber
→ Non-matching messages are silently dropped for that subscriber (not lost — other subscribers may still receive)

CRITICAL: aws:PrincipalOrgID vs aws:SourceAccount:

Condition KeyWhat It ChecksWhen to Use
aws:PrincipalOrgIDThe Organization ID of the CALLERWhen the caller is an IAM principal (role, user)
aws:SourceAccountThe account ID of the SERVICE that initiated the callWhen an AWS service (EventBridge, S3) publishes on behalf of an account
aws:PrincipalOrgPathsThe OU path of the callerWhen you want to restrict to specific OUs within the Organization

→ For EventBridge service principal targets → use aws:SourceAccount (the service publishes)
→ For IAM role direct publishing → use aws:PrincipalOrgID (the role publishes)
→ For comprehensive coverage → include BOTH conditions in separate statements

  • KMS must allow the SNS service to use the key

KMS Key Policy for Cross-Account SNS/SQS Encryption:

→ Navigate to KMS in Account 555555555555 → Select shopnow-shared-services-key

Add Key Policy Statements:

StatementPrincipalActionsConditionPurpose
Allow SNS to use keysns.amazonaws.comkms:Decrypt, kms:GenerateDataKey*NoneSNS encrypts/decrypts messages
Allow SQS to use keysqs.amazonaws.comkms:Decrypt, kms:GenerateDataKey*NoneSQS encrypts/decrypts messages

Why service principals need KMS access:
→ When SNS receives a message → it encrypts with KMS before storing
→ When SQS delivers a message to a consumer → it decrypts with KMS
→ These are SERVICE-level operations → the service principal needs key access
→ This is different from user-level KMS access (Project 36)

Consumer KMS Access:
→ Lambda functions or EC2 instances CONSUMING messages from the encrypted SQS queue also need kms:Decrypt
→ These consumers are in Account 555555555555 → standard KMS key user setup (Project 36)

✅ WHAT SHOULD DO

→ Always use aws:SourceAccount condition on service principal statements — prevents unauthorized account publishing
→ Always use aws:PrincipalOrgID condition when accounts are in the same Organization — simplifies policies
→ Always enable SNS delivery status logging — critical for debugging cross-account delivery failures
→ Always configure DLQ at EVERY layer — EventBridge rule DLQ + SQS redrive DLQ
→ Always use message attributes for filtering — enables selective subscription delivery
→ Always encrypt SNS topics and SQS queues with KMS — add service principals to key policy
→ Always use raw message delivery for SQS subscriptions — avoids double JSON wrapping
→ Always use input transformation on EventBridge rules — clean payloads for downstream consumers
→ Always create dedicated IAM roles for EventBridge cross-account targets — don't reuse application roles
→ Always test cross-account messaging with a simple test message before building complex flows
→ Always monitor DLQ queues with CloudWatch alarms — messages in DLQ indicate broken pipelines
→ Always document the cross-account messaging topology — which accounts publish where and subscribe to what

❌ WHAT SHOULD NOT DO

→ Never use Principal: * without a condition — allows ANY AWS account to publish/send
→ Never skip aws:SourceAccount or aws:SourceArn conditions on service principal statements
→ Never allow sns:* or sqs:* in cross-account policies — only grant specific actions (Publish, SendMessage, Subscribe)
→ Never forget KMS key policy updates when encrypting SNS/SQS — cross-account delivery silently fails
→ Never use the same SQS queue for both monitoring and incident processing — separate by purpose
→ Never skip DLQ configuration — cross-account failures are silent without DLQ
→ Never hardcode account IDs in policies when Organization conditions are available
→ Never allow cross-account sns:DeleteTopic or sqs:DeleteQueue — only the owning account should manage lifecycle
→ Never subscribe sensitive queues to broad topics without filter policies — causes noise and potential data exposure
→ Never assume cross-account SNS → SQS delivery is instant — there can be seconds of latency, more for cross-region
→ Never forget that SNS → SQS cross-account requires BOTH topic policy (Publish) AND queue policy (SendMessage from SNS)
→ Never create circular event forwarding — Account A → Central Bus → Rule targets Account A's bus → infinite loop

MECHANIC 3 — SNS MESSAGE ATTRIBUTE FILTERING INTERNALS

How Filter Policies are Evaluated:

Message arrives at SNS with attributes:
  severity = "CRITICAL"
  sourceAccount = "111111111111"
  sourceService = "glue"
        │
For each subscription:
        │
        ├── Subscription 1 (Monitoring Queue):
        │     Filter policy: {} (empty — no filter)
        │     Empty filter = match ALL messages
        │     → DELIVER ✅
        │
        ├── Subscription 2 (Incident Queue):
        │     Filter policy: {"severity": ["CRITICAL", "HIGH"]}
        │     Check: message.severity ("CRITICAL") IN ["CRITICAL", "HIGH"]?
        │     → YES → DELIVER ✅
        │
        ├── Subscription 3 (Email):

        │     Check: message.severity ("CRITICAL") IN ["CRITICAL"]?
        │     → YES → DELIVER ✅
        │

              Filter policy: {"severity": ["INFO"]}
              Check: message.severity ("CRITICAL") IN ["INFO"]?
              → NO → DO NOT DELIVER ❌
              (Message silently dropped for this subscriber)

Filter Policy Operators:

OperatorSyntaxExampleMatches
Exact match["value"]{"severity": ["CRITICAL"]}severity = "CRITICAL"
OR match["val1", "val2"]{"severity": ["CRITICAL", "HIGH"]}severity = "CRITICAL" OR "HIGH"
Prefix match[{"prefix": "val"}]{"sourceService": [{"prefix": "glue"}]}sourceService starts with "glue"
Anything-but[{"anything-but": "val"}]{"severity": [{"anything-but": "INFO"}]}severity is NOT "INFO"
Numeric match[{"numeric": [">=", 100]}]{"errorCount": [{"numeric": [">=", 5]}]}errorCount >= 5

→ These are evaluated AT THE SNS LEVEL — before delivery
→ Filtered messages are NOT stored or retried for non-matching subscribers
→ Cost optimization: Subscribers only receive relevant messages → fewer SQS messages → lower cost

              Timestamp: ...

📈 SCALABILITY CONSIDERATIONS

→ Message Volume Scaling

  • SNS: Virtually unlimited throughput for Standard topics
  • SQS: Virtually unlimited throughput for Standard queues (nearly unlimited sends/receives per second)
  • EventBridge: 10,000+ events/second default (can increase)
  • For millions of events/second → Kinesis Data Streams (Project 52)

→ Account Count Scaling

  • 5 accounts → Individual account IDs in policies (current approach)
  • 20+ accounts → aws:PrincipalOrgID condition (Step 6)
  • 100+ accounts → Organizational conditions + SCP governance (Project 54)
  • SNS topic policy size limit: 30 KB → can hold ~100 account-specific statements
  • With aws:PrincipalOrgID → ONE statement covers unlimited accounts

→ Subscription Scaling

  • SNS subscriptions per topic: 12.5 million (soft limit)
  • SQS queues per account: Unlimited (Standard), 100,000 (FIFO)
  • EventBridge rules per bus: 300 (soft limit, can increase to 2,000+)

→ Cross-Region Scaling

  • SNS → SQS cross-region: Supported, slight latency increase
  • EventBridge bus-to-bus cross-region: Supported via rules targeting remote bus ARN
  • For low-latency requirements → Deploy regional SNS topics with regional subscribers

→ Cost Scaling

  • Costs scale linearly with message volume
  • KMS costs can become significant at high volume → Consider using SQS SSE-SQS (no KMS) for non-sensitive operational messages
  • EventBridge custom events ($1/million) are more expensive than SNS ($0.50/million) → use SNS for simple fan-out, EventBridge for complex routing
  • SNS → SQS delivery is FREE → the fan-out pattern is cost-effective
  • Archive costs in EventBridge scale with event volume × retention period → set appropriate retention

→ Reliability Scaling

  • SNS: 4 nines availability (99.99%) — messages durably stored across multiple AZs
  • SQS: 4 nines availability — messages replicated across multiple AZs
  • EventBridge: 4 nines availability — at-least-once delivery guarantee
  • Multi-layer DLQ architecture ensures NO message is silently lost
  • SNS retries SQS delivery for up to 23 days — extremely resilient
  • For mission-critical messages → Add CloudWatch alarm on DLQ depth → auto-remediation Lambda

→ Fan-Out Scaling

  • Single SNS Publish → delivered to ALL matching subscribers simultaneously
  • 1 message × 100 subscribers = 100 deliveries (parallel, not sequential)
  • No degradation with increasing subscriber count
  • Filter policies reduce unnecessary deliveries → cost savings at scale
  • Central EventBridge bus with 50 rules → all evaluated in parallel per event

🎯 KEY TAKEAWAY

→ Cross-account messaging follows the SAME dual-authorization model as cross-account S3 access (Project 39) and IAM role assumption (Project 40) — resource policy + IAM policy must BOTH allow
→ THREE patterns exist for cross-account event flow: SNS Publish, SQS SendMessage, EventBridge bus-to-bus — each has its sweet spot
→ SNS topic policy and SQS queue policy are RESOURCE-BASED policies — same concept as S3 bucket policy (Project 10) and KMS key policy (Project 36)
→ Service principals (events.amazonaws.com, sns.amazonaws.com) need SEPARATE policy statements from IAM principals — always add aws:SourceAccount condition
→ aws:PrincipalOrgID condition simplifies multi-account policies from N statements to ONE — essential for scaling beyond 5 accounts
→ KMS encryption adds a THIRD authorization layer — service principals must be added to the KMS key policy for encrypted SNS/SQS
→ Multi-layer DLQ architecture (EventBridge DLQ → SQS redrive DLQ) ensures NO message is silently lost across account boundaries
→ SNS filter policies enable SELECTIVE fan-out — different subscribers receive different message subsets based on attributes
→ EventBridge bus-to-bus pattern is the most powerful — central bus applies rich content-based routing to events from ALL accounts
→ The hub-and-spoke messaging pattern mirrors the hub-and-spoke IAM pattern from Project 40 — Shared Services is the central hub
