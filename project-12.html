STEP 1 — ENABLE S3 VERSIONING

→ S3 Console → Select bucket shopnow-productimages-prod-aps1 → "Properties" tab
→ "Bucket Versioning" → "Edit" → Enable → "Save changes"

Versioning States:

StateBehaviorCan You Revert?
Unversioned (default)Objects have no version ID — overwrites replace permanentlyN/A (initial state)
EnabledEvery PUT creates a new version — old versions preservedCan SUSPEND (not disable)
SuspendedNew objects get version ID "null" — existing versions preservedCan RE-ENABLE

→ CRITICAL: Once versioning is ENABLED → it can NEVER be fully disabled
→ You can SUSPEND versioning → new objects get null version ID
→ But existing versioned objects remain with their version IDs
→ To truly remove all versions → you must delete each version individually

What Happens the Moment Versioning is Enabled:

BEFORE enabling versioning:
  shoe.jpg → No version ID (technically version ID = "null")

AFTER enabling versioning:
  shoe.jpg → Still has version ID "null" (pre-existing objects keep null)
  
Upload shoe.jpg again:
  shoe.jpg → Version ID: "aaaa1111" (NEW version — CURRENT)
  shoe.jpg → Version ID: "null" (OLD version — NONCURRENT)

Upload shoe.jpg again:
  shoe.jpg → Version ID: "bbbb2222" (NEWEST — CURRENT)
  shoe.jpg → Version ID: "aaaa1111" (NONCURRENT)
  shoe.jpg → Version ID: "null" (NONCURRENT)

→ Each version is a COMPLETE copy of the object — not a diff/delta
→ Each version is independently stored and billed
→ 3 versions of a 10 MB file = 30 MB of storage billed

STEP 3 — UNDERSTAND DELETE MARKERS

→ This is the most CONFUSING concept in S3 versioning
→ When you "delete" an object in a versioned bucket → the object is NOT actually deleted
→ Instead, S3 places a DELETE MARKER on top of the version stack

What Happens When You Delete an Object (Simple DELETE):

State BEFORE delete:
  shoe.jpg → Version "bbbb2222" (CURRENT)
  shoe.jpg → Version "aaaa1111" (NONCURRENT)

Action: DELETE shoe.jpg (without specifying version ID)

State AFTER delete:
  shoe.jpg → DELETE MARKER (version "dmrk4444") ← NEW — this is the "CURRENT" version
  shoe.jpg → Version "bbbb2222" (NONCURRENT) ← still exists!
  shoe.jpg → Version "aaaa1111" (NONCURRENT) ← still exists!

What happens now:
  GET shoe.jpg → Returns 404 "Not Found" (delete marker makes it appear deleted)
  GET shoe.jpg?versionId=bbbb2222 → Returns the object! (specific version still exists)
  LIST objects → shoe.jpg does NOT appear (hidden by delete marker)
  LIST versions → Shows ALL versions INCLUDING the delete marker

→ A delete marker is a ZERO-SIZE "version" that hides the object
→ The actual data versions are still there → still stored → still billed
→ The object APPEARS deleted but is actually RECOVERABLE

How to Recover a "Deleted" Object:

→ Method 1: Delete the delete marker

DELETE shoe.jpg?versionId=dmrk4444 (delete the delete marker specifically)

State AFTER removing delete marker:
  shoe.jpg → Version "bbbb2222" (CURRENT again!) ← object is back!
  shoe.jpg → Version "aaaa1111" (NONCURRENT)

GET shoe.jpg → Returns version bbbb2222 ✓
Object is "undeleted" — recovered!

→ Method 2: Copy a specific version to the same key

COPY shoe.jpg (source: versionId=aaaa1111) → shoe.jpg (destination — same key)

State AFTER copy:
  shoe.jpg → Version "eeee5555" (CURRENT — copy of aaaa1111)
  shoe.jpg → DELETE MARKER (dmrk4444) (NONCURRENT)
  shoe.jpg → Version "bbbb2222" (NONCURRENT)
  shoe.jpg → Version "aaaa1111" (NONCURRENT)

GET shoe.jpg → Returns version eeee5555 (which is the content of aaaa1111)
Object is restored to a previous version ✓

PERMANENT Deletion (Cannot Be Recovered):

→ To PERMANENTLY delete a specific version:

DELETE shoe.jpg?versionId=bbbb2222

This permanently removes version bbbb2222 — NO recovery possible
The other versions (aaaa1111, dmrk4444) are unaffected

→ CRITICAL: Only specifying a version ID in the DELETE request causes permanent deletion
→ A simple DELETE (without version ID) only creates a delete marker → recoverable
→ This is the safety mechanism that solves Problem 3 (intern's accidental deletion)

Delete Marker Cost:

→ Delete markers have ZERO size → no storage cost for the marker itself
→ BUT the hidden versions underneath still consume storage → still billed
→ Delete markers accumulate over time → need lifecycle rules to clean them up

✅ WHAT SHOULD DO

→ Always enable versioning on production buckets — it's the ONLY protection against accidental overwrites and deletions
→ Always create lifecycle rules for EVERY prefix — unmanaged data grows costs indefinitely
→ Always include noncurrent version expiration rules — without them, old versions accumulate forever
→ Always set "number of newer versions to retain" as a safety net — prevents accidental loss of all history
→ Always include incomplete multipart upload cleanup — invisible storage waste
→ Always include delete marker cleanup — prevents orphaned markers cluttering the bucket
→ Always consider minimum storage duration charges before transitioning — transitioning then deleting wastes money
→ Always consider minimum object size for IA transitions — tiny files cost MORE in IA classes
→ Always analyze access patterns BEFORE creating lifecycle rules — use S3 Analytics or S3 Storage Lens
→ Always test lifecycle rules on a non-production bucket first — rules are irreversible once objects are deleted
→ Always document lifecycle rules and their rationale — future team members need to understand why
→ Always use S3 Intelligent-Tiering when access patterns are unpredictable

❌ WHAT SHOULD NOT DO

→ Never leave versioning enabled without noncurrent version expiration rules — storage costs explode
→ Never transition objects to Glacier classes if they need frequent access — retrieval fees are expensive
→ Never transition objects smaller than 128 KB to IA classes — charged as 128 KB minimum
→ Never transition objects that will be deleted within the minimum storage duration — you pay for the full minimum period
→ Never assume "delete" means permanently gone with versioning enabled — it creates a delete marker
→ Never rely on lifecycle rules as the ONLY backup — they delete data permanently when expiration triggers
→ Never set expiration rules without understanding the data's compliance requirements — legal may require 7+ years
→ Never create overlapping lifecycle rules on the same prefix with conflicting actions — unpredictable behavior
→ Never suspend versioning expecting old versions to disappear — they persist until manually deleted
→ Never forget that EVERY version is billed independently — 10 versions of 1 GB = 10 GB billed
→ Never set lifecycle expiration to 1 day for important data — allow buffer for recovery
→ Never ignore S3 Storage Lens recommendations — it identifies optimization opportunities

2 → HOW LIFECYCLE EVALUATION WORKS

LIFECYCLE EVALUATION ENGINE (runs periodically):

Step 1: S3 scans the bucket for lifecycle-eligible objects
        (This is NOT instant — runs in batches, typically once per day)
        │
        ▼
Step 2: For each object:
        │
        ├── Match lifecycle rules by prefix and tags
        │
        ├── For CURRENT versions:
        │   │
        │   ├── Check transition rules:
        │   │   Object age = current_date - creation_date
        │   │   If age >= transition_days → Transition to specified storage class
        │   │   Multiple transitions → apply in order (smallest days first)
        │   │
        │   └── Check expiration rules:
        │       If age >= expiration_days → Create delete marker (versioned) or delete (unversioned)
        │
        ├── For NONCURRENT versions:
        │   │
        │   ├── Check noncurrent transition rules:
        │   │   Noncurrent age = current_date - date_became_noncurrent
        │   │   If age >= noncurrent_transition_days → Transition
        │   │
        │   └── Check noncurrent expiration rules:
        │       If age >= noncurrent_expiration_days → Permanently delete
        │       BUT keep minimum N newer noncurrent versions (if configured)
        │
        └── For DELETE MARKERS:
            Check if expired (only version remaining) → Delete if expired
        │
        ▼
Step 3: Apply actions in batches
        Actions are applied asynchronously
        May take 24-48 hours for all objects to be processed

Lifecycle Timing — NOT Instant:

→ Lifecycle rules are evaluated periodically — NOT continuously
→ AWS documentation states: "Changes may take up to 24-48 hours to complete"
→ An object that becomes eligible at Day 30 may not transition until Day 31 or 32
→ This is NORMAL — lifecycle is designed for cost optimization, not real-time processing
→ Do NOT rely on lifecycle for time-sensitive operations

3 → HOW RETRIEVAL FROM GLACIER WORKS

Glacier Instant Retrieval:

→ Retrieval: Same as Standard (milliseconds)
→ No restore process needed — direct GET request works
→ Higher per-GB retrieval fee than Standard
→ Best for: Data accessed once per quarter

Glacier Flexible Retrieval:

→ Data is NOT directly accessible — must be "restored" first
→ Restore process:

Step 1: Initiate restore request
        Specify: Number of days to keep restored copy, retrieval tier

Step 2: S3 retrieves data from Glacier storage (background process)
        
        Retrieval Tiers:
        ┌──────────────┬─────────────────┬───────────────┐
        │ Tier         │ Retrieval Time  │ Cost          │

Step 3: Restored copy placed in S3 Standard (temporarily)
        Original remains in Glacier
        Restored copy available for specified number of days

🚨 COMMON PITFALLS AND FAILURE MODES

#PitfallWhat HappensHow to Avoid
1Enabling versioning without noncurrent expiration rulesStorage grows indefinitely — 10 versions of every fileAlways add noncurrent version lifecycle rules
2Assuming "delete" removes data with versioningDelete only creates a delete marker — data still exists and is billedUnderstand delete markers — permanently delete specific versions if needed
3Transitioning tiny files to IA classesFiles < 128 KB charged as 128 KB — costs MORE than StandardOnly transition files > 128 KB to IA classes
4Transitioning then deleting within minimum durationCharged for full minimum period (30/90/180 days)Don't transition objects that will be deleted soon
5Expecting lifecycle rules to act instantlyRules may take 24-48 hours to processDon't rely on lifecycle for time-sensitive operations
6Overlapping rules on same prefix with different actionsUnpredictable behavior — S3 applies the most restrictiveUse non-overlapping prefixes or consolidate rules
7Forgetting incomplete multipart cleanupInvisible storage waste accumulatesAlways add abort incomplete multipart rule
8Not calculating retrieval costs for Glacier dataRestoring 1 TB from Glacier Flexible = significant costFactor retrieval fees into total cost analysis
9Suspending versioning thinking versions disappearExisting versions remain — only new uploads get null version IDMust explicitly delete old versions to reclaim storage
10Lifecycle expiration on compliance dataLegal requires 7-year retention — lifecycle deletes after 1 yearReview compliance requirements before setting expiration
11No version retention minimum setAll noncurrent versions deleted — no recovery possibleSet "newer versions to retain" as safety net
12Forgetting delete marker cleanupThousands of orphaned delete markers clutter listingsAdd "delete expired object delete markers" rule

📈 SCALABILITY CONSIDERATIONS

→ Version Scalability

  • No limit on versions per object key
  • No limit on total objects/versions per bucket
  • Each version is independently stored and accessed
  • Lifecycle rules handle cleanup at any scale — millions of objects

→ Lifecycle Rule Scalability

  • Up to 1,000 rules per bucket
  • Rules can use prefix and tag filters for granularity
  • S3 processes lifecycle rules in batches — large buckets may take longer
  • For buckets with billions of objects → lifecycle still works but may take multiple days for a full pass

→ Cost Scalability

  • Without lifecycle → costs grow linearly with data volume
  • With lifecycle → costs grow sub-linearly (data auto-moves to cheaper tiers)
  • For very large datasets (PBs) → lifecycle savings become millions of dollars annually
  • Use S3 Storage Lens (Project 56) for bucket-level cost analysis across accounts

→ Operational Scalability

  • Lifecycle rules require ZERO manual intervention after creation
  • No cron jobs, no scripts, no monitoring needed
  • Changes to rules take effect on the next evaluation cycle
  • New objects automatically matched by existing rules

→ Multi-Bucket Scalability

  • Each bucket has its own lifecycle rules
  • For organizational standards → use AWS Config rules to enforce lifecycle policy existence
  • S3 Storage Lens provides cross-bucket, cross-account analytics

→ How to debug "lifecycle rule not working":

STEP 1: Is the rule enabled?
        S3 Console → Management → Lifecycle rules → Status: Enabled?

STEP 2: Does the prefix match?
        Rule prefix: "products/"
        Object key: "Products/image.jpg" (capital P)
        → MISMATCH (S3 keys are case-sensitive) → rule doesn't apply

STEP 3: Are tag filters correct?
        Rule requires tag: Environment=Production
        Object doesn't have this tag → rule doesn't apply

STEP 4: Has enough time passed?
        Rule says: transition after 30 days
        Object was uploaded 25 days ago → not yet eligible
        Lifecycle evaluation delay → add 1-2 more days

STEP 5: Is the transition valid?
        Standard → Standard-IA requires minimum 30-day age
        Standard → Glacier requires minimum 90-day age (if transitioning via IA)
        Invalid transitions are silently skipped

STEP 6: Is the object large enough?
        Objects < 128 KB are transitioned but charged minimum 128 KB
        Very small objects may not show expected savings

STEP 7: Check S3 Server Access Logs
        Look for lifecycle-related entries → confirms S3 attempted the action
