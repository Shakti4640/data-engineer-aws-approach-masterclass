PROJECT 44: THE MICROSERVICES ACROSS ACCOUNTS THAT NEEDED TO COMMUNICATE — Multi-Account VPC Peering

🔴 HYPOTHETICAL PROBLEM

→ A fintech company "PayScale" operates a microservices architecture
→ They have grown to 3 AWS accounts for isolation and billing separation

AccountPurposeVPC CIDRRegion
Account A — Platform Team (111111111111)Core Payment Processing Service10.0.0.0/16ap-south-1
Account B — Risk Team (222222222222)Fraud Detection Service10.1.0.0/16ap-south-1
Account C — Analytics Team (333333333333)Reporting Dashboard + MariaDB (from Project 9)10.2.0.0/16ap-south-1

→ The Fraud Detection Service in Account B needs to call the Payment Processing API in Account A over private IPs
→ The Reporting Dashboard in Account C needs to read from the MariaDB in Account A's private subnet
→ Currently — all inter-service communication goes over the PUBLIC INTERNET
→ This means:

  • Traffic leaves AWS → traverses the internet → re-enters AWS
  • Adds 15–30ms latency per hop
  • Exposes traffic to potential interception even with TLS
  • Compliance team flagged it — PCI-DSS requires private network communication for payment data
  • Data transfer costs are higher (internet egress charges)
    → The CTO says: "All microservice communication must happen over private AWS networking — no internet traversal"
    → They need Account A ↔ Account B and Account A ↔ Account C private connectivity
    → They do NOT need Account B ↔ Account C direct communication

🟡 HOW TO APPROACH IT

→ First — understand what private inter-VPC communication options exist

OptionWhat It DoesWhen to Use
VPC Peering1-to-1 private connection between two VPCsSmall number of VPC pairs (< 20 connections)
Transit GatewayHub-and-spoke model — central router for many VPCsMany VPCs (10+), mesh connectivity needed (Project 50)
PrivateLinkExpose a specific service endpoint across VPCs/accountsService-to-service, not full network access (covered in Project 27)
VPN over InternetEncrypted tunnel over public internetHybrid cloud, on-premises connectivity

→ PayScale needs only 2 connections (A↔B and A↔C)
→ VPC Peering is the right choice — simple, no extra infrastructure cost, no hourly charge
→ Transit Gateway would be overkill for 2 connections (explored in Project 50 when they grow to 20 VPCs)

→ The approach should be:

  • Step 1 — Validate CIDR blocks do NOT overlap across all three VPCs
  • Step 2 — Create VPC Peering Connection (Requester side)
  • Step 3 — Accept VPC Peering Connection (Accepter side — different account)
  • Step 4 — Update Route Tables on BOTH sides
  • Step 5 — Update Security Groups on BOTH sides to allow peered traffic
  • Step 6 — Enable DNS Resolution across the peering connection
  • Step 7 — Test private connectivity
  • Step 8 — Repeat for second peering connection (A↔C)

🟢 HOW CAN WE SOLVE IT — Step-by-Step Configuration

PRE-REQUISITE VALIDATION — CIDR OVERLAP CHECK

→ This is the MOST CRITICAL step — if CIDRs overlap, peering is IMPOSSIBLE

AccountVPC CIDRIP Range
A10.0.0.0/1610.0.0.0 → 10.0.255.255 (65,536 IPs)
B10.1.0.0/1610.1.0.0 → 10.1.255.255 (65,536 IPs)
C10.2.0.0/1610.2.0.0 → 10.2.255.255 (65,536 IPs)

→ No overlap ✅ — Peering is possible

→ AWS checks this during peering creation and REJECTS overlapping CIDRs
→ This is why VPC CIDR planning (from Project 5) is foundational — mistakes here are unfixable without VPC recreation

IMPORTANT — Cross-Account Security Group Referencing:

→ In SAME-ACCOUNT peering → You can reference a security group ID directly (e.g., sg-0abc123)
→ In CROSS-ACCOUNT peering → You CANNOT reference the other account's security group ID
→ You MUST use CIDR blocks as the source/destination in security group rules
→ This is a hard AWS limitation for cross-account peering

STEP 5 — ENABLE DNS RESOLUTION ACROSS PEERING

Why this matters:

→ Account B's Fraud Detection service wants to call Account A's Payment API
→ The API is behind an internal DNS name: payment-api.internal.payscale.com
→ This DNS name resolves to a private IP like 10.0.1.45
→ Without DNS resolution enabled on the peering connection → Account B cannot resolve Account A's private DNS names
→ The DNS query returns nothing or a public IP

Configuration:

→ Navigate to VPC Console → Peering Connections → Select the peering connection
→ Actions → Edit DNS Settings

SettingAccount A SideAccount B Side
Allow VPC-A to resolve VPC-B private DNS✅ Enable—
Allow VPC-B to resolve VPC-A private DNS—✅ Enable

→ BOTH sides must enable DNS resolution independently
→ Each account owner enables their own side

Pre-requisite for DNS resolution:
→ Both VPCs must have these VPC settings enabled (from Project 5):

  • enableDnsSupport = true
  • enableDnsHostnames = true
    → If either is false → DNS resolution across peering will not work

STEP 7 — UNDERSTANDING THE TRANSITIVE LIMITATION

→ Account B can reach Account A ✅
→ Account C can reach Account A ✅
→ Can Account B reach Account C through Account A? ❌ NO

This is the NON-TRANSITIVE nature of VPC Peering

Account B → Account A → Account C
              ↑
         TRAFFIC STOPS HERE
         A does NOT forward B's traffic to C

→ VPC Peering is a DIRECT 1-to-1 connection
→ Traffic entering through one peering connection CANNOT exit through another peering connection
→ AWS enforces this at the network fabric level — not configurable

If Account B needs to reach Account C:
→ Option 1: Create a THIRD peering connection (B ↔ C directly)
→ Option 2: Use Transit Gateway (Project 50) — which DOES support transitive routing

The Scaling Problem:
→ 3 VPCs needing full mesh = 3 peering connections
→ 5 VPCs needing full mesh = 10 peering connections
→ 10 VPCs needing full mesh = 45 peering connections
→ 20 VPCs needing full mesh = 190 peering connections
→ Formula: n × (n-1) / 2
→ This is exactly why Transit Gateway exists (Project 50)

✅ WHAT SHOULD DO

→ Plan VPC CIDR blocks across ALL accounts BEFORE creating any VPCs — overlapping CIDRs are unfixable
→ Use a centralized IP Address Management (IPAM) spreadsheet or AWS VPC IPAM service
→ Document every peering connection with both account IDs, VPC IDs, and CIDRs
→ Update route tables on BOTH sides — forgetting one side is the #1 debugging nightmare
→ Use the most specific CIDR possible in security group rules (subnet CIDR, not entire VPC CIDR)
→ Enable DNS resolution on both sides if services use private DNS names
→ Tag peering connections with source/destination account names for visibility
→ Use NACLs (from Project 4) as an additional layer — NACLs are stateless so add BOTH inbound AND outbound rules
→ Monitor peering connection metrics in CloudWatch:

  • BytesIn / BytesOut — traffic volume
  • PacketsIn / PacketsOut — packet counts
    → Set up CloudWatch Alarms (from Project 8) for unexpected traffic spikes across peering
    → Use VPC Flow Logs (enable on both VPCs) to audit cross-account traffic

❌ WHAT SHOULD NOT DO

→ Never assume peering is "active = working" — routes and security groups are mandatory
→ Never peer VPCs with overlapping CIDRs — AWS rejects it but some secondary CIDRs can cause partial overlap confusion
→ Never use 0.0.0.0/0 as destination pointing to a peering connection — AWS rejects this (peering cannot be a default route)
→ Never expect transitive routing — VPC A peered with B and C does NOT make B↔C connected
→ Never reference cross-account security group IDs in rules — use CIDR blocks only
→ Never forget to update NACL rules if NACLs are customized — NACLs are stateless (from Project 4), so both inbound AND outbound rules must allow the traffic AND the return traffic
→ Never create peering connections without the accepter account's knowledge — this is a security anti-pattern even within the same organization
→ Never rely on peering for more than ~20-30 VPC pairs — use Transit Gateway (Project 50) instead
→ Never peer just to access a single service — use PrivateLink (from Project 27) for service-specific access
→ Never leave unused peering connections active — they don't cost money but create security surface area

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

1 → THE PEERING CONNECTION IS NOT A VPN OR PHYSICAL LINK

→ VPC Peering is a LOGICAL networking construct within the AWS backbone network
→ There is no physical cable or tunnel created
→ AWS's internal network fabric (which interconnects all VPCs in a region) simply allows traffic between the two VPCs based on the peering configuration
→ For cross-region peering → AWS uses its private inter-region backbone (NOT the public internet)
→ This backbone is the same one used for S3 cross-region replication (from Project 30) and other AWS inter-region services

2 → THE REQUESTER-ACCEPTER HANDSHAKE

REQUESTER (Account A)                          ACCEPTER (Account B)
        |                                               |
        |  1. CreateVpcPeeringConnection API call        |
        | -------------------------------------------→  |
        |     (specifies accepter account + VPC ID)     |
        |                                               |
        |  2. Peering object created in BOTH accounts   |
        |     Status: "pending-acceptance"               |
        |                                               |
        |  3. AcceptVpcPeeringConnection API call       |
        |  ←------------------------------------------- |
        |                                               |
        |  4. Status changes to "active" in BOTH        |
        |     AWS updates internal routing fabric        |
        |                                               |

→ The peering object is visible in BOTH accounts simultaneously
→ Either side can DELETE the peering connection — it's immediately torn down on both sides
→ No approval needed from the other side to delete

3 → HOW ROUTING WORKS INTERNALLY

→ When Account B sends a packet to 10.0.1.45 (Account A):

Step 1: Application in Account B generates packet
        Source: 10.1.2.100 → Destination: 10.0.1.45

Step 2: OS routing table sends packet to the ENI (Elastic Network Interface)

Step 3: VPC route table evaluation in Account B:
        - 10.1.0.0/16 → local (no match — destination is 10.0.x.x)
        - 10.0.0.0/16 → pcx-0aaabbb1234567890 ✅ MATCH
        
Step 4: Packet enters the peering connection

Step 5: AWS network fabric validates:
        - Is peering connection active? ✅
        - Does source IP belong to requester/accepter VPC? ✅
        - Does destination IP fall within the peer VPC's CIDR? ✅
        
Step 6: Packet arrives at Account A's VPC

Step 7: Account A's route table evaluation:
        - 10.0.0.0/16 → local ✅ MATCH (destination 10.0.1.45 is local)
        
Step 8: Packet delivered to the target ENI

Step 9: Security Group on target ENI evaluated:
        - Inbound rule allows TCP 443 from 10.1.0.0/16? ✅
        
Step 10: Packet reaches the application

→ Return traffic follows the REVERSE path
→ Security Groups are STATEFUL (from Project 4) — so return traffic is automatically allowed if the initial request was allowed
→ NACLs are STATELESS — so return traffic MUST have explicit outbound rules on the destination side AND explicit inbound rules on the source side for the ephemeral port range

4 → WHY TRANSITIVE ROUTING IS IMPOSSIBLE

→ When AWS processes a packet entering through a peering connection:

  • It checks: "Is the destination IP within the LOCAL VPC CIDR?"
  • If YES → deliver locally
  • If NO → DROP the packet
  • It does NOT check other route table entries pointing to other peering connections
    → This is a deliberate security design — prevents one peered VPC from becoming an unauthorized router
    → AWS enforces this at the hypervisor/network fabric level — no configuration can override it

6 → DNS RESOLUTION MECHANICS

→ When DNS resolution is enabled on the peering connection:

  • Account B makes a DNS query for ip-10-0-1-45.ap-south-1.compute.internal
  • Normally → This resolves ONLY within Account A's VPC
  • With DNS resolution enabled → AWS Route 53 Resolver forwards the query through the peering connection
  • Account A's VPC DNS resolver responds with the private IP
  • Account B receives the private IP and routes traffic through the peering connection

→ For custom private hosted zones (Route 53):

  • If Account A has a private hosted zone internal.payscale.com associated with its VPC
  • Account B CANNOT automatically resolve these names even with peering DNS enabled
  • Account A must ALSO associate the private hosted zone with Account B's VPC (cross-account hosted zone association)
  • This requires a separate authorization flow in Route 53

📈 SCALABILITY CONSIDERATIONS

→ Connection Scalability

  • 50 peering connections per VPC (default), extendable to 125
  • Full mesh formula: n × (n-1) / 2 connections
  • 10 VPCs = 45 connections → Manageable
  • 20 VPCs = 190 connections → Exceeds per-VPC limit → Transit Gateway required
  • 50 VPCs = 1,225 connections → Completely unmanageable with peering

→ Bandwidth Scalability

  • No aggregate bandwidth limit on a peering connection
  • Individual instance bandwidth is the bottleneck (e.g., m5.xlarge = ~10 Gbps)
  • Cross-region peering bandwidth is limited by inter-region backbone capacity
  • No throttling or QoS configuration available

→ Latency Scalability

  • Same-region peering: Sub-millisecond to low milliseconds (same as intra-VPC)
  • Cross-region peering: Regional latency (e.g., Mumbai ↔ Ireland ≈ 120ms)
  • Peering adds negligible overhead — it's the DISTANCE that adds latency

→ Route Table Scalability

  • Each route table supports 50 routes (default), extendable to 1,000
  • Each peering connection needs at least one route per route table
  • With 50 peering connections → 50 routes consumed just for peering
  • Plan route table capacity early

→ Operational Scalability

  • Each peering connection requires manual route + SG updates on both sides
  • No "auto-propagation" of routes (unlike Transit Gateway)
  • For 45 peering connections → 90 route table updates + 90 security group updates
  • Infrastructure as Code (CloudFormation/Terraform) becomes mandatory at scale

🎯 KEY TAKEAWAY

→ VPC Peering is a 1-to-1 private networking construct between exactly two VPCs
→ It works ACROSS accounts (using the requester-accepter handshake from this project) and ACROSS regions
→ Three things must ALL be configured for traffic to flow: Peering Active + Routes Added + Security Groups Allow
→ Peering is NON-TRANSITIVE — A↔B and A↔C does NOT give B↔C
→ CIDR overlap = peering impossible — plan CIDRs before creating VPCs (from Project 5)
→ Cross-account security group rules MUST use CIDR blocks — no SG ID referencing
→ VPC Peering has zero hourly cost — you pay only for data transfer
→ This approach works perfectly for small-scale (2-10 VPCs) cross-account connectivity
