PROJECT 49: THE COMPLEX PIPELINE — S3 Upload → EventBridge → Step Functions → Glue ETL → Redshift → SNS Alert

🔴 HYPOTHETICAL PROBLEM

→ PayScale's data platform has matured — individual services are configured and working
→ But the platform operates as DISCONNECTED ISLANDS:

Current State — Manual & Fragmented:

Island 1: Partner uploads CSV to S3 landing zone
    → Nobody knows it arrived until someone checks manually

Island 2: Data engineer manually triggers Glue ETL job
    → Sometimes forgets → Data is 2 days stale
    → Sometimes triggers twice → Duplicate processing

Island 3: After Glue completes, engineer manually runs Redshift COPY
    → Sometimes Glue failed but engineer doesn't know → COPY loads corrupt data
    → Sometimes COPY fails → Nobody is notified for hours

Island 4: After Redshift load, engineer manually sends Slack message
    → "Data is ready" — but sometimes it isn't
    → No audit trail of what happened, when, or why

Island 5: When anything fails, the failure is discovered HOURS later
    → By then, the fraud team has been making decisions on stale data
    → No automatic retry — engineer must manually re-run failed steps

The Business Impact:

ProblemImpactFrequency
File arrives but nobody processes itFraud detection delayed by 4-8 hours3x per week
Glue job fails silentlyData lake has stale data2x per week
Redshift COPY runs on incomplete dataIncorrect fraud scores in dashboard1x per week
No failure notificationTeam discovers issues hours laterDaily
Manual re-runs create duplicatesDuplicate transactions in analytics2x per month
No audit trailCompliance team cannot verify data lineageEvery audit

The CTO's Requirement:

→ "Build ONE automated pipeline that does everything end-to-end"
→ "When a file lands in S3 → everything happens automatically"
→ "If anything fails → retry automatically → if retry fails → notify immediately"
→ "I want to see the entire pipeline status in one place"
→ "Zero human intervention for the happy path"

STEP 5 — CONFIGURE ERROR HANDLING AND RETRY LOGIC

This is where Step Functions truly differentiates from manual orchestration

RETRY CONFIGURATION ON GLUE ETL STATE

Retry Policy:

SettingValueWhy
ErrorEqualsStates.ALLCatch any error type
IntervalSeconds60Wait 1 minute before first retry
MaxAttempts2Try up to 2 additional times (3 total)
BackoffRate2.0Double the wait between retries (60s → 120s → 240s)

How Retry Works:

Attempt 1: Glue ETL starts → FAILS (e.g., transient S3 error)
    → Wait 60 seconds
Attempt 2: Glue ETL starts → FAILS (e.g., still transient)
    → Wait 120 seconds (60 × 2.0 backoff)
Attempt 3: Glue ETL starts → FAILS (persistent issue)
    → MaxAttempts exhausted → Fall through to CATCH

Catch Configuration (After Retries Exhausted):

SettingValueWhy
ErrorEqualsStates.ALLCatch any unhandled error
NextNotifyGlueFailureRoute to SNS notification state
ResultPath$.errorInfoPreserve error details in state data

Error Information Available in $.errorInfo:

{
    "Error": "Glue.JobRunException",
    "Cause": "Job run failed: OutOfMemoryError in Spark executor"
}

→ This error detail is included in the SNS notification
→ The team knows exactly WHY the pipeline failed — not just THAT it failed

RETRY CONFIGURATION ON REDSHIFT COPY STATE

SettingValueWhy
ErrorEqualsStates.ALLCatch any error
IntervalSeconds30Redshift errors often resolve faster
MaxAttempts1Only 1 retry — Redshift failures are usually data-related, not transient
BackoffRate1.0No backoff — fixed 30-second wait

Why fewer retries for Redshift?
→ Glue failures are often transient (S3 throttling, temporary resource constraints)
→ Redshift COPY failures are usually persistent (wrong schema, missing files, permission errors)
→ Retrying a persistent error 3 times wastes time — better to notify immediately

GLOBAL TIMEOUT ON STATE MACHINE

SettingValueWhy
TimeoutSeconds7200 (2 hours)Entire pipeline should complete within 2 hours

→ If the pipeline runs for > 2 hours → Something is fundamentally wrong
→ State machine is automatically aborted → Error notification sent

STEP 8 — PREVENT DUPLICATE PIPELINE EXECUTIONS

Problem: What if the same file triggers the pipeline twice?
→ Partner re-uploads the same file (overwrite)
→ EventBridge fires again → Second execution starts
→ Both executions process the same file → Duplicate data in Redshift

Solution 1 — EventBridge Deduplication (Limited):
→ EventBridge does NOT have built-in deduplication
→ Each S3 event triggers a new execution

Solution 2 — Step Functions Execution Name Deduplication:

→ Use the S3 object key as the execution name
→ Step Functions rejects duplicate execution names within 90 days

SettingConfiguration
Execution nameDerived from S3 key: txn-2025-01-15-partner-feed

→ If the same file triggers again → Step Functions returns: "Execution already exists"
→ EventBridge receives the error but the duplicate execution is prevented

Solution 3 — DynamoDB Tracking Table (Most Robust):

→ The ValidateFile Lambda checks a DynamoDB table before proceeding

DynamoDB TableSchema
Table namepipeline-executions
Partition keyfile_key (String) — e.g., transactions/2025-01-15/partner-feed.csv
Attributesstatus, execution_id, started_at, completed_at

Lambda Logic:

Check DynamoDB: Does file_key exist with status = "COMPLETED"?
    → YES → Return validation: "DUPLICATE" → Pipeline ends gracefully
    → NO → Insert record with status = "PROCESSING" → Continue pipeline
    
At pipeline end (success): Update status to "COMPLETED"
At pipeline end (failure): Update status to "FAILED" (allows re-processing)

✅ WHAT SHOULD DO

→ Always use the .sync integration pattern for Glue Jobs — Step Functions handles polling automatically
→ Always build manual polling loops for services without .sync support (Crawlers, Redshift Data API)
→ Always configure Retry with exponential backoff — transient errors often resolve themselves
→ Always configure Catch as the final safety net — every error should produce a notification
→ Always set a global timeout on the state machine — prevents infinite runs
→ Always pass minimal, structured input between states — avoid passing entire event payloads
→ Always use Input Transformers in EventBridge — clean data before it reaches Step Functions
→ Always implement deduplication — S3 events can fire multiple times for the same file
→ Always include the execution ID and file name in every SNS notification — enables quick debugging
→ Always enable CloudWatch logging at ALL level during development — reduce to ERROR in production
→ Always enable X-Ray tracing — provides end-to-end visibility across all integrated services
→ Always scope IAM permissions to specific resources — never use * for actions
→ Always test each state independently before testing the full pipeline
→ Always use the Step Functions visual editor for initial design — switch to ASL JSON for version control

❌ WHAT SHOULD NOT DO

→ Never use Express workflows for pipelines involving Glue or Redshift — 5-minute limit is too short
→ Never skip error handling — a state without Retry/Catch fails the ENTIRE execution silently
→ Never use fire-and-forget (.waitForTaskToken) for Glue jobs when .sync is available — unnecessary complexity
→ Never hardcode S3 paths, job names, or ARNs in the state machine — use parameters and input mapping
→ Never route ALL EventBridge events to Step Functions — use event patterns to filter precisely
→ Never forget the EventBridge IAM role — without it, EventBridge cannot start Step Functions
→ Never put sensitive data (credentials, PII) in state machine input/output — it's logged in CloudWatch
→ Never set retry MaxAttempts too high for persistent errors — wastes time and money
→ Never skip the data quality check — loading corrupt data is worse than loading no data
→ Never ignore the cost of Step Functions state transitions — optimize by combining states where possible
→ Never build complex logic inside Step Functions states — use Lambda for complex decisions
→ Never assume S3 events fire exactly once — design for at-least-once delivery

2 → HOW .SYNC INTEGRATION WORKS INTERNALLY

Step Functions calls glue:StartJobRun
    |
    ↓
Glue returns: {JobRunId: "jr_xxx"}
    |
    ↓
Step Functions stores JobRunId internally
    |
    ↓
Step Functions creates an INTERNAL polling schedule:
    Every 10-30 seconds (adaptive):
        Call glue:GetJobRun with JobRunId
        Check JobRunState:
            STARTING → Continue polling
            RUNNING → Continue polling
            SUCCEEDED → Transition to next state with job output
            FAILED → Throw error (triggers Retry/Catch)
            TIMEOUT → Throw error
            STOPPED → Throw error
    |
    ↓
Polling continues until terminal state reached
    No Step Functions state transitions consumed during polling
    (Polling is FREE — not charged as state transitions)

→ Key insight: The polling during .sync is NOT charged as state transitions
→ Only the initial StartJobRun and the final result processing count as transitions
→ This makes .sync cost-effective for long-running tasks

→ Scaling Patterns:

Pattern 1: Sequential (Current — One File at a Time)

File 1 arrives → Pipeline runs → Completes
File 2 arrives → Pipeline runs → Completes

→ Simple but slow if many files arrive simultaneously

Pattern 2: Parallel (Multiple Files Simultaneously)

File 1 arrives → Pipeline 1 runs ──────→ Completes
File 2 arrives → Pipeline 2 runs ──────→ Completes
File 3 arrives → Pipeline 3 runs ──────→ Completes

→ Each file triggers its OWN state machine execution
→ All run in parallel — limited only by Glue concurrent job limit
→ This is the DEFAULT behavior — EventBridge triggers a new execution per event

Pattern 3: Batched (Collect Files, Process Together)

File 1 arrives → SQS queue
File 2 arrives → SQS queue    → Lambda polls SQS every 5 min
File 3 arrives → SQS queue       → Starts ONE pipeline with list of files
                                   → Glue processes all files in one job

→ More efficient for high-volume scenarios (hundreds of small files)
→ Reduces Glue job overhead (fewer job starts)
→ Adds complexity (SQS + Lambda + batch logic)
