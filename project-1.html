PROJECT 1: THE STARTUP THAT LOST FILES — Understanding S3 Bucket Creation & Object Storage

🔴 HYPOTHETICAL PROBLEM

→ A startup called "ShopNow" has 15 employees
→ They store product images, invoices, and customer reports on local laptops
→ One developer's hard drive crashed — 6 months of product images gone permanently
→ Another employee accidentally deleted an invoices folder — no backup existed
→ The CEO asks: "Where should we store all our files so this never happens again?"
→ They need a central, durable, accessible-from-anywhere storage solution
→ Budget is tight — they cannot afford expensive SAN/NAS hardware
→ Files range from 10 KB images to 500 MB video demos
→ Multiple team members need access simultaneously
→ Some files are accessed daily, some once a year

🟡 HOW TO APPROACH IT

→ First — understand what kind of storage problem this is
→ This is NOT a database problem (they are not querying structured rows)
→ This is NOT a block storage problem (they don't need a virtual hard disk attached to a server)
→ This IS an object storage problem — files (objects) that need to be stored, retrieved, and organized

→ Object Storage means

  • Every file is an "object"
  • Every object has a unique key (like a file path)
  • Every object sits inside a container (called a "bucket")
  • No folder hierarchy actually exists — it's a flat namespace with prefix illusion

→ AWS S3 (Simple Storage Service) is purpose-built for this exact scenario

→ The approach should be

  • Step 1 — Create a bucket (the container)
  • Step 2 — Define the region (where physically the data lives)
  • Step 3 — Configure basic settings (versioning, tags, encryption)
  • Step 4 — Upload objects (files)
  • Step 5 — Understand how retrieval works
  • Step 6 — Understand storage classes for cost optimization

🟢 HOW CAN WE SOLVE IT — Step-by-Step Configuration

STEP 1 — CREATE AN S3 BUCKET

→ Navigate to AWS Management Console → Search "S3" → Click "Create bucket"

Setting 1: Bucket Name
→ Must be globally unique across ALL AWS accounts in the entire world
→ Must be 3–63 characters
→ Only lowercase letters, numbers, hyphens, dots
→ Cannot look like an IP address (e.g., 192.168.1.1)
→ Example: shopnow-product-images-2025

Why globally unique?
→ Because S3 bucket names map to DNS entries
→ shopnow-product-images-2025.s3.amazonaws.com
→ Two DNS entries cannot have the same name on the internet
→ If someone else already took the name — you cannot use it

Setting 2: AWS Region
→ This decides WHERE your data physically resides
→ Dropdown shows regions like us-east-1 (N. Virginia), ap-south-1 (Mumbai), eu-west-1 (Ireland)

How to choose a region:

FactorDecision Logic
User LocationChoose region closest to majority of users for lowest latency
ComplianceSome regulations require data to stay in specific countries
CostPricing varies by region — us-east-1 is generally cheapest
Service AvailabilitySome AWS features launch in us-east-1 first

→ ShopNow is based in India → Choose ap-south-1 (Mumbai)
→ Data stays in Mumbai data centers → Satisfies Indian data residency preferences

Setting 3: Object Ownership
→ Option: "ACLs disabled (recommended)"
→ This means the bucket owner owns ALL objects regardless of who uploaded
→ Keep this as recommended — ACLs are legacy

Setting 4: Block Public Access Settings
→ This is a SAFETY NET — 4 checkboxes appear

CheckboxWhat It Does
Block all public accessMaster switch — overrides everything below
Block public access granted through NEW ACLsPrevents any new ACL from making objects public
Block public access granted through ANY ACLsIgnores all existing and future ACLs that grant public access
Block public access granted through NEW public bucket policiesPrevents new bucket policies from granting public access
Block public access granted through ANY public bucket or access point policiesIgnores all policies that grant public access

→ For ShopNow → Keep ALL CHECKED (Block all public access = ON)
→ These are internal company files — nobody outside should access them
→ This single setting has prevented thousands of data breaches globally

Setting 5: Bucket Versioning
→ Option: Enable or Disable
→ For now → Keep DISABLED (we will enable it in Project 12)
→ Quick understanding: When enabled, S3 keeps every version of every object
→ Delete a file? The old version still exists
→ Overwrite a file? The previous version is preserved

Setting 6: Tags
→ Key-value pairs for organization and billing tracking

KeyValue
DepartmentMarketing
ProjectProductCatalog
EnvironmentProduction

→ Tags help when you have 200 buckets and need to track costs per department

Setting 7: Default Encryption
→ Option 1: SSE-S3 (Amazon S3 managed keys) — AWS handles everything
→ Option 2: SSE-KMS (AWS KMS managed keys) — More control, audit trail
→ Option 3: DSSE-KMS (Dual-layer encryption) — Ultra-sensitive data

→ For ShopNow → Choose SSE-S3
→ Simplest option — every object encrypted at rest automatically
→ No extra cost — no extra configuration
→ KMS will be explored in Project 36

Setting 8: Bucket Key
→ Enable → Reduces KMS API call costs when using SSE-KMS
→ For SSE-S3 → Not applicable but keep enabled (no harm)

→ Click "Create bucket" → Bucket is created

STEP 5 — UNDERSTAND "FOLDERS" IN S3

→ There are NO real folders in S3
→ S3 is a flat key-value store

→ When you create a "folder" called invoices/ in the console
→ AWS actually creates a zero-byte object with key invoices/
→ When you upload invoices/jan-2025.pdf
→ The object key is literally invoices/jan-2025.pdf
→ The console DISPLAYS it as a folder structure for human convenience
→ Internally — it's just a string key with a / delimiter

→ Why this matters:

  • You cannot "rename" a folder — you'd have to copy every object with new prefix and delete old ones
  • Listing objects with prefix invoices/ returns all objects "inside" that "folder"
  • Performance is not affected by "folder depth" — S3 doesn't traverse directories

✅ WHAT SHOULD DO

→ Always enable "Block All Public Access" unless you explicitly need public hosting
→ Always choose the region closest to your primary user base
→ Always add tags from Day 1 — retrofitting tags on 500 buckets is painful
→ Always enable default encryption (SSE-S3 minimum)
→ Use meaningful, consistent bucket naming conventions

  • Pattern: {company}-{project}-{environment}-{region}
  • Example: shopnow-productimages-prod-aps1
    → Use prefixes (pseudo-folders) to organize objects logically
  • raw/, processed/, archive/
  • year=2025/month=01/day=15/ (for partitioned data — critical in Project 17, 18)
    → Always check the region selector in the console before creating — wrong region = data in wrong country

❌ WHAT SHOULD NOT DO

→ Never use personal identifiers in bucket names (john-test-bucket) — it's globally visible in ARNs
→ Never leave Block Public Access OFF unless you have a documented reason
→ Never store credentials, passwords, or API keys as plain text files in S3
→ Never use periods (.) in bucket names if you plan to use HTTPS — SSL wildcard certificates don't support dots in subdomains
→ Never assume S3 has folder rename capability — it doesn't
→ Never ignore the region — creating a bucket in us-east-1 when your users are in Mumbai adds 200ms+ latency
→ Never create one giant bucket for everything — separate by concern (logs bucket, data bucket, backup bucket)
→ Never skip encryption — even if data seems "non-sensitive" today, compliance requirements change

🔵 WHEN TO USE S3

ScenarioUse S3?
Store files/images/videos/documents✅ Yes — primary use case
Data lake storage (CSV, Parquet, JSON, ORC)✅ Yes — foundation of AWS data lakes
Static website hosting✅ Yes — with public access enabled
Backup & archive✅ Yes — Glacier tiers for cost savings
Log storage (CloudWatch, ALB, CloudTrail)✅ Yes — most AWS services can write logs to S3
Intermediate storage between ETL steps✅ Yes — Glue, EMR, Redshift all use S3 as staging
Need POSIX file system (mount as drive)❌ No — Use EFS or FSx
Need block storage for database files❌ No — Use EBS
Need sub-millisecond key-value lookups❌ No — Use DynamoDB or ElastiCache
Need relational queries on structured data❌ No — Use RDS (Project 9) or Redshift (Project 16)
Need real-time streaming data ingestion❌ No — Use Kinesis (Project 52)

🔴 WHEN NOT TO USE S3

→ When you need a mounted file system that multiple EC2 instances share simultaneously → Use EFS
→ When you need transactional database operations (INSERT/UPDATE/DELETE with ACID) → Use RDS
→ When you need to process streaming data in real-time → Use Kinesis
→ When you need block-level storage attached to a single EC2 → Use EBS
→ When your application needs consistent sub-millisecond response for key lookups → Use DynamoDB
→ When you need Windows-compatible file shares → Use FSx for Windows

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

THE ARCHITECTURE BENEATH S3

1 → Object Storage Model
→ S3 is NOT a file system — it is a distributed object store
→ Each object is stored as a complete unit — you cannot "append" to a file or modify byte 500
→ To change an object — you re-upload the entire object (overwrite)

2 → Key-Value Store Internally
→ The "key" is the full object path: invoices/jan-2025.pdf
→ The "value" is the binary content of the file + metadata
→ S3 uses a hash of the key to determine which partition stores the object
→ This is why S3 scales infinitely — keys are distributed across massive partition infrastructure

3 → Durability Mechanism — 99.999999999% (11 9s)
→ When you upload an object to S3 Standard:

  • S3 automatically replicates the data across minimum 3 Availability Zones
  • Each AZ is a physically separate data center with independent power, cooling, networking
  • The upload API returns HTTP 200 only AFTER the object is durably stored across all AZs
  • This means — even if an entire data center burns down, your data survives

→ Math: 11 9s durability = if you store 10 million objects, you expect to lose 1 object every 10,000 years

4 → Availability vs Durability

ConceptS3 Standard ValueMeaning
Durability99.999999999%Probability your data is NOT lost
Availability99.99%Probability you CAN access your data at any given moment

→ Durability ≠ Availability
→ Your object is always SAFE (durability) but the service might be briefly unreachable (availability)
→ 99.99% availability = up to ~52 minutes of downtime per year

5 → Consistency Model (Current)
→ S3 provides STRONG read-after-write consistency as of December 2020
→ What this means:

  • Upload an object → Immediately read it → You get the latest version
  • Overwrite an object → Immediately read it → You get the new version
  • Delete an object → Immediately try to read it → You get 404
    → Before 2020 — S3 had "eventual consistency" for overwrites/deletes (you might read stale data)
    → This is no longer the case — strong consistency is now the default with zero performance impact

6 → Request Routing
→ When you make a request to shopnow-product-images-2025.s3.ap-south-1.amazonaws.com/shoe.jpg
→ DNS resolves to an S3 endpoint in ap-south-1
→ S3 front-end receives the request
→ The key shoe.jpg is hashed to determine which internal partition holds the object
→ The partition serves the object from the nearest available AZ replica

7 → Multipart Upload (Important for large files)
→ For objects > 100 MB, AWS recommends multipart upload
→ For objects > 5 GB, multipart upload is MANDATORY
→ How it works:

  • File is split into parts (5 MB – 5 GB each, up to 10,000 parts)
  • Each part is uploaded independently (can be parallel)
  • If one part fails — only that part is re-uploaded
  • After all parts succeed — a "complete" API call assembles them
  • Maximum object size: 5 TB

📈 SCALABILITY CONSIDERATIONS

→ Storage Scalability

  • S3 has NO storage limit — you can store 1 byte or 1 exabyte
  • No provisioning required — it scales automatically
  • No "disk full" errors ever

→ Request Scalability

  • S3 supports 3,500 PUT/COPY/POST/DELETE requests per second per prefix
  • S3 supports 5,500 GET/HEAD requests per second per prefix
  • If you need more → spread objects across multiple prefixes
  • Example: Instead of data/file1.csv, use data/a1/file1.csv, data/b2/file2.csv
  • S3 automatically partitions by prefix for parallel throughput

→ Bandwidth Scalability

  • Single object download speed: up to ~100 Gbps (using byte-range fetches in parallel)
  • S3 Transfer Acceleration: Uses CloudFront edge locations for faster uploads from distant locations
  • Multipart upload: Parallelizes upload bandwidth

→ Cost Scalability

  • Costs scale linearly with storage and requests
  • No "committed capacity" required
  • Pay only for what you use
  • Lifecycle policies (Project 12) auto-move data to cheaper tiers

→ Organizational Scalability

  • Default: 100 buckets per account (soft limit — can request increase to 1,000+)
  • No limit on objects per bucket
  • Use prefixes and tags for organization — not more buckets
