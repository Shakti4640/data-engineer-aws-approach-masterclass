The Current Problem:

→ Today, access is controlled through IAM policies (from Project 2)
→ IAM can control:

  • ✅ "Can this user use Athena?" → athena:StartQueryExecution
  • ✅ "Can this user access this S3 bucket?" → s3:GetObject on bucket ARN
  • ✅ "Can this user access the Glue Catalog?" → glue:GetTable on database ARN

→ IAM CANNOT control:

  • ❌ "Can this user see columns A, B, C but not column D within the same table?"

→ The compliance officer says: "Either everyone sees everything or nobody sees anything — that's not governance, that's a liability"

→ The auditor says: "I need access from MY AWS account to ONLY the aggregate table — nothing else"

🟡 HOW TO APPROACH IT

→ First — understand the governance gap

IAM Alone (What We Have):

IAM Policy → Controls SERVICE-level access
    ├── Can user call Athena? YES/NO
    ├── Can user access S3 bucket? YES/NO
    └── Can user access Glue database? YES/NO
    
    CANNOT answer:
    ├── Can user access TABLE X within database? ???
    ├── Can user see COLUMN Y within table? ???
    └── Can user see ROWS matching condition Z? ???

Lake Formation (What We Need):

Lake Formation → Controls DATA-level access
    ├── Database-level: Can user access this database? YES/NO
    ├── Table-level: Can user access this table? YES/NO
    ├── Column-level: Can user see these specific columns? YES/NO (Include or Exclude)
    ├── Row-level: Can user see rows matching this filter? YES/NO
    └── Cross-account: Can external account access this specific table? YES/NO

→ Lake Formation sits ON TOP of IAM — it does NOT replace IAM
→ Both IAM AND Lake Formation must allow access for the request to succeed
→ Lake Formation adds a FINER grain of control that IAM alone cannot provide

→ The approach should be:

  • Step 1 — Understand the Lake Formation permission model
  • Step 2 — Register the S3 data location with Lake Formation
  • Step 3 — Set up Lake Formation Admins
  • Step 4 — Convert from IAM-only to Lake Formation permission model
  • Step 5 — Grant database-level permissions
  • Step 6 — Grant table-level permissions
  • Step 7 — Grant column-level permissions (include/exclude)
  • Step 8 — Configure row-level security with data filters
  • Step 9 — Configure Tag-Based Access Control (TBAC) for scalable governance
  • Step 10 — Configure cross-account access for the external auditor

🟢 HOW CAN WE SOLVE IT — Step-by-Step Configuration

STEP 1 — UNDERSTAND THE LAKE FORMATION PERMISSION MODEL

The Two Permission Models in AWS:

ModelHow It WorksGranularityDefault State
IAM-Only (Legacy)Access controlled entirely through IAM policies + S3 bucket policies + Glue resource policiesService and resource levelThis is the DEFAULT for all new AWS accounts
Lake Formation + IAM (Modern)IAM controls service access + Lake Formation controls data accessDatabase, table, column, row levelMust be explicitly enabled

The Critical Concept — IAMAllowedPrincipals:

→ When you first create a Glue database/table → Lake Formation automatically grants a SUPER permission to a special group called IAMAllowedPrincipals
→ This group means: "Anyone whose IAM policy allows Glue access can access this table"
→ This is why Lake Formation appears to do nothing by default — it's passing everything through to IAM

Default State (New Account):
    Glue Table: transactions
    Lake Formation Permission: IAMAllowedPrincipals → ALL access
    Effect: Anyone with IAM glue:GetTable permission → Full access to table
    
    Lake Formation is PRESENT but TRANSPARENT — IAM controls everything

→ To activate Lake Formation governance → You must REMOVE the IAMAllowedPrincipals permission
→ Once removed → ONLY explicit Lake Formation grants allow access
→ This is the "switch" from IAM-only to Lake Formation governance

STEP 4 — REVOKE IAMAllowedPrincipals (THE CRITICAL SWITCH)

This is the single most important step — and the most dangerous if done incorrectly

→ Navigate to Lake Formation Console → Data lake permissions

Option A — Revoke Per-Database (Surgical):

→ Select the permission entry for IAMAllowedPrincipals on database fraud_db
→ Click "Revoke"
→ Confirm

What this does:

BEFORE Revocation:
    IAMAllowedPrincipals → ALL on fraud_db → ALL tables
    Effect: Anyone with IAM Glue permissions → Full access

AFTER Revocation:
    No default permission exists
    Effect: NOBODY can access fraud_db unless explicitly granted by Lake Formation
    
    ⚠️ This IMMEDIATELY breaks access for everyone who was relying on IAM-only

Option B — Change Default Settings (Global):

→ Navigate to Lake Formation Console → Settings (under Administration)
→ Uncheck: "Use only IAM access control for new databases"
→ Uncheck: "Use only IAM access control for new tables in new databases"

SettingDefault (Checked)Recommended (Unchecked)
Use only IAM access control for new databasesIAMAllowedPrincipals auto-granted on new databasesNew databases require explicit Lake Formation grants
Use only IAM access control for new tablesIAMAllowedPrincipals auto-granted on new tablesNew tables require explicit Lake Formation grants

→ This affects FUTURE databases/tables — existing ones retain IAMAllowedPrincipals until manually revoked

THE DANGER:
→ If you revoke IAMAllowedPrincipals BEFORE granting explicit permissions → ALL users lose access IMMEDIATELY
→ Glue jobs fail, Athena queries fail, Redshift Spectrum fails
→ This is the #1 Lake Formation deployment failure

SAFE MIGRATION ORDER:

Step 4a: Grant explicit Lake Formation permissions to ALL current users (Steps 5-8)
Step 4b: VERIFY all users can access through Lake Formation grants
Step 4c: THEN revoke IAMAllowedPrincipals
Step 4d: VERIFY nothing broke

→ Grant FIRST → Verify → THEN Revoke → Verify again
→ NEVER Revoke first

→ The user cannot see, bypass, or disable the filter
→ Even SELECT COUNT(*) FROM transactions returns only the APAC count

How Row Filters Work Internally:

User Query: SELECT * FROM transactions WHERE amount > 1000

Lake Formation Injects Filter:
    SELECT * FROM transactions 
    WHERE (country IN ('IN','SG','JP','AU','NZ'))   ← Row filter (injected)
    AND (amount > 1000)                               ← User's filter (preserved)
    
    Columns returned: All EXCEPT credit_card_number   ← Column filter (applied)

→ Lake Formation modifies the query plan BEFORE it reaches Athena/Redshift Spectrum
→ The modification is invisible to the user
→ The query results appear as if the filtered-out rows simply don't exist

STEP 9 — CONFIGURE TAG-BASED ACCESS CONTROL (TBAC)

The Scalability Problem with Named Resource Grants:

→ PayScale has 5 tables → 6 user groups → ~15 individual grant entries
→ What if PayScale grows to 50 tables and 20 user groups?
→ 50 × 20 = potentially 1,000 individual grants → Unmanageable

Tag-Based Access Control (TBAC) Solution:

→ Instead of granting access per-table per-user → assign TAGS to tables and users
→ Create a POLICY that says: "If user has tag X, they can access tables with tag X"

✅ WHAT SHOULD DO

→ Always designate a Data Lake Administrator ROLE (not user) before any Lake Formation configuration
→ Always register S3 data locations BEFORE granting table-level permissions
→ Always grant explicit permissions BEFORE revoking IAMAllowedPrincipals — grant first, revoke second
→ Always test permission grants with actual Athena queries from the target user's session
→ Always use EXCLUDE columns for tables with few sensitive columns (simpler, auto-includes new columns)
→ Always use INCLUDE columns for tables with few safe columns (more restrictive, new columns are hidden by default)
→ Always design a tag taxonomy BEFORE implementing TBAC — retrofitting tags is painful
→ Always document the complete permission matrix: WHO has access to WHAT at which level
→ Always use resource links for cross-account sharing — direct table references don't work in Athena
→ Always grant permissions to IAM ROLES not individual IAM users — roles survive employee changes
→ Always keep the Glue ETL service role's permissions updated — Crawlers and Jobs need Lake Formation grants too
→ Always verify that both IAM AND Lake Formation allow the action — both must permit for access to succeed

❌ WHAT SHOULD NOT DO

→ Never revoke IAMAllowedPrincipals before granting explicit Lake Formation permissions — causes immediate access loss for ALL users
→ Never make individual IAM users Data Lake Administrators — use roles
→ Never grant "Grantable" permissions to end users — they could escalate access for themselves or others
→ Never forget to grant permissions to the Glue ETL service role — Crawlers and Jobs will fail silently
→ Never mix TBAC and Named Resource grants for the same table without understanding precedence — both are evaluated, most permissive wins
→ Never assume column exclusion on a table applies to future columns — EXCLUDE mode auto-includes new columns (potential security gap)
→ Never forget the resource link step for cross-account access — without it, Athena in the consumer account cannot query
→ Never skip the RAM acceptance step — the sharing won't work until accepted
→ Never use Lake Formation for S3 paths that aren't registered — permissions won't be enforced
→ Never assume row-level filters prevent data inference — aggregation queries (COUNT, SUM) on filtered data may still reveal patterns
→ Never change Lake Formation settings during business hours without testing — one wrong revocation breaks all queries

ScenarioUse Lake Formation?Why

PII columns must be hidden from certain users✅ YesColumn-level exclusion
Regional teams should only see their region's data✅ YesRow-level data filters

All users need identical full access❌ NoIAM alone is simpler

🔴 WHEN NOT TO USE LAKE FORMATION

→ When you have a single team with uniform access needs → IAM is simpler and sufficient
→ When your data is not in S3 + Glue Catalog → Lake Formation doesn't govern RDS, DynamoDB, etc. directly
→ When you need real-time row-level security for application-level queries → Application-layer authorization is more appropriate
→ When your organization has < 3 distinct access roles → Overhead of Lake Formation exceeds benefit
→ When you need encryption-based access control → KMS key policies (from Project 36) handle this
→ When data sharing is purely at the S3 bucket level → Cross-account bucket policies (from Project 39) are simpler

Key Insight — S3 Permission Delegation:
→ When Lake Formation is active → Users do NOT need direct S3 access
→ Lake Formation uses ITS OWN registered IAM role to access S3 on behalf of the user
→ This is called "credential vending"
→ Lake Formation generates temporary S3 credentials scoped to ONLY the permitted data

WITHOUT Lake Formation:
    User → IAM (Athena + Glue + S3 permissions) → S3
    User needs: s3:GetObject on the actual data paths

WITH Lake Formation:
    User → IAM (Athena + Glue permissions only) → Lake Formation → S3
    User does NOT need: s3:GetObject
    Lake Formation's role provides S3 access

→ This is a major security improvement — users never directly touch S3
→ Even if a user discovers the S3 path — they cannot access it without Lake Formation's credential vending

2 → HOW COLUMN-LEVEL FILTERING WORKS

User Query: SELECT * FROM transactions

Lake Formation Permission: EXCLUDE credit_card_number

Internal Query Rewrite:
    SELECT transaction_id, customer_id, amount, merchant, city, country, 
           risk_score, created_at 
    FROM transactions

→ The * is EXPANDED to only permitted columns
→ The excluded column is removed from the column list BEFORE the query engine processes it
→ Athena's query plan never sees the excluded column
→ The Parquet file reader in S3 uses column projection to skip reading excluded columns
→ This means: excluded columns are not just hidden — they are never READ from storage
→ Performance benefit: Less data scanned = lower Athena cost (Athena charges per TB scanned)

3 → HOW ROW-LEVEL FILTERING WORKS

User Query: SELECT * FROM transactions WHERE amount > 1000

Data Filter: country IN ('IN', 'SG', 'JP', 'AU', 'NZ')

Internal Query Rewrite:
    SELECT transaction_id, customer_id, amount, merchant, city, country,
           risk_score, created_at
    FROM transactions
    WHERE (country IN ('IN', 'SG', 'JP', 'AU', 'NZ'))    ← INJECTED by Lake Formation
      AND (amount > 1000)                                   ← Original user filter

→ Lake Formation's row filter is injected as an AND condition
→ The user's filter is preserved
→ Both must be true for a row to be returned
→ If the table is partitioned by country (from Project 25)
  → Athena uses partition pruning → Only reads APAC partitions
  → Double efficiency: Less partitions scanned + row filter applied

Row Filter Limitations:

SupportedNot Supported
Simple comparisons (=, >, <, IN, BETWEEN)Subqueries in filter expression
AND/OR combinationsJOINs in filter expression
String functions (LIKE, starts_with)Aggregate functions (COUNT, SUM)
Column references from the same tableReferences to other tables
NULL checks (IS NULL, IS NOT NULL)User-defined functions

4 → HOW TAG-BASED ACCESS CONTROL RESOLVES

User: DataScientistRole
    Tags: sensitivity IN (public, internal, confidential)

Table: fraud_db.transactions
    Tags: sensitivity = confidential

Column: transactions.credit_card_number
    Tags: sensitivity = restricted

Resolution:
    1. Table tag: confidential → User has confidential → TABLE ACCESS GRANTED ✅
    2. Column tag: restricted → User does NOT have restricted → COLUMN ACCESS DENIED ❌
    3. Other columns: Inherit table tag (confidential) → User has confidential → GRANTED ✅

Result: DataScientist can query transactions table, all columns EXCEPT credit_card_number

Tag Resolution Rules:

RuleBehavior
Column with explicit tagColumn's own tag is evaluated (overrides table tag)
Column without explicit tagInherits the table's tag
Table with explicit tagTable's own tag is evaluated (overrides database tag)
Table without explicit tagInherits the database's tag
Multiple tags on same resourceALL tag conditions must be satisfied (AND logic)
No matching tag grant existsACCESS DENIED

🎯 KEY TAKEAWAY

→ Lake Formation adds DATA-LEVEL governance on top of IAM's SERVICE-LEVEL access control
→ IAM answers: "Can this user call Athena?" → Lake Formation answers: "Can this user see THIS table's THIS column's THIS row?"
