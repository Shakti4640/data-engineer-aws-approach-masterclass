PROJECT 37: THE CREDENTIALS HARDCODED IN CONFIG FILES — Secrets Manager for Database Credentials

🔴 HYPOTHETICAL PROBLEM

→ The auditor David (from Project 36) has signed off on encryption at rest
→ But his final finding is equally alarming — credential management is non-existent

Finding 1 — Glue ETL Jobs (Project 24)
→ The Glue JDBC connection to MariaDB stores the database password in the connection properties
→ Anyone with glue:GetConnection IAM permission can read the password in plain text
→ The password has not been changed in 14 months
→ Three former employees who set up the connection still know this password

Finding 2 — Athena Federated Query Connector (Project 35)
→ The Lambda connector references a Secrets Manager secret (correctly set up in Project 35)
→ But the secret was created manually with a password that the developer chose
→ That same password is also written in a Confluence wiki page titled "Database Credentials"
→ No automatic rotation is configured
→ If the password is compromised — manual update required across multiple services

Finding 3 — EC2 Application (Project 14)
→ The application running on EC2 connects to MariaDB using credentials stored in a .env file on the instance
→ The .env file contains:

DB_HOST=shopnow-mariadb-prod-encrypted.abc123.ap-south-1.rds.amazonaws.com
DB_PORT=3306
DB_USER=app_user
DB_PASSWORD[REDACTED:PASSWORD]3!SecurePass
DB_NAME=shopnow_prod

→ Anyone who SSH into the instance (or gains shell access via exploit) reads credentials instantly
→ The AMI snapshot of this instance also contains these credentials
→ If the instance is compromised — attacker has direct database access

Finding 4 — Lambda Functions (Project 26, 34)
→ Two Lambda functions that write to MariaDB store credentials as Lambda environment variables
→ Environment variables are visible in the Lambda console to anyone with lambda:GetFunction permission
→ Environment variables are included in deployment packages and version histories

Finding 5 — Shared Credentials Anti-Pattern
→ All services use the SAME database username and password: admin / ShopNow2023!SecurePass
→ If one credential is leaked — everything is compromised
→ No way to know WHICH service accessed the database — all connections look identical in MariaDB logs
→ Cannot revoke access for one service without breaking all others

The Auditor's Mandate:

"Centralize all credentials. Eliminate hardcoded secrets everywhere.
Implement automatic rotation. Use unique credentials per service.
Prove you can answer: WHO retrieved WHICH credential, WHEN."

→ This requires AWS Secrets Manager as the centralized credential vault

🟡 HOW TO APPROACH IT

→ First — understand why this is a SEPARATE problem from encryption (Project 36)

Encryption (Project 36) protects:
→ Data AT REST on disk — S3 objects, RDS volumes, Redshift blocks
→ If someone steals a hard drive → data is unreadable
→ Encryption does NOT prevent authorized applications from reading data
→ If an application has valid credentials → encryption is transparent

Secrets Management (This project) protects:
→ The CREDENTIALS that grant access to data
→ If someone steals credentials → they bypass encryption entirely (they authenticate as a valid user)
→ Secrets Manager ensures credentials are:

  • Never stored in plain text anywhere
  • Fetched dynamically at runtime
  • Rotated automatically
  • Audited for every access
  • Independently revocable per consumer

→ The approach:

  • Step 1 — Understand Secrets Manager architecture
  • Step 2 — Create dedicated MariaDB users per service
  • Step 3 — Store each credential set as a separate secret
  • Step 4 — Configure secret encryption with KMS (connecting to Project 36)
  • Step 5 — Configure IAM policies for secret access
  • Step 6 — Update Glue JDBC connection to use Secrets Manager
  • Step 7 — Verify Athena connector secret configuration (from Project 35)
  • Step 8 — Update EC2 application to fetch secrets dynamically
  • Step 9 — Update Lambda functions to fetch secrets dynamically
  • Step 10 — Configure automatic rotation
  • Step 11 — Verify audit trail in CloudTrail

🟢 HOW CAN WE SOLVE IT — Step-by-Step Configuration

STEP 1 — UNDERSTAND SECRETS MANAGER ARCHITECTURE

What Secrets Manager actually stores:
→ A "secret" is a JSON blob containing key-value pairs
→ For a database credential, the standard structure is:

{
  "username": "app_user",
  "password": "auto-generated-complex-password",
  "engine": "mysql",
  "host": "shopnow-mariadb-prod-encrypted.abc123.ap-south-1.rds.amazonaws.com",
  "port": 3306,
  "dbname": "shopnow_prod"
}

→ This JSON is encrypted using a KMS key (from Project 36)
→ Stored in Secrets Manager's internal storage (not in S3, not in DynamoDB)
→ Each secret has:

  • A name (path-like): shopnow/prod/mariadb/app-user
  • An ARN: arn:aws:secretsmanager:ap-south-1:123456789012:secret:shopnow/prod/mariadb/app-user-AbCdEf
  • Version stages: AWSCURRENT (active), AWSPREVIOUS (last rotation), AWSPENDING (during rotation)
  • Metadata: Description, tags, rotation configuration, KMS key ID

How Secrets Manager differs from SSM Parameter Store (SecureString):

FeatureSecrets ManagerSSM Parameter Store (SecureString)
Automatic rotation✅ Built-in with Lambda❌ Manual implementation required
RDS/MariaDB integration✅ Native rotation templates❌ No database integration
Cross-region replication✅ Built-in replica secrets❌ Not supported
Versioning with staging labels✅ AWSCURRENT/AWSPREVIOUS/AWSPENDING❌ Basic version history only
Cost$0.40 per secret per month + $0.05 per 10,000 API callsFree tier for Standard, $0.05 per Advanced parameter per month
Maximum secret size64 KB8 KB (Advanced: 8 KB)
Random password generation✅ Built-in API❌ Not available

→ For database credentials with rotation → Secrets Manager is the correct choice
→ For simple configuration values (non-sensitive) → SSM Parameter Store is cheaper

STEP 4 — UNDERSTAND SECRET ENCRYPTION WITH KMS

→ Connecting to Project 36's KMS concepts:

How Secrets Manager uses KMS:

You store a secret (JSON with credentials)
        ↓
Secrets Manager calls KMS: GenerateDataKey(KeyId = shopnow-rds-mariadb-key)
        ↓
KMS returns:
  1. Plaintext DEK
  2. Encrypted DEK
        ↓
Secrets Manager encrypts the secret JSON using the plaintext DEK (AES-256-GCM)
        ↓
Secrets Manager stores:
  - Encrypted secret value
  - Encrypted DEK
        ↓
Plaintext DEK discarded from memory

When a service retrieves the secret:

Service calls: GetSecretValue(SecretId = shopnow/prod/mariadb/ec2-app)
        ↓
Secrets Manager retrieves encrypted secret + encrypted DEK
        ↓
Secrets Manager calls KMS: Decrypt(CiphertextBlob = encrypted DEK)
        ↓
KMS checks:
  1. Caller has kms:Decrypt in key policy? (Project 36 key user check)
  2. Caller has kms:Decrypt in IAM policy?
  3. Key is enabled?
        ↓
KMS returns plaintext DEK
        ↓
Secrets Manager decrypts secret JSON using plaintext DEK
        ↓
Returns plain text secret to the caller
        ↓
CloudTrail logs BOTH:
  - secretsmanager:GetSecretValue event
  - kms:Decrypt event

→ This means: To retrieve a secret, the caller needs BOTH:

  • secretsmanager:GetSecretValue permission (IAM policy)
  • kms:Decrypt permission on the encryption key (IAM policy + Key policy)
    → This is TRIPLE-gated security:
  1. Secrets Manager resource policy (optional)
  2. IAM policy on the caller role
  3. KMS key policy on the encryption key

STEP 11 — CONFIGURE AUTOMATIC ROTATION

→ This is the most powerful feature of Secrets Manager — and the auditor's primary requirement
→ Navigate to each secret → "Rotation configuration" → Edit rotation

Understanding How Rotation Works:

Day 0: Secret has passwor[REDACTED:PASSWORD]123" (AWSCURRENT)
        ↓
Rotation triggered (automatic schedule or manual)
        ↓
Step 1 — createSecret:
  Secrets Manager generates new password "NewPass456"
  Stores it as version stage AWSPENDING
  Old password "OldPass123" remains AWSCURRENT
  Both passwords are valid in the secret store
        ↓
Step 2 — setSecret:
  Rotation Lambda connects to MariaDB
  Changes the database user's password TO "NewPass456"
  Now MariaDB accepts "NewPass456"
  MariaDB no longer accepts "OldPass123"
        ↓
Step 3 — testSecret:
  Rotation Lambda tests connection to MariaDB
  using the new password "NewPass456"
  Verifies it works (SELECT 1)
        ↓
Step 4 — finishSecret:
  Secrets Manager moves version labels:
    "NewPass456" → AWSCURRENT
    "OldPass123" → AWSPREVIOUS
  Rotation complete

→ Applications fetching AWSCURRENT now get "NewPass456"
→ Applications with cached "OldPass123" will get authentication errors
→ Those applications retry → fetch fresh secret → get "NewPass456" → reconnect successfully
→ The cache TTL (Step 10) determines maximum disruption window

Rotation Configuration Settings:

SettingValueExplanation
Automatic rotation✅ Enable

Rotation window3:00 AM - 5:00 AM UTCChoose low-traffic window

Rotation strategy"Single user" or "Alternating users"See explanation below

Two Rotation Strategies:

Strategy 1 — Single User Rotation:

One MariaDB user: glue_etl_user
        ↓
Rotation changes THIS user's password
        ↓
Brief moment where old password is invalid, new password is active
        ↓
Risk: If application connects with old cached password 
      during the exact moment of password change → connection fails
        ↓
Application retries → fetches new password → succeeds
ProsCons
Simple — one user to manageBrief disruption window during password change
Lower MariaDB user countApplications with aggressive connection pooling may see errors
Sufficient for most use cases

Strategy 2 — Alternating Users Rotation:

Two MariaDB users: glue_etl_user and glue_etl_user_clone
        ↓
Rotation 1: Changes glue_etl_user_clone's password, makes it AWSCURRENT
Rotation 2: Changes glue_etl_user's password, makes it AWSCURRENT
Rotation 3: Changes glue_etl_user_clone's password, makes it AWSCURRENT
... alternates forever
        ↓
The user NOT being rotated still has a valid password
        ↓
Zero disruption — old password remains valid until next rotation cycle
ProsCons
Zero disruption during rotationRequires TWO MariaDB users per service
Old password valid until next rotationMore complex setup
Recommended for production workloadsDoubles MariaDB user management

→ For ShopNow production → Use Alternating Users strategy
→ For dev/test → Single User is sufficient

STEP 13 — SECRET VERSIONING DEEP DIVE

→ Every secret has version stages that control which value is "active"

Version Stage Lifecycle:

Initial Secret Creation:
  Version 1 → AWSCURRENT (this is what GetSecretValue returns by default)

After First Rotation:
  Version 1 → AWSPREVIOUS
  Version 2 → AWSCURRENT

After Second Rotation:
  Version 2 → AWSPREVIOUS
  Version 3 → AWSCURRENT
  Version 1 → (no stage — still exists but not labeled)

During Rotation (brief window):
  Version 2 → AWSCURRENT (old password — still works in DB)
  Version 3 → AWSPENDING (new password — being set in DB)

How Applications Should Handle Versions:

ScenarioWhat to Do
Normal operationFetch default (AWSCURRENT) — always returns latest valid credential
Connection fails after rotationCatch authentication error → fetch fresh secret (invalidate cache) → retry
Need to rollback rotationManually set AWSCURRENT back to previous version
Testing new credential before cutoverFetch AWSPENDING version explicitly

✅ WHAT SHOULD DO

→ Always use separate database users per service — never share credentials
→ Always use hierarchical secret naming: {company}/{env}/{engine}/{consumer}
→ Always exclude JDBC-breaking characters from auto-generated passwords
→ Always encrypt secrets with Customer Managed KMS keys (Project 36)
→ Always configure automatic rotation with Alternating Users strategy for production
→ Always set cache TTL to balance freshness vs performance (5 minutes recommended)
→ Always configure resource policies on secrets as defense-in-depth
→ Always set up CloudTrail monitoring for Secrets Manager events
→ Always alert on RotationFailed events — they mean passwords are NOT being changed
→ Always test rotation manually before enabling automatic schedule
→ Always verify rotation Lambda has VPC access to RDS and Secrets Manager
→ Always delete .env files, config files, and environment variables after migration to Secrets Manager

❌ WHAT SHOULD NOT DO

→ Never store database passwords in Lambda environment variables — visible in console and API
→ Never store credentials in .env files on EC2 instances — survives in AMI snapshots
→ Never store credentials in Glue connection properties without Secrets Manager integration
→ Never share one database user across multiple services — prevents audit and blast radius containment
→ Never use the admin/root database user for application access — use least-privilege dedicated users
→ Never skip password character exclusion — JDBC URL parsing breaks on @, /, \ characters
→ Never set rotation interval longer than 90 days for production — compliance frameworks require shorter
→ Never forget to update RDS security group for rotation Lambda access — rotation fails silently
→ Never delete a secret without verifying no services depend on it — check CloudTrail for recent GetSecretValue
→ Never hardcode secret ARNs — use secret names (Secrets Manager resolves names to ARNs)
→ Never grant secretsmanager:* to application roles — only grant GetSecretValue
→ Never store non-credential configuration in Secrets Manager — use SSM Parameter Store (cheaper)

MECHANIC 2 — GetSecretValue API FLOW

Application calls GetSecretValue(SecretId = "shopnow/prod/mariadb/ec2-app")
        ↓
Secrets Manager checks:
  1. Resource policy on the secret → Allow or Deny?
  2. IAM policy on the caller → secretsmanager:GetSecretValue allowed?
  3. Any SCP restrictions? (Project 54 preview)
        ↓
If authorized:
  Secrets Manager retrieves encrypted blob + encrypted DEK
        ↓
  Secrets Manager calls KMS Decrypt internally:
    - Uses the caller's identity for KMS authorization check
    - KMS checks key policy + IAM policy for kms:Decrypt
        ↓
  If KMS authorizes:
    KMS returns plaintext DEK
    Secrets Manager decrypts the secret blob
    Returns plaintext JSON to caller
        ↓
  CloudTrail records:
    Event 1: secretsmanager:GetSecretValue (who, when, which secret)
    Event 2: kms:Decrypt (who, when, which key)

CRITICAL DETAIL: KMS Authorization Uses Caller Identity
→ When Secrets Manager calls KMS Decrypt → KMS sees the ORIGINAL caller (e.g., EC2AppRole)
→ NOT the Secrets Manager service identity
→ This means: Even if Secrets Manager tries to decrypt, it checks YOUR role's KMS permissions
→ If your role does not have kms:Decrypt on the key → Secrets Manager cannot decrypt → GetSecretValue fails
→ This is why BOTH secretsmanager:GetSecretValue AND kms:Decrypt are required

MECHANIC 3 — ROTATION LAMBDA INTERNAL FLOW

Rotation Triggered (schedule or manual)
        │
        ├── Step 1: createSecret
        │   └── Lambda calls GetRandomPassword → generates "NewPass456"
        │   └── Lambda calls PutSecretValue(AWSPENDING) → stores new password in secret
        │   └── At this point: AWSCURRENT = "OldPass123", AWSPENDING = "NewPass456"
        │   └── MariaDB still only accepts "OldPass123"
        │
        ├── Step 2: setSecret
        │   └── Lambda connects to MariaDB using admin credentials
        │   └── Lambda executes ALTER USER to change password to "NewPass456"
        │   └── At this point: MariaDB accepts "NewPass456"
        │   └── MariaDB may still accept "OldPass123" briefly (connection caching)
        │
        ├── Step 3: testSecret
        │   └── Lambda fetches AWSPENDING version from Secrets Manager
        │   └── Lambda connects to MariaDB using "NewPass456"
        │   └── Lambda runs test query (SELECT 1 or similar)
        │   └── If test fails → rotation aborted → AWSCURRENT stays as "OldPass123"
        │
        └── Step 4: finishSecret
            └── Lambda calls UpdateSecretVersionStage:
                  Move AWSCURRENT label to new version ("NewPass456")
                  Move AWSPREVIOUS label to old version ("OldPass123")
            └── At this point: GetSecretValue returns "NewPass456" by default
            └── Rotation complete → CloudTrail logs RotationSucceeded

What if rotation fails at Step 2 or 3?
→ AWSCURRENT still points to "OldPass123"
→ Applications continue working with old password
→ CloudTrail logs RotationFailed event
→ Secret remains in "rotation in progress" state
→ Fix the issue → retry rotation manually or wait for next scheduled attempt

MECHANIC 4 — ALTERNATING USERS INTERNAL FLOW

Secret stores TWO usernames: glue_etl_user and glue_etl_user_clone
        │
Rotation 1:
        ├── Currently active: glue_etl_user (AWSCURRENT)
        ├── createSecret → generate new password for glue_etl_user_clone
        ├── setSecret → change glue_etl_user_clone's password in MariaDB
        ├── testSecret → verify glue_etl_user_clone can connect
        ├── finishSecret → AWSCURRENT now returns glue_etl_user_clone credentials
        │
Rotation 2:
        ├── Currently active: glue_etl_user_clone (AWSCURRENT)
        ├── createSecret → generate new password for glue_etl_user
        ├── setSecret → change glue_etl_user's password in MariaDB
        ├── testSecret → verify glue_etl_user can connect
        ├── finishSecret → AWSCURRENT now returns glue_etl_user credentials
        │
... alternates forever

→ The KEY ADVANTAGE: During rotation, the currently active user's password is NOT being changed
→ Applications using the current user experience ZERO disruption
→ Only the inactive user's password is changed
→ After rotation completes → applications fetch the newly activated user on next secret retrieval

📈 SCALABILITY CONSIDERATIONS

→ Secret Count Scaling

  • Default: 500,000 secrets per region per account
  • Practically unlimited for most organizations
  • Naming convention becomes critical at scale — use hierarchical paths
  • Use tags for filtering and cost allocation

→ API Request Scaling

  • Default: 10,000 GetSecretValue requests per second per account per region
  • With caching → actual API calls are a fraction of application requests
  • Example: 1,000 Lambda concurrent executions with 5-minute cache → ~3-4 API calls/second average
  • Without caching → 1,000 Lambda invocations/second → 1,000 API calls/second → within limits but wasteful

→ Rotation Scaling

  • Each secret has its own rotation schedule
  • Multiple secrets can share one rotation Lambda function
  • Rotation Lambda invoked once per rotation per secret
  • 100 secrets rotating every 30 days → ~3 rotation Lambda invocations per day → negligible

→ Multi-Region Scaling

  • Each secret can be replicated to multiple regions
  • Replication is asynchronous — eventual consistency (typically seconds)
  • Applications in each region read from local replica → low latency
  • Rotation in source region propagates to replicas automatically

→ Multi-Account Scaling (Preview for Projects 39-40)

  • Secrets CAN be shared cross-account via resource policy
  • Or each account stores its own secrets (preferred for isolation)
  • Central security account manages rotation configuration
  • Application accounts only have GetSecretValue permission

→ Connection Pool Scaling with Alternating Users

  • After rotation → applications gradually migrate connections to new user
  • Connection pools may hold connections with OLD user → these remain valid until next rotation
  • No pool draining required — connections gracefully transition
