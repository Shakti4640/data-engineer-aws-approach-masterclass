PROJECT 6: THE TEAM THAT MISSED EVERY ALERT — SNS Topic Creation & Email Subscription

🔴 HYPOTHETICAL PROBLEM

→ ShopNow (from Projects 1-5) now has:

  • S3 bucket for product images (Project 1)
  • IAM users with proper permissions (Project 2)
  • EC2 instances in a properly architected VPC (Projects 3, 4, 5)

→ Last Friday at 11 PM — the web server's disk filled up → application crashed
→ Nobody noticed until Monday morning → 60 hours of downtime
→ Customers couldn't browse products → estimated revenue loss: ₹4,00,000

→ The CTO asks: "Why didn't anyone know the server was down?"
→ Answer: There was no alerting system — nobody was watching at 11 PM

→ Then Tuesday — the marketing team uploaded 200 new product images to S3
→ The catalog team didn't know → the images sat in S3 for 3 days before being processed
→ Marketing says: "We emailed the catalog team" — but the email went to spam

→ Then Wednesday — a batch job failed at 3 AM
→ The developer who owns it was asleep → the failed job blocked the morning reports
→ The finance team discovered the issue at 10 AM → 7 hours wasted

→ Three problems — ONE root cause: No automated notification system

→ The CTO demands:

  • When the server has issues → notify the on-call engineer IMMEDIATELY
  • When files are uploaded to S3 → notify the catalog team AUTOMATICALLY
  • When batch jobs fail → notify the responsible developer AND the ops team
  • ONE notification → MULTIPLE recipients → different channels (email, SMS, potentially more)

→ The team needs a PUBLISH-SUBSCRIBE messaging service

🟡 HOW TO APPROACH IT

→ This is a PUB/SUB (Publish-Subscribe) problem
→ PUB/SUB means:

  • A PUBLISHER sends a message to a central TOPIC
  • All SUBSCRIBERS to that topic receive the message
  • The publisher doesn't know or care who the subscribers are
  • Subscribers don't know or care who the publisher is
  • The TOPIC is the decoupling layer between them

→ AWS SNS (Simple Notification Service) is purpose-built for this pattern

│ CloudWatch   │──────────►│           │──────────►│ Email        │

→ ONE message published → ALL subscribers receive it simultaneously
→ This is called FAN-OUT

→ The approach should be:

  • Step 1 — Create an SNS Topic
  • Step 2 — Understand topic types (Standard vs FIFO)
  • Step 3 — Create subscriptions (Email, SMS, and other protocols)

  • Step 5 — Configure message attributes and filtering

STEP 4 — SUBSCRIPTION CONFIRMATION WORKFLOW

→ For Email and HTTP/HTTPS protocols → SNS requires EXPLICIT confirmation
→ This prevents abuse — you can't subscribe someone else's email without their consent

Email Confirmation Flow:

Step 1: You create subscription → SNS sends confirmation email to endpoint
        │
        ▼
Step 2: Email arrives with subject: "AWS Notification - Subscription Confirmation"
        Body contains a "Confirm subscription" link
        │
        ▼
Step 3: Recipient clicks the link
        │
        ├── Link leads to AWS SNS endpoint
        ├── SNS marks subscription as "Confirmed"
        └── Subscription status changes: "Pending confirmation" → "Confirmed"
        │
        ▼
Step 4: Subscription is ACTIVE — future messages will be delivered

IF recipient never clicks the link:
        │
        ▼
Subscription stays "Pending" for 3 days → then AUTOMATICALLY DELETED
Messages published during "Pending" state are NOT delivered to this subscriber

→ CRITICAL: Until the email recipient clicks "Confirm" → they receive NOTHING
→ This is the #1 reason people think SNS "isn't working" → they forgot to confirm

HTTP/HTTPS Confirmation Flow:

Step 1: SNS sends HTTP POST to the endpoint URL with:
        - Type: "SubscriptionConfirmation"
        - SubscribeURL: URL to visit to confirm
        - Token: Confirmation token
        │
        ▼
Step 2: Your application must:
        Option A: Make a GET request to the SubscribeURL
        Option B: Call SNS ConfirmSubscription API with the Token
        │
        ▼
Step 3: SNS marks subscription as "Confirmed"

→ HTTP/HTTPS endpoints must be programmed to handle the confirmation
→ This is why automated integrations (SQS, Lambda) auto-confirm — they're AWS-internal

Unsubscribe:
→ Every email notification includes an "Unsubscribe" link at the bottom
→ Clicking it removes the subscription immediately
→ This is AWS's compliance with anti-spam requirements

STEP 5 — MESSAGE ATTRIBUTES AND SUBSCRIPTION FILTERING

→ Not every subscriber wants EVERY message
→ The ops team wants CRITICAL alerts but not INFO messages
→ The CTO wants only CRITICAL alerts

Message Attributes:

→ When publishing a message → you can attach key-value metadata called "Message Attributes"
→ These attributes do NOT appear in the message body — they are metadata used for filtering

Example Message Attributes:

Attribute NameTypeExample Value
severityStringCRITICAL
serviceStringweb-server
environmentStringproduction
error_codeNumber500

Subscription Filter Policy:

→ A filter policy tells SNS: "Only deliver messages to this subscriber IF the message attributes match"

→ SNS Console → Select subscription → "Edit" → Subscription filter policy

Filter Policy for CTO (only CRITICAL alerts):

Filter Policy:
{
  "severity": ["CRITICAL"]
}

→ This means: Only deliver messages to the CTO's email if severity attribute = CRITICAL
→ Messages with severity: WARNING or severity: INFO are NOT delivered to the CTO
→ Messages with NO severity attribute are also NOT delivered (unless policy uses exists: false)

Filter Policy for Ops Team (CRITICAL and WARNING):

Filter Policy:
{
  "severity": ["CRITICAL", "WARNING"]
}

→ Ops team gets CRITICAL and WARNING but not INFO

Filter Policy for Developer (specific service):

Filter Policy:
{
  "severity": ["CRITICAL", "WARNING", "INFO"],
  "service": ["batch-processor"]
}

→ Developer gets all severities BUT only for the batch-processor service
→ Multiple attributes = AND logic (both conditions must match)
→ Multiple values within an attribute = OR logic (any value matches)

Advanced Filter Operators:

OperatorExampleMeaning
Exact match"severity": ["CRITICAL"]severity must be exactly "CRITICAL"
Anything-but"severity": [{"anything-but": ["INFO"]}]severity can be anything EXCEPT "INFO"
Prefix"service": [{"prefix": "web-"}]service must start with "web-"
Numeric match"error_code": [{"numeric": [">=", 500]}]error_code must be >= 500
Exists"severity": [{"exists": true}]Message must HAVE a severity attribute
Not exists"severity": [{"exists": false}]Message must NOT have a severity attribute

Filter Policy Scope:

ScopeFilters OnDefault
MessageAttributesMessage metadata (attributes)YES (default)
MessageBodyActual message content (JSON body)Must explicitly set

→ Filtering on MessageBody allows filtering on the actual JSON payload
→ Example: Filter on {"status": "FAILED"} in the message body
→ Set via: Subscription → Edit → "Filter policy scope" → "Message body"

STEP 6 — TOPIC ACCESS POLICY (WHO CAN PUBLISH)

→ The topic access policy controls:

  • Who can PUBLISH messages to the topic
  • Who can SUBSCRIBE to the topic
  • Who can manage the topic

→ For ShopNow → we need S3 and CloudWatch to publish to our topics
→ These are AWS services — not IAM users — so we need resource-based policies

Allowing S3 to Publish to SNS Topic:

→ SNS Console → Select shopnow-s3-events → "Edit" → Access policy

Policy to Allow S3 Events (needed for Project 15):

Statement:
  Sid: AllowS3ToPublish
  Effect: Allow
  Principal:
    Service: s3.amazonaws.com
  Action: sns:Publish
  Resource: arn:aws:sns:ap-south-1:123456789012:shopnow-s3-events
  Condition:
    ArnLike:
      aws:SourceArn: arn:aws:s3:::shopnow-productimages-prod-aps1
    StringEquals:
      aws:SourceAccount: "123456789012"

→ Principal: Service: s3.amazonaws.com → allows the S3 service to publish
→ Condition: ArnLike: aws:SourceArn → only from the specific bucket (not ANY bucket)
→ Condition: StringEquals: aws:SourceAccount → only from your account (prevents confused deputy)

Why Both Conditions:
→ Without SourceArn → ANY S3 bucket in your account could publish → too broad
→ Without SourceAccount → an S3 bucket in ANOTHER account could publish if they know your topic ARN → security risk
→ Both together = defense in depth

Allowing CloudWatch to Publish to SNS Topic:

Statement:
  Sid: AllowCloudWatchToPublish
  Effect: Allow
  Principal:
    Service: cloudwatch.amazonaws.com
  Action: sns:Publish
  Resource: arn:aws:sns:ap-south-1:123456789012:shopnow-ops-alerts
  Condition:
    ArnLike:
      aws:SourceArn: arn:aws:cloudwatch:ap-south-1:123456789012:alarm:*

→ This allows CloudWatch alarms (Project 8) to send notifications to this topic
→ The wildcard alarm:* allows any alarm in the account to publish

Allowing Another AWS Account to Publish (Preview for Project 41):

Statement:
  Sid: AllowCrossAccountPublish
  Effect: Allow
  Principal:
    AWS: "arn:aws:iam::987654321098:root"
  Action: sns:Publish
  Resource: arn:aws:sns:ap-south-1:123456789012:shopnow-ops-alerts

→ This allows account 987654321098 to publish to your topic
→ Used in inter-account notification patterns (Project 41)

STEP 8 — DELIVERY RETRY AND DEAD LETTER QUEUE

What Happens When Delivery Fails:

→ SNS attempts to deliver → endpoint is unavailable → delivery fails
→ SNS retries based on the delivery protocol:

Retry Policies by Protocol:

ProtocolRetry Behavior
HTTP/HTTPSConfigurable: 0-100 retries over minutes/hours
SQS3 retries immediately (SQS is almost always available)
Lambda3 retries (2 internal + 1 SNS)
EmailNo retries — one attempt only (relies on email infrastructure)
SMSNo retries — one attempt only

→ Email and SMS: If delivery fails → message is LOST
→ HTTP/SQS/Lambda: SNS retries multiple times

Dead Letter Queue (DLQ) for Failed Deliveries:

→ For critical notifications → configure a DLQ to capture failed deliveries
→ The DLQ is an SQS queue (Project 7) that receives messages SNS couldn't deliver

Configuration:

→ SNS Console → Select subscription → "Edit" → "Redrive policy (dead-letter queue)"

SettingValue
Dead-letter queuearn:aws:sqs:ap-south-1:123456789012:shopnow-sns-dlq

→ The SQS queue must have a policy allowing SNS to send messages to it:

Statement:
  Sid: AllowSNSToSendToDLQ
  Effect: Allow
  Principal:
    Service: sns.amazonaws.com
  Action: sqs:SendMessage
  Resource: arn:aws:sqs:ap-south-1:123456789012:shopnow-sns-dlq
  Condition:
    ArnEquals:
      aws:SourceArn: arn:aws:sns:ap-south-1:123456789012:shopnow-ops-alerts

→ Now if SNS can't deliver to an HTTP endpoint after all retries → message goes to DLQ
→ You can monitor the DLQ for failed deliveries and investigate
→ DLQ is NOT supported for Email and SMS subscriptions (they don't retry)

✅ WHAT SHOULD DO

→ Always create separate topics for separate concerns (ops alerts, business events, batch notifications)
→ Always use subscription filter policies to reduce noise for each subscriber
→ Always confirm email subscriptions immediately — pending subscriptions auto-delete after 3 days
→ Always use message attributes for categorization (severity, service, environment)
→ Always configure access policies with conditions (SourceArn, SourceAccount) when allowing services to publish
→ Always enable delivery status logging for debugging
→ Always set SMS account spend limits to prevent cost surprises
→ Always use custom payloads per protocol when email and programmatic consumers need different formats
→ Always configure DLQ for HTTP/SQS/Lambda subscriptions to capture failed deliveries
→ Always encrypt topics with KMS when messages contain sensitive data
→ Always use descriptive topic names that indicate their purpose
→ Always test the full flow: Publish → Confirm delivery → Verify filter policies

❌ WHAT SHOULD NOT DO

→ Never create one giant topic for ALL notifications — subscribers get overwhelmed with irrelevant messages
→ Never skip subscription confirmation — unconfirmed subscriptions receive nothing and auto-delete
→ Never allow sns:Publish to a topic without conditions — anyone who knows the ARN could publish
→ Never rely solely on SMS for critical alerts — SMS has no retry and can fail silently
→ Never use Promotional SMS type for critical operational alerts — lower delivery priority
→ Never assume message order in Standard topics — messages may arrive out of order
→ Never publish sensitive data (passwords, PII) in SNS messages without encryption
→ Never ignore the SNS SMS sandbox for new accounts — messages won't deliver to unverified numbers
→ Never forget that SNS is REGIONAL — a topic in ap-south-1 cannot be subscribed to from us-east-1 directly
→ Never use SNS for large payloads — maximum message size is 256 KB
→ Never assume email delivery is instant — email goes through standard email infrastructure with possible delays

🔴 WHEN NOT TO USE

→ Do NOT use SNS for large file transfer → messages max 256 KB → Use S3 with presigned URLs
→ Do NOT use SNS for guaranteed message ordering → Standard topics don't guarantee order → Use SQS FIFO
→ Do NOT use SNS for point-to-point messaging (one sender → one receiver) → Use SQS directly
→ Do NOT use SNS for real-time streaming data → Use Kinesis (Project 52)
→ Do NOT use SNS for complex event processing / routing → Use EventBridge (Project 38)
→ Do NOT use SNS for workflow orchestration → Use Step Functions (Project 32)
→ Do NOT use SNS when you need custom sender email addresses → Use SES (Simple Email Service)
→ Do NOT use SNS for chat/interactive messaging → Use Amazon Chime or third-party services

2 → WHAT HAPPENS WHEN A MESSAGE IS PUBLISHED

Publisher calls sns:Publish API
        │
        ▼
Step 1: SNS authenticates the request (IAM — Project 2)
        │
        ▼
Step 2: SNS checks Topic Access Policy
        Does the publisher have permission to publish to this topic?
        │
        ├── IAM policy says Allow? AND Topic policy doesn't Deny? → Proceed
        │
        └── Denied → Return 403 error
        │
        ▼
Step 3: Message stored redundantly across multiple AZs
        (ensures message is not lost if one AZ fails)
        │
        ▼
Step 4: SNS retrieves ALL confirmed subscriptions for the topic
        │
        ▼
Step 5: For EACH subscription:
        │
        ├── Step 5a: Evaluate filter policy (if any)
        │   Does message attributes match filter? 
        │   ├── YES → Proceed to delivery
        │   └── NO → Skip this subscription (message filtered out)
        │
        ├── Step 5b: Format message for the protocol
        │   Email → Subject + Body + Unsubscribe link
        │   SMS → Truncated to 160 chars (or multi-part for longer)
        │   SQS → Full JSON envelope
        │   Lambda → Full JSON envelope (invoked asynchronously)
        │   HTTP → Full JSON POST to endpoint URL
        │
        └── Step 5c: Deliver to endpoint
            │
            ├── Success → Log delivery status (if enabled)
            │
            └── Failure → Retry based on protocol policy
                │
                ├── Retries exhausted → Send to DLQ (if configured)
                │
                └── No DLQ → Message LOST for this subscription
        │
        ▼
Step 6: Return MessageId to publisher
        (Publisher gets MessageId BEFORE all deliveries complete)
        (Publishing is ASYNCHRONOUS — publisher doesn't wait for all deliveries)

→ KEY: Publishing is ASYNCHRONOUS
→ The Publish API returns immediately after the message is stored
→ Actual delivery to subscribers happens in the background
→ Publisher does NOT know if all subscribers received the message

3 → HOW FILTER POLICIES WORK INTERNALLY

Message arrives with attributes:
  severity = "CRITICAL"
  service = "web-server"

Subscription filter policy:
  {"severity": ["CRITICAL", "WARNING"]}

Evaluation:
  1. Message has attribute "severity"? → YES
  2. Value "CRITICAL" in allowed list ["CRITICAL", "WARNING"]? → YES
  3. All filter conditions satisfied? → YES
  4. RESULT: DELIVER

Another subscription filter:
  {"severity": ["INFO"], "service": ["batch-processor"]}

Evaluation:
  1. Message has "severity"? → YES. Value "CRITICAL" in ["INFO"]? → NO
  2. RESULT: DO NOT DELIVER (first condition failed — short-circuit)

→ Filter evaluation happens INSIDE the SNS service — BEFORE delivery
→ Filtered-out messages are NOT counted as delivery failures
→ Filtered-out messages are NOT sent to DLQ
→ They simply don't match → SNS skips that subscription silently

4 → HOW EMAIL DELIVERY WORKS

SNS decides to deliver to email subscription
        │
        ▼
Step 1: SNS formats the email
        From: no-reply@sns.amazonaws.com
        To: ops-team@shopnow.com
        Subject: [from Publish call]
        Body: [message + unsubscribe link]
        │
        ▼
Step 2: SNS sends via AWS's internal SMTP infrastructure
        │
        ▼
Step 3: Email enters standard internet email routing
        SNS → AWS SMTP → Recipient's MX server → Recipient's inbox
        │
        ▼
Step 4: Email may be:
        ├── Delivered to inbox ✓
        ├── Delivered to spam/junk folder (common with automated emails)
        ├── Bounced (invalid email address)
        └── Silently dropped by recipient's email server

→ SNS has NO visibility into whether the email actually reached the inbox
→ "Delivery status logging" for email shows SNS sent it — not that recipient read it
→ Email delivery is BEST-EFFORT — no retries, no DLQ support
→ For reliable email delivery → use Amazon SES (Simple Email Service) with bounce/complaint handling

5 → HOW SMS DELIVERY WORKS

SNS decides to deliver SMS
        │
        ▼
Step 1: SNS determines message type (Transactional/Promotional)
        │
        ▼
Step 2: SNS routes through carrier aggregator
        │
        ├── Transactional: Higher priority route → faster delivery
        └── Promotional: Standard route → may be slower, may be filtered
        │
        ▼
Step 3: Carrier aggregator → Mobile carrier → Recipient's phone
        │
        ▼
Step 4: SMS may be:
        ├── Delivered ✓
        ├── Failed (phone off, invalid number)
        ├── Blocked by carrier (DND — Do Not Disturb registry in India)
        └── Delayed (carrier congestion)

→ India has DND (Do Not Disturb) regulations
→ Promotional SMS to DND-registered numbers are BLOCKED by carriers
→ Transactional SMS can bypass DND (for OTPs, alerts) but requires proper sender ID registration
→ SMS delivery rates vary by country and carrier

📈 SCALABILITY CONSIDERATIONS

→ Message Throughput

  • Standard topics: Virtually unlimited publishing rate
  • FIFO topics: 300 msg/sec (3,000 with batching) — significant limitation
  • Fan-out to millions of subscribers is supported
  • SNS scales automatically — no capacity planning needed

→ Subscription Scalability

  • 12.5 million subscriptions per topic
  • Filter policies allow selective delivery without creating multiple topics
  • For very different subscriber groups → use separate topics (cleaner than complex filters)

→ Cost Scalability

  • Email/HTTP are extremely cheap → scale freely
  • SMS costs scale linearly with volume → budget carefully for large-scale SMS
  • SQS/Lambda deliveries add minimal SNS cost → most cost is on the SQS/Lambda side

→ Multi-Region Scalability

  • Topics are regional → create topics in each region where publishers exist
  • Cross-region: Publisher in us-east-1 can publish to topic in ap-south-1 (higher latency)
  • For global architectures → consider EventBridge global endpoints

→ Operational Scalability

  • Hundreds of topics per account are manageable
  • Use naming conventions: {team}-{env}-{purpose}
  • Use tags for cost allocation and organizational management
  • Infrastructure as Code (CloudFormation/Terraform) for managing topics at scale

        Check subscription filter policy
        Compare with message attributes used during publish
        If mismatch → message was filtered → not a bug

STEP 3: Did SNS attempt delivery?
        Check delivery status logs in CloudWatch
        If no log entry → message was filtered or subscription wasn't matched

STEP 4: Did delivery fail?
        Check CloudWatch delivery status for failures
        Check DLQ (if configured) for failed messages

STEP 5: For email — check spam/junk folder
        AWS SNS emails are often flagged as spam

STEP 6: For SMS — check SMS delivery logs
        SNS Console → Mobile → Text messaging → Statistics
        Check for failures by country/carrier

STEP 7: For SQS — check the SQS queue
        SQS Console → Select queue → Check "Messages available"
        If message is in queue → SNS delivered successfully → issue is on consumer side

STEP 8: For cross-service publishing (S3, CloudWatch)
        Check topic access policy → Verify service principal and conditions
        Check CloudTrail for SNS:Publish calls → Look for errors

🎯 KEY TAKEAWAY

→ SNS is the PUB/SUB backbone of AWS — one message published, many subscribers notified simultaneously
→ Topics are the central routing point — publishers and subscribers are completely decoupled
→ Subscription confirmation is MANDATORY for Email/HTTP — unconfirmed = no delivery
→ Filter policies enable surgical message delivery without creating many topics
→ Topic access policies control which AWS services and accounts can publish
→ SNS is asynchronous — publisher doesn't wait for delivery, gets MessageId immediately
→ Standard topics have virtually unlimited throughput but no ordering guarantee
→ FIFO topics guarantee ordering but are limited to 300 msg/sec

🔗 BRIDGE TO NEXT PROJECT

→ ShopNow now has SNS topics for notifications
→ But the catalog team says: "We don't just want to be NOTIFIED about new images — we want to PROCESS them"
→ When a notification comes in — they need to:

  • Resize the image
  • Generate a thumbnail
  • Update the product catalog database
    → But what if 1,000 images are uploaded simultaneously?
  • The processing system can only handle 10 images per second
  • The remaining 990 messages need to WAIT in line
  • If the processor crashes — the messages should NOT be lost
  • When the processor restarts — it should pick up where it left off

→ This is NOT a notification problem — this is a QUEUING problem
→ SNS says "here's a message — deal with it NOW"
→ What they need is: "here's a message — hold it until I'm READY to process it"

→ This is exactly what Project 7 solves
→ Project 7 introduces SQS (Simple Queue Service) — the message queuing service
→ SQS holds messages until a consumer is ready to process them
→ It provides durability (messages survive consumer crashes), ordering (FIFO), and backpressure (queue absorbs spikes)
→ Together, SNS + SQS (Project 13) form the most powerful messaging pattern in AWS
