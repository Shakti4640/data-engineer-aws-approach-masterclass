→ Problem 1: ALL S3 Traffic Flows Through NAT Gateway

  • EC2 instances in private subnets access S3 via NAT Gateway → Internet Gateway → S3 public endpoint
  • Glue ETL jobs (Project 24) write Parquet to S3 via NAT Gateway
  • Redshift COPY (Project 22) and UNLOAD (Project 23) access S3 via NAT Gateway
  • CloudWatch Agent (Project 19) sends logs via NAT Gateway

→ NAT Gateway data transfer costs:

Glue ETL (6 jobs)15 GB write450 GB$20.25
Redshift COPY10 GB read300 GB$13.50

→ Plus NAT Gateway hourly cost: 2 NAT Gateways × $0.045/hour × 730 hours = $65.70/month
→ Total NAT Gateway cost: $114.30/month — and 90%+ of this traffic is S3

→ Problem 2: Security Concern — S3 Traffic Traverses Public Internet

  • Data flows: Private subnet → NAT Gateway → Internet Gateway → PUBLIC internet → S3 endpoint
  • Even though HTTPS encrypts the data, the TRAFFIC PATH crosses the public internet
  • The security team flags this: "Customer data should NEVER traverse the public internet"
  • Compliance requirements (PCI-DSS, HIPAA) may require private-only data paths
  • If NAT Gateway fails → ALL S3 access fails → entire pipeline breaks

→ Problem 3: Lambda Cannot Reach Private Resources AND S3 Simultaneously

  • Lambda order-fulfillment-processor (Project 26) needs to:
    1. Read messages from SQS (AWS service — public endpoint)
    2. Check inventory in MariaDB (private subnet — VPC resource)
    3. Write fulfillment records to S3 (AWS service — public endpoint)
  • Without VPC configuration → Lambda can reach SQS and S3 but NOT MariaDB
  • With VPC configuration → Lambda gets ENI in private subnet → can reach MariaDB but LOSES internet access
  • Without internet access → Lambda cannot reach S3 or SQS
  • This is the "Lambda VPC networking dilemma" (deep dive in Project 45)

→ Problem 4: Glue ENI S3 Access Failure

  • In Project 24, we noted: Glue ENIs in private subnets need S3 access to write output
  • Currently relies on NAT Gateway for this access
  • If NAT Gateway is removed (cost optimization) → Glue JDBC jobs fail on S3 write
  • Need a NAT-independent path to S3

→ Problem 5: DynamoDB Access from Private Subnets

  • ShopNow recently added a DynamoDB table for session management (caching user sessions)
  • EC2 application in private subnet needs to read/write DynamoDB
  • DynamoDB is a public AWS service — same NAT Gateway dependency as S3
  • Additional traffic flowing through NAT → additional costs

→ The CTO and CISO jointly demand: "All AWS service traffic from private subnets must use PRIVATE network paths — no public internet, no NAT Gateway dependency for AWS services, and reduce our networking costs"

🟡 HOW TO APPROACH IT

→ First — understand the networking problem clearly

Current traffic path (Private Subnet → S3):

EC2 in Private Subnet
  → Private subnet route table → NAT Gateway (public subnet)
    → Internet Gateway
      → PUBLIC INTERNET
        → S3 public endpoint (s3.amazonaws.com)
          → Response travels back the same path

→ Problems with this path:

  1. Data crosses the public internet (security concern)
  2. NAT Gateway charges $0.045/GB for data processing (cost concern)
  3. NAT Gateway is a single point of failure per AZ (reliability concern)
  4. NAT Gateway has bandwidth limit of 45 Gbps (scalability concern for large transfers)
  5. Added latency from internet routing (performance concern)

Desired traffic path (Private Subnet → S3 via VPC Endpoint):

EC2 in Private Subnet
  → Private subnet route table → VPC Endpoint (within AWS network)
    → S3 (NEVER leaves AWS backbone network)
      → Response travels back the same private path

→ Benefits of this path:

  1. Data NEVER leaves AWS private network (security ✅)
  2. NO data processing charges (cost ✅)
  3. Highly available — managed by AWS (reliability ✅)
  4. AWS backbone bandwidth — no NAT bottleneck (scalability ✅)
  5. Lower latency — direct AWS network path (performance ✅)

→ AWS provides two types of VPC Endpoints to achieve this:

Endpoint TypeSupported ServicesMechanismCost
Gateway EndpointS3, DynamoDB (only these two)Route table entryFREE
Interface Endpoint100+ AWS services (SQS, SNS, KMS, CloudWatch, Glue, etc.)ENI with private IP$0.01/hour + $0.01/GB

→ The approach should be:

  • Step 1 — Understand Gateway Endpoints vs Interface Endpoints deeply
  • Step 2 — Create S3 Gateway Endpoint
  • Step 3 — Create DynamoDB Gateway Endpoint
  • Step 4 — Configure Endpoint Policies (security controls)
  • Step 5 — Verify traffic routing
  • Step 6 — Evaluate Interface Endpoints for other services
  • Step 7 — Assess NAT Gateway optimization after endpoints are configured
  • Step 8 — Integrate with existing services (Glue, Redshift, Lambda)

🟢 HOW CAN WE SOLVE IT — Step-by-Step Configuration

What happens after creation:

→ AWS automatically adds a route to each selected route table:

DestinationTargetDescription
pl-78a54011 (S3 prefix list)vpce-0abc123 (endpoint ID)S3 traffic routed to Gateway Endpoint

→ pl-78a54011 is a "prefix list" — a managed list of S3's IP ranges for this region
→ AWS maintains this list — you never need to update it
→ When S3 adds new IP ranges, the prefix list auto-updates

Before Gateway Endpoint (route table):

DestinationTargetNotes
10.0.0.0/16localVPC internal traffic
0.0.0.0/0nat-gw-xxxAll internet traffic → NAT Gateway

→ S3 traffic matches 0.0.0.0/0 → goes to NAT Gateway → internet → S3

After Gateway Endpoint (route table):

DestinationTargetNotes
10.0.0.0/16localVPC internal traffic
pl-78a54011vpce-0abc123S3 traffic → Gateway Endpoint (private)
0.0.0.0/0nat-gw-xxxAll OTHER internet traffic → NAT Gateway

→ S3 traffic matches the more specific S3 prefix list → goes to Gateway Endpoint → AWS backbone → S3
→ Route table precedence: most specific route wins
→ S3 prefix list is more specific than 0.0.0.0/0 → S3 traffic ALWAYS uses the endpoint

Immediate effects (no code changes, no application restart):

→ Applications still use s3.amazonaws.com — DNS resolution doesn't change
→ Only the NETWORK PATH changes — from internet to private backbone

STEP 4 — CONFIGURE ENDPOINT POLICIES

→ Endpoint Policies control WHAT operations are allowed THROUGH the endpoint
→ They are an ADDITIONAL layer of security on top of IAM policies

Default Endpoint Policy: Full Access

→ By default, the endpoint allows ALL S3 operations to ALL buckets
→ Any IAM principal in the VPC that has S3 permissions can access any bucket through the endpoint

Custom Endpoint Policy: Restrict to Specific Buckets

→ Navigate to VPC → Endpoints → Select S3 endpoint → Edit Policy

Restrictive Policy for ShopNow:

ElementValuePurpose
EffectAllowGrant access
Principal*Any principal (IAM policies still apply)
Actions3:GetObject, s3:PutObject, s3:ListBucket, s3:DeleteObjectAllowed S3 operations
Resourcearn:aws:s3:::shopnow-data-lake, arn:aws:s3:::shopnow-data-lake/*Only ShopNow's data lake bucket
Resourcearn:aws:s3:::shopnow-fulfillment-records, arn:aws:s3:::shopnow-fulfillment-records/*Fulfillment bucket
Resourcearn:aws:s3:::shopnow-glue-scripts, arn:aws:s3:::shopnow-glue-scripts/*Glue scripts bucket
Resourcearn:aws:s3:::shopnow-athena-results, arn:aws:s3:::shopnow-athena-results/*Athena results bucket

→ Now even if an EC2 instance's IAM role has s3:* permissions on ALL buckets:

  • Through this endpoint → can ONLY access the 4 specified buckets
  • Direct internet access (if available) → full IAM permissions apply
  • This prevents data exfiltration to unauthorized S3 buckets

Why Endpoint Policies matter for security:

ScenarioWithout Endpoint PolicyWith Endpoint Policy
Compromised EC2 instance tries to copy data to attacker's S3 bucketSucceeds (if IAM allows s3:PutObject on *)BLOCKED — attacker's bucket not in endpoint policy
Insider threat copies company data to personal S3 bucketSucceedsBLOCKED
Legitimate access to company bucketsSucceedsSucceeds ✅

→ Endpoint Policy + IAM Policy = defense in depth
→ BOTH must allow the action for it to succeed:

Request: EC2 wants to PUT object to s3://shopnow-data-lake/file.parquet

IAM Policy check: Does the EC2 role allow s3:PutObject on shopnow-data-lake/*?
  → YES ✅

Endpoint Policy check: Does the endpoint allow s3:PutObject on shopnow-data-lake/*?
  → YES ✅

Result: ALLOWED ✅

---

Request: EC2 wants to PUT object to s3://attacker-bucket/stolen-data.csv

IAM Policy check: Does the EC2 role allow s3:PutObject on attacker-bucket/*?
  → YES (if role has s3:PutObject on *)

Endpoint Policy check: Does the endpoint allow s3:PutObject on attacker-bucket/*?
  → NO ❌ (bucket not in endpoint policy)

Result: DENIED ❌

→ Always configure Endpoint Policies to restrict access to authorized buckets/tables — defense in depth
→ Always enable Private DNS for Interface Endpoints — eliminates code changes
→ Always verify DNS settings on VPC (enableDnsHostnames and enableDnsSupport = true) before creating Interface Endpoints

❌ WHAT SHOULD NOT DO

→ Never forget to associate Gateway Endpoints with route tables — endpoint exists but no traffic routes to it

→ Never disable Private DNS on Interface Endpoints unless you have specific DNS requirements
→ Never place Interface Endpoint in only one AZ — single AZ failure = service disruption
→ Never create Gateway Endpoint without testing — verify S3 access still works after creation
→ Never assume endpoint policies replace IAM policies — both must allow the action
→ Never confuse Gateway Endpoints with Interface Endpoints — they work completely differently

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

1 → GATEWAY ENDPOINT ROUTING INTERNALS

EC2 instance makes S3 API call: GET s3://shopnow-data-lake/transformed/orders/file.parquet
       ↓
AWS SDK resolves DNS: s3.ap-south-1.amazonaws.com → 52.219.64.xxx (S3 public IP)
       ↓
EC2 creates TCP packet:
  Source: 10.0.1.50 (EC2 private IP)
  Destination: 52.219.64.xxx (S3 public IP)
       ↓
Packet enters VPC router
       ↓
Router checks route table for Private Subnet A:
  ├── 10.0.0.0/16 → local                  (no match — destination is public IP)
  ├── pl-78a54011 → vpce-0abc123 (S3 endpoint)  
  │     ├── Does 52.219.64.xxx match S3 prefix list? 
  │     └── YES ✅ (AWS maintains S3 IP list in the prefix list)
  └── 0.0.0.0/0 → nat-gw-xxx              (would match — but less specific)
       ↓
Most specific match wins: pl-78a54011 → vpce-0abc123
       ↓
Packet routed to S3 Gateway Endpoint
       ↓
Gateway Endpoint:
  ├── Checks Endpoint Policy: Is this action allowed? ✅
  ├── Forwards packet to S3 via AWS backbone network
  ├── NEVER touches the public internet
  └── Source IP in S3 logs: 10.0.1.50 (EC2 private IP!)
       ↓
S3 processes request:
  ├── Checks IAM authentication (SigV4 signature)
  ├── Checks bucket policy
  ├── Returns response
  └── Response routes back through Gateway Endpoint → EC2

→ Key insight: DNS still resolves to S3 PUBLIC IPs
→ The Gateway Endpoint intercepts the traffic at the ROUTE TABLE level
→ The packet's destination IP is a public S3 IP — but it never reaches the internet
→ This is why no DNS changes or code changes are needed

Source IP preservation:
→ When traffic goes through NAT Gateway → S3 sees NAT Gateway's public IP as source
→ When traffic goes through Gateway Endpoint → S3 sees the EC2's PRIVATE IP as source
→ This matters for S3 bucket policies that restrict by source IP
→ If your bucket policy allows "aws:SourceIp": "52.xx.xx.xx" (NAT Gateway IP):

  • After switching to Gateway Endpoint → access DENIED (source IP changed!)
  • Update bucket policy to use "aws:sourceVpce": "vpce-0abc123" instead

2 → INTERFACE ENDPOINT (PRIVATELINK) INTERNALS

EC2 instance makes KMS API call: kms.ap-south-1.amazonaws.com
       ↓
DNS Resolution (with Private DNS enabled):
  ├── VPC DNS resolver intercepts the query
  ├── Checks: Is there an Interface Endpoint for kms.ap-south-1.amazonaws.com?
  ├── YES → Returns private IP: 10.0.1.200 (Interface Endpoint ENI IP)
  └── Instead of returning KMS public IP
       ↓
EC2 creates TCP packet:
  Source: 10.0.1.50 (EC2 private IP)
  Destination: 10.0.1.200 (Interface Endpoint private IP)
       ↓
Packet enters VPC router
       ↓
Router checks route table:
  ├── 10.0.0.0/16 → local ✅ (destination is within VPC CIDR!)
  └── Packet stays within VPC
       ↓
Packet reaches Interface Endpoint ENI (10.0.1.200)
       ↓
ENI Security Group check:
  ├── Inbound rule: Allow HTTPS (443) from 10.0.0.0/16? ✅
  └── Packet accepted
       ↓
Interface Endpoint:
  ├── Checks Endpoint Policy: Is this action allowed? ✅
  ├── Forwards to KMS via AWS PrivateLink (private backbone)
  └── NEVER touches the public internet
       ↓
KMS processes request → response returns same private path

→ Key difference from Gateway: Interface Endpoints work via DNS manipulation
→ DNS resolves to a PRIVATE IP instead of public IP
→ Traffic is VPC-internal (10.x.x.x → 10.x.x.x) → never needs route to internet
→ This is why Interface Endpoints work even WITHOUT NAT Gateway or Internet Gateway

3 → ENDPOINT POLICY EVALUATION FLOW

Request: S3 PutObject to shopnow-data-lake

STEP 1: IAM Authentication
  ├── Is the request signed with valid credentials?
  └── YES → Continue

STEP 2: IAM Policy Evaluation
  ├── Does the caller's IAM policy allow s3:PutObject on this resource?
  ├── Is there an explicit DENY? → DENIED
  ├── Is there an explicit ALLOW? → Continue to next check
  └── No matching statement? → DENIED (implicit deny)

STEP 3: VPC Endpoint Policy Evaluation (if request comes through endpoint)
  ├── Does the endpoint policy allow s3:PutObject on this resource?
  ├── Is there an explicit DENY? → DENIED
  ├── Is there an explicit ALLOW? → Continue to next check
  └── No matching statement? → DENIED (implicit deny)

STEP 4: S3 Bucket Policy Evaluation
  ├── Does the bucket policy allow this action?
  ├── Can include VPC Endpoint conditions:
  │     ├── "aws:sourceVpce": "vpce-0abc123" (specific endpoint)
  │     └── "aws:sourceVpc": "vpc-0abc123" (any endpoint in this VPC)
  └── Evaluate as per standard bucket policy logic

STEP 5: All checks passed → REQUEST ALLOWED ✅

→ Three layers of policy evaluation for endpoint-routed traffic:

  1. IAM Policy (identity-based)
  2. Endpoint Policy (network path-based)
  3. Resource Policy — bucket policy (resource-based)
    → ALL THREE must allow → any deny = request denied
    → This is powerful defense in depth

5 → PREFIX LIST INTERNALS

→ The S3 Gateway Endpoint uses a "prefix list" in the route table
→ A prefix list is a managed collection of CIDR blocks

Prefix List: pl-78a54011 (S3 in ap-south-1)
  ├── 52.219.64.0/21    (S3 IP range 1)
  ├── 52.219.72.0/22    (S3 IP range 2)
  ├── 52.219.76.0/22    (S3 IP range 3)
  ├── 3.5.164.0/22      (S3 IP range 4)
  └── ... (50+ CIDR blocks)

→ AWS AUTOMATICALLY updates this list when S3 adds new IP ranges
→ You never need to manage these IPs
→ The route table entry references the prefix list ID, not individual IPs
→ When evaluating routing → VPC router expands the prefix list and matches against destination IP

Viewing prefix lists:
→ Navigate to VPC → Managed Prefix Lists
→ Or: VPC → Route Tables → Select table → Routes tab → see pl-xxxxx entries

6 → HOW GATEWAY ENDPOINTS HANDLE MULTI-AZ

VPC with 2 AZs:

AZ-a:
  Private Subnet A (route table: rtb-private-a)
    → S3 prefix list → vpce-0abc123 (Gateway Endpoint)
    
AZ-b:
  Private Subnet B (route table: rtb-private-b)
    → S3 prefix list → vpce-0abc123 (SAME Gateway Endpoint)

→ Gateway Endpoint is NOT per-AZ — it's per-VPC
→ A single Gateway Endpoint serves ALL subnets in the VPC (that have the route)
→ The endpoint itself is highly available across all AZs
→ If one AZ has an outage → endpoint still works for other AZs
→ This is different from Interface Endpoints which ARE per-AZ (one ENI per AZ)
