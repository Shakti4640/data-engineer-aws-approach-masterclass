PROJECT 23: THE ANALYTICS EXPORT NOBODY COULD AUTOMATE — Redshift UNLOAD to S3

🔴 HYPOTHETICAL PROBLEM

→ ShopNow's Redshift cluster (Project 16) is now loaded with optimized data via COPY (Project 22)
→ The warehouse contains orders, customers, products, and clickstream data
→ Analysts run complex queries daily — joins, window functions, aggregations
→ But now multiple teams need query RESULTS outside of Redshift:

→ Problem 1: Marketing Team — External Partner Export

  • Marketing runs a customer segmentation query in Redshift
  • Result: 2 million rows identifying high-value customers with purchase patterns
  • They need this as a CSV file in S3 to share with an external advertising platform
  • Currently: An analyst runs the query → copies results from Query Editor → pastes into Excel → uploads to S3
  • This manual process takes 45 minutes and fails for results > 100,000 rows (browser memory limit)

→ Problem 2: Finance Team — Monthly Archive

  • Finance needs monthly revenue aggregations exported to S3 for long-term archival (7 years — compliance)
  • Result: Aggregated revenue by region, product category, and payment method per month
  • Currently: No export process exists — data lives only in Redshift
  • If the Redshift cluster is decommissioned or resized → historical aggregations are lost

→ Problem 3: Data Science Team — ML Training Data

  • Data scientists need feature-engineered data from Redshift for ML model training in SageMaker
  • Result: 50 million rows with 30 computed features (rolling averages, ratios, ranks)
  • They need this in Parquet format in S3 (SageMaker reads from S3)
  • Currently: They query Redshift from a notebook → download results → re-upload to S3
  • This fails for 50M rows — the notebook runs out of memory

→ Problem 4: Data Lake Feedback Loop

  • The data engineering team wants to write Redshift-computed aggregations BACK to the data lake in S3
  • These aggregations become the "curated" layer (referenced in Project 21's data lake zones)
  • Other teams query these curated datasets via Athena (Project 18) without needing Redshift access
  • Currently: No mechanism to push data from Redshift to S3

→ Common Thread Across All Problems:

  • Extracting large result sets from Redshift to S3
  • Manual methods don't scale beyond a few thousand rows
  • Different teams need different formats (CSV, Parquet)
  • Some need partitioned output, some need single files
  • All need automation — not manual analyst effort

→ The data engineering lead says: "We need the reverse of COPY — a way to parallel-export query results from Redshift to S3"

🟡 HOW TO APPROACH IT

→ First — understand why naive export methods fail

MethodMechanismLimitProblem
Query Editor copy-pasteBrowser clipboard~100K rowsBrowser crashes on large results
Application SELECT → write to file → upload S3Single connection, row-by-row~1M rowsMemory exhaustion, hours of runtime
JDBC fetch → Python script → S3 uploadSingle thread pull~5M rowsExtremely slow, single-threaded bottleneck
UNLOAD commandAll compute nodes write to S3 in PARALLELBillions of rowsNone — purpose-built for this

→ UNLOAD is the mirror image of COPY (Project 22):

  • COPY: S3 → Parallel Read → Compute Nodes → Redshift Tables
  • UNLOAD: Redshift Query → Compute Nodes → Parallel Write → S3

→ Just as COPY leverages all compute nodes to READ from S3
→ UNLOAD leverages all compute nodes to WRITE to S3

→ The approach should be:

  • Step 1 — Understand UNLOAD architecture (how parallel writes work)
  • Step 2 — Configure IAM Role for Redshift to WRITE to S3 (extends Project 22 role)
  • Step 3 — Execute basic UNLOAD for each use case
  • Step 4 — Configure format options (CSV, Parquet, JSON)
  • Step 5 — Configure partitioning for data lake integration
  • Step 6 — Configure compression and encryption
  • Step 7 — Handle overwrite and append scenarios
  • Step 8 — Verify output and integrate with downstream services
  • Step 9 — Automate UNLOAD operations

🟢 HOW CAN WE SOLVE IT — Step-by-Step Configuration

STEP 1 — UNDERSTAND UNLOAD ARCHITECTURE

→ When UNLOAD executes, this is what happens internally:

UNLOAD command received by Leader Node
       ↓
Leader parses the embedded SELECT query
       ↓
Leader creates a distributed execution plan for the query
       ↓
Compute nodes execute the query in parallel
  ├── Node 1: Processes its portion of the data
  ├── Node 2: Processes its portion of the data
  └── Node 3: Processes its portion of the data
       ↓
Each node's results are formatted (CSV/Parquet/JSON)
       ↓
Each node writes its results DIRECTLY to S3
  ├── Node 1 → s3://bucket/prefix/0000_part_00
  ├── Node 1 → s3://bucket/prefix/0001_part_00
  ├── Node 2 → s3://bucket/prefix/0000_part_01
  ├── Node 2 → s3://bucket/prefix/0001_part_01
  └── Node 3 → s3://bucket/prefix/0000_part_02
       ↓
Leader confirms all writes complete
       ↓
UNLOAD returns success with row count

→ Key insight: Each compute node writes its own files to S3 simultaneously
→ A 4-node cluster writes 4+ files in parallel → massive throughput
→ Each slice within each node can write its own file → even more parallelism

Default output file naming:

s3://bucket/prefix/
  ├── 0000_part_00.csv       (Slice 0, Part 0)
  ├── 0001_part_00.csv       (Slice 0, Part 1)
  ├── 0000_part_01.csv       (Slice 1, Part 0)
  ├── 0001_part_01.csv       (Slice 1, Part 1)
  ├── 0000_part_02.csv       (Slice 2, Part 0)
  └── 0001_part_02.csv       (Slice 2, Part 1)

→ File count = number of slices × number of parts per slice
→ Typical: 4-16 files for a small-medium cluster
→ You can control this with the PARALLEL parameter (Step 3)

Partition columns are NOT included in the Parquet file dataThey exist only in the S3 path

→ PARTITION BY + Parquet format + UNLOAD = Redshift becomes a data lake WRITER
→ This completes the bidirectional flow:

  • S3 → COPY → Redshift (Project 22)
  • Redshift → UNLOAD → S3 (this project)

STEP 9 — HANDLE OVERWRITE AND APPEND SCENARIOS

→ Critical: By default, UNLOAD REFUSES to write to a path that already contains files

ALLOWOVERWRITE Parameter:

SettingBehavior
Without ALLOWOVERWRITEUNLOAD fails if ANY file exists at the target path
With ALLOWOVERWRITEUNLOAD overwrites existing files with same names

→ Important nuance: ALLOWOVERWRITE overwrites files with matching names
→ If previous run created files 0000_part_00 through 0007_part_03 (8 files)
→ And new run creates only 0000_part_00 through 0003_part_01 (4 files)
→ Files 0004_part_02 through 0007_part_03 from the PREVIOUS run STILL EXIST
→ You now have STALE data mixed with fresh data

Safe overwrite pattern:

Step 1: Delete all existing files in the target path
Step 2: Run UNLOAD with ALLOWOVERWRITE
Step 3: Verify output

→ Or use date-based paths that are unique per run:

  • s3://bucket/exports/marketing/segments/run_date=2025-01-15/
  • Each run writes to a new path → no overwrite concern
  • Old runs are preserved for comparison/audit

CLEANPATH Parameter (Newer Redshift versions):

SettingBehavior
CLEANPATHDeletes ALL existing files in the target path BEFORE writing

→ This is the safest option for "replace all" semantics
→ Atomically cleans old files and writes new ones
→ Preferred over manual delete + ALLOWOVERWRITE

Append pattern (adding to existing data):
→ Use different prefixes per run:

  • Run 1: s3://bucket/curated/daily-summary/run=2025-01-15/
  • Run 2: s3://bucket/curated/daily-summary/run=2025-01-16/
    → Downstream consumers (Athena, Glue) read ALL prefixes as a single dataset

✅ WHAT SHOULD DO

→ Always use UNLOAD for exporting large result sets — never manual copy-paste or row-by-row fetch
→ Always use PARQUET format for data lake integration — columnar, compressed, schema-embedded
→ Always use CSV format for external sharing with non-technical consumers
→ Always use PARTITION BY for data lake curated layer — enables Athena partition pruning
→ Always use MANIFEST VERBOSE for production exports — enables validation and auditability
→ Always use CLEANPATH (or manual delete) before overwriting — prevents stale file contamination
→ Always use ADDQUOTES and ESCAPE for CSV with text fields — prevents parsing errors
→ Always use date-based or version-based paths — enables historical comparison and rollback
→ Always encrypt sensitive exports with SSE-KMS — especially finance and PII data
→ Always verify row counts after UNLOAD — compare with source query count
→ Always set MAXFILESIZE to 128-256 MB for data lake exports — optimal for Athena
→ Always run Glue Crawler on UNLOAD output paths to register in catalog
→ Always align UNLOAD schedule AFTER COPY/ETL completes — ensure data is fresh

❌ WHAT SHOULD NOT DO

→ Never use PARALLEL OFF for large exports (> 5M rows) — leader node becomes bottleneck
→ Never forget ALLOWOVERWRITE or CLEANPATH — UNLOAD fails silently on existing paths
→ Never use UNLOAD without checking the output — verify file count, size, and content
→ Never export Parquet with GZIP parameter — Parquet has built-in Snappy; external compression not supported
→ Never export to a path that mixes files from different queries — downstream consumers read ALL files
→ Never skip MANIFEST for production exports — you lose auditability
→ Never use UNLOAD as a real-time data pipeline — it's batch-oriented; use Kinesis for streaming
→ Never give Redshift write access to raw/transformed S3 zones — UNLOAD should only write to exports/curated
→ Never hardcode dates in UNLOAD paths for scheduled queries — use date functions for dynamic paths
→ Never ignore file count in PARALLEL ON mode — too many tiny files degrades downstream performance
→ Never UNLOAD during peak query hours without resource management — UNLOAD consumes WLM queue slots

Export < 100 rows for quick inspection❌ NoJust use Query Editor → download results
Real-time data streaming to S3❌ NoUse Kinesis Firehose
Continuous replication to S3❌ NoUse Redshift Streaming Ingestion or CDC
Export to a database (not S3)❌ NoUse Glue ETL with JDBC (Project 43)

🔴 WHEN NOT TO USE UNLOAD

→ When you need real-time continuous export → Use Kinesis Firehose or change data capture
→ When target is another database (not S3) → Use Glue ETL JDBC connection (Project 43)
→ When result set is tiny (< 100 rows) → Download from Query Editor directly
→ When you want to query S3 data from Redshift WITHOUT loading → Use Spectrum (Project 28) — that's read, not write
→ When you need to export to formats not supported (Avro, ORC) → UNLOAD supports CSV, Parquet, JSON only → use Glue to convert after UNLOAD

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

1 → UNLOAD EXECUTION ENGINE

UNLOAD command parsed by Leader Node
       ↓
Leader compiles the SELECT query into a distributed execution plan
  (same as any regular query — same optimizer, same plan)
       ↓
Compute nodes execute the query:
  ├── Table scans with zone map filtering (SORTKEY benefit)
  ├── Joins with co-located data (DISTKEY benefit)
  ├── Aggregations distributed across nodes
  └── Window functions computed in parallel
       ↓
Query results exist in memory on each compute node
  (Each node holds a PORTION of the result set)
       ↓
FORMATTING PHASE:
  ├── CSV: Convert each row to delimited text
  │     ├── Apply DELIMITER
  │     ├── Apply ADDQUOTES
  │     ├── Apply ESCAPE
  │     ├── Apply NULL AS
  │     └── Apply HEADER (first file only)
  │
  ├── PARQUET: Convert columnar result into Parquet format
  │     ├── Organize by column
  │     ├── Apply Snappy compression per column chunk
  │     ├── Compute column statistics (min/max/null_count)
  │     └── Write file footer with schema
  │
  └── JSON: Convert each row to JSON object
        ├── Key names from column names
        └── One JSON object per line
       ↓
WRITING PHASE:
  ├── PARALLEL ON:
  │     ├── Each slice opens S3 connection
  │     ├── Each slice writes its portion directly
  │     ├── File names: {sequence}_{part}_{slice}.{format}
  │     └── All slices write simultaneously
  │
  └── PARALLEL OFF:
        ├── All compute nodes send results to Leader Node
        ├── Leader aggregates all results
        ├── Leader writes single file to S3
        └── Bottleneck: Leader node network/memory
       ↓
PARTITION BY (if specified):
  ├── Before writing, each node groups its rows by partition values
  ├── Each unique partition value combination gets a separate S3 "directory"
  ├── Files within each partition follow the same PARALLEL rules
  └── Partition columns are REMOVED from the file content (exist only in path)
       ↓
MANIFEST (if specified):
  ├── Leader collects all output file paths from all nodes
  ├── Creates manifest JSON file
  ├── If VERBOSE: includes file sizes and row counts
  └── Writes manifest to S3 at target path + /manifest
       ↓
Leader confirms all S3 writes completed
       ↓
Transaction committed → UNLOAD returns row count

2 → PARALLEL ON: DEEP DIVE INTO FILE DISTRIBUTION

→ With a 4-slice cluster and a query returning 10 million rows:

Slice 0 holds ~2.5M rows of query result
Slice 1 holds ~2.5M rows of query result
Slice 2 holds ~2.5M rows of query result
Slice 3 holds ~2.5M rows of query result

MAXFILESIZE = 256 MB

Slice 0: 2.5M rows = ~600 MB → splits into 3 files (200 MB each)
  ├── 0000_part_00 (200 MB)
  ├── 0001_part_00 (200 MB)
  └── 0002_part_00 (200 MB)

Slice 1: 2.5M rows = ~600 MB → splits into 3 files
  ├── 0000_part_01 (200 MB)
  ├── 0001_part_01 (200 MB)
  └── 0002_part_01 (200 MB)

Slice 2: 2.5M rows = ~600 MB → splits into 3 files
  ...

Slice 3: 2.5M rows = ~600 MB → splits into 3 files
  ...

Total: 12 files × ~200 MB = ~2.4 GB
All 12 files written SIMULTANEOUSLY

→ File naming convention: {sequence_within_slice}_part_{slice_number}
→ Sequence resets for each slice
→ This naming is NOT configurable — Redshift controls it

3 → PARALLEL OFF: WHY IT'S SLOWER

All 4 slices execute query in parallel
  ├── Slice 0: 2.5M rows ready
  ├── Slice 1: 2.5M rows ready
  ├── Slice 2: 2.5M rows ready
  └── Slice 3: 2.5M rows ready
       ↓
All results sent to Leader Node over cluster interconnect
  (10M rows × ~240 bytes/row = ~2.4 GB of data transfer)
       ↓
Leader Node receives all data
  (Must fit in Leader's memory — can be a constraint for very large results)
       ↓
Leader writes single file to S3
  (Single S3 PutObject or multipart upload)
  (Upload speed: ~100 MB/s for single stream)
       ↓
2.4 GB / 100 MB/s = ~24 seconds for S3 write alone
  Plus: data transfer from compute to leader: ~10 seconds
  Plus: formatting overhead: ~5 seconds
  Total: ~40 seconds
       ↓
Compare to PARALLEL ON:
  2.4 GB / 4 parallel streams / 100 MB/s = ~6 seconds for S3 write
  No leader bottleneck
  Total: ~10 seconds

→ PARALLEL OFF is 3-4x slower for this example
→ For 100M+ rows → PARALLEL OFF may run out of Leader memory → job fails
→ Rule: Use PARALLEL OFF only for small exports (< 5M rows) where single file convenience matters

4 → PARTITION BY: INTERNAL ROUTING

Query result row: {order_date: '2025-01-15', region: 'US-West', revenue: 5000}

PARTITION BY (year, month):
  ├── Extract year = 2025
  ├── Extract month = 01
  └── Target path: s3://bucket/prefix/year=2025/month=01/

Slice 0 processes its 2.5M rows:
  ├── Group by (year, month)
  │     ├── (2024, 12): 500K rows → write to year=2024/month=12/0000_part_00
  │     ├── (2025, 01): 2M rows → write to year=2025/month=01/0000_part_00
  │     └── ... other months
  └── Each partition group gets its own output file(s)

Slice 1 does the same independently:
  ├── (2024, 12): 480K rows → write to year=2024/month=12/0000_part_01
  ├── (2025, 01): 2.02M rows → write to year=2025/month=01/0000_part_01
  └── ...

Result: Each partition directory has one file per slice
  year=2025/month=01/
    ├── 0000_part_00 (from slice 0)
    ├── 0000_part_01 (from slice 1)
    ├── 0000_part_02 (from slice 2)
    └── 0000_part_03 (from slice 3)

→ Partition routing happens WITHIN each compute node — no data redistribution between nodes
→ This is why partitioned UNLOAD is nearly as fast as non-partitioned

6 → WLM (WORKLOAD MANAGEMENT) IMPACT

→ UNLOAD runs as a regular query within Redshift's WLM system
→ It consumes a WLM queue slot (just like any SELECT query)

WLM queue behavior:

ScenarioImpact
UNLOAD on small result setMinimal — completes quickly, releases slot
UNLOAD on huge result set (1B rows)Long-running — holds WLM slot for duration
Multiple UNLOADs simultaneouslyEach takes a slot — may exhaust queue capacity
UNLOAD during peak query hoursCompetes with analyst queries for slots

→ Best practice: Create a dedicated WLM queue for UNLOAD operations

WLM QueueConcurrencyMemory %Purpose
Default560%Analyst ad-hoc queries
ETL230%COPY and UNLOAD operations
Superuser110%Administrative tasks

→ Assign UNLOAD queries to the ETL queue via query group or user group
→ This prevents UNLOAD from stealing resources from analyst queries

→ WLM configuration: Redshift Console → Cluster → Properties → Parameter Group → WLM Configuration
→ From Project 16 — parameter groups were introduced; WLM is the most important parameter group setting
