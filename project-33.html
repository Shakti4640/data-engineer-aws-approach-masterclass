PROJECT 33: THE APPLICATION THAT CRASHED EVERY BLACK FRIDAY — EC2 Auto Scaling with CloudWatch Triggers

🔴 HYPOTHETICAL PROBLEM

→ ShopNow's data pipeline (Project 32) runs flawlessly every morning
→ Enriched data flows into Redshift → dashboards are accurate → CEO is happy

→ But the CUSTOMER-FACING side has a catastrophic problem:

The Architecture Today:
→ One EC2 instance (from Project 3) runs the ShopNow web application
→ It sits in a public subnet (from Project 5) behind an Elastic IP
→ It connects to RDS MariaDB (from Project 14) in a private subnet
→ Security Groups (from Project 4) allow HTTP/HTTPS traffic from the internet

What happened on Black Friday 2024:

→ Normal daily traffic: ~500 concurrent users
→ Black Friday 8 AM: Marketing sends email blast → 1,500 users hit the site
→ Black Friday 10 AM: Social media campaign goes viral → 5,000 concurrent users
→ Black Friday 11 AM: Flash sale announcement → 12,000 concurrent users

Business Impact:
→ 3 hours of downtime during peak shopping
→ $200,000 in lost revenue
→ 4,000 customers abandoned carts permanently
→ Social media backlash — "#ShopNowDown" trending
→ Customer trust damaged

→ The CTO mandates: "Build infrastructure that automatically scales from 2 to 20 instances based on demand, distributes traffic, replaces unhealthy instances, and does all of this WITHOUT human intervention"

→ This requires THREE interconnected AWS services:

  1. Launch Template — defines WHAT to launch (instance blueprint)
  2. Application Load Balancer (ALB) — distributes traffic across instances
  3. Auto Scaling Group (ASG) — decides WHEN and HOW MANY instances to run

Understanding Health Check Grace Period:

→ When a new instance launches:

  1. EC2 instance starts (30-60 seconds)
  2. Operating system boots (30-60 seconds)
  3. Application starts (30-120 seconds)
  4. Application becomes ready to serve traffic

→ Total: 2-4 minutes before the instance can pass health checks
→ If grace period is too short (e.g., 60 seconds) → ASG checks health before app is ready → marks as unhealthy → TERMINATES the instance → launches another → same thing → infinite loop

→ Grace period = 300 seconds → ASG waits 5 minutes before starting health checks on new instances
→ This gives the application time to fully start

→ Set this based on your application's actual startup time + buffer
→ If your custom AMI boots and serves in 2 minutes → grace period = 180 seconds (3 minutes)

POLICY 1 — Target Tracking (Primary — Reactive Scaling)

→ In ASG configuration → Scaling Policies → Add Policy

SettingValue
Policy Nameshopnow-cpu-target-tracking
Policy TypeTarget Tracking Scaling
Metric TypeAverage CPU Utilization
Target Value60
Instance warmup300 seconds

What this does internally:

→ ASG continuously monitors average CPU across ALL instances in the group
→ If average CPU > 60% → ASG calculates how many instances needed to bring CPU back to 60%
→ Example: 2 instances at 80% CPU → ASG determines 3 instances would bring average to ~53% → launches 1
→ If average CPU < 60% → ASG calculates if instances can be removed while staying at/below 60%
→ Example: 4 instances at 30% CPU → ASG determines 2 instances would be at ~60% → terminates 2

Understanding Instance Warmup:

→ When a new instance launches → it takes time to reach normal load
→ During warmup period → the new instance's metrics are NOT included in the average calculation
→ Without warmup: New instance has 0% CPU → drags average down → ASG thinks load decreased → terminates another instance → load spikes again → oscillation
→ With warmup (300s): New instance ignored for 5 minutes → average calculated from existing instances only → stable scaling

Choosing the Target Value:

Target CPUBehaviorTrade-off
40%Scales out aggressively — more instances runningHigher cost, very responsive
60%✅ Balanced — good headroom for spikesRecommended for most web apps
80%Scales out conservatively — fewer instancesLower cost, less headroom for sudden spikes

→ 60% chosen because: Web apps have bursty traffic → need headroom between average and 100%
→ If target is 80% and a sudden spike hits → instances go to 100% before new ones launch → user impact

STEP 7 — CONFIGURE SCALING COOLDOWNS AND PROTECTIONS

Cooldown Settings:

SettingValuePurpose
Default cooldown300 secondsAfter a scaling activity → wait 5 minutes before another
Scale-in protectionDisabled for nowWhen enabled → specific instances cannot be terminated during scale-in
Warm poolSee belowPre-initialized instances ready to launch faster

Understanding Cooldown:

Alarm fires → Scale-out: +2 instances → COOLDOWN STARTS (300s)
                                              │
During cooldown → No additional scaling       │
                  activities occur             │
                                              │
Cooldown ends → Evaluate metrics again ◄──────┘

→ Without cooldown: Alarm fires → add 2 → metrics still high (new instances warming up) → alarm fires again → add 2 more → overscaling
→ With cooldown: Alarm fires → add 2 → wait 5 minutes → new instances absorb load → metrics normalize → no further action needed

→ Target Tracking policies manage their own cooldown internally → you don't set cooldown for them
→ Step Scaling policies USE the ASG's default cooldown

Warm Pool Configuration (Optional — Advanced):

→ A warm pool keeps pre-initialized instances in a STOPPED or RUNNING state
→ When scale-out happens → ASG pulls from warm pool instead of launching fresh

✅ WHAT SHOULD DO

→ Always use Launch Template (never Launch Configuration)
→ Always use custom AMI with application pre-installed for faster scaling response
→ Always deploy across minimum 2 Availability Zones
→ Always set Minimum capacity ≥ 2 for production (single instance = single point of failure)
→ Always enable detailed monitoring (1-minute metrics) for responsive scaling
→ Always set health check grace period longer than application startup time
→ Always enable BOTH EC2 and ELB health checks
→ Always create a dedicated /health endpoint — lightweight, no database-heavy queries
→ Always set Maximum capacity as a COST SAFETY NET — prevents runaway scaling
→ Always use Target Tracking as primary policy — simplest and most effective
→ Always add Scheduled Scaling for known events (sales, campaigns, launches)
→ Always configure scale-in to be CONSERVATIVE (remove slowly) and scale-out to be AGGRESSIVE (add fast)
→ Always restrict EC2 Security Group to accept traffic ONLY from ALB Security Group
→ Always use instance warmup to prevent oscillation
→ Always set up ASG notifications for launch failures

❌ WHAT SHOULD NOT DO

→ Never set Minimum to 1 in production — one instance = one AZ = one failure away from downtime
→ Never set health check grace period too short — causes termination loop (launch → fail check → terminate → launch)
→ Never use only Simple Scaling — it's the least responsive policy type
→ Never set Target Tracking target too high (>80%) — no headroom for sudden spikes
→ Never forget to update the AMI in Launch Template when deploying new application versions
→ Never allow EC2 instances to be directly accessible from internet when behind ALB
→ Never enable stickiness unless your application truly requires it — breaks even load distribution
→ Never set Maximum too low — during a genuine traffic spike, hitting the ceiling means dropped requests
→ Never set Maximum too high without cost alerts — a bug in your app could trigger infinite scaling
→ Never mix Launch Template and manual instance registration in the same Target Group
→ Never skip the warm pool evaluation for latency-sensitive applications
→ Never ignore Spot Instance interruption handling if using mixed fleet
→ Never deploy a new AMI to ALL instances simultaneously — use Instance Refresh with checkpoints

🔵 WHEN TO USE AUTO SCALING

ScenarioUse Auto Scaling?

Unpredictable traffic spikes (viral content, flash sales)✅ Yes — Target Tracking + Step Scaling

🔴 WHEN NOT TO USE AUTO SCALING

→ When your application is NOT horizontally scalable

  • If your app can't run on multiple instances (singleton processes, file locks) → fix the app first
  • Auto Scaling adds instances — if the app can't share load → more instances don't help

→ When you're running a managed service (RDS, Redshift, ElastiCache)

  • These have their own scaling mechanisms
  • Don't put managed services behind ASG

→ When your application has extremely long startup time (>15 minutes)

  • By the time new instances are ready, the spike may have passed
  • Use warm pools or consider container-based deployment (ECS/EKS)

→ When cost is more important than availability

  • Dev/test environments → single instance is fine
  • ASG with Min=2 doubles your baseline cost

→ When your workload is purely serverless

  • Lambda auto-scales per request
  • Fargate auto-scales per task
  • No need for EC2 Auto Scaling

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

MECHANIC 1 — ASG CONTROL LOOP

→ The ASG runs a continuous control loop:

EVERY 10-60 SECONDS:
    │
    ├─ Check: Is current running count == Desired?
    │   ├─ Running < Desired → LAUNCH instances (scale out)
    │   ├─ Running > Desired → TERMINATE instances (scale in)
    │   └─ Running == Desired → No action
    │
    ├─ Check: Are all instances healthy?
    │   ├─ Unhealthy instance found → TERMINATE it → Desired stays same → replacement launches
    │   └─ All healthy → No action
    │
    ├─ Check: Are instances balanced across AZs?
    │   ├─ Imbalanced → Launch in under-represented AZ, terminate in over-represented AZ
    │   └─ Balanced → No action
    │
    └─ Check: Any scaling policies triggered?
        ├─ Scale-out policy → Increase Desired → triggers launch in next loop iteration
        ├─ Scale-in policy → Decrease Desired → triggers termination in next loop iteration
        └─ No policy triggered → No action

→ This loop is the HEART of Auto Scaling — it's always running, always evaluating
→ The Desired count is the SINGLE number that controls everything
→ Scaling policies MODIFY Desired → the control loop ACTS on it

MECHANIC 2 — ALB REQUEST ROUTING

→ When a request arrives at the ALB:

Client Request → DNS → ALB Endpoint
    │
    ▼
ALB receives request on Listener (port 80/443)
    │
    ▼
Listener evaluates Rules (in priority order):
    │
    ├─ Rule 1: If path = /api/* → Forward to API Target Group
    ├─ Rule 2: If path = /static/* → Forward to Static Target Group
    └─ Default Rule: Forward to shopnow-web-tg
    │
    ▼
Target Group selects a healthy target using ROUTING ALGORITHM:
    │
    ├─ Round Robin (default): Distribute requests evenly across all healthy targets
    └─ Least Outstanding Requests: Send to target with fewest in-flight requests
    │
    ▼
ALB forwards request to selected EC2 instance
    │
    ▼
EC2 instance processes request → returns response → ALB forwards to client

Understanding Routing Algorithms:

AlgorithmBehaviorBest For
Round RobinAlternates between targets sequentiallyUniform request processing time
Least Outstanding RequestsRoutes to target with fewest active connections✅ Variable request processing time (recommended)

→ ShopNow: Use Least Outstanding Requests
→ Some pages (product detail) take 100ms, others (checkout) take 2 seconds
→ Round Robin would overload instances processing slow requests

MECHANIC 3 — HEALTH CHECK LIFECYCLE

New instance launched by ASG
    │
    ▼
Instance enters "Pending" state
    │
    ▼
EC2 starts → OS boots → Application starts
    │
    ▼
Health check grace period countdown (300 seconds)
    │  (NO health checks during this period)
    ▼
Grace period ends → Health checks begin
    │
    ├─ ALB sends GET /health every 30 seconds
    │   ├─ Response 200 within 5s → Healthy count +1
    │   └─ No response / non-200 → Unhealthy count +1
    │
    ├─ Healthy threshold reached (3 consecutive) → Instance marked HEALTHY
    │   └─ ALB starts routing traffic to this instance
    │
    └─ Unhealthy threshold reached (2 consecutive) → Instance marked UNHEALTHY
        └─ ALB stops routing traffic → ASG terminates instance → launches replacement

Two-Layer Health Checks:

LayerCheckCatches
EC2 Health CheckIs the VM running? System status OK?Hardware failure, kernel panic, network unreachable
ELB Health CheckDoes the application respond on /health?Application crash, memory exhaustion, deadlock, dependency failure

→ EC2 check passes but ELB check fails = Application is broken but OS is fine
→ Without ELB health check → ASG thinks instance is healthy → keeps routing traffic → users see errors
→ With ELB health check → ASG detects application failure → replaces instance automatically

MECHANIC 4 — TARGET TRACKING INTERNAL ALGORITHM

→ Target Tracking uses a proportional scaling algorithm:

Target: 60% CPU
Current instances: 4
Current average CPU: 80%

Required instances = Current instances × (Current metric / Target metric)
Required instances = 4 × (80 / 60)
Required instances = 5.33
Required instances = 6 (rounded UP for scale-out)

Action: Increase Desired from 4 to 6 → launch 2 instances

Scale-in calculation:

Current instances: 6
Current average CPU: 35%

Required instances = 6 × (35 / 60)
Required instances = 3.5
Required instances = 4 (rounded UP for scale-in — conservative)

Action: Decrease Desired from 6 to 4 → terminate 2 instances

→ Notice: Scale-out rounds UP → aggressive
→ Scale-in also rounds UP → conservative (keeps more instances than minimum needed)
→ This asymmetry is by design — AWS prioritizes availability over cost savings

Target Tracking also creates TWO CloudWatch Alarms automatically:

AlarmThresholdAction
TargetTracking-shopnow-web-asg-AlarmHighCPU > 60% for 3 minutesScale out
TargetTracking-shopnow-web-asg-AlarmLowCPU < 42% for 15 minutesScale in

→ The LOW alarm threshold is NOT 60% — it's intentionally lower (~70% of target)
→ And it requires 15 minutes (not 3) before triggering
→ This prevents oscillation: scale out at 60%, scale in only when well below 60% for extended period

MECHANIC 5 — INSTANCE LAUNCH SEQUENCE

→ When ASG decides to launch a new instance:

1. ASG selects target AZ (least instances or rebalancing)
2. ASG reads Launch Template (latest version)
3. ASG calls EC2 RunInstances API with template parameters
4. EC2 allocates resources in selected AZ
5. EC2 creates ENI (network interface) in specified subnet
6. EC2 attaches Security Groups
7. EC2 attaches IAM Instance Profile
8. EC2 creates EBS volumes
9. EC2 starts the instance
10. User Data script executes (if configured)
11. Instance enters "Pending" state in ASG
12. ASG registers instance with Target Group
13. Health check grace period begins
14. Grace period ends → health checks begin
15. Health checks pass → instance marked InService
16. ALB starts routing traffic to instance

→ Total time: 2-5 minutes (without warm pool)
→ With warm pool (stopped): Steps 1-10 already done → starts at step 11 → 30-60 seconds

MECHANIC 6 — AZ REBALANCING

→ ASG automatically distributes instances evenly across AZs

Desired: 4 instances
AZs: ap-south-1a, ap-south-1b

Balanced state:
    AZ-1a: [Instance-1] [Instance-2]
    AZ-1b: [Instance-3] [Instance-4]

If Instance-2 fails:
    AZ-1a: [Instance-1] ← only 1
    AZ-1b: [Instance-3] [Instance-4] ← has 2

ASG detects imbalance → launches new instance in AZ-1a:
    AZ-1a: [Instance-1] [Instance-5] ← rebalanced
    AZ-1b: [Instance-3] [Instance-4]

→ During rebalancing → ASG may temporarily EXCEED Desired capacity
→ It launches first (in under-represented AZ) → then terminates (in over-represented AZ)
→ This ensures availability is maintained during rebalancing

→ If an entire AZ goes down:

Before AZ failure:
    AZ-1a: [Instance-1] [Instance-2]
    AZ-1b: [Instance-3] [Instance-4]

AZ-1a goes down:
    AZ-1a: ❌ ❌
    AZ-1b: [Instance-3] [Instance-4]

ASG detects: Running (2) < Desired (4)
ASG launches 2 new instances in AZ-1b:
    AZ-1a: ❌ (unavailable)
    AZ-1b: [Instance-3] [Instance-4] [Instance-5] [Instance-6]

→ Service continues with all 4 instances in the surviving AZ
→ When AZ-1a recovers → ASG gradually rebalances back

📈 SCALABILITY CONSIDERATIONS

→ Instance Count Scaling

  • ASG supports up to 500 instances per group (soft limit — can request increase)
  • For larger fleets → multiple ASGs behind same ALB Target Group
  • OR use ECS/EKS for container-based scaling (thousands of containers)

→ ALB Scaling

  • ALB scales AUTOMATICALLY based on traffic — no configuration needed
  • For sudden massive spikes (millions of requests/second) → ALB may take 1-5 minutes to scale
  • For known events → "pre-warm" the ALB by contacting AWS Support with expected traffic pattern
  • OR gradually ramp up traffic before the event

→ AZ Scaling

  • More AZs = better availability but potentially higher cross-AZ data transfer costs
  • 2 AZs: Good for most applications
  • 3 AZs: Better availability, recommended for critical production
  • All AZs in region: Maximum availability, higher complexity

→ Cost Scaling

→ Metric Scaling

🎯 KEY TAKEAWAY

→ Auto Scaling is NOT just about adding instances — it's a THREE-COMPONENT system:

  1. Launch Template — the WHAT (instance blueprint with AMI, type, security, IAM)
  2. Application Load Balancer — the HOW (traffic distribution + health detection)
  3. Auto Scaling Group — the WHEN and HOW MANY (scaling policies + fleet management)

→ All three must work together — removing any one breaks the system:

  • Without Launch Template → ASG doesn't know what to launch
  • Without ALB → traffic can't reach new instances → scaling is useless
  • Without ASG → no automatic scaling → back to manual intervention

→ The FOUR scaling policy types serve different purposes:

  • Target Tracking → "Keep metric at X" → handles 80% of scenarios → START HERE
  • Step Scaling → "At different thresholds, respond differently" → emergency response
  • Scheduled Scaling → "At known times, pre-scale" → predictable events
  • Predictive Scaling → "Learn from history, pre-scale automatically" → repeating patterns

→ The GOLDEN RULE of scaling:

  • Scale OUT aggressively (add many instances quickly)
  • Scale IN conservatively (remove few instances slowly)
  • Users feel scale-out delays (errors, timeouts)
  • Users don't feel unnecessary extra instances (just costs you money)
  • It's always better to over-provision briefly than under-provision during a spike

→ Health checks are the AUTOMATIC HEALING mechanism:

  • EC2 health check → detects infrastructure failure
  • ELB health check → detects application failure
  • BOTH enabled → comprehensive detection → automatic replacement

→ Security fundamentally changes when introducing ALB:

  • Internet → ALB (public) → EC2 (private from internet)
  • EC2 Security Group: Accept traffic ONLY from ALB Security Group
  • This is a significant improvement over Project 4's direct internet-to-EC2 model

→ Connection to Previous Projects:

  • Project 3 → Single EC2 instance was the starting point — now replaced by managed fleet
  • Project 4 → Security Groups evolve — EC2 SG now references ALB SG instead of 0.0.0.0/0
  • Project 5 → VPC subnets used for ALB (public) and potentially EC2 (public or private)
  • Project 8 → CloudWatch Alarms now TRIGGER scaling actions, not just send emails
  • Project 11 → IAM Instance Profile applied via Launch Template to all instances
  • Project 6 → SNS notifications for scaling events

→ What this project fundamentally changes:

  • Before: One server → it dies → site dies → manual recovery in hours
  • After: Fleet of servers → one dies → automatically replaced in minutes → users never notice
  • Before: Traffic spike → server overwhelmed → site crashes
  • After: Traffic spike → CloudWatch detects → ASG adds instances → ALB distributes → site stays fast
  • Before: Black Friday requires manual preparation → human error → disaster
  • After: Scheduled + predictive scaling → infrastructure ready BEFORE traffic arrives
