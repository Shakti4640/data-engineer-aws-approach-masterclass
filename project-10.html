PROJECT 10: THE PUBLIC BUCKET THAT COST $50,000 — S3 Bucket Policies & Access Control Lists

🔴 HYPOTHETICAL PROBLEM

→ ShopNow (from Projects 1-9) now has:

  • S3 bucket for product images (Project 1)
  • IAM users with proper permissions (Project 2)
  • EC2 instances in a properly architected VPC (Projects 3, 4, 5)
  • SNS topics and SQS queues for messaging (Projects 6, 7)
  • CloudWatch monitoring and alerting (Project 8)
  • RDS MariaDB in private subnets (Project 9)

→ Incident 1: ShopNow hired a freelance developer to build a mobile app
→ The freelancer needed access to product images in S3
→ priya-dev created an IAM user for the freelancer with s3:GetObject permission
→ But the freelancer's contract ended 3 months ago → the IAM user was never deleted
→ The freelancer's access keys were found in a public GitHub repository
→ A bot discovered the keys → downloaded the ENTIRE bucket contents including internal documents
→ 50,000 customer records with personal information were exposed
→ Data breach notification cost + legal fees + reputation damage = ₹50,00,000

→ Incident 2: A junior developer was testing a static website hosting feature
→ He enabled "public access" on the PRODUCTION bucket to test
→ He forgot to disable it after testing
→ Google indexed the bucket contents → customer invoices appeared in search results
→ A security researcher found it → posted on Twitter → media picked it up
→ AWS sent a "Your bucket is public" warning email → nobody read it for 2 weeks

→ Incident 3: The marketing team uploads product images to the marketing/ prefix
→ Some images were uploaded WITHOUT encryption → violating company policy
→ The compliance team says: "EVERY object in this bucket MUST be encrypted with KMS"
→ But IAM policies (Project 2) cannot enforce encryption on uploads — they control WHO can upload, not HOW

→ Incident 4: ShopNow's partner company "LogiShip" (different AWS account: 987654321098) needs to read shipping labels from ShopNow's S3 bucket
→ IAM policies are IDENTITY-based — they live in the CALLER'S account
→ ShopNow cannot attach IAM policies to LogiShip's users
→ They need a way to grant access FROM THE BUCKET SIDE

→ The CTO demands:

  • The bucket itself must enforce security rules — regardless of who accesses it
  • ALL uploads must be encrypted — rejected if not
  • Specific IP ranges should be blocked at the bucket level
  • Cross-account access must be controlled from the bucket
  • Public access must be permanently blocked with multiple layers of protection

🟡 HOW TO APPROACH IT

→ This is a RESOURCE-BASED POLICY problem
→ AWS has TWO types of policies:

Policy TypeAttached ToControlsAnswers
Identity-based (IAM — Project 2)User, Group, or RoleWhat the IDENTITY can do"What can this person do?"
Resource-based (Bucket Policy)The S3 bucket itselfWhat the RESOURCE allows"Who can access this bucket and how?"

→ Think of it like a building:

  • IAM Policy = Your employee badge (determines which buildings YOU can enter)
  • Bucket Policy = Building's guest list (determines who the BUILDING allows in)
  • BOTH must agree for access to succeed (with one important exception — cross-account)

STEP 1 — UNDERSTAND BUCKET POLICY STRUCTURE

→ A bucket policy is a JSON document attached directly to an S3 bucket
→ It uses the SAME policy language as IAM (Project 2) with one key addition: Principal

→ S3 Console → Select bucket → "Permissions" tab → "Bucket policy" → "Edit"

Anatomy of a Bucket Policy:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowWebServerRead",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:role/shopnow-web-server-role"
      },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::shopnow-productimages-prod-aps1/*",
      "Condition": {
        "StringEquals": {
          "aws:SourceVpc": "vpc-0abc123def456"
        }
      }
    }
  ]
}

Fields — What's NEW compared to IAM policies (Project 2):

FieldIAM PolicyBucket PolicyExplanation
VersionSameSameAlways "2012-10-17"
StatementSameSameArray of permission rules
SidSameSameHuman-readable statement ID
EffectSameSameAllow or Deny
PrincipalNOT presentREQUIREDWHO this statement applies to
ActionSameSameWhat operations are allowed/denied
ResourceSameSameWhich objects/bucket this applies to
ConditionSameSameAdditional constraints

→ The KEY difference: Bucket policies have a Principal field
→ IAM policies don't need Principal because they're attached to the identity (the principal IS the user/role)
→ Bucket policies need Principal because they're attached to the resource — they must specify WHO

Principal Types:

Principal FormatWho It RepresentsExample
"AWS": "arn:aws:iam::123456789012:root"Entire AWS accountAllow any identity in account 123456789012
"AWS": "arn:aws:iam::123456789012:user/priya-dev"Specific IAM userAllow only priya-dev
"AWS": "arn:aws:iam::123456789012:role/web-server-role"Specific IAM roleAllow only this role
"AWS": "*"Any AWS identityAllow anyone (DANGEROUS without conditions)
"Service": "s3.amazonaws.com"AWS S3 serviceAllow S3 replication (Project 30)
"Service": "sns.amazonaws.com"AWS SNS serviceAlready used in Project 6 topic policies
"Service": "cloudwatch.amazonaws.com"AWS CloudWatchAllow CloudWatch to read/write
"*"ANYONE (including anonymous)Public access — extremely dangerous

CRITICAL: "*" vs "AWS": "*":

PrincipalMeaningRisk Level
"*"ANYONE — including unauthenticated users (no AWS account needed)EXTREME — public internet access
"AWS": "*"Any AUTHENTICATED AWS identity (must have an AWS account)HIGH — any AWS customer

→ Using "*" as principal = making the bucket publicly accessible to the entire internet
→ This is what caused Incident 2 — public access without authentication

STEP 2 — HOW IAM POLICIES AND BUCKET POLICIES INTERACT

→ This is the MOST important concept in this project
→ Understanding this prevents 90% of S3 access issues

SAME-ACCOUNT ACCESS (IAM user in Account A accessing bucket in Account A):

priya-dev (Account 123456789012) → GetObject → shopnow-bucket (Account 123456789012)

EVALUATION:
Step 1: Gather ALL applicable policies
        - IAM policy on priya-dev (identity-based)
        - Bucket policy on shopnow-bucket (resource-based)
        - Permission boundary on priya-dev (if set — Project 2)

Step 2: Is there an EXPLICIT DENY in ANY policy?
        YES → DENIED (game over)
        NO → Continue

Step 3: Is there an EXPLICIT ALLOW in ANY policy?
        IAM policy allows? → ALLOWED ✓
        OR Bucket policy allows? → ALLOWED ✓
        (Either one is sufficient for same-account)

Step 4: Neither allows? → DENIED (implicit deny)

→ KEY: For same-account access → EITHER the IAM policy OR the bucket policy can grant access
→ You don't need BOTH to allow — one is enough
→ But an explicit DENY in either one blocks access regardless

Example Scenarios (Same Account):

IAM PolicyBucket PolicyResultExplanation
Allow s3:GetObjectNo bucket policyALLOWEDIAM allows → sufficient
No IAM policy for S3Allow Principal: priya-dev GetObjectALLOWEDBucket policy allows → sufficient
Allow s3:GetObjectDeny Principal: priya-devDENIEDExplicit deny in bucket policy wins
Deny s3:GetObjectAllow Principal: priya-devDENIEDExplicit deny in IAM policy wins
No IAM policyNo bucket policyDENIEDNo allow anywhere → implicit deny
Allow s3:GetObjectAllow Principal: priya-devALLOWEDBoth allow → definitely allowed

CROSS-ACCOUNT ACCESS (IAM user in Account B accessing bucket in Account A):

logiship-user (Account 987654321098) → GetObject → shopnow-bucket (Account 123456789012)

EVALUATION:
Step 1: BOTH must explicitly allow
        - IAM policy in Account B (on logiship-user): Must allow s3:GetObject on shopnow-bucket
        AND
        - Bucket policy in Account A (on shopnow-bucket): Must allow Principal from Account B

Step 2: If EITHER is missing the allow → DENIED

Step 3: Explicit deny in EITHER → DENIED

→ KEY: For cross-account access → BOTH the IAM policy AND the bucket policy must explicitly allow
→ One is NOT sufficient — you need both sides to agree
→ This is the fundamental difference from same-account evaluation

Cross-Account Matrix:

IAM Policy (Account B)Bucket Policy (Account A)Result
Allow s3:GetObjectAllow Principal: Account BALLOWED ✓
Allow s3:GetObjectNo bucket policyDENIED ✗
No IAM policyAllow Principal: Account BDENIED ✗
Allow s3:GetObjectDeny Principal: Account BDENIED ✗
No IAM policyNo bucket policyDENIED ✗

→ This is why LogiShip needs BOTH:

  1. Their IAM admin creates a policy allowing access to ShopNow's bucket
  2. ShopNow's bucket policy explicitly allows LogiShip's account
    → Project 39 covers this in full depth

STEP 3 — CREATE BUCKET POLICIES FOR SHOPNOW

POLICY 1: Allow Web Server Role to Read All Objects

→ The EC2 web server (with IAM Role — will be configured in Project 11) needs to read product images

Statement:
  Sid: AllowWebServerRead
  Effect: Allow
  Principal:
    AWS: arn:aws:iam::123456789012:role/shopnow-web-server-role
  Action:
    - s3:GetObject
  Resource:
    - arn:aws:s3:::shopnow-productimages-prod-aps1/*

→ Only the specific IAM Role can read objects
→ Not all users — just the web server role
→ Resource uses /* for object-level access (Project 2 ARN lesson)

POLICY 2: Allow Finance Team to Read/Write Only invoices/ Prefix

→ Supplements the IAM policy from Project 2 with bucket-level enforcement

Statement 1:
  Sid: AllowFinanceListInvoices
  Effect: Allow
  Principal:
    AWS: arn:aws:iam::123456789012:root
  Action:
    - s3:ListBucket
  Resource:
    - arn:aws:s3:::shopnow-productimages-prod-aps1
  Condition:
    StringLike:
      s3:prefix:
        - "invoices/*"
        - ""

Statement 2:
  Sid: AllowFinanceReadWriteInvoices
  Effect: Allow
  Principal:
    AWS: arn:aws:iam::123456789012:root
  Action:
    - s3:GetObject
    - s3:PutObject
  Resource:
    - arn:aws:s3:::shopnow-productimages-prod-aps1/invoices/*

→ Wait — Principal is root (entire account) but IAM policies restrict per-user
→ This is the LAYERED approach:

  • Bucket policy: Allows the account broadly (for invoices prefix)
  • IAM policy (Project 2): Restricts specific users to specific prefixes
  • BOTH work together — IAM policy is the fine-grained control

→ Alternatively — you can specify individual user ARNs as principal for even tighter bucket-level control
→ For most organizations → IAM policies handle per-user restrictions, bucket policies handle broad resource-level rules

POLICY 3: Explicit Deny — Block Specific IAM User (Terminated Freelancer)

→ The freelancer's IAM user should have been deleted — but as defense in depth:

Statement:
  Sid: DenyFreelancerAccess
  Effect: Deny
  Principal:
    AWS: arn:aws:iam::123456789012:user/freelancer-dev
  Action:
    - s3:*
  Resource:
    - arn:aws:s3:::shopnow-productimages-prod-aps1
    - arn:aws:s3:::shopnow-productimages-prod-aps1/*

→ Even if the freelancer's IAM policy says Allow → the bucket's explicit Deny overrides it
→ This is defense in depth — even if IAM user deletion is delayed, the bucket is protected
→ Best practice: Delete the IAM user AND add a bucket policy deny as double protection

STEP 4 — ENFORCE ENCRYPTION ON ALL UPLOADS

→ This solves Incident 3: Some objects uploaded without encryption

Policy: Deny Any Upload Without Server-Side Encryption

Statement 1:
  Sid: DenyUnencryptedUploads
  Effect: Deny
  Principal: "*"
  Action: s3:PutObject
  Resource: arn:aws:s3:::shopnow-productimages-prod-aps1/*
  Condition:
    StringNotEquals:
      s3:x-amz-server-side-encryption: "aws:kms"

Statement 2:
  Sid: DenyUploadWithoutEncryptionHeader
  Effect: Deny
  Principal: "*"
  Action: s3:PutObject
  Resource: arn:aws:s3:::shopnow-productimages-prod-aps1/*
  Condition:
    "Null":
      s3:x-amz-server-side-encryption: "true"

How These Two Statements Work Together:

UPLOAD SCENARIO 1: No encryption header at all
  → Statement 2: Condition checks if header is NULL → YES → DENY ✗

UPLOAD SCENARIO 2: Encryption header = "AES256" (SSE-S3)
  → Statement 2: Header is NOT null → Statement 2 doesn't apply
  → Statement 1: Header value = "AES256" ≠ "aws:kms" → DENY ✗
  (We require KMS encryption specifically, not just SSE-S3)

UPLOAD SCENARIO 3: Encryption header = "aws:kms"
  → Statement 2: Header is NOT null → doesn't apply
  → Statement 1: Header value = "aws:kms" → StringNotEquals fails → Statement doesn't apply
  → No deny → Upload proceeds ✓

→ This enforces: EVERY upload MUST use KMS encryption
→ Anyone uploading without encryption → request rejected with "Access Denied"
→ This applies to ALL principals (including admins) — Principal: "*" in a Deny statement
→ Even the bucket owner cannot upload unencrypted objects

Why Deny Instead of Allow:

→ You CANNOT "require" something with an Allow statement
→ Allow says: "You may do this" — it doesn't say "You MUST do it this way"
→ Deny with condition says: "You CANNOT do this UNLESS you meet the condition"
→ The Deny + condition pattern is how you ENFORCE requirements

STEP 5 — RESTRICT ACCESS BY IP ADDRESS

→ Block access from specific IP ranges or allow only from known IPs

Policy: Allow Access Only from Office IP Range

Statement:
  Sid: DenyAccessFromOutsideOffice
  Effect: Deny
  Principal: "*"
  Action: s3:*
  Resource:
    - arn:aws:s3:::shopnow-productimages-prod-aps1
    - arn:aws:s3:::shopnow-productimages-prod-aps1/*
  Condition:
    NotIpAddress:
      aws:SourceIp:
        - "203.0.113.0/24"
        - "198.51.100.0/24"
    StringNotEquals:
      aws:PrincipalServiceName: ""

→ NotIpAddress: aws:SourceIp → Deny if the request comes from an IP NOT in the listed ranges
→ This blocks: Freelancers at home, stolen keys used from random IPs, bots
→ The StringNotEquals: aws:PrincipalServiceName condition prevents blocking AWS service calls
→ Without it → S3 replication, CloudTrail logging, and other services would be blocked

CAUTION: IP-based restrictions and AWS service access:

→ When AWS services access your bucket (CloudTrail, S3 Replication, Glue, etc.):

  • They don't come from your office IP
  • They use AWS internal IPs
  • IP-based deny would block them

→ Solution: Add conditions to exclude AWS service principals:

Condition:
  NotIpAddress:
    aws:SourceIp: "203.0.113.0/24"
  StringNotLike:
    aws:userId:
      - "AROA*"

→ AROA* is the prefix for role session IDs → this excludes calls made by IAM roles
→ Alternatively, use aws:ViaAWSService: "true" condition to detect service-to-service calls

→ IP-based restrictions are COMPLEX — test thoroughly before applying

STEP 6 — RESTRICT ACCESS BY VPC ENDPOINT

→ A more reliable approach than IP-based restrictions
→ Instead of trusting IP addresses → trust the VPC endpoint through which the request arrives
→ VPC Endpoints are covered in Project 27 — this is a preview of the bucket policy side

Policy: Allow Access Only from Specific VPC

Statement:
  Sid: DenyAccessFromOutsideVPC
  Effect: Deny
  Principal: "*"
  Action: s3:*
  Resource:
    - arn:aws:s3:::shopnow-productimages-prod-aps1
    - arn:aws:s3:::shopnow-productimages-prod-aps1/*
  Condition:
    StringNotEquals:
      aws:SourceVpc: "vpc-0abc123def456"

→ aws:SourceVpc → the VPC ID from which the request originates
→ This blocks ALL access that doesn't come through the specified VPC
→ Even valid IAM credentials from outside the VPC → DENIED

Or restrict to specific VPC Endpoint:

Condition:
  StringNotEquals:
    aws:SourceVpce: "vpce-0abc123def456"

→ aws:SourceVpce → the specific VPC endpoint ID
→ More granular than VPC-level restriction

CAUTION with VPC restrictions:

→ Console access (from your browser) does NOT go through VPC endpoints
→ If you restrict to VPC only → you CANNOT manage the bucket from the AWS Console
→ Solution: Add an exception for specific IAM roles used for administration:

Statement:
  Sid: DenyAccessFromOutsideVPCExceptAdmin
  Effect: Deny
  Principal: "*"
  Action: s3:*
  Resource:
    - arn:aws:s3:::shopnow-productimages-prod-aps1
    - arn:aws:s3:::shopnow-productimages-prod-aps1/*
  Condition:
    StringNotEquals:
      aws:SourceVpc: "vpc-0abc123def456"
    ArnNotLike:
      aws:PrincipalArn:
        - "arn:aws:iam::123456789012:role/shopnow-admin-role"

→ This denies non-VPC access EXCEPT when the caller is the admin role
→ Admin can manage from console → everyone else must go through VPC endpoint

STEP 7 — BLOCK PUBLIC ACCESS SETTINGS (DEFENSE IN DEPTH)

→ S3 provides FOUR levels of public access blocking
→ These are OVERRIDES that supersede bucket policies and ACLs

→ S3 Console → Select bucket → "Permissions" → "Block public access" → "Edit"

The Four Settings:

#SettingWhat It Blocks
1Block public access granted through NEW ACLsPrevents adding ACLs that grant public access
2Block public access granted through ANY ACLsIgnores ALL existing and future ACLs that grant public access
3Block public access granted through NEW public bucket policiesPrevents saving bucket policies that grant public access
4Block public access granted through ANY public bucket or access point policiesIgnores ALL existing and future policies that grant public access

How "Block All Public Access" Works:

SCENARIO: Block All Public Access = ON (all 4 settings enabled)

Developer tries to add bucket policy:
  Principal: "*"
  Effect: Allow
  Action: s3:GetObject

Step 1: Developer clicks "Save"
Step 2: S3 evaluates the policy → detects public access grant
Step 3: S3 REJECTS the policy change → Error: "Block Public Access settings prevent this"
Step 4: Policy is NOT saved → bucket remains private

Even if policy was ALREADY present before Block Public Access was enabled:
Step 1: Block Public Access enabled
Step 2: Existing public-granting policy is OVERRIDDEN → effectively ignored
Step 3: Public access requests → DENIED regardless of bucket policy

→ Block Public Access is the NUCLEAR OPTION — it overrides everything
→ This is what would have prevented Incident 2 — even if the developer enabled public access in the bucket policy, Block Public Access would have rejected it

Account-Level Block Public Access:

→ S3 Console → "Block Public Access settings for this account" (left sidebar)
→ You can enable Block Public Access at the ACCOUNT level
→ This applies to ALL buckets in the account — overrides individual bucket settings

LevelScopeOverride Hierarchy
Account-level Block Public AccessALL buckets in the accountHighest priority — overrides bucket-level
Bucket-level Block Public AccessThis specific bucketOverrides bucket policies and ACLs
Bucket PolicyThis specific bucketCan be overridden by Block Public Access
ACLsSpecific objects or bucketCan be overridden by Block Public Access

→ For ShopNow → Enable Block Public Access at BOTH account level AND bucket level
→ Belt AND suspenders approach

STEP 8 — UNDERSTAND ACLs (LEGACY) AND WHY TO AVOID THEM

What Are ACLs?

→ ACLs (Access Control Lists) are the ORIGINAL S3 access control mechanism
→ They predate bucket policies and IAM policies
→ They are MUCH less flexible than bucket policies

ACL Grantees:

GranteeMeaning
Bucket ownerThe AWS account that owns the bucket
Authenticated usersANY authenticated AWS account (DANGEROUS)
All users (public)EVERYONE including anonymous (EXTREMELY DANGEROUS)
Log deliveryS3 log delivery service
Specific AWS accountIdentified by canonical user ID or email

ACL Permissions:

PermissionBucket LevelObject Level
READList objects in bucketRead the object data
WRITECreate/delete objects in bucketN/A
READ_ACPRead the bucket ACLRead the object ACL
WRITE_ACPModify the bucket ACLModify the object ACL
FULL_CONTROLAll of the aboveAll of the above

Why ACLs Are Problematic:

IssueExplanation
Coarse-grainedOnly 5 permission types — no conditional logic
Confusing grantees"Authenticated users" means ANY AWS account — not "your" users
Object-level ACLsEach object can have its own ACL — impossible to manage at scale
No conditionsCannot restrict by IP, VPC, encryption, time, etc.
Cross-account confusionObjects uploaded by other accounts may be owned by them, not you
LegacyAWS recommends disabling ACLs for all new buckets

AWS Recommendation: Disable ACLs

→ S3 Console → Create bucket → Object Ownership → "ACLs disabled (recommended)"
→ This was already configured in Project 1

→ When ACLs are disabled:

  • The bucket owner owns ALL objects regardless of who uploaded
  • ACLs cannot be set on objects or the bucket
  • All access control is through IAM policies and bucket policies
  • This is the CLEANEST and MOST SECURE approach

If ACLs Are Still Enabled (Legacy Buckets):

→ S3 Console → Select bucket → "Permissions" → "Access control list (ACL)"
→ Verify:

  • Bucket owner: FULL_CONTROL (default, cannot remove)
  • No other grantees listed
  • "All users" (public) and "Authenticated users" should NOT have any permissions

→ If you find "All users: READ" → the bucket is PUBLIC → remove immediately
→ Better yet → switch to "ACLs disabled" in Object Ownership settings

STEP 9 — CROSS-ACCOUNT ACCESS VIA BUCKET POLICY (Preview for Project 39)

→ LogiShip (Account 987654321098) needs to read shipping labels from ShopNow's bucket

Bucket Policy on ShopNow's Bucket:

Statement:
  Sid: AllowLogiShipRead
  Effect: Allow
  Principal:
    AWS: "arn:aws:iam::987654321098:root"
  Action:
    - s3:GetObject
  Resource:
    - arn:aws:s3:::shopnow-productimages-prod-aps1/shipping-labels/*
  Condition:
    StringEquals:
      aws:PrincipalOrgID: "o-abc123def4"

→ Principal: Account 987654321098 → allows LogiShip's account
→ Resource: /shipping-labels/* → only the shipping-labels prefix — not the entire bucket
→ Condition: aws:PrincipalOrgID → only if LogiShip is in the same AWS Organization (Project 54)
→ Without the org condition → any account you specify can access

What LogiShip Must Also Do (Their Side):

→ LogiShip's IAM admin must create an IAM policy for their user:

Effect: Allow
Action: s3:GetObject
Resource: arn:aws:s3:::shopnow-productimages-prod-aps1/shipping-labels/*

→ Remember: Cross-account requires BOTH sides (Step 2)
→ ShopNow's bucket policy allows LogiShip
→ LogiShip's IAM policy allows their user to access ShopNow's bucket
→ If either side is missing → access denied

→ Full cross-account patterns are covered in Project 39

→ Combining all the policies from Steps 3-6 into a single bucket policy:

{
  "Version": "2012-10-17",
  "Statement": [

    Statement 1: Allow web server role to read all objects
    Sid: AllowWebServerRead
    Effect: Allow
    Principal: AWS: arn:aws:iam::123456789012:role/shopnow-web-server-role
    Action: s3:GetObject
    Resource: arn:aws:s3:::shopnow-productimages-prod-aps1/*

    Statement 2: Deny unencrypted uploads (wrong encryption type)
    Sid: DenyNonKMSEncryptedUploads
    Effect: Deny
    Principal: "*"
    Action: s3:PutObject
    Resource: arn:aws:s3:::shopnow-productimages-prod-aps1/*
    Condition:
      StringNotEquals:
        s3:x-amz-server-side-encryption: "aws:kms"

    Statement 3: Deny uploads without any encryption header
    Sid: DenyUnencryptedUploads
    Effect: Deny
    Principal: "*"
    Action: s3:PutObject
    Resource: arn:aws:s3:::shopnow-productimages-prod-aps1/*
    Condition:
      "Null":
        s3:x-amz-server-side-encryption: "true"

    Statement 4: Deny HTTP (non-HTTPS) access
    Sid: DenyInsecureTransport
    Effect: Deny
    Principal: "*"
    Action: s3:*
    Resource:
      - arn:aws:s3:::shopnow-productimages-prod-aps1
      - arn:aws:s3:::shopnow-productimages-prod-aps1/*
    Condition:
      Bool:
        aws:SecureTransport: "false"

    Statement 5: Deny access from terminated freelancer
    Sid: DenyFreelancerAccess
    Effect: Deny
    Principal: AWS: arn:aws:iam::123456789012:user/freelancer-dev
    Action: s3:*
    Resource:
      - arn:aws:s3:::shopnow-productimages-prod-aps1
      - arn:aws:s3:::shopnow-productimages-prod-aps1/*

    Statement 6: Allow LogiShip to read shipping labels
    Sid: AllowLogiShipReadShippingLabels
    Effect: Allow
    Principal: AWS: arn:aws:iam::987654321098:root
    Action: s3:GetObject
    Resource: arn:aws:s3:::shopnow-productimages-prod-aps1/shipping-labels/*

  ]
}

Statement 4 Explained: Deny HTTP (Enforce HTTPS)

→ aws:SecureTransport: "false" → true when the request uses HTTP instead of HTTPS
→ Denying when SecureTransport is false → forces all requests to use HTTPS
→ This ensures data in transit is ALWAYS encrypted (TLS/SSL)
→ Combined with KMS encryption on upload → data is encrypted BOTH at rest AND in transit

IMPORTANT: Bucket Policy Size Limit:

LimitValue
Maximum bucket policy size20 KB
Maximum statementsNo hard limit (but policy size constrains)

→ 20 KB is substantial but can be reached with many detailed statements
→ If you hit the limit → consolidate statements, use fewer conditions, or use IAM policies for per-user rules
→ Bucket policies should handle RESOURCE-level rules (encryption, transport, VPC)
→ IAM policies should handle IDENTITY-level rules (per-user, per-group permissions)

✅ WHAT SHOULD DO

→ Always enable Block Public Access at BOTH account level AND bucket level
→ Always enforce HTTPS with the aws:SecureTransport condition
→ Always enforce encryption with Deny + condition pattern on PutObject
→ Always use specific principals in Allow statements — avoid "*" or "AWS": "*"
→ Always use both bucket ARN and bucket ARN/* in Deny statements (covers both bucket and object operations)
→ Always add Deny statements for terminated employees as defense in depth
→ Always disable ACLs on new buckets — use bucket policies and IAM policies instead
→ Always test bucket policies with a test user before applying to production
→ Always use the Policy Simulator (IAM Console) to verify combined policy effects
→ Always include Sid on every statement for debugging and audit
→ Always version control your bucket policies (store in Git alongside infrastructure code)
→ Always review bucket policies quarterly for stale principals or overly broad permissions

❌ WHAT SHOULD NOT DO

→ Never use "*" as Principal in an Allow statement without strict conditions
→ Never use ACLs for new access control — they are legacy and confusing
→ Never grant "Authenticated users" ACL access — it means ANY AWS customer
→ Never forget that cross-account access requires BOTH sides to explicitly allow
→ Never rely solely on IAM policies for sensitive buckets — add bucket policy Deny statements
→ Never add a bucket policy that contradicts Block Public Access settings — it will be rejected
→ Never use HTTP for S3 access — always enforce HTTPS via bucket policy
→ Never leave encryption enforcement to "hope" — use Deny policies to REQUIRE it
→ Never create bucket policies larger than 20 KB — simplify or split concerns
→ Never forget to include both arn:aws:s3:::bucket AND arn:aws:s3:::bucket/* in resource
→ Never assume deleting an IAM user immediately prevents access — add bucket-level deny for safety
→ Never test bucket policy changes on production first — test on a non-production bucket

🔵 WHEN TO USE WHAT

ScenarioUse Bucket PolicyUse IAM Policy
Enforce encryption on all uploads✅ Deny without encryption header❌ IAM can't enforce upload conditions
Enforce HTTPS on all access✅ Deny when SecureTransport=false❌ IAM can't enforce transport method
Restrict access to VPC only✅ Deny when SourceVpc doesn't match❌ IAM can't reference VPC conditions easily
Block specific IP ranges✅ Deny from specific IPs❌ Possible but less natural
Per-user access control⚠️ Possible but hard to manage✅ IAM policies on users/groups (Project 2)
Cross-account access✅ Must have bucket policy allowing✅ Must also have IAM policy in other account
Grant AWS service access (S3 replication, CloudTrail)✅ Service principal in bucket policy❌ Services don't have IAM users
Temporary access for contractors❌ Bucket policy is persistent✅ IAM user with time-limited credentials
Block public access as override✅ Block Public Access settings❌ IAM can't override bucket-level access

Decision Framework:

Question 1: Does the rule apply to the RESOURCE regardless of who accesses it?
  YES → Bucket Policy (encryption, HTTPS, VPC, IP restrictions)
  
Question 2: Does the rule apply to a specific IDENTITY?
  YES → IAM Policy (per-user, per-group, per-role permissions)
  
Question 3: Does the rule involve cross-account access?
  YES → BOTH (bucket policy allows the account, IAM policy allows the user)
  
Question 4: Does the rule involve AWS service access?
  YES → Bucket Policy (service principal)
  
Best Practice: Use BOTH for defense in depth
  IAM: Controls what identities CAN do
  Bucket: Controls what the resource ALLOWS
  Block Public Access: Prevents accidental public exposure

🔴 WHEN NOT TO USE

→ Do NOT use bucket policies for fine-grained per-user permissions → Use IAM policies (Project 2) + Groups
→ Do NOT use bucket policies as the ONLY security mechanism → Always combine with IAM policies
→ Do NOT use ACLs for anything → Use bucket policies instead
→ Do NOT use bucket policies to replace VPC-level security → VPC endpoints (Project 27) + bucket policy conditions together
→ Do NOT use bucket policies for temporary access grants → Use IAM policies with time conditions or presigned URLs
→ Do NOT use bucket policies for complex organizational hierarchies → Use IAM Groups + AWS Organizations SCPs (Project 54)

1 → THE COMPLETE S3 ACCESS EVALUATION FLOW

S3 Request arrives (e.g., GetObject)
        │
        ▼

        │
        ├── Signed request (SigV4 — Project 2)?
        │   → Verify signature → Identify IAM principal
        │

        │   → Verify signature + expiration → Identify original signer
        │

            → Principal = anonymous ("*")
        │
        ▼
Step 2: BLOCK PUBLIC ACCESS CHECK
        │
        ├── Account-level Block Public Access enabled?
        │   → Is this a public access request? → DENIED
        │
        └── Bucket-level Block Public Access enabled?
            → Is this a public access request? → DENIED
        │
        ▼
Step 3: GATHER ALL APPLICABLE POLICIES
        │
        ├── Identity-based policies (IAM policies on the caller)
        ├── Resource-based policies (S3 bucket policy)
        ├── Permission boundaries (if set on the caller — Project 2)
        ├── Session policies (if using temporary credentials)
        ├── Service Control Policies (if in AWS Organization — Project 54)
        └── VPC endpoint policies (if request through VPC endpoint — Project 27)
        │
        ▼
Step 4: EVALUATE — EXPLICIT DENY CHECK
        │
        ├── Any policy has an explicit Deny that matches?
        │   → DENIED (game over — nothing overrides explicit deny)
        │
        └── No explicit deny → Continue
        │
        ▼
Step 5: EVALUATE — EXPLICIT ALLOW CHECK
        │
        ├── SAME ACCOUNT:
        │   Any policy (IAM OR bucket) has explicit Allow?
        │   → Check permission boundary (must also allow)
        │   → Check SCP (must also allow)
        │   → If all checks pass → ALLOWED
        │
        └── CROSS ACCOUNT:
            BOTH IAM policy AND bucket policy must explicitly allow
            → Check permission boundary
            → Check SCP
            → If all checks pass → ALLOWED
        │
        ▼
Step 6: NO EXPLICIT ALLOW FOUND
        → DENIED (implicit deny — default)

→ This flow executes in MILLISECONDS for every single S3 API call
→ Block Public Access is checked EARLY — before policy evaluation
→ Explicit Deny is checked BEFORE Allow — deny always wins
→ The cross-account vs same-account distinction is evaluated at Step 5

2 → HOW BLOCK PUBLIC ACCESS WORKS INTERNALLY

What S3 Considers "Public":

→ S3 analyzes your bucket policy and ACLs to determine if they grant public access
→ "Public" means: access to anyone (Principal "*") without restrictive conditions

POLICY ANALYSIS ENGINE:

Bucket policy statement:
  Principal: "*"
  Action: s3:GetObject
  Condition: None

Analysis: This grants PUBLIC access (unrestricted "*" with no conditions)
→ If Block Public Access is ON → This policy is BLOCKED

Bucket policy statement:
  Principal: "*"
  Action: s3:GetObject
  Condition:
    StringEquals:
      aws:SourceVpc: "vpc-0abc123"

Analysis: This is NOT considered public (restricted by VPC condition)
→ Block Public Access does NOT block this

Bucket policy statement:
  Principal: "arn:aws:iam::123456789012:user/priya-dev"
  Action: s3:GetObject

Analysis: This is NOT public (specific principal, not "*")
→ Block Public Access does NOT block this

The Analysis Logic:

PrincipalConditionsConsidered Public?Blocked by BPA?
"*"NoneYESYES
"*"aws:SourceIpDepends on IP rangePartial
"*"aws:SourceVpcNO (VPC is specific)NO
"*"aws:SourceAccountNO (account is specific)NO
"AWS": "*"NoneYESYES
Specific IAM ARNAnyNONO
Specific account ARNAnyNONO

→ Block Public Access uses heuristics to determine "public"
→ VPC-restricted "*" is NOT considered public → allowed even with BPA
→ Unrestricted "*" IS considered public → blocked by BPA

4 → HOW POLICY CONFLICTS ARE RESOLVED

Scenario: Multiple Statements with Conflicting Effects

Statement 1: Allow Principal: priya-dev, Action: s3:*, Resource: bucket/*
Statement 2: Deny Principal: "*", Action: s3:PutObject, Resource: bucket/* (without encryption)

priya-dev tries to upload without encryption:
  Statement 1: ALLOW (s3:* includes PutObject, priya-dev matches)
  Statement 2: DENY (Principal "*" matches everyone, condition met — no encryption)
  
  RESULT: DENIED (Explicit Deny wins over Explicit Allow — ALWAYS)

Scenario: Allow in Bucket Policy, Deny in IAM Policy

Bucket Policy: Allow Principal: priya-dev, Action: s3:GetObject
IAM Policy on priya-dev: Deny Action: s3:GetObject on this bucket

priya-dev tries to GetObject:
  Bucket Policy: ALLOW
  IAM Policy: DENY
  
  RESULT: DENIED (Explicit Deny in ANY policy wins)

The Universal Rule (Repeated from Project 2):

EXPLICIT DENY > EXPLICIT ALLOW > IMPLICIT DENY

1. Any explicit Deny ANYWHERE → DENIED
2. Any explicit Allow (with no Deny) → ALLOWED (same-account) or requires BOTH (cross-account)
3. No Allow found → DENIED (implicit deny)

5 → BUCKET POLICY vs ACL — EVALUATION ORDER

S3 Request arrives
        │
        ▼
Step 1: Block Public Access evaluated
        │
        ▼
Step 2: Bucket Policy evaluated
        │
        ├── Explicit Deny found → DENIED
        ├── Explicit Allow found → ALLOWED (if no deny elsewhere)
        └── No match → Continue
        │
        ▼
Step 3: ACLs evaluated (if ACLs are enabled)
        │
        ├── ACL grants access → ALLOWED (if no deny from bucket policy)
        └── No ACL match → Continue
        │
        ▼
Step 4: IAM Policy evaluated (for the calling identity)
        │
        ├── Explicit Deny → DENIED
        ├── Explicit Allow → ALLOWED
        └── No match → DENIED (implicit deny)

→ In practice, with ACLs disabled (recommended):

  • Block Public Access → Bucket Policy → IAM Policy → Decision
  • Much simpler evaluation path

6 → BUCKET POLICY STORAGE AND PROPAGATION

→ Bucket policies are stored as part of the bucket's metadata
→ When you save a policy:

  • S3 validates the JSON syntax
  • S3 validates the policy logic (valid principals, valid actions)
  • S3 checks against Block Public Access settings
  • If valid → policy is stored and becomes effective within seconds
  • S3 replicates the policy across its distributed infrastructure

→ Changes are EVENTUALLY CONSISTENT (usually seconds)
→ There may be a brief window where old policy is still being served
→ In practice → changes take effect within 1-5 seconds

→ Policy validation catches:

  • Invalid JSON syntax → error on save
  • Invalid principal ARNs → error on save
  • Actions that don't exist → error on save
  • Resources that don't match the bucket → error on save
  • Public access when Block Public Access is on → error on save

📈 SCALABILITY CONSIDERATIONS

→ Policy Complexity Scalability

  • Maximum policy size: 20 KB → limits number of statements
  • For many principals → use account-level (arn:aws:iam::ACCOUNT:root) instead of per-user
  • For many conditions → consolidate with wildcards and prefixes
  • For per-user control → use IAM policies (unlimited policies per account)
  • For organizational access → use aws:PrincipalOrgID condition (one statement covers all accounts in org)

→ Cross-Account Scalability

  • Each cross-account principal needs an entry in the bucket policy
  • With 50+ partner accounts → policy becomes large
  • Solution: Use AWS Organization ID condition → one statement covers all member accounts
  • Solution: Use S3 Access Points (each access point has its own policy — 10 KB limit each)

→ S3 Access Points (Scalability Solution):

  • Create separate access points for different use cases
  • Each access point has its own name, policy, and network configuration
  • Example: shopnow-bucket-web-ap for web server, shopnow-bucket-logistics-ap for LogiShip
  • Each access point can have VPC-only access → no internet access
  • Up to 10,000 access points per region per account

→ Evaluation Performance

  • S3 evaluates policies in milliseconds — no noticeable latency impact
  • Complex policies (many conditions) add negligible evaluation time
  • S3 handles billions of requests per day with policy evaluation

→ Audit Scalability

  • CloudTrail logs every S3 API call (who, what, when, from where)
  • S3 Server Access Logging provides request-level detail
  • For high-volume buckets → CloudTrail logs in S3 → Athena queries (Project 18) for analysis
  • IAM Access Analyzer identifies resources shared externally

🎯 KEY TAKEAWAY

→ Bucket policies are RESOURCE-BASED policies — they control what the BUCKET allows regardless of who's asking
→ IAM policies are IDENTITY-BASED policies — they control what the IDENTITY can do regardless of which resource
→ SAME-ACCOUNT: Either policy allowing is sufficient (unless explicit deny exists)
→ CROSS-ACCOUNT: BOTH policies must explicitly allow (both sides must agree)
→ EXPLICIT DENY always wins — in any policy, from any source
→ Block Public Access is the ultimate safety net — overrides policies and ACLs
→ ACLs are legacy — disable them and use bucket policies + IAM policies exclusively
→ Deny + condition pattern ENFORCES requirements (encryption, HTTPS, VPC, IP)
→ Bucket policies have a 20 KB limit — use IAM for per-user rules, bucket policy for resource-level rules
→ The Principal field is what makes bucket policies different from IAM policies
