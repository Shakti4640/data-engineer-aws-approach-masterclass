STEP 1 — UNDERSTAND WHAT JOB BOOKMARKS ARE

→ A Job Bookmark is a PERSISTENCE MECHANISM built into AWS Glue
→ It stores a "checkpoint" after each successful job run
→ On the next run — Glue reads the checkpoint and processes only data AFTER that point

Conceptual Model:

Run 1 (First Ever):
    → Bookmark: "No previous checkpoint"
    → Glue reads ALL data from source
    → Processes everything
    → Bookmark saved: "Processed up to record X / file Y / timestamp Z"

Run 2:
    → Bookmark: "Last processed up to X"
    → Glue reads ONLY data AFTER X
    → Processes only new data
    → Bookmark updated: "Processed up to record X+N"

Run 3:
    → Bookmark: "Last processed up to X+N"
    → Glue reads ONLY data AFTER X+N
    → And so on...

→ The bookmark is stored INTERNALLY by AWS Glue — you do NOT manage a file or database for this
→ It is tied to the specific Glue Job name + Job Run
→ It survives job restarts, account reboots, and console refreshes

STEP 2 — UNDERSTAND THE TWO SOURCE TYPES AND THEIR BOOKMARK BEHAVIOR

Glue supports bookmarks for TWO types of sources — and they work COMPLETELY DIFFERENTLY:

SOURCE TYPE A: S3 (File-Based Bookmark)

How Glue determines "new" for S3:
→ Glue tracks which FILES it has already processed
→ It uses a combination of:

  • File path (S3 key)
  • File modification timestamp
  • File size

Behavior:

ScenarioWhat Glue Does
New file added to S3 path since last run✅ Processes the new file
Existing file modified (overwritten) since last run✅ Processes the modified file
Existing file unchanged since last run❌ Skips — already processed
File deleted since last run❌ Ignores — bookmark doesn't handle deletes

Example:

Run 1 (Day 1):
    S3 Input: transactions/file1.csv, transactions/file2.csv
    Glue processes: file1.csv, file2.csv
    Bookmark saved: {file1.csv: processed, file2.csv: processed}

Run 2 (Day 2):
    S3 Input: transactions/file1.csv, file2.csv, file3.csv (NEW)
    Glue processes: file3.csv ONLY
    Bookmark updated: {file1.csv, file2.csv, file3.csv: all processed}

Run 3 (Day 3):
    S3 Input: file1.csv (MODIFIED), file2.csv, file3.csv, file4.csv (NEW)
    Glue processes: file1.csv (re-processes because modified) + file4.csv (new)
    Bookmark updated accordingly

→ S3 bookmarks are FILE-LEVEL — not record-level
→ If a file has 1 million records and 1 record changes → the ENTIRE file is reprocessed
→ This is why partitioned file structures matter (from Project 25)

SOURCE TYPE B: JDBC (Database — MariaDB, PostgreSQL, MySQL, etc.)

How Glue determines "new" for JDBC sources:
→ Glue tracks using a BOOKMARK KEY — a column in the database table
→ The bookmark key must be a column that is:

  • Monotonically increasing (always goes up, never goes down)
  • Sequential or timestamp-based
  • NOT updated on existing records (or updates would be missed)

Common Bookmark Key Choices:

Column TypeExampleWorks as Bookmark Key?Why
Auto-increment primary keytransaction_id INT AUTO_INCREMENT✅ Yes — always increasesNew records always have higher IDs
Created timestampcreated_at TIMESTAMP DEFAULT NOW()✅ Yes — always increasesNew records have later timestamps
Updated timestampupdated_at TIMESTAMP ON UPDATE NOW()⚠️ Partially — catches updates but may reprocessUpdated records get new timestamps
Status columnstatus ENUM('new','processed')❌ No — not monotonically increasingValues go back and forth
Random UUIDid UUID❌ No — not sequentialNo ordering possible
Name or text columncustomer_name VARCHAR❌ No — not orderedAlphabetical order is meaningless for "new"

Behavior with JDBC Bookmark:

Table: transactions
Bookmark Key: transaction_id (AUTO_INCREMENT)

Run 1 (Day 1):
    SELECT * FROM transactions WHERE transaction_id > 0
    → Reads records 1 through 10,000
    → Bookmark saved: {transaction_id: 10000}

Run 2 (Day 2):
    SELECT * FROM transactions WHERE transaction_id > 10000
    → Reads records 10,001 through 20,000 ONLY
    → Bookmark updated: {transaction_id: 20000}

Run 3 (Day 3):
    SELECT * FROM transactions WHERE transaction_id > 20000
    → Reads records 20,001 through 30,000 ONLY
    → Bookmark updated: {transaction_id: 30000}

→ Glue AUTOMATICALLY generates the WHERE clause based on the bookmark
→ You do NOT write the WHERE clause yourself — Glue injects it
→ The MariaDB only returns new records → less data transferred → less processing time

Understanding Bookmark Key Selection:

PayScale's MariaDB transactions table:

Best Bookmark Key: transaction_id

Criteriontransaction_idcreated_atupdated_at
Monotonically increasing?✅ Yes (AUTO_INCREMENT)✅ Yes (always later)⚠️ Yes but changes on updates
Unique per record?✅ Yes (primary key)❌ No (same timestamp possible)❌ No

Decision Matrix:

RequirementBest Bookmark Key
Only need to detect NEW records (INSERTs)transaction_id (simplest, most reliable)
Need to detect NEW + UPDATED recordsupdated_at (but will reprocess updated records)
Need to detect NEW + UPDATED + DELETEDBookmark CANNOT handle this → use CDC (Project 58 — DMS)

→ PayScale's fraud scoring only needs NEW transactions → Use transaction_id

Specifying the Bookmark Key:

→ In the Glue job script → The bookmark key is configured within the DynamicFrame creation context
→ Since we are configuration-focused (no code) → this is set through:

Job Parameters (Key-Value pairs in Job Details):

KeyValuePurpose
--job-bookmark-optionjob-bookmark-enableEnables bookmark (same as UI toggle)

→ The bookmark key itself is specified in the TRANSFORMATION CONTEXT within the job script
→ For JDBC sources, Glue needs to know which column(s) to use for bookmarking

Bookmark Key Configuration for JDBC:

→ The transformation_ctx parameter in the DynamicFrame reader is what links a specific data read to a specific bookmark
→ Each transformation_ctx string must be UNIQUE within the job
→ Glue stores the bookmark AGAINST this context string

How Glue selects the bookmark key for JDBC automatically:

ConditionWhat Glue Uses as Bookmark Key
Table has a primary key columnUses the primary key automatically
Table has multiple primary key columns (composite key)Uses all columns in the composite key
Table has no primary key❌ Bookmark DOES NOT WORK — falls back to full load
You explicitly specify bookmark keysUses your specified columns (overrides auto-detection)

→ PayScale's transactions table has transaction_id as PRIMARY KEY
→ Glue will automatically use it as the bookmark key ✅
→ No additional configuration needed IF the primary key is the correct incremental column

What if the primary key is NOT suitable (e.g., UUID)?
→ You must explicitly specify bookmark keys in the job script
→ The column must still be monotonically increasing
→ If no suitable column exists → bookmarks cannot help → consider adding a created_at timestamp column

STEP 5 — CONFIGURE S3 SOURCE BOOKMARK BEHAVIOR

→ PayScale also has a second Glue job that reads CSV files from an S3 landing zone
→ External partners drop daily CSV files into s3://payscale-landing/transactions/

S3 Bookmark Configuration:

→ For S3 sources — bookmark tracking is AUTOMATIC when enabled
→ No bookmark key needed — Glue tracks by file path + modification timestamp

How it works:

S3 Path: s3://payscale-landing/transactions/

Run 1 (Day 1):
    Files present: 2025-01-01.csv, 2025-01-02.csv
    Glue processes: BOTH files
    Bookmark: {2025-01-01.csv: timestamp_A, 2025-01-02.csv: timestamp_B}

Run 2 (Day 2):
    Files present: 2025-01-01.csv, 2025-01-02.csv, 2025-01-03.csv (NEW)
    Glue checks bookmark:
        → 2025-01-01.csv: same timestamp → SKIP
        → 2025-01-02.csv: same timestamp → SKIP
        → 2025-01-03.csv: NOT in bookmark → PROCESS ✅
    Bookmark updated: {all three files tracked}

Critical S3 Bookmark Behaviors:

ScenarioWhat HappensWhy
New file added✅ ProcessedNot in bookmark
Existing file overwritten (new timestamp)✅ Re-processedTimestamp changed — Glue detects modification
Existing file content changed but timestamp same❌ SkippedGlue uses timestamp, not content hash
File deleted from S3❌ No actionBookmark doesn't track deletions
File renamed✅ Processed as new fileNew key = new file in S3 terms

S3 File Ordering:
→ Glue processes S3 files in LEXICOGRAPHIC ORDER of the S3 key
→ This means: 2025-01-02.csv is processed BEFORE 2025-01-10.csv (because "0" < "1" character-wise)
→ But 2025-1-2.csv would be processed AFTER 2025-1-10.csv (because "2" > "1")
→ Always use zero-padded dates: 2025-01-02 not 2025-1-2
→ File ordering affects bookmark checkpointing — if a job fails mid-way, it resumes from the last successfully processed file in order

STEP 6 — UNDERSTAND BOOKMARK STORAGE AND PERSISTENCE

Where is the bookmark stored?

→ Glue stores bookmarks in an INTERNAL metadata store managed by AWS
→ You CANNOT directly read or edit the bookmark value
→ You can only:

  • Reset the bookmark (start over)
  • Pause the bookmark (process all but don't advance)
  • Enable/disable the bookmark

Bookmark Lifecycle:

Job Created → Bookmark: Empty (no history)
    |
    ↓
Run 1 (Success) → Bookmark: Checkpoint A saved
    |
    ↓
Run 2 (Success) → Bookmark: Checkpoint B saved (replaces A)
    |
    ↓
Run 3 (FAILURE mid-way) → Bookmark: Checkpoint B remains (NOT updated)
    |                        → Next run will re-process from B
    ↓
Run 4 (Success) → Bookmark: Checkpoint C saved
    |
    ↓
Manual Reset → Bookmark: Cleared → Next run processes ALL data again

Critical behavior on failure:
→ If a job run FAILS → the bookmark is NOT updated
→ The next run re-processes the same data the failed run was working on
→ This is EXACTLY CORRECT behavior — prevents data loss
→ But it means: failed runs + successful retries may produce DUPLICATE output files in S3
→ Downstream consumers (Athena, Redshift) must handle or deduplicate

STEP 7 — BOOKMARK PAUSE, RESUME, AND RESET OPERATIONS

Navigate to: Glue Console → Jobs → [Job Name] → Actions

RESET BOOKMARK:

SettingWhat It DoesWhen to Use
Reset bookmarkClears ALL bookmark history — next run processes everything from scratchAfter schema changes, after data corrections, when you need a full reload

→ Navigate to Glue Console → Jobs → Select Job → Actions → Reset job bookmark
→ Confirm → Bookmark is cleared

Use Cases for Reset:

ScenarioAction
Source table schema changed (new columns added)Reset → Full reprocessing with new schema
Historical data was corrected/backfilledReset → Reprocess everything to capture corrections
Output S3 path was accidentally deletedReset → Regenerate all output files
Bookmark got corrupted (rare)Reset → Start fresh
Moving to a new S3 output pathReset → Ensure all data lands in new location

PAUSE BOOKMARK:

SettingWhat It DoesWhen to Use
Pause (set bookmark option to "pause")Job processes ALL data but does NOT save a new bookmark checkpointTesting, validation, debugging

Example:

Current bookmark: transaction_id = 50,000

Run with PAUSE:
    → Reads ALL records from transaction_id > 0 (ignores bookmark)
    → Processes everything
    → Bookmark STILL says: transaction_id = 50,000 (not updated)

Next run with ENABLE:
    → Reads records from transaction_id > 50,000 (bookmark unchanged)
    → Processes only new records since 50,000

→ Pause is useful when:

  • You want to test a full reprocessing without losing your bookmark position
  • You want to validate output quality on the full dataset
  • You want to run a one-time report on all historical data

STEP 8 — HANDLE MULTIPLE SOURCES IN ONE JOB

→ PayScale's Glue job reads from TWO sources:

  1. MariaDB JDBC (transactions table)
  2. S3 CSV (partner transaction files)

→ Each source has its OWN bookmark tracked INDEPENDENTLY

How this works:

→ Each DynamicFrame read operation has a transformation_ctx parameter
→ This is a STRING that uniquely identifies the read operation
→ Glue maintains a SEPARATE bookmark for each transformation_ctx

Configuration:

Read Operationtransformation_ctxBookmark Tracks
Read from MariaDB transactions table"ctx_jdbc_transactions"transaction_id checkpoint
Read from S3 partner files"ctx_s3_partner_files"File paths + timestamps

→ If you use the SAME transformation_ctx for two different reads → bookmarks COLLIDE → unpredictable behavior
→ Always use UNIQUE, DESCRIPTIVE context strings

What happens with multiple contexts on bookmark reset:
→ Resetting the job bookmark clears ALL transformation contexts
→ You CANNOT reset just one context and keep the other
→ This is a limitation — if you need to reset only the JDBC source, you must reset both

STEP 9 — CONFIGURE OUTPUT TO AVOID DUPLICATES

The Duplicate Problem:

Run 1: Processes records 1-10,000 → Writes: s3://output/part-00001.parquet
Run 2: Processes records 10,001-20,000 → Writes: s3://output/part-00001.parquet (DIFFERENT content)

→ Wait — if both runs write to the same path → Run 2 OVERWRITES Run 1's output
→ This means records 1-10,000 are LOST from S3

Solution: Partition Output by Run Date or Bookmark Range

Approach 1 — Date-based partitioning (Recommended):

RunOutput PathContent
Run 1 (2025-01-01)s3://output/date=2025-01-01/part-00001.parquetRecords 1-10,000
Run 2 (2025-01-02)s3://output/date=2025-01-02/part-00001.parquetRecords 10,001-20,000
Run 3 (2025-01-03)s3://output/date=2025-01-03/part-00001.parquetRecords 20,001-30,000

→ Each run writes to a UNIQUE partition → No overwrites → No duplicates
→ Athena (from Project 18, 25) can use partition projection on the date column for efficient querying

Approach 2 — Append mode with unique file names:

→ Glue generates unique file names (part-00001, part-00002, etc.) per run
→ If writing to the SAME path → files accumulate
→ But if a job FAILS and RETRIES → the retry writes duplicate files
→ Less clean than date-based partitioning

PayScale's Configuration:

→ Output S3 path includes a date partition derived from the job run timestamp
→ Glue Workflow (from Project 31) passes the run date as a job parameter

Job ParameterValuePurpose
--run_date2025-01-15 (passed by scheduler/workflow)Used to construct output partition path

→ Output path: s3://payscale-curated/fraud-transactions/date=2025-01-15/

STEP 10 — HANDLING SCHEMA EVOLUTION WITH BOOKMARKS

Scenario: MariaDB table gets a new column

→ On Day 100, the development team adds a column risk_score DECIMAL(5,2) to the transactions table
→ The Glue job has been running for 100 days with bookmark enabled

What happens:

RunSchemaBookmark BehaviorOutput
Run 1-996 columns (no risk_score)Normal incrementalParquet with 6 columns
Run 1007 columns (risk_score added)Normal incremental — only reads new recordsParquet with 7 columns

→ Records before Day 100 have 6-column Parquet files
→ Records from Day 100 onward have 7-column Parquet files
→ Athena handles this gracefully — missing columns return NULL
→ Redshift Spectrum handles this too — schema-on-read

BUT — what if you need the new column for ALL historical records?
→ Reset the bookmark → Full reprocessing → All Parquet files regenerated with 7 columns
→ Delete old output files first to avoid duplicates
→ This is the trade-off: bookmark reset = full cost but complete schema consistency

What about column REMOVAL or TYPE CHANGE?

ChangeBookmark ImpactAction Needed
Column addedNo bookmark impact — new records have new columnNo action (or reset for historical consistency)
Column removedBookmark still works — new records just don't have the columnNo action (or reset)
Column type changed (e.g., INT → BIGINT)May cause DynamicFrame errors if types are incompatibleReset bookmark + update job mapping
Column renamedGlue treats it as a new column (old name missing, new name added)Reset bookmark + update job mapping
Table renamedBookmark is tied to the old table name → new table = fresh startUpdate job source configuration

✅ WHAT SHOULD DO

→ Always enable Job Bookmarks for any recurring/scheduled Glue ETL job
→ Always verify the bookmark key column is monotonically increasing before relying on JDBC bookmarks
→ Always use PRIMARY KEY columns as bookmark keys when possible — Glue auto-detects them
→ Always use UNIQUE transformation_ctx strings for each DynamicFrame read operation
→ Always partition output by date/run to prevent overwrites and enable efficient querying
→ Always test with "Pause" mode first to verify behavior without advancing the bookmark
→ Always reset bookmark after schema changes that affect historical data
→ Always monitor bookmark behavior through CloudWatch metrics and job run details
→ Always handle the "failed run + retry = potential duplicates" scenario in downstream consumers
→ Always use zero-padded date formats in S3 keys for correct lexicographic ordering
→ Always document the bookmark key column for each Glue job — future engineers need to know what drives incrementality

❌ WHAT SHOULD NOT DO

→ Never assume bookmark works without a primary key on JDBC sources — it silently falls back to full load
→ Never use non-monotonic columns (status, name, UUID) as bookmark keys — produces incorrect incremental behavior
→ Never use the same transformation_ctx for different read operations — bookmark collision causes data loss or duplication
→ Never rely on bookmark for detecting DELETED records — bookmarks only track forward progress, not removals
→ Never rely on bookmark for detecting UPDATED records when using auto-increment PK — updates don't change the PK
→ Never reset bookmark without understanding the impact — full reprocessing may take hours and cost significantly more
→ Never forget that bookmark reset clears ALL transformation contexts — you cannot selectively reset one source
→ Never write incremental output to a non-partitioned path — overwrites or duplicates will occur
→ Never ignore failed job runs — the bookmark doesn't advance on failure, but partial output files may exist in S3
→ Never change the bookmark key column on an existing job without resetting — mismatched checkpoints cause skipped or duplicated data
→ Never assume S3 bookmark tracks content changes — it tracks file timestamp only

🔵 WHEN TO USE GLUE JOB BOOKMARKS

ScenarioUse Bookmarks?Why
Daily incremental load from MariaDB to S3✅ YesOnly process new records — saves cost and time
Hourly processing of new S3 files from a landing zone✅ YesOnly process newly arrived files
One-time full data migration❌ NoNo need for incrementality — disable bookmark
Detecting UPDATED records in source⚠️ PartiallyOnly if bookmark key is updated_at timestamp — not with auto-increment PK
Detecting DELETED records in source❌ NoBookmarks cannot track deletions — use CDC (DMS, Project 58)
Source has no suitable monotonic column❌ NoBookmark will silently do full load every time
Job reads from multiple sources with different schemas✅ YesUse separate transformation_ctx per source
Job writes to Redshift directly (not S3)✅ YesBookmark still tracks source reading — destination doesn't matter
Data needs exact deduplication guarantee❌ Not sufficientBookmark prevents most duplicates but failed retries can cause some — add dedup logic

🔴 WHEN NOT TO USE GLUE JOB BOOKMARKS

→ When you need Change Data Capture (INSERT + UPDATE + DELETE tracking) → Use DMS (Project 58)
→ When the source has no monotonically increasing column → Bookmark falls back to full load silently
→ When you need exactly-once processing guarantees → Bookmarks provide at-least-once (retries may duplicate)
→ When you need to reprocess specific date ranges selectively → Bookmarks are all-or-nothing (reset = full reprocess)
→ When source data is modified in place (same file overwritten with corrections) → Bookmark may miss changes if timestamp doesn't change
→ When the job is a one-time operation → Disable bookmark to avoid unnecessary overhead

2 → JDBC BOOKMARK INTERNAL QUERY GENERATION

What Glue actually does internally for MariaDB:

Bookmark State: {transaction_id: 50000}
Table: transactions
Bookmark Key: transaction_id (auto-detected from PRIMARY KEY)

Glue internally generates:
    SELECT * FROM transactions 
    WHERE transaction_id > 50000 
    ORDER BY transaction_id ASC

→ This is pushed down to MariaDB as a SQL query
→ MariaDB uses the PRIMARY KEY index to efficiently find records > 50000
→ Only matching records are transferred over JDBC
→ Network transfer is minimal — only new data crosses the wire

For composite primary keys:

Table: order_items
Primary Key: (order_id, item_id)

Bookmark State: {order_id: 1000, item_id: 5}

Glue internally generates:
    SELECT * FROM order_items
    WHERE (order_id > 1000) 
       OR (order_id = 1000 AND item_id > 5)
    ORDER BY order_id ASC, item_id ASC

→ Composite keys use lexicographic comparison
→ This ensures correct incremental behavior even with multi-column keys

For timestamp-based bookmark keys:

Bookmark Key: created_at (TIMESTAMP)
Bookmark State: {created_at: '2025-01-14 23:59:59'}

Glue internally generates:
    SELECT * FROM transactions
    WHERE created_at > '2025-01-14 23:59:59'
    ORDER BY created_at ASC

→ WARNING: If multiple records have the EXACT same timestamp (same second)
→ Some may be in the previous batch, some in the next
→ Timestamp precision matters — DATETIME(6) with microseconds is safer than DATETIME

    └── File key NOT in bookmark → PROCESS (new file)

S3 Listing Behavior:
→ Glue performs a LIST operation on the S3 path
→ For paths with millions of files → listing itself can take minutes
→ This is why partitioned S3 paths (year/month/day) are important
→ If the Glue job reads from s3://bucket/data/ with 10 million files → listing takes 10+ minutes before any processing starts
→ If partitioned as s3://bucket/data/year=2025/month=01/day=15/ and the crawler/job targets a specific partition → listing is fast

4 → BOOKMARK AND JOB FAILURE INTERACTION

Run N starts
    |
    ↓
Reads bookmark: checkpoint = 50000
    |
    ↓
Reads records 50001-60000 from MariaDB
    |
    ↓
Processes records... writing to S3...
    |
    ↓ (At record 55000 — JOB FAILS)
    |
    ↓
Output in S3: Records 50001-55000 (PARTIAL — some files written)
Bookmark: STILL at 50000 (NOT updated because job failed)
    |
    ↓
Run N+1 starts (retry)
    |
    ↓
Reads bookmark: checkpoint = 50000 (unchanged)
    |
    ↓
Reads records 50001-60000 from MariaDB (SAME as failed run)
    |
    ↓
Processes ALL records 50001-60000
    |
    ↓
Output in S3: Records 50001-60000 (COMPLETE)
BUT: Records 50001-55000 exist TWICE in S3 (from failed run + retry)
Bookmark: Updated to 60000

This is the AT-LEAST-ONCE delivery problem:
→ Bookmarks guarantee NO data is MISSED
→ But they do NOT guarantee no DUPLICATES on failure/retry
→ The partial output from the failed run remains in S3

Mitigation Strategies:

StrategyHowComplexity
Partition by run date (from Step 9)Failed run's output is in a different partition than retryLow
Delete partial output before retryJob starts by cleaning its output partitionMedium
Idempotent writesUse Glue's overwrite mode per partitionMedium
Downstream deduplicationAthena/Redshift query with DISTINCT or ROW_NUMBER()Low

5 → BOOKMARK VERSUS GLUE CRAWLER INTERACTION

→ Glue Crawler (from Project 17) catalogs S3 data into the Glue Data Catalog
→ When bookmark-enabled Glue job writes NEW Parquet files to S3:

  • The Crawler must run AFTER the ETL job to detect new partitions
  • If Crawler runs before the ETL job completes → new data not yet cataloged
  • Use Glue Workflow (from Project 31) to sequence: ETL Job → Crawler
Glue Workflow Sequence:
    Step 1: Glue ETL Job (bookmark-enabled) → Writes new Parquet files to S3
    Step 2: Glue Crawler → Detects new files/partitions → Updates Glue Catalog
    Step 3: Athena/Redshift Spectrum can now query the new data

→ Alternatively: Use enableUpdateCatalog and partitionKeys in the Glue job itself to update the catalog without a separate Crawler run

📈 SCALABILITY CONSIDERATIONS

→ Data Volume Scalability

  • Without bookmarks: Cost and time scale with TOTAL data volume (linear growth forever)
  • With bookmarks: Cost and time scale with INCREMENTAL data volume (constant per run if daily volume is constant)
MetricWithout Bookmark (Day 365)With Bookmark (Day 365)
Records read from MariaDB3,650,00010,000
Data transferred over JDBC~3.6 GB~10 MB
Glue DPU-hours consumed10+ hours2 minutes
Daily cost$50+$0.15
Annual cost$18,000+$55
MariaDB read loadHeavy (10 hours of full scan)Minimal (index scan on PK > N)

→ S3 File Scalability

  • Bookmark tracks every processed file in its internal state
  • With millions of files → bookmark state grows large → job startup time increases
  • Mitigation: Use partitioned paths and target specific partitions instead of root paths
  • Archive old partitions to reduce file listing overhead

→ Concurrent Job Scalability

  • Each Glue job has its own independent bookmark
  • 10 jobs running concurrently → 10 independent bookmarks → no conflict
  • However: 10 jobs reading from the same MariaDB → 10 concurrent JDBC connections
  • Monitor MariaDB connection limits (from Project 45 — RDS connection considerations)

→ Bookmark State Scalability

  • JDBC bookmarks: Tiny state (just one or two column values) → No scalability concern
  • S3 bookmarks: State grows with number of files processed → Can become large over months
  • Mitigation for S3: Periodically reset bookmark + archive old files to a different path

→ Cross-Account Scalability

  • Bookmarks work identically for cross-account sources
  • Glue job in Account B with JDBC connection to Account A's RDS (from Project 24, 44, 45)
  • Bookmark tracks Account A's data independently — no cross-account bookmark interference

→ How to debug bookmark issues:

Step 1: Is bookmark enabled in Job settings?
    → No → Enable it
    → Yes → Go to Step 2

Step 2: Does the source table have a PRIMARY KEY?
    → No → Add one OR explicitly specify bookmark keys
    → Yes → Go to Step 3

Step 3: Is the PRIMARY KEY monotonically increasing?
    → No (UUID, random) → Change bookmark key to auto-increment or timestamp column
    → Yes → Go to Step 4

Step 4: Is transformation_ctx unique per read operation?
    → No (same string for multiple reads) → Make them unique
    → Yes → Go to Step 5

Step 5: Was bookmark recently reset?
    → Yes → First run after reset does full load — this is expected
    → No → Go to Step 6

Step 6: Check Glue job run logs in CloudWatch
    → Look for bookmark-related log entries
    → Check if WHERE clause is being applied to JDBC query
    → Check if file filtering is happening for S3 source

🎯 KEY TAKEAWAY

→ Job Bookmarks transform Glue from a "brute force full load" tool into an efficient incremental ETL engine
→ JDBC bookmarks use a monotonically increasing column (usually PRIMARY KEY) to generate WHERE clause filters
→ S3 bookmarks track files by path + modification timestamp to skip already-processed files
→ Bookmarks are stored internally by AWS — you control them through enable/disable/pause/reset only
→ The transformation_ctx string is the critical link between a read operation and its bookmark — must be unique
→ Failed job runs do NOT advance the bookmark — this prevents data loss but can cause duplicates on retry
→ Output partitioning by date is ESSENTIAL to prevent overwrites and enable clean incremental consumption
→ Without bookmarks → cost scales with TOTAL data volume (grows forever)
→ With bookmarks → cost scales with INCREMENTAL data volume (stays constant if daily volume is constant)
