→ Challenge: Athena queries are ASYNCHRONOUS

  • You submit a query → Athena returns a Query Execution ID
  • You poll for completion → Athena returns results when ready
  • Query time: 2-30 seconds depending on data volume
  • API consumers expect synchronous response (request → wait → response)

→ Two patterns to handle this:

Pattern 1: Synchronous (Lambda waits for Athena)

Client → API Gateway → Lambda → Start Athena Query → Poll until complete → Return results

→ Pros: Simple, single request/response
→ Cons: Lambda runs for entire query duration (2-30 seconds), higher cost, API Gateway 29-second timeout

Pattern 2: Asynchronous (Two API calls)

Call 1: Client → API Gateway → Lambda → Start Athena Query → Return Query ID
Call 2: Client → API Gateway → Lambda → Check Query Status → Return results (if ready)

→ Pros: Lambda runs briefly each call, no timeout issues
→ Cons: Client must poll, more complex client implementation

→ DataPulse decision: Pattern 1 (Synchronous) for refined/aggregated queries (< 5 seconds)
→ Pattern 2 (Asynchronous) for ad-hoc complex queries (potentially > 29 seconds)
→ Pre-aggregated refined tables ensure most queries complete in 2-5 seconds

→ How caching works:

  • First request: API Gateway → Lambda → Athena → Response cached
  • Subsequent identical requests within 5 minutes: Response served from cache
  • No Lambda invocation, no Athena query — instant response
  • Dramatically reduces Athena costs and improves latency

→ Always use separate Athena workgroups per consumer type — cost isolation and query limits
→ Always enable API Gateway caching for frequently requested endpoints — massive cost and latency savings
→ Always use Lambda Proxy Integration — simplest pattern, full request context in Lambda

2 → Throttling: Token Bucket Algorithm

→ API Gateway uses a token bucket algorithm for rate limiting:

  • Bucket holds up to burst tokens
  • Tokens are added at rate tokens per second
  • Each request consumes one token
  • If bucket is empty → request is throttled (429 response)

→ Example: Rate = 10 req/s, Burst = 20

  • Bucket starts full: 20 tokens
  • 20 requests arrive simultaneously → all served (bucket now empty)
  • Tokens refill at 10/second
  • 11th request 0.5 seconds later → only 5 tokens available → served
  • If 100 requests arrive at once → 20 served, 80 throttled

2 → SPICE Data Processing

→ During import, SPICE:

  1. Reads data from source (Athena query, S3 file, Redshift query)
  2. Encodes each column independently (columnar format)
  3. Applies automatic compression (dictionary encoding, run-length encoding)
  4. Creates internal indexes for filter and aggregation operations
  5. Distributes data across SPICE cluster nodes
  6. Stores entirely in memory (with SSD spillover for very large datasets)

→ Compression ratio: Typically 10:1 to 20:1
→ 100 GB of raw CSV data → ~5-10 GB in SPICE
→ This is why SPICE capacity is measured differently than raw data size

5 → SPICE Refresh Internals

→ Full Refresh:

  • SPICE runs the source query (Athena query against refined_db)
  • Downloads ALL data
  • Replaces the entire in-memory dataset atomically
  • During refresh: Users see OLD data until refresh completes
  • After refresh: Users instantly see NEW data (no partial state)

→ Incremental Refresh (Enterprise):

  • SPICE runs a filtered query (e.g., WHERE date > last_refresh_date)
  • Downloads only NEW/CHANGED data
  • Merges with existing SPICE data
  • Faster than full refresh for large datasets with small daily additions

→ Refresh failure handling:

  • If refresh fails → old data remains (users see stale but valid data)
  • QuickSight sends failure notification to configured email/SNS
  • Common failure causes: Athena timeout, S3 permission change, schema change

2 → JWT Token Validation at API Gateway

→ API Gateway does NOT call Cognito to validate each token
→ Instead, it uses JWT signature verification:

Cognito signs tokens with RSA private key
           │
           ▼
API Gateway has Cognito's public key (JWKS - JSON Web Key Set)
           │
           ▼
For each request:
  1. Extract JWT from Authorization header
  2. Decode JWT header → get key ID (kid)
  3. Look up public key matching kid from JWKS
  4. Verify JWT signature using public key
  5. Check exp (expiration) claim
  6. Check iss (issuer) claim matches User Pool URL
  7. Check aud (audience) claim matches App Client ID
  8. If ALL checks pass → request authorized

→ This is STATELESS validation — no database lookup, no network call to Cognito
→ Sub-millisecond authorization overhead
→ Scales infinitely — no bottleneck at Cognito

📈 SCALABILITY CONSIDERATIONS

→ API Gateway Scaling

  • API Gateway scales automatically — no provisioning
  • Default account limit: 10,000 requests/second across all APIs
  • Can be increased via service limit request
  • Regional API: Scales within the region
  • Edge-optimized API: Uses CloudFront — global scale with 400+ edge locations
  • At 1 million requests/month → ~$3.50 cost
  • At 100 million requests/month → ~$350 cost
  • Linear cost scaling with usage

→ Lambda Scaling

  • Lambda scales automatically — one invocation per concurrent request
  • Default concurrency limit: 1,000 per region per account
  • Can be increased to tens of thousands via limit request
  • Cold start consideration:
    • First invocation after idle: 200ms-2s initialization
    • Subsequent invocations: <1ms initialization (warm container reused)
    • Provisioned Concurrency: Pre-warm N containers → eliminates cold starts → costs ~$15/month per provisioned instance
  • At 10,000 concurrent API users → need ~100-500 concurrent Lambda executions
  • Lambda handles this automatically (within concurrency limit)

→ Athena Scaling

  • Athena scales automatically — parallel query execution
  • Default: 20 concurrent queries per account (DML)
  • Can be increased to 200+ via limit request
  • Each query independently scaled — more data = more parallel workers
  • Cost scales with data scanned — not with query count
  • At 100,000 queries/month scanning 1 MB each → $0.50/month
  • At 100,000 queries/month scanning 1 GB each → $500/month
  • Optimization: Caching + partitioning + Parquet = minimal scan per query

→ QuickSight Scaling

  • SPICE: Up to 500 GB per dataset (Enterprise)
  • Concurrent dashboard users: Thousands (managed by AWS)
  • Reader pricing caps at $5/month → cost-predictable
  • At 1,000 readers: Maximum $5,000/month (if all very active)
  • At 1,000 readers with moderate usage: ~$1,500/month (many sessions < $5)
  • SPICE capacity: Purchased in GB increments → scales independently
