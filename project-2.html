PROJECT 2: THE INTERN WHO DELETED PRODUCTION — IAM Users, Groups, Policies & Permission Boundaries

🔴 HYPOTHETICAL PROBLEM

→ ShopNow (from Project 1) now has their S3 bucket running with product images and invoices
→ The company has 15 employees sharing ONE AWS root account login
→ Everyone logs in with the same email and password
→ Last Tuesday — an intern was exploring the AWS console
→ He saw the S3 bucket shopnow-productimages-prod-aps1
→ He thought it was a test bucket → Clicked "Delete" → Confirmed deletion
→ 6 months of product images — gone
→ The CTO screams: "WHO DID THIS?"
→ Nobody knows — because everyone used the same login — no audit trail
→ The CEO asks three questions:

  • How do we give each person their OWN login?
  • How do we RESTRICT what each person can do?
  • How do we TRACK who did what?
    → Additionally — the marketing team should ONLY see the marketing/ prefix in S3
    → The finance team should ONLY access the invoices/ prefix
    → Developers should have broader access but NEVER be able to delete buckets
    → The intern should have READ-ONLY access to everything

🟡 HOW TO APPROACH IT

→ This is an Identity and Access Management (IAM) problem
→ IAM answers THREE fundamental questions in AWS:

QuestionIAM Concept
WHO is making the request?Identity (User, Role, Federated Identity)
WHAT are they trying to do?Action (s3:GetObject, s3:DeleteBucket, ec2:StartInstances)
ON WHICH resource?Resource (specific bucket ARN, specific EC2 instance ARN)

→ The approach should be:

  • Step 1 — Secure the root account (never use it for daily work)
  • Step 2 — Create IAM users (individual identities)
  • Step 3 — Create IAM groups (logical grouping by role/department)
  • Step 4 — Create/attach IAM policies (permission documents)
  • Step 5 — Understand policy evaluation logic (how AWS decides Allow/Deny)
  • Step 6 — Understand ARN format (how resources are identified)
  • Step 7 — Understand Permission Boundaries (ceiling on permissions)

STEP 4 — UNDERSTAND IAM POLICY STRUCTURE (THE CORE OF IAM)

→ Before creating policies — you MUST understand the JSON structure
→ Every permission in AWS is defined as a JSON policy document

The Anatomy of an IAM Policy:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowReadProductImages",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::shopnow-productimages-prod-aps1",
        "arn:aws:s3:::shopnow-productimages-prod-aps1/*"
      ],
      "Condition": {
        "StringLike": {
          "s3:prefix": ["marketing/*"]
        }
      }
    }
  ]
}

Every field explained:

Field 1: Version
→ Always "2012-10-17" — this is the policy language version
→ There is an older version "2008-10-17" — NEVER use it
→ The old version doesn't support policy variables like ${aws:username}
→ This is NOT the date you created the policy

Field 2: Statement
→ An array of permission rules
→ A single policy can have multiple statements
→ Each statement is independently evaluated

Field 3: Sid (Statement ID)
→ Optional — human-readable label for the statement
→ Must be unique within the policy
→ Example: "AllowReadProductImages", "DenyBucketDeletion"
→ Has NO functional impact — purely for readability and debugging

Field 4: Effect

ValueMeaning
"Allow"Grants the specified permissions
"Deny"Explicitly blocks the specified permissions

→ There are ONLY two values — no "Maybe" or "Conditional Allow"
→ Explicit Deny ALWAYS wins over Allow (critical — explained in evaluation logic)

Field 5: Action
→ The specific AWS API operations being allowed or denied
→ Format: service:APIOperation

Example ActionWhat It Permits
s3:GetObjectDownload an object from S3
s3:PutObjectUpload an object to S3
s3:DeleteObjectDelete an object from S3
s3:DeleteBucketDelete an entire S3 bucket
s3:ListBucketList objects inside a bucket
s3:*ALL S3 actions (wildcard)
*ALL actions on ALL services (God mode)

→ Wildcards are powerful but dangerous
→ s3:Get* matches s3:GetObject, s3:GetBucketPolicy, s3:GetBucketLocation, etc.
→ Always prefer specific actions over wildcards for security

Field 6: Resource
→ The specific AWS resource(s) the action applies to
→ Uses ARN (Amazon Resource Name) format

ARN Format Deep Dive:

arn:partition:service:region:account-id:resource-type/resource-id
ComponentExampleExplanation
arnarnFixed prefix — always "arn"
partitionawsaws for standard regions, aws-cn for China, aws-us-gov for GovCloud
services3The AWS service name
region`` (empty for S3)S3 buckets are globally named, so region is omitted
account-id`` (empty for S3)S3 bucket names are globally unique, so account is omitted
resource-type/resource-idshopnow-productimages-prod-aps1The bucket name

S3 ARN Patterns — CRITICAL to understand:

ARNWhat It Refers To
arn:aws:s3:::shopnow-productimages-prod-aps1The BUCKET itself (for bucket-level operations like ListBucket)
arn:aws:s3:::shopnow-productimages-prod-aps1/*ALL OBJECTS inside the bucket (for object-level operations like GetObject)
arn:aws:s3:::shopnow-productimages-prod-aps1/invoices/*Only objects under the invoices/ prefix
arn:aws:s3:::*ALL buckets in ALL accounts (extremely broad)

→ Most common mistake: Forgetting that bucket-level and object-level ARNs are DIFFERENT
→ s3:ListBucket needs the bucket ARN (without /*)
→ s3:GetObject needs the object ARN (with /*)
→ If you mix them → "Access Denied" errors that are very hard to debug

Field 7: Condition (Optional)
→ Adds extra requirements that must be true for the statement to apply

Condition OperatorExampleMeaning
StringEquals"aws:RequestedRegion": "ap-south-1"Only allow actions in Mumbai region
StringLike"s3:prefix": ["marketing/*"]Only allow listing objects with marketing prefix
IpAddress"aws:SourceIp": "203.0.113.0/24"Only allow from specific IP range
Bool"aws:MultiFactorAuthPresent": "true"Only allow if MFA is active on the session
DateGreaterThan"aws:CurrentTime": "2025-01-01T00:00:00Z"Only allow after a specific date
StringEquals"aws:PrincipalTag/Department": "Finance"Only allow if user has tag Department=Finance

→ Conditions make policies surgical — instead of broad allow/deny, you add precision

Policy 2: Developer Access — Broad S3 but No Bucket Deletion (for developers group)

→ Statement 1: Allow all S3 actions on the ShopNow bucket
→ Statement 2: Explicitly DENY bucket deletion

  Sid: AllowS3BucketAccess
  Effect: Allow
  Action: s3:*
  Resource:

→ Statement 2 explicitly denies DeleteBucket

STEP 7 — UNDERSTAND POLICY EVALUATION LOGIC (HOW AWS DECIDES ALLOW OR DENY)

→ This is the MOST important concept in IAM
→ When any IAM entity makes an API call, AWS evaluates permissions in this EXACT order:

START
  │
  ▼
Is there an EXPLICIT DENY in any applicable policy?
  │
  ├── YES → DENIED (game over — nothing overrides explicit deny)
  │
  └── NO ──▶ Is there an EXPLICIT ALLOW in any applicable policy?
                │
                ├── YES → ALLOWED
                │
                └── NO → DENIED (implicit deny — default state)

Three States of Permission:

StateMeaningExample
Explicit AllowA policy statement says Effect: Allow for this action+resourceDeveloper policy allows s3:PutObject
Explicit DenyA policy statement says Effect: Deny for this action+resourceDeveloper policy denies s3:DeleteBucket
Implicit DenyNo policy says anything about this action+resourceNo policy mentions ec2:TerminateInstances → denied by default

The Golden Rule:
→ By default — EVERYTHING is denied (implicit deny)
→ An explicit Allow overrides implicit deny
→ An explicit Deny overrides EVERYTHING (even explicit Allow)
→ Deny > Allow > Implicit Deny

Real Scenario with ShopNow:

→ priya-dev tries to delete the bucket
→ AWS checks: Policy Statement 1 says Allow s3:* → this includes s3:DeleteBucket
→ AWS checks: Policy Statement 2 says Deny s3:DeleteBucket
→ Explicit Deny found → DENIED
→ Priya cannot delete the bucket even though s3:* includes it

→ arjun-intern tries to delete an object
→ AWS checks: ReadOnlyAccess policy — no s3:DeleteObject action mentioned
→ No explicit Allow → Implicit Deny → DENIED
→ No explicit Deny needed — absence of Allow is enough

STEP 8 — UNDERSTANDING PERMISSION BOUNDARIES

→ Permission Boundary is a CEILING on what an IAM user or role CAN EVER have
→ Even if someone attaches AdministratorAccess to the user — the boundary limits them

How it works:

Effective Permissions = Identity Policy ∩ Permission Boundary
                        (Intersection — only what BOTH allow)

Scenario:
→ The CTO wants to let priya-dev create IAM users for her team
→ But he's worried she might create users with AdministratorAccess
→ Solution: Set a Permission Boundary on priya-dev

Configuration:
→ IAM Console → Users → priya-dev → Permissions tab → "Set permissions boundary"
→ Choose a policy that defines the MAXIMUM permissions

Permission Boundary Policy Example:

Policy Name: ShopNow-Developer-Boundary

Statement 1:
  Sid: AllowOnlyS3AndEC2
  Effect: Allow
  Action:
    - s3:*
    - ec2:*
    - iam:CreateUser
    - iam:AttachUserPolicy
  Resource: *

Statement 2:
  Sid: DenyChangingOwnBoundary
  Effect: Deny
  Action:
    - iam:DeleteUserPermissionsBoundary
    - iam:PutUserPermissionsBoundary
  Resource: arn:aws:iam::123456789012:user/priya-dev

→ Now — even if someone attaches AdministratorAccess to priya-dev
→ She can ONLY use S3, EC2, and limited IAM actions
→ She CANNOT use RDS, Redshift, Lambda, or any other service
→ She CANNOT remove her own boundary (Statement 2)

Visual:

┌─────────────────────────────────────┐
│     Permission Boundary (Ceiling)   │
│     S3 + EC2 + Limited IAM          │
│                                     │
│   ┌───────────────────────┐         │
│   │  Identity Policy      │         │
│   │  AdministratorAccess  │         │
│   │  (Allow: *)           │         │
│   └───────────────────────┘         │
│                                     │
│   Effective = ONLY S3 + EC2 + IAM   │

✅ WHAT SHOULD DO

→ Always secure root account with MFA and never use it for daily operations
→ Always create individual IAM users — one person = one user
→ Always use Groups to manage permissions — never attach policies directly to users
→ Always follow least privilege — give MINIMUM permissions needed for the job
→ Always use Customer Managed Policies for organization-specific permissions
→ Always use Permission Boundaries when delegating IAM administration
→ Always enable MFA for all human users — especially administrators
→ Always set a strong password policy at the account level
→ Always use tags on IAM users for organizational tracking
→ Always use descriptive Sids in policy statements for debugging
→ Always test policies using IAM Policy Simulator before applying
→ Always create an account alias for human-friendly sign-in URL

❌ WHAT SHOULD NOT DO

→ Never share root account credentials with anyone
→ Never create access keys for root account
→ Never attach policies directly to individual users — use groups
→ Never use * in Action AND Resource simultaneously unless it's a true admin policy
→ Never hardcode Access Key ID and Secret Access Key in application code
→ Never create one "god user" that everyone shares
→ Never forget the difference between bucket ARN and object ARN (Project 1 reference)
→ Never use inline policies for standard permissions — use managed policies
→ Never give iam:* permissions to non-admin users — they could escalate their own privileges
→ Never skip MFA for admin users — a compromised admin account = entire AWS account compromised
→ Never forget that implicit deny is the default — if nothing says Allow, it's denied
→ Never assume that removing an Allow = adding a Deny — they are different concepts

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

1 → THE IAM SERVICE ARCHITECTURE

→ IAM is a GLOBAL service — it is NOT region-specific
→ When you create an IAM user in ap-south-1 console → it's available in ALL regions
→ IAM data is replicated across all AWS regions automatically
→ This is why IAM has no region selector in the console (always shows "Global")

→ IAM is eventually consistent
→ If you create a user and immediately try to use it → might get a brief delay
→ Typically < 1 second — but in rare cases, up to a few seconds
→ This is because the change needs to propagate across all global endpoints

2 → HOW AUTHENTICATION WORKS

Console Login (Username + Password):

User enters credentials at sign-in URL
        │
        ▼
AWS IAM authentication service validates
        │
        ├── Password correct + MFA valid → Session token issued
        │
        └── Failed → "Invalid credentials" error
        
Session token = temporary credential that the browser uses for all subsequent API calls
Console session default duration: 12 hours (configurable: 1-12 hours)

API/CLI Authentication (Access Keys):

CLI command: aws s3 ls
        │
        ▼
AWS SDK reads Access Key ID + Secret Access Key from ~/.aws/credentials
        │
        ▼
SDK creates a "Signature Version 4" (SigV4) signed request
        │
        ▼
The signature includes:
  - Access Key ID (identifies WHO)
  - Timestamp (prevents replay attacks)
  - Region + Service (scope)
  - Hash of request body
  - All signed with Secret Access Key using HMAC-SHA256
        │
        ▼
AWS receives request → looks up Access Key ID → retrieves corresponding Secret Access Key
        │
        ▼
AWS recreates the signature using the same algorithm
        │
        ├── Signatures match → Authenticated → Proceed to authorization
        │
        └── Signatures don't match → 403 Forbidden

→ The Secret Access Key NEVER travels over the network
→ Only the computed signature travels
→ This is why SigV4 is secure even without HTTPS (though HTTPS is always used)

3 → HOW AUTHORIZATION WORKS (POLICY EVALUATION ENGINE)

→ After authentication succeeds → AWS determines what the caller is allowed to do

Full Evaluation Flow (Single Account):

Request: priya-dev calls s3:DeleteBucket on shopnow-productimages-prod-aps1
        │
        ▼
Step 1: Gather ALL applicable policies
        │
        ├── Identity-based policies (attached to user/groups)
        ├── Resource-based policies (S3 bucket policy — Project 10)
        ├── Permission boundaries (if set)
        ├── Session policies (if using temporary credentials)
        └── Service Control Policies (if using AWS Organizations — Project 54)
        │
        ▼
Step 2: Evaluate all policies
        │
        ▼
Step 2a: Any EXPLICIT DENY across ALL policies?
        │
        ├── YES → FINAL DECISION: DENIED
        │        (Developer policy has Deny s3:DeleteBucket → DENIED)
        │
        └── NO → Continue
                │
                ▼
Step 2b: For identity-based policies + resource-based policies:
         Is there an EXPLICIT ALLOW?
                │
                ├── YES → Check boundaries
                │         │
                │         ▼
                │   Does Permission Boundary also ALLOW this action?
                │         │
                │         ├── YES → FINAL DECISION: ALLOWED
                │         └── NO → FINAL DECISION: DENIED
                │
                └── NO → FINAL DECISION: DENIED (implicit deny)

→ This flow executes in MILLISECONDS for every single API call
→ AWS evaluates potentially dozens of policies per request
→ This is why well-structured policies matter — complex policies = slower evaluation

5 → IAM IDENTITY CENTER vs IAM USERS

→ IAM Users (what we created) are the CLASSIC approach
→ AWS now recommends IAM Identity Center (formerly AWS SSO) for human users
→ IAM Identity Center provides:

  • Single sign-on across multiple AWS accounts
  • Integration with corporate directories (Active Directory, Okta)
  • Temporary credentials automatically (no long-lived passwords/keys)

→ However — IAM Users are still fundamental to understand because:

  • IAM Roles (Project 11) are built on the same policy model
  • Service-to-service authentication uses the same concepts
  • Cross-account access (Project 40) requires understanding trust policies
  • Every AWS certification tests IAM User concepts extensively

🚨 COMMON PITFALLS AND FAILURE MODES

#PitfallWhat HappensHow to Avoid
1Using root account for daily workNo audit trail per person, maximum blast radius if compromisedCreate IAM users immediately, lock root away
2Attaching policies directly to usersUnmanageable at scale — 50 users = 50 individual policy attachments to trackAlways use groups
3Using Action: * and Resource: * for non-admin usersUser can do anything — delete databases, terminate instances, modify billingAlways specify exact actions and resources
4Confusing bucket ARN with object ARNs3:GetObject on bucket ARN → Access DeniedBucket ARN for bucket operations, bucket ARN/* for object operations

7Giving iam:* to developersDeveloper creates admin user for themselves → privilege escalationRestrict IAM actions + use Permission Boundaries

📈 SCALABILITY CONSIDERATIONS

→ User Scalability

  • 5,000 IAM users per account hard limit
  • For organizations with > 5,000 people → use IAM Identity Center or Federation (SAML/OIDC)
  • IAM Users are best for service accounts and small-to-medium teams

→ Policy Scalability

  • Maximum 10 managed policies per entity (user/group/role)
  • Maximum 6,144 characters per policy
  • For complex permissions → combine multiple policies on a group
  • For very granular access → use ABAC (Attribute-Based Access Control) with tags instead of writing one policy per user

→ Group Scalability

  • 300 groups per account
  • A user in 10 groups × 10 policies per group = 100 policies evaluated per API call
  • AWS handles this efficiently but keep policy count reasonable

→ Multi-Account Scalability

  • Single AWS account IAM works for small organizations
  • Medium/large organizations → multiple accounts (dev/staging/prod) → AWS Organizations (Project 54)
  • Cross-account roles (Project 40) enable access without creating users in every account

→ Audit Scalability

  • CloudTrail logs every IAM-authenticated API call
  • At scale → millions of API calls per day → store CloudTrail logs in S3 (Project 1) → analyze with Athena (Project 18)
  • IAM Access Analyzer automatically identifies resources shared externally

→ How to debug "Access Denied":

Step 1: Identify the EXACT API call being denied
        (Error message usually says which action was denied)

Step 2: Check all policies attached to the user (direct + via groups)

Step 3: Check resource-based policies (bucket policy for S3)

Step 4: Check Permission Boundaries if set

Step 5: Check Service Control Policies if using Organizations

Step 6: Verify ARN format — bucket vs object ARN is #1 cause

Step 7: Use Policy Simulator with the exact action + resource

Step 8: Check for explicit Deny — remember Deny always wins
