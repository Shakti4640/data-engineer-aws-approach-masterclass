PROJECT 30: THE COMPLIANCE TEAM THAT DEMANDED GEO-REDUNDANCY — Cross-Region S3 Replication

🔴 HYPOTHETICAL PROBLEM

→ ShopNow's data infrastructure has matured significantly:

  • S3 data lake: 8 TB of Parquet data across raw, transformed, operational, curated zones
  • RDS MariaDB: Multi-AZ with Read Replicas (Project 29) — database tier is resilient
  • Redshift: Loaded with optimized warehouse data (Project 22)
  • All infrastructure runs in ap-south-1 (Mumbai)

→ Then the compliance team drops a bombshell:

→ Problem 1: Regulatory Requirement — Geographic Data Redundancy

  • ShopNow processes payments → falls under PCI-DSS compliance
  • The compliance audit finding states:
    • "Critical business data must be stored in at least TWO geographically separate locations"
    • "Recovery from a regional disaster must be possible within 4 hours (RTO)"
    • "Maximum acceptable data loss in a regional disaster: 15 minutes (RPO)"
  • Currently: ALL S3 data exists ONLY in ap-south-1
  • S3's 11 9s durability is within ONE region — if the entire region is unavailable, data is inaccessible
  • The auditor's question: "If Mumbai experiences a catastrophic event — earthquake, widespread power failure, or regional AWS outage — how do you recover?"
  • ShopNow's answer today: "We can't."

→ Problem 2: The Accidental Deletion Incident

  • A junior engineer was cleaning up old test data in S3
  • They intended to delete: s3://shopnow-data-lake/raw/test-data/
  • They accidentally deleted: s3://shopnow-data-lake/raw/orders/ (3 months of raw order files)
  • Versioning (Project 12 concept) was enabled → delete markers were placed → objects still recoverable
  • But restoring 3 months of versioned objects was a manual, tedious process (90,000 files)
  • Recovery took 8 hours of engineering time
  • If versioning had NOT been enabled → data would be permanently lost
  • The CTO asks: "What if versioning was off? What if the entire BUCKET was deleted?"

→ Problem 3: Global Analyst Access

  • ShopNow expanded to the US market — 5 analysts now work from New York
  • They query the data lake via Athena (Project 25)
  • Every Athena query from US analysts scans S3 data in Mumbai
  • Latency: 200-300ms round-trip per S3 request
  • Large analytical queries that make thousands of S3 requests are noticeably slower
  • Cross-region data transfer: $0.09/GB from ap-south-1 to US
  • US analysts scanning 50 GB/day = $4.50/day = $135/month in cross-region transfer

→ Problem 4: Business Continuity Plan Gap

  • The board requires a documented Business Continuity Plan (BCP)
  • The BCP must include:
    • Where backup data is stored
    • How quickly data can be recovered
    • How operations resume in a disaster
  • Currently the BCP says: "S3 data is durable within the Mumbai region"
  • The board says: "That's not a plan — that's a hope"

→ Problem 5: Data Sovereignty Complexity

  • Indian customer data must remain in India (data residency requirement)
  • But the BCP requires a copy OUTSIDE India for disaster recovery
  • Contradiction? Not quite — the DR copy can exist in another region IF:
    • Access is restricted to DR scenarios only
    • Encryption is maintained
    • Proper legal agreements are in place
  • The replication target must be carefully chosen: Singapore (ap-southeast-1) — closest region, similar regulatory environment

→ The CISO says: "I need automated, continuous replication of our entire data lake to a second AWS region — encrypted in transit and at rest, with monitoring that guarantees RPO < 15 minutes, and the ability to fail over our analytics operations within 4 hours"

🟡 HOW TO APPROACH IT

→ First — understand what S3 Cross-Region Replication (CRR) is and isn't

AspectWhat CRR ISWhat CRR IS NOT
PurposeAutomatic, asynchronous copy of objects to another regionReal-time synchronous replication
TriggerEvery new object uploaded → automatically replicatedRetroactive — existing objects NOT replicated by default
GranularityPer-object replicationBucket-level snapshot
DirectionSource bucket → destination bucketNOT bidirectional (by default)
LatencyMost objects replicated within 15 minutesNOT instant — asynchronous
ScopeNew objects after rule is enabledExisting objects require separate "Existing Object Replication" job
VersioningREQUIRED on both source and destinationCannot work without versioning

→ Prerequisites that must be in place BEFORE configuring CRR:

  1. Versioning enabled on SOURCE bucket (Project 12)
  2. Versioning enabled on DESTINATION bucket
  3. IAM Role for S3 to perform replication
  4. Destination bucket in a DIFFERENT region

→ The approach should be:

  • Step 1 — Enable versioning on source bucket (if not already — Project 12)
  • Step 2 — Create destination bucket in target region
  • Step 3 — Create IAM Role for replication
  • Step 4 — Configure replication rules
  • Step 5 — Handle existing objects (Batch Replication)
  • Step 6 — Configure Replication Time Control (RTC) for SLA compliance
  • Step 7 — Configure replication metrics and monitoring
  • Step 8 — Configure encryption in replication
  • Step 9 — Test replication and verify
  • Step 10 — Plan for failover scenarios

Why versioning is required for CRR:
→ CRR replicates specific VERSION IDs of objects
→ Without versioning → S3 can't track which version has been replicated
→ Version ID is the replication "cursor" — S3 knows "I've replicated up to version X"

SettingOptionsRecommendation

STEP 8 — CONFIGURE ENCRYPTION IN REPLICATION

→ ShopNow uses SSE-KMS encryption (Project 36 concept)
→ CRR must handle the encryption transition between regions

Encryption Flow During Replication:

SOURCE (ap-south-1):
  Object stored encrypted with Mumbai KMS key
       ↓
S3 Replication:
  ├── S3 decrypts object using Mumbai KMS key
  │     (Replication role needs kms:Decrypt on Mumbai key)
  ├── S3 transfers decrypted data over TLS (encrypted in transit)
  └── S3 encrypts object using Singapore KMS key
        (Replication role needs kms:Encrypt on Singapore key)
       ↓
DESTINATION (ap-southeast-1):
  Object stored encrypted with Singapore KMS key

→ Key point: Objects are RE-ENCRYPTED with the destination region's KMS key
→ The source key CANNOT be used in the destination region (KMS keys are regional)

Replication Rule Encryption Configuration:

SettingValueExplanation
Replicate objects encrypted with KMSEnabledRequired for SSE-KMS objects
Source KMS keyarn:aws:kms:ap-south-1:…:key/source-key-idKey used to encrypt source objects
Destination KMS keyarn:aws:kms:ap-southeast-1:…:key/dest-key-idKey to encrypt destination objects

→ If this setting is NOT enabled → SSE-KMS encrypted objects are SKIPPED (not replicated!)
→ This is a common mistake: CRR appears to work but KMS-encrypted objects silently fail to replicate

What about SSE-S3 encrypted objects?
→ SSE-S3 objects replicate automatically — no special configuration needed
→ S3 manages the keys in both regions transparently

Encryption summary:

Source EncryptionCRR Configuration NeededDestination Encryption
SSE-S3None (automatic)SSE-S3 (destination default)
SSE-KMSMust enable KMS replication + specify keysSSE-KMS (destination KMS key)
SSE-C (customer-provided)NOT supported by CRRCannot replicate

STEP 10 — PLAN FOR FAILOVER SCENARIOS

Scenario A: Source Region Temporarily Unavailable (Hours)

ap-south-1 (Mumbai) becomes unavailable
       ↓
Analysts cannot access source bucket
       ↓
FAILOVER STEPS:
  1. Verify destination bucket is accessible
  2. Update Athena workgroup → point to destination bucket
       ├── Create Glue Catalog in ap-southeast-1
       ├── Run Crawler on destination bucket
       └── Configure Athena in Singapore region
  3. Analysts query destination data via Athena in Singapore
  4. RPO: Up to 15 minutes of data loss (last unreplicated objects)
  5. RTO: ~2-4 hours (time to set up Athena in Singapore)
       ↓
WHEN SOURCE RECOVERS:
  1. Verify source data integrity
  2. Switch analysts back to source
  3. Any objects created in destination during outage → replicate back (manual)

Scenario B: Source Bucket Accidentally Deleted

Someone deletes shopnow-data-lake bucket entirely
       ↓
RECOVERY STEPS:
  1. Destination bucket still has ALL data (delete markers not replicated)
  2. Create new source bucket with same name
  3. Enable versioning on new bucket
  4. Copy data from destination back to source
     (S3 Batch Copy or S3 Sync via CLI)
  5. Re-enable CRR rules
  6. RPO: Up to 15 minutes (RTC) or potentially longer for non-RTC prefixes
  7. RTO: 4-24 hours depending on data volume

Scenario C: Data Corruption in Source

ETL bug corrupts Parquet files in transformed/ prefix
       ↓
RECOVERY STEPS:
  1. Identify corruption timeframe
  2. In destination bucket:
     ├── Corrupted files were replicated (replication doesn't validate content)
     ├── BUT versioning preserves pre-corruption versions
     └── Retrieve previous versions of corrupted files from DESTINATION
  3. Copy correct versions back to source
  4. Re-run ETL on corrected data
  5. RPO: Depends on when corruption was detected

→ Key insight: CRR + Versioning provides the strongest protection:

  • CRR protects against regional failure
  • Versioning protects against corruption and accidental deletion
  • Together they cover almost every data loss scenario

✅ WHAT SHOULD DO

→ Always enable versioning on BOTH source and destination buckets — CRR requires it
→ Always create granular replication rules per prefix — different SLAs and storage classes per data zone
→ Always disable delete marker replication for DR buckets — preserves data even if source is accidentally deleted
→ Always enable RTC for compliance-critical data — guarantees 15-minute RPO
→ Always configure replication metrics and CloudWatch alarms — detect failures immediately
→ Always handle existing objects via Batch Replication — CRR only covers new objects
→ Always test replication with actual files — verify end-to-end before relying on it
→ Always document the failover runbook — step-by-step instructions for regional disaster
→ Always test failover quarterly — an untested DR plan is not a DR plan
→ Always use a restrictive endpoint policy on the destination bucket — DR data should not be casually accessed
→ Always configure KMS replication settings for SSE-KMS encrypted objects — otherwise they silently fail to replicate
→ Always monitor OperationsFailedReplication — even one failure means data gap in DR

❌ WHAT SHOULD NOT DO

→ Never assume CRR replicates existing objects — only NEW objects after rule creation
→ Never forget to enable versioning on both buckets — CRR will fail without it
→ Never enable delete marker replication for DR — accidental source deletion propagates to DR
→ Never skip KMS encryption configuration — SSE-KMS objects silently fail to replicate
→ Never use the same KMS key for source and destination — KMS keys are regional
→ Never rely on CRR without monitoring — silent failures create false sense of security
→ Never set RTC on ALL prefixes — RTC costs $0.015/GB; use only for critical data
→ Never use One Zone-IA for DR destination — single AZ storage defeats geographic redundancy
→ Never treat CRR as a backup solution alone — it replicates deletions too (unless delete markers disabled)
→ Never skip batch replication for existing data — your DR copy is incomplete
→ Never forget cross-region transfer costs — $0.09/GB adds up for large datasets
→ Never store DR bucket credentials or access in the same region as source — if source region fails, you need independent access

🔵 WHEN TO USE CROSS-REGION REPLICATION

ScenarioUse CRR?Why
Compliance requires geographic data redundancy✅ YesPrimary use case
Disaster recovery for critical data✅ YesRegional failure protection
Global user access (reduce latency for remote users)✅ YesData closer to users
Protection against accidental deletion (with delete markers disabled)✅ YesDR copy preserved
Cross-region analytics (Athena in multiple regions)✅ YesLocal data access in each region
Same-region redundancy❌ NoUse Same-Region Replication (SRR) instead
Real-time synchronization requirement❌ NoCRR is asynchronous (seconds to minutes lag)
Backup of transactional database❌ NoUse RDS automated backups, snapshots, or DMS
Cost-constrained environment with no compliance requirement❌ NoS3 durability (11 9s) within region may be sufficient

🔴 WHEN NOT TO USE CROSS-REGION REPLICATION

→ When compliance doesn't require geographic redundancy AND budget is tight → S3's within-region durability is sufficient
→ When data can be easily regenerated → replicating derived data (CTAS tables, Athena results) may not be worth the cost
→ When real-time sync is required → CRR lag (seconds to minutes) may be too high → consider DynamoDB Global Tables or Aurora Global Database
→ When data volume is enormous (100+ TB) and budget is constrained → cross-region transfer at $0.09/GB = $9,000+ one-time + ongoing costs

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

1 → CRR REPLICATION ENGINE

Object uploaded to source bucket:
  PutObject → s3://shopnow-data-lake/transformed/orders/year=2025/month=01/day=16/part-00000.parquet
       ↓
S3 assigns Version ID: v_abc123
       ↓
S3 checks replication rules:
  ├── Rule: "replicate-transformed-data"
  ├── Prefix filter: "transformed/" → MATCHES ✅
  ├── Status: Enabled ✅
  └── Object eligible for replication
       ↓
S3 Replication Engine (AWS-managed):
  ├── Queues object for replication
  ├── Priority: Based on rule priority and RTC status
  │
  ├── READ PHASE:
  │     ├── Assume IAM Role: S3-CrossRegion-Replication-Role
  │     ├── Read object from source bucket
  │     ├── If SSE-KMS: Decrypt using source KMS key (kms:Decrypt)
  │     └── Object content in memory/buffer
  │
  ├── TRANSFER PHASE:
  │     ├── Establish TLS connection to destination region
  │     ├── Transfer object over AWS backbone network
  │     ├── NOT over public internet — AWS private infrastructure
  │     └── Transfer is encrypted in transit (TLS)
  │
  └── WRITE PHASE:
        ├── If SSE-KMS: Encrypt using destination KMS key (kms:Encrypt)
        ├── Write object to destination bucket
        ├── Preserve: object key, metadata, tags, ACL
        ├── Version ID in destination: NEW (different from source)
        └── Replication Status on source: COMPLETED
       ↓
CloudWatch Metrics updated:
  ├── ReplicationLatency: Time from upload to replication completion
  ├── BytesPendingReplication: Decremented
  └── OperationsPendingReplication: Decremented

2 → REPLICATION TIME CONTROL (RTC) INTERNALS

WITHOUT RTC (Best Effort):
  ├── Objects queued in standard replication pipeline
  ├── Most objects replicate within minutes
  ├── No guarantee — could take hours during S3 service degradation
  └── No SLA or financial backing

WITH RTC:
  ├── Objects enter PRIORITY replication pipeline
  ├── S3 commits to 99.99% of objects replicated within 15 minutes
  ├── RTC objects get dedicated replication bandwidth
  ├── S3 monitors per-rule compliance
  │     ├── If approaching 15-minute threshold → escalate priority
  │     └── CloudWatch metric: ReplicationLatency tracks actual time
  └── Backed by S3 SLA — credits if commitment missed

RTC SLA breakdown:
→ 99.99% within 15 minutes means:

  • Out of 10,000 objects → 9,999 replicate within 15 minutes
  • 1 object MAY take longer (but typically still within 30 minutes)
  • This meets ShopNow's RPO requirement of 15 minutes

3 → VERSIONING AND REPLICATION INTERACTION

SOURCE BUCKET:
  Object: transformed/orders/part-00000.parquet
    ├── Version 1 (v_001): Original upload → REPLICATED
    ├── Version 2 (v_002): Overwrite (new ETL run) → REPLICATED
    └── Delete Marker (v_003): Object deleted → NOT replicated (disabled)

DESTINATION BUCKET:
  Object: transformed/orders/part-00000.parquet
    ├── Version 1 (v_abc): Replica of source v_001
    └── Version 2 (v_def): Replica of source v_002
    (No delete marker — object still accessible in destination!)

→ Each source version is replicated as a NEW version in the destination
→ Version IDs differ between source and destination (they're independent)
→ Delete marker replication disabled → destination retains all versions even after source deletion

Version lifecycle with replication:

Source ActionSource StateDestination State (delete markers disabled)
Upload filev1 createdv1 replicated
Overwrite filev2 created (v1 retained)v2 replicated (v1 retained)
Delete fileDelete marker (v1, v2 still exist under versions)NO delete marker — v1, v2 still directly accessible
Permanently delete v1v1 removed from sourcev1 STILL exists in destination

→ This makes the destination a SUPERSET of the source in terms of data retention
→ Even permanent deletions in source don't affect destination

4 → CROSS-REGION DATA TRANSFER PATH

Source: ap-south-1 (Mumbai)
Destination: ap-southeast-1 (Singapore)

  S3 Mumbai data center
    → AWS Mumbai backbone router
      → AWS inter-region backbone fiber
        (Mumbai → Singapore: ~3,800 km undersea cable)
      → AWS Singapore backbone router
    → S3 Singapore data center
  
  Characteristics:
  ├── PRIVATE AWS network — never touches public internet

→ The transfer is over AWS's PRIVATE global backbone
→ Not the public internet — higher bandwidth, lower latency, more reliable
→ This is why CRR is faster and more reliable than manual cross-region copy

5 → BATCH REPLICATION INTERNALS

Batch Replication job initiated
       ↓
S3 generates INVENTORY of all existing objects in source bucket
  ├── Lists all objects across all prefixes
  ├── Checks each object's replication status
  └── Identifies objects NOT yet replicated
       ↓
S3 creates a MANIFEST of objects to replicate
  ├── Each object: key, version ID, size
  └── Total manifest may contain millions of entries
       ↓
S3 Batch Operations processes the manifest:
  ├── Reads objects from source (in parallel)
  ├── Transfers to destination region
  ├── Writes to destination bucket
  └── Updates replication status on source objects
       ↓
Completion report generated:
  ├── Successful replications
  ├── Failed replications (with error reasons)
  └── Total objects/bytes processed
       ↓
Ongoing CRR takes over for new objects

→ Batch Replication is a ONE-TIME job to catch up existing objects
→ After completion → CRR handles everything going forward
→ If batch replication fails for some objects → review completion report → retry failed objects
