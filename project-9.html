PROJECT 9: THE E-COMMERCE APP THAT NEEDED A DATABASE — RDS MariaDB Instance Setup

🔴 HYPOTHETICAL PROBLEM

→ ShopNow (from Projects 1-8) now has:

  • S3 bucket for product images (Project 1)
  • IAM users with proper permissions (Project 2)
  • EC2 instances in a properly architected VPC (Projects 3, 4, 5)
  • SNS topics and SQS queues for messaging (Projects 6, 7)
  • CloudWatch monitoring and alerting (Project 8)

→ Currently — ShopNow runs MariaDB on an EC2 instance in the private data subnet (Project 5)
→ The database stores: products, customers, orders, inventory, and payment records

→ Incident 1: The database EC2 instance's EBS volume failed at 2 AM
→ Last manual backup was 3 days ago → 3 days of order data LOST
→ Recovery took 14 hours → manual restore from backup → reconfigure application connections
→ 14 hours of complete application downtime → estimated loss: ₹12,00,000

→ Incident 2: MariaDB needed a critical security patch (CVE with severity: CRITICAL)
→ Nobody on the team knew HOW to patch MariaDB without breaking it
→ The DBA had resigned 2 months ago → no database expertise remained
→ The patch sat uninstalled for 6 weeks → security audit flagged it as HIGH RISK

→ Incident 3: During Diwali sale — database CPU hit 99% due to heavy read queries
→ The team wanted to add a read replica to offload read traffic
→ Setting up MariaDB replication manually requires: binary log configuration, replica server setup, CHANGE MASTER TO command, monitoring replication lag
→ Nobody knew how → took 3 days to configure → sale was already over

→ Incident 4: The application team asked for a staging database identical to production
→ Manually creating a copy of a 500 GB database took 8 hours
→ During the copy → production database slowed down (backup I/O competing with application I/O)

→ The CTO says: "I don't want us to MANAGE a database server. I want us to USE a database. Someone else should handle:

  • Automated backups every day
  • Automated patching of the database engine
  • One-click read replicas
  • Automatic failover if hardware fails
  • Monitoring built-in
  • Encryption at rest and in transit
  • Point-in-time recovery to any second in the last 35 days"

🟡 HOW TO APPROACH IT

→ This is a MANAGED DATABASE problem
→ AWS RDS (Relational Database Service) provides fully managed relational databases
→ YOU manage: Schema, queries, application logic, connection strings
→ AWS manages: Hardware, patching, backups, replication, failover, monitoring, encryption

→ A DB Subnet Group tells RDS WHICH subnets to place the database in
→ It MUST include subnets in at least 2 different Availability Zones
→ This is required even for Single-AZ deployments (RDS enforces it for future Multi-AZ capability)

→ RDS Console → "Subnet groups" → "Create DB subnet group"

Why Private Subnets?
→ Databases should NEVER be in public subnets (Project 5 design principle)
→ Private subnets have NO route to Internet Gateway → database is unreachable from internet
→ Only instances within the VPC (via Security Group referencing — Project 4) can connect
→ This is ARCHITECTURAL security — not just firewall rules

Why 2 AZs Minimum?
→ Even for Single-AZ deployment → RDS requires 2 AZ subnets in the group
→ If you later enable Multi-AZ → RDS needs to place the standby in a different AZ
→ If you don't have 2 AZ subnets → Multi-AZ conversion fails
→ Always include all data-tier subnets across all AZs

Storage Auto-Scaling:

→ When enabled → RDS automatically increases storage when:

  • Free storage falls below 10% of allocated storage
  • Low storage condition lasts at least 5 minutes
  • At least 6 hours since last auto-scaling event

→ Storage ONLY scales UP — it NEVER scales down
→ Once auto-scaled from 100 GB to 150 GB → it stays at 150 GB
→ To reduce → you must create a new instance with smaller storage and migrate data
→ The maximum threshold prevents runaway growth (important for cost control)

How Automated Backups Work:

Daily Backup Process:
1. RDS takes a SNAPSHOT of the entire database storage volume
2. RDS captures TRANSACTION LOGS continuously throughout the day
3. Snapshot + Transaction Logs = Point-in-time recovery capability

Point-in-Time Recovery:
  → Can restore to ANY SECOND within the retention period
  → Example: "Restore to 2025-01-15 14:23:47 UTC"
  → RDS restores from latest snapshot → replays transaction logs to exact second
  → Creates a NEW RDS instance with the restored data (doesn't overwrite original)

→ This is what ShopNow was missing → automatic daily backups + point-in-time recovery
→ No more "last backup was 3 days ago" → recovery to any second within 7 days

Backup Window:

→ A 30-minute window during which the daily backup runs
→ Brief I/O suspension may occur (for Single-AZ) → schedule during low-traffic hours
→ For Multi-AZ → backup taken from standby → no performance impact on primary
→ If "No preference" → AWS assigns a random window based on region

→ For ShopNow (India): Set to 18:30-19:00 UTC (midnight-12:30 AM IST) → lowest traffic

How the Endpoint Works:

→ The endpoint is a DNS CNAME (alias) that resolves to the private IP of the RDS instance
→ You NEVER connect to an IP address directly → always use the endpoint
→ Why: If the instance fails over to another AZ (Multi-AZ — Project 29) → the DNS CNAME updates to the new IP
→ Applications using the endpoint → automatically connect to the new instance after failover
→ Applications hardcoding an IP → break during failover

Connection String Format:

Port:     3306

→ The application (running on EC2 in the public or app subnet) uses this connection string
→ Traffic flows: EC2 → Security Group allows → Route table routes within VPC → RDS endpoint resolves → Connection established

STEP 6 — UNDERSTAND RDS SNAPSHOTS

Two Types of Backups:

TypeCreated ByRetentionStored WhereCost
Automated BackupsRDS automatically (daily)0-35 days (configurable)AWS managed — same regionFree (up to 100% of DB storage)
Manual SnapshotsYou (on-demand)Forever (until you delete)Your account — can copy cross-regionStorage cost per GB

Automated Backup:
→ Daily snapshot + continuous transaction log capture
→ Enables point-in-time recovery to any second
→ Deleted when: Instance is deleted (if "retain automated backups" is not checked) OR retention period expires
→ CANNOT be shared cross-account → copy to manual snapshot first

Manual Snapshot:
→ Created on-demand: RDS Console → Select instance → Actions → "Take snapshot"
→ Retained until YOU delete it → even if the RDS instance is deleted
→ CAN be shared cross-account
→ CAN be copied to other regions (for disaster recovery — Project 55)

Restoring from Snapshot:

→ CRITICAL UNDERSTANDING: Restoring creates a NEW RDS instance
→ It does NOT overwrite the existing instance
→ The new instance gets a new endpoint
→ You must update your application's connection string to point to the new instance

Original Instance: shopnow-prod-mariadb (endpoint: abc.rds.amazonaws.com)
        │
        ▼
Restore from snapshot → Creates NEW instance
New Instance: shopnow-prod-mariadb-restored (endpoint: xyz.rds.amazonaws.com)
        │
        ▼
You must: Update application → point to new endpoint → verify → delete old instance (if desired)

✅ WHAT SHOULD DO

→ Always use private subnets for database placement — never public subnets
→ Always create custom Parameter Groups before launching — default groups are read-only
→ Always enable automated backups with retention of at least 7 days (35 for critical data)
→ Always enable encryption at creation time — cannot add later without migration
→ Always enable deletion protection for production databases
→ Always set Public Access to "No" for production databases
→ Always use Security Group referencing (Project 4) to control database access
→ Always enable Enhanced Monitoring and Performance Insights for production
→ Always export slow query logs and error logs to CloudWatch
→ Always create CloudWatch alarms for: CPU, storage, connections, latency (Project 8)
→ Always set the maintenance window during lowest-traffic hours
→ Always use gp3 storage instead of gp2 for new instances
→ Always use the RDS endpoint hostname — never hardcode IP addresses
→ Always take a manual snapshot before making significant changes (version upgrade, parameter changes)
→ Always subscribe to RDS events via SNS for lifecycle notifications
→ Always set character set to utf8mb4 for proper Unicode support

❌ WHAT SHOULD NOT DO

→ Never set Public Access to "Yes" for production databases
→ Never use the default Parameter Group for production — it's read-only
→ Never set backup retention to 0 — this disables all automated backups
→ Never hardcode the database IP in application configuration — use the endpoint DNS
→ Never use the master username/password in application code — use Secrets Manager (Project 37)
→ Never enable the General Log in production — extreme I/O overhead
→ Never use burstable instances (db.t3) for production with consistent load — use db.r6g or db.m6g
→ Never skip encryption — compliance and security require it
→ Never ignore the slow query log — it's the primary tool for performance optimization
→ Never forget that storage auto-scaling only grows UP — plan initial size carefully
→ Never restore a snapshot expecting it to overwrite the original — it creates a NEW instance
→ Never skip deletion protection — accidental deletion = catastrophic data loss
→ Never connect from the internet to a private RDS instance and wonder why it times out — it has no public IP

2 → HOW AUTOMATED BACKUPS WORK INTERNALLY

CONTINUOUS OPERATION:
Transaction logs are captured and uploaded to S3 every 5 minutes
        │
        ▼
DAILY BACKUP WINDOW (e.g., 18:30-19:00 UTC):
        │
        ▼
Step 1: RDS initiates EBS snapshot of the database volume
        │
        ├── Single-AZ: Brief I/O suspension (~seconds) during snapshot initialization
        │   After initialization → snapshot proceeds in background → no I/O impact
        │
        └── Multi-AZ: Snapshot taken from STANDBY instance → ZERO impact on primary
        │
        ▼
Step 2: Snapshot stored in S3 (AWS-managed, not visible in your S3 console)
        │
        ▼
Step 3: Transaction logs since last snapshot → also stored in S3
        │
        ▼
Step 4: Old backups beyond retention period → automatically deleted

POINT-IN-TIME RECOVERY:
User requests: "Restore to 2025-01-15 14:23:47 UTC"
        │
        ▼
Step 1: RDS identifies the closest daily snapshot BEFORE the target time
        (e.g., snapshot from 2025-01-15 06:00 UTC)
        │
        ▼
Step 2: RDS restores the snapshot → creates new instance
        │
        ▼
Step 3: RDS replays transaction logs from 06:00 to 14:23:47
        │
        ▼
Step 4: New instance is available with data as of 14:23:47 UTC
        Data after 14:23:47 is NOT present (that's the recovery point)

→ Backups are stored in S3 but you CANNOT see or access them directly
→ Transaction log uploads every 5 minutes → recovery point objective (RPO) is ~5 minutes
→ The most you can lose in a disaster = last 5 minutes of transactions

Why DNS-based Endpoint:

→ During Multi-AZ failover (Project 29):

BEFORE FAILOVER:
  Endpoint DNS → 10.0.11.47 (Primary in AZ-a)

FAILOVER OCCURS (primary AZ-a fails):
  RDS promotes standby in AZ-b to primary
  RDS updates DNS: Endpoint → 10.0.12.23 (New primary in AZ-b)

AFTER FAILOVER:
  Endpoint DNS → 10.0.12.23
  Application reconnects using same endpoint → reaches new primary
  DNS TTL is low (typically 5 seconds) → propagation is fast

→ Applications using the endpoint → automatically failover
→ Applications using hardcoded IP → break → manual intervention needed

4 → HOW STORAGE WORKS

RDS Storage Layer:

→ RDS uses EBS volumes under the hood
→ The EBS volumes are in the same AZ as the RDS instance
→ For Multi-AZ → separate EBS volumes in each AZ (synchronously replicated)

Storage Auto-Scaling Internal Process:

Free storage monitoring runs every minute
        │
        ▼
Check: Is free storage < 10% of allocated?
        │
        ├── NO → Do nothing
        │
        └── YES → Has this condition persisted for 5+ minutes?
            │
            ├── NO → Wait
            │
            └── YES → Was last auto-scale > 6 hours ago?
                │
                ├── NO → Wait (cooldown period)
                │
                └── YES → Trigger auto-scaling
                    │
                    ▼
                    Calculate new size:
                    New = MAX(10% increase, 5 GB, free storage trigger amount)
                    New = MIN(New, Maximum storage threshold)
                    │
                    ▼
                    Modify storage → transparent to application
                    (Brief I/O optimization period after resize)

→ The 6-hour cooldown prevents rapid successive scaling events
→ The maximum threshold prevents unbounded growth
→ Storage modification happens online → no downtime

📈 SCALABILITY CONSIDERATIONS

→ Vertical Scaling (Scale Up)

  • Change instance class: db.r6g.large → db.r6g.xlarge → db.r6g.2xlarge
  • Requires brief downtime (Single-AZ) or ~60-second failover (Multi-AZ)
  • Schedule during maintenance window
  • Maximum: db.r6g.16xlarge (64 vCPU, 512 GB RAM)

→ Storage Scaling

  • Increase storage: 100 GB → 200 GB → 500 GB (online, no downtime)
  • Storage auto-scaling handles growth automatically
  • Maximum: 64 TB
  • Storage CANNOT be reduced — only increased

→ Read Scaling (Horizontal)

  • Read Replicas (Project 29): Up to 5 replicas for MariaDB
  • Each replica has its own endpoint → application reads from replicas
  • Reduces read load on primary → primary handles writes
  • Replicas can be in different AZs or different regions

→ Connection Scaling

  • max_connections limited by instance memory
  • db.r6g.large: ~700 default max connections
  • For more connections: Scale up instance OR use RDS Proxy (managed connection pooling)
  • RDS Proxy: Maintains a pool of database connections → shares them across many application connections
  • Reduces connection overhead on the database

→ Write Scaling

  • RDS MariaDB: Single writer (primary instance)
  • Cannot horizontally scale writes with standard RDS
  • For write-heavy workloads: Scale up instance class OR use Aurora (up to 128 TB, distributed storage)
  • For extreme write scaling: Consider DynamoDB (NoSQL) or application-level sharding

→ Cross-Region Scaling

  • Cross-region Read Replicas for global read performance
  • Cross-region snapshots for disaster recovery (Project 55)

STEP 4: Check route table

        NACL on RDS subnet → allows inbound on port 3306?
        NACL on RDS subnet → allows outbound on ephemeral ports (return traffic)?

        Wrong username/password → "Access denied for user"
        Correct endpoint but wrong database name → "Unknown database"

STEP 8: Check parameter group
        max_connections reached → "Too many connections" error

STEP 9: Check SSL/TLS
