PROJECT 14: THE WEB APP THAT COULDN'T TALK TO ITS DATABASE — EC2 to RDS MariaDB Private Connectivity

🔴 HYPOTHETICAL PROBLEM

→ ShopNow (from Projects 1-13) now has:

  • EC2 web servers in public subnets with IAM Roles (Projects 3, 5, 11)
  • RDS MariaDB in private data subnets (Project 9)
  • Security Groups configured for multi-tier access (Project 4)
  • VPC with proper routing and NAT Gateways (Project 5)
  • SNS → SQS fan-out for order processing (Project 13)

→ The web application on EC2 needs to connect to RDS MariaDB for:

  • Reading product catalog (SELECT queries)
  • Writing new orders (INSERT queries)
  • Updating inventory (UPDATE queries)
  • Running reports (complex JOINs)

→ Problem 1: priya-dev configured the application with the RDS IP address
→ She ran nslookup shopnow-prod-mariadb.cdefghijklmn.ap-south-1.rds.amazonaws.com
→ Got IP: 10.0.11.47 → hardcoded it in the application config
→ One month later → AWS performed maintenance → RDS instance moved to a different AZ
→ New IP: 10.0.12.23 → application connection broke → 2 hours of downtime
→ The DNS endpoint auto-updated but the hardcoded IP didn't

→ Problem 2: The application opens a new database connection for EVERY request
→ Each HTTP request → new TCP connection → MariaDB authentication → SQL query → close connection
→ During peak: 500 requests/second → 500 database connections opened and closed per second
→ Connection overhead: ~50ms per connection establishment
→ MariaDB max_connections hit (default ~700) → new connections rejected → "Too many connections" error
→ Application returns 500 errors to customers

→ Problem 3: The order processor (EC2 in private app subnet) also needs database access
→ But it uses a DIFFERENT Security Group than the web server
→ The RDS Security Group only allows shopnow-web-sg → order processor gets "Connection timed out"
→ Nobody thought to add the order processor's Security Group

→ Problem 4: Database credentials are stored in a plain text config file on EC2
→ Username: shopnow_admin, Password: [REDACTED:PASSWORD]!`
→ The config file is readable by the application user
→ If the server is compromised → attacker gets database credentials → full database access
→ The same credentials are used by web servers, order processors, and batch jobs → no isolation

→ Problem 5: Data travels between EC2 and RDS over the VPC network UNENCRYPTED
→ The security consultant says: "Even within a VPC, you should encrypt data in transit"
→ Compliance requires SSL/TLS for all database connections
→ The team doesn't know how to configure SSL for MariaDB connections

→ Problem 6: The batch reporting job runs at midnight → executes a massive query for 45 minutes
→ The query locks tables → web application INSERT queries queue up → customer-facing latency spikes
→ The team asks: "Can we run reports on a COPY of the database instead of the production primary?"

→ The CTO demands:

  • Application must use the DNS endpoint — NEVER hardcode IPs
  • Connection pooling must be implemented — stop creating new connections per request
  • ALL application tiers must be properly authorized in Security Groups
  • Database credentials must NOT be in plain text on servers
  • ALL database connections must be encrypted with SSL/TLS
  • Reporting must NOT impact production performance

🟡 HOW TO APPROACH IT

→ This is a DATABASE CONNECTIVITY problem spanning:

  • Network layer (VPC routing, Security Groups)
  • Application layer (connection strings, pooling, SSL)
  • Security layer (credentials management, encryption in transit)
  • Performance layer (connection management, read/write separation)

→ The approach should be:

  • Step 1 — Verify and fix network connectivity (Security Groups, routing)
  • Step 2 — Configure the correct connection string using the RDS endpoint
  • Step 3 — Understand and configure SSL/TLS for database connections
  • Step 4 — Implement connection pooling strategies
  • Step 5 — Manage database credentials securely
  • Step 6 — Configure multi-application access patterns
  • Step 7 — Understand read/write separation (Read Replica preview)
  • Step 8 — Monitor and troubleshoot database connectivity

CRITICAL: Always Use the DNS Endpoint — NEVER Hardcode IPs

→ The RDS endpoint is a DNS CNAME → resolves to the current private IP
→ During maintenance, failover, or AZ migration → IP changes → DNS auto-updates
→ Applications using the endpoint → automatically follow the IP change
→ Applications using hardcoded IPs → break on every infrastructure event

DNS TTL (Time-To-Live):

→ RDS endpoint DNS has a TTL of typically 5 seconds
→ This means: After a failover, DNS resolves to the new IP within ~5 seconds
→ Application must: Close stale connections → reconnect → DNS gives new IP
→ Connection pooling libraries handle this automatically (Step 4)

SSL Mode Options:

DISABLEDNoNoNoNONE — plain text
PREFERREDYes (if server supports)NoNoLOW — falls back to unencrypted

Step 3c: Enforce SSL at the Database Level

→ By default → RDS MariaDB ALLOWS both SSL and non-SSL connections
→ To REQUIRE SSL for all connections → modify the RDS parameter group (Project 9)

Parameter Group Change:

ParameterDefaultNew ValueEffect
require_secure_transportOFFONALL connections MUST use SSL — non-SSL rejected

→ RDS Console → Parameter groups → Select shopnow-mariadb-params → Edit
→ Set require_secure_transport = ON
→ This is a DYNAMIC parameter → takes effect immediately (no reboot needed)

→ After this change:

  • Application connecting with SSL → works ✓
  • Application connecting without SSL → "Access denied" error ✗
  • Any client without SSL certificate → rejected ✗

→ Instead of requiring SSL for ALL connections → require for specific users:

SQL: ALTER USER 'shopnow_admin'@'%' REQUIRE SSL;

→ Only shopnow_admin requires SSL → other users can connect without SSL
→ This is more granular but harder to manage

→ For ShopNow → Use the parameter group approach (enforce for ALL connections)

STEP 4 — CONNECTION POOLING STRATEGIES

→ This solves Problem 2: Opening new connections per request

Why Connection Pooling Matters:

WITHOUT POOLING:
Request 1 → Open Connection (50ms) → Query (5ms) → Close Connection (10ms) = 65ms
Request 2 → Open Connection (50ms) → Query (5ms) → Close Connection (10ms) = 65ms
Request 3 → Open Connection (50ms) → Query (5ms) → Close Connection (10ms) = 65ms

500 requests/sec → 500 × 50ms = 25,000ms of connection overhead per second
500 simultaneous connections → hits max_connections → errors

WITH POOLING:
Startup → Open 20 connections → Keep them alive in pool
Request 1 → Borrow connection from pool (0ms) → Query (5ms) → Return to pool (0ms) = 5ms
Request 2 → Borrow connection from pool (0ms) → Query (5ms) → Return to pool (0ms) = 5ms
Request 3 → Borrow connection from pool (0ms) → Query (5ms) → Return to pool (0ms) = 5ms

500 requests/sec → 20 pooled connections handle them all
Connection overhead: ZERO (connections already established)

Connection Pool Configuration Parameters:

ParameterRecommended ValueExplanation
Minimum pool size5Keep at least 5 connections alive even when idle
Maximum pool size20Never exceed 20 connections from this application
Connection timeout30 secondsWait max 30s for an available connection from pool
Idle timeout300 seconds (5 minutes)Close idle connections after 5 minutes
Max lifetime1800 seconds (30 minutes)Force-close and recreate connections every 30 minutes
Validation querySELECT 1Verify connection is alive before using
Validation interval30 secondsCheck connection health every 30 seconds

Why Max Lifetime Matters:

→ RDS MariaDB has wait_timeout = 300 seconds (set in Project 9)
→ If a pooled connection is idle for > 300 seconds → MariaDB closes it server-side
→ But the pool doesn't know → next use of that connection → "MySQL has gone away" error
→ Setting max lifetime < wait_timeout prevents this → pool recreates connections before server closes them
→ Also handles: DNS changes during failover → new connections go to new IP

Connection Pool Sizing Formula:

Pool Size = (Number of concurrent requests) × (Average query time) / (1 second)

ShopNow:
  Peak concurrent requests: 100
  Average query time: 50ms (0.05 seconds)
  Pool size = 100 × 0.05 = 5 connections

  But add buffer for:
  - Slow queries during peak
  - Connection validation overhead
  - Burst handling

  Recommended: 15-20 connections per application instance

Multi-Instance Pool Management:

→ ShopNow has 5 web server EC2 instances behind a load balancer
→ Each instance has its own connection pool: 20 connections each
→ Total database connections: 5 × 20 = 100 connections to RDS

→ Plus order processor (2 instances × 10 connections): 20 connections
→ Plus batch reporting (1 instance × 5 connections): 5 connections
→ Total: 100 + 20 + 5 = 125 concurrent connections

→ MariaDB max_connections = 700 (for db.r6g.large — Project 9)
→ 125 / 700 = 18% utilization → comfortable headroom

RDS Proxy (Managed Connection Pooling):

→ For more advanced scenarios → AWS RDS Proxy manages connection pooling as a SERVICE

WITHOUT RDS Proxy:
  EC2-1 (20 connections) ─┐

WITH RDS Proxy:
  EC2-1 (20 connections) ─┐

  EC2-4 (20 connections) ─┤

→ RDS Proxy sits between applications and the database
→ It maintains a SHARED pool of database connections
→ 100 application connections are multiplexed onto ~20 actual database connections
→ Reduces database connection overhead by 80%+

→ Secret ARN: arn:aws:secretsmanager:ap-south-1:123456789012:secret:shopnow/prod/mariadb/admin-AbCdEf

How Application Reads Credentials:

Application startup:
  1. EC2 IAM Role (Project 11) has permission: secretsmanager:GetSecretValue
  2. Application calls Secrets Manager API: GetSecretValue(SecretId="shopnow/prod/mariadb/admin")
  3. Secrets Manager returns: {"username": "shopnow_admin", "password": "..."}
  4. Application uses these credentials for database connection pool
  5. Credentials are cached in memory — NOT written to disk

Credential rotation:
  1. Secrets Manager automatically rotates password every 30 days
  2. Lambda rotation function: Changes password in RDS → updates secret
  3. Application's connection pool max lifetime (30 min) → new connections get new password
  4. Old connections continue working until they expire → graceful transition

IAM Role Permission for Secrets Manager:

→ Add to shopnow-web-server-role (Project 11):

Statement:
  Sid: AllowReadDatabaseSecret
  Effect: Allow
  Action:
    - secretsmanager:GetSecretValue
  Resource:
    - arn:aws:secretsmanager:ap-south-1:123456789012:secret:shopnow/prod/mariadb/admin-*

→ The -* at the end matches the random suffix Secrets Manager adds to the ARN
→ This allows the web server role to read ONLY this specific secret

Option 2: IAM Database Authentication (No Passwords At All)

→ RDS MariaDB supports IAM authentication → no passwords needed
→ The EC2 IAM Role generates a temporary authentication TOKEN
→ This token is used instead of a password

How It Works:

Step 1: EC2 IAM Role has permission: rds-db:connect
Step 2: Application calls RDS: generate-db-auth-token
        → Returns a temporary token (valid for 15 minutes)
Step 3: Application connects to RDS using:
        Username: shopnow_app_user (IAM-authenticated DB user)
        Password: (the temporary token)
Step 4: RDS validates the token against IAM
        → Connection established

Token rotation:
  Tokens expire after 15 minutes
  Application requests new token before expiry
  No password storage needed — ANYWHERE

IAM DB Auth Configuration:

→ Step 1: Enable IAM authentication on RDS (already done in Project 9, Step 4i)

→ Step 2: Create a database user linked to IAM

SQL: CREATE USER 'shopnow_app_user'@'%' IDENTIFIED WITH AWSAuthenticationPlugin AS 'RDS';
SQL: GRANT SELECT, INSERT, UPDATE ON shopnow.* TO 'shopnow_app_user'@'%';

→ Step 3: Add IAM permission to the EC2 role

Statement:
  Sid: AllowRDSIAMAuth
  Effect: Allow
  Action:
    - rds-db:connect
  Resource:
    - arn:aws:rds-db:ap-south-1:123456789012:dbuser:dbi-resource-id/shopnow_app_user

→ dbi-resource-id is the DBI resource ID from RDS Console → Instance → Configuration → Resource ID

Creating Separate Users (SQL — run as admin):

User 1: Web server
  CREATE USER 'shopnow_web_user'@'10.0.1.%' IDENTIFIED BY '(password)';
  GRANT SELECT ON shopnow.products TO 'shopnow_web_user'@'10.0.1.%';
  GRANT SELECT ON shopnow.categories TO 'shopnow_web_user'@'10.0.1.%';
  GRANT SELECT, INSERT ON shopnow.orders TO 'shopnow_web_user'@'10.0.1.%';
  GRANT SELECT, INSERT ON shopnow.order_items TO 'shopnow_web_user'@'10.0.1.%';

User 2: Order processor
  CREATE USER 'shopnow_processor_user'@'10.0.21.%' IDENTIFIED BY '(password)';
  GRANT SELECT, INSERT, UPDATE ON shopnow.orders TO 'shopnow_processor_user'@'10.0.21.%';
  GRANT SELECT, UPDATE ON shopnow.inventory TO 'shopnow_processor_user'@'10.0.21.%';
  GRANT SELECT ON shopnow.products TO 'shopnow_processor_user'@'10.0.21.%';

User 3: Batch reporting
  CREATE USER 'shopnow_reporting_user'@'10.0.%.%' IDENTIFIED BY '(password)';
  GRANT SELECT ON shopnow.* TO 'shopnow_reporting_user'@'10.0.%.%';

Connection-Specific Troubleshooting:

PROBLEM: "Too many connections" error
DIAGNOSIS:
  1. Check CloudWatch: DatabaseConnections metric → is it near max_connections?
  2. SQL: SHOW PROCESSLIST → identify connections by user/host
  3. Are all expected applications using connection pooling?
  4. Is any application leaking connections (opening but not closing)?
  5. Is connection pool max size too large across all applications?
SOLUTION:
  - Implement/fix connection pooling in application
  - Reduce pool max size per application
  - Increase max_connections in parameter group (if instance memory allows)
  - Scale up instance class for more memory → supports more connections
  - Consider RDS Proxy for connection multiplexing

PROBLEM: "MySQL has gone away" error
DIAGNOSIS:
  1. Connection was idle longer than wait_timeout (300 seconds)
  2. Connection pool kept stale connection → tried to reuse → server had closed it
SOLUTION:
  - Set pool max lifetime < wait_timeout (e.g., 250 seconds)
  - Enable connection validation (SELECT 1 before use)
  - Set validation interval to 30 seconds

PROBLEM: "Connection timed out" (new connections fail)
DIAGNOSIS:
  1. Security Group not allowing source → Check shopnow-db-sg inbound
  2. NACL blocking → Check db subnet NACL (stateless — check BOTH directions)
  3. RDS instance in different VPC → Verify VPC matches
  4. RDS instance stopped/rebooting → Check RDS status
  5. DNS not resolving → Check VPC DNS settings (Project 5)
SOLUTION:
  - Follow the 6-step network verification (Step 1 of this project)

PROBLEM: "Access denied for user" (authentication failure)
DIAGNOSIS:
  1. Wrong username or password
  2. User doesn't exist or wrong host pattern
  3. SSL required but client not using SSL
  4. IAM auth token expired (if using IAM DB auth)
SOLUTION:
  - Verify credentials in Secrets Manager
  - Check user's host pattern: 'user'@'10.0.1.%' vs 'user'@'%'
  - Ensure SSL is configured if require_secure_transport = ON
  - Refresh IAM auth token if using IAM DB auth

PROBLEM: Queries are slow (latency spikes)
DIAGNOSIS:
  1. Check Performance Insights → identify slow SQL
  2. Check CloudWatch → CPU, ReadLatency, WriteLatency, DiskQueueDepth
  3. Check Enhanced Monitoring → memory pressure, swap usage
  4. Check slow query log → queries exceeding long_query_time
  5. Check lock waits → batch reporting locking tables
SOLUTION:
  - Optimize slow queries (add indexes, rewrite queries)
  - Move read-only workloads to Read Replica (Project 29)
  - Scale up instance class
  - Increase IOPS (gp3 → io2 if needed)

✅ WHAT SHOULD DO

→ Always use the RDS DNS endpoint — NEVER hardcode IP addresses
→ Always use connection pooling — NEVER create a new connection per request
→ Always encrypt connections with SSL/TLS — set require_secure_transport = ON
→ Always create separate database users per application tier with least-privilege grants
→ Always store credentials in Secrets Manager — NEVER in config files on servers
→ Always configure Security Groups for ALL application tiers that need database access
→ Always verify both NACL directions (stateless) when troubleshooting connectivity
→ Always set connection pool max lifetime LESS than MariaDB wait_timeout
→ Always enable connection validation in the pool to detect stale connections
→ Always monitor DatabaseConnections metric — approaching max_connections causes outages
→ Always plan connection budgets across all applications before deploying
→ Always use Performance Insights to identify slow queries before they become outages
→ Always test connectivity from each application tier before deploying application code

❌ WHAT SHOULD NOT DO

→ Never hardcode the RDS IP address — it changes during maintenance and failover
→ Never open a new database connection for every HTTP request — connection overhead is expensive
→ Never use the same database user for all applications — no isolation, no audit trail
→ Never store database credentials in plain text config files — use Secrets Manager
→ Never use the master/admin user for application connections — create dedicated users
→ Never skip SSL/TLS for database connections — even within a VPC
→ Never set connection pool max size larger than needed — wastes database connections
→ Never forget to add new application SGs to the RDS Security Group — connections will time out
→ Never ignore the slow query log — it's your primary performance optimization tool
→ Never run heavy reporting queries on the production primary — use Read Replicas (Project 29)
→ Never set wait_timeout too low — connections drop unexpectedly
→ Never set wait_timeout too high — idle connections waste resources

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

1 → THE COMPLETE CONNECTION ESTABLISHMENT FLOW

Application on EC2 requests a database connection from the pool
        │
        ▼
Step 1: Connection pool checks for available idle connection
        │
        ├── Idle connection available → Validate it (SELECT 1)
        │   ├── Valid → Return to application (0ms overhead)
        │   └── Invalid (stale) → Discard → Create new (below)
        │
        └── No idle connection available → Create new connection
        │
        ▼
Step 2: Read credentials from Secrets Manager (or cache)
        IAM Role → secretsmanager:GetSecretValue → Returns username/password
        (Cached in memory for ~1 hour)
        │
        ▼
Step 3: DNS resolution
        Resolve: shopnow-prod-mariadb.cdefghijklmn.ap-south-1.rds.amazonaws.com
        VPC DNS (10.0.0.2) → Returns: 10.0.11.47
        DNS TTL: ~5 seconds (cached by OS resolver)
        │
        ▼
Step 4: TCP three-way handshake
        EC2 (10.0.1.50) → SYN → RDS (10.0.11.47:3306)
        │
        ├── Packet leaves EC2 ENI → Security Group outbound: ALLOW ✓
        ├── Packet enters public subnet NACL outbound: ALLOW ✓
        ├── VPC Router: 10.0.11.47 matches 10.0.0.0/16 → local route
        ├── Packet enters private data subnet NACL inbound: ALLOW ✓
        ├── Packet reaches RDS ENI → Security Group inbound: Source SG matches → ALLOW ✓
        └── TCP handshake completes (~1-2ms within same AZ, ~2-5ms cross-AZ)
        │
        ▼
Step 5: TLS handshake (if SSL enabled)
        Client Hello → Server Hello + Certificate → Key Exchange → Finished
        RDS presents its server certificate (signed by RDS CA)
        Client verifies certificate against downloaded CA bundle
        TLS 1.2 session established (~5-10ms)
        │
        ▼
Step 6: MariaDB authentication
        Client sends: username + password (encrypted via TLS)
        MariaDB validates credentials → checks user privileges → checks host pattern
        Authentication successful → MariaDB sends OK packet
        (~2-5ms)
        │
        ▼
Step 7: Connection established
        Total time: ~10-20ms (within same AZ) or ~15-30ms (cross-AZ)
        Connection returned to application
        │
        ▼
Step 8: Application sends SQL query over the established connection
        Query → Execute → Return results
        Connection stays open in pool for reuse

2 → HOW RDS PROXY MULTIPLEXING WORKS

WITHOUT PROXY:
  App Connection 1 ──────────► DB Connection 1 (dedicated)
  App Connection 2 ──────────► DB Connection 2 (dedicated)
  App Connection 3 ──────────► DB Connection 3 (dedicated)
  ...
  App Connection 100 ────────► DB Connection 100 (dedicated)
  
  100 app connections = 100 DB connections

WITH PROXY:
  App Connection 1 ─┐
  App Connection 2 ─┤         ┌──► DB Connection 1
  App Connection 3 ─┤  PROXY  ├──► DB Connection 2
  ...               ├────────►├──► DB Connection 3
  App Connection 98 ─┤         ├──► ...
  App Connection 99 ─┤         └──► DB Connection 20
  App Connection 100 ┘
  
  100 app connections = 20 DB connections (5:1 multiplexing)

How Multiplexing Works:

App Connection 1 sends query: SELECT * FROM products WHERE id=1
        │
        ▼
Proxy receives query
        │
        ▼
Proxy checks: Is App Connection 1 in a transaction?
        │
        ├── NO (simple query) → Use ANY available DB connection from pool
        │   Proxy borrows DB Connection 3 (currently idle)
        │   Sends query on DB Connection 3
        │   Returns result to App Connection 1
        │   Returns DB Connection 3 to proxy's pool
        │
        └── YES (in transaction) → Must use the SAME DB connection for entire transaction
            Proxy "pins" App Connection 1 to DB Connection 5
            All queries in this transaction go through DB Connection 5
            After COMMIT/ROLLBACK → unpin → DB Connection 5 returns to pool

→ Simple queries (no transaction) → proxy can multiplex aggressively
→ Transactions → proxy must pin the connection → less multiplexing
→ Applications with many short, non-transactional queries benefit most from proxy

3 → HOW DNS FAILOVER WORKS

NORMAL OPERATION:
  RDS Primary in AZ-a → IP: 10.0.11.47
  DNS: shopnow-prod-mariadb.xxx.rds.amazonaws.com → 10.0.11.47

FAILOVER TRIGGERED (Primary AZ-a fails):
  
  Time 0s:   Failure detected
  Time 10s:  RDS promotes standby in AZ-b to primary
  Time 30s:  New primary ready → IP: 10.0.12.23
  Time 31s:  RDS updates DNS: endpoint → 10.0.12.23
  Time 36s:  DNS TTL expires (5 seconds) → clients resolve new IP
  
  Application behavior:
  Time 0-36s: Active connections fail → "MySQL has gone away"
  Time 36s+:  Connection pool detects failures → closes stale connections
              Pool creates new connections → DNS resolves to 10.0.12.23
              New connections go to new primary in AZ-b
              Application resumes normal operation

WITH RDS PROXY:
  Time 0s:   Failure detected
  Time 1-2s: Proxy detects failure → holds incoming queries
  Time 10s:  New primary ready
  Time 11s:  Proxy establishes connections to new primary
  Time 12s:  Proxy resumes processing held queries
  
  Application sees: ~10 second pause, then normal operation
  NO connection errors → proxy absorbed the failover transparently

→ Without proxy: ~30-60 seconds of connection errors during failover
→ With proxy: ~10-15 seconds of query delay, no connection errors
→ This is why RDS Proxy is valuable for production applications

4 → CROSS-AZ CONNECTIVITY LATENCY

SAME AZ (EC2 in ap-south-1a, RDS in ap-south-1a):
  Network latency: ~0.5-1ms
  Connection establishment: ~10-15ms (TCP + TLS + auth)
  Query round-trip: ~1-5ms + query execution time

CROSS-AZ (EC2 in ap-south-1a, RDS in ap-south-1b):
  Network latency: ~1-3ms
  Connection establishment: ~15-25ms (TCP + TLS + auth)
  Query round-trip: ~2-6ms + query execution time
  Data transfer cost: ~$0.01/GB each direction

SAME REGION, DIFFERENT VPC (via VPC Peering — Project 44):
  Network latency: ~1-5ms (similar to cross-AZ)
  Additional: VPC Peering route must exist

→ Cross-AZ adds ~1-3ms per query round-trip
→ For latency-sensitive applications → place EC2 and RDS in the same AZ
→ BUT: Same AZ = single point of failure → cross-AZ is better for availability
→ Trade-off: Latency (same AZ) vs Availability (cross-AZ)

→ For ShopNow: Cross-AZ is acceptable — 2ms extra per query is negligible
→ Multi-AZ RDS (Project 29) provides automatic failover → availability is critical
