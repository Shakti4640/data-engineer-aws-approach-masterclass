PROJECT 4: THE DATABASE THAT NOBODY COULD REACH — Security Groups vs NACLs

🔴 HYPOTHETICAL PROBLEM

→ ShopNow (from Projects 1, 2, 3) now has:

  • S3 bucket for product images (Project 1)
  • IAM users with proper permissions (Project 2)
  • EC2 instance running the product catalog web app (Project 3)

→ The CTO decides to add a database server (another EC2 instance for now — RDS comes in Project 9)
→ The database runs MariaDB on port 3306
→ The web app EC2 instance needs to connect to the database EC2 instance on port 3306

→ She puts it as rule number 200 — but rule number 100 already allows all traffic
→ The deny rule NEVER triggers — the malicious traffic gets through
→ She doesn't understand why

→ The CTO demands: "Explain to me EXACTLY how network security works in AWS — all of it"

🟡 HOW TO APPROACH IT

→ Every EC2 instance has at least one ENI → every ENI has at least one Security Group

The "ALLOW ONLY" Rule — Critical Understanding:

→ Security Groups can ONLY have ALLOW rules
→ There is NO way to write a DENY rule in a Security Group
→ If you want to BLOCK a specific IP → you CANNOT do it with Security Groups
→ For blocking specific IPs → you need NACLs (Step 2)

→ The logic is:

  • If traffic matches ANY inbound rule → ALLOWED
  • If traffic matches NO inbound rule → DENIED (implicit deny)
  • There is no explicit deny capability

STEP 2 — NETWORK ACLs (NACLs) DEEP DIVE

→ A NACL is a firewall at the SUBNET level
→ Every subnet has exactly ONE NACL associated with it
→ A NACL can be associated with MULTIPLE subnets
→ Traffic entering or leaving the subnet passes through the NACL

Default NACL (created automatically with VPC):

→ VPC Console → Network ACLs → You see the default NACL

Default NACL Inbound Rules:

Rule #TypeProtocolPort RangeSourceAllow/Deny

Default NACL Outbound Rules:

Rule #TypeProtocolPort RangeDestinationAllow/Deny
100All trafficAllAll0.0.0.0/0ALLOW
*All trafficAllAll0.0.0.0/0DENY

→ The default NACL allows ALL traffic in and out — effectively transparent
→ The * rule is the DEFAULT rule — cannot be modified or deleted — catches anything not matched by numbered rules
→ This is why many people think NACLs "don't do anything" — the default NACL is wide open

WAIT — What is Rule 130 Inbound (Ports 1024-65535)?

→ This is the EPHEMERAL PORT rule — critical for NACLs and confusing for beginners
→ Because NACLs are STATELESS (explained in Step 3)
→ When your web server responds to an HTTP request:

  • The CLIENT connects FROM a random high port (e.g., port 52431) TO port 80
  • The SERVER responds FROM port 80 TO the client's port 52431
  • The response traffic is on a port in the range 1024-65535

→ In a STATEFUL firewall (Security Group) — the response is automatically allowed
→ In a STATELESS firewall (NACL) — the response is treated as NEW traffic → needs its own rule
→ Without Rule 130 → responses to HTTP/HTTPS requests are BLOCKED → website appears down

Ephemeral Port Ranges by OS:

Operating SystemEphemeral Port Range
Amazon Linux / RHEL32768-60999
Windows Server49152-65535
Many older systems1024-65535
Safe catch-all1024-65535

→ Use 1024-65535 to be safe across all client operating systems

NACL Rule Evaluation — NUMBERED ORDERING:

→ This is where Priya's Problem 4 occurred
→ NACL rules are evaluated IN ORDER from lowest rule number to highest

Incoming packet arrives at NACL
        │
        ▼
Check Rule 100: Does packet match?
        │
        ├── YES → Apply the rule's action (ALLOW or DENY) → STOP evaluating
        │
        └── NO → Continue to next rule
                │
                ▼
Check Rule 110: Does packet match?
        │
        ├── YES → Apply the rule's action → STOP evaluating
        │
        └── NO → Continue to next rule
                │
                ▼
        ... (continue through all numbered rules)
                │
                ▼
Check Rule *: DEFAULT rule → DENY
        (Always reached if no numbered rule matched)

→ FIRST MATCH WINS — once a rule matches, no further rules are evaluated
→ This is completely different from Security Groups (where ALL rules are evaluated)

Priya's Problem Explained:

Rule 100: ALLOW all traffic from 0.0.0.0/0       ← Matches EVERYTHING
Rule 200: DENY all traffic from 198.51.100.0/24   ← NEVER reached

Traffic from 198.51.100.0/24 arrives
→ Rule 100 matches (0.0.0.0/0 includes all IPs) → ALLOW → Stop
→ Rule 200 is never evaluated

The Fix:

Rule 90:  DENY all traffic from 198.51.100.0/24   ← Evaluated FIRST
Rule 100: ALLOW all traffic from 0.0.0.0/0        ← Evaluated SECOND (for everything else)

Traffic from 198.51.100.0/24 arrives
→ Rule 90 matches → DENY → Stop
→ Rule 100 never reached for this traffic

Traffic from any other IP arrives
→ Rule 90 does NOT match → Continue
→ Rule 100 matches → ALLOW

→ DENY rules must have LOWER numbers than ALLOW rules to take effect
→ Best practice: Leave gaps between rule numbers (100, 110, 120) so you can insert rules later (105, 115)

STEP 3 — STATEFUL vs STATELESS — THE CRITICAL DIFFERENCE

→ This is the MOST important concept in this project
→ It explains 90% of NACL-related confusion

STATEFUL (Security Groups):

→ "Remembers" the connection
→ If inbound traffic is ALLOWED → the return (outbound) traffic is AUTOMATICALLY allowed
→ If outbound traffic is ALLOWED → the return (inbound) traffic is AUTOMATICALLY allowed
→ You do NOT need to create rules for response traffic

Example:

Web client sends request to EC2 on port 80
        │
        ▼
Security Group INBOUND: Rule allows port 80 from 0.0.0.0/0 → ALLOWED
        │
        ▼
EC2 processes request → sends response back to client on ephemeral port
        │
        ▼
Security Group OUTBOUND: No rule needed for this response
→ SG "remembers" the inbound connection → automatically allows the response
→ THIS IS STATEFUL BEHAVIOR

What the SG Tracks:

Tracked AttributePurpose
Source IPWhich IP initiated the connection
Source PortWhich port the client used
Destination IPWhich IP was the target
Destination PortWhich port was the target
ProtocolTCP/UDP/ICMP
Connection stateESTABLISHED, RELATED

→ This tracking is maintained in a CONNECTION TRACKING TABLE
→ The table is maintained per-ENI by the Nitro card (Project 3 reference)

STATELESS (NACLs):

→ "Forgets" immediately — treats every packet independently
→ Inbound traffic allowed does NOT mean return traffic is allowed
→ You MUST create explicit rules for BOTH directions

Example:

Web client sends request to EC2 on port 80
        │
        ▼
NACL INBOUND: Rule 100 allows port 80 from 0.0.0.0/0 → ALLOWED
        │
        ▼
EC2 processes request → sends response back to client on ephemeral port (e.g., 52431)
        │
        ▼
NACL OUTBOUND: Does a rule allow TCP to destination port 52431?
        │
        ├── If Rule exists allowing ports 1024-65535 → ALLOWED → Response reaches client
        │
        └── If NO such rule → DENIED → Client never gets the response → Connection hangs

→ THIS is why NACL configurations always need ephemeral port rules
→ THIS is why NACLs are more complex to manage than Security Groups

Side-by-Side Comparison:

STEP 4 — SECURITY GROUP REFERENCING (SOLVING THE ARCHITECTURE)

→ This is the most POWERFUL feature of Security Groups
→ And the reason Security Groups are the PRIMARY firewall in AWS

The Problem Without Referencing:

→ Web server IP: 10.0.1.50
→ Database SG allows port 3306 from 10.0.1.50/32
→ Web server gets replaced → new IP: 10.0.1.75
→ Database SG still references old IP → connection fails
→ Must manually update the database SG every time

→ With Auto Scaling (Project 33) → instances launch/terminate constantly → IPs change every minute
→ Managing IPs in SG rules is IMPOSSIBLE at scale

The Solution With Referencing:

→ Database SG allows port 3306 from shopnow-web-sg (Security Group ID)
→ ANY instance with shopnow-web-sg attached → can connect to port 3306
→ New instance launches with shopnow-web-sg → automatically gets database access
→ Old instance terminates → automatically loses database access
→ ZERO manual updates needed

How Referencing Works Internally:

Instance A (Web) has SG: shopnow-web-sg (sg-0abc1234)
Instance B (DB) has SG: shopnow-db-sg

shopnow-db-sg Inbound Rule:
  Port 3306, Source: sg-0abc1234

Instance A sends packet to Instance B on port 3306
        │
        ▼
Packet arrives at Instance B's ENI
        │
        ▼
Security Group evaluation:
  "Is the SOURCE of this packet an ENI that has sg-0abc1234 attached?"
        │
        ├── YES (Instance A has sg-0abc1234) → ALLOWED
        │
        └── NO → Check other rules → If no match → DENIED

→ AWS resolves the SG reference by checking the source ENI's attached Security Groups
→ This happens at the Nitro card level — extremely fast

Multi-Tier Architecture Using SG Referencing:

INTERNET

│ PUBLIC SUBNET                                           │

│  ┌─────────────────┐    shopnow-web-sg                  │
│  │  Web Server EC2 │    Inbound: 80, 443 from 0.0.0.0/0│
│  │                 │    Inbound: 22 from admin IPs      │

            │ Port 3306

→ The database has NO public IP and NO internet-facing rules
→ ONLY the web server (identified by its SG) can reach port 3306
→ This is called a "Multi-Tier Security Architecture"
→ This pattern is FOUNDATIONAL — used in every production AWS deployment
→ VPC with public and private subnets (Project 5) enables this physical separation

STEP 6 — TRAFFIC FLOW WALKTHROUGH (COMPLETE PATH)

→ Understanding exactly what happens when a user accesses ShopNow's website

USER'S BROWSER → http://13.234.176.89 (Elastic IP from Project 3)

STEP 1: Packet reaches VPC
        Source: User's IP (e.g., 45.67.89.100)
        Destination: 13.234.176.89 (maps to private IP 10.0.1.50)
        Port: 80

STEP 2: NACL INBOUND evaluation (shopnow-web-nacl)
        Rule 50: Source 45.67.89.100 ∈ 198.51.100.0/24? NO → Next rule
        Rule 100: Port 80, TCP, from 0.0.0.0/0? YES → ALLOW → Stop
        ✅ Packet enters the subnet

STEP 3: Security Group INBOUND evaluation (shopnow-web-sg)
        Rule 1: Port 80, TCP, from 0.0.0.0/0? YES → ALLOW
        ✅ Packet reaches the EC2 instance

STEP 4: Web application processes request
        App queries database at 10.0.2.100:3306

STEP 5: Security Group OUTBOUND evaluation (shopnow-web-sg)
        Rule 1: Port 3306, TCP, to shopnow-db-sg? YES → ALLOW
        ✅ Packet leaves web instance

STEP 6: NACL OUTBOUND evaluation (shopnow-web-nacl)
        Rule 120: Port 3306, TCP, to 10.0.2.0/24? YES → ALLOW → Stop
        ✅ Packet leaves web subnet

STEP 7: NACL INBOUND evaluation (shopnow-db-nacl)
        Rule 100: Port 3306, TCP, from 10.0.1.0/24? YES → ALLOW → Stop
        ✅ Packet enters database subnet

STEP 8: Security Group INBOUND evaluation (shopnow-db-sg)
        Rule 1: Port 3306, TCP, from shopnow-web-sg? 
        Is source ENI attached to shopnow-web-sg? YES → ALLOW
        ✅ Packet reaches database instance

STEP 9: Database processes query → sends response
        Response goes to 10.0.1.50 (web server) on ephemeral port (e.g., 45678)

STEP 10: Security Group OUTBOUND evaluation (shopnow-db-sg)
         STATEFUL: Inbound was allowed → outbound response AUTO-ALLOWED
         ✅ No outbound rule needed

STEP 11: NACL OUTBOUND evaluation (shopnow-db-nacl)
         STATELESS: Must match a rule
         Rule 100: TCP, port 1024-65535, to 10.0.1.0/24? YES → ALLOW → Stop
         ✅ Response leaves database subnet

STEP 12: NACL INBOUND evaluation (shopnow-web-nacl)
         STATELESS: Must match a rule
         Rule 130: TCP, port 1024-65535, from 0.0.0.0/0? YES → ALLOW → Stop
         ✅ Response enters web subnet

STEP 13: Security Group INBOUND evaluation (shopnow-web-sg)
         STATEFUL: Outbound to database was allowed → inbound response AUTO-ALLOWED
         ✅ Response reaches web instance

STEP 14: Web application builds HTTP response → sends back to user

STEP 15: Security Group OUTBOUND evaluation (shopnow-web-sg)
         STATEFUL: Inbound HTTP was allowed → outbound response AUTO-ALLOWED
         ✅ Response leaves web instance

STEP 16: NACL OUTBOUND evaluation (shopnow-web-nacl)
         STATELESS: Must match a rule
         Rule 130: TCP, port 1024-65535 (user's ephemeral port), to 0.0.0.0/0? YES → ALLOW
         ✅ Response leaves subnet → reaches user

TOTAL: 16 evaluation steps for ONE HTTP request that queries a database

→ This is why understanding BOTH SGs and NACLs is essential
→ A failure at ANY step = connection fails
→ Security Groups handle 4 evaluations (but auto-allow return traffic due to statefulness)
→ NACLs handle 8 evaluations (must explicitly allow every direction including returns)

STEP 7 — SECURITY GROUP LIMITS AND ADVANCED CONFIGURATIONS

Limits:

Security Groups per VPC2,500

→ If you need more than 60 rules per SG → split across multiple SGs and attach all to the ENI
→ If you need more than 5 SGs per ENI → consolidate rules into fewer SGs

Multiple Security Groups on One Instance:

→ When an ENI has multiple SGs → all inbound rules from ALL SGs are COMBINED
→ If SG-A allows port 80 and SG-B allows port 443 → the instance allows BOTH 80 and 443
→ This is an ADDITIVE (union) model — SGs can only ADD permissions, never remove

Example:

Instance has: SG-A and SG-B attached

SG-A Inbound: Allow port 80 from 0.0.0.0/0
SG-B Inbound: Allow port 22 from 10.0.0.0/8

Effective Inbound: 
  Allow port 80 from 0.0.0.0/0  (from SG-A)
  Allow port 22 from 10.0.0.0/8  (from SG-B)

→ You CANNOT use SG-B to "override" or "deny" what SG-A allows
→ Once allowed by any SG → it's allowed → period

STEP 8 — NACL LIMITS AND ADVANCED CONFIGURATIONS

Limits:

ResourceDefault Limit
NACLs per VPC200
Rules per NACL (inbound + outbound)20 each (can increase to 40 — but impacts network performance)
NACLs per subnetExactly 1
Subnets per NACLUnlimited (one NACL can serve many subnets)

→ The 20 rule limit per direction is quite restrictive
→ This is why NACLs are used for BROAD rules (block IP ranges) not fine-grained control
→ Fine-grained control belongs to Security Groups

NACL Association:

→ Every subnet MUST have a NACL
→ If you don't explicitly associate one → the DEFAULT NACL (allows all) is used
→ When you associate a custom NACL → the previous NACL is automatically disassociated
→ One subnet = one NACL at a time (always)

✅ WHAT SHOULD DO

→ Always use Security Groups as your PRIMARY firewall — they are easier and more powerful
→ Always use SG referencing instead of IP addresses when resources communicate within AWS
→ Always use NACLs as a SECONDARY defense layer for subnet-level blocking
→ Always put DENY rules at LOWER rule numbers than ALLOW rules in NACLs
→ Always include ephemeral port rules (1024-65535) in NACLs for return traffic
→ Always leave gaps between NACL rule numbers (10, 20, 30 or 100, 110, 120) for future insertions
→ Always restrict SSH/RDP to specific IP ranges — never 0.0.0.0/0
→ Always restrict database ports to only the application tier SG — never the internet
→ Always restrict outbound SG rules for sensitive tiers (database should not reach arbitrary internet)
→ Always document which SGs reference which other SGs — this becomes complex in large environments
→ Always use descriptive SG names and descriptions — debugging at 3 AM requires clarity

❌ WHAT SHOULD NOT DO

→ Never open database ports (3306, 5432, 1433, 6379) to 0.0.0.0/0
→ Never open SSH (22) or RDP (3389) to 0.0.0.0/0
→ Never rely ONLY on NACLs — they are harder to manage due to stateless nature
→ Never put a DENY rule at a HIGHER number than a matching ALLOW rule in NACLs — it will never trigger
→ Never forget ephemeral port rules in NACLs — your connections will appear to "hang"
→ Never use the default NACL for production — it allows everything
→ Never create NACLs with too many rules — 20 per direction is the default limit and performance degrades beyond that
→ Never assume Security Groups can DENY traffic — they can only ALLOW
→ Never reference NACLs from Security Groups or vice versa — they are independent systems
→ Never use individual IPs in SG rules when SG referencing is possible — IPs change, SGs don't

🔴 WHEN NOT TO USE

→ Do NOT use NACLs as your only firewall — their stateless nature makes management error-prone
→ Do NOT use Security Groups for IP-based blocking — they lack deny rules
→ Do NOT use NACLs for fine-grained per-instance control — they apply to entire subnets
→ Do NOT use either for application-layer filtering (SQL injection, XSS) → Use AWS WAF
→ Do NOT use either for DDoS protection → Use AWS Shield
→ Do NOT use either for DNS-based filtering → Use Route 53 Resolver DNS Firewall
→ Do NOT use either for deep packet inspection → Use AWS Network Firewall

2 → CONNECTION TRACKING DEEP DIVE (SECURITY GROUPS)

→ The Nitro card maintains a connection tracking table:

FieldExample
ProtocolTCP
Source IP45.67.89.100
Source Port52431
Destination IP10.0.1.50
Destination Port80
StateESTABLISHED
Timeout350 seconds (TCP) / 30 seconds (UDP)

→ When a NEW inbound packet arrives → SG rules are evaluated
→ If ALLOWED → entry added to connection tracking table
→ When RESPONSE packet goes outbound → table is checked
→ Entry found → response automatically allowed WITHOUT checking outbound rules
→ This is the mechanics of "stateful" behavior

Connection Tracking Limits:

ParameterValue
Maximum tracked connections per ENI~130,000 (varies by instance type)
Timeout for established TCP connections350 seconds of idle
Timeout for UDP30 seconds
Timeout for ICMP30 seconds

→ If connection tracking table is FULL → new connections are DROPPED
→ Symptoms: Random connection failures under very high connection counts
→ Monitoring: CloudWatch metric ConnTrackAllowanceExceeded (available on Nitro instances)

Untracked Connections:

→ When a Security Group rule allows ALL traffic (protocol: all, port: all, source: 0.0.0.0/0)
→ Both inbound and outbound → connections become UNTRACKED
→ This means: No connection tracking table entry → slightly better performance
→ But: Response traffic is also not tracked → works because everything is allowed anyway
→ Relevant for high-performance scenarios (hundreds of thousands of connections)

3 → PACKET EVALUATION ORDER (COMPLETE)

INBOUND TRAFFIC (from internet to instance):
1. Route Table → determines which subnet to send to
2. NACL Inbound → stateless evaluation (first matching rule)
3. Security Group Inbound → stateful evaluation (all rules checked)
4. Traffic reaches instance

OUTBOUND TRAFFIC (from instance to internet):
1. Security Group Outbound → stateful evaluation (all rules checked)
   OR auto-allowed if response to tracked inbound connection
2. NACL Outbound → stateless evaluation (first matching rule)
3. Route Table → determines where to send
4. Traffic leaves VPC

→ For INBOUND: NACL is checked FIRST, then SG
→ For OUTBOUND: SG is checked FIRST, then NACL
→ Traffic must pass BOTH → failure at either = blocked

📈 SCALABILITY CONSIDERATIONS

→ Security Group Scalability

  • 2,500 SGs per VPC (can be increased)
  • 60 rules per SG (can be increased to 200 but reduces SGs per ENI)
  • 5 SGs per ENI → maximum 300 combined rules per instance
  • At enterprise scale → use SG referencing chains (web-sg → app-sg → db-sg)
  • SG changes propagate in seconds to all associated instances

→ NACL Scalability

  • 200 NACLs per VPC
  • 20 rules per direction per NACL (can increase to 40)
  • 20 rules is limiting for complex blocking requirements
  • For complex IP blocking → use AWS Network Firewall or AWS WAF instead
  • NACLs are best for BROAD subnet-level rules, not fine-grained filtering

→ Performance Impact

  • Security Groups: Minimal — evaluated on Nitro hardware
  • NACLs: Minimal — evaluated on network fabric
  • Connection tracking: Can become bottleneck at very high connection counts (>100K concurrent)
  • More rules = slightly more evaluation time but negligible in practice

→ Operational Scalability

  • 50 instances with same SG → change rule once → all 50 updated instantly
  • SG referencing eliminates IP management → scales with Auto Scaling
  • Infrastructure as Code (CloudFormation/Terraform) essential for managing SGs at scale
  • Tagging SGs with purpose and tier helps organizational management

🎯 KEY TAKEAWAY

→ AWS has TWO layers of network firewall: Security Groups (instance-level) and NACLs (subnet-level)
→ Security Groups are STATEFUL (track connections, auto-allow return traffic)
→ NACLs are STATELESS (every packet evaluated independently, need ephemeral port rules)
→ Security Groups can only ALLOW — NACLs can ALLOW and DENY
→ Security Group rules evaluate ALL rules (any match = allow) — NACL rules evaluate in NUMBER ORDER (first match wins)
→ Security Group REFERENCING is the most powerful feature — enables dynamic instance-to-instance access
→ NACLs are best for BROAD blocking (malicious IP ranges) — SGs are best for fine-grained access control
→ Connection to Project 3: The EC2 instance's connectivity is controlled by SGs we configured here
→ Connection to Project 2: IAM controls WHO can modify SG and NACL rules (ec2:AuthorizeSecurityGroupIngress, ec2:CreateNetworkAclEntry)
→ Connection to Project 1: S3 is accessed over HTTPS (port 443) — SG outbound must allow this
