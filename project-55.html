What Replication Lag Means for RPO:

Replication Lag = 30 seconds

    → After promotion → You lose the last 30 seconds of transactions
    → RPO achieved: 30 seconds ✅ (within 1-hour target)

Replication Lag = 45 minutes
    → If disaster happens NOW → DR replica is 45 minutes behind
    → After promotion → You lose the last 45 minutes of transactions

Replication Lag = 2 hours
    → If disaster happens NOW → DR replica is 2 hours behind

    → RPO VIOLATED ❌ — investigate and fix replication lag

How DNS Failover Works:

NORMAL OPERATION:
    Client → DNS query for api.payscale.com
    Route 53 → Health check on primary: HEALTHY ✅
    Route 53 → Returns: ALB-ap-south-1 IP address
    Client → Connects to ap-south-1

DURING DISASTER:
    Client → DNS query for api.payscale.com
    Route 53 → Health check on primary: UNHEALTHY ❌ (3 consecutive failures)
    Route 53 → Returns: ALB-ap-southeast-1 IP address
    Client → Connects to ap-southeast-1 (DR region)

AFTER RECOVERY:
    Client → DNS query for api.payscale.com
    Route 53 → Health check on primary: HEALTHY ✅ (recovered)
    Route 53 → Returns: ALB-ap-south-1 IP address (back to primary)
    Client → Connects to ap-south-1 (primary restored)

STEP 7 — DESIGN THE FAILBACK PROCEDURE

Failback = Returning to primary region after it recovers

Failback is MORE COMPLEX than failover because:
→ During failover → DR region accumulated NEW data
→ This new data must be synchronized BACK to the primary region
→ The primary region's old data may be stale or inconsistent

Failback Steps:

Phase 1: PRIMARY REGION RECOVERY VERIFICATION
    → Verify ap-south-1 is fully recovered
    → Verify all services are operational
    → Run health checks for 24 hours minimum

Phase 2: DATA SYNCHRONIZATION (DR → Primary)
    |
    ├── MariaDB:
    │   → Create new Cross-Region Read Replica from DR instance → Primary region
    │   → Wait for replica to catch up (replication lag → 0)
    │   → This may take hours depending on data volume
    │
    ├── S3:
    │   → Enable CRR from DR buckets → Primary buckets
    │   → Run Batch Replication for objects created during failover
    │   → Verify all objects replicated
    │
    └── Redshift:
        → Take snapshot in DR → Copy to primary
        → OR rebuild from primary S3 buckets after S3 sync

Phase 3: TRAFFIC CUTOVER
    → Promote primary region MariaDB replica to standalone
    → Update Route 53 to point back to primary
    → Verify primary is serving traffic correctly
    → Monitor for 24 hours

Phase 4: DR RE-ESTABLISHMENT
    → Create new Cross-Region Read Replica (primary → DR)
    → Re-enable S3 CRR (primary → DR)
    → Re-enable Redshift cross-region snapshots
    → Verify DR is synchronized
    → Ready for next disaster

Failback Timeline: 24-48 hours (not urgent — DR is already serving traffic)

✅ WHAT SHOULD DO

→ Always choose a DR region in a different geographic fault domain from primary
→ Always maintain Cross-Region Read Replica for critical databases — it's the fastest failover path
→ Always enable S3 Cross-Region Replication with Replication Time Control for RPO guarantees
→ Always configure cross-region Redshift snapshots — even if RPO for analytics is less strict
→ Always set low DNS TTL (60 seconds) on Route 53 failover records — enables fast DNS propagation
→ Always create and maintain a detailed failover runbook with specific steps, owners, and durations
→ Always pre-configure DR VPC, subnets, security groups, and VPC endpoints — eliminate setup time during crisis
→ Always monitor replication lag for MariaDB and S3 continuously — lag = RPO risk
→ Always test DR at least quarterly — untested DR plans fail when needed most
→ Always replicate existing S3 objects via Batch Replication — CRR only handles new objects
→ Always use different KMS keys per region — KMS keys are regional
→ Always document the failback procedure — getting back to primary is harder than failing over
→ Always keep DR costs visible in budget reporting — prevents cost-cutting that removes DR protection
→ Always notify teams and customers before planned DR tests

❌ WHAT SHOULD NOT DO

→ Never assume Multi-AZ within a region is sufficient DR — regional outages affect all AZs
→ Never skip DR testing — the first time you test DR should NOT be during an actual disaster
→ Never promote a read replica without checking replication lag first — high lag = significant data loss
→ Never forget that MariaDB promotion is IRREVERSIBLE — plan the failback path before promoting
→ Never use the same KMS key across regions — KMS keys are regional resources
→ Never set Route 53 TTL to 3600 seconds (1 hour) — failover DNS propagation will take an hour
→ Never assume S3 CRR replicates existing objects — it only replicates NEW objects after rule creation
→ Never run DR infrastructure at full production scale during pilot light — waste of money
→ Never skip the failback plan — you'll be stuck in DR region indefinitely
→ Never rely solely on automated DNS failover — verify DR resources are actually ready before traffic switches
→ Never ignore replication lag alerts — growing lag means RPO is at risk
→ Never test DR only once — systems change, and DR configuration must be validated regularly
→ Never assume DR will work because it worked last time — infrastructure drift happens

    |

    |
    ↓

Key Differences from Same-Region Replica:

AspectSame-Region ReplicaCross-Region Replica
Network pathWithin region (< 1ms)Cross-region backbone (~60ms)

KMS encryptionSame keyDifferent key (re-encrypted)

During Promotion:

BEFORE PROMOTION:
    DR Replica: Read-Only, replicating from primary
    
PROMOTION INITIATED:
    Step 1: Replica stops replication (breaks link to primary)
    Step 2: Replica applies remaining relay log entries
    Step 3: Replica becomes standalone Read-Write instance
    Step 4: Replica gets new endpoint DNS name
    Step 5: Backup is enabled on the new standalone instance
    
    Duration: 5-15 minutes
    
AFTER PROMOTION:
    DR Instance: Read-Write, INDEPENDENT (no longer connected to primary)
    Primary: Still running (if region recovered) but no longer replicated to DR

3 → ROUTE 53 FAILOVER MECHANICS

NORMAL STATE:
    Route 53 Health Checkers (in 3+ regions worldwide)
        → Every 10 seconds: HTTPS GET to primary ALB /health endpoint
        → Response 200 OK → PRIMARY HEALTHY ✅
        → DNS resolution: api.payscale.com → PRIMARY ALB IP

FAILURE DETECTED:
    T+0s:   Health checker #1 (us-east-1) gets timeout → Mark as failed
    T+10s:  Health checker #2 (eu-west-1) gets timeout → Mark as failed
    T+20s:  Health checker #3 (ap-northeast-1) gets timeout → Mark as failed
    
    3 consecutive failures from multiple regions → PRIMARY UNHEALTHY ❌
    
    T+20s:  Route 53 updates DNS:
            api.payscale.com → SECONDARY ALB IP (DR region)
    
    T+20s to T+80s: DNS propagation
            Clients with cached DNS → Still go to primary (for up to TTL=60s)
            New DNS lookups → Get DR region IP
    
    T+80s:  Most clients now reaching DR region

Health Check Considerations:

ConfigurationImpact
Request interval: 10s (fast)Faster detection but higher cost ($1/month vs $0.50 for 30s)
Failure threshold: 33 × 10s = 30 seconds to detect failure
Multiple regions: 3+Prevents false positive from single region network issue
String matching (optional)Check response body for specific string — catches application errors not just network failures
