PROJECT 31: THE ETL THAT HAD 5 STEPS IN SEQUENCE — Glue Workflow Orchestration

🔴 HYPOTHETICAL PROBLEM

→ ShopNow (from Project 1) has grown significantly
→ Their data engineering team has built multiple individual Glue components over time:

  • A Crawler that catalogs raw CSV files landing in S3 (built in Project 17)
  • A Glue ETL Job that converts CSV to Parquet (built in Project 21)
  • Another Crawler that catalogs the newly created Parquet files
  • A second Glue ETL Job that enriches Parquet data by joining with MariaDB product master (built in Project 24)
  • A final Crawler that catalogs the enriched dataset for Athena consumption (Project 18 pattern)

→ Currently — a data engineer manually runs each step one after another
→ He runs Crawler 1 → waits → checks if succeeded → runs Job 1 → waits → checks → runs Crawler 2 → waits → runs Job 2 → waits → runs Crawler 3
→ This takes 45 minutes of babysitting every single morning

→ Problems compounding:

  • The engineer went on vacation — nobody knew the sequence — pipeline didn't run for 5 days
  • One day Job 1 failed silently — Job 2 ran on stale data — wrong reports reached the CEO
  • On weekends nobody triggers the pipeline — Monday analytics are always 2 days behind
  • The team tried running steps in parallel where possible but couldn't coordinate dependencies
  • No visibility into which step failed, when, or why — they check each component's logs individually

→ The Head of Data asks: "Can we define the entire 5-step pipeline as ONE unit that runs automatically, handles failures, and shows me the full picture in one place?"

→ This is EXACTLY what AWS Glue Workflows solve

🟡 HOW TO APPROACH IT

→ First — understand what orchestration means in the ETL context
→ Individual Glue Crawlers and Jobs (from Projects 17, 21, 24) are like individual musicians
→ A Workflow is the conductor — it decides WHO plays WHEN and WHAT happens if someone misses a note

→ The mental model:

WORKFLOW = A directed acyclic graph (DAG) of:
    TRIGGERS → that start
    ACTIONS → which are Crawlers or Jobs
    CONDITIONS → that determine if next step should fire

→ Three core components inside a Glue Workflow:

ComponentRoleAnalogy
WorkflowThe container that holds everythingThe orchestra performance
TriggerThe signal that starts one or more actionsThe conductor's baton signal
ActionThe Crawler or Job that executesThe musician playing their part

→ Triggers come in 3 types — this is the KEY decision point:

Trigger TypeWhen It FiresUse Case
On-DemandManually or via API callTesting, ad-hoc runs
ScheduledAt a cron/rate expressionDaily/hourly automated runs
ConditionalWhen previous action(s) succeed/fail/completeChaining steps in sequence

→ The approach for ShopNow's 5-step pipeline:

[Scheduled Trigger - 6 AM daily]
        │
        ▼
   [Crawler 1: Catalog raw CSV]
        │
        ▼ (Conditional Trigger: fires when Crawler 1 SUCCEEDS)
   [Job 1: CSV → Parquet conversion]
        │
        ▼ (Conditional Trigger: fires when Job 1 SUCCEEDS)
   [Crawler 2: Catalog Parquet files]
        │
        ▼ (Conditional Trigger: fires when Crawler 2 SUCCEEDS)
   [Job 2: Parquet + MariaDB enrichment]
        │
        ▼ (Conditional Trigger: fires when Job 2 SUCCEEDS)
   [Crawler 3: Catalog enriched dataset]

→ If ANY step fails → subsequent steps DO NOT fire → workflow status shows FAILED → team gets visibility

STEP 12 — ADDING PARALLEL BRANCHES (Advanced Pattern)

→ ShopNow's pipeline is purely sequential — each step depends on the previous
→ But what if Job 1 produces TWO outputs and you need TWO crawlers to run simultaneously?

                [Job 1]
               /       \
              ▼         ▼
        [Crawler 2A] [Crawler 2B]    ← PARALLEL
              \       /
               ▼     ▼
          [Conditional Trigger: ALL SUCCEEDED]
                  │
                  ▼
               [Job 2]

→ Configuration for parallel execution:

Trigger for parallel start:
→ Single conditional trigger watching Job 1 → SUCCEEDED
→ But with TWO actions: Crawler 2A AND Crawler 2B
→ Both crawlers start simultaneously when Job 1 succeeds

Trigger for convergence (fan-in):
→ Single conditional trigger with:

  • Trigger Logic: "Start after ALL watched events"
  • Watched Event 1: Crawler 2A → SUCCEEDED
  • Watched Event 2: Crawler 2B → SUCCEEDED
    → This trigger fires ONLY when BOTH crawlers have succeeded
    → Then it starts Job 2

"ALL" vs "ANY" logic:

LogicBehaviorUse Case
ALLFires when every watched event has completed with specified conditionFan-in — wait for all parallel branches
ANYFires when at least one watched event has completed with specified conditionFirst-responder pattern — proceed as soon as any branch finishes

→ ShopNow doesn't need parallel branches yet — but this pattern is critical for complex pipelines

✅ WHAT SHOULD DO

→ Always name workflows, triggers, and actions with consistent naming conventions

  • Pattern: {project}-{pipeline}-{component-type}-{step-description}
  • Example: shopnow-daily-orders-trigger-after-crawler1

→ Always set Max Concurrent Runs to 1 for pipelines that write to same output paths
→ Always use conditional triggers with SUCCEEDED condition — never assume previous step worked

→ Always set workflow default run properties for shared configuration

  • Processing dates
  • Environment identifiers
  • Source/target paths
  • Any value that multiple steps need

→ Always test with an on-demand run before activating the schedule
→ Always verify each individual Crawler and Job works standalone BEFORE adding them to a workflow
→ Always check that individual Crawlers and Jobs do NOT have their own independent schedules

  • If Crawler 1 has its own schedule AND is in a workflow → it runs twice (once independently, once via workflow)

→ Always monitor workflow runs — set up CloudWatch Alarms on workflow failure (ties to Project 8)
→ Always tag workflows with Department, Project, Environment for cost tracking and governance

❌ WHAT SHOULD NOT DO

→ Never create a workflow with more than 100 triggers — Glue has a soft limit
→ Never put ALL your company's ETL into ONE workflow — separate by domain/pipeline
→ Never assume conditional triggers fire instantly — there's a small polling interval (seconds)
→ Never set Crawler schedules AND include them in a workflow — duplicate execution
→ Never chain more than 10-15 sequential steps in a single workflow without evaluating Step Functions (Project 32) as alternative
→ Never use workflow run properties for sensitive data (credentials, API keys) — use Secrets Manager (Project 37)
→ Never skip the testing phase — a misconfigured trigger condition can silently skip steps
→ Never forget that if the starting trigger is not activated, nothing runs — activation is a separate step from creation
→ Never use "ANY" logic when you mean "ALL" for convergence triggers — one failed branch could let the pipeline continue with incomplete data
→ Never hardcode S3 paths inside Jobs when you can pass them via workflow run properties — makes the workflow inflexible

🔴 WHEN NOT TO USE GLUE WORKFLOWS

→ When your pipeline includes non-Glue steps

  • Need to trigger a Lambda after ETL? → Step Functions (Project 32)
  • Need to send SNS notification on completion? → Step Functions or EventBridge (Project 38)
  • Need to run a Redshift COPY command? → Step Functions with Redshift Data API

→ When you need dynamic runtime decisions

  • "If data volume > 1M rows, use Job A; else use Job B" → Step Functions Choice State
  • Glue Workflows have static DAGs — the graph is fixed at design time

→ When you need retry with exponential backoff per step

  • Glue Workflows do not have built-in retry on trigger level
  • Jobs have their own retry config (max retries setting on the Job itself)
  • For sophisticated retry patterns → Step Functions Retry/Catch

→ When you need cross-account pipeline coordination

  • Workflow runs within a single AWS account
  • For cross-account patterns → EventBridge (Project 38) + Step Functions (Project 32)

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

MECHANIC 1 — WORKFLOW EXECUTION ENGINE

→ When the scheduled trigger fires at 6 AM:

  1. Glue Workflow Engine creates a new "Workflow Run" with a unique Run ID
  2. The starting trigger evaluates and fires its actions (Crawler 1)
  3. Crawler 1 transitions to RUNNING state
  4. The Workflow Engine polls the status of Crawler 1 periodically
  5. When Crawler 1 completes → the engine checks all conditional triggers watching Crawler 1
  6. Trigger condition is evaluated: "Did Crawler 1 SUCCEED?"
  7. If YES → the trigger fires its action (Job 1)
  8. If NO (FAILED/STOPPED/TIMEOUT) → the trigger does NOT fire → downstream chain stops
  9. This cascading evaluation continues through every trigger-action pair

→ The polling interval is internal to AWS — typically seconds, not configurable
→ There is no SLA on exact trigger evaluation latency — expect 5-30 seconds between step completion and next step starting

MECHANIC 2 — TRIGGER STATE MACHINE

→ Each trigger in a workflow has its own lifecycle:

CREATED → ACTIVATED → FIRING → SUCCEEDED/FAILED
              ↑                      │
              └──────────────────────┘
              (re-arms for next run)
StateMeaning
CREATEDTrigger exists but is not active
ACTIVATEDTrigger is listening for its condition
FIRINGCondition met — starting its actions
SUCCEEDEDActions started successfully
FAILEDCould not start actions (e.g., action doesn't exist)

→ Scheduled triggers re-arm after each firing — they fire again at the next scheduled time
→ Conditional triggers re-arm for the next workflow run — they fire once per run

MECHANIC 3 — RUN PROPERTIES PROPAGATION

→ When a workflow run starts:

  1. Default run properties (defined at workflow creation) are loaded
  2. If overrides were provided (via API/console), they replace defaults
  3. Every Job in the workflow receives additional system arguments:
    • --workflow_name: Name of the workflow
    • --workflow_run_id: Unique ID of this specific run
  4. Inside the Job → these two parameters are used to FETCH the run properties via Glue API
  5. The Job calls glue.get_workflow_run_properties(Name, RunId) → gets the key-value pairs
  6. The Job uses these properties to determine paths, dates, configurations

→ This is NOT automatic injection into job arguments
→ The Job must EXPLICITLY fetch run properties using the workflow name and run ID
→ If the Job runs OUTSIDE the workflow (standalone), these parameters won't exist → Job should handle this gracefully

MECHANIC 4 — WORKFLOW RUN STATUS DETERMINATION

→ The overall workflow run status is determined by the collective status of ALL actions:

ScenarioWorkflow Run Status
All actions completed successfullyCOMPLETED
Any action failed AND downstream triggers didn't fireERROR
Workflow was manually stoppedSTOPPED
Some actions completed, pipeline stopped due to failureERROR
Workflow is still executingRUNNING

→ Important: Glue Workflow does NOT have a native "FAILED" notification mechanism
→ To get notified on failure → you must:

  • Set up a CloudWatch Events rule (EventBridge) watching for Glue Workflow run state changes
  • Route to SNS for email notification (Project 6, 38 patterns)
  • OR use a separate monitoring Lambda that polls workflow run status

MECHANIC 5 — HOW THE VISUAL GRAPH IS BUILT

→ The Glue console renders the workflow graph by:

  1. Reading all triggers associated with the workflow
  2. For each trigger → mapping its watched events (upstream) and actions (downstream)
  3. Building a DAG (Directed Acyclic Graph) from these relationships
  4. Color-coding nodes based on the current/last run status

→ The graph is READ-ONLY during execution — you cannot modify a running workflow
→ To modify → stop the workflow → edit triggers/actions → reactivate

MECHANIC 6 — TRIGGER EVALUATION FOR PARALLEL FAN-IN

→ When a conditional trigger watches MULTIPLE events with "ALL" logic:

Trigger watches: [Crawler 2A: SUCCEEDED] AND [Crawler 2B: SUCCEEDED]

→ Internal behavior:

  1. Crawler 2A finishes → engine checks trigger → Crawler 2B not done yet → trigger does NOT fire
  2. Crawler 2B finishes → engine checks trigger → both conditions met → trigger FIRES
  3. If Crawler 2A SUCCEEDED but Crawler 2B FAILED → trigger NEVER fires

→ The trigger maintains internal state tracking which watched events have been satisfied
→ This state resets at the beginning of each workflow run

MECHANIC 7 — INTERACTION WITH INDIVIDUAL JOB/CRAWLER SETTINGS

→ Glue Jobs and Crawlers have their own configurations (timeout, retries, DPU allocation)
→ The Workflow does NOT override these settings
→ It simply starts them — the Job/Crawler runs with its own defined configuration

SettingControlled ByNOT By
Job timeoutJob definitionWorkflow
Job max retriesJob definitionWorkflow
Job DPU/worker countJob definitionWorkflow
Crawler timeoutCrawler definitionWorkflow
Execution sequenceWorkflow triggersJob/Crawler definition
Shared parametersWorkflow run propertiesJob definition

→ If a Job has Max Retries: 2 → and it fails → Glue retries the job 2 times
→ The conditional trigger watching this Job waits until ALL retries are exhausted
→ Only after the final retry fails does the trigger see the FAILED condition

📈 SCALABILITY CONSIDERATIONS

→ Number of Steps per Workflow

  • Soft limit: 100 triggers per workflow
  • Practical limit: 15-20 steps before the visual graph becomes hard to manage
  • Beyond 20 steps → consider splitting into sub-workflows or using Step Functions

→ Number of Workflows per Account

  • Soft limit: 25 workflows per account (can request increase)
  • For organizations with many pipelines → request limit increase to 100+

→ Concurrent Workflow Runs

  • Configurable via Max Concurrent Runs
  • Each run consumes resources for every Job/Crawler it triggers
  • 10 workflows each running 5 Jobs = 50 concurrent Glue Jobs → check your account's Glue Job concurrency limit

→ Execution Duration

  • No timeout on the workflow itself — it runs until all steps complete or fail
  • Individual Jobs and Crawlers have their own timeouts
  • A workflow with 5 sequential steps, each taking 30 minutes = 2.5 hours total
  • Plan scheduling accordingly — don't schedule next run before previous can finish

→ Cross-Region Limitation

  • Workflows are regional — they can only orchestrate Crawlers and Jobs in the same AWS region
  • For cross-region ETL → use Step Functions or EventBridge cross-region patterns

→ Run History Retention

  • Glue retains workflow run history for 90 days
  • After 90 days → run details are purged
  • For long-term audit → export run metadata to S3 via API

→ Cost Scaling

  • The Workflow itself has NO cost — it's a free orchestration layer
  • You pay ONLY for the Crawlers and Jobs that execute within it
  • Crawlers: Billed per DPU-hour
  • Jobs: Billed per DPU-hour (or per DPU-second for Glue 2.0+)
  • More steps = more DPU consumption = higher cost
  • Optimize individual Job DPU allocation to control costs
