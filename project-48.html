PROJECT 48: THE REDSHIFT CLUSTER THAT NEEDED LIVE OPERATIONAL DATA — Redshift Federated Query to RDS MariaDB

🔴 HYPOTHETICAL PROBLEM

→ PayScale's data architecture now has two major data stores:

Store 1 — Redshift Data Warehouse (Account B)
→ Set up in Project 16, loaded via COPY in Project 22
→ Contains 3 years of historical transaction data (1+ billion rows)
→ Optimized for analytical queries — aggregations, trend analysis, pattern detection
→ Data is loaded nightly via Glue ETL (Project 24, 46)
→ Latest data is always 24 hours old (yesterday's transactions)

Store 2 — MariaDB Operational Database (Account B)
→ Set up in Project 9, connected to EC2 in Project 14
→ Contains LIVE transaction data — updated in real-time as payments happen
→ Optimized for transactional queries — INSERT, UPDATE, SELECT by primary key
→ Contains current transaction STATUS (pending, processing, completed, failed, reversed)

The Current Data Flow:

Live Transaction → MariaDB (real-time) → Glue ETL (nightly) → S3 → COPY → Redshift (24h delay)

The Business Problem:

→ The fraud team runs a critical query every hour:

"For each customer who had a high-risk transaction in the LAST 3 YEARS (Redshift),
 show me their CURRENT pending transactions and LIVE status (MariaDB)"

→ This requires JOINING:

  • Redshift historical_risk_scores table (3 years, 1B rows) — only in Redshift
  • MariaDB transactions table (live status) — only in MariaDB

Current Workaround:

StepActionTimeProblem
1Export MariaDB transactions to S3 via Glue45 minutesData is stale by the time export completes
2COPY S3 data into Redshift staging table15 minutesAdditional storage cost in Redshift
3Run JOIN query in Redshift2 minutesQuery runs on 1-hour-old data, not live
4Drop staging table1 minuteCleanup overhead
Total~63 minutesData is 1+ hour stale by the time results are available

→ The fraud team says: "We need this in REAL-TIME — a single query that joins Redshift history with MariaDB LIVE data"
→ They cannot wait 63 minutes — fraud decisions must happen within seconds
→ They don't want to replicate all MariaDB data into Redshift — it's constantly changing

→ The data architect says: "We need Redshift to reach INTO MariaDB and query it directly — without moving data"

🟡 HOW TO APPROACH IT

→ First — understand the options for querying external data from Redshift

    |   AND m.status = 'pending'                   |

    Redshift processes local table scan             |

    high-risk customers)                           |

STEP 6 — QUERY MARIADB TABLES FROM REDSHIFT

Simple Query — Read MariaDB Data from Redshift:

Redshift SQL:
    SELECT transaction_id, customer_id, amount, status
    FROM mariadb_live.transactions
    WHERE status = 'pending'
    LIMIT 100;

What Happens Internally:

Redshift Query Planner
    |
    ↓
Detects: mariadb_live.transactions is an EXTERNAL table (federated)
    |
    ↓
Redshift generates a remote query for MariaDB:
    SELECT transaction_id, customer_id, amount, status
    FROM payscale_prod.transactions
    WHERE status = 'pending'
    LIMIT 100
    |
    ↓
Sends query to MariaDB over JDBC (TCP 3306)
    |
    ↓
MariaDB executes query using its own indexes/optimizer
    |
    ↓
MariaDB returns result set to Redshift
    |
    ↓
Redshift formats results and returns to user

→ The query is PUSHED DOWN to MariaDB — Redshift doesn't pull all data and filter locally
→ This is critical for performance — only matching rows cross the network

STEP 7 — EXECUTE CROSS-DATABASE JOINS (THE REAL POWER)

The Query the Fraud Team Needs:

Redshift SQL:
    SELECT 
        r.customer_id,
        r.avg_risk_score,
        r.total_flagged_transactions,
        m.transaction_id AS live_txn_id,
        m.amount AS live_amount,
        m.status AS live_status,
        m.created_at AS live_created_at
    FROM public.historical_risk_summary r          -- LOCAL Redshift table (3 years, aggregated)
    JOIN mariadb_live.transactions m                -- REMOTE MariaDB table (live)
        ON r.customer_id = m.customer_id
    WHERE r.avg_risk_score > 0.8                   -- High-risk customers (Redshift filters locally)
        AND m.status = 'pending'                   -- Only pending transactions (pushed to MariaDB)
    ORDER BY r.avg_risk_score DESC
    LIMIT 50;

How Redshift Executes This JOIN:

Step 1: Redshift scans LOCAL table: public.historical_risk_summary
        WHERE avg_risk_score > 0.8
        Result: ~5,000 high-risk customer_ids (out of 500K total)
        
Step 2: Redshift generates REMOTE query for MariaDB:
        SELECT transaction_id, customer_id, amount, status, created_at
        FROM payscale_prod.transactions
        WHERE status = 'pending'
        AND customer_id IN (list of 5,000 customer_ids)
        
        → This is PUSHDOWN — Redshift pushes the filter to MariaDB
        → MariaDB uses its customer_id index to efficiently find matching rows
        
Step 3: MariaDB returns ~200 matching pending transactions
        (only 200 out of potentially millions — highly filtered)
        
Step 4: Redshift JOINs the 5,000 local rows with 200 remote rows
        Result: ~200 joined rows
        
Step 5: Redshift sorts by avg_risk_score DESC, applies LIMIT 50
        Returns final 50 rows to user
        
Total time: ~3-5 seconds (vs 63 minutes with the old ETL approach)

The Key Insight:
→ Redshift processes the LOCAL table first (large historical data — this is what Redshift is good at)
→ Then sends a TARGETED query to MariaDB (small filtered result set)
→ MariaDB only processes a small, indexed query
→ The JOIN happens in Redshift's memory — minimal data crosses the network

STEP 8 — UNDERSTAND QUERY PUSHDOWN OPTIMIZATION

What Gets Pushed Down to MariaDB:

SQL ElementPushed Down?Example
WHERE clause filters✅ YesWHERE status = 'pending' → Executed on MariaDB
Column projection✅ YesSELECT col1, col2 → Only requested columns fetched
LIMIT✅ YesLIMIT 100 → MariaDB returns only 100 rows
Aggregations (COUNT, SUM, AVG)✅ YesSELECT COUNT(*) FROM table → Computed on MariaDB
ORDER BY⚠️ PartialSimple ORDER BY may be pushed; complex may not
JOINs between two remote tables❌ NoJOIN of two MariaDB tables → Both fetched to Redshift, joined locally
Subqueries⚠️ PartialSimple subqueries may be pushed
Window functions❌ NoROW_NUMBER(), RANK() → Computed in Redshift
UDFs (User-Defined Functions)❌ NoRedshift UDFs don't exist on MariaDB

How to Verify Pushdown:

→ Use Redshift's EXPLAIN command to see the query plan:

EXPLAIN SELECT * FROM mariadb_live.transactions WHERE status = 'pending';

→ Look for: XN Foreign Scan → Indicates federated query
→ Look for: Filter conditions listed under the Foreign Scan → Indicates pushdown

Optimizing for Pushdown:

Do ThisWhy
Put WHERE filters on remote table columnsGets pushed to MariaDB — reduces data transfer
Select only needed columnsColumn projection reduces data transfer
Use LIMIT on remote-only queriesPrevents pulling entire remote table
Filter local table FIRST, then join remoteReduces the customer_id IN (...) list sent to MariaDB
Ensure MariaDB has indexes on filtered columnsPushdown filters are only fast if MariaDB can use indexes
Avoid ThisWhy
SELECT * FROM remote table without WHEREPulls ENTIRE table over network — very slow
Joining two large remote tablesBoth tables fetched entirely to Redshift — defeats purpose
Complex functions on remote columns in WHEREMay prevent pushdown — filter applied locally after full fetch
DISTINCT on remote table without filtersFetches all rows, deduplicates locally

STEP 9 — MARIADB INDEX OPTIMIZATION FOR FEDERATED QUERIES

Why MariaDB Indexes Matter:

→ Redshift pushes WHERE clauses to MariaDB
→ If MariaDB doesn't have indexes on the filtered columns → Full table scan on MariaDB
→ Full table scan on MariaDB = slow response = slow federated query = defeats the purpose

Recommended MariaDB Indexes for Federated Query Patterns:

Query Pattern from RedshiftMariaDB Index NeededType
WHERE status = 'pending'Index on statusSingle column
WHERE customer_id IN (...)Index on customer_idSingle column (likely PK or FK)
WHERE status = 'pending' AND customer_id IN (...)Composite index on (status, customer_id)Multi-column
WHERE created_at > '2025-01-15'Index on created_atSingle column
WHERE status = 'pending' AND created_at > '2025-01-15'Composite index on (status, created_at)Multi-column

→ Work with the DBA to create appropriate indexes
→ Monitor MariaDB's slow query log during federated queries
→ If a federated query takes > 10 seconds → likely missing index on MariaDB side

STEP 10 — CROSS-DATABASE QUERY PATTERNS

Pattern 1 — Local Redshift + Remote MariaDB JOIN (Most Common)

SELECT r.*, m.status
FROM public.redshift_table r
JOIN mariadb_live.mariadb_table m ON r.id = m.id
WHERE r.local_filter = 'value'
    AND m.remote_filter = 'value';

→ Best performance — Redshift filters locally first, then queries MariaDB for matching subset

Pattern 2 — Remote MariaDB + Redshift Spectrum (S3) JOIN

SELECT m.transaction_id, m.status, s.risk_model_output
FROM mariadb_live.transactions m
JOIN spectrum_schema.model_scores s ON m.transaction_id = s.transaction_id
WHERE m.status = 'pending';

→ Joins LIVE MariaDB data with S3 data lake (via Spectrum from Project 28)
→ Three data sources accessible from single Redshift query: Local + MariaDB + S3

Pattern 3 — Three-Way JOIN (Local + MariaDB + Spectrum)

SELECT 
    r.customer_id,
    r.lifetime_value,                        -- From Redshift local table
    m.current_balance,                       -- From MariaDB (live)
    s.latest_risk_score                      -- From S3 via Spectrum
FROM public.customer_summary r               -- LOCAL
JOIN mariadb_live.customer_accounts m        -- REMOTE (MariaDB)
    ON r.customer_id = m.customer_id
JOIN spectrum_schema.risk_scores s           -- REMOTE (S3)
    ON r.customer_id = s.customer_id
WHERE r.lifetime_value > 100000
    AND s.latest_risk_score > 0.7;

→ This is the ULTIMATE power of Redshift — querying three different data stores in one SQL statement
→ No data movement, no ETL, no staging tables

Pattern 4 — Materialized View on Federated Query (Caching)

→ If the same federated query runs frequently → Create a materialized view

SettingValuePurpose
Materialized view namemv_high_risk_pendingCached result of federated join
Auto-refreshEnabled (schedule-based)Refresh every 15 minutes

→ First query hits MariaDB → Result cached in materialized view
→ Subsequent queries read from materialized view → No MariaDB hit
→ Trade-off: Data freshness (up to 15 min stale) vs query performance

✅ WHAT SHOULD DO

→ Always store MariaDB credentials in Secrets Manager — never hardcode in external schema definition
→ Always create a dedicated read-only MariaDB user for Redshift federation — never use admin/root
→ Always verify Enhanced VPC Routing is enabled on the Redshift cluster
→ Always ensure security groups allow Redshift → MariaDB on port 3306
→ Always use the RDS DNS endpoint in the URI — never use IP addresses (IPs change during failover from Project 29)
→ Always push filters to the remote side by placing WHERE conditions on remote table columns
→ Always select only the columns you need from remote tables — minimize data transfer
→ Always filter the LOCAL Redshift table FIRST in JOINs — reduce the dataset before hitting MariaDB
→ Always create appropriate indexes on MariaDB for columns used in federated query WHERE clauses
→ Always monitor MariaDB connection count during federated queries — each query opens connections
→ Always test federated queries during low-traffic MariaDB periods first — measure impact on operational workload
→ Always use EXPLAIN to verify query pushdown behavior
→ Always consider materialized views for frequently-run federated queries

❌ WHAT SHOULD NOT DO

→ Never run SELECT * FROM mariadb_live.large_table without WHERE — pulls entire table over network
→ Never join two large remote MariaDB tables — both are fetched entirely to Redshift before joining
→ Never use the MariaDB root/admin credentials — use a dedicated read-only user
→ Never forget to update security groups — federated query silently times out without clear error
→ Never assume federated query performance equals local Redshift query performance — network latency exists
→ Never use federated queries for bulk historical analysis — use COPY/Glue ETL for that (Project 22, 24)
→ Never run hundreds of concurrent federated queries — each opens MariaDB connections (connection limit from Project 45)
→ Never use federated queries as a replacement for proper ETL — federation is for LIVE lookups, not bulk processing
→ Never skip Enhanced VPC Routing — without it, Redshift cannot reach private RDS instances
→ Never hardcode MariaDB IPs in URI — Multi-AZ failover (from Project 29) changes the IP
→ Never ignore the impact on MariaDB — federated queries add read load to your operational database
→ Never use federated query for data that doesn't change frequently — COPY it into Redshift instead (cheaper, faster)

🔴 WHEN NOT TO USE REDSHIFT FEDERATED QUERY

→ When you need to scan millions of rows from MariaDB → Network transfer becomes the bottleneck → Use ETL
→ When the same query runs hundreds of times per hour → MariaDB connection overload → Use materialized views or cache
→ When data freshness of 1 hour is acceptable → Nightly/hourly ETL is simpler and cheaper
→ When MariaDB is already under heavy write load → Federated queries add read pressure → Risk to operational stability
→ When complex aggregations are needed on remote data → Pushdown may not apply → Full table fetched → Slow
→ When you need to join two large remote tables → Both fully fetched → Worse than local join
→ When Redshift Serverless is used with very small RPU allocation → Federated query overhead may be disproportionate

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

1 → THE FEDERATED QUERY EXECUTION ENGINE

User submits SQL to Redshift
    |
    ↓
Redshift Query Parser
    |
    ├── Identifies LOCAL tables (public schema, local schemas)
    ├── Identifies EXTERNAL tables (mariadb_live schema)
    └── Identifies SPECTRUM tables (spectrum schemas)
    |
    ↓
Redshift Query Optimizer
    |
    ├── Determines optimal join order
    ├── Decides what to push down to remote sources
    ├── Creates execution plan with:
    |   ├── Local scan nodes (for Redshift tables)
    |   ├── Foreign scan nodes (for MariaDB tables)
    |   └── Spectrum scan nodes (for S3 tables)
    |
    ↓
Execution Engine
    |
    ├── LOCAL scan: Executed on Redshift compute nodes
    |   → Uses columnar storage, zone maps, sort keys
    |   → Highly optimized — this is Redshift's strength
    |
    ├── FOREIGN scan: Executed via JDBC to MariaDB
    |   → Redshift leader node establishes JDBC connection
    |   → Sends pushed-down SQL to MariaDB
    |   → Receives result set over network
    |   → Result materialized in Redshift memory
    |
    └── SPECTRUM scan: Executed via Spectrum layer on S3
        → Uses Redshift Spectrum compute (from Project 28)
        → Reads Parquet/ORC from S3
        → Returns filtered results to Redshift
    |
    ↓
JOIN execution (all results now in Redshift memory)
    |
    ↓
Final result returned to user

Connection Pooling Behavior:
→ Redshift maintains a CONNECTION POOL for federated queries
→ Connections are reused across queries to the same external schema
→ Pool size is managed by Redshift internally — not configurable
→ However — during high concurrency → multiple connections opened simultaneously
→ This impacts MariaDB's max_connections limit (from Project 45)

4 → QUERY PUSHDOWN MECHANICS IN DETAIL

How Redshift Decides What to Push Down:

Original Query:
    SELECT t.transaction_id, t.amount
    FROM mariadb_live.transactions t
    WHERE t.status = 'pending'
        AND t.amount > 1000
        AND UPPER(t.merchant) = 'AMAZON'

Pushdown Analysis:
    ├── t.status = 'pending'          → Simple equality → PUSHED DOWN ✅
    ├── t.amount > 1000               → Simple comparison → PUSHED DOWN ✅
    └── UPPER(t.merchant) = 'AMAZON'  → Function on column → DEPENDS
        → UPPER() is a standard SQL function → MariaDB supports it → PUSHED DOWN ✅
        → If it were a Redshift-specific function → NOT pushed down ❌

Query sent to MariaDB:
    SELECT transaction_id, amount
    FROM payscale_prod.transactions
    WHERE status = 'pending'
        AND amount > 1000
        AND UPPER(merchant) = 'AMAZON'

Functions That ARE Pushed Down (standard SQL):

FunctionExample
UPPER, LOWERWHERE UPPER(name) = 'X'
TRIM, LTRIM, RTRIMWHERE TRIM(status) = 'active'
CASTWHERE CAST(amount AS INTEGER) > 100
COALESCEWHERE COALESCE(status, 'unknown') = 'pending'
Mathematical (+, -, *, /)WHERE amount * 1.1 > 1000
Comparison (=, >, <, >=, <=, <>)All standard comparisons
IN, BETWEEN, LIKEStandard predicates
IS NULL, IS NOT NULLNULL checks

Functions That Are NOT Pushed Down (Redshift-specific):

FunctionWhy Not
GETDATE()Redshift-specific function
DATEADD, DATEDIFFRedshift syntax differs from MariaDB
REGEXP_REPLACEDifferent regex implementations
Custom UDFsOnly exist in Redshift
Window functions (ROW_NUMBER, RANK)Complex — processed locally

→ When a function cannot be pushed down:

  • Redshift fetches ALL rows from MariaDB (or rows matching other pushable filters)
  • Applies the non-pushable function LOCALLY on the fetched data
  • This can dramatically increase data transfer and slow down the query

→ Three layers of authorization:

  1. Redshift schema permissions (who can query the external schema)
  2. IAM role permissions (what the role can access)
  3. MariaDB user permissions (what the DB user can SELECT)

→ All three must allow the request — any denial stops the query
