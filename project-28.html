PROJECT 28: THE WAREHOUSE THAT NEEDED TO QUERY THE LAKE — Redshift Spectrum Querying S3

🔴 HYPOTHETICAL PROBLEM

→ ShopNow's analytics infrastructure now has TWO powerful query engines:

Engine 1: Redshift (The Warehouse)
→ Contains 2 years of operational data loaded via COPY (Project 22)
→ Tables: orders, customers, products, payments (from MariaDB extraction — Project 24)
→ Optimized with DISTKEY, SORTKEY, compression (Project 22)
→ Sub-second queries for complex JOINs and aggregations
→ Total data: ~500 GB compressed in Redshift storage

Engine 2: Athena (The Data Lake)
→ Queries S3 data via Glue Catalog (Projects 17, 18, 25)
→ Tables: clickstream (5 years), historical orders (5 years), vendor data, log archives
→ Parquet format, partitioned, compressed (Project 21)
→ Total data: ~8 TB in S3

→ Problem 1: The JOIN That Spans Two Worlds

  • Marketing wants: "Show me customers who clicked on a product ad in the last 90 days AND made a purchase — ranked by lifetime value"
  • This requires:
    • clickstream data (S3 — 5 years, 8 TB) → ad click events
    • orders data (Redshift — 2 years, optimized) → purchase records
    • customers data (Redshift — optimized) → lifetime value calculations
  • Current approach: UNLOAD orders from Redshift to S3 → query everything in Athena
  • Problem: Athena is slow for complex window functions across 8 TB
  • Alternative: Load ALL clickstream into Redshift → 8 TB × Redshift storage pricing = $2,000/month
  • Neither option is acceptable

→ Problem 2: Historical Data Too Expensive to Keep in Redshift

  • Redshift stores 2 years of orders (500 GB compressed)
  • Business needs: 5 years of order history for compliance and trend analysis
  • Loading 5 years: ~1.2 TB compressed → requires upgrading cluster from 2 to 4 ra3.xlplus nodes
  • Cost increase: $2,500/month → $5,000/month (doubling cluster cost)
  • But 80% of queries only touch last 6 months of data
  • Keeping 5 years in Redshift means paying premium storage for rarely-queried historical data

→ Problem 3: New Data Sources Without Loading Overhead

  • Data science team produces ML model scores weekly — stored in S3 as Parquet
  • Vendor sends competitive pricing data monthly — stored in S3 as CSV
  • IoT sensors from warehouses send inventory telemetry — stored in S3 via Kinesis
  • Each new data source currently requires:
    1. Create Redshift table
    2. Set up COPY job
    3. Schedule and monitor the loading
    4. Manage storage growth
  • The data engineering team is overwhelmed with loading jobs
  • Some datasets are queried once a month — not worth the loading overhead

→ Problem 4: Athena Lacks Redshift's Advanced Features

  • Complex window functions (LEAD, LAG, NTH_VALUE, PERCENTILE_CONT) across large datasets
  • Recursive CTEs for hierarchical data
  • Stored procedures for complex business logic
  • Materialized views that auto-refresh
  • Fine-grained workload management (WLM queues — Project 16)
  • Athena is powerful but Redshift has deeper analytical SQL capabilities

→ The data engineering lead says: "We need ONE query engine that can access BOTH Redshift local tables AND S3 data lake tables — without loading S3 data into Redshift — and use Redshift's powerful SQL engine for the query execution"

What is Redshift Spectrum?
→ A feature of Redshift that extends query execution to S3
→ Redshift's SQL engine plans the query
→ For local tables: Compute nodes read from local storage (fast)
→ For external tables (S3): Spectrum workers read from S3 in parallel
→ Results are combined seamlessly — the query sees both as regular tables

STEP 1 — UNDERSTAND SPECTRUM ARCHITECTURE AND PRICING

Spectrum Execution Model:

Query: SELECT c.customer_name, SUM(o.total_amount), COUNT(cl.event_id)
       FROM local_schema.customers c                    ← LOCAL Redshift table
       JOIN local_schema.orders o ON c.customer_id = o.customer_id  ← LOCAL
       JOIN spectrum_schema.clickstream cl ON c.customer_id = cl.user_id  ← EXTERNAL S3
       WHERE cl.event_date >= '2024-10-01'
       GROUP BY c.customer_name
       ORDER BY 2 DESC
       LIMIT 100

EXECUTION:
  Step 1: Leader Node creates distributed execution plan
    ├── Identifies local tables: customers, orders
    ├── Identifies external table: clickstream (S3)
    └── Creates hybrid plan

  Step 2: Spectrum Layer handles external table
    ├── Reads Glue Catalog → identifies S3 file locations
    ├── Applies partition pruning: event_date >= '2024-10-01'
    │     → Only reads partitions from Oct 2024 to present (not all 5 years)
    ├── Spawns Spectrum workers proportional to data volume
    ├── Each worker reads Parquet files from S3
    ├── Workers apply column pruning: only reads event_id, user_id, event_date
    ├── Workers apply predicate pushdown: filters within Parquet row groups
    └── Workers stream filtered results to Redshift compute nodes

  Step 3: Compute Nodes handle local tables + JOIN
    ├── Read customers and orders from local storage (fast — DISTKEY/SORTKEY optimized)
    ├── Receive filtered clickstream data from Spectrum workers
    ├── Execute JOIN: customers ⋈ orders ⋈ clickstream (on customer_id/user_id)
    ├── Execute GROUP BY and ORDER BY
    └── Return top 100 results to Leader Node

  Step 4: Leader Node returns results to client

→ The magic: Spectrum workers do the heavy lifting of S3 scanning
→ Only FILTERED, PROJECTED results flow to Redshift compute nodes
→ Compute nodes handle the complex SQL operations (JOINs, aggregations, window functions)

STEP 5 — WRITE QUERIES JOINING LOCAL AND EXTERNAL TABLES

→ This is where Spectrum's power becomes visible
→ A single query seamlessly spans local Redshift storage and S3

Query 1: The Cross-World JOIN (Marketing's Original Request)

→ "Customers who clicked on product ads in last 90 days AND made a purchase — ranked by lifetime value"

Query Logic:

ComponentSourceSchemaLocation
Customer profilesLocal Redshift tablepublic.customersRedshift storage
Order historyLocal Redshift tablepublic.ordersRedshift storage
Clickstream eventsExternal S3 tablestaging_lake.clickstreamS3 Parquet

What Spectrum does:

Query parsed by Leader Node
       ↓
Execution plan created:
  ├── LOCAL: Scan public.customers (DISTKEY = customer_id) → fast
  ├── LOCAL: Scan public.orders (SORTKEY = order_date) → fast with zone map pruning
  └── EXTERNAL: Scan staging_lake.clickstream WHERE event_date >= 90 days ago
                 AND event_type = 'ad_click'
       ↓
Spectrum handles external scan:
  ├── Partition pruning: Only read last 90 days of partitions
  │     → 5 years of data → only 90 days scanned → 95% data skipped
  ├── Column pruning: Only read user_id, event_date, product_id columns
  │     → 30 columns in table → only 3 read → 90% columns skipped
  ├── Predicate pushdown: Filter event_type = 'ad_click' within Parquet row groups
  │     → Skip row groups where max(event_type) < 'ad_click' or min(event_type) > 'ad_click'
  └── Stream filtered results to Redshift compute nodes
       ↓
Compute nodes execute JOIN:
  ├── clickstream.user_id = customers.customer_id (DISTKEY optimization — co-located)
  ├── orders joined on customer_id
  ├── Aggregate: lifetime_value = SUM(orders.total_amount) per customer
  └── RANK by lifetime_value DESC
       ↓
Results returned: Top customers with ad click → purchase conversion

Performance characteristics:

ComponentData ScannedTimeCost

STEP 6 — OPTIMIZE SPECTRUM QUERIES

→ Spectrum charges per TB scanned — optimization directly reduces cost

Optimization 1: Partition Pruning (Most Important)

QueryPartitions ScannedData ScannedCost
No WHERE on partition columnALL partitions (5 years)8 TB$40
WHERE year = 20251 year of partitions~1.6 TB$8
WHERE year = 2025 AND month = 11 month~130 GB$0.65
WHERE year = 2025 AND month = 1 AND day = 151 day~4.3 GB$0.02

→ Always include partition columns in WHERE clause for external tables
→ This is the same principle as Athena (Project 18) — Spectrum uses Glue Catalog partitions identically

Optimization 2: Column Pruning (Parquet Benefit)

QueryColumns ReadData Scanned (1 month)Cost
SELECT * (all 30 columns)30130 GB$0.65
SELECT user_id, event_type, event_date (3 columns)3~13 GB$0.065
SELECT COUNT(*)Metadata only~1 GB$0.005

→ Always SELECT only needed columns — never SELECT * on external tables
→ Parquet's columnar format means unused columns are never read from S3

Optimization 3: Predicate Pushdown

→ Spectrum pushes filter predicates down to the S3 scan layer

Filter LocationWhat Happens
No filterSpectrum reads all rows from all files
Filter in WHERE clauseSpectrum uses Parquet row group statistics to skip irrelevant row groups
Filter on sort column of ParquetMaximum skip efficiency — row groups are organized by sort order

→ Example: Clickstream Parquet files sorted by event_date within each partition
→ Query: WHERE event_date = '2025-01-15' AND event_type = 'ad_click'
→ Spectrum:

  1. Partition pruning: Only read year=2025/month=1/day=15 partition
  2. Row group pruning: Check Parquet statistics — skip row groups where event_type min/max doesn't include 'ad_click'
  3. Column pruning: Only read needed columns

Optimization 4: File Format Impact

FormatSpectrum Scan EfficiencyRecommendation
CSV (uncompressed)Worst — scans entire file, all columns❌ Never use for Spectrum
CSV (GZIP)Better — smaller files but still row-based❌ Avoid
JSONPoor — row-based, no column pruning❌ Avoid
Parquet (Snappy)Best — columnar, compressed, statistics✅ Always use
ORC (Snappy)Excellent — similar to Parquet✅ Good alternative

→ This is why Project 21 (CSV → Parquet conversion) was critical groundwork
→ Spectrum on Parquet vs CSV: 10-50x less data scanned for typical analytical queries

Optimization 5: File Size for Spectrum

File SizeSpectrum PerformanceIssue
< 10 MBPoor — too many S3 LIST and GET requestsOverhead per file
10-128 MBGoodAcceptable
128 MB - 512 MBOptimalBest balance of parallelism and overhead
512 MB - 1 GBGoodSlightly less parallelism
> 1 GBAcceptableFewer files, less parallelism

→ Same guidance as Athena (Project 18) — Spectrum uses similar scan infrastructure
→ Glue ETL (Project 21) should produce files in the 128-512 MB range

Optimization 6: Late Materialization

→ Spectrum sends only filtered/projected results to Redshift compute nodes
→ The less data flowing from Spectrum to compute nodes → the faster the query

PatternData Flow to ComputePerformance
JOIN external with local, then filterLarge data flow → filter after JOINSlow
Filter external first, then JOIN with localSmall data flow → JOIN on filtered resultFast

→ Redshift's optimizer usually handles this — but complex queries may benefit from explicit subquery ordering
→ Pre-filter and pre-aggregate external tables before JOINing with local tables

→ Always use Glue Catalog as the metastore for Spectrum external tables — shared with Athena, Glue, EMR
→ Always include partition columns in WHERE clauses for external tables — prevents full S3 scans
→ Always SELECT only needed columns from external tables — Parquet column pruning saves 80%+ scan cost
→ Always use Parquet format for S3 data queried by Spectrum — CSV costs 10-50x more
→ Always implement hot/cold architecture — keep recent data local, historical data in S3
→ Always create UNION views that transparently combine local and external tables — seamless for analysts
→ Always monitor SVL_S3QUERY_SUMMARY — track scan efficiency and cost per query
→ Always pre-filter external tables before JOINing with local tables — reduce data flow to compute nodes
→ Always verify Spectrum IAM Role has both S3 and Glue Catalog permissions — common oversight
→ Always ensure S3 files are 128-512 MB — optimal for Spectrum parallelism
→ Always use the same Glue Catalog databases that Athena uses — single source of truth
→ Always compare Spectrum cost vs loading cost for frequently-queried external data — if Spectrum cost exceeds loading cost, load the data

❌ WHAT SHOULD NOT DO

→ Never use SELECT * on external tables — scans all columns, maximum cost
→ Never query external tables without partition filters — scans entire table history
→ Never store Spectrum data as CSV — 10-50x more expensive than Parquet
→ Never use Spectrum for sub-second latency requirements — S3 scan adds 3-10 seconds minimum
→ Never JOIN large external tables with large external tables — performance is poor; pre-aggregate one side first
→ Never create external schemas pointing to Glue Catalog databases you don't own without coordination — schema changes may break your queries
→ Never assume Spectrum is always cheaper than loading — for heavily-queried data, loading into Redshift may be more cost-effective
→ Never ignore the SVL_S3QUERY_SUMMARY table — it reveals whether optimizations are working
→ Never forget that Spectrum workers are shared infrastructure — during peak AWS usage, Spectrum may be slower
→ Never create manual external tables when Glue Catalog is available — manual partition management is error-prone
→ Never mix local and external tables with the same name in different schemas without clear naming — causes analyst confusion

  │     └── Leverage SORTKEY zone maps for filtering

        ├── Consult Glue Catalog for table metadata
        ├── Identify partitions matching WHERE clause
        ├── Generate S3 file list for qualifying partitions
        └── Create Spectrum scan requests
       ↓

  ├── Receives scan requests from Leader
  ├── Spawns Spectrum workers (auto-scaled based on data volume)

  │     ├── Medium query (1-100 GB): 5-50 workers
  │     └── Large query (100 GB+): 50-1000+ workers
  ├── Each worker:
  │     ├── Downloads assigned S3 files
  │     ├── Reads Parquet footer → identifies relevant row groups
  │     ├── Reads only requested columns from qualifying row groups
  │     ├── Applies predicate filters
  │     ├── Projects selected columns
  │     └── Streams filtered/projected results to Redshift compute nodes
  └── Workers operate in PARALLEL — more data = more workers
       ↓
COMPUTE NODES receive Spectrum results:
  ├── Treat Spectrum results like any other data source
  ├── Execute JOINs between local data and Spectrum results
  ├── Execute aggregations, window functions, CTEs
  └── Send final results to Leader Node
       ↓
Leader returns results to client

5 → DATA FLOW VOLUME ANALYSIS

External table: clickstream (5 years)
  Total data in S3: 8 TB

Query: SELECT user_id, COUNT(*) 
       FROM staging_lake.clickstream 
       WHERE year = 2025 AND month = 1 AND event_type = 'purchase'
       GROUP BY user_id

STEP 1: Partition pruning
  8 TB → year=2025/month=1 → ~130 GB (98.4% eliminated)

STEP 2: Column pruning
  130 GB → only user_id, event_type columns → ~8.7 GB (93.3% eliminated)

STEP 3: Predicate pushdown (row group filtering)
  8.7 GB → event_type = 'purchase' (5% of events) → ~0.5 GB (94.3% eliminated)

STEP 4: Spectrum sends filtered results to Redshift
  0.5 GB of (user_id, event_type) tuples → Redshift compute nodes

STEP 5: Redshift executes GROUP BY
  0.5 GB → aggregated to ~50 MB of results

FINAL: $5/TB × 8.7 GB scanned = $0.0435 per query
  (Spectrum charges for column-pruned data BEFORE predicate pushdown)
  (Partition-pruned data is NOT charged — partitions are skipped before scan)

→ Billing point: Spectrum charges for data SCANNED after partition and column pruning
→ Predicate pushdown reduces data TRANSFERRED to Redshift but not necessarily data BILLED
→ However, Parquet row group statistics CAN reduce bytes scanned (skipped row groups aren't billed)
