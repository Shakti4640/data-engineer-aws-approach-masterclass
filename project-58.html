PROJECT 58: THE ON-PREMISES MIGRATION — On-Prem MariaDB → DMS → RDS → Glue → Redshift Full Pipeline

🔴 HYPOTHETICAL PROBLEM

→ DataPulse Corp (from Projects 56 and 57) acquires a smaller analytics company called "InsightWorks"
→ InsightWorks runs a MariaDB 10.6 database on a physical server in their office data center
→ The database contains:

Table CategoryTablesRowsSize
Customer Records12 tables45 million rows120 GB
Transaction History8 tables380 million rows280 GB
Product Catalog6 tables2 million rows15 GB
Analytics Aggregates15 tables90 million rows85 GB
Total41 tables517 million rows500 GB

→ InsightWorks' application is LIVE — serving 200 concurrent users during business hours
→ The application writes approximately 50,000 INSERT/UPDATE/DELETE operations per hour during peak

Business Requirements:
→ Migrate ALL 500 GB to AWS RDS MariaDB (already set up in Project 9, secured in Project 57)
→ Maximum acceptable downtime: 30 minutes (application cutover window)
→ After migration to RDS → data must flow into existing Glue ETL pipeline (Project 24)
→ Glue transforms and loads into Redshift (Project 22) for DataPulse's unified analytics
→ Migration must be encrypted in transit (TLS) — per security policies from Project 57
→ All migration activities must be auditable via CloudTrail (Project 57)
→ Migration must be repeatable — InsightWorks has 3 more regional databases to migrate later

Technical Challenges:
→ On-premises server is behind a corporate firewall
→ No existing VPN or Direct Connect to AWS
→ Database is actively receiving writes during migration
→ Schema differences exist between InsightWorks' MariaDB and DataPulse's target schema
→ Some tables have foreign keys and triggers that complicate migration order
→ Binary logs (binlog) are enabled but in MIXED format — CDC requires ROW format
→ Character set is latin1 on some tables — target uses utf8mb4

What the team considered and rejected:
→ mysqldump export → S3 upload → RDS import

  • Rejected: Would require 6+ hours of downtime for 500 GB
    → Manual application-level data sync
  • Rejected: Error-prone, doesn't handle ongoing changes
    → Shut down application → migrate → restart
  • Rejected: 6+ hour downtime unacceptable

What they chose:
→ AWS Database Migration Service (DMS) with Full Load + Change Data Capture (CDC)
→ This allows migrating the entire database while the source continues receiving writes
→ Cutover window reduced to minutes instead of hours

Step C3 — Create Target Schema

→ Two approaches:

Approach 1: Let DMS Create Tables Automatically
→ DMS can create tables on the target with basic structure
→ Limitation: DMS creates tables with MINIMAL structure — no secondary indexes, no foreign keys, no triggers
→ Suitable for simple migrations

Approach 2: Pre-Create Schema Using SCT (Recommended)
→ AWS Schema Conversion Tool converts source schema to target-compatible DDL
→ SCT handles:

  • Data type mappings (latin1 → utf8mb4)
  • Index definitions
  • Constraint definitions
  • Stored procedure conversion (if applicable)

→ Process:

  1. SCT connects to source MariaDB
  2. SCT generates converted DDL for target
  3. Review and apply DDL to target RDS
  4. Drop secondary indexes (will be recreated after full load)
  5. Disable triggers (will be re-enabled after cutover)
  6. Keep primary keys (DMS needs them for CDC)

→ DataPulse chooses Approach 2 — pre-create schema for full control

✅ WHAT SHOULD DO

→ Always use Full Load + CDC for production migrations — minimizes downtime to minutes
→ Always test endpoint connectivity BEFORE creating the migration task
→ Always pre-create schema on target using SCT — gives you full control over schema design
→ Always drop secondary indexes before full load — recreate after load completes (10x+ speed improvement)
→ Always disable triggers on target during migration — prevents duplicate side effects
→ Always enable DMS validation — catches data mismatches automatically
→ Always configure CloudWatch logging and alarms for DMS tasks
→ Always use SSL/TLS for both source and target endpoints — encrypt data in transit
→ Always encrypt the replication instance storage with KMS
→ Always keep the source database running for 30 days after cutover — safety net for rollback
→ Always partition large tables for parallel load — dramatically reduces load time
→ Always document the migration runbook — essential for repeatable migrations
→ Always verify binlog_format = ROW before starting CDC — MIXED or STATEMENT format will cause failures
→ Always set binlog retention to cover the entire migration duration plus buffer
→ Always configure the replication instance in a private subnet — never publicly accessible
→ Always revert target parameter group to production values after migration completes

❌ WHAT SHOULD NOT DO

→ Never start migration without testing endpoint connectivity first — silent failures waste hours
→ Never use DMS to migrate stored procedures, views, or triggers — they must be migrated separately via SCT
→ Never keep secondary indexes during full load — each INSERT triggers index update → 5-10x slower
→ Never ignore CDC latency metrics — growing latency means replication is falling behind
→ Never cutover while CDCIncomingChanges > 0 — target is not caught up
→ Never decommission source immediately after cutover — always keep as rollback option
→ Never use public internet for DMS endpoints without SSL — data traverses unencrypted
→ Never run DMS replication instance with insufficient storage — if local storage fills up, task fails
→ Never skip the SCT assessment — it identifies migration issues BEFORE they become runtime failures
→ Never use binlog_format = STATEMENT for DMS CDC — non-deterministic statements cause data inconsistency
→ Never create the DMS replication instance in a public subnet — security violation
→ Never assume DMS handles all data types identically — test edge cases (BLOBs, JSON, spatial data)
→ Never run multiple large migration tasks on an undersized replication instance — resource contention causes failures
→ Never forget to re-enable foreign key checks and triggers after cutover — application behavior depends on them

STATEMENTThe SQL statement itself❌ Incompatible — non-deterministic functions (NOW(), RAND()) produce different results on target

5 → DMS Validation Internal Mechanics

VALIDATION PROCESS:
            │
Step 1:     ├── DMS selects a batch of rows from SOURCE
            │   (Using primary key range)
            │
Step 2:     ├── DMS selects the same rows from TARGET
            │   (Using same primary key values)
            │

            │   │

            │   │
            │   ├── Row exists in target but NOT in source → "Extra" finding  
            │   │

            │   │

            │   │   (Created on target database)
            │   │
            │   ├── CloudWatch Logs (if enabled)
            │   │
            │   └── DMS Console → Table Statistics → Validation columns
            │
Step 5:     └── Validation runs CONTINUOUSLY during CDC
                │
                ├── Re-validates previously mismatched rows on next pass
                │
                └── Accounts for CDC lag (waits for target to catch up
                    before comparing recently changed rows)

→ Validation does NOT compare every column of every row every time
→ It uses a hash-based comparison:

  • Computes hash of row on source
  • Computes hash of same row on target
  • If hashes match → row is validated (no column-by-column comparison needed)
  • If hashes differ → performs column-by-column comparison to identify which columns mismatch
  • This makes validation efficient even for tables with many columns

→ Validation limitations:

  • Tables WITHOUT primary keys cannot be validated (DMS needs PK to identify rows)
  • LOB columns are not included in validation by default (too large for hash comparison)
  • Validation adds ~10-20% overhead to replication instance CPU

#PitfallImpactPrevention
1binlog_format not set to ROWCDC fails completely — DMS cannot parse STATEMENT or MIXED format changesVerify SHOW VARIABLES LIKE 'binlog_format' = ROW before starting
2binlog_row_image set to minimalUPDATE events missing before-image — DMS cannot identify which row to update on targetSet binlog_row_image = FULL
3Binlog expired before CDC catches upDMS loses position in binlog → cannot continue CDC → must restart full loadSet expire_logs_days to cover migration duration + 50% buffer
4Replication instance storage fills upTask fails with storage error → CDC events lostMonitor FreeStorageSpace → set CloudWatch alarm at 20% remaining → increase storage proactively
5Source table has no primary keyDMS cannot perform CDC on tables without PK — full load works but CDC skips the tableAdd primary key to source tables before migration OR use unique index
6Foreign key constraints on target during full loadChild table loaded before parent → FK violation → task fails for that tableDisable FK checks via parameter group (Step C2) OR DMS task setting
7Triggers firing on target during full loadTrigger creates duplicate records or side effectsDisable all triggers on target before migration → re-enable after cutover
8LOB columns with "Limited LOB mode"Data truncated at LOB size limit (default 32 KB)Use "Full LOB mode" for tables with large LOB data — slower but complete
9Character set mismatch (latin1 source → utf8mb4 target)Special characters display as garbled textConfigure DMS endpoint extra connection attributes for character set conversion → validate sample data
10VPN tunnel drops during full loadDMS loses source connection → task failsUse redundant VPN tunnels → DMS retries connection → if persistent failure, task pauses and can be resumed
11Replication instance undersized for CDC change rateCDCLatencyTarget grows continuously → never catches up → cutover impossibleMonitor CDCLatencyTarget trend → if consistently increasing, upgrade replication instance class
12Source database performance degradation during full loadDMS SELECT queries compete with application queries → application slows downReduce parallel load threads → schedule full load during low-traffic hours → use read replica as source
13Cutover without verifying CDC lag = 0Target missing recent changes → data inconsistency → application errorsAlways verify CDCIncomingChanges = 0 AND CDCLatencyTarget < 5 seconds before cutover

15DMS task configured with wrong schema/table mappingWrong tables migrated OR tables migrated to wrong target schemaTest table mapping with "Premigration assessment" feature before starting full load
16Auto-increment values reset on targetNew inserts on target start from 1 instead of continuing from source's max valueDMS preserves auto-increment values during full load — but verify after migration
17Timezone mismatch between source and targetTimestamp columns shift by hours → data appears incorrectSet both endpoints to UTC via extra connection attributes
18Running migration without premigration assessmentIssues discovered during runtime → task fails mid-migration → wasted hoursAlways run DMS premigration assessment first → fix all FAILED checks

🎯 KEY TAKEAWAY

→ Database migration is a MULTI-PHASE operation — not a single step
→ DMS Full Load + CDC is the gold standard for near-zero-downtime migrations
