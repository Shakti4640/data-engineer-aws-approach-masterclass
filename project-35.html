PROJECT 35: THE ANALYST WHO WANTED TO JOIN LAKE AND DATABASE DATA — Athena Federated Query to RDS MariaDB

🔴 HYPOTHETICAL PROBLEM

→ ShopNow has matured significantly since the early projects
→ Their data architecture now has TWO distinct data stores:

Store 1 — The Data Lake (S3 + Glue Catalog + Athena)
→ Built across Projects 17, 18, 21, 25
→ Contains historical order data in Parquet format
→ Contains clickstream logs, product catalog snapshots, marketing campaign data
→ Queried via Athena using Glue Catalog tables
→ Holds 18 months of analytical data — billions of rows
→ Partitioned by year/month/day in S3

Store 2 — The Operational Database (RDS MariaDB)
→ Built across Projects 9, 14, 29
→ Contains LIVE transactional data
→ Current customer profiles, real-time inventory, active order statuses, shipping addresses
→ Updated thousands of times per minute by the application layer
→ Holds only current-state data — not historical

→ The lead analyst "Priya" gets a business request:

"Show me all customers who placed orders in the last 18 months (historical data in S3)
whose current loyalty tier is 'Platinum' (live data in MariaDB)
along with their current shipping address (live data in MariaDB)
grouped by the marketing campaign that acquired them (historical data in S3)"

→ This query REQUIRES joining data across BOTH stores simultaneously
→ Priya currently does this manually:

  • Runs Athena query on S3 → exports CSV
  • Connects to MariaDB via MySQL client → exports CSV
  • Opens both CSVs in Excel → does VLOOKUP
  • Takes 4 hours every time
  • Error-prone — mismatched customer IDs, stale exports

→ Priya asks: "Can I write ONE SQL query in Athena that joins my S3 data lake with my live MariaDB database?"

→ The answer is YES — using Athena Federated Query

🟡 HOW TO APPROACH IT

→ First — understand the fundamental limitation being overcome

Without Federated Query (What Athena normally does — Project 18):
→ Athena ONLY queries data registered in the Glue Data Catalog
→ Glue Data Catalog ONLY points to S3 locations
→ Therefore — Athena can only reach S3
→ Any data in RDS, DynamoDB, Redshift, or other stores is invisible to Athena

With Federated Query (What this project enables):
→ Athena gains the ability to query EXTERNAL data sources beyond S3
→ It does this by delegating the query execution to a Lambda function
→ That Lambda function acts as a "connector" — it knows how to talk to the external source
→ The connector translates Athena's query into the external source's language
→ Results flow back through Lambda to Athena
→ From the analyst's perspective — it's just another table in a SQL query

→ The approach:

  • Step 1 — Understand the Federated Query architecture
  • Step 2 — Deploy the MySQL connector Lambda (MariaDB is MySQL-compatible)
  • Step 3 — Configure the Spill Bucket for large result sets
  • Step 4 — Store MariaDB credentials securely (via Secrets Manager — Project 37)
  • Step 5 — Register the connector as an Athena Data Source
  • Step 6 — Write cross-source JOIN queries

STEP 2 — CREATE THE SPILL BUCKET

→ Navigate to S3 → Create a new bucket

What is a Spill Bucket and why is it needed?
→ Lambda functions have a response payload limit (6 MB for synchronous invocation)
→ When the federated connector queries MariaDB, the result set may be far larger than 6 MB
→ The connector "spills" (writes) overflow data to this S3 bucket temporarily
→ Athena reads the spilled data from S3 instead of from Lambda directly
→ Without a spill bucket → any query returning more than a few thousand rows FAILS

Configuration:

SettingValue
Bucket Nameshopnow-athena-federation-spill-prod
RegionSame region as your Athena workgroup and RDS instance
Block Public AccessAll ON (as per Project 1)
Default EncryptionSSE-S3 minimum (SSE-KMS recommended — see Project 36)
VersioningDisabled (spill data is temporary)

→ Add a lifecycle rule (as per Project 12 concepts):

  • Rule name: auto-delete-spill-data
  • Scope: Entire bucket
  • Action: Expire current versions after 1 day
  • This ensures spill data doesn't accumulate and incur storage costs

CRITICAL: Understanding the Connection String Format

→ The DefaultConnectionString has a specific syntax:

mysql://jdbc:mysql://<RDS_ENDPOINT>:3306/<DATABASE_NAME>?${SECRET_NAME}

→ Breaking it down:

ComponentPurpose
mysql://Protocol identifier for the connector
jdbc:mysql://JDBC driver protocol
<RDS_ENDPOINT>The RDS MariaDB endpoint (e.g., shopnow-db.abc123.ap-south-1.rds.amazonaws.com)

/<DATABASE_NAME>Default database to connect to
?${SECRET_NAME}Tells connector to fetch credentials from this Secrets Manager secret

→ The ${shopnow/prod/mariadb/analytics} syntax is NOT a variable substitution in the traditional sense
→ The connector Lambda code recognizes this pattern and calls Secrets Manager API at runtime
→ Credentials are NEVER hardcoded — they are fetched dynamically on every invocation

CRITICAL: Understanding the Networking Requirements

→ The Lambda connector function MUST reach two things:

STEP 7 — WRITE CROSS-SOURCE JOIN QUERIES (The actual goal)

→ Now Priya can write the query she originally needed

Conceptual Query Structure:

SELECT
    lake_orders.customer_id,
    lake_orders.campaign_name,
    lake_orders.order_total,
    lake_orders.order_date,
    live_customers.loyalty_tier,
    live_shipping.address_line1,
    live_shipping.city,
    live_shipping.country
FROM
    "AwsDataCatalog"."shopnow_datalake"."fact_orders" AS lake_orders
JOIN
    "mariadb-operational"."shopnow_prod"."customers" AS live_customers
    ON lake_orders.customer_id = live_customers.customer_id
JOIN
    "mariadb-operational"."shopnow_prod"."shipping_addresses" AS live_shipping
    ON live_customers.customer_id = live_shipping.customer_id
    AND live_shipping.is_default = 1
WHERE
    live_customers.loyalty_tier = 'Platinum'
    AND lake_orders.order_date >= DATE '2024-01-01'
GROUP BY
    lake_orders.campaign_name,
    lake_orders.customer_id,
    lake_orders.order_total,
    lake_orders.order_date,
    live_customers.loyalty_tier,
    live_shipping.address_line1,
    live_shipping.city,
    live_shipping.country

Understanding the three-part table reference:

"<DATA_SOURCE>"."<DATABASE>"."<TABLE>"
ComponentFor S3 Data LakeFor MariaDB
Data SourceAwsDataCatalogmariadb-operational
DatabaseGlue Catalog database nameMariaDB schema name
TableGlue Catalog table nameMariaDB table name

→ This three-part naming is what makes cross-source JOINs possible
→ Athena knows which engine to send each table reference to
→ It fetches data from both sources and performs the JOIN in Athena's own engine

✅ WHAT SHOULD DO

→ Always use Secrets Manager for database credentials — never hardcode in Lambda environment variables
→ Always create a dedicated spill bucket with lifecycle expiration (1 day)
→ Always deploy the connector Lambda in the SAME VPC and private subnets as RDS
→ Always configure VPC Endpoints for S3 and Secrets Manager to avoid NAT Gateway costs and latency
→ Always use the three-part naming convention in queries: "data_source"."database"."table"
→ Always push down filters to the federated source — filter MariaDB tables in the WHERE clause
→ Always test the connector with a simple SELECT LIMIT 10 before writing complex JOINs
→ Always monitor Lambda invocation duration and errors in CloudWatch (Project 19 concepts)
→ Always set appropriate Lambda memory (3008 MB recommended for large result sets)
→ Always set Lambda timeout to maximum (900 seconds) for complex queries
→ Always verify security group cross-referencing — connector SG → RDS SG on port 3306

❌ WHAT SHOULD NOT DO

→ Never use federated query as a replacement for ETL — it is for ad-hoc analytical JOINs, not bulk data movement
→ Never run federated queries that do full table scans on large MariaDB tables — this hammers the operational database
→ Never skip the spill bucket — without it, any query returning > 6 MB of federated data FAILS
→ Never deploy the Lambda connector in public subnets — it needs private subnets with VPC endpoints or NAT
→ Never forget to update the RDS security group to allow inbound from the connector Lambda security group
→ Never use the same security group for the connector Lambda and RDS — separate them for auditability
→ Never expect federated query performance to match native Athena-on-S3 performance — it is inherently slower
→ Never store the spill bucket in a different region than Athena — cross-region data transfer adds cost and latency
→ Never create multiple connector Lambdas for the same MariaDB — one connector can serve multiple databases
→ Never run concurrent heavy federated queries without monitoring MariaDB CPU — each query opens JDBC connections

ScenarioUse Federated Query?

🔴 WHEN NOT TO USE

→ When query frequency exceeds a few times per day on the same data → ETL the data to S3 instead
→ When the federated source has millions of rows and no filter pushdown is possible → Full scan kills MariaDB performance
→ When latency requirements are under 2 seconds → Federated queries typically take 5-30+ seconds
→ When the operational database is already under high load → Federated queries add concurrent connections
→ When you need to materialize results for repeated use → Use CTAS (Project 25) to save results as S3 table
→ When the data source is in a different AWS region → Cross-region JDBC connections are slow and unreliable
→ When data needs transformation before joining → ETL first (Project 21, 24), then join natively in Athena

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

MECHANIC 1 — THE LAMBDA CONNECTOR ARCHITECTURE

→ Athena Federated Query is built on the Athena Query Federation SDK
→ The SDK defines a contract between Athena and any external data source
→ This contract is implemented as a Lambda function with specific handler methods:

Handler MethodWhat Athena AsksWhat Lambda Responds
GetDataSourceCapabilities"What can you do?"Supported operations, data types
ListSchemas"What databases exist?"List of MariaDB schemas
ListTables"What tables exist in this schema?"List of MariaDB tables
GetTable"What columns does this table have?"Column names, data types, partition info
GetSplits"How should I parallelize reading this table?"Split definitions (query partitions)
ReadRecords"Give me data for this split"Apache Arrow formatted record batches

→ Athena calls these Lambda handlers in sequence during query execution
→ The MySQL connector implements all these handlers using JDBC internally

MECHANIC 2 — PREDICATE PUSHDOWN

→ This is the MOST IMPORTANT performance concept in federated queries

Without pushdown:
→ Athena asks Lambda: "Give me ALL rows from the customers table"
→ Lambda fetches 5 million rows from MariaDB
→ Athena filters them in its own engine
→ Result: Slow, expensive, hammers MariaDB

With pushdown:
→ Athena asks Lambda: "Give me rows from customers WHERE loyalty_tier = 'Platinum'"
→ Lambda translates this to a MariaDB SQL query WITH the WHERE clause
→ MariaDB returns only 50,000 matching rows
→ Lambda sends 50,000 rows to Athena
→ Result: Fast, efficient, minimal MariaDB impact

What gets pushed down:

Filter TypePushed Down?
Simple equality (column = value)✅ Yes
Range filters (column > value)✅ Yes
IN lists (column IN (a, b, c))✅ Yes
LIMIT✅ Yes
LIKE patterns⚠️ Sometimes (connector-dependent)
Complex functions (UPPER(column) = 'X')❌ No — executed in Athena
JOINs between federated tables❌ No — each table fetched separately, joined in Athena
Aggregations (SUM, COUNT)❌ No — executed in Athena

→ To maximize performance → Always filter federated tables in the WHERE clause
→ This reduces data transfer from MariaDB → Lambda → Spill Bucket → Athena

MECHANIC 3 — SPLIT PARALLELISM

→ For large tables, the connector can "split" the read into parallel chunks

How splits work:
→ Athena calls GetSplits → Connector decides how to partition the read
→ For MySQL connector — splits are typically based on:

  • Row ranges (e.g., id 1-100000, id 100001-200000)
  • Partition keys if MariaDB table is partitioned
    → Athena invokes Lambda MULTIPLE TIMES — once per split — in parallel
    → Each Lambda invocation reads its assigned split
    → Athena merges all split results

Parallelism configuration:

FactorImpact
Lambda concurrency limitLimits how many splits run in parallel
RDS max_connectionsLimits how many JDBC connections can be opened simultaneously
Athena DML query concurrencyDefault 25 concurrent DML queries per account per region

MECHANIC 4 — SPILL MECHANISM DEEP DIVE

→ Each Lambda invocation has a 6 MB response payload limit (Lambda service constraint)
→ The Athena Federation SDK handles this automatically:

Lambda starts reading from MariaDB
        ↓
Accumulates rows in memory as Apache Arrow blocks
        ↓
If accumulated data < 6 MB → return directly to Athena
        ↓
If accumulated data > 6 MB → encrypt and write to spill bucket
        ↓
Return to Athena: "Data is at s3://spill-bucket/athena-spill/<query-id>/<split-id>.spill"
        ↓
Athena reads spilled data from S3 directly

→ Spill data is encrypted using:

  • AES-GCM encryption by default (within the connector)
  • Additionally SSE-S3 or SSE-KMS if bucket encryption is configured

→ Spill data is NOT automatically cleaned up by the connector
→ This is why the lifecycle policy from Step 2 is essential

→ Important: Athena does NOT charge per-TB-scanned for federated data

📈 SCALABILITY CONSIDERATIONS

→ Query Concurrency Scaling

  • Athena allows 25 concurrent DML queries per account per region (soft limit — can request increase)
  • Each federated query may invoke Lambda 1-N times (based on splits)
  • Lambda concurrent execution default: 1,000 per account per region
  • If 10 analysts run federated queries simultaneously → 10 × N splits = many concurrent Lambdas
  • Monitor Lambda concurrent executions in CloudWatch

→ RDS Connection Scaling

  • Each Lambda invocation opens 1 JDBC connection to MariaDB
  • MariaDB max_connections default varies by instance class (typically 150-16,000)
  • 10 concurrent queries × 5 splits each = 50 simultaneous connections
  • If application already uses 100 connections → total = 150 → may hit limit
  • Solution: Use Read Replica (Project 29) as the federated query target
  • Point the connector's connection string to the Read Replica endpoint
  • This isolates analytical load from transactional load

→ Data Volume Scaling

  • Small result sets (< 6 MB) → Direct Lambda response → Fast
  • Medium result sets (6 MB – 1 GB) → Spill to S3 → Moderate
  • Large result sets (> 1 GB) → Multiple spill files → Slow, expensive Lambda compute
  • For large data volumes → Do NOT use federated query → Use Glue ETL (Project 24) to copy data to S3

→ Cost Scaling

  • Lambda cost scales linearly with query duration and memory
  • A query that runs Lambda for 60 seconds at 3008 MB ≈ $0.05
  • 100 such queries per day ≈ $5/day ≈ $150/month
  • If usage exceeds this → ETL the data to S3 and query natively (free Lambda cost)

→ Multi-Database Scaling

  • One connector Lambda can serve multiple MariaDB databases on the same instance
  • For multiple RDS instances → Deploy separate connector Lambdas (one per instance)
  • Each gets its own data source name in Athena

🎯 KEY TAKEAWAY

→ Athena Federated Query breaks the "Athena can only query S3" limitation
→ It works through a Lambda connector that acts as a translator between Athena and external data sources
→ The connector handles schema discovery, predicate pushdown, split parallelism, and spill management
→ The three-part naming convention ("data_source"."database"."table") enables cross-source JOINs in a single SQL
→ Performance depends critically on predicate pushdown — always filter federated tables
→ The connector Lambda must live in the same VPC as RDS with proper security group cross-referencing (Project 14, 24)
→ Credentials are managed via Secrets Manager (Project 37) — never hardcoded
→ VPC Endpoints (Project 27) are essential for Lambda-in-VPC to reach S3 and Secrets Manager
→ This is for AD-HOC analytics — not for production ETL or high-frequency dashboards
→ For repeated data needs → ETL the data to S3 first (Project 24), then query natively
